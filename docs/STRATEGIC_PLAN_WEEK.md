## 2. Привязка к стратегической цели

### Текущий этап дорожной карты
Мы находимся **на стыке Этапа 0 и Этапа 1** (WHITE_PAPER, раздел 8):
- ✅ **Этап 0 (фундамент)** — практически завершён:

  - `fft-analyzer` выдаёт спектр в реальном времени.
  - Клиент показывает спектр через `microphone-stream-viz`.

- 🔄 **Этап 1 (одинокий слушатель)** — начинается:
  - Нужен `drone-detector-service` для бинарной классификации «есть аэро-источник / нет».
  - UI должна показывать индикатор уверенности.
  - Цель: валидировать акустический признак в полевых условиях.

### Что приближает к цели
1. **Модульная архитектура UI** (микрофонный модуль, плагины) — позволяет быстро добавлять новые аналайзеры без переписывания App.tsx.
2. **Выделение `audio-engine` как foundation** — даёт правильную границу между инфраструктурой и специализированными анализаторами.
3. **Слабая связанность через хаб событий** (microphoneStreamHub) — соответствует принципу 2 из WHITE_PAPER (локальная автономность узла).
4. **Специализированная библиотека визуализации** — подготавливает слой presentation к сложным сценариям (треки, счётчики, карта).

### Что нейтрально (инженерное совершенство)
- CI/CD доработки, линтеризация, типизация — укрепляют основу, не приближают напрямую.
- Скрипты стратегического планирования — мета-инструмент для скоординированной работы виртуальной команды.

## 3. Риски и долг

### Технические риски

| Риск | Уровень | Суть | Что делаем |
|------|---------|------|-----------|
| **Синхронизация времени между узлами** | 🔴 High | TDOA требует точности мс; NTP даёт мс, но не µs; GPS-PPS нужна на узлах | На Этапе 2 добавить поддержку PPS в `audio-engine`; описать калибровку в device-board |
| **Качество TDOA при многолучёвости** | 🔴 High | Отражения от зданий искажают time-of-arrival; GCC-PHAT даёт ошибки | На Этапе 2 использовать медианную фильтрацию, обучать классификатор на синтетических отражениях |
| **Масштабирование UI к многим целям** | 🟡 Medium | Сейчас вся визуализация заточена на один спектр; при 100+ дронах упадёт | На Этапе 4 вести счётчик + агрегированную карту; детали по клику |
| **Потеря узла в сети** | 🟡 Medium | Один узел без связи = потеря триангуляции; нужна отказоустойчивость | На Этап 4+ добавить локальные буферы узлов и синхронизацию при восстановлении |
| **Классификация при тишине дрона** | 🟡 Medium | Электрические БПЛА на низких оборотах шумят < 60 дБ; микрофон не видит | На Этап 5 добавить второй модальный канал (RF-приёмник или оптику) |

### Накопленный долг

1. **Нет юнит-тестов для audio-engine и fft-analyzer** — основные сервисы проверяются только интеграционно через UI. Нужны unit-тесты для `LiveSampler`, `FftAnalyzer`, вычислений спектра.
2. **Отсутствует обработка ошибок getUserMedia** — если микрофон не разрешен, модуль падает. Нужна graceful degradation и UI-ошибка.
3. **Кольцевой буфер аудио не реализован** — WHITE_PAPER требует локального хранения сырого звука на узле для доказательной базы. Это критично для Этапа 1.
4. **Нет типов для наблюдений (AcousticObservation)** — в WHITE_PAPER (раздел 7) описан контракт наблюдений, но в коде его ещё нет. Нужно в `@membrana/core`.

### Нарушения границ пакетов

- **`microphone-stream-viz` зависит от `audioDataViz`** — это нарушает правило слабой связанности плагинов. Решение: `audioDataViz` — это не плагин, а shared library (OK); но плагин не должен импортировать другой плагин.
- **`MicrophoneModule` импортирует `useFftAnalyzer()` напрямую** — хорошо, это разрешено (модуль может тянуть сервисы). Но если появится второй анализатор, нужен registry, а не прямой импорт.
