# Техническое задание: установка и настройка MCP-серверов для проекта Membrana

**Проект:** Membrana — распределённая акустическая разведка нижнего неба
**Дата:** 20 мая 2026
**Версия:** 3.0 (адаптация под Membrana)
**Целевая платформа:** Windows 10/11 (основной сценарий)

> **Стратегия использования MCP в процессе разработки:** [`MCP_INTEGRATION_STRATEGY.md`](./MCP_INTEGRATION_STRATEGY.md).
> Настоящий файл — **runbook установки**; стратегия — **политика, роли, безопасность, сценарии по типам задач**.

---

## Контекст и обоснование

Membrana — TypeScript-монорепозиторий с многоуровневой архитектурой (audio-engine, семейство детекторов трёх классов, benchmark-инфраструктура). Разработка ведётся компактным core-составом 4–7 человек с опорой на дисциплину виртуальной команды специализированных агентов (стратег, архитектор, математик, верификатор). Эта модель работы означает, что инструменты агента — часть инженерного процесса; их качество прямо влияет на скорость прохождения **stage-gate** (precision ≥ 85%, recall ≥ 90% на одном узле), за которым следует переход к двухузловой локализации.

Настоящее ТЗ описывает развёртывание локальной MCP-среды для Claude Desktop и Cursor под этот сценарий работы. От стандартной установки оно отличается:

- привязкой каждого сервера к конкретной роли виртуальной команды;
- набором приёмочных тестов под текущую фазу проекта (single-node detection);
- осознанным **исключением** Chrome MCP из основного контура (модель угроз — доступ ко всем залогиненным сессиям пользовательского Chrome — несовместима с инженерной дисциплиной проекта; Playwright покрывает требуемые сценарии браузерной автоматизации в изолированном окружении);
- использованием **gitnexus** для навигации по графу кода (зависимости, цепочки вызовов, реализации контрактов) и **GlyphMCP** для генерации Excalidraw-диаграмм при подготовке заданий для агентов и внутренней документации. Эти два сервера часто путают по названию, но решают разные задачи и оба нужны.

---

## Цель

Развернуть стабильный набор MCP-серверов, обеспечивающий следующие возможности агента:

- актуальный исследовательский поиск с цитированием источников (отраслевые публикации по акустике БПЛА, спецификации YAMNet/CLAP, нормативные документы);
- автоматизация изолированного браузера для скрапинга открытых датасетов и навигации по технической документации;
- **граф-навигация по TS-монорепозиторию Membrana**: символы, цепочки вызовов, реализации интерфейсов, кластеры зависимостей;
- работа с историей репозитория: коммиты, log, diff без выхода из диалога;
- безопасный доступ к локальным каталогам с датасетами полевых записей;
- **генерация Excalidraw-диаграмм** для заданий агентам, внутренней документации и презентационных материалов из текстовых описаний.

## Решения по безопасности

### Почему Chrome MCP исключён

Chrome MCP подключается к работающему Chrome пользователя через расширение и получает доступ ко всем активным сессиям: GitHub, корпоративная почта, банковские кабинеты, любые SaaS-инструменты. Для проекта, чей публичный сигнал зрелости — инженерная дисциплина, такая модель угроз создаёт несоразмерный риск. Если в будущем появится сценарий, не покрываемый Playwright (залогиненные дашборды поставщиков железа, личные кабинеты грантовых программ), Chrome MCP добавляется отдельно — **на изолированном профиле Chrome без рабочих и финансовых аккаунтов**, с явной отметкой в этом ТЗ.

### Изоляция датасетов

Filesystem MCP конфигурируется с **whitelist каталогов** — доступ только к корню проекта Membrana и к каталогу датасетов. Сервер не получает доступа к домашнему каталогу пользователя целиком.

### Индекс gitnexus и приватность кода

gitnexus работает локально: индекс хранится в `.gitnexus/` внутри репозитория (добавляется в `.gitignore`), глобальный реестр путей — в `~/.gitnexus/registry.json`. Код в облако не отправляется. Веб-интерфейс на `gitnexus.vercel.app` — отдельный SPA, который команда не использует для инженерной работы.

### GlyphMCP: что уходит на сторонний сервер

GlyphMCP — коммерческий SaaS. При генерации диаграммы текстовое описание (а это формулировка задания для агента или фрагмент внутренней документации) отправляется на серверы `glyphmcp.dev`. Из этого следует операционное правило: **в промптах для GlyphMCP не упоминать имена потенциальных заказчиков, конкретные объекты внедрения, точные метрики, конфиденциальные параметры архитектуры**. Описания формулируются на уровне обобщённых компонентов («узел акустической разведки», «слой слияния», «оператор объекта»), без привязки к реальным контрактам и контрагентам. Архитектурно это совместимо с публичной частью документации и с уровнем детализации one-pager.

### Секреты в конфиге

`PERPLEXITY_API_KEY` и `GLYPHMCP_API_TOKEN` хранятся только в `claude_desktop_config.json` (вне git, см. Этап 9). Не вставляются в чаты, не публикуются в README. При компрометации — отозвать в личном кабинете соответствующего сервиса и перевыпустить.

## Этап 1. Подготовка runtime (Windows)

Открыть **PowerShell от администратора** и выполнить:

```powershell
winget install OpenJS.NodeJS.LTS
winget install Git.Git
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

**Перезапустить PowerShell** (важно — переменные PATH обновляются только в новой сессии), затем проверить:

```powershell
node --version
uv --version
git --version
```

**Критерий приёмки этапа:** все три команды возвращают версии без ошибок. Node должен быть ≥ 18.

## Этап 2. Получение API-ключа Perplexity

1. Зарегистрироваться/залогиниться на `perplexity.ai`
2. Settings → API → Generate Key
3. Привязать платёжный метод (требование Perplexity для API доступа)
4. Сохранить ключ формата `pplx-...` в менеджере паролей

**Критерий приёмки:** ключ получен, не закоммичен в git, не вставлен в чаты.

## Этап 3. Получение API-токена GlyphMCP

GlyphMCP — коммерческий сервис генерации Excalidraw-диаграмм через MCP. Веб-приложение для просмотра и редактирования результатов — `glyphmcp.dev`.

1. Перейти на `glyphmcp.dev`, зарегистрироваться (электронной почтой или через провайдера SSO).
2. Выбрать тарифный план. На момент подготовки ТЗ актуальные опции уточняются на сайте — для команды из 4–7 человек, использующей сервис эпизодически (задания для агентов, иллюстрации к документации), достаточен начальный платный тариф или free tier при наличии такового. Зафиксировать выбранный тариф во внутренней документации команды.
3. В личном кабинете сгенерировать API-токен (Settings → API Tokens или эквивалент).
4. Сохранить токен в менеджере паролей.

**Замечание о конфиденциальности.** См. раздел «GlyphMCP: что уходит на сторонний сервер» выше. Перед началом работы с сервисом ознакомить команду с правилом: «не отправлять в GlyphMCP имена заказчиков, конкретные метрики и иные конфиденциальные параметры».

**Критерий приёмки:** аккаунт создан, тариф выбран, токен сохранён, операционное правило конфиденциальности донесено до команды.

## Этап 4. Установка gitnexus и индексация репозитория

В исходном `.cursor/mcp.json` проекта gitnexus запускается через `npx -y gitnexus@latest mcp`. Эта форма рабочая, но имеет известное ограничение: на холодном npm-кэше первый запуск сервера может превысить дефолтный MCP_TIMEOUT (~30 секунд), особенно под Windows через `cmd /c`. Поэтому ставим пакет **глобально**.

### 4.1. Глобальная установка

```powershell
npm install -g gitnexus
gitnexus --version
```

### 4.2. Индексация репозитория Membrana

gitnexus — не «zero-config» сервер; перед первым запуском MCP нужно построить граф кода. Из **корня репозитория Membrana**:

```powershell
cd <путь_к_репозиторию_Membrana>
gitnexus analyze
```

Команда:

- проходит файловое дерево, строит AST через Tree-sitter (TypeScript/JavaScript из коробки),
- извлекает функции, классы, методы, интерфейсы и их связи,
- строит граф зависимостей в локальной KuzuDB в каталоге `.gitnexus/`,
- регистрирует репозиторий в глобальном реестре `~/.gitnexus/registry.json`.

Для среднего TS-проекта это занимает менее минуты.

### 4.3. Добавление в .gitignore

```powershell
Add-Content -Path .\.gitignore -Value "`n# gitnexus local index`n.gitnexus/"
```

Проверить, что строка добавлена, и закоммитить изменение `.gitignore` (но НЕ сам каталог `.gitnexus/`).

### 4.4. Проверка

```powershell
gitnexus status
gitnexus list
```

`status` должен показать состояние индекса для текущего репо, `list` — все индексированные репо.

**Критерий приёмки этапа:** репозиторий проиндексирован, `.gitnexus/` создан и попал в `.gitignore`, `gitnexus list` показывает Membrana.

## Этап 6. Конфигурация Claude Desktop

### 6.1. Расположение конфига

```
%APPDATA%\Claude\claude_desktop_config.json
```

Если файла нет — создать. Альтернативно: Claude Desktop → **Settings → Developer → Edit Config**.

### 6.2. Содержимое конфига (Windows)

```json
{
  "mcpServers": {
    "perplexity": {
      "command": "cmd",
      "args": ["/c", "npx", "-y", "@perplexity-ai/mcp-server"],
      "env": {
        "PERPLEXITY_API_KEY": "pplx-ЗАМЕНИТЬ_НА_РЕАЛЬНЫЙ_КЛЮЧ"
      }
    },
    "playwright": {
      "command": "cmd",
      "args": ["/c", "npx", "@playwright/mcp@latest"]
    },
    "gitnexus": {
      "command": "cmd",
      "args": ["/c", "gitnexus", "mcp"]
    },
    "git": {
      "command": "uvx",
      "args": ["mcp-server-git", "--repository", "C:\\Users\\username\\projects\\Membrana"]
    },
    "filesystem": {
      "command": "cmd",
      "args": [
        "/c",
        "npx",
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "C:\\Users\\username\\projects\\Membrana",
        "C:\\Users\\username\\projects\\Membrana\\datasets"
      ]
    },
    "glyph": {
      "command": "cmd",
      "args": ["/c", "npx", "glyph-mcp@latest"],
      "env": {
        "GLYPHMCP_API_TOKEN": "ЗАМЕНИТЬ_НА_РЕАЛЬНЫЙ_ТОКЕН"
      }
    }
  }
}
```

### 6.3. Замены перед сохранением

| Плейсхолдер                                         | Источник значения           |
| --------------------------------------------------- | --------------------------- |
| `pplx-ЗАМЕНИТЬ_НА_РЕАЛЬНЫЙ_КЛЮЧ`                    | Этап 2                      |
| `ЗАМЕНИТЬ_НА_РЕАЛЬНЫЙ_ТОКЕН` (GLYPHMCP_API_TOKEN)   | Этап 3                      |
| `C:\\Users\\username\\projects\\Membrana`           | Этап 5 (корень репозитория) |
| `C:\\Users\\username\\projects\\Membrana\\datasets` | Этап 5 (каталог датасетов)  |

**Внимание к слешам.** Windows-пути в JSON требуют **двойного обратного слеша** (`\\`). Одинарный обратный слеш — это escape-последовательность; парсер JSON падает или интерпретирует путь неверно. Альтернатива — прямые слеши (`/`), Node их принимает.

**Про gitnexus.** В отличие от исходного `.cursor/mcp.json`, где использовался `npx -y gitnexus@latest mcp`, здесь команда — глобально установленный бинарь `gitnexus`. Это устраняет риск таймаута на холодном npm-кэше. Графовый сервер один на все индексированные репо (registry global), поэтому путь к репозиторию в его args передавать не нужно — он подхватит проиндексированный на Этапе 4 проект автоматически.

**Про gitnexus и Glyph.** Несмотря на схожие задачи «дать агенту представление о коде», это **разные инструменты**:

- gitnexus — навигация по существующему коду (запросы к графу зависимостей);
- Glyph — генерация **новых** диаграмм по текстовому описанию (Excalidraw как формат вывода).

Оба идут в основной конфиг.

### 6.4. Перезапуск Claude Desktop

Правый клик по иконке Claude в системном трее → **Quit**, затем заново открыть приложение. Просто закрыть окно недостаточно — приложение продолжит работать в фоне.

### 6.5. Проверка запуска

В окне ввода Claude должна появиться иконка инструментов. Развернуть список — увидеть инструменты с префиксами `perplexity_*`, `browser_*` (Playwright), `gitnexus_*` (поиск по графу, Cypher-запросы, навигация по символам), `git_*`, файловые операции от Filesystem, и инструменты Glyph для генерации диаграмм.

При ошибках: **Settings → Developer → MCP Log** показывает stdout/stderr каждого сервера при запуске.

**Критерий приёмки этапа:** все шесть серверов видны в списке инструментов, MCP Log не содержит красных ошибок при старте.

## Этап 7. Приёмочные тесты под задачи Membrana

Выполнить в Claude Desktop по очереди. Тесты сформулированы под текущую фазу проекта (single-node detection, подготовка к stage-gate).

### Тест 7.1. Perplexity — отраслевой research

> «Используя Perplexity, найди публикации за последние 12 месяцев по акустическому обнаружению малых БПЛА: гармонический анализ несущей частоты винтов, использование YAMNet или CLAP для классификации звука дронов. Дай 3–5 источников с DOI или ссылками, краткую сводку по каждому.»

**Ожидается:** ответ с цитатами и URL, источники датированы 2025–2026 годами, упомянуты конкретные диапазоны частот или метрики.

### Тест 7.2. Playwright — открытые датасеты

> «Открой через Playwright страницу публичного датасета акустических записей дронов (например, DREGON), извлеки описание условий записи и формат файлов, сделай скриншот страницы.»

**Ожидается:** при первом запуске Playwright скачивает Chromium (~300 МБ). Затем — описание датасета и изображение. Подтверждает работу браузерной автоматизации в изолированной среде.

### Тест 7.3. gitnexus — граф реализаций контракта

> «Через gitnexus покажи все модули в репозитории Membrana, которые реализуют интерфейс детектора (наиболее похожий на `Detector` или `IDetector`). Для каждой реализации покажи, в каких файлах она используется (кто её вызывает или импортирует).»

**Ожидается:** структурированный ответ — список реализаций, граф зависимостей в обе стороны. Должны быть видны границы между DSP-детекторами, нейросетевыми и агентными. Это базовый признак того, что архитектурная дисциплина «единый контракт для всего семейства» соблюдается на уровне кода.

### Тест 7.4. Git — история изменений контрактов

> «Через Git MCP покажи последние 20 коммитов в репозитории Membrana, затрагивающие пакет с контрактами детекторов. Покажи diff коммита, в котором был зафиксирован контракт detector interface.»

**Ожидается:** список коммитов с хешами, авторами, датами, сообщениями; конкретный diff отображается. Это даёт верификатору возможность отслеживать стабильность интерфейсов между релизами.

### Тест 7.5. Filesystem — метаданные датасетов

> «Через Filesystem прочитай содержимое каталога datasets. Покажи список .wav файлов с их размерами и датами. НЕ загружай содержимое самих файлов — только метаданные.»

**Ожидается:** список файлов без попыток парсинга бинарного содержимого. Подтверждает, что доступ к датасету предоставлен корректно и ограничен whitelist'ом каталогов.

### Тест 7.6. Glyph — диаграмма для задания агенту

> «Через Glyph создай Excalidraw-диаграмму: схема обработки сигнала на одном узле акустической разведки. Слои: захват с микрофона, FFT-окно, ветка DSP-детектора (гармонический анализ), ветка нейросетевого детектора (классификация спектра), слой объединения предсказаний. Стрелки — поток данных. Используй обобщённые названия без привязки к конкретным заказчикам.»

**Ожидается:** ссылка на сгенерированную Excalidraw-диаграмму в web-приложении Glyph. На диаграмме видны указанные слои и направления потока данных. Это типовая заготовка, из которой архитектор формирует задание для агента разработки.

### Тест 7.7. Композитный тест — связка серверов под задачу stage-gate

> «Задача: подготовить отчёт о готовности к stage-gate. Через gitnexus найди все реализации интерфейса детектора в репозитории. Через Git покажи, когда каждый из них был добавлен и последний раз изменён. Через Filesystem проверь, есть ли в датасете записи, на которых эти детекторы были оценены. Сформулируй краткую сводку. Через Glyph создай итоговую диаграмму: реализации детекторов как узлы, состояние покрытия датасетом как метки.»

**Ожидается:** агент использует четыре сервера последовательно, синтезирует данные в один связный ответ и финальную диаграмму. Это рабочий сценарий верификатора и архитектора перед прохождением stage-gate.

**Критерий приёмки этапа:** все семь тестов проходят без ручного вмешательства. Композитный тест 7.7 — обязательный.

## Чего не делаем

- **Не пишем собственные `.mjs` или `.sh` скрипты-обёртки** для запуска серверов. MCP-клиент сам управляет жизненным циклом процессов через `command` + `args` в конфиге.
- **Не запускаем серверы вручную в терминале** для «теста». Они работают через stdio и ожидают подключения клиента; запуск без клиента просто повиснет.
- **Не коммитим конфиг с реальными секретами** (Perplexity, GlyphMCP).
- **Не ставим Chrome MCP** в основной конфиг (см. «Решения по безопасности»). Если в будущем понадобится — отдельная процедура с изолированным профилем Chrome.
- **Не ставим mcp-firewall** на этом этапе — он обоснован только перед выходом в production или при предоставлении доступа третьим лицам.
- **Не подключаем Filesystem MCP к корню диска или к домашнему каталогу пользователя.** Только whitelist каталогов проекта.
- **Не отправляем в GlyphMCP конфиденциальные параметры** — имена заказчиков, точные метрики, детали контрактов. Только обобщённые архитектурные описания.

## Откат и удаление

В случае проблем:

1. Закрыть Claude Desktop (правый клик по иконке в трее → Quit, не просто закрыть окно)
2. Переименовать `%APPDATA%\Claude\claude_desktop_config.json` → `claude_desktop_config.json.bak`
3. Запустить Claude Desktop — приложение стартует без MCP-серверов
4. Диагностировать причину по `MCP Log`, починить конфиг, переименовать обратно

Типовые проблемы и где их искать:

- Не находит `uv` / `uvx` — PowerShell не была перезапущена после установки uv; PATH не обновился.
- `gitnexus mcp` падает с «no indexed repos» — пропущен Этап 4.2 (индексация); запустить `gitnexus analyze` из корня репо.
- Filesystem не видит датасеты — путь в args написан с одинарными слешами или указан несуществующий каталог.
- Perplexity молчит — ключ не привязан к платёжному методу.
- Glyph падает с 401/403 — токен неверный или истёк; перевыпустить на `glyphmcp.dev`.
- Glyph падает с таймаутом — холодный `npx`-кэш; либо подождать первый запуск, либо аналогично gitnexus поставить пакет глобально (если он опубликован под этим именем) и переписать команду.
- Git MCP падает на `--repository` — путь указывает не на git-репозиторий (нет `.git` директории).

## Самоописание изменений относительно исходного ТЗ

1. Целевая ОС зафиксирована — Windows; macOS-секции убраны.
2. Состав серверов перестроен с трёх до шести: вместо изначального трио Perplexity / Playwright / Glyph-outline (`benmyles/glyph`) получилось **Perplexity / Playwright / gitnexus / Git / Filesystem / GlyphMCP (Excalidraw)**.
3. **Уточнено разделение gitnexus и Glyph** — это разные инструменты с разными задачами: gitnexus читает существующий код как граф, Glyph генерирует новые диаграммы для документации и заданий агентам. В исходном ТЗ и v2 настоящего ТЗ под «Glyph» ошибочно фигурировал `benmyles/glyph` (outline символов через Tree-sitter), что не соответствовало реальному назначению инструмента в проекте.
4. Chrome MCP **исключён**: явное обоснование в разделе «Решения по безопасности».
5. Введена таблица **привязки серверов к четырём ролям** виртуальной команды (стратег, архитектор, математик, верификатор) с описанием сценариев.
6. Приёмочные тесты переформулированы под текущую фазу проекта: вместо абстрактных «найди новости» — конкретные сценарии по акустическому обнаружению дронов, навигации по контракту детекторов, проверке стабильности интерфейсов, метаданным датасетов, генерации диаграмм заданий.
7. Добавлен **композитный тест 7.7** — связка четырёх серверов под рабочий сценарий «отчёт перед stage-gate».
8. Введено **операционное правило конфиденциальности для GlyphMCP** (Этап 3 и Этап 8).
9. Заменена форма запуска gitnexus с `npx -y gitnexus@latest mcp` (как в `.cursor/mcp.json` проекта) на глобально установленный бинарь — устраняет риск превышения MCP_TIMEOUT под Windows.

### Что осталось как заглушка или требует уточнения

- Конкретный путь к каталогу датасетов полевых записей. На текущий момент проект только подходит к этапу полевых испытаний — каталог может ещё не существовать. В этом случае `datasets/` создаётся пустым на Этапе 5, и Filesystem MCP начинает работать «в режиме ожидания данных».
- Точное имя пакета и тарифные условия GlyphMCP уточняются на `glyphmcp.dev` при выполнении Этапа 3. В конфиге зафиксирована форма `glyph-mcp@latest` как наиболее вероятная по документации сервиса; при расхождении заменить на ту, что выдаст сервис при настройке аккаунта.
- Имя ветки/директории внутри Membrana с пакетом контрактов детекторов — указано обобщённо. Перед выполнением тестов 7.3 и 7.4 уточнить точное имя по `ARCHITECTURE.md`.
- Конкретный публичный датасет для теста 7.2 (DREGON упомянут как ориентир) — выбрать актуальный на момент выполнения.
