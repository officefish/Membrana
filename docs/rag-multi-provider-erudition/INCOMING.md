## Суть идеи (формулировка автора)

> Voyage AI работает как альтернатива OpenAI когда тот заблокирован,  
> а nomic-embed — это экспериментальный RAG, который работает для прикладной задачи:  
> обслуживает эрудицию каждого отдельного участника виртуальной команды.

---

## Два независимых контура

### Контур 1 — Voyage AI как fallback для OpenAI

- RAG уже поддерживает `RAG_EMBEDDING_PROVIDER=voyage` (адаптер есть)
- Идея: при недоступности OpenAI (geo-block, quota, outage) автоматически или вручную переключаться на Voyage AI
- Модели сопоставимы: `voyage-3-lite` ↔ `text-embedding-3-small` (1536-dim)
- Индексы совместимы при одинаковой размерности

### Контур 2 — nomic-embed как локальный per-member эрудит

- `nomic-embed-text` (ollama) — локальная модель, без API-ключей, без quota
- Каждый участник виртуальной команды (Vesnin, Ozhegov, Dynin, Kuryokhin, Rodchenko) имеет **собственный LanceDB-индекс** со своим корпусом знаний
- При вызове агента через consilium/VIRTUAL_TEAM_PROMPT — подгружается его персональный индекс
- Экспериментальный контур: не заменяет основной RAG, работает параллельно

## Что нужно оценить агенту

1. **Feasibility Контура 1**: нужен ли runtime-fallback или достаточно конфига `.env`? Как детектировать недоступность OpenAI?
2. **Совместимость индексов**: можно ли использовать один LanceDB-файл для обоих провайдеров, или нужны отдельные namespaces?
3. **Архитектура per-member индексов**: как связать `memberId` → `lanceDbPath` → `nomic-embed` в текущем `RagService`?
4. **Corpus per member**: откуда берётся корпус знаний каждого участника? Какие источники, как обновляется?
5. **Интеграция с consilium**: как RagService вызывается при генерации голоса участника в виртуальной команде?
6. **Производительность**: nomic-embed через ollama — latency при query? Приемлемо для интерактивного consilium?
