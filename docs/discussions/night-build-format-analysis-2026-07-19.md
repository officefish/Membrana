# Разбор формата Night Build — заметки к обсуждению (19.07)

> Не вердикт. Сырьё для отдельного разговора с владельцем.
> Повод: каскадное приземление висящих night-triage PR (#481→#527→#602→#638).

## Что есть сейчас (канон)

Носитель: [`docs/NIGHT_SPRINT_REGULATION.md`](../NIGHT_SPRINT_REGULATION.md) + скилл `membrana-night-sprint`.

| Слой | Суть |
|------|------|
| Несущий инвариант | G1∧G2∧G3: always-yes deny / worktree / ветка от `origin/main` — **техническая невозможность** вредить общему состоянию |
| Жизненный цикл | evening → `night:open` → NB0…NBn + checkpoint → `night:close` → утро: HANDOFF → review/merge |
| Вес кванта | W = R+C+D; каденс чекпоинта по классу XS/S/M |
| Тест-гейт | класс правки (`code`/`config`/…) определяет **какой** гейт, не «любой тест» |
| Процессное следствие | «ночь не мёржит» — **норма**, не hard-гейт (следствие изоляции) |

Параллельно живёт **другой** ночной контур: `night-triage` / `night-hunt` на office — cron → отчёт → **draft PR**. Это не Night Build эпик.

## Что показала очередь triage-PR

Четыре PR (14–18.07): каждый = **один** markdown-отчёт, CI зелёный, MERGEABLE, на main файлов не было. Они висели днями при готовом merge.

Наблюдения:

1. **Контур triage умеет производить, но не умеет закрывать цикл.** «sink not source» правильный для детекции; для артефакта в репо нужен явный **утренний sink** (человек/каскад/auto-merge docs-only).
2. **Night Build и night-triage путаются в языке.** Оба «ночные», оба дают PR, но разный контракт: эпик с фазами vs рутинный отчёт. В очереди PR это выглядит одинаково («Night triage…»), и morning review их не подхватывает.
3. **Каскад вручную trivial** (разные файлы → нет конфликтов), но регламент Night Build на это не заточен: нет фазы «приземлить накопившиеся docs-only draft PR».
4. **CI на момент открытия зелёный ≠ CI сейчас.** Для docs-only риск низок; для code-night PR каскад без rebase на свежий main опасен. Формат должен различать классы PR ночи.

## Напряжения формата (к обсуждению)

### A. Два ночных жанра под одним словом

| | Night Build | Night Triage / Hunt |
|--|-------------|---------------------|
| Цель | замороженный эпик разработки | регулярный срез/охота |
| Выход | ветка `night/<epic>-<date>` + HANDOFF | draft PR с отчётом |
| Merge | утро, человек | **не специфицирован** → очередь |
| Scope | NB-фазы | один файл, sink not source |

Вопрос: завести общий зонтик «ночные артефакты» с двумя подтипами и общим утренним приёмником?

### B. Утро как единственный merge-шлюз — перегруз

Инвариант «ночь не мёржит» защищает main. Цена: docs-only отчёты копятся и вытесняют внимание. Варианты:

1. **Утренний каскад-ритуал** — шаг в `ritual:day`: список open `night-triage/*` / `docs-only` → squash по дате.
2. **Авто-merge docs-only** при зелёном CI + path-filter `docs/reports/night-triage/**` (жёсткий allowlist).
3. **Не PR, а commit в protected branch через office** с подписью / отдельно от GitHub PR (меняет модель).

### C. HANDOFF vs draft PR

Night Build требует HANDOFF; triage даёт PR без HANDOFF и без записи в morning plan. Hermes уже перечисляет эти PR как «висят» — сигнал есть, **действия нет**.

Вопрос: должен ли `yarn ritual:day` / `main-day-issue` иметь обязательный блок «ночные draft PR: merge / defer / close»?

### D. Каскад как формат приёмки

Сегодняшний каскад (ревью заголовка → squash oldest-first → delete branch) ближе к **утреннему batch-приёмнику**, чем к Night Build. Имеет смысл описать его отдельно:

```text
NB-land-0  инвентарь open night-* PR (класс: docs-report | code-epic)
NB-land-1  docs-report: secret-scan path + squash cascade по дате
NB-land-2  code-epic: rebase origin/main + adversarial review + один merge
NB-land-3  запись в archive/night-build/<date>/LANDING.md
```

Не смешивать с `night:open` ночного эпика.

### E. Что в формате Night Build сильно

- G1/G2/G3 как несущие гейты (не «дисциплина мёржа»).
- Разделение вес W vs класс правки/гейт (M2/M3) — лечит фиктивные тесты.
- Делегирование субагенту + утренняя адверсариальная проверка (ADR-0009).

### F. Что слабо / дыряво относительно triage-очереди

- Нет контракта на **lifecycle draft PR после зелёного CI**.
- Нет различия «отчётный PR» vs «эпический PR» в UI/ритуале.
- `night:open` не знает про чужие висящие night-triage PR (соседи есть в Hermes, не в гейте открытия).

## Предлагаемые развилки (не решать здесь)

1. Docs-only night-triage: auto-merge vs утренний каскад-шаг ритуала.
2. Переименовать/пометить PR title префиксом `[report]` / `[night-build]` для фильтра.
3. Нужен ли `yarn night:land-reports` как детерминированный каскад.
4. Связка с `night-build-format-v2` (active в реестре) — туда ли эти выводы.
