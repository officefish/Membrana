**Вопрос:** Визуальный скриптинг будет осуществляться через раздел scenarios клиентского приложения. В нем пользователь прогрмаммы сможет составлять разные сценарии обработки входящих сигналов с помощью нод, которые он будет соединять друг с другом

**Ответ:**

Уточняю несколько моментов, чтобы правильно выстроить архитектуру:

1. **Где живёт граф нод?** Состояние графа (ноды, рёбра, сохранённые сценарии) — в сторе клиента (`@membrana/agenda` или отдельный `scenarios`), или это отдельный сервис в `packages/services`?

2. **Исполнение графа — синхронно или через event-driven?** То есть, когда сигнал приходит на вход сценария, ноды срабатывают цепочкой, или каждая нода — отдельный плагин/сервис, который слушает шину?

**Предварительный каркас (если я угадал правильно):**

- **`packages/services/scenarios`** — фасад для CRUD сценариев и их компиляции в граф.
- **Store `scenarios`** — список сценариев, текущий граф (ноды/рёбра), активный сценарий.
- **`packages/plugins/scenarios-viz`** — UI с drag-and-drop нод в приложении.
- **`packages/plugins/scenarios-executor`** — исполнитель: перевод графа в цепь обработчиков и запуск при входящем сигнале.
- **Типы нод** — в отдельном `packages/types/scenario-nodes` (FFT, фильтр, громкость и т.д.).

Ноды, которые обрабатывают звук (FFT, фильтры), должны делегировать вычисления математику, а не дублировать его код.

Правильно ли я понял логику? Тогда спроектирую структуру по ARCHITECTURE.
