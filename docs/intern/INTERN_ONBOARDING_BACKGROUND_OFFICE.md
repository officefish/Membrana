# Входящий документ: онбординг в `background-office`

**Дата:** 27 июня 2026 · **Версия:** 0.2 · **Статус:** рабочий
**Кому:** второй разработчик (парт-тайм), стартовый набор задач

> Документ вводит в работу над сервисом `background-office`. Три стартовые задачи
> идут вертикалью: проверяем внешние каналы → выставляем готовность наружу →
> прогоняем по проверенному каналу первую реальную джобу. Каждую задачу можно
> сдать отдельной PR.

---

## 1. Контекст

**Membrana** — распределённая сеть акустического обнаружения малых БПЛА.
Тебе для старта важна не вся система, а один её сервис.

**`background-office`** — NestJS-сервис (порт 3000), отвечающий за интеграции:
Claude, Linear, GitHub. Он **stateless** и поднимается локально без хостинга.

Границы, которые нельзя нарушать:

- В office **не** кладём хранение аудио и пользовательские шаблоны — это
  `background-media`.
- В office **не** кладём auth, мембраны, ключи пользователей — это
  `background-cabinet`.
- office — это data-plane интеграций, и только.

Монорепо: Turborepo + **Yarn 4** (через Corepack), TypeScript.

## 2. Перед стартом

### 2.1 Доступ к репозиторию

У тебя **write** в `officefish/Membrana`, но `main` защищён ruleset'ом: прямой
push в него блокируется, всё идёт через фиче-ветку и Pull Request с ревью
куратора. Точная последовательность команд — в 2.4. Главное правило:
**одна ветка на задачу**.

### 2.2 Свои ключи (не копии чужих)

Ты работаешь под **своими** ключами — так каждый запрос атрибутируется к тебе,
а лимит ограничивает риск:

- **Perplexity (Sonar)** — свой ключ, нужен для задачи 3 (дайджест).
- **Anthropic** — тебе заводят отдельный workspace в Console со своим ключом и
  месячным spend-лимитом; нужен для проверок в задачах 1–2.
- **GitHub** — свой аккаунт.

Правила: ключи только в `.env` или секрет-хранилище, **никогда** в репозитории;
никогда не использовать продакшен-ключ проекта.

### 2.3 Локальный запуск

```bash
# Yarn 4 через Corepack (в VM может стоять Yarn 1.x)
corepack enable
corepack prepare yarn@4.5.0 --activate

# Зависимости
yarn install

# Свой .env рядом с примером: ANTHROPIC_API_KEY, PERPLEXITY_API_KEY,
# GITHUB_TOKEN, порты. Сервис без них должен падать внятно (см. задачи).

# Поднять office (порт 3000)
yarn office:dev
```

Хостинг для разработки не нужен: office stateless, гоняешь его у себя.

### 2.4 Git-процесс: точная последовательность

Правило простое: **одна ветка на задачу**, имя ветки — по задаче. В `main`
напрямую не коммитим никогда.

**Один раз — настройка:**

```bash
gh auth login                 # авторизуйся своим GitHub-аккаунтом
git clone https://github.com/officefish/Membrana.git
cd Membrana
```

**На каждую задачу — с начала:**

```bash
# 1. Стартуй всегда со свежего main
git checkout main
git pull

# 2. Заведи ветку под задачу (имя — по задаче)
git checkout -b task-1-self-check      # см. имена ниже

# 3. Работай, потом коммить
git add -A
git commit -m "task 1: outbound self-check"

# 4. Запушь ветку
git push -u origin task-1-self-check

# 5. Открой PR в main
gh pr create --base main --head task-1-self-check \
  --title "Task 1: outbound self-check" --fill
```

**Когда ревью просит правки — в ту же ветку, НЕ вторую PR:**

```bash
git add -A
git commit -m "review: исправления"
git push                       # PR обновится сам
```

**После мержа — следующая задача с начала:** снова `git checkout main && git pull`,
затем новая ветка.

**Жёсткие правила (чтобы не напутать):**

- Одна ветка = одна задача. Не смешивай задачи в одной ветке.
- Имена веток: `task-1-self-check`, `task-2-health-ready`, `task-3-research-digest`.
- Всегда начинай ветку со свежего `main` (`checkout main && pull`).
- Никогда не коммить в `main` напрямую. Если push в `main` отлетел — это
  ожидаемо: ты должен быть на ветке задачи.
- Правки по ревью — новые коммиты в ту же ветку, не новая PR.
- Не делай force-push.
- Секреты не коммить; `.env` в `.gitignore`.

**Если ошибся:**

- _Случайно закоммитил в `main` локально (ещё не запушил):_
  ```bash
  git switch -c task-x          # перенести коммиты на ветку
  git checkout main
  git reset --hard origin/main  # вернуть main в чистое состояние
  ```
- _PR показывает конфликты / ветка отстала от `main`:_
  ```bash
  git checkout main && git pull
  git checkout task-x
  git merge main                # разрешить конфликты, затем commit + push
  ```

## 3. Стартовые задачи

Формат каждой: **цель — объём — критерий готовности — не-цели**. Не-цели важны
не меньше целей: они держат задачу в размере.

### Задача 1. Outbound self-check

**Цель.** Переиспользуемая диагностика: пингует внешние эндпоинты, от которых
зависит office, и отдаёт по каждому статус и латентность. Превращает разовый
ручной `curl` в инструмент.

**Объём.**

- Проверяет достижимость и латентность: `api.anthropic.com`, Linear API,
  GitHub API, Perplexity (`api.perplexity.ai`).
- Отдаёт компактную сводку: эндпоинт / достижим / латентность / код ответа.
- Недостижимый эндпоинт обрабатывается аккуратно (таймаут, не виснет).
- Форма — скрипт (`yarn`-команда); опционально та же логика переиспользуется в
  задаче 2.

**Критерий готовности.** Запуск даёт читаемую сводку по всем четырём
эндпоинтам; недостижимый эндпоинт не роняет проверку, а помечается.

**Не-цели.** Не чинить связность; не делать ретраи фичей; не слать алерты; не
ставить на расписание.

### Задача 2. `/health` и `/ready`

**Цель.** Базовая наблюдаемость для сервиса, который пойдёт в фоновые джобы.

**Объём.**

- `/health` — живость: процесс поднят, отвечает быстро, ничего наружу не дёргает.
- `/ready` — готовность: доступны ли вниз по течению Claude / Linear / GitHub /
  Perplexity. Переиспользует логику проверки из задачи 1.
- Внятный формат ответа (что готово, что нет).

**Критерий готовности.** Оба эндпоинта отвечают; `/ready` отражает реальное
состояние зависимостей; поведение кратко задокументировано.

**Не-цели.** Никакого стека метрик и дашбордов; никакой авторизации (это office,
не cabinet); readiness не «чинит», только сообщает.

### Задача 3. Ресёрч-дайджест — «утренняя пресса на стол»

**Ведущий образ.** Это не отчёт о работе, а свежая пресса к утру: директор за
минуту видит, что сдвинулось в индустрии, и не тонет во вкладках. Регистр —
секретарский, не инженерный. Сжато, по убыванию важности, с источником под
каждым пунктом — чтобы при желании «развернуть газету» и прочитать целиком.

**Цель.** Периодическая обоснованная заметка, синтезирующая внешние новости по
одной теме, релевантной Мембране, со ссылками. Питает `analyzers:research:week`.

**Объём.**

- **Вход:** одна тема из конфига (например, «открытые модели аудио-классификации»
  или «акустическое обнаружение малых БПЛА»). Начинаем с одной, не динамической.
- **Запрос:** Perplexity Sonar, с обязательными источниками в ответе.
- **Артефакт:** markdown по фиксированному пути `docs/research/digest-YYYY-MM-DD.md`.
  Короткий синтез + список ссылок.
- **Выжимка, а не подшивка:** жёсткий потолок 5–7 пунктов, 1–2 предложения на
  каждый, по убыванию значимости. Принёс Sonar двадцать — твоя работа отобрать
  семь. Редактура — часть задачи.
- **Подпись секретаря:** у каждого пункта источник и дата. Что нельзя сослать —
  в газету не идёт.
- **Каденс:** пока вручную, одной `yarn`-командой. Без планировщика.
- **Идемпотентность:** повтор за ту же дату — переписать либо пропустить; флаг
  `--dry-run`.

**Критерий готовности.** Команда создаёт файл по пути; в нём ≤ 7 пунктов по
убыванию важности, у каждого источник и дата; повторный запуск идемпотентен;
`--dry-run` работает.

**Не-цели.** Никакого расписания; никакой мультимодельности; никакой записи
обратно в Linear/GitHub; никаких действий. Только read → synthesize → write-file.

## 4. Как сдавать

Точные команды — в 2.4. Кратко:

- Одна задача — одна ветка — одна PR в `main`.
- В PR — короткое описание: что делает, как проверить локально.
- Ревью обязательно; правки по комментариям идут в **ту же** ветку и PR.
- Секреты не коммитим; в репозитории только `.env.example`.

## 5. Порядок и приоритет

1. Задача 1 — самая изолированная, с неё начни.
2. Задача 2 — переиспользует проверку из задачи 1.
3. Задача 3 — первая «настоящая» джоба, опирается на проверенный канал.

Что дальше (не сейчас, для ориентира): расписание и хостинг дайджеста — фаза два,
когда джоба обкатана; мультимодельность (OpenRouter) — отдельный апгрейд, когда
понадобится сравнивать модели, а не синтезировать прессу.
