# Паттерн: версионирование по пинованному подграфу

Статус: **рекомендация** (SHOULD). Родился 2026-07-21 из беседы владельца и агента
о китах (issue [#761](https://github.com/officefish/Membrana/issues/761)) — семя и
первый адресат паттерна. Применяется к китам **с момента их появления** (решение
владельца 21.07).

**Область vs контейнер** (T9 / m4 · #930):

1. **Список/реестр контейнера** ([`GROUP_CONTAINERIZATION`](./GROUP_CONTAINERIZATION.md)) —
   снимковая дисциплина (overwrite-реестр + Meta + dated-опционально) — **вне
   паттерна**.
2. **Сабграф ПОД контейнером** (провода, отрезки, поддерживающий код) — **законный
   адресат** паттерна.

Оси ортогональны: контейнеризация — про *пространство* (где живут артефакты, кто
может писать), версионирование — про *идентичность во времени* (что считать той же
вещью, когда она воспроизводима), а операции над домом — про *чем работают*
([`HOME_WORKSHOP`](./HOME_WORKSHOP.md), третья ось).

### Доменные имена родового узора

| домен | слой | что пинится | механизм |
|-------|------|-------------|----------|
| **кит** | скрипты (`kits/`) | файл целиком (git SHA) | `auditPins` + file-resolver (`kits:audit`) |
| **фрейм** | процедуры (`docs/procedures/`) | отрезок (git SHA файла + `segmentHash` + якорь) | `auditPins` + `resolveSegment` |

*Новый домен заводит своё имя только при новой гранулярности пина* (файл — кит,
отрезок — фрейм; третий домен — только при третьей гранулярности).

## Контекст

Переиспользуемая единица собирается из связной группы файлов: набор скриптов под
задачу (кит) тянет транзитивные импорты `lib/*.mjs` и конфиги. Единицу потребляют
процессы, требующие воспроизводимости, — прежде всего автономные прогоны (ночной
спринт, cron, office), где некому заметить, что зависимость «уехала» вчера вечером.

## Проблема

Три наивных подхода к версии — три знакомых дефекта:

1. **Пин одного файла** — иллюзия чистоты: поведение определяется замыканием по
   подграфу зависимостей. Файл заморожен, `lib/…` уехал — «чистая функция»
   изменилась, не изменившись.
2. **Живые ссылки (виртуальный неймспейс без версий)** — действие на расстоянии:
   правка под нужды одного потребителя молча меняет поведение другого; поведение
   единицы меняется без изменения её манифеста («объявленное ≠ работающее»).
3. **Копии** — stale-dist на новом уровне: протухшие дубли, бэкпорт фиксов в N мест
   (живой прецедент — гейт cg6).

## Решение

**Значение без копий — пины через git:**

- **Единица версии — связный подграф**, не файл: корневой скрипт + транзитивные
  импорты + читаемые конфиги. Подграф перечислен **манифестом**: `путь → SHA`
  (git-объекты уже иммутабельны — хранилище остаётся виртуальным, семантика по
  значению).
- **Полнота проверяется через SHA**: аудит строит фактический граф зависимостей
  корня и сверяет с манифестом. Недостающий узел или уехавший SHA — находка,
  инструмент той же audit-семьи (по духу `tasks:audit` / decompose «вне категорий»).
- **Два режима исполнения:**
  - *latest* (dev) — интерактивные сессии, владелец рядом: скорость фиксов важнее;
  - *pinned* (release) — автономные прогоны: **только от пина**.
- **Обновление пина — осознанный акт**: отдельный диффабельный, ревьюируемый
  коммит манифеста; не побочный эффект правки скрипта.
- **Владелец пина назван** в манифесте (домен или сам кит) — от этого зависит,
  чьим инструментом ловится дрейф «манифест ↔ реальность».

## Чеклист соответствия

Реализация ссылается на этот паттерн и копирует чеклист в свой README, заменяя
каждый `[ ]` на ✅ либо ⚠ с причиной отступления:

1. [ ] Единица версии — подграф, перечисленный манифестом (корень + импорты + конфиги).
2. [ ] Пины без копий файлов: **кит** — git SHA файла; **фрейм** — git SHA файла + `segmentHash` отрезка + якорь (`segmentHash` = дайджест, не копия).
3. [ ] Есть аудит полноты: фактический граф зависимостей == манифест; расхождение всплывает, не прячется.
4. [ ] Режимы latest/pinned разделены; автономное исполнение — только от пина.
5. [ ] Обновление пина — отдельный ревьюируемый коммит, не побочный эффект.
6. [ ] Владелец пина назван в манифесте.
7. [ ] Дрейф ловится инструментом audit-семьи с табличным выводом.

## Границы применимости

- **Список/реестр контейнера — вне области** (T9): снимковая дисциплина достаточна;
  сам контейнер как список этим паттерном не версионируется.
- **Сабграф под контейнером — в области**: поддерживающая структура (в т.ч. фреймы
  процедур с `segmentHash`) — законный адресат; см. таблицу доменных имён выше.
- Интерактивная сессия с владельцем рядом → latest, пин избыточен.
- Маленький одноразовый скрипт без потребителей → не пиновать.
- Пин ≠ замена тестам: SHA / `segmentHash` гарантируют идентичность входов, не корректность поведения.
