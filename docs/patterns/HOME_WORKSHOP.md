# Паттерн: домашняя мастерская

Статус: **рекомендация** (SHOULD). Родился 2026-07-22 из заседания
[`home-workshop`](../meeting/home-workshop/EPIC.md) (пять вердиктов Ф1–Ф5,
ратифицированы владельцем), обобщившего две независимо построенные мастерские —
контейнера веток ([`docs/audit/git/`](../audit/git/README.md)) и контейнера задач
([`docs/audit/tasks/`](../audit/tasks/README.md)). Вход — шторм
[`storm-home-workshop-2026-07-22`](../storm/storm-home-workshop-2026-07-22/REPORT.md).

## Три оси (навигатор)

Паттерн — третья ось родового узора «дом и его оснастка». Оси ортогональны; каждое
правило живёт в одном документе, соседи держат только ссылку.

| Ось | Паттерн | Про что |
|-----|---------|---------|
| **Пространство** | [`GROUP_CONTAINERIZATION`](./GROUP_CONTAINERIZATION.md) | где живёт группа, кто пишет, что коммитить |
| **Время / идентичность** | [`PINNED_SUBGRAPH_VERSIONING`](./PINNED_SUBGRAPH_VERSIONING.md) | какая ревизия оснастки, пины подграфа (кит) |
| **Операции** | **этот паттерн** | чем над домом работают: осмотр, декомпозиция, рассмотрение |

## Контекст

У дома (контейнера) накапливается оснастка — команды осмотра, раскладки, разбора его
содержимого. Оснастка расползается по скиллам и скриптам без общей формы; работая с
двумя домами, агент переносит на второй привычки первого, а инструмент случайно
садится не в тот дом (прецедент — [#915](https://github.com/officefish/Membrana/issues/915)).

## Проблема

Без выделенной формы «набор инструментов дома» порождает дефекты:

1. **Инструмент без адреса** — непонятно, чьей мастерской он принадлежит; обзор,
   работающий над одним домом, оседает в мастерской другого (мис-филинг).
2. **Плоское мышление на неплоском доме** — инструменты, заточенные под одномерный
   список, ломаются на доме, чей элемент сам контейнер (двумерный дом).
3. **Немой отказ оснастки** — недоступный кит или потерянный якорь дают тихо пустой
   результат вместо видимой ошибки.

## Решение

**Мастерская — именованный набор инструментов, поставляемый для работы с ОДНИМ домом.**
Мастерская — сторона **спроса** (заказчик операций), в отличие от контейнера (дом,
пространство) и кита (поставка, `PINNED_SUBGRAPH_VERSIONING`).

### Контракт (Ф1)

Мастерская несёт манифест `workshop.manifest.json` в доме контейнера:

| Поле | Смысл | Кратность |
|------|-------|-----------|
| `pattern` | ссылка на этот документ | 1 |
| `name` | имя мастерской | 1 |
| `worksOn` | дом, над содержимым которого работает | **ровно 1** (иначе не мастерская) |
| `verbs` | словарь инструментов (см. ниже) | 1 |
| `kit` | заказанный кит (`kits/<id>`) или `null` | 1 (значение опционально) |

Стрелки: `контейнер-скриптов —поставляет→ кит —заказывает← мастерская —worksOn→ дом`.
`worksOn` и `kit` — наши; `supplies`/`contains` — ссылки на `GROUP_CONTAINERIZATION` /
`PINNED_SUBGRAPH_VERSIONING`, без копии правил. Недоступен кит → мастерская `unequipped`
(видимо, без тихой подмены соседним).

### Словарь инструментов (Ф2)

**Каноническая тройка инвентаря** (различны по арности) — ориентир audit-family:

| Глагол | Тип | Смысл |
|--------|-----|-------|
| `audit` | `House → Report` | осмотр: что в доме и в каком состоянии |
| `decompose` | `House → Element[]` | раскладка по удобным ЭТОМУ дому правилам |
| `inspectElement` | `Element → Detail \| recurse` | рассмотрение одного элемента вглубь |

- **Покрытие дома инвентарём — MUST на уровне дома, не каждой мастерской.**
  `audit` и `decompose` обязаны существовать *для дома*: либо как `verbs` **этой**
  мастерской (классика audit-family: git, `docs/audit/tasks`, bestiary, precedents,
  procedures, llm-calls), либо **вынесены вовне** — производная мастерская, CI-гейт
  или парный audit-контур — с явной пометкой ⚠ и указателем куда (не молча). Ключи
  `verbs.audit` / `verbs.decompose` в манифесте остаются (контракт зуба): значение —
  непустая строка (**✅ в этой мастерской**) или `null` (**⚠ вне этой мастерской**).
  Вердикт владельца 23.07 (g0 / [#1056](https://github.com/officefish/Membrana/issues/1056),
  **V2 wins**): первичная мастерская `docs/tasks` несёт decision-verbs; `audit` /
  `decompose` живут вне неё (соседний контур `docs/audit/tasks` + CI), вместе с
  писателями / sync / `start`. Живые дома с `audit`+`decompose` в своих verbs **не**
  становятся невалидны — это по-прежнему нормальный (и предпочтительный) профиль.
- **`inspectElement` — SHOULD** (его отсутствие — ⚠ с причиной, не провал; шов
  вердиктов Ф2↔Ф5 разрешён Ф5 в пользу SHOULD, т.к. у живущих мастерских его пока
  нет — достройка отдельной задачей).
- **Стек-подобный слой — SHOULD-ориентир, тождественный но НЕ идентичный.** Инструменты
  *соответствуют* `map`/`filter`/`reduce` по роли (гомоморфизм смысла), но не обязаны
  реализовывать эти сигнатуры (не MUST-интерфейс). Рекомендация — сходство, не контракт.
- **Декомпозиция НЕ поглощается стек-примитивами** (`domainPreserved`): доменная
  раскладка (ветки — 7 категорий гигиены, задачи — префиксы конфига) остаётся своим
  глаголом, её не переписывают через `reduce`.
- **Класс `domain` открыт**: специализированные инструменты сверх инвентарного
  минимума (в т.ч. decision-verbs вроде `list` / `board` / `bookkeeping` / `reviewing`),
  каждый несёт `worksOn`.

### Полиморфизм и рекурсия в двумерном доме (Ф3)

Дом, чей элемент сам контейнер (процедурный — 2D-массив), обрабатывается полиморфно:

- **Диспетчер — общий интерфейс, не свитч**: род элемента приносит свою реализацию
  (живёт в ките рода); мастерская зовёт по интерфейсу, не зная чей кит. Новый род не
  трогает мастерскую (открыто для расширения).
- **База рекурсии — якорный документ**: `inspectElement`/`decompose` спускаются по
  вложенным домам, пока элемент — контейнер; упёрлись в якорный документ (не контейнер) —
  терминал. Конечность гарантирована фундированностью (строго убывающая глубина на
  конечном дереве).
- **Отказы видимы**: `anchor-missing` — находка обзора (не гейт, легально для
  недостроенного дома); `kit-missing` — жёсткий стоп, не тихо пустой обзор.
- Операции обзора — **чтение**, идемпотентны.

### Принадлежность инструмента (Ф4)

Правило: **`owner(tool) = workshop(worksOn(tool))`** — инструмент принадлежит мастерской
того дома, над содержимым которого работает. Зуб `checkOwnership` (блокирующий CI):
приписка `expected == actual` и уникальность (инструмент ровно в одном манифесте).
Инструмент без дома-приёмника — видимый `Allowlist` с `reason` и `expiresWhen`, без
тихого pending.

## Чеклист соответствия

Реализация ссылается на этот паттерн и копирует чеклист в свой носитель, заменяя каждый
`[ ]` на ✅ либо ⚠ с причиной:

1. [ ] `workshop.manifest.json` с полями `pattern`/`name`/`worksOn`(1)/`verbs`/`kit`.
2. [ ] `worksOn` — ровно один дом; ссылка на паттерн резолвится; правила не скопированы.
3. [ ] Инвентарное покрытие дома: `audit`+`decompose` ✅ в verbs этой мастерской **или**
   ⚠ = `null` с указателем на соседний контур (производная / CI); `inspectElement` —
   ✅ либо ⚠ с причиной. Молчаливый пропуск ключей — запрещён.
4. [ ] Доменная декомпозиция не переписана через стек-примитивы (`domainPreserved`).
5. [ ] `kit` объявлен явно (`null` допустим для интерактивной мастерской).
6. [ ] Каждый доменный инструмент несёт `worksOn`; приписка совпадает (зуб `checkOwnership`).
7. [ ] Отказы видимы: недоступный кит/потерянный якорь — ошибка, не тихо пустой результат.
8. [ ] Провода: мастерская видна из паттерна (таблица реализаций) и дома контейнера.

## Границы применимости

- Дом маленький/короткоживущий → мастерская избыточна, хватает пары скриптов.
- Мастерская — форма *операций* над домом, не *пространства* (это
  `GROUP_CONTAINERIZATION`) и не *идентичности во времени* (это
  `PINNED_SUBGRAPH_VERSIONING`, кит — поставляемый мастерской набор).
- Стек-подобный словарь — ориентир по сходству (SHOULD), не обязательный интерфейс:
  не переименовывать доменные инструменты в `map/filter/reduce` насильно.
