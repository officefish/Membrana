# docs/patterns — архитектурные паттерны Membrana

Каталог именованных **рекомендаций** (уровень SHOULD по RFC 2119): мягче интерфейса,
твёрже устного обычая. Паттерн не принуждается гейтом — он назван, записан и проверяем
по чеклисту; на него можно ссылаться в ревью, консилиуме и ADR.

Отличие от соседних жанров:

- **ADR** (`docs/adr/`) фиксирует *одно принятое решение* в конкретном месте;
  паттерн описывает *повторяемое решение* и контекст его применимости.
- **Регламент** (`docs/prompts/*_REGULATION.md`) обязателен и имеет зубы;
  паттерн — рекомендация: отступление допустимо, но должно быть осознанным
  и объяснимым (SHOULD, не MUST).
- **Скилл** (`.cursor/skills/`) — плейбук «как делать»; паттерн — «какую форму
  принять». Скилл может реализовывать паттерн.

| Паттерн | Суть |
|---------|------|
| [`GROUP_CONTAINERIZATION.md`](./GROUP_CONTAINERIZATION.md) | Накопившейся обобщённой группой артефактов управлять через контейнер (контракт · реестр · кеш · инструменты · агент). Ось **пространства** |
| [`PINNED_SUBGRAPH_VERSIONING.md`](./PINNED_SUBGRAPH_VERSIONING.md) | Единица версии — связный подграф (корень + импорты + конфиги), пины через git SHA, полнота проверяется аудитом; для китов (#761), контейнеры — вне области. Ось **времени/идентичности** |
| [`HOME_WORKSHOP.md`](./HOME_WORKSHOP.md) | Мастерская — именованный набор инструментов, поставляемый для работы с ОДНИМ домом (осмотр · декомпозиция · рассмотреть-элемент; кит опц.). Ось **операций** над домом (#915) |

## Как добавить паттерн

1. Файл `UPPER_SNAKE.md` по структуре: Контекст → Проблема → Решение → Чеклист
   соответствия → Известные реализации → Границы применимости.
2. Строка в таблицу выше.
3. Каждая реализация паттерна ссылается на него и показывает, по какому чеклисту
   соответствует (и где осознанно отступает).
