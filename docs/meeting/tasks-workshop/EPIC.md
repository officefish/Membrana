# EPIC — мастерская контейнера задач (вердикты заседания `tasks-workshop`)

> Сборка активных вердиктов девяти фаз. Задание — [`MEETING_BRIEF.md`](./MEETING_BRIEF.md),
> ход заседания и долги процедуры — [`MEETING_ACTIVE.md`](./MEETING_ACTIVE.md).
> Заседание вердикты **выносит**, исполнение **не запускает**: карточки заводит владелец.

## Картина целиком

Мастерская дома `docs/tasks` получает адрес и иерархию (M1), состав (M2), оси раскладки (M3),
три контракта — паспорт элемента (M4A), валидность (M4B), инварианты сцепки (M4C) — плюс
гигиену производного файла (M4D) и два поведенческих контракта под формат one shot (M5A, M5B).

## Вердикты

### V1 · Адрес мастерской и режим манифеста (M1)

Оба манифеста **остаются**, но получают явную иерархию: `role: primary | derivative`.
Первичный — `docs/tasks` (реестр, истина), производный — `docs/audit/tasks` (разборы, снимки).
Производный несёт `dependentOn: ["docs/tasks"]` и `mirrorsFrom`.

Нынешняя приписка признана **не ошибкой**, а недостроенной иерархией: мастерская разборов
существует законно, но без явных ролей возникали две точки истины.

Зуб: новый `check:workshop-dependencies` — ровно одна первичная мастерская на дом, у любой
производной есть `dependentOn`, `mirrorsFrom` актуально. Правила валидации манифестов — в
одном носителе (`docs/audit/workshop-semantics.json`), манифесты ссылаются на `rulesVersion`.

### V2 · Состав базового набора (M2)

В мастерскую входят пять команд: `inspectElement`, `list`, `board`, `bookkeeping`, `reviewing`.
Писатели (`register`, `archive`, `close-github`) и синхронизаторы (`sync-readme`, `sync-issues`),
а также исполнитель `start` — **вне**: они исполняют решения, а не принимают их.

**Признак границы (правило для будущих команд):** команда входит в мастерскую тогда и только
тогда, когда решает о мета-структуре реестра (целостность графа, категоризация, пересмотр)
**или** порождает отчёты, используемые в таких решениях.

> ⚠ **Расхождение с паттерном, не заглаженное при сборке.** Вердикт относит `audit` и
> `decompose` к «вне мастерской» (как CI-гейты), тогда как `HOME_WORKSHOP` называет их
> **обязательными глаголами самой мастерской** (MUST). Либо комната ошиблась, либо паттерн
> требует уточнения формулировки. Решение — за владельцем; в реализацию это расхождение
> тащить нельзя.

### V3 · Оси декомпозиции (M3, канал `hand`)

Оси: `category` (по умолчанию), `size`, `age`, `lead`, `kind`, `links`. Задаются флагом
`--by`; параметры — в конфиге, не в коде (продолжение нормы Р5). **Одна ось за прогон**,
комбинации запрещены. Правило «ВНЕ КАТЕГОРИЙ» распространяется на любую ось.

Ось `health` **отложена** до вердикта валидности. Проведена граница: `decompose` смотрит
**наличие** полей (факт реестра), **истинность** связей — работа валидности.

### V4 · Контракт `inspectElement` (M4A)

`inspectElement(registry, cardId, { depth })` — **в сеть не ходит**, читает только реестр;
живые данные — отдельной командой с `--refresh`. Паспорт: идентичность, статус, владелец,
родитель, ссылки, маркеры `[ORPHANED]` / `[INCONSISTENT]`. База рекурсии — живые карточки,
в архив и отклонённые спуска нет. Чистая функция внутри, CLI-оболочка снаружи.

### V5 · Контракт валидности (M4B, канал `hand`)

Две чистых функции: `validateTask(card, links)` и `validateRegistry(cards, links)`. Сеть — в
отдельном грязном слое, который отдаёт слепок `links`; предикат тестируется готовым слепком.

Шкала из трёх уровней, каждый означает действие: **blocker** (работа не начинается),
**warning** (работать можно, след неполон), **note** (заметка). **Неизвестность ≠ нарушение**:
недоступность внешней системы даёт `unknown` и не поднимает уровень выше `warning` — упавший
прокси не должен красить весь реестр.

Граница с `tasks:audit`: `audit` отвечает «что из `active` закрыто на самом деле» (соответствие
внешнему миру), валидность — «цела ли карточка как запись». Поглощения нет; `audit` может
использовать предикат, обратное — нет.

Валидность — **зрение, а не забор**: даёт вердикт, но сама ритуал не останавливает.

### V6 · Инварианты сцепки (M4C)

Три обязательных инварианта с уровнями:

| Инвариант | Уровень |
|---|---|
| `active` карточка с `linearId` → тикет существует и открыт | **HARD_BLOCK** |
| Карточка ссылается на иссью → иссью существует и доступна | **WARNING** (мёртвая ссылка) |
| Иссью закрыта → `githubIssueClosedAt` заполнена | **DATALOSS** (не блокирует, флагируется при экспорте провенанса) |

Проверка в двух режимах: быстрый (только реестр, для утра) и полный (с API, для вечера), с
кэшем и TTL. Восстановление мёртвых ссылок — интерактивно (обнулить / создать новый / указать
вручную), автовосстановление без явного одобрения запрещено.

### V7 · README против реестра (M4D, канал `hand`)

Проверка **уходит в групповую валидность** — по природе она групповая. `task:sync-readme --check`
допустим только как **тонкая обёртка** над предикатом (проверяемо: удаление предиката обязано
ломать `--check`).

Зуб — в **pre-commit**, срабатывает только при затронутых `registry.json` / `README.md`; падает,
а **не чинит**: автофикс уничтожил бы сигнал. Иссью **#1014 не закрывается ссылкой**, а
переформулируется в «поставить зуб» — логика поглощена, место осталось.

### V8 · Контракт подбора для one shot (M5A)

Функция возвращает **ранжированный список с обоснованием, не вердикт**. Допуск остаётся у
тимлида — самоаттестация исключена (тезис T4 формата). Каждый источник суждения несёт
`dataReadiness`, чтобы тимлид видел дыры в данных.

Суждение о серверности до появления диффа берётся по слоям приоритета: дифф (если готов) →
ключевые слова описания → историческая репутация → роль; при отсутствии всех — `unknown`.
Жёсткая отсечка по критическому серверному риску, мягкие признаки поднимают ранг.

### V9 · Защита от дробления (M5B)

Историю шотов подбор смотрит **обязательно**. Окно — **7 дней**. Смежность путей — общий
префикс папок длиной ≥ 2. При обнаружении цепочки ранг снижается на уровень; владелец может
перебить флагом, и тогда в приказ добавляется видимый маркер `[risk-override]`.

Источник истории — **отдельный журнал** `docs/audit/one-shot-trail.jsonl`, не реестр: формат
one shot регистрацию карточки отменяет, поэтому след обязан жить независимо. Запись
идемпотентна.
