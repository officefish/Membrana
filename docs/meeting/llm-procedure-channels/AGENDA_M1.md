## **P1 — Каков контракт «процедуры» как единицы маршрутизации и учёта?**

Комната фиксирует: схему стабильного id, что входит в запись реестра процедур,
кто регистрирует новых жильцов, версионируется ли привязка к скрипту/релизу,
и что явно **не** является процедурой (чтобы не раздуть поверхность панели).

### Кандидаты существа (просто текст, без жирных ID)

1. Id = kebab имя скрипта/yarn-команды (`code-review`, `consilium`) без отдельного реестра.
2. Id = запись в git-реестре процедур (`docs/…` или `scripts/lib/…-registry`) с полями
   `{id, yarnScript?, entryMjs, owners[], meters: bool}`; новые id — PR + LGTM.
3. Id выдаёт office/panel API; git только клиент; регистрация через админ UI.
4. Версия: id неизменен, а `bindingVersion` (hash entry / semver пакета скриптов)
   уходит в телеметрию и в снимок маршрута — или версия не нужна в v1.
5. Не процедура: одноразовый `yarn ask`, experimental `llm-proxy:ask`, ручной чат IDE,
   внутренние хелперы без пользовательского ритуала — или наоборот, всё с LLM = процедура.

### Форма вердикта

- Одна схема id + место реестра (или явный отказ от реестра).
- Минимальные поля записи v1.
- Кто добавляет процедуру (PR / админ / автоматически из package.json).
- Версионирование: да/нет/отложить.
- Явный exclude-list «не процедура».
- **Список посылок обязателен** в том же протоколе (правило 6).
