## Вход (не переоткрывать)

- P1: git-реестр id процедур; регистрация id = PR; админ UI не порождает id в v1.
- Требование владельца: на mmbrn.tech админ переключает канал процедуры.
- Агент работает в worktree/Windows; office = `office.mmbrn.tech` / panel уже жив.

## **C1 — Где живёт источник истины «процедура → канал» и кто его меняет?**

Комната фиксирует: git-конфиг vs office API vs UI-only; что побеждает при конфликте
локальный `.env` vs серверная политика; минимальный контракт чтения агентом и записи админом.

### Кандидаты (просто текст)

1. SoT = git-файл маршрутов (рядом с реестром процедур); панель только читает/показывает;
   смена канала = PR (админ = merge).
2. SoT = office API (persist); панель пишет; агент pull при старте процедуры; git держит
   только default/fallback snapshot для cold start без сети.
3. SoT = только `.env` / `.env.llm-proxy` на машине агента; панель — дашборд usage без write.
4. Гибрид: defaults в git; overlay (per-procedure channel) в office; merge-правило
   «overlay > default»; `.env` только секреты ключей, не выбор канала.

### Форма вердикта

- Выбранный SoT и роль git / office / `.env`.
- Правило конфликта (что побеждает).
- Минимальные операции API или файлового контракта v1 (read path агента, write path админа).
- Что отложено.
- **Список посылок обязателен.**

### Запреты

- Не решать схему полей телеметрии (комната 2).
- Не решать fallback-цепочку детально (комната 5) — только где хранится выбор канала.
- Не решать шов llm-proxy ключей (комната 6) сверх «секреты ≠ маршрут».
- Не проектировать экраны панели (комната 3) — только write-path админа в принципе.
