## **O1 — Какой порядок зависимостей у вопросов заседания?**

Комната держит все кандидаты разом и выдаёт **DAG** (слои, корень, что параллельно,
что запрещено решать раньше фундамента). Порядок утверждает **владелец**, не председатель.

### Кандидаты (без ID — просто текст)

1. Где живёт control plane маршрутизации процедур → канал (git-конфиг · office API ·
   только UI панели · иное), и что является источником истины при конфликте
   локальный `.env` vs серверная политика.
2. Модель телеметрии вызова: событие (процедура, provider, model, tokens in/out,
   latency, outcome), где хранится, срок хранения, что нельзя писать (секреты, полный
   промпт?).
3. Поверхность панели на mmbrn.tech: host (`panel.mmbrn.tech` vs иной), роль/grant
   админа, минимальный UI v1 (сводка дня + переключатель канал×процедура).
4. Контракт «процедура» как единица маршрутизации: стабильный id (`code-review`,
   `consilium`), версия скрипта, кто регистрирует новые процедуры.
5. Поведение при отказе канала: fallback-цепочка vs жёсткий STOP; кто меняет цепочку
   (админ панели vs только деплой).
6. Шов с существующим `.env.llm-proxy` / experimental OpenRouter: поглотить, оставить
   рядом, или объявить deprecated для ритуалов.
7. Граница v1: только code-review + consilium — или сразу каркас «все ритуальные
   LLM-скрипты», а провода CR/consilium как первые жильцы.
8. Как локальный агент (Windows, worktree) докладывает usage на панель (push в office
   vs batch-файл vs только серверные вызовы через office).
9. Критерий готовности к спринту исполнения: какие вердикты обязательны до первой
   строки кода транспорта в `code-review.mjs`.

### Форма вердикта M0

- Корень DAG и слои.
- Явный список рёбер «A до B».
- Что сознательно **вне** заседания (follow-up / другой эпик).
- **Список посылок обязателен** в том же протоколе (правило 6): полный список
  входов вердикта, без добора задним числом.
