[Структурщик]: Снова две леммы id. **Имя вызова** — что печатает человек (`yarn code-review`). **Единица учёта** — ключ в маршруте и в событии телеметрии. Они могут совпадать строкой, но не обязаны совпадать механизмом появления: yarn-скрипт можно переименовать, ключ учёта ломать нельзя. Кандидат 1 (id = имя yarn без реестра) смешивает леммы: rename скрипта = разрыв истории токенов.

[Teamlead]: Функция контракта — дать стабильный ключ для панели и для агента. Прецедент в репо: `docs/tasks/registry.json`, `docs/procedures/*/MANIFEST.json` — git-реестры с явным добавлением через PR. Кандидат 3 (id выдаёт office) создаёт зависимость панели от сервера до первого провода code-review — тяжёлый фундамент. Склоняюсь к **кандидату 2**: git-реестр процедур, id стабилен, yarnScript — поле, не ключ.

[Математик]: Формализую. Нужна инъекция `id: ProcedureId → Record`. Стабильность: ∀ события e с procedure=id история суммируется без rename-миграций. Если id:=yarnName, то rename — это смена ключа агрегации → разрыв ряда. Реестр с неизменяемым id и опциональным `yarnScript` сохраняет ряд. Минимальные поля v1: `{id, entryMjs, yarnScript?, title?, meters: boolean}`. `owners[]` можно отложить — не блокер маршрута.

[Музыкант]: Флаг мёртвой двери: автоматическая регистрация «всего с LLM из package.json» (край кандидата 5 наоборот) засорит пульт процедурами вроде одноразовых smoke. На слух нужны только ритуальные/агентские вызовы с политикой канала. Exclude: `llm-proxy:ask`, `anthropic:smoke/task`, сырой чат IDE, `yarn ask` без отдельного решения. code-review и consilium — первые две записи вручную в том же PR, что заведёт реестр.

[Верстальщик]: Для панели id должен быть **человекочитаемым kebab** (`code-review`, `consilium`), не uuid. Uuid хорош для офиса, плох на переключателе. Git-реестр рядом с procedures — привычная витрина в PR: diff виден, LGTM виден. Office как единственный регистратор (кандидат 3) прячет появление процедуры от code review репо.

[Структурщик]: Версия. Два слоя: **id** (логический жилец) и **binding** (какой файл/ревизия реально позвала модель). В телеметрию v1 достаточно класть `entryMjs` + опционально git sha запуска — это не semver реестра. Отдельный `bindingVersion` в записи реестра в v1 **не обязателен**: усложняет без выигрыша, пока SoT маршрутов ещё не выбран (комната 1). Вердикт: версию записи реестра отложить; в событии — пути/sha как факты рантайма, не как контракт P1.

[Teamlead]: Кто добавляет. Только PR в реестр + LGTM — согласован с «админ на панели переключает канал», но **не** плодит id из UI в v1. Иначе панель становится источником истины имён раньше control plane (кандидат 1). Админ UI позже может предложить «добавить», но запись всё равно приезжает git-ом — или это уже существо комнаты 1/3. Для P1: **регистрация = PR в git-реестр**.

[Математик]: Опровергну кандидата 1 численно. Пусть за день N вызовов; доля rename скриптов за квартал p>0 на живом монорепо. Матожидание разрыва ряда при id=yarn растёт с p·N. При стабильном id стоимость — один PR на нового жильца, O(1) на квартал. Реестр дешевле разрывов.

[Музыкант]: Кандидат 5 (что не процедура). ## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Схема id | Стабильный **kebab-id** (`^[a-z][a-z0-9-]*$`), immutable; не uuid; не обязан равняться имени yarn навсегда. |
| Место реестра | Git-канон: JSON под `scripts/lib/` (рабочее имя `llm-procedures.json`); не office как единственный регистратор в v1. |
| Поля записи v1 | `{id, entryMjs, yarnScript?, title?, meters}` — `meters: true` для учёта в дневной сводке. |
| Регистрация | Новый id только **PR + LGTM** в реестр; админ UI не порождает id в v1. |
| Первые записи | `code-review`, `consilium` (meters true) — в том же контуре, что заводит реестр; состав «только они» = комната scope (7), не P1. |
| Версионирование | Отдельный bindingVersion в реестре **отложен**; рантайм может слать path/sha в телеметрии (комната 2) без поля реестра. |
| Не процедура (exclude) | experimental `llm-proxy:*` ask/smoke; `anthropic:smoke`/`anthropic:task`; сырой IDE-чат; одноразовые хелперы без ритуала. |
| Тот же id | `consilium` secretary и LLM — один id; `local-code-review` **не** отдельный id (канал/fallback того же code-review — комнаты 1/5). |
| Отвергнуто | Кандидат 1 (id≡yarn без реестра); кандидат 3 (office выдаёт id) для v1. |

**Definition of Done (только P1):**
- Вердикт называет схему id, путь/принцип реестра, поля v1, правило регистрации, exclude, судьбу local-code-review.
- Не утверждает SoT маршрутов, схему событий, scope «навсегда только двое», UI панели.
- Список посылок в этом протоколе.
