## Вопрос заседания

**E3 — конституция границы слоёв: чем обеспечивается направленность
«процедуры → киты → скрипты», чем даётся зуб на нарушение, и где живёт парсер
ответственности как жилец (адрес, не механизм)?**

Ответ обязан назвать: правило направленности и его машинную проверку; статус
кит-манифеста в ревью; адрес парсера ответственности с маршрутом расследования
прецедента (вердикт → SHA правил → история правок). Механизм парсера — под
запретом T9.

## Решено владельцем — посылки, не предмет спора

- Скрипты — плоский контейнер; процедуры управляют скриптами через ссылку;
  группировка — виртуальные наборы, киты (T11).
- Контейнер процедуры без собственного кода и тестов (T12).
- Киты версионируются с момента появления (`PINNED_SUBGRAPH_VERSIONING`:
  единица версии — связный подграф, пины git SHA, полнота через аудит);
  контейнеры — нет.
- По сложным коммитам процедурных веток ответственность распределяет парсер
  на code-review (T6); сейчас проектируется его место жительства, не механизм (T9).

## Вердикты M1–M2 — применить, не переспорить

- Дом: `docs/procedures/<id>/` — `README.md` + `MANIFEST.json`
  (`id`, `leadPersona`, `kitVersion`, `engines[]`, `precedents[]`);
  зуб заселённости `validateProcedure(dir)` в CI.
- Словарь: полуоткрытый, ядро 4 категории, машиночитаемый источник в доме слоя,
  маркеры `@cat:`, чистый `check()` со списком нарушений.
- Семантика обращения процедур с категориями (род: объект/отношение/ориентир) —
  предмет ЭТОЙ комнаты (передано из M2).

## Фактура (проверяемо)

- `engines[]` манифеста M1 — уже ссылки «процедура → скрипты»; обратного механизма
  («скрипт знает процедуру») в контракте нет — направленность заложена структурой.
- Прецеденты зубов на границу: `check-package-boundaries` (границы пакетов),
  eslint-гейт singleton §3.5 (TD3), pre-push хуки — образцы механизма.
- issue #761 — кит: виртуальный неймспейс, SHA-пины подграфа; статус: идея
  зарегистрирована, реализации нет (факт сессии 21.07).
- Паттерн `PINNED_SUBGRAPH_VERSIONING` в `docs/patterns/` — чеклист из 7 пунктов,
  «Известные реализации: пока нет — ждёт первого кита».
- Расследования прецедентов ответственности вести некуда: контейнера домена
  «ответственность» не существует (факт).

## Что комната должна выдать — РОВНО ТРИ пункта

1. Правило направленности границы и его зуб (инструмент, где вызывается).
2. Статус кит-манифеста как объекта ревью.
3. Адрес парсера ответственности (определение / правила / прецеденты) и маршрут
   расследования прецедента. Механизм — не проектировать.

Долг «род категорий» из M2 в этой комнате НЕ разбирается — возвращён в хвост
заседания (см. MEETING_ACTIVE). Судьба K9 считается решённой выбором адреса.

## Структура финала — жёсткая, ровно два блока

**Definition of Done НЕ писать** — вместо него финал состоит ровно из двух блоков:

1. Секция с заголовком ровно **«Список посылок»** — до 10 позиций, каждая одной
   строкой, пометка **факт** или **норма**. Только входы, не выводы комнаты.
2. Таблица **«Итоговое решение»** — не более 7 строк.

Четыре прошлые комнаты потеряли посылки, растратив финал на DoD, — их вердикты
отвергнуты гейтом целиком.

## Запреты этой комнаты

- Не проектировать механизм парсера (T9) — только адреса и маршрут.
- Не трогать грамматику веток и каскад вывода ответственного (M4), не планировать
  миграцию (M5).
- Не пересматривать вердикты M1–M2 и тезисы владельца.
