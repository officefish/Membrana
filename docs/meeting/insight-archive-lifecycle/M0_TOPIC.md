# M0 — порядок вопросов жизненного цикла инсайтов

> Общее задание: `docs/meeting/insight-archive-lifecycle/MEETING_BRIEF.md`  
> Фактура: внутреннее расследование + external deep research из ссылок в brief.  
> На этом прогоне запрещены содержательные решения по кандидатам: нужен только DAG.

## Вопрос

**O1 — какой порядок зависимостей должен иметь разбор жизненного цикла инсайтов, чтобы каждый последующий вердикт опирался только на уже принятые основания?**

## Кандидаты (материал, не отдельные вопросы повестки)

- Определить объект закрытия: INSIGHT, REVIEW-мандат или outcome.
- Развести состояния решения, delivery/outcome и archive/visibility.
- Определить типизированный evidence contract и полноту task graph.
- Определить модель истории, производных views и обратимых событий.
- Определить forensic migration legacy-связей и классификацию четырёх кандидатов.
- Замкнуть workflow и enforcement для Cursor, Claude и Codex.
- Определить operational safety: concurrency, live work, idempotency, rollback/reopen.

## Требуемая форма единственного вердикта

1. Эхо `O1` и явная строка `ВЕРДИКТ O1`.
2. DAG: корень, слои, зависимости каждого кандидата.
3. Обоснование, почему ни один вопрос не поставлен раньше собственной посылки.
4. Никаких решений по существу кандидатов.
5. Список посылок, реально использованных для DAG.
