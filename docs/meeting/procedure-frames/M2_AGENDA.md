# Повестка M2 — механика пина отрезков

> Заседание `procedure-frames`, фаза M2 (вторая по порядку M0, ратифицирован 22.07).
> Задание: [`MEETING_BRIEF.md`](./MEETING_BRIEF.md) — прочесть целиком до реплик.
> Предшественники: вердикт M0 (порядок K1→K4), вердикт M1 (`m1-frames-contract`:
> элемент фрейма `{id, holder, pin?}`, `pin` — непрозрачная ссылка, битый пин = block,
> резолв пина — предикат P3) — посылки этой комнаты.

---

## Вопрос заседания

**F2 — какова механика пина отрезка: форма якоря (файл, якорь, хэш отрезка),
правило «дрейф отрезка ≠ дрейф файла» и инструмент аудита дрейфа?**

---

## Фактура (замер 22.07)

- **Киты пинят целые файлы**: `kits/<id>/MANIFEST.json` несёт `pins: {путь → git
  blob SHA-1}`; `kit-subgraph-audit.mjs` считает blob-хэш рабочего дерева
  (`git hash-object`-эквивалент) и сверяет с манифестом; расхождение — находка
  `sha_drift` (в latest — предупреждение, недостающий узел — блок).
- **Тезис T8**: целые документы в подграф фрейма не входят — документ шире
  процедуры; часть фрейма — конкретная строчка/абзац под версией и лицом.
  Фрейм — набор **разрозненных отрезков** (команды + отрезки документов).
- **Кандидат K1-шторма** (Математик, не подтверждён — проверить здесь): SHA файла
  меняется от правки соседних строк → пин отрезка = (файл, якорь/диапазон, хэш
  текста отрезка); дрейф отрезка ≠ дрейф файла.
- **Вердикт M1**: `pin` в элементе фрейма опционален; не резолвится → block
  (громкий стоп); сетевой/внешний слой недоступен → «не проверен», не ложно-зелёный.
- **Родовой узор T7**: кит и фрейм — доменные имена одного паттерна; вопрос
  переиспользования — общий аудит или доменный близнец `kits:audit`.

## Что комната должна выдать

- Форму якоря отрезка: диапазон строк, заголовок секции, текстовый маркер или
  комбинация — и почему выбранная форма переживает правки соседних строк.
- Что именно хэшируется (сырой текст отрезка, нормализация переводов
  строк/пробелов) и каким хэшем — совместимость с git blob SHA-1 китов или
  осознанное отличие.
- Правило дрейфа: отрезок уехал / файл правлен рядом / файл переименован /
  якорь не найден — какой из случаев block, какой предупреждение, какой тишина.
- Инструмент аудита: имя в audit-семье, табличный вывод, кто его гоняет
  (CI / `validateProcedure` P3 / отдельная команда), отношение к `kits:audit`
  (общая библиотека, форк или расширение).
- Как обновляется пин при законной правке отрезка: осознанный акт (отдельный
  диффабельный коммит — принцип паттерна), не побочный эффект.

## Список посылок — обязателен

Вердикт обязан нести секцию с заголовком ровно **«Список посылок»**: полный список
входов, у каждого пометка **факт** (проверяемо по коду или замеру) или **норма**
(наше решение). Только входы — выводы этой комнаты в посылки не вносить.
Вердикты M0 и M1 — легальные посылки-предшественники.

## Запреты этой комнаты

- Не собирать состав первого фрейма «подводка утра» (какие именно отрезки) —
  комната K3; здесь механика, не наполнение.
- Не переписывать формулировку паттерна PINNED_SUBGRAPH_VERSIONING — комната K4.
- Не пересматривать контракт M1 (`{id, holder, pin?}`, block на битом пине) —
  это посылка.
- Не пересматривать тезисы владельца (MEETING_BRIEF, «переспорить нельзя»).
