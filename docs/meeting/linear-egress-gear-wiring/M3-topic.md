Место в DAG: после формы egress и trust boundary. Предшественники: M1 (producer pull =

## Единственный вопрос заседания M3

**E5 — какой контракт `linear-snapshot@1` и какой DoD первого боевого pull через
media-NL: какие поля обязательны в honest-шапке (включая producedBy и регион egress),
какой проверяемый критерий «боевой pull успешен», и что именно гейты читают офлайн
без обращения к api.linear.app?**

Ответ — одна схема контракта + один предикат успеха + явное офлайн-правило гейта,
не меню без выбора. Живой ответ Linear с NL комнате не добыт — DoD задаёт
**проверяемые признаки артефакта**, не выдумывает замер сети.

## Кандидаты (простым текстом — не вопросы этой комнаты)

1) Минимальная honest-шапка: `format`, `capturedAt`, `sourceRevision`, `producedBy`
   (= media-NL), `egressRegion` (= NL), `mode=batch-full-pull`, `trigger`, `recordCount`.
   Поле `source: office-batch` из текущего типа — смысловая замена / расширение под
   producer media (аксиома «гейты читают снимок» не отменяется). Боевой pull успешен
   ⇔ артефакт с полной шапкой ∧ producedBy=media-NL ∧ egressRegion=NL ∧
   sourceRevision непуст ∧ recordCount согласован с телом. Гейт в момент вердикта
   читает только файл снимка (+ опционально отдельный freshness-запрос вне тела гейта
   по паспорту) — не GraphQL Linear.

2) Расширенная шапка: всё из пункта выше плюс хэш тела (`contentDigest`) и
   `keyFingerprint` (не сам ключ) как доказательство легального узла. DoD первого
   pull жёстче: без digest — SNAPSHOT_NO_INPUT.

3) Шапка как в коде сегодня (`source: office-batch` без producedBy/region) + только
   смена docs; DoD = «HTTP 200 от Linear». Отклоняется, если комната сочтёт, что
   это ломает видимость границы M1/M2.

## Вход (факты / нормы — не вердикты)

- Норма владельца: гейты читают снимок, не live fetch; egress = media-NL.
- Вердикт M1: producer pull = media-NL; office видит только артефакт; «office-батч»
  → `mode=batch-full-pull`, producer на media.
- Вердикт M2: ключ только media; в К3 ушли `producedBy: media-NL` + регион egress.
- Факт кода: `LinearSnapshotHeader` в office уже несёт `format`, `capturedAt`,
  `sourceRevision`, `source: 'office-batch'`, `trigger`, `recordCount` — **без**
  `producedBy` / `egressRegion` (пакет `linear-snapshot`).
- Паспорт §6 / INTERFACE: провенанс `(capturedAt, sourceRevision)`; коды 10/20;
  честная шапка обязательна; слово «кэш» запрещено.
- Факт: живой боевой pull с media-NL ещё не проведён (пилот — stub).

## Что комната должна выдать

- Обязательные поля honest-шапки `linear-snapshot@1` после проводки media-NL.
- Судьбу литерала `source: office-batch` (сохранить / заменить / дополнить).
- Предикат «первый боевой pull успешен» (булевы условия на артефакте, не на настроении).
- Правило: что гейт читает офлайн; что (если что) допустимо вне тела гейта для свежести.
- Что уходит в К5 / К4b одной строкой каждый (критерий снятия stub; гейт closure) —
  без вердикта по ним.
- DoD контракта без деплоя и без выдуманного live-ответа Linear.
