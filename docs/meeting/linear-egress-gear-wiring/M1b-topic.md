## Кандидаты (простым текстом — не вопросы этой комнаты)

1) Ввести `yarn task:start`: до кода создаёт/требует GitHub Issue, пишет биекцию в
   registry, печатает шаблон acceptance `{acceptedBy,headRev}`, движение Linear —
   честный stub (`movement: deferred-egress`) пока нет боевого pull. Отказ active без
   Issue — зуб в register/pre-push. AGENTS.md и lifecycle skill ссылаются на паспорт
   и словарь, не на «register in registry.json» как единственный старт.

2) Оставить `task:register` + ручной Issue, но запретить `githubIssue: null` на active
   жёстким гейтом; посев в AGENTS — только ссылка на паспорт без новой команды старта.
   Контейнер спринта = опциональный Linear parent, не обязателен до egress.

3) Развести «регистрацию содержания» и «старт движения»: registry + prompt остаются
   содержанием; `task:start` — только событие старта (Issue обязателен, stub движения,
   leadPersona). MAIN_DAY остаётся фокусом дня; единица работы вне магистрали —
   явное поле override со ссылкой на Issue владельца (урок пилота O8), иначе конфликт норм.
