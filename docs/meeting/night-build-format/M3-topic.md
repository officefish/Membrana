# Заседание `night-build-format` — фаза M3 (edit-class taxonomy)

Задание: `MEETING_BRIEF.md`. Порядок из M0 (ратифицирован): guardrails закрыт,
каденс закрыт, тулинг 4+5 смёржен → последний узел DAG, кандидат 1.
Вход: `docs/tasks/research/night-build-format-v2.md` (Q1 §4, Q2 §2).

Разбирается РОВНО один вопрос — тест-гейт по классу правки. Guardrails, каденс,
тулинг — закрыты, сюда не тянуть.

## Единственный вопрос

**T1 — Как регламент должен различать классы правки при тест-гейте (код / конфиг /
инфраструктура / интеграция), что считать достаточной проверкой для каждого класса,
и как честно оформлять квант, который юнит-тестом не проверяется в принципе —
чтобы «каждая правка с тестом» не порождало ни фиктивных тестов ради галочки, ни
молчаливого нарушения?**

## Обоснование (ночь 17.07 + research)

Ночь дала три кванта, где безусловный тест-гейт не работал:
- `NB1` `.markdownlint.json` — конфиг из 12 строк; юнит-тест на него искусственен
  (проверка = валидность JSON + docs:lint зелёный).
- `NB6` клиентский вывод ласточки — печать строки; проверялась глазами и
  синтаксисом, юнит фиктивен.
- `NB5` libuv-гонка `process.exit` после fetch — **в принципе** не проверяется без
  живого LLM-вызова; я закоммитил с честной пометкой «верифицируется живым
  прогоном», то есть фактически обошёл гейт.

Research: код → юнит обязателен; конфиг → линтер/схема; инфра → plan/симуляция;
интеграции → sandbox/интеграционный, юнит не проверяет реальное взаимодействие.
Q1 §4 добавляет: «для всех классов тесты обязательный гейт» — но гейт РАЗНЫЙ.

## Что должен вернуть вердикт

- Таблица `класс правки → достаточная проверка` (операционально, без синонимов).
- Как квант объявляет свой класс (кто и где помечает; связь с весом W из каденса —
  но НЕ пересматривать сам вес).
- Честная форма для «юнитом не проверяется»: что писать вместо фиктивного теста,
  чтобы обход был видимым, а не молчаливым.
- Какие разделы `NIGHT_SPRINT_REGULATION.md` меняются.
- Полный список посылок.
