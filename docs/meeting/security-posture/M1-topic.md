# Заседание `security-posture` — M1 (дозор за зависимостями)

Задание: `docs/meeting/security-posture/MEETING_BRIEF.md`.
Место в DAG (вердикт M0, ратифицирован 19.07): №2 «дозор» — КОРЕНЬ; его порог
severity и каденс — вход для нарезки №1b (апгрейд-спринт, M2). Протокол M0:
`docs/seanses/security-posture-m0-order-run2-2026-07-19.md`.

## Единственный вопрос заседания M1

**A1 — Каким должен быть постоянный дозор за зависимостями Membrana: где живёт
проверка (Dependabot alerts + место `yarn npm audit` в ритуале — утро, вечер или
гейт), какой порог реагирования по severity превращает advisory в задачу реестра
(candidate M0: critical/high → задача сразу; moderate → планово; low —
накопительно к ближайшему апгрейду), и какой каденс пересмотра порога?**

## Вход (факты)

- Аудит 19.07: 96 advisories — 2 critical, 34 high, 49 moderate, 11 low.
  Разрез: рантайм прода (fastify-стек, electron, tar, multer, ws, undici) vs
  dev-тулинг (vite, vitest).
- Ритуал дня существует: `yarn ritual:day` (утро) и `yarn ritual:evening`
  (вечер) — кандидаты на место audit-шага.
- Dependabot alerts в GitHub-репозитории officefish/Membrana сейчас не
  подтверждены как включённые.
- Гигиена-патчи (№1a) и critical рантайма исполняются листами параллельно
  этому заседанию — дозор их не блокирует.
- Урок инцидента patchhog 19.07: снаружи о дырах узнаём от спамеров — свой
  дозор должен опережать чужие «отчёты».

## ⛔ ОГРАДА

Вопрос РОВНО ОДИН — дозор (№2). НЕ выносить вердиктов по: существу апгрейд-спринта
№1b (версии vite/vitest/fastify — это M2), протоколу входящих №3, политике
third-party доступов №4, секрет-гигиене №5 (это M3), узлам чужих заседаний
(night-build, scripts-boundary, truth-graph, evening-auditor). Упомянуть смежное
можно, вердикт — нельзя.

## Что должен вернуть M1

- Где живёт дозор: Dependabot (вкл/выкл, какие экосистемы) + место
  `yarn npm audit` в ритуале (утро/вечер/оба/гейт) + кто читает вывод.
- Порог: severity → действие (задача сразу / планово / накопительно), с явным
  разрезом рантайм-прод vs dev-тулинг, если комната сочтёт его значимым.
- Каденс пересмотра порога.
- DoD операционных шагов (что именно включить/встроить), без чужих вопросов.

## ⚠️ Обязательная форма вердикта (правило 6, гейт заворачивает без неё)

После итоговой таблицы протокол ОБЯЗАН нести отдельную секцию с заголовком
ровно такого вида (строка-заголовок, не упоминание в прозе):

`## Список посылок`

В теле секции — только ВХОДЫ вердикта (факты аудита 19.07, вердикт M0 как
посылка-предшественник, нормы ритуала, факты инцидента), пронумерованные.
ЗАПРЕЩЕНО включать выводы этой же комнаты («здесь принято», «решено здесь»).
Протокол без этой секции гейт отклоняет, заседание не состоится.
