# Заседание `security-posture` — M0 (установочное)

Задание: `docs/meeting/security-posture/MEETING_BRIEF.md`.
Вход: аудит `yarn npm audit --all --recursive` от 19.07 (96 advisories: 2 critical,
34 high, 49 moderate, 11 low) + инцидент с холодным письмом patchhog (выдуманная
версия vite, токен-трекинг, воронка GitHub OAuth с write-доступом).

M0 держит пять кандидатов в одной комнате и выносит **единственный** вердикт —
порядок зависимостей (DAG): что фундамент, что зависит, что параллельно, какие
кандидаты не требуют консилиума (тривиальный фикс / готовый канон), порядок M1..Mn.

## Единственный вопрос заседания M0

**O1 — Каков порядок зависимостей (DAG) между пятью security-кандидатами: какой
вопрос фундамент, что от чего зависит, что параллельно, какие исполнить без
отдельного заседания, и в каком порядке созывать M1..Mn?**

## Кандидаты (простым текстом, ранг назначает M0)

1. Триаж аудита зависимостей: разрез «рантайм прода vs dev-тулинг». Critical
   рантайма — @fastify/middie auth bypass (+6 high обходов middleware в
   fastify/nestjs-стеке); critical dev — vitest 1.6.1 (UI-сервер исполняет
   произвольные файлы); high dev — vite 5.4.21 fs.deny bypass на Windows (патча
   в 5.x нет, нужен мажор 7+). Что чинится гигиеной (патч/минор: tar, multer,
   ws, undici, form-data, fast-uri), что требует апгрейд-спринта (vite 7 +
   vitest 3, fastify-стек), и как порезать на задачи реестра.
2. Постоянный дозор за зависимостями: включить Dependabot alerts в GitHub;
   определить место `yarn npm audit` в ритуале (утро/вечер/гейт) и порог
   реагирования по severity (critical/high → задача сразу, moderate → планово).
3. Протокол входящих security-сообщений (урок инцидента patchhog 19.07): как
   проверять заявленное (сверка с репо, декодирование ссылок, локальный аудит
   вместо чужого сканера), что запрещено (переход по токен-ссылкам, Sign in with
   GitHub на сторонних «отчётах»), куда фиксировать эпизод.
4. Политика third-party доступов: правило выдачи GitHub Apps / OAuth с доступом
   к репозиториям (гейт: владелец/консилиум), ревизия уже установленных
   приложений и выданных токенов.
5. Секрет-гигиена как машинный гейт: secret-scan (gitleaks — тот же движок, что
   продаёт patchhog, бесплатен) в pre-commit или CI; дисциплина `.env` в
   worktree (память: office-token-in-main-worktree).

## ⛔ ОГРАДА

Кандидатов РОВНО ПЯТЬ, перечислены выше. НЕ импортировать узлы соседних
заседаний и эпиков: night-build (guardrails, каденс, taxonomy), scripts-boundary
(версионирование схем, ретраи, реестр процессов), truth-graph (форма вердиктов),
evening-auditor. Существо версий (какую именно версию ставить) M0 не решает —
только порядок разбора. Если роль называет узел не из пяти выше — это дрейф,
вернуть к списку.

## Что должен вернуть M0

- DAG: рёбра «X — фундамент для Y».
- Корень (фундамент) и что параллельно.
- Кандидаты без консилиума (тривиальный фикс / готовый канон) — исполнять, не
  заседать по ним.
- Порядок созыва M1..Mn.
- Вердикт называет полный список посылок.

## ⚠️ Обязательная форма вердикта (правило 6, гейт заворачивает без неё)

После итоговой таблицы протокол ОБЯЗАН нести отдельную секцию с заголовком
ровно такого вида (строка-заголовок, не упоминание в прозе):

`## Список посылок`

В теле секции — только ВХОДЫ вердикта (факты аудита 19.07, факты инцидента
patchhog, нормы регламентов, память проекта), пронумерованные. ЗАПРЕЩЕНО
включать в посылки выводы этой же комнаты («здесь принято», «решено здесь») —
посылка обязана быть входом, иначе тест на отзыв не запускается. Протокол без
этой секции гейт отклоняет, заседание не состоится.
