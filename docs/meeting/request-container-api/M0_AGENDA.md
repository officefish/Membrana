## Что уже подтверждено штормом (не переоткрывать)

| # | Тезис |
|---|-------|
| Т1 | Пара «модель+провайдер» выясняется **у сервера** через мастерскую контейнера запросов; процедура провайдера не носит. |
| Т2 | **В процедурах агент остаётся без ключей** — отъём секретов, не удобство. |
| Т3 | Вызов внешнего API — **заявка**: агент формулирует, сервер исполняет. |
| Т4 | Ответственность **переезжает на сервер**: `countersign` нужен, билеты раздаются там, у агента — кабинет. |
| Т5 | **Кабинет — зона полномочий** (не место и не личность). |
| Т6 | **Серверную механику держит Фаррелл**. |
| Т7 | **Контейнер запросов регулирует очередь** (порядок, задержка, отказ) **и считает токен-экономику**; Фаррелл работает глаголами его мастерской. |

## Фактура на входе (замерено, не гипотезы)

- Контейнер запросов **существует**: `docs/audit/llm-calls`, семья `audit-family` в атласе контейнеров. Оснащён **на два глагола из трёх** — `inspectElement` отсутствует.
- Слой каналов **частично построен**: `consilium` резолвит провайдера через `llm-procedure-resolve`; в дефолтах у процедур прописана **цепочка** `anthropic → openrouter` (фолбэк при лимите).
- Дыра актора **не закрыта**: полномочия кабинета не дают отличимости субъекта; второй актор на сервере не назван.

## Вопросы-кандидаты в комнаты (порядок — предмет вердикта M0)

1. **Контракт кабинета** как зоны полномочий: что перечисляется, кто выдаёт, чем проверяется граница, что происходит при выходе за неё.
2. **Границы мастерской контейнера запросов**: какие глаголы она несёт, что Фаррелл делает её глаголами, а что — вне; и достройка третьего глагола.
3. **Второй актор на сервере**: где живёт `countersign`, кто подписывает приёмку, раз у клиента ключей нет и полномочия отличимости не дают.
4. **Токен-экономика**: что есть единица счёта, кто её читает, чем она гейтит (отказ по исчерпанию — состояние очереди или отказ процедуры).
5. **Счёт вдохов на сервере** — хвост шторма после потолка: переносится ли механика вдоха в контейнер запросов и что это меняет для форматов.

## Единственный вопрос комнаты M0

**O1 — Каков порядок разбора перечисленных кандидатов: какие из них становятся комнатами, в какой последовательности, какие сливаются между собой и какие отклоняются как не-вопросы?**

Обоснование порядка обязано опираться на зависимости (что чем питается), а не на удобство. Комната не решает сами вопросы — только их очередь.

## Список посылок — обязателен

Вердикт обязан встать на **явно названном фундаменте**. Комната завершает протокол разделом **«Список посылок»**, где каждая посылка помечена родом:

- **факт** — замеренное состояние мира (со ссылкой на вещдок);
- **норма** — действующее правило проекта (со ссылкой на канон или вердикт).

Жёсткое правило: посылки — **только входы**, а не выводы самой комнаты. Вывод, вписанный в посылки, заражает DAG и делает вердикт самообоснованным; такое возвращается на доработку. Если нужного входа нет — посылка объявляется отсутствующей явно, а не додумывается.
