## Вопрос заседания

**F5 — как две живущие мастерские (ветки, задачи) становятся ссылающимися реализациями
паттерна: чеклист соответствия, форма потребления кита (замер: не потребляют) и что
дописать каждой, чтобы обобщение состоялось?**

---

## Фактура (замер 22.07 — блокер M0 закрыт)

- **Мастерская веток**: `repo:branches` (осмотр) + `repo:branches:decompose` (раскладка
  по 7 категориям гигиены) + скиллы branch-audit/decompose (PR #755, #758). `worksOn` =
  git-контейнер веток.
- **Мастерская задач**: `tasks:audit` (осмотр) + `tasks:decompose` (раскладка по префиксам
  конфига) + скиллы tasks-audit/decompose (PR #752, #757). `worksOn` = реестр карточек.
- **Кит НЕ потребляют** (замер): обе зовут скрипты напрямую, `kitVersion`/pins в скриптах
  нет, скиллы кит не упоминают. Киты в репо — только процедурные (angelina-morning,
  dream-master). Значит потребление кита у мастерских = null (легально, аналог
  `kitVersion: null` процедур).
- **Третий глагол `inspectElement`**: у обеих — есть ли? осмотр+декомпозиция налицо;
  «рассмотреть один элемент» (одна ветка / одна карточка вглубь) — проверить наличие.

## Что комната должна выдать

- Чеклист соответствия паттерну (по образцу `GROUP_CONTAINERIZATION`): пункты, которые
  реализация копирует в свой носитель, заменяя `[ ]` на ✅/⚠ с причиной.
- Отображение двух живущих на контракт M1/M2: `worksOn` каждой; какие из трёх глаголов
  уже есть, какого не хватает (кандидат: `inspectElement` у обеих — назвать явно).
- Форма потребления кита: `kit: null` у обеих — зафиксировать как легальное состояние
  (потребление кита нужно, когда мастерская уходит в автономный прогон; интерактивные —
  зовут скрипты напрямую). Провод «мастерская → кит» опционален.
- Что дописать каждой мастерской, чтобы стать ссылающейся реализацией: манифест
  мастерской (поля из M2: mandatory/stackLike/domain, worksOn), ссылка на паттерн,
  копия чеклиста. Объём — минимальный, без переписывания живых скриптов.
- Границу: обобщение = придание формы уже работающему, НЕ переделка поведения веток/задач.

## Список посылок — обязателен

Секция с заголовком ровно **«Список посылок»**: полный список входов, каждый — **факт**
или **норма**. Только входы. Вердикты M0, M1, M2, M4 — легальные посылки-предшественники
(M3 независим — не тащить как посылку).

## Запреты этой комнаты

- Не пересматривать контракт (M1), словарь (M2), правило принадлежности (M4) — посылки.
- Не проектировать полиморфизм 2D (M3) и достройку процедурной мастерской по существу.
- Не переписывать поведение живых скриптов веток/задач — только придать форму.
- Не пересматривать посылки MEETING_BRIEF («переспорить нельзя»).
