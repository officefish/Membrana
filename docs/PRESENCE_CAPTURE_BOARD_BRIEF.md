## Повестка

Два обязательных результата + пул улучшений кабинета:
1. **Живость присутствия должна быть достоверной**: кабинет обязан видеть узел online, когда клиент реально подключён (и offline, когда нет) — устойчиво к перезапускам клиента и рестарту сервера.
2. **Захват → view-only борд на устройстве**: при `board.capture` клиент принудительно открывает device-board в режиме просмотра, редактирование заблокировано. «Мы обязательно должны этого добиться».
3. Предложения по кабинету (см. §Предложения).

## Факты репозитория (проверено 2026-07-04)

### Присутствие (offline держится после PL1)

| Факт | Источник |
|------|----------|
| Клиент в paired-режиме поднимает node-WS **на загрузке** (`reconfigureNodeRealtimeFromConnection`) | `NodeConnectionShell.tsx:36` |
| Кабинет: `online = isDeviceLive(node.device.mediaDeviceId)` — id совпадает с presence-событиями (обе стороны `mediaDeviceId`) | `NodesPage.tsx:185-188` |
| PL1: снапшот при подключении кабинета + не обнуляем при реконнекте — **merged** | #242 |
| Node-WS auth: `authenticateNode` → `validateSessionToken`; при невалидной сессии WS закрывается **4401**, `registerNode` не вызывается → узел не в `nodeSockets` → кабинету не видно | `node-realtime-auth.service.ts:20`, `gateway:87-101` |
| Клиентский статус «связан» — из **localStorage** (credentials), НЕ из живости WS; рассинхрон не сверяется/не показывается | `nodeConnectionStore` (mode/pairing), `nodeRealtimeClient` state отдельно |
| Источник истины presence — **in-memory `nodeSockets`**; переживает клиентские реконнекты, НЕ переживает рестарт сервера. `Device.lastSeenAt` есть, но heartbeat-обновление (**PL2b**) отложено | `node-realtime.service.ts`, консилиум pairing-lifecycle |
| Dev WS-URL выводится из `getCabinetApiBase()` (HTTP→ws) | `nodeRealtimeUrl.ts` |

**Гипотезы persistent-offline (проверить в диагностике):**
- **H1 (наиболее вероятная):** клиент «связан» по localStorage, но node-WS не зарегистрирован на сервере — истёкшая/удалённая pairing-сессия (auth 4401), либо WS не доходит в dev-сетапе. Клиент этого не показывает; перезапуск не помогает, т.к. credentials те же.
- **H2:** рестарт сервера (пока шли работы) обнулил in-memory `nodeSockets`; без heartbeat/`lastSeenAt`-bootstrap (PL2b) кабинет не восстановил online до нового `nodeOnline` от узла.
- **H3:** `registerNode` ранний выход `if (!meta.mediaDeviceId) return` — устройство без `mediaDeviceId`.
- **H4:** в песочнице (Vite HMR) node-WS не переустанавливается на hot-reload, только на полный reload.

### View-only борд под захватом

| Факт | Источник |
|------|----------|
| Board canvas умеет `readOnly` (structure lock) | `board-flow-canvas.tsx:175,194` |
| Сессия `cabinet-view` / `isViewOnlySession` (system-preview, cabinet-view) | `device-board-session.ts:18-23` |
| `resolveDeviceCaptureFlags` — флаги enforcement по захвату (edit/pause) | `packages/device-board` (CT tariff v2) |
| **Нет** авто-открытия device-board модуля на клиенте при `board.capture` — захват гейтит редактирование, но не навигирует/не форсит открытие борда | нет кода force-open |

**Итог:** блоки для view-only есть; недостаёт «на `board.capture` клиент форсит открытие device-board в view-only и держит его, снимая при release».

| # | Тема | Суть |

| A1 | Диагностика присутствия | Инструментировать node-WS lifecycle (connect/auth-fail/register/close) в лог клиента+сервера; воспроизвести H1–H4 на связке 1573↔кабинет |
| A2 | Реальная живость WS на клиенте | Отделить «связан» (credentials) от «на связи» (WS registered); показывать оператору фактическое состояние; при auth-fail — явный статус (сессия истекла → пере-пейринг) |

## Предложения по кабинету (мои, на обсуждение)

- **P1.** Индикатор живости с `lastSeenAt` («виден 8с назад») — оператор отличает «реально online» от «застряло».
- **P2.** Реконсиляция на клиенте: баннер «сопряжён, но нет связи с кабинетом» при paired-без-WS — прямо снимает жалобу.
- **P3.** Кнопка «переподключить» узел в кабинете (принудительный ping/health) для диагностики оператором.
- **P4.** Тех-долг: формальные closure-review PL2–PL5 (были смёржены при инфра-флейке review-runner, домен cg2) — до-оформить или явно списать.
- **P5.** Второй флейк `background-cabinet` (vitest-воркер под нагрузкой, «Unexpected end of JSON input») — отдельная cg-задача.
