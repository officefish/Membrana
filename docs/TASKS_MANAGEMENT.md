# TASKS_MANAGEMENT — методология работы с задачами

> Стратегический документ. Описывает, **где** и **как** живут задачи в проекте Membrana,
> чтобы они не растворялись в чатах с агентами и не дублировались между инструментами.
>
> Статус: **v0.1 — концепт.** Хранитель: Teamlead. Правки — через PR с пометкой `/architect`.

---

## 1. Зачем этот документ

В проекте параллельно работают человек и несколько AI-агентов. Без единых правил задачи
рассыпаются: что-то обсуждается в чате с агентом, что-то в комментарии к коммиту,
что-то живёт «в голове». Этот документ фиксирует **одно место для входа, одно — для работы
и один обязательный артефакт на выходе**.

---

## 2. Принципы

1. **Один вход — GitHub Issues.** Любое пожелание, замечание, недоделка, идея —
   создаётся как GitHub Issue по одному из трёх шаблонов
   (`wish`, `bug`, `imperfection`). Чаты с агентами — инструмент формулирования,
   но **итог обсуждения обязан стать Issue**.
2. **Одна рабочая зона — Linear.** Реальная декомпозиция, статусы, циклы,
   внутренние обсуждения хода работы — в Linear. Это снимает шум с GitHub
   и даёт нормальный workflow исполнителю.
3. **Обязательный отчёт перед закрытием.** Прежде чем закрыть GitHub Issue,
   автор PR оставляет в нём формальный отчёт (раздел 6). Без отчёта Issue
   не считается закрытой — Teamlead может переоткрыть.
4. **Архив — закрытые Issues.** GitHub хранит их вечно, ищет фильтрами,
   поэтому отдельные «архивные» папки не нужны.
5. **Каждая задача знает свою привязку к стратегии.** В Issue по возможности
   указан этап или раздел `WHITE_PAPER.md`. Это не бюрократия — это позволяет
   `STRATEGIC_PLAN_WEEK.md` ссылаться на задачи и видеть, какие из них приближают
   нас к целям, а какие — нет.

---

## 3. Жизненный цикл задачи

```
┌──────────────────────────────────────────────────────────────────┐
│ [1] Идея / пожелание / замечание / недоделка                     │
│     —— могут родиться где угодно (чат, code review, разговор) —— │
└────────────────────────────────┬─────────────────────────────────┘
                                 │ ОБЯЗАТЕЛЬНО оформляется
                                 ▼
┌──────────────────────────────────────────────────────────────────┐
│ [2] GitHub Issue (wish / bug / imperfection)                     │
│     status:triage, минимальный набор полей                       │
└────────────────────────────────┬─────────────────────────────────┘
                                 │ Teamlead на триаже
                                 ▼
┌──────────────────────────────────────────────────────────────────┐
│ [3] Linear ticket — создан, привязан к Issue по ссылке/ID        │
│     На GitHub Issue ставится label `status:linear`               │
└────────────────────────────────┬─────────────────────────────────┘
                                 │ работа исполнителя
                                 ▼
┌──────────────────────────────────────────────────────────────────┐
│ [4] Внутренние обсуждения, декомпозиция, статусы — В LINEAR.     │
│     GitHub Issue «спит» — никакой активности.                    │
└────────────────────────────────┬─────────────────────────────────┘
                                 │ реализация
                                 ▼
┌──────────────────────────────────────────────────────────────────┐
│ [5] PR в GitHub. В описании PR:                                  │
│       - Linear-ID (например MEM-42)                              │
│       - "Closes #N" — номер GitHub Issue                         │
└────────────────────────────────┬─────────────────────────────────┘
                                 │ merge
                                 ▼
┌──────────────────────────────────────────────────────────────────┐
│ [6] Формальный отчёт в комментарии GitHub Issue → Issue closed.  │
│     Linear ticket — Done (через интеграцию).                     │
└──────────────────────────────────────────────────────────────────┘
```

## 4. Этап 1–2: создание Issue в GitHub

### Шаблоны

Лежат в `.github/ISSUE_TEMPLATE/`:

| Шаблон | Когда использовать |
|--------|--------------------|
| **Wish** (`wish.yml`) | Пожелание, идея, фича, расширение возможностей. |
| **Bug** (`bug.yml`) | Что-то работает не так, как ожидалось; регрессия. |
| **Imperfection** (`imperfection.yml`) | Недоделка, технический долг, дублирование, пропущенные тесты, устаревшая документация. |

Создание пустого Issue **отключено**. Это намеренно: помогает не превращать Issues
в свалку.

### Что обязано быть в любом Issue

- **Заголовок в форме «глагол + цель».** Не «FFT», а «Добавить SNR-индикатор на
  спектр FFT в `microphone-stream-viz`».
- **Контекст.** Зачем это нужно, какую боль решает. Один абзац — достаточно.
- **Acceptance criteria.** 2–5 проверяемых пунктов: что считается «сделано».
- **Связь с `WHITE_PAPER.md`** — по возможности (этап / раздел).

### Чего в Issue быть **не должно**

- **Подробного плана реализации.** Декомпозиция — задача Linear.
- **Сроков в днях/неделях.** Размер задачи (S/M/L) и зависимости — да; календарь — нет.
- **Длинных переписок.** Если обсуждение пошло — это сигнал перенести его в Linear.

## 5. Этап 3–4: работа в Linear

Linear — единственное место, где задача:

- декомпозируется на подзадачи,
- получает исполнителя и роль виртуальной команды
  (Teamlead / Структурщик / Математик / Музыкант / Верстальщик
  — см. `VIRTUAL_TEAM_PROMPT.md`),
- переходит по статусам (Backlog → To Do → In Progress → Review → Done),
- обсуждается внутренне (комментарии в Linear).

GitHub Issue в это время **тихая**. Никаких параллельных дискуссий.

### Связка Linear ↔ GitHub

Linear имеет встроенную интеграцию с GitHub. Настраивается **один раз**:

1. Linear → Settings → Integrations → GitHub → Connect.
2. Подключить репозиторий `officefish/Membrana`.
3. Включить:
   - **Auto-link issues**: при упоминании `MEM-42` в названии ветки / PR /
     commit сообщения Linear автоматически линкует их.
   - **Auto-close on merge**: после merge PR Linear-ticket переходит в Done.
   - **Sync labels** — опционально.

После настройки достаточно упоминать ID Linear-ticket'а где угодно
(ветка, PR, commit) — связь подхватится сама.

### Конвертация GitHub Issue → Linear ticket

Доступны три способа:

1. **Через интеграцию.** В GitHub Issue нажать `…` → `Create Linear issue`
   (если включена соответствующая опция в Linear).
2. **Вручную.** Создать ticket в Linear, в его описании дать прямую ссылку на
   GitHub Issue. Далее — в GitHub Issue добавить комментарий с Linear-ID.
3. **При сильной интеграции** — через webhook (не обязательно для v0.1).

В любом случае после конвертации:

- на GitHub Issue ставится label **`status:linear`**;
- в комментарии указывается Linear-ID и ссылка.

## 6. Этап 5–6: PR, отчёт, закрытие

### PR

Любой PR, связанный с задачей, **обязан** содержать:

- В описании или заголовке — Linear-ID (например `MEM-42`).
- В описании — строку `Closes #<number>`, где `<number>` — GitHub Issue.
  Это автоматически закроет Issue при merge.

### Формальный отчёт перед закрытием

Перед самим закрытием (или сразу после авто-закрытия) автор PR оставляет
в GitHub Issue **последний комментарий** по форме:

```markdown
## Отчёт о выполнении

**Что сделано.** Краткое (3–6 строк) описание реализованного.

**PRs.** Список ссылок на PR.

**Linear ticket.** Прямая ссылка (`https://linear.app/.../MEM-42`).

**Связь со стратегией.** Какой этап дорожной карты WHITE_PAPER был продвинут.

**Известные нюансы / отложено.** Что осталось за рамками или известные
ограничения. Если ничего — «—».
```

Этот отчёт — единственная вещь, которая остаётся доступной всем, в том числе
будущим агентам, без доступа в Linear. Поэтому он обязан быть осмысленным,
а не отпиской.

### Когда **не** оставлять отчёт

Только в трёх случаях:

- Issue закрыт как `wontfix` — с короткой причиной в комментарии.
- Issue закрыт как `duplicate` — со ссылкой на оригинал.
- Issue закрыт как `out-of-scope` — с пояснением, почему вынесено.

## 7. Label-структура GitHub Issues

Минимальная и ортогональная. Тонкая классификация уже в Linear.

| Группа | Label'ы | Назначение |
|--------|---------|------------|
| **Тип** | `wish`, `bug`, `imperfection` | Категория Issue. Ставится шаблоном автоматически. |
| **Статус** | `status:triage`, `status:linear`, `status:done` | `triage` — только создан, ждёт триажа; `linear` — переведён в Linear, идёт работа; `done` — ждёт отчёта и закрытия. |
| **Пакет** | `package:core`, `package:agenda`, `package:device-board`, `package:audio-engine-service`, `package:fft-analyzer-service`, `package:client`, `package:infra`, `package:docs` | Удобная фильтрация по пакетам монорепо. |
| **Стоп-сигналы** | `blocked`, `wontfix`, `duplicate`, `out-of-scope` | По мере необходимости. |

Этап (`stage:*`), роль (`role:*`), размер (`size:*`) **не** дублируются в GitHub-labels —
это поля Linear-ticket'а.

## 8. Когда какой инструмент использовать

| Сценарий | Куда |
|----------|------|
| «Хочу добавить X» (от тебя, коллеги, пользователя) | GitHub Issue → `wish` |
| «Тут баг» / «ведёт себя странно» | GitHub Issue → `bug` |
| «Стоит почистить foo.ts» / «забыли тесты» | GitHub Issue → `imperfection` |
| «Как лучше реализовать классификатор?» | Linear ticket comments (после триажа) |
| «План на неделю» | `STRATEGIC_PLAN_WEEK.md` (автогенерация) — может ссылаться на номера Issues |
| Стратегическая цель проекта | `WHITE_PAPER.md` |
| Большое архитектурное решение | пока — `docs/ARCHITECTURE.md` или PR с пометкой `/architect`. ADR-формат можно ввести позже отдельным документом. |

---

## 9. Что **не** делаем

- **Не превращаем чат с агентом в задачу.** Чат — инструмент формулирования.
  Если разговор привёл к ясному пункту — оформи Issue. Если нет — закрой чат.
- **Не дублируем обсуждение** в GitHub Issue и Linear. Создание + отчёт —
  GitHub. Внутренняя кухня — Linear.
- **Не закрываем Issue молча.** Даже при `wontfix` — короткое объяснение.
- **Не создаём задачи в личке / DM.** Просьба коллеги в DM — это сигнал
  попросить его открыть Issue по шаблону. Если он не может — открой сам
  и тегни автора.
- **Не вешаем `status:linear` на Issue, для которого Linear-ticket не создан.**
  Это нарушает целостность связки и сбивает фильтры.

## 11. Чек-листы

### Создатель Issue

- [ ] Выбран правильный шаблон (`wish` / `bug` / `imperfection`).
- [ ] Заголовок в форме «глагол + цель».
- [ ] Заполнен контекст (зачем).
- [ ] Указаны acceptance criteria (2–5 пунктов).
- [ ] По возможности — этап / раздел `WHITE_PAPER.md`.

### Triage (Teamlead)

- [ ] Проставлены `package:*` (если применимо).
- [ ] Создан Linear ticket и оставлена ссылка в комментарии.
- [ ] Метка `status:triage` снята, поставлена `status:linear`.
- [ ] Если задача нерелевантна — закрыта с одной из меток `wontfix` / `duplicate` / `out-of-scope` и короткой причиной.

### Закрытие Issue после реализации

- [ ] Все упомянутые PR смержены.
- [ ] Linear ticket — Done.
- [ ] Оставлен формальный отчёт по форме из раздела 6.
- [ ] Issue закрыта (или закрыта автоматически через `Closes #N`).
