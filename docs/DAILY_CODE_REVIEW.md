## Что проверено глубоко — и держится

**Разделение привилегий partner-access (главный риск эпика #463).** Проверял путь
самоэскалации внешнего партнёра:

- **Гранты и роль — разные оси.** `grants` гейтят разделы UI (`grantsAllowSection`),
  `role` гейтит admin-API. Партнёр по промокоду всегда `role: 'ally'`; даже с
  wildcard-грантом `'*'` он получает **404 на `/v1/panel/admin/*`** (`@OwnerAdmin` =
  `MinRole('owner')` + `denyAs404`). Пути «грант → owner-действие» нет.
- **`@OwnerAdmin` корректен** (`panel-auth.decorators.ts:17`): не-owner не только не
  проходит `canAccess(role,'owner')`, но и не подтверждает существование ручки (404, Q3).
- **Гонка `usedCount` промокода невозможна:** `PanelUsersStore.mutate` синхронный
  (`fn(state)` → sync `persist` tmp→rename → присвоение, без `await`); `register`/
  `createCode` тоже не-async. Однопоточный Node сериализует read-modify-write.
- **Регистрация закалена:** отдельный лимитер 5/час на клиента (жёстче общего
  панельного), ошибка не различает причину наружу (`code was not accepted` на
  нет-кода/истёк/исчерпан/отозван — Q3), промокод показывается наружу целиком
  ровно один раз при чеканке (дальше только префикс).

**Экспортёр detector-compare (#452).** Переиспользует `benchmark-detectors.mjs`
(те же детекторы/манифест/пороги — паритет, не копия), пояснения детерминированными
шаблонами без LLM, идемпотентность через `reportsEqualIgnoringGeneratedAt`
(повторный прогон не трогает `generatedAt`). Границы консилиума соблюдены.

## P2 — закалка (не блокеры)

1. **`panel-users.controller.ts` — snapshot-first blind-write** (`register:93-94`,
   `createCode:190-196`): `mutate(() => result.state)` игнорирует текущий стейт и
   пишет предвычисленный из `snapshot()`. **Сейчас безопасно** (нет `await` между
   snapshot и mutate — код это комментирует), но хрупко: если кто-то позже вставит
   `await` в этот промежуток, параллельная запись будет затёрта (lost update). Совет:
   пересчитывать мутацию ВНУТРИ колбэка `mutate` (`mutate(state => redeemCode(state,…))`)
   — тогда инвариант держится структурно, а не по дисциплине.
2. **`panel-users-core.ts:75-80` — `parseState` не валидирует элементы:**
   `s.users as PanelUser[]` доверяет форме объектов. Валидный-JSON-но-битый
   store-файл (частичная порча полей) пройдёт. Риск низкий (файл app-owned,
   atomic-запись), но поэлементная проверка укрепила бы «видимую деградацию».
3. **`panel-users.store.ts:90-96` — `persist` без fsync:** tmp→rename даёт
   атомарность *свопа*, но не *долговечности содержимого* при потере питания.
   Для реестра партнёров некритично; отметить в runbook про бэкап (он уже есть).
