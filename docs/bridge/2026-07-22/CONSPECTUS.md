## Тезис мостика: реестр имущества

Капитан (22.07): «Мы заведём реестр имущества в том числе».

Механизм верификации завоза станков (принцип factory-before-product): перепись
активов системы — станков/оборудования (генераторы, фреймы, гейты, инструменты,
процедуры), по которой система **удостоверяется**, что оборудование реально
установлено и отвечает на вопрос «под какую продукцию». Край фазы оснастки
становится проверяемым фактом, не декларацией.

Связи: инсайт change-routing-office (именные гейты, атрибуция), фреймы #900
(лицо + версия у каждого станка), реестр процедур (17 жильцов — прообраз части
имущества), реестр долгов попугая (обратная сторона: чего НЕ хватает).
Статус: тезис капитана, не спроектирован. Кандидат в шторм/инсайт.

### Опись имущества: чек + фотография (капитан, 22.07)

Капитан: «Могу предложить чеки и фотографию». Двухчастная запись актива:
- **Чек** = провенанс приобретения: git-SHA + PR + дата въезда станка. Неизменяемо
  («откуда/когда взяли»). Уже есть: провенанс-строки, owner-choice с хешем, append-only.
- **Фотография** = снимок ТЕКУЩЕГО состояния: станок на полке сейчас — файл есть,
  тест зелёный, validateProcedure проходит в эту секунду. «Сверка с полками» (заноза
  Фаррелла). Фото устаревает → переснимать (re-probe периодически).

Инвариант: чек без фото = «купили, а стоит ли ещё?»; фото без чека = «стоит, но
чьё/откуда?». Оба обязательны. Механики существуют (провенанс + artifact-freshness/
validateProcedure) — реестр имущества их СВОДИТ, не изобретает.

Капитан: «Дата сегодняшняя. И подтверждать всё нужно по идее сегодняшним же числом».

### Развилка: контейнер имущества → в СОВЕЩАНИЕ (капитан, 22.07)

Капитан: «Давай отправим в совещание, пусть команда обсуждает. Там всё по аналогии

Ключевые семена для заседания:
- Контейнер имущества строится ПО АНАЛОГИИ (GROUP_CONTAINERIZATION): контейнер +
  инструменты + Google Drive «под капотом» (вещдоки-бинарь наружу, метаданные в git).
- **Родовой узор расширяется третьим доменом.** Версионируемый сабграф контейнера:
  в скриптах = КИТ, в процедурах = ФРЕЙМ, **в имуществе = НАБОР**. Все три —
  PINNED_SUBGRAPH_VERSIONING.
- **Набор формируется на основании ЧЕКА**: один чек (покупка) → один набор активов.
  Чек — организующий принцип группировки/версии набора (как импорты у кита,
  отрезки у фрейма).
- Размещение реестра: git-дом docs/assets/ (выбор капитана); вещдоки — Drive-ссылками.
- Подтверждение — датированный акт «сегодняшним числом» (гейт RT-9).

Статус: НЕ строим соло — вход в заседание «Контейнер имущества». Преждевременное
ядро (scripts/lib/assets.mjs) снято: архитектуру решает команда.
