# RESEARCH: Ресёрч: Linear — личность агента, состояние подзадач, блокирующие связи

> Сгенерировано `yarn research` (Perplexity API), 2026-07-19.
> Выжимка — вход для решения, а не решение: проверяй утверждения по нашему коду.
## Q1 — Landscape

**Запрос:** как в Linear в 2025–2026 оформляется агент или бот как отдельный субъект действия — какие сущности для этого существуют (agent, application, OAuth actor, интеграция), и как ДЕЙСТВИЕ такого субъекта выглядит в исходящем вебхуке и в ответе GraphQL API: какие поля идентифицируют актора и чем они отличаются от полей действия человека-пользователя?

**Выжимка:**

В Linear на 2025–2026 год агент оформляется как отдельный субъект действия через новую сущность **Agent** (также называемую **«app user»**), которая реализуется технически как специальный пользователь-аккаунт внутри воркспейса, созданный по OAuth-режиму с параметром **`actor=app`** [1][3].

### 1. Сущности и иерархия
Для оформления агента используются следующие ключевые компоненты:
*   **Agent (App User)**: Основная сущность. Это не просто «интеграция» или «бот», а полноценный пользователь с уникальным `id`, именем и аватаром, который отображается в списке назначаемых (assignee) и может быть @-mention [1][3].
*   **OAuth Actor (`actor=app`)**: Специальный режим OAuth-флага, который при установке интеграции создает выделенного пользователя агента [3].
*   **Application**: Базовая OAuth-интеграция, которая выступает «оберткой» для агента, но действие выполняет именно созданный внутри неё **Agent-user** [3].
*   **Инструменты доступности (Scopes)**: Агент получает права через специфические селекторы, например `app:assignable` (для назначения задач) и `app:mentionable` (для упоминаний), что отличает его от обычных интеграций [3].

### 2. Как действие агента выглядит в исходящем вебхуке
При создании **Agent Session** (сессии взаимодействия агента) Linear отправляет вебхук со структурированным контекстом. В этом вебхуке актор идентифицируется не как человек, а как приложение:
*   **Идентификатор актора**: В поле, обозначающем триггер (кто вызвал событие), передается `id` созданного **Agent-user** (тот же `user.id`, что и у людей, но с типом «app») [3].
*   **Отличие от человека**: Контекст содержит явное указание на тип сессии («agent session»). Если событие вызвано делегированием задачи (delegation), в объекте задачи будет два владельца:
    1.  `assignee`: Человек (ответственный за результат).
    2.  `delegatedAgent`: Агент (ответственный за выполнение действия) [3].
*   **Состояние**: Вебхук также включает состояние сессии актора (waiting, working, completed, errored), что не применяется к действиям обычных пользователей [3].

### 3. Как действие агента выглядит в ответе GraphQL API
В GraphQL-ответах (например, при запросе `issue` или `user`) действия агента идентифицируются через те же поля ID, но с критическими семантическими отличиями:
*   **Поля идентификации**:
    *   `id`: Уникальный UUID сущности (совпадает с форматом у людей).
    *   `name`: Имя агента (например, «Linear Agent»).
    *   `isAgent` (или аналогичный флаг типа): Поле, которое явно возвращает `true` для агентов, отличая их от людей (`isUser: false`) [3].
*   **Визуальное отличие**: В объекте пользователя (например, в `issue.assignee`) агент всегда отображается с явной маркировкой **«agent, not person»**, чтобы отличие было всегда видно [3].
*   **Семантика действия**:
    *   Человек: `assignee` = человек.
    *   Агент: `assignee` = человек, но появляется новое поле или связь `delegatedTo` = агент. Агент не может быть единственным «assignee» в традиционном смысле ответственности, только «делегатом» [3].
*   **Ограничения**: В API агенты могут выполнять любые мутации, доступные людям, но у них **нет возможности** входить в интерфейс Linear как пользователь (нет UI-сессии) [4].

### Ключевые отличия полей актора (Агент vs Человек)

| Характеристика | Человек (Human User) | Агент (Agent / App User) |
| :--- | :--- | :--- |
| **OAuth режим** | `actor=user` (обычный) | `actor=app` [3] |
| **Тип в объекте** | `isUser: true` | `isAgent: true` (или `isApp: true`) [3] |
| **Статус ответственности** | Единый `assignee` | `assignee` (человек) + `delegatedAgent` (агент) [3] |
| **Вход в UI** | Да (логин по паролю/SSO) | Нет (ограничение на вход в UI) [4] |
| **Сессия** | `userSession` | `agentSession` с трекингом состояния [3] |
| **Семантика упоминания** | Стандартное упоминание | Требует селектора `app:mentionable` [3] |

Таким образом, Linear в 2026 году формализирует агента не как внешнюю интеграцию, а как **виртуального пользователя** с уникальным токеном, явным флагом типа и специальной семантикой делегирования задач, что четко отражается в `webhook.context.actor.type` и GraphQL-ответах через поля типа `isAgent` и структуру делегирования.

## Q2 — Fit (Membrana)

**Запрос:** в модели данных Linear — есть ли у элемента подзадачи или чеклиста внутри issue собственное состояние жизненного цикла и собственный исполнитель, доступные через API, или состояние и исполнитель существуют только на уровне самой issue; и как принято моделировать «контейнер работ с одной центральной задачей и несколькими подчинёнными»?

**Выжимка:**

В модели данных Linear у каждой **подзадачи (sub-issue)** есть **собственное статусное состояние** (например, Backlog, Todo, In Progress, Done) и **собственный исполнитель (assignee)**, которые доступны через API как независимые свойства этого элемента, а не только на уровне родительской issue [2][4].

Встроенный элемент **чеклиста** (текстовый список внутри описания задачи) в Linear не является отдельной сущностью с собственным жизненным циклом или исполнителем; для моделирования таких работ используется именно иерархия **sub-issues** [4].

### Детали доступности через API и статусы
*   **Собственное состояние:** Подзадача отслеживает свой статус независимо от родительской. Например, родительская задача может быть в статусе `In Progress`, пока одна из её подзадач находится в `Todo`, а другая — в `Done` [4].
*   **Собственный исполнитель:** У каждой подзадачи можно назначить отдельного `assignee`, что позволяет распределять роли внутри одной крупной задачи [2][4].
*   **API:** Поскольку sub-issues являются полноценными объектами `Issue` в системе, все их свойства (status, assignee, priority) доступны через стандартные методы API Linear так же, как и у родительских задач [2].

### Моделирование «контейнера работ с одной центральной задачей и подчинёнными»
В Linear принято моделировать такую структуру через иерархическую связь **Parent Issue (центральная задача) → Sub-issues (подчинённые)**, где:

| Компонент | Роль в модели | Особенности |
|-----------|---------------|-------------|
| **Родительская Issue** | Центральная задача (Feature/Milestone) | Предоставляет **агрегированный вид** общего прогресса фичи; её статус часто обновляется автоматически в зависимости от завершения подзадач [4] |
| **Sub-issues** | Подчинённые работы (Design, API, UI, Testing) | Каждый элемент — **атомарная задача** с приоритетом, статусом, исполнителем и оценкой, выполняемая independently [4] |
| **Projects** | Группировка нескольких таких иерархий | Если работ много, их объединяют в **Project** (контейнер уровня команды), который имеет свой таймлайн и статус прогресса (On Track/At Risk) для всей группы [2][4] |

**Рекомендация по моделированию:**
Для реализации «контейнера с центральной задачей» создайте крупную **Feature Issue** (родитель), а внутри неё разбейте работу на **Sub-issues** (например, «Design review», «Backend API», «Frontend UI», «Testing», «Documentation»). Это позволяет сохранять granular task tracking (детальную учёт задач) при одновременном high-level feature visibility (видимости общего прогресса) [4].

Если задача требует стратегического планирования (квартальный роадмэп), такую группу можно поместить в **Initiative**, которая объединяет несколько Projects под общей целью [2][5].

## Q3 — Risk

**Запрос:** как в Linear выражаются блокирующие отношения между задачами (blocks/blocked-by, зависимости) и наблюдаемы ли они через API и вебхуки для внешнего гейта; какие известны ограничения — rate limits, потеря событий, отсутствие истории переходов — при построении внешних проверок поверх Linear как источника истины?

**Выжимка:**

В Linear блокирующие отношения между задачами выражаются через поля **`blockedBy`** (заблокировано) и **`blocking`** (блокирует), которые являются атрибутами задачи (issue), а не отдельным типом связи на уровне проектов. Эти зависимости **наблюдаемы через API** (GraphQL), но **не поддерживаются вебхуками** для событий типа «блокировка/разблокировка» как отдельного триггера, что требует построения внешних гейтов на основе периодического мониторинга состояния задач.

### 1. Как выражаются зависимости в Linear

В Linear зависимости между задачами (issues) реализуются как прямые ссылки на другие задачи:
*   **Поля задачи:** У каждой задачи есть список `blockedBy` (задачи, которые её блокируют) и `blocking` (задачи, которые она блокирует) [1].
*   **Тип зависимости:** На уровне задач (issues) поддерживается только отношение «финиш — старт» (finish-to-start), то есть задача B может начаться только после завершения задачи A. На уровне проектов (Project Dependencies) также поддерживается только `end -> start` зависимость [1].
*   **Визуализация:** В таймлайнах проектов зависимости отображаются линиями (синие — валидны, красные — нарушены) [1], но на уровне отдельных задач это просто список ID связанных задач.

### 2. Доступность через API и вебхуки

#### API (GraphQL)
Зависимости полностью доступны через GraphQL API:
*   Вы можете запросить поля `blockedBy` и `blocking` в объекте `Issue` [1].
*   Пример запроса:
    ```graphql
    query {
      issue(id: "TASK_ID") {
        blockedBy { id title }
        blocking { id title }
        state { name }
      }
    }
    ```
Это позволяет внешнему гейту (например, CI/CD пайплайну) проверять: «Заблокирована ли задача X?», «Когда ожидается завершение задачи в `blockedBy`?».

#### Вебхуки (Webhooks)
На текущий момент **вебхуки Linear не эмиттируют событие изменения зависимостей** как отдельный триггер.
*   Вебхуки эмиттируются при изменениях задачи (`issue.created`, `issue.updated`, `issue.deleted`), но в payload события `issue.updated` нет специального флага, указывающего, что изменились именно зависимости [3].
*   **Проблема:** Чтобы обнаружить блокировку, внешний сервис должен получать общее событие `issue.updated` и сравнивать старое и новое состояние списков `blockedBy`/`blocking`. Это требует сохранения истории состояний или повторного запроса задачи через API после каждого вебхука.
*   **Отсутствие истории переходов:** В вебхуках не передается история изменений (change log). Вы не получите информацию о том, *кто* и *когда* добавил зависимость, только факт обновления задачи.

### 3. Ограничения при построении внешних проверок

При использовании Linear как «источника истины» для внешних гейтов (например, запрет мерджа кода, если задача блокирует релиз) существуют следующие ограничения:

| Ограничение | Описание и влияние на гейт |
|-------------|----------------------------|
| **Rate Limits** | Лимит составляет **3,000,000 точек в час** для API-ключей (на пользователя) и **100,000 точек** для неавторизованных запросов (на IP) [8]. Сложность одного запроса ограничена **10,000 точек** [8]. Если гейт будет часто проверять состояние задач (polling), он может быстро превысить лимит, что приведет к ответу `400` с ошибкой `RATELIMITED` [8]. |
| **Потеря событий (Deduplication)** | Вебхуки могут доставляться повторно, но не гарантируют доставку 100% событий в случае сбоя сети. Если вебхук пропущен, и гейт не ведет периодический polling (synchronization), он может не заметить изменение зависимостей. |
| **Отсутствие истории переходов** | Вебхуки не содержат `audit log` изменений. Нельзя автоматически определить момент начала блокировки без повторного запроса к API и сравнения состояния. Это усложняет триггеры типа «задача заблокирована более 2 часов». |
| **Асинхронность данных** | API Linear может иметь небольшую задержку синхронизации сразу после вебхука. Прямой запрос к API сразу после получения `issue.updated` вебхука может вернуть «старые» данные зависимости. Рекомендуется задержка (debouncing) или повторный запрос. |
| **Ограничение сложности** | Запрос к GraphQL, который пытается загрузить все задачи проекта с зависимостями (например, `issues { blockedBy { ... } }`), может превысить лимит сложности 10,000 точек, если проект большой [8]. |

### Рекомендация для внешнего гейта
Для надежной проверки блоков поверх Linear:
1.  **Не используйте только вебхуки:** Implement `polling` (периодический запрос) критических задач каждые N минут для синхронизации состояния, так как вебхуки не дают детальной истории изменений.
2.  **Используйте GraphQL:** Запрашивайте поля `blockedBy` и `state` в одном запросе, чтобы минимизировать количество вызовов и избежать превышения лимитов [8].
3.  **Учитывайте лимит сложности:** Если задача зависит от множества других, разбивайте запросы на части или используйте фильтрацию, чтобы не превысить 10,000 точек сложности [8].

Важно уточнить: если вам нужны зависимости именно между **проектами** (Project Dependencies), они также доступны через API, но логика их блокирования (timeline violation) работает на уровне дат проектов, а не статусов задач [1]. Для гейтинга задач (issues) используйте поля задачи.
