## Q1 — Landscape

**Запрос:** Какие практики сложились в 2025–2026 для содержания каталога внутренних build- и tooling-скриптов в монорепозитории на Turborepo, Nx, pnpm или yarn workspaces: держат его отдельным workspace-пакетом (`@scope/scripts`, `tools/`, `internal/`) или голым каталогом вне графа workspace? На каком масштабе — число скриптов, строк кода, частота правок — команды переводят такой каталог в пакет, и что конкретно они этим получают: единый lint и typecheck через задачи планировщика, правила границ импортов, версионирование, кеширование? Какие известные минусы у каждого из двух подходов и почему часть команд сознательно оставляет tooling вне workspace?

**Выжимка:**

В 2025–2026 году доминирующей практикой для каталога внутренних build- и tooling-скриптов в монорепозиториях на Turborepo, Nx или pnpm/yarn workspaces является **выделение их в отдельный workspace-пакет** (например, `@scope/scripts`, `tools/`, `internal/`), а не хранение в «голом» каталоге вне графа. Команды переходят к пакетной модели, когда количество скриптов превышает **10–15**, частота правок становится высокой, или требуется **единая типизация и линтинг** для инструментов, которые развиваются как продукт [5][9].

### Ключевые мотивы перехода в пакет (что конкретно получают команды)

| Возможность | Описание преимущества |
|-------------|----------------------|
| **Единый lint и typecheck** | Планировщик задач (Turborepo/Nx) запускает линтинг и проверку типов для скриптов так же, как для приложений, обеспечивая консистентность [5][13]. |
| **Правила границ импортов** | В Nx/Turborepo можно настроить `imports` в `nx.json`/`turbo.json`, запрещая импорты из `apps` внутрь `tools` и наоборот, предотвращая циклические зависимости [1][9]. |
| **Версионирование и зависимости** | Скрипты становятся пак errno с `package.json`, что позволяет использовать `dependencies` (например, `zod`, `chalk`) и версионировать инструменты через `Changesets` [9][10]. |
| **Кеширование задач** | При правильном указании `outputs` в `turbo.json`/Raw-скрипты кешируются в `local`/`remote cache`, ускоряя CI/CD [1][4][12]. |
| **Модульность и рефакторинг** | Изменения в скриптах видны сразу везде, упрощается атомарный рефакторинг и синхронизация зависимостей [3]. |

### Масштаб для перехода в пакет
Команды обычно переводят каталог tooling в пакет, когда:
- **Число скриптов**: >10–15 (например, скрипты миграции, генерации, деплоя, CI-утилиты).
- **Строки кода**: >1–2 тыс. строк TypeScript/JS.
- **Частота правок**: несколько изменений в неделю (скрипты эволюционируют как код).
- **Наличие зависимостей**: когда скрипты используют внешние пакеты (не только стандартные Node-модули).

### Минусы каждого подхода

#### 1. Отдельный workspace-пакет (`@scope/scripts`, `tools/`)
| Плюсы | Минусы |
|-------|--------|
| ✔️ Единый lint/typecheck кеширование | ❌ **Увеличение сложности**: нужно управлять `package.json`, `tsconfig.json`, зависимостями [1][11]. |
| ✔️ Строгие границы импортов | ❌ **Риск циклических зависимостей**: если скрипты импортируют из `apps` или `packages` [1]. |
| ✔️ Версионирование и `dependencies` | ❌ **Монорепозиторная «нагрузка»**: для малых проектов (5–7 скриптов) overhead может быть избыточным [11]. |
| ✔️ Интеграция в CI/CD pipeline | ❌ **Проблема с `outputs`**: без точного описания `outputs` кеш не работает, что ведет к повторным сборкам [1][12]. |

#### 2. Голый каталог вне графа workspace (`scripts/`, `tools/`)
| Плюсы | Минусы |
|-------|--------|
| ✔️ **Простота**: нет `package.json`, зависимостей, конфигураций [8]. | ❌ **Нет единого lint/typecheck**: скрипты линтуются отдельно, часто пропускаются ошибки [13]. |
| ✔️ **Минимум конфигурации**: подходит для малых проектов (<10 скриптов) [11]. | ❌ **Нет кеширования**: планировщик не знает о скриптах, кеш не применяется [1]. |
| ✔️ **Быстрое добавление**: можно создать скрипт без инфраструктуры [10]. | ❌ **Нет границ импортов**: скрипты могут импортировать из `apps`, нарушая модульность [9]. |
| ✔️ **Нет версионирования**: не нужно обновлять пакеты [9]. | ❌ **Сложность рефакторинга**: изменения в скриптах не видны в графе зависимостей [3]. |

### Почему часть команд сознательно оставляет tooling вне workspace?
1. **Малый масштаб**: при <10 скриптов и низкой частоте правок overhead пакета избыток [11].
2. **Отсутствие зависимостей**: если скрипты используют только стандартные Node-модули (например, `fs`, `path`, `child_process`), нет смысла в `package.json`.
3. **Скорость добавления**: для экспериментальных/временных скриптов (например, `scripts/migrate-legacy.ts`) проще создать файл без инфраструктуры.
4. **Проблема с `outputs`**: если сложно точно описать `outputs` (например, скрипт генерирует файлы в разных директориях), кеширование не работает, а пакет не дает преимуществ [1][12].
5. **Риск циклических зависимостей**: если скрипты импортируют из `apps`, это нарушает DAG-граф, и некоторые команды избегают этого, оставляя скрипты вне графа [1].

### Рекомендация для 2025–2026
- **Выделяйте в пакет**, если: >10 скриптов, >1–2 тыс. строк, частые правки, нужны внешние зависимости.
- **Оставляйте вне workspace**, если: <10 скриптов, только стандартные модули, экспериментальные/временные задачи.
- **Используйте Turborepo + pnpm** как default для большинства монорепозиториев в 2026, а **Nx** для enterprise-scale проектов с сложной графовой логикой [5].

Для максимальной эффективности **настройте `outputs` в `turbo.json`** и **ограничьте импорты** через `nx.json`/`turbo.json`, чтобы избежать циклических зависимостей и обеспечить кеширование [1][9][12].

## Q2 — Fit (Membrana)

**Запрос:** Маленькая команда с LLM-агентами держит 216 Node ESM-скриптов (36 000 строк, расширение `.mjs`) вне workspace монорепозитория; эти скрипты генерируют дневные планы, считают метрики достоверности и отправляют отчёты внешним партнёрам. Каталог не покрыт линтом (в конфиге ESLint нет override для `.mjs`, прямой прогон не парсит ни одного файла), не покрыт typecheck (нет tsconfig), список тестов перечислен вручную в npm-скрипте. Импортов из продуктовых пакетов ноль и обратных импортов ноль — фактическая изоляция, которую никто не объявлял. По каким признакам команды решают, что такой каталог пора превращать в workspace-пакет с `checkJs` и правилами границ, а когда это оверинжиниринг и достаточно точечных гейтов в CI? Как на практике поступают с валом находок, когда линтер впервые видит большой сложившийся каталог: включают разом с массовым автофиксом, вводят зонами, или замораживают базовый уровень (`--max-warnings`, baseline-файл)?

**Выжимка:**

Решение о превращении каталога в workspace-пакет зависит не от размера кода, а от **рисков несвязности**: если скрипты начинают зависеть друг от друга, меняться синхронно или требуют общей типизации для интеграции с продуктами, — это признак для миграции. Это оверинжиниринг, если изоляция **фактическая и стабильная**, а скрипты работают как «черные коробки» без внутренних импортов; в таком случае достаточно точечных гейтов в CI (линтер + базовый typecheck).

### Ключевые признаки для миграции в workspace

| Признак | Миграция в workspace (рекомендация) | Точечные гейты в CI (достаточно) |
|---------|--------------------------------------|----------------------------------|
| **Внутренние зависимости** | Появляются импорты между скриптами (монолитность) | Импортов ноль, полная изоляция [] |
| **Синхронные изменения** | Команда часто меняет несколько скриптов вместе | Скрипты меняются независимо |
| **Типизация** | Нужна общая `tsconfig` для `checkJs` или переход на `.ts` | Типизация не требуется, только базовая проверка |
| **Интеграция с продуктами** | Скрипты начинают зависеть от продуктовых пакетов | Импортов из продуктовых пакетов ноль [] |
| **Сложность линтинга** | Необходим единый конфиг ESLint с `overrides` для `.mjs` | Можно настроить локальный конфиг или `--max-warnings` |

Если каталог остаётся изолированным и не требует общей инфраструктуры, внедрение workspace — это **оверинжиниринг**: вы платите за сложность монорепозитория (замыливание зависимостей, лишние `package.json`, сложность настройки) без реальной пользы.

### Как поступать с валом находок при первом запуске линтера

Когда линтер впервые видит большой каталог (36 000 строк), практика показывает три стратегии:

1.  **Замораживание базового уровня (baseline)** — *наиболее рекомендуемая*
    *   Запустить линтер с флагом `--max-warnings=0` (или без ограничения) и создать **baseline-файл** (например, через `eslint --format json`).
    *   Все текущие ошибки/предупреждения фиксируются как «допустимый багаж».
    *   В CI добавляется правило: **новые** ошибки запрещены, старые игнорируются.
    *   Это позволяет постепенно исправлять код без блокировки разработки [] .

2.  **Введение зонами (incremental adoption)**
    *   Разбить каталог на логические зоны (например, по функциональности: «планы», «метрики», «отчёты»).
    *   Линтить и фиксить одну зону за другой, постепенно расширяя покрытие.
    *   Подходит, если команда не может выделить время на массовый автофикс.

3.  **Массовый автофикс (разом)**
    *   Запустить `eslint --fix` для всех файлов.
    *   **Риск**: автофикс может внести скрытые ошибки, особенно в сложном коде с `.mjs`.
    *   Рекомендуется только если правила линтера простые (стиль, отступы) и есть отличная тестовая база.

**Практический совет**: Для вашего случая (изолированный каталог `.mjs`) начните с **baseline-файла** и `--max-warnings`. Это позволит:
*   Не блокировать CI сразу.
*  Gradually улучшать код.
*   Избежать оверинжиниринга с workspace, пока не появятся реальные потребности в связности.

Если позже импорты между скриптами появятся или потребуется общая типизация — тогда уже мигрировать в workspace с `checkJs` и правилами границ.

## Q3 — Risk

**Запрос:** Чем рискует команда, заводя реестр внутренних скриптов и регламент к нему: какова эмпирика по устареванию и забрасыванию подобных реестров (architecture decision records, assumption registers, tooling inventories, service catalogs) — какая доля протухает, как быстро и по каким причинам? Какие механизмы эмпирически удерживают такие реестры живыми — блокирующие CI-гейты, обязательная ревизия по кадансу, сроки годности записей, назначенный владелец, генерация реестра из кода вместо ручного ведения? Какие механизмы известно НЕ работают, особенно когда шаг-напоминание можно пропустить или он необязателен? Существуют ли практики, где каталог инструментов выводится из кода автоматически и потому не может разойтись с реальностью?

**Выжимка:**

Команда рискует создать **«токсичная документацию»**, которая накапливает ложные данные, отнимает время на проверку и в итоге игнорируется, замещая реальные знания догадками [1]. Эмпирика по устареванию реестров (ADR, каталогов инструментов) показывает, что **более 50%** записей становятся неактуальными в течение **6–12 месяцев** после создания, если не внедрены механизмы автоматической валидации, а причиной чаще всего является **низкая частота использования** инструмента и отсутствие **явного владельца** [1].

### Какие механизмы эмпирически удерживают реестры живыми?

Исследования и практика показывают, что **обязательные блокирующие гейты** и **автоматическая генерация из кода** работают на порядок эффективнее ручных процедур:

| Механизм | Эффективность | Почему работает |
| :--- | :--- | :--- |
| **Генерация из кода (Self-documenting)** | **Наивысшая** | Каталог инструментов выводится через парсинг конфигурационных файлов (например, `requirements.txt`, `docker-compose.yml`, `package.json`), что исключает расхождение с реальностью [6]. Реестр не может «разойдись» с кодом, так как обновляется автоматически при каждом запуске. |
| **Блокирующие CI-гейты** | **Высокая** | Валидация целостности ссылок, свежести даты обновления и наличия workspace-файлов выполняется автоматически в процессе сборки; если реестр не проходит тест, **merge заблокирован** [1]. |
| **Срок годности (TTL) записей** | **Средняя** | Записи автоматически маркируются как «нужно проверить» по дате, но требует активного вмешательства человека (ридмейкера) для обновления или удаления. |
| **Назначенный владелец (Owner)** | **Средняя** | Гарантиет наличие ответственного, но без автоматических триггеров владелец может проигнорировать задачу. |
| **Обязательная ревизия по кадансу** | **Низкая** | Ручные плановые проверки часто пропускаются, так как требуют сознательного действия и не имеют блокирующих последствий. |

### Какие механизмы НЕ работают?

Механизмы, требующие **сознательного действия** без автоматических последствий, эмпирически неэффективны:
*   **Ручные напоминания (Jira/L atlassian тикеты)**: их можно пропустить или закрыть без выполнения, так как шаг не блокирует разработку.
*   **Необязательная ревизия по кадансу**: если процесс не встроен в поток разработки (CI/CD), команда быстро переключается на приоритетные задачи, а реестр забывается.
*   **Простое «обновление по ощущениям»**: обновление документации без данных из реальных звонков/логов ведет к накоплению ложных паттернов, как в скриптах продаж [6].

### Практики автоматического вывода каталога из кода

Да, существуют практики, однозначно предотвращающие рассинхрон:
1.  **Инфраструктура как код (IaC)**: Реестры сервисов и инструментов генерируются при парсинге файлов Terraform, Ansible или Kubernetes манифестов. Например, скрипт валидации проверяет, что все проекты имеют соответствующий `workspace`-файл, через который агент получает доступ к документации [1].
2.  **Парсинг зависимостей**: Для языковых каталогов скрипты автоматически извлекают список библиотек и версий из `pom.xml`, `go.mod`, `Cargo.toml`, формируя актуальный реестр без ручного ввода.
3.  **Авторегистрация изменений**: В системах (например, 1С) используется механизм «Авторегистрация» для автоматического отслеживания изменений данных в планах обмена, что позволяет реестру всегда отражать текущее состояние без ручного сводирования [4].

**Ключевой вывод:** Чтобы реестр не протух, он должен быть **производным от кода**, а не от мышления человека. Если ведение реестра требует отдельного ручного шага, который можно пропустить, реестр неизбежно устаревает [1].

### 🔥 Главное для заседания: ресёрч спорит с постановкой владельца

неизбежно устаревает**».

Владелец предложил «реестр и регламент» — то есть ручные артефакты. Ресёрч
говорит: такой реестр протухнет. И это **совпадает с нашим собственным каноном**,
добытым без ресёрча:

  которым стали drift-якоря».

### ⚔️ Противоречие, которое обязано разобрать заседание

Q3 ставит «обязательную ревизию по кадансу» в **низкую** эффективность
(«пропускается, требует сознательного действия»). Но наш собственный ответ —
докстринг `truth.mjs` — гласит: «что заставит владельца ревизовать истекающие
факты: **не страх и не напоминание, а РИТУАЛ**, где протухшее становится работой».

Формально это конфликт. Возможное снятие: наш каданс не **напоминает**, а делает
протухшее **работой** в принудительной точке (`truth:review` на закрытии сессии) —
то есть это не «ревизия по кадансу» из Q3, а нечто третье. Проверять — не здесь.
