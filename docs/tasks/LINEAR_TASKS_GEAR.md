# LINEAR_TASKS_GEAR — концептуальный паспорт движка задач

> Этот документ — **полная картина движка задач Membrana для агента, который с ним
> не работает через процедуры**. Здесь нет команд и чек-листов — только устройство,
> смыслы и границы. Процедурные входы — в конце (§10).
>
> Рождение движка: штормы и заседания 19.07.2026 → Cowork Sprint
> `cowork-execution-registry` (#660). Вердикты закрыты и не пересматриваются —
> только применяются.

## 1. Зачем движок существует

Два корня, вскрытые одним днём:

1. **У процессов не было субъекта.** Утренние и вечерние документы собирала
   детерминированная склейка; виртуальная команда голосовала, но не исполняла;
   ни у одного шага не было того, кто решает и отвечает.
2. **Реестр задач упёрся в свой носитель.** `docs/tasks/registry.json` — сотни
   карточек в одном JSON: движение задач, их содержание и их историю нёс один
   файл, который не масштабируется и не различает «удостоверение» и «движение».

Ответ движка: **развести задачу на слои** — кто отвечает (субъект), что считается
(учёт), где лежат факты (данные) — и переселить движение во внешний трекер
(Linear), оставив удостоверение в GitHub, а долговременность — в своём холоде.

## 2. Аксиома счёта: что такое «задача»

Центральное различение, на котором стоит всё:

| Сущность | Носитель | Роль |
|---|---|---|
| **Центральная задача** | **GitHub-issue** | удостоверение: конкретная работа, конкретный ответственный. **Единица ответственности и счёта.** Задача без Issue незаконна (норма владельца) |
| **Контейнер** | Linear-issue (parent) | единица движения: ползёт по колонкам, агрегирует. Ответственности не несёт, в счёт не входит |
| **Центральная под-задача** | Linear-под-задача | та же единица работы в носителе движения: связана **биекцией** со своей GitHub-issue |
| **Дополнительная под-задача** | Linear-под-задача | атрибут центральной; блокирующая держит закрытие, но отдельной строкой счёта не становится |

- **Биекция** «центральная под-задача ⟺ GitHub-issue» исключает двойной счёт по
  построению: каждая работа — ровно одна единица счёта.
- **«Закрыто»** = закрытие центральной задачи. Закрытие контейнера — производное;
  закрытие подчинённой под-задачи **событием не является**.
- Канон: [`UNITS_DICTIONARY.md`](./UNITS_DICTIONARY.md). Всё, что считает или
  гейтит, обязано импортировать значения слов оттуда, а не вводить свои.

## 3. Субъект: кто отвечает и кто делает

Три роли, которые нельзя склеивать:

- **Ответственный** — ровно одна персона, **принявшая выход** (`leadPersona`
  карточки, `assignee` в Linear). Инвариант: на единице работы один ответственный.
- **Исполнитель** — тот, кто произвёл работу. Может быть **обезличенным
  субагентом**: у него нет имени, нет подписи, и требовать её = сочинять имя.
  Его единственный публичный признак — **род**: `analyst` (параллельная фактура)
  или `scribe` (долгоживущий документ). В Linear слот исполнителя — поле
  `delegate` (проверено живым API 19.07; ресёрч ошибочно звал его `delegatedAgent`).
- **Ведущая персона — Ангелина** (`docs/virtual-team/PROMPT_ANGELINA.md`) — первый
  субъект процессов: ведёт диалог, решает и делегирует. Она **НЕ шестая
  советующая**: у промпта нет предметной зоны, в консилиумах не голосует.
  Делегирует по предикату трёх наблюдаемых входов
  (`¬fitsOneContext ∨ needsParallelFactua ∨ isLongLivedDocument`); возврат
  субагента валидируется структурно (`isValid`) ДО приёмки. Память — журнал из
  трёх полей (хвосты, напоминания владельца, итог дня) с провенансом «день + источник».

## 4. След: чем доказывается ответственность

След единицы работы живёт **ровно в двух носителях**:

1. **Карточка реестра** — `leadPersona` (назначение);
2. **Артефакт закрытия** — `{acceptedBy, headRev}` (приёмка: кто дал LGTM и на
   какой ревизии), опционально append-only `acceptanceHistory`.

**Коммит-сообщение и тело PR носителями НЕ являются** — это обезличенные каналы,
уязвимые к сборке чужой рукой; гейты их не читают. Приёмка происходит строго
ПОСЛЕ валидного возврата; подпись субагента в след не попадает никогда.

## 5. Зубы: как регламент отказывает

Отказ отличается от замечания **кодом, не тоном**. Два пространства кодов:

- **вердикт-код** — в структуре `{code, verdict}` возврата чистой функции;
- **exit процесса** — проекция: жёсткий вердикт → `exit = код`; мягкий → `exit 0`
  + обязательная печать замечания; чистый → `exit 0`.

Единая таблица спринта (новый код — только правкой контракта):

| Код | Имя | Смысл |
|---|---|---|
| 0 | OK | чисто |
| 10 | SNAPSHOT_STALE | снимок протух — аннотация, не отказ |
| 11 | LEGALITY_MIGRATING | гейт законности в миграционном режиме |
| 12 | ACCEPTANCE_UNCONFIRMED | приёмка не подтверждена (soft-фаза) |
| 20 | SNAPSHOT_NO_INPUT | входа нет — жёсткий отказ |
| 21 | LEGALITY_HARD | незаконные записи при жёстком режиме |
| 22 | LEAD_PERSONA_MISSING | ответственный не назначен (отказ-I, hard сразу) |
| 23 | ACCEPTANCE_FALSE | приёмка ложна: LGTM не на той ревизии (отказ-II hard) |

Живые точки: отказ-I стоит в `pre-push` (по всем active-карточкам) и в
`task:archive` (по архивируемой, включая `--dry-run` — показ не врёт); отказ-II —
в closure review (пока soft; hard — после миграции карточек, по предикату, не по дате).

## 6. Данные: откуда факты и где лежат

Три закона слоя данных:

1. **Вход гейтов — снимок, не сеть и не кэш.** Живой запрос дисквалифицирован
   (`gate(fetch()) ≠ gate(fetch())`). Снимок — контракт: адресуемый,
   версионированный, с провенансом `(capturedAt, sourceRevision)` и honest-шапкой
   `producedBy`/`egressRegion`/`mode`; слово «кэш» в подсистеме запрещено.
   Формат `linear-snapshot@1`; производитель — **media-NL** режимом
   `batch-full-pull` (вердикты M1/M3 `linear-egress-gear-wiring`); office —
   потребитель артефакта + trigger через `X-Membrana-Token`, в GraphQL Linear
   **не ходит** (ни snapshot-путь, ни issue-view — К1; refuse с
   `LINEAR_OFFICE_EGRESS_DISABLED`). Вебхук — только триггер «пора снимать»,
   телом источника не является. Предикат `pullOk(S)` — чистая функция от файла;
   freshness — один дешёвый запрос ВНЕ тела гейта. Владелец времени —
   производитель снимка (media-NL, UTC). Канон полей:
   [`LINEAR_SNAPSHOT_CONTRACT.md`](./LINEAR_SNAPSHOT_CONTRACT.md).
   Режим движения после первого честного `pullOk`: атомарный флаг
   `{movementMode, snapshotRef, switchedAt}` в
   [`docs/tasks/movement-mode.json`](./movement-mode.json) (К5/M4) —
   явный lift (`yarn movement:lift`), silent-flip запрещён; см.
   [`MOVEMENT_MODE.md`](./MOVEMENT_MODE.md).
2. **Расслоение носителей:** содержание → git; движение → снимок Linear;
   долговременность → **холод** `docs/tasks/archive/archive.jsonl` (append-only,
   один писатель — office-батч; принимает записи только с honest-шапкой; хранит
   содержание + доменные поля `dependsOn`/`strategicWave`/`leadPersona`;
   **движение не дублирует**). Формат: [`ARCHIVE_FORMAT.md`](./archive/ARCHIVE_FORMAT.md).
   Назначение холода: независимость от поставщика · офлайн-детерминизм гейтов ·
   жизнь за горизонтом подписки. Postgres/MongoDB отклонены как избыточные.
3. **Общий закон провенанса:** вход без honest-шапки отвергается, выход несёт
   honest-шапку. Это верно для снимка, холода и любого отчёта.

## 7. Измерение: что считается и что — принципиально нет

Пять величин, все из носителей следа (§4) и снимков (§6), ни одна не смотрит в git:

| Величина | Смысл | Пара против Гудхарта |
|---|---|---|
| `leadTime` | время жизни единицы | `reworkRate` |
| `ownerlessRate` | доля без ответственного | ограничитель, норма 0% |
| `wip(p)` | очередь намерений персоны | `throughput` |
| `throughput(p,w)` | закрытых за окно | `reworkRate` |
| `escalationRate` | смены ответственного/приёмки | ограничитель, вниз |

- Обязательная подпись любого отчёта: **«wip = очередь намерений, не
  производительность»**. Честного источника производительности нет — заявлено явно.
- Шестая величина «доля переделок» — **риторическая**: живёт только в аудите
  конца дня, чистой функцией от тех же двух снимков, хранимым событием не
  подкрепляется, не оптимизируется.
- **Намеренно неизмеримое** (в отчёте явным текстом): производительность аватара ·
  вклад в строках/коммитах · «качество мышления» персоны.
- Вычисление — вечерний батч из **двух материализованных снимков** + артефактов
  закрытия, чистой функцией, бит-в-бит детерминированно.

## 8. Миграция и законность (переходный период)

- `wellArchived(r) = inCold(r) ∧ hasPassport(r) ∧ archivedAt(r) ≠ null` — три
  предиката на снимке, без сети.
- Исторический долг классифицирован ДО переноса: `debtClass ∈ {record, work,
  legality, owner-knowledge}`. **Счётчик незакрытых работ читает только `work`** —
  архивные записи без паспорта его не поднимают.
- Три режима гейта законности: мягкий (0) → миграционный (11) → жёсткий (21);
  переключение на жёсткий — по предикату «незаконных ноль», **не по дате**.
- `parked` — состояние долга владельческого знания: не мигрирует, не блокирует,
  из счётчиков исключено.

## 9. Внешняя реальность (проверено живьём 19.07)

- **Linear заблокирован для России по IP** (`403 RESTRICTED_COUNTRY_BLOCKED`) —
  и с рабочих машин, и с мск-VDS офиса. Ключ `LINEAR_API_KEY` валиден на
  **media-NL**; office ключ для GraphQL не использует. Боевой pull:
  `POST /v1/linear-snapshots/capture` на media (live `pullOk` с
  `producedBy=media-NL` — 2026-07-20, #692). Режим движения:
  `movementMode: live-snapshot` после явного К5 lift (не silent).
- Поле исполнителя в API — `delegate`; под-задачи `parent`/`children` — полноценные
  объекты; `relations` (blockedBy/blocking) в GraphQL есть, вебхуками — допущение.
- **Старые карточки Linear-workspace (до 19.07) — расходные** (слово владельца):
  миграции ИЗ Linear нет, workspace начинается заново.
- Открытые гейты владельца: egress + расширение issue-view; дата рубежа
  законности; разбор `parked`; ратификация уточнения `insight-task-archive-storage`.

  застаблен, ждёт гейтов.
