# Памятка союзника Membrana

> Для тех, кто знакомится с проектом с нуля и читает отчёты бота в группе.
> Термины и формулировки — по канону [`GLOSSARY.md`](./canon/GLOSSARY.md) / [`FACTS_SHEET.md`](./canon/FACTS_SHEET.md).
> Версия: 2026-07-14.

---

## 1. Что такое Membrana

> **Membrana — распределённая акустическая разведка нижнего неба: обнаружение, классификация
> и трекинг малых БПЛА по звуку, в реальном времени, без радара.**

По-простому: это сеть недорогих «ушей». Каждое «ухо» — **сенсорный узел**: микрофон плюс
маленький компьютер уровня Raspberry Pi. Узлы слушают **нижнее небо** (от земли до 1500 м)
и в реальном времени распознают звук малого дрона — у него характерный акустический
«портрет» (низкий гул мотора и его обертона).

Принцип — **«дешёвый узел, умная сеть»**: сам узел прост и дёшев, а вся сложность
(объединение данных, определение координат) живёт на уровне сети. Это **наблюдательная
система**: она фиксирует факт полёта и предупреждает — ничем не управляет и ничего не глушит.

## 2. Из чего состоит продукт

- **Клиентское приложение с конструктором сценариев** — визуальные «кубики»
  (микрофон → распознавание → тревога), из которых оператор собирает поведение системы
  без программирования. Готовый собранный набор кубиков называется **сценарий**.
- **Личный кабинет** — привязка устройств, тарифы, наблюдение за узлами.
- **Распознавание — семейство детекторов, а не одна нейросеть.** Эшелон A — классический
  анализ звука (объяснимый, дешёвый, работает на слабом железе); эшелон B — готовые
  нейросетевые модели (например, YAMNet). Их мнения объединяются (**fusion**) в общий
  балл уверенности, по которому сценарий решает, поднимать ли тревогу.
- **Два служебных сервера:** «media» хранит записи и данные, «office» — рабочие
  инструменты команды (ночные проверки, отчёты — в том числе этот бот).

## 3. Кто и как это делает

Проект ведёт владелец вместе с **виртуальной командой** — пятью ролями на базе языковых
моделей (Teamlead, Структурщик, Математик, Музыкант, Верстальщик). Ритм дня:

- **Утро** — план дня: одна главная задача + список поддерживающих.
- **Вечер** — итоги: что реально сделано, каждый участник оценивает полезность дня
  по 10-балльной шкале, формируются предложения на завтра.
- Бот пересылает выжимки утреннего и вечернего итогов в эту группу.
- Серьёзные развилки решает **консилиум** — совет всех пяти ролей **до** написания кода;
  принятое архитектурное решение фиксируется документом (**ADR**), чтобы его никто
  незаметно не «переиграл».

## 4. Мини-словарь для чтения отчётов

| В отчёте | По-простому |
|----------|-------------|
| PR #N, merge | «Порция изменений» с номером и её принятие в основную версию проекта |
| Issue #N | Карточка задачи с номером в трекере |
| Консилиум | Совет пяти ролей команды до кода по спорному вопросу |
| ADR | Короткий документ «какое решение приняли и почему» |
| Ритуал (утренний/вечерний) | Ежедневная процедура планирования / подведения итогов |
| Сценарий (UserCase, UC) | Готовая сборка «кубиков» под конкретную задачу пользователя |
| Детектор | Отдельный распознаватель звука дрона (классический или нейросетевой) |
| fusion / combinedScore | Объединение мнений детекторов в один балл уверенности |
| drift-anchor, «якорь» | Ежедневная автоматическая самопроверка качества (см. §5) |
| S3 | Полоса работ «упаковать готовые сценарии в каталог продукта» |
| VDR | Проверка качества на реальном полевом оборудовании и записях |
| office / media | Служебные серверы проекта (инструменты команды / данные) |

## 5. Что уже сделано — состояние на середину июля

Расшифровка недавних итогов (это и был последний вечерний отчёт):

1. **«drift-anchor замкнут»** — построена и включена ежедневная **самопроверка качества**:
   система каждый день сравнивает своё распознавание с эталоном и поднимает флаг, если
   что-то незаметно испортилось. Это особенно важно, потому что большой объём кода пишут
   агенты на базе языковых моделей — «якорь» страхует от тихой деградации качества.
   *Что даёт: изменения в коде больше не могут незаметно ухудшить распознавание.*
2. **«live-neural fusion в prod»** — в живом распознавании теперь совместно работают
   классический анализ звука и нейросетевая модель, их мнения объединяются в общий балл.
   *Что даёт: точнее распознавание, меньше ложных тревог.*
3. **Телеграм-бот отчётов** — канал, в котором вы это читаете: автоматические выжимки
   утреннего плана и вечерних итогов.
   *Что даёт: пассивная лента прогресса без GitHub и технических знаний.*
4. **Ночные помощники** — сервер по ночам сам наводит порядок в задачах и ищет
   расхождения в проекте, к утру готовит предложения.
   *Что даёт: рутина не съедает дневное время команды.*

Фраза **«день продуктивный, но дрейфующий»** в отчёте — это честная самооценка: сделано
много, но не всё из утреннего плана, часть сил ушла на внеплановые (хотя и одобренные
владельцем) задачи. «Дрейф» здесь — расхождение плана и факта; команда его отслеживает
и называет вслух.

## 6. Что планируется — расшифровка пунктов «что дальше»

| Из отчёта | По-простому | Что это даст |
|-----------|-------------|--------------|
| «Консилиум-гейт drift-panel-placement ДО кода» | Прежде чем строить экран наблюдения за качеством («панель дрейф-якоря»), команда советом решает, где ему жить: внутри личного кабинета или отдельной простой страницей для оператора | Наглядный экран «всё ли в порядке с качеством» — без чтения технических журналов |
| «Закрыть перф-замер latency yamnet» | Измерить, за сколько миллисекунд нейросетевая модель обрабатывает звук на обычном компьютере | Уверенность, что тревога не запаздывает; понимание требований к железу |
| «Таблица trends DRONE_TIGHT vs yamnet» | Честное сравнение двух распознавателей на одном и том же наборе проверочных записей: кто точнее и где ошибается | Обоснованный выбор состава распознавания для бесплатной версии; шаг к шлюзу качества |
| «Вернуть в фокус S3» | Упаковать четыре готовых сценария (3 базовых + комбинированный) в каталог, чтобы пользователь включал их в один клик | Осязаемое наполнение бесплатной версии — то, что реально увидит пользователь |
| «Owner-хвосты llm-providers» | Административные шаги владельца: активировать двух поставщиков языковых моделей (баланс/оплата) | Полные ночные отчёты и поиск по истории проекта без ограничений |
| «Держать границу ADR 0004» | Соблюдать принятое правило: служебный сервер только передаёт данные о качестве, выводы делает получатель | Систему проще проверять, её труднее незаметно усложнить или сломать |

## 7. Куда всё это ведёт

Последовательность ближайших целей:

1. **Бесплатная версия** с готовыми сценариями «из коробки» — ближайшая продуктовая цель.
2. **Полевое подтверждение качества (VDR)** — проверка распознавания на реальном
   оборудовании и оператор-размеченных записях (ориентир — вторая половина июля).
3. **Шлюз качества (stage-gate):** переход к сети узлов заблокирован, пока один узел
   не подтвердит пороги точности (precision ≥ 85 %, recall ≥ 90 %) на едином датасете.
   Это осознанный барьер против самообмана.
4. После шлюза — **сеть узлов**: локализация по разнице времён прихода звука (TDOA),
   трекинг, «умная сеть» в полном смысле.

## 8. Об этих отчётах

Канал только что запущен: первые дайджесты собираются автоматически из внутренних
документов и местами звучат технически. Это известно и будет дорабатываться — следующий
шаг: пересказ «по-простому» силами языковой модели поверх тех же фактов. Вопросы и
пожелания по формату — владельцу.
