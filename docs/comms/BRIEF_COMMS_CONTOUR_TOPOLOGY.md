# Бриф: контур внешних коммуникаций — топология и изоляция

**Тип:** бриф для консилиума
**Формат работы:** консилиум пяти ролей + жюри
**Статус решения:** открытый вопрос (топология не зафиксирована)
**Дата:** [заполняется]

---

## 1. Сигнал

В команду вводится новый узел — партнёр по внешним коммуникациям. Он не пишет код. Он работает над презентациями, лендингами, брендбуком, инфографикой. Ему нужен полный контекст проекта и мета-каркас, но без доступа к продуктовому коду на запись.

Задача консилиума — спроектировать **выделенный контур communications**: пространство, которое умеет на основе общего контекста проекта выполнять задания (описывать компоненты, готовить outline, копирайт, визуальные спецификации) и при этом не представляет угрозы для остального кода.

Один вопрос остаётся **открытым и выносится на консилиум**: где этот контур живёт — внутри монорепозитория как изолированный воркспейс, или отдельным репозиторием, в который импортируется часть канонических документов из центрального. Ниже — рамка для решения, не само решение.

## 2. Инвариант (не обсуждается)

Что бы ни решил консилиум по топологии, один инвариант держит всю конструкцию:

> **Контур communications — это сток, а не исток.**
> Читает много, пишет мало. Может читать весь контекст проекта, чтобы описывать компоненты точно. Пишет только в свой каталог. Ни один продуктовый пакет от него не зависит.

Пока стрелка зависимости направлена в одну сторону, угрозы остальному коду нет по построению, а не по договорённости. Топология — это способ реализовать инвариант, а не отменить его.

Второй инвариант, из канона проекта: **single source of truth.** Факт про проект рождается в одном месте. Все документы и рендеры на него ссылаются, не дублируют. Контур communications — потребитель истины, а не второй её источник.

## 3. Что контур покупает

Прежде чем спорить о топологии, зафиксируем, ради чего вообще нужен контур внутри инфраструктуры проекта, а не просто Claude Project + Google Drive.

Единственная реальная выгода — **живое чтение актуального кода и документов.** Knowledge Base в Project — снимок; он дрейфует относительно реализации. Контур, читающий актуальные исходники, описывает компоненты по факту, а не по устаревшей копии. Это снимает человеческий шаг верификации «соответствует ли описание текущей реализации».

Проверочный вопрос для консилиума: **снимает ли контур человеческий шаг на последовательном критическом пути, или только добавляет поверхность обслуживания?** Живое чтение — снимает. Git-ветки и PR для нетехнического партнёра — добавляют. Отсюда следствие для обеих топологий: в контуре работает агент; партнёр касается контура тонким срезом (монитор каталога + выход в Drive для ручной правки), а не через прямой git.

## 4. Открытый вопрос: топология

Консилиум рассматривает две топологии и приходит к единому решению с обоснованием. Ниже — затравка аргументов для обеих сторон. Список не исчерпывающий; роли добавляют своё.

### Вариант A. Изолированный воркспейс внутри монорепо

Пример: `apps/comms-studio` или `docs/comms` как leaf-workspace без входящих рёбер.

**За:**
- Живое чтение кода без синхронизации — контур видит актуальное состояние по определению.
- Механизмы изоляции уже есть в стеке: `yarn check:boundaries`, CODEOWNERS, branch protection, отдельная CI-полоса. Нового тулинга не требуется.
- Единый ритм: правки идут через тот же процесс, канон не расщепляется.

**Против:**
- Нетехнический партнёр рядом с продуктовым git — источник шума и риска ошибочной записи (митигируется CODEOWNERS, но остаётся когнитивная нагрузка).
- Продуктовый CI и comms-CI надо явно развести, иначе сломанный слайд блокирует сборку кода, а правка копирайта гоняет продуктовый пайплайн.
- Разрастание монорепо: внешний маркетинговый контент внутри инженерного канона смешивает два разных риск-профиля и ритма.

### Вариант B. Отдельный репозиторий с импортом канона

Отдельный репо, в который переносится минимальный набор канонических документов из центрального (см. второй артефакт — `CANON_MINIMUM_FOR_COMMS_CONTOUR.md`).

**За:**
- Чистая физическая изоляция: партнёр не имеет доступа к продуктовому коду вообще. Инвариант «сток, не исток» выполняется тривиально.
- Отдельный риск-профиль, ритм, права доступа — маркетинговый контур не притворяется частью инженерного.
- Проще отдать партнёру полный контроль над его репо без риска для основного.

**Против:**
- **Теряется главная выгода — живое чтение.** Импортированный канон становится снимком, который дрейфует. Возникает шаг синхронизации, то есть ровно тот человеческий шаг, который контур должен был снять.
- Нужен механизм синхронизации канона (submodule, subtree, CI-зеркало, ручной перенос) — каждый вариант имеет свою цену обслуживания.
- Риск расщепления истины: две копии канона в двух репо. Прямое нарушение single source of truth, если синхронизация не автоматизирована и не односторонняя.

### Гибрид (роли могут предложить)

Возможен промежуток: канон остаётся в монорепо и экспортируется вниз односторонним автоматом (read-only зеркало), а рабочий контур партнёра — отдельно. Консилиум оценивает, покупает ли гибрид что-то сверх сложности, или это худшее из двух миров.

## 5. Критерий решения

Решение принимается не по вкусу, а по одному вопросу:

> **Какая топология сохраняет живое чтение канона при минимальной поверхности синхронизации и при выполненном инварианте «сток, не исток»?**

Если синхронизация в варианте B оказывается односторонней, автоматической и дешёвой — расщепления истины нет, и физическая изоляция может перевесить. Если синхронизация тянет ручной шаг или двусторонний дрейф — вариант B покупает изоляцию ценой той самой выгоды, ради которой контур затевался, и тогда A предпочтительнее.

## 6. Дисциплина стадии (ограничение сверху)

Напоминание, которое консилиум держит в рамке: реальный лимит проекта — последовательный человеческий критический путь, не число инструментов и не машинная пропускная способность. Контур оправдан ровно настолько, насколько снимает человеческий шаг. Инфраструктура, которая только добавляет поверхность обслуживания, откладывается — как и всё, что конкурирует с блокирующим шлюзом одного узла.

Отдельно: **решение об авторстве контура не равно решению о хостинге продакшен-лендинга.** Контур производит описания, outline, копирайт, макеты. Где и как деплоится боевой сайт `membrana.space` — вопрос отдельный, и stage-gate его откладывает. Не смешивать эти два решения в одном обсуждении.

## 7. Что требуется от консилиума

1. Единое решение по топологии (A / B / гибрид) с явным обоснованием через критерий §5.
2. Список механизмов изоляции для выбранной топологии (границы, CI, права доступа, скоуп секретов).
3. Явные non-goals: что контур **не** делает (не деплоит продакшен, не второй источник истины, не сажает партнёра в продуктовый git).
4. Если выбран вариант с импортом канона — механизм синхронизации и доказательство его односторонности.
5. Открытый риск: где именно возможен дрейф истины и что его блокирует.

Оформление результата — INSIGHT-артефакт в каноническом формате. Пока решение устное — оно не канон.

## Приложение: параметры контура (общие для любой топологии)

- **Доступ на чтение:** полный контекст проекта (код, документы, канон).
- **Доступ на запись:** только каталог контура.
- **Входящие зависимости от контура:** запрещены (проверяется автоматом границ).
- **CI:** отдельная полоса, не пересекается с продуктовым `lint typecheck test build`.
- **Секреты:** изолированный `.env`; вычистка секретов — блокирующее условие перед любым архивом или переиспользованием (канон проекта, не исключение).
- **Партнёр как узел:** тонкий срез — монитор каталога + Drive для ручной правки. Не прямой git.
- **Канон коммуникаций** (тон, dual-use, запрещённые слова, лист фактов, глоссарий) применяется к выходу контура ровно как к слайдам. Инструмент рендерит канон, не отменяет его.
