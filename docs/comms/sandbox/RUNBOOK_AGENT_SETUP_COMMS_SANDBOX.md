**Тип:** task-prompt для Claude-агента (исполняемый runbook)
**Исполнитель:** Claude Code
**Парный документ:** `CHECKLIST_PARTNER_MANUAL_SETUP.md` (человеческие шаги)
**Проверено против:** документация Claude Code / Agent Skills / MCP на дату составления
**Дата:** 2026-07-06 (адаптация к реализации CC1–CC9, спринт CD1–CD6)

## 0. Контракт исполнения

Ты — агент настройки. Твоя задача — привести песочницу communications в рабочее состояние: репозиторий, среда, зависимости, MCP-серверы, скиллы, хуки. Выполняй шаги по порядку. На каждом шаге, помеченном **STOP → партнёр**, ты физически не можешь продолжить (нужен ввод учётных данных или OAuth-подтверждение человека) — остановись, выведи, что именно требуется, и жди. Не пытайся обойти: ввод паролей, токенов и прохождение OAuth — не твоя операция.

**Инвариант, который ты не нарушаешь:** песочница — сток, не исток. Ты подключаешь генеративные инструменты и инструменты чтения. Ты **не** подключаешь и **не** активируешь публикующие/деплой-инструменты. Из песочницы выходят файлы, не публикации.

Переменные окружения этого runbook (значения зафиксированы — топология решена, **Вариант A**, см. INSIGHT `insight-comms-contour-topology`):
- `CONTOUR_DIR=apps/comms-studio` — leaf-воркспейс контура внутри монорепо. Не выбирай другой каталог: контур уже реализован здесь (CC1–CC9).
- `REPO_URL=github.com/officefish/Membrana` — монорепо (отдельного comms-репо нет; изоляция контура держится инвариантом границ `check:boundaries`, а не отдельным репозиторием).

## 1. Проверка среды

Прежде чем ставить — измерь, что уже есть. Не переустанавливай то, что в норме.

```bash
node --version      # цель: LTS ≥ 20 (см. §2 — какая именно версия и почему)
git --version       # цель: ≥ 2.0
claude --version    # есть ли уже Claude Code
```

Зафиксируй результат каждой команды. Дальнейшие шаги — только для того, чего не хватает.

## 2. Node.js

Ставим **Node.js LTS**, даже если Claude Code поставится нативным установщиком без Node. Причина: MCP-серверы запускаются через `npx`, а хуки — через Node; без него они не заработают. Это не зависимость самого `claude`, а зависимость контура.

Какую версию (по фактам репо, не на глаз):
- **Минимум — Node 20** (`engines.node` монорепо = `>=20.0.0`; `@types/node` = `^20`; CI-полоса контура `.github/workflows/comms-studio.yml` валидирует на `node-version: '20'`).
- **Локально — по `.nvmrc` репо (сейчас `22`)**, чтобы совпасть с dev-машиной. `nvm use` в корне репо подхватит `.nvmrc` сам.

Оба LTS работают: репо собирается на 20 (CI-контракт) и разрабатывается на 22 (`.nvmrc`). Не завышай минимум до 22 в требованиях — CI гейтит на 20.

Через nvm (изолирует Node в домашнем каталоге, без прав root — не используй `sudo`):

```bash
# nvm, если отсутствует
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.0/install.sh | bash
# перезапусти шелл, затем — из корня репо (подхватит .nvmrc):
nvm install
nvm use
node --version      # подтверди: LTS ≥ 20 (репо .nvmrc → 22)
```

## 3. Claude Code

Если `claude --version` уже дал результат в §1 — пропусти. Иначе — нативный установщик (рекомендованный, без зависимостей, с автообновлением):

```bash
# нативный установщик (macOS/Linux)
curl -fsSL https://claude.ai/install.sh | bash
claude --version    # подтверди
```

Альтернатива (npm, официально устаревший путь; требует Node — как раз есть из §2):

```bash
npm install -g @anthropic-ai/claude-code
```

Не запускай `npm install -g` под `sudo`. При ошибке прав — настрой пользовательский каталог npm (`~/.npm-global`), не повышай привилегии.

**STOP → партнёр:** аутентификация Claude Code (`claude` → вход через аккаунт Anthropic). Ввод учётных данных — не твоя операция. Выведи партнёру: «нужен вход в аккаунт Anthropic», жди.

## 4. Репозиторий и зависимости монорепо

```bash
git clone $REPO_URL
cd <корень репозитория>
```

Монорепо на Yarn 4 через Corepack (по канону проекта):

```bash
corepack enable
corepack prepare yarn@4.5.0 --activate
yarn install
```

Убедись, что `yarn -v` показывает 4.x, а не 1.x (иначе — `corepack enable` вручную). При падении `--immutable` — рассинхрон `yarn.lock`; не чини на ходу, сообщи.

Проверь, что сборка проходит, прежде чем строить контур поверх:

```bash
yarn build
yarn typecheck
```

## 5. MCP-серверы

Подключаем ровно четыре. Каждый — по одной функции. Публикующих нет.

### 5.1. Filesystem — чтение **живого** канона и журнала

Канон **не копируется** в контур — он читается живьём из монорепо. Перечень канонических путей — источник истины `apps/comms-studio/src/canon-sources.ts` (Слой 1–3). Область filesystem-MCP — подкаталог `docs/` монорепо, где эти пути лежат, не `$CONTOUR_DIR/canon` (такого каталога нет и быть не должно — это была бы дрейфующая копия):

```bash
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem \
  "$(git rev-parse --show-toplevel)/docs"
```

Живой канон (по `canon-sources.ts`), read-only:
- **Слой 3 (производный, специфичный для comms):** `docs/comms/canon/{FACTS_SHEET,GLOSSARY,BRAND_TOKENS}.md` — наполнен (CC5–CC7).
- **Слой 1 (носитель истины):** `docs/foundation/PROMPT_WHITE_PAPER.md`, `docs/ARCHITECTURE.md`, `docs/INTEGRATIONS_STRATEGY.md`, `docs/prompts/SINGLE_NODE_DETECTION_FIRST_PROMPT.md`.
- **Слой 2 (носитель формы):** `docs/foundation/CLAUDE_PROJECT_SYSTEM_PROMPT.md`, `docs/foundation/MEMBRANA_ONE_PAGER.md`.

Read (+ append для журнала дизайн-решений, когда он появится — отдельным каталогом, не поверх канона). Это прямое следствие инварианта single source of truth: канон живёт в git один раз, контур на него **ссылается**, а не хранит вторую версию.

### 5.2. Playwright — сверка рендера (обратная связь)

```bash
claude mcp add playwright -- npx -y @playwright/mcp@latest
```

**`[сверить перед запуском]`** — фактическое имя пакета Playwright MCP: экосистема менялась, `@playwright/mcp` дан как ориентир, не как проверенная константа. Проверь актуальный пакет и подставь; не полагайся на это имя вслепую.

### 5.3. Figma remote — дизайн-контекст

```bash
claude mcp add figma --transport http https://mcp.figma.com/mcp
```

**STOP → партнёр:** откроется OAuth-вход Figma. Подтверждает партнёр (чеклист, Блок B). Ты не проходишь авторизацию.

### 5.4. Replit — генерация лендингов (allow-list, без деплоя)

```bash
claude mcp add replit --transport http <официальный remote-URL Replit MCP>   # [сверить перед запуском]
```

**`[сверить перед запуском]`** — `<официальный remote-URL Replit MCP>` не подставлен намеренно: не выдумывай URL. Возьми актуальный remote-endpoint из официальной документации Replit MCP на момент сборки и подставь.

**STOP → партнёр:** OAuth-вход Replit. Партнёр подтверждает доступ **только на создание/правку файлов, не на публикацию** (чеклист, Блок B).

**После подключения — критично:** ограничь набор инструментов Replit до генеративных (create, write, read, fork); deploy/publish — в deny. Пропиши это в конфигурации разрешений Claude Code для этого сервера.

**Проверка, а не вера** (эмпирический шаг, по канону — как `curl` перед архитектурой): убедись, что deny реально **запрещает вызов**, а не просто прячет инструмент из подсказок. Попробуй вызвать публикующий инструмент напрямую — он должен быть отклонён. Если клиент только скрывает, но вызов проходит — граница дырявая: не подключай Replit в этот профиль вовсе, вынеси в отдельный профиль без публикующего сервера и сообщи.

### Проверка всех серверов

```bash
claude mcp list     # все четыре присутствуют и в статусе connected
```

## 6. Скиллы

Все проектные, в `.claude/skills/` контура (шарятся через git — канон живёт с контуром, не с машиной).

```bash
mkdir -p $CONTOUR_DIR/.claude/skills
```

### 6.1. Готовый скилл — подключить

- **`frontend-design`** — публичный. Установи из доступного источника скиллов, не пиши с нуля.

### 6.2. Собственные скиллы — создать каталоги и SKILL.md

Для каждого создай `<skill>/SKILL.md` с YAML-фронтматтером (`name`, `description`) и телом. **Описание пиши настойчиво, с триггерами** — модель склонна недосрабатывать скиллы; вялое описание не запустится.

Эшелон 1 (кодификация канона) — `references/` **указывают на живой канон** `docs/comms/canon/`, не хранят копий:
- `brand-kit` — визуальные токены; источник — `docs/comms/canon/BRAND_TOKENS.md` (палитра/шрифты/do-don't). В `assets/` кладём только бинарные ассеты (шрифты, логотип), которых нет в markdown-каноне.
- `lexicon-guard` — запрещённые слова; источник — `docs/comms/canon/GLOSSARY.md` **§3** (не отдельный сочинённый `blocked-terms.md`).
- `glossary-lock` — термины; источник — `docs/comms/canon/GLOSSARY.md` (§1–§2).
- `facts-guard` — метрики только из `docs/comms/canon/FACTS_SHEET.md`.

Эшелон 2 (workflow):
- `gamma-outline`, `landing-structure`, `diagram-vocabulary`.

Эшелон 3 (обратная связь):
- `render-check` — task-скилл, вызывается явно, использует Playwright из §5.2.

Содержимое канона — **не выдумывай и не копируй**. Три канон-файла уже наполнены в репо (CC5–CC7): `docs/comms/canon/{FACTS_SHEET,GLOSSARY,BRAND_TOKENS}.md`. Скилл в `references/` **ссылается** на эти живые пути (относительно корня монорепо), а не хранит вторую версию — копия дрейфует, ссылка — нет. Если понадобился канон-факт, которого в этих файлах нет — не заполняй по догадке: пометь `[TBD]` и сообщи, чего не хватает в каноне.

### Проверка

```bash
ls $CONTOUR_DIR/.claude/skills/*/SKILL.md    # все девять
```

## 7. Хук на запрещённые слова — переиспользует наш guard

Скилл `lexicon-guard` направляет генерацию; хук — детерминированный барьер на выходе, вне решения модели. Это разные слои, нужны оба.

**Не пиши список запрещённого заново.** Детерминированный барьер уже реализован в контуре: `apps/comms-studio/src/tone-guard.ts` — функция `checkTone(text)` возвращает нарушения по трём категориям (хайп, военная риторика, эмодзи); `apps/comms-studio/src/out-writer.ts` (`writeArtifact`) отклоняет запись при любом нарушении. Источник термов — `docs/comms/canon/GLOSSARY.md §3`, вшитый в `tone-guard`.

- Настрой хук Claude Code, который на выходе прогоняет текст **через тот же `checkTone`** (импортируй/вызови из контура, не дублируй список). Хук **блокирует** выход при непустом списке нарушений.
- Так список термов и логика границы слов (стем с левой границей; «удар» намеренно исключён как ложно-срабатывающий — см. комментарий в `tone-guard.ts`) остаются в одном месте. Дубликат неизбежно разойдётся с каноном.
- **Проверка (сохранить эмпирику):** сгенерируй текст с заведомо запрещённым словом. Хук должен **остановить** выход, а не пометить. Если не блокирует — хук неплотный, чини до того, как контур пойдёт в работу.

## 8. Финальная верификация

Пройди чек-лист состояния. Каждый пункт — фактом, не предположением.

- [ ] `node -v` → LTS ≥ 20 (репо `.nvmrc` → 22); `claude --version` → есть; `yarn -v` → 4.x.
- [ ] `yarn build` и `yarn typecheck` проходят.
- [ ] `claude mcp list` → filesystem, playwright, figma, replit — все connected.
- [ ] Replit deny на публикацию **запрещает вызов** (проверено попыткой, §5.4).
- [ ] `ls .claude/skills/*/SKILL.md` → девять скиллов.
- [ ] Скилл срабатывает без явного вызова (дай задачу, проверь автоподхват; если недосрабатывает — усиль `description`).
- [ ] Хук на запрещённые слова блокирует, а не помечает (§7).
- [ ] Каналы канона (`facts-sheet`, `glossary`, `blocked-terms`, бренд-токены) наполнены партнёром либо явно помечены `[ожидает передачи]`.

## 9. Отчёт

По завершении выведи короткий отчёт: что установлено (с версиями), какие MCP подключены и в каком статусе, какие скиллы созданы и какие ждут наполнения канона, результат каждой проверки §8, и список всех **STOP → партнёр**, которые остались незакрытыми. Не рапортуй «готово», если хоть одна проверка §8 не прошла или хоть один канон-файл остался заглушкой — сообщи фактическое состояние, а не желаемое.

## Замечание для исполнителя

Ты донастраиваешь машинную часть — среду и инструменты. Это поднимает качество и воспроизводимость проходов, но не ускоряет человеческий критический путь проекта; помни об этом, если возникнет соблазн добавить «ещё один полезный сервер» сверх четырёх. Каждый лишний инструмент — поверхность обслуживания и поверхность prompt injection между серверами. Ставь ровно то, что в runbook. Расширение — отдельным решением, не по ходу установки.
