## 1. Формализация

Качество сгенерированного лендинга или дека — функция трёх величин, и только трёх. Каждая интеграция оправдана ровно настолько, насколько поднимает одну из них.

- **Точность контекста** на входе — агент знает токены, компоненты, факты.
- **Обратная связь** — агент видит, что произвёл, и сверяет с эталоном.
- **Кодифицированность ограничений** — правила применяются детерминированно, а не «агент вспомнил».

Всё, что не бьёт ни в одну из трёх, — поверхность обслуживания, не пропускная способность. Не входит в песочницу.

## 2. Несущий инвариант

> **Из песочницы выходят только файлы.**
> Ни один инструмент в ней не производит необратимого действия во внешнем мире. Генерация — да. Публикация — нет.

Деплой на продакшен — отдельный шаг, человеческий: его выполняет основатель и виртуальная команда, забирая готовые файлы из песочницы. Пока инвариант держится, песочница может читать канон и генерировать смело — цена ошибки равна «не забрали файл», а не «выложили брак на `membrana.space`».

Это то же разделение «сток / исток», что в брифе по топологии контура, приложенное здесь к набору инструментов MCP-клиента, а не к репозиторию.

## 3. Состав песочницы

### 3.1. Скиллы (кодификация ограничений)

| Скилл | Величина | Роль |
|---|---|---|
| `frontend-design` | кодификация | Несущий для лендингов и HTML-деков: дизайн-токены, типографика, уход от шаблонных дефолтов. Без него — «ещё один бутстрап» с ручной правкой. |
| **brand-kit** (написать) | кодификация | Бренд-токены (палитра графит/тёмно-синий/серый + один акцент; шрифты Inter/Manrope/IBM Plex Sans; визуальные do/don't) и запрещённые слова — не документ для чтения, а контракт в исполнимой форме. Первый по приоритету собственный артефакт. |
| скиллы Figma-плагина | точность | Приходят с remote-сервером Figma, повышают надёжность write-to-canvas. Отдельного решения не требуют. |
| `pptx` | — | **Условно.** Только если выход — настоящий `.pptx`. Рабочий процесс идёт через Gamma (markdown-outline), где скилл не задействован. Не класть превентивно. |

### 3.2. MCP-серверы

| Сервер | Величина | Роль | Scope |
|---|---|---|---|
| **Figma remote MCP** | точность + обратная связь | Переменные, компоненты, layout как источник истины дизайн-системы; capture live UI возвращает рендер в Figma слоями. OAuth, бета (write-to-canvas — будущая платная функция). | read + write-to-canvas |
| **Replit MCP** | генерация | Создание реплов из описания, запись/правка файлов, форк, чтение. Производит статику лендингов, которую забирают руками. | **allow-list: create, write, read, fork. deploy/publish — запрещены** |
| **Playwright MCP** | обратная связь | Замыкает петлю «сгенерировал → посмотрел → исправил»: агент открывает страницу, снимает скриншот, сверяет с брендом, правит сам. Убирает человеческий шаг визуальной приёмки с критического пути. Уже в стеке. | read |
| **Filesystem MCP** | точность | Читает канонический минимум и журнал дизайн-решений. Канон мал, влезает в контекст — вектор не нужен. Уже в стеке. | read + append (журнал) |

### 3.3. Hook (детерминированная отбраковка)

**Claude Code hook на выходе** прогоняет сгенерированный текст через список запрещённых слов и блокирует «революционный», «AI-powered», военную лексику до того, как артефакт покинет песочницу.

Обоснование: двадцать слов со синонимами модель рано или поздно пропустит. Если ошибку можно поймать проверкой — не полагайся на канал без проверки. Скилл генерирует, hook отбраковывает. Дешевле и надёжнее, чем дисциплина в каждом проходе.

## 4. Память

Три канала памяти с разным временем жизни; смешивать их — ошибка.

| Канал | Хранилище | Статус |
|---|---|---|
| **Канон** (токены, факты, глоссарий, запрещённые слова) | Filesystem MCP, read-only | Есть. Источник истины, приходит сверху. |
| **Журнал дизайн-решений** (принятый паттерн / отвергнутый, с причиной) | **отдельный** Obsidian vault, Filesystem MCP, read + append | Заводить, когда появится первое решение, которое дороже записать, чем перегенерировать. Растёт снизу. |
| **Вектор** (Pinecone) | — | **Не сейчас.** Триггер — не объём, а поломка прямой адресации. Песочница до этого не дорастёт в обозримом горизонте. |

Ключевой момент: журнал дизайн-решений — не «агент вспоминает», а **канон в становлении.** Принятый паттерн карточек, отвергнутый градиент — это INSIGHT-артефакты. Память песочницы и канонический минимум — один механизм single source of truth, две скорости наполнения. Vault журнала — **отдельный** от основного Obsidian-корпуса разработки: смешивать дизайн-заметки с оперативной памятью инженерии — нарушить границу контура.

## 5. Что НЕ класть (сигнал против шума)

| Исключено | Причина |
|---|---|
| **Pinecone MCP** | Канон мал, влезает в контекст. RAG поверх семи документов — точность ценой лишнего слоя и недетерминированной выборки. Overengineering для этого контура. |
| **Replit deploy/publish** | Необратимый side-effect. Публикация — человеческий шаг, вне песочницы. В allow-list этих инструментов нет. |
| **Perplexity, git/gitnexus MCP** | Research и AST репозитория — другая полоса. Не поднимают ни одну из трёх величин для генерации слайда. |
| **Общий Obsidian vault с разработкой** | Нарушает границу контура: песочница получит на чтение лишнее, инженерная память замусорится дизайн-заметками. |
| **Продуктовый GitHub MCP в одном профиле с write-инструментами** | Соединяет чтение продуктового кода с правом записи вовне. Поверхность prompt injection между инструментами. |

## 6. Эмпирические проверки перед сборкой

По каноническому принципу — проверить, прежде чем архитектурить вокруг.

1. **Allow-list Replit реально фильтрует вызов, а не прячет из подсказок.** Если MCP-клиент умеет только скрывать инструменты, а модель всё же может вызвать скрытый — граница дырявая; тогда Replit уходит в отдельный профиль без публикующих инструментов вовсе. Увидеть, что deny запрещает вызов.
2. **Полнота generate-контура Replit** — какие именно инструменты сервер отдаёт на создание и запись файлов (официальный remote даёт три публичных инструмента; проверить, покрывают ли они нужный generate-workflow без публикации).
3. **Figma write-to-canvas** — статус беты и модель тарификации на момент сборки (заявлена как будущая платная usage-based функция); не строить критический процесс вокруг бесплатности.
4. **Доступность внешних endpoint'ов из сетевого периметра** — `mcp.figma.com`, Replit remote — через прокси, как проверяется `api.anthropic.com`.

## 7. Минимальная сборка

Пять сущностей, каждая закрывает конкретную величину:

- `frontend-design` + собственный **brand-kit** скилл — кодификация.
- **Figma remote MCP** (read + write-to-canvas) — точность дизайн-контекста.
- **Replit MCP** (allow-list на генеративные инструменты) — генерация статики.
- **Playwright MCP** — сверка рендера, обратная связь.
- **Filesystem MCP** — канон + журнал дизайн-решений.
- **Hook** на запрещённые слова — детерминированная отбраковка на выходе.

Journal-vault и `pptx` — по мере того, как конкретный артефакт потребует. Не раньше.

## Замечание

Ни одна интеграция здесь не покупает пропускную способность человеческого критического пути напрямую — они поднимают качество машинного прохода и убирают отдельные человеческие шаги (визуальную приёмку — Playwright; вычитку запрещённых слов — hook). Это оправдано. Но реальный лимит проекта остаётся прежним: последовательный человеческий путь, не число подключённых серверов. Песочница масштабируется по качеству, не по скорости основателя. Держать это в рамке при добавлении любого шестого инструмента.
