# Инструкция: внедрение скиллов песочницы communications

**Тип:** руководство по внедрению
**Контур:** песочница генерации презентаций, лендингов, инфографики
**Клиент:** Claude Code
**Дата:** [заполняется]
**Проверено против:** официальная документация Agent Skills (Claude Code) на дату составления

---

## 0. Формализация

Скилл — не память, а контракт. Механически: имя и описание грузятся в системный промпт при старте (стоят единицы токенов), полный SKILL.md подгружается только когда модель сочла скилл релевантным, вложенные файлы — только когда до них дошло. Это трёхуровневое раскрытие: пятьдесят скиллов не давят на контекст, пока не сработали.

Отсюда два следствия, которые определяют всю инструкцию:

- **Триггер — это поле `description`, а не тело.** Модель склонна *недосрабатывать* — не звать скилл, когда он к месту. Описание пишется «настойчиво»: с перечислением ситуаций-триггеров, а не одной вежливой фразой. Вялое описание — главная причина, по которой скилл не запускается.
- **Скилл ≠ хук ≠ MCP.** Скилл даёт экспертизу по запросу — модель *решает* его применить. Хук выносит принуждение в harness — политика держится детерминированно, вне решения модели. Это разделение load-bearing: часть из девяти «скиллов» на самом деле должна быть хуками. См. §5.

## 1. Топология размещения

Два класса по времени жизни и области видимости.

| Класс | Каталог | Область | Что кладём |
|---|---|---|---|
| **Проектные** | `.claude/skills/<name>/` | шарятся через git, живут с песочницей | Всё, что специфично для Membrana: канон, глоссарий, факты, структуры документов |
| **Персональные** | `~/.claude/skills/<name>/` | все проекты пользователя | Ничего из этих девяти — они все проектные по определению |

Все девять — **проектные**. Они несут канон Membrana, а канон живёт с контуром, а не с машиной партнёра. Это прямое следствие инварианта single source of truth: скилл в git — часть источника истины, скилл на ноутбуке — его копия, которая дрейфует.

## 2. Структура каждого скилла

Минимум — каталог с одним файлом:

```
<skill-name>/
└── SKILL.md          # обязательно: YAML frontmatter + markdown-инструкции
```

Расширение по мере надобности (не превентивно):

```
<skill-name>/
├── SKILL.md
├── references/       # доки, грузятся в контекст по запросу (глоссарий, лист фактов)
├── scripts/          # детерминированный код (проверки, преобразования)
└── assets/           # шаблоны, шрифты, палитры для выхода
```

Frontmatter — обязательны только `name` и `description`:

```yaml
---
name: brand-kit
description: <настойчивое описание с триггерами>
---
```

Тело — 500–5000 слов инструкций. Когда SKILL.md разбухает — выносить в `references/` и ссылаться. Взаимоисключающие ветки держать в разных файлах, чтобы не грузить лишнее.

## 3. Девять скиллов: назначение и содержание

Порядок — приоритет внедрения, не алфавит. Первые — кодификация канона, наибольший прирост качества на проход.

### Эшелон 1 — кодификация канона

**1. `brand-kit`** — палитра (графит/тёмно-синий/серый + один акцент), шрифты (Inter/Manrope/IBM Plex Sans), визуальные do/don't. Держит палитру и шрифты в `assets/`, чтобы выход был воспроизводим.
*Триггеры в description:* лендинг, слайд, дек, инфографика, любой визуальный артефакт Membrana.

**2. `frontend-design`** — публичный скилл (не писать, подключить). Типографика, уход от шаблонных дефолтов для HTML-деков и лендингов. Дополняет brand-kit на уровне вёрстки.

**3. `lexicon-guard`** — запрещённые слова и dual-use переписывание над *текстом*. Список запрещённого — в `references/blocked-terms.md`.
*Важно:* это скилл-подсказка для генерации; жёсткая отбраковка — хук (§5). Скилл помогает не сгенерировать плохое, хук ловит то, что всё же прошло.

**4. `glossary-lock`** — «нижнее небо», «сенсорный узел» (не «нода», не «датчик»). Глоссарий — в `references/glossary.md`.

**5. `facts-guard`** — ни одной цифры из воздуха: метрика из листа фактов или `[TBD]`. Лист фактов — в `references/facts-sheet.md`.
*Это барьер против главного риска нового узла.*

### Эшелон 2 — кодификация workflow

**6. `gamma-outline`** — формат outline под Gamma: 13 слайдов, телеграфные буллеты 12–15 слов, без finishing slides. Кодифицирует то, что сейчас в промпте `PROMPT_PITCH_DECK.md`.

**7. `landing-structure`** — каноническая секционная структура `membrana.space`: проблема → решение → подход → стадия → ask. Фиксирует скелет.

**8. `diagram-vocabulary`** — единый язык для четырёх диаграмм: схема системы, семейство детекторов, TDOA, таймлайн. Шаблоны — в `assets/`.

### Эшелон 3 — обратная связь

**9. `render-check`** — workflow сверки через Playwright: снять скриншот, сверить с брендом, отбраковать самому. Замыкает петлю «сгенерировал → посмотрел → исправил».
*Это task-скилл:* вызывается явно (`/render-check`), а не по усмотрению модели.

## 4. Правило описания (где чаще всего ломается)

Плохо (недосрабатывает):
```yaml
description: Бренд-гайд Membrana.
```

Хорошо (настойчиво, с триггерами):
```yaml
description: >
  Дизайн-токены Membrana: палитра, шрифты, визуальные do/don't.
  Обязательно применять при любой генерации лендинга, слайда, дека,
  инфографики или HTML-артефакта — даже если пользователь не сказал
  «бренд» явно. Если создаётся что-либо визуальное для Membrana —
  этот скилл релевантен.
```

Тело SKILL.md — детали реализации; описание — триггер. Не путать.

## 5. Что на самом деле должно быть хуком, а не скиллом

Скилл модель *может* применить или забыть. Хук срабатывает *всегда*, вне решения модели. Три из девяти несут проверки, которые нельзя доверять усмотрению:

| Функция | Скилл (генерация) | Хук (отбраковка) |
|---|---|---|
| Запрещённые слова | `lexicon-guard` подсказывает при письме | Хук на выходе блокирует «революционный», «AI-powered», военную лексику детерминированно |
| Факты из воздуха | `facts-guard` направляет к листу фактов | (проверка сложнее автоматизировать; остаётся скиллом + ручная сверка) |
| Глоссарий | `glossary-lock` держит термины | Хук может ловить «нода»/«датчик» по списку замен |

**Принцип:** если ошибку ловит детерминированная проверка — она в хуке, не в скилле. Скилл генерирует правильно, хук гарантирует, что неправильное не выйдет. Запрещённые слова — классический случай: двадцать терминов со синонимами модель рано или поздно пропустит. Хук — это тот же список, но применяемый harness'ом, а не памятью.

Порядок: сначала скиллы (§3), затем хук на запрещённые слова поверх `lexicon-guard`. Скилл без хука работает, но течёт; хук без скилла ловит, но не учит генерировать лучше. Нужны оба.

## 6. Порядок внедрения

1. **`frontend-design`** — подключить готовый (быстрая победа, ноль написания).
2. **Эшелон 1** (`brand-kit`, `lexicon-guard`, `glossary-lock`, `facts-guard`) — написать. Наибольший прирост качества.
3. **Хук на запрещённые слова** — поверх `lexicon-guard`. Детерминированный барьер.
4. **Эшелон 2** (`gamma-outline`, `landing-structure`, `diagram-vocabulary`) — workflow.
5. **`render-check`** — когда Playwright MCP подключён и проверен.

## 7. Проверка после внедрения

По каноническому принципу — эмпирически, не на веру.

- **Скиллы обнаружены:** `ls .claude/skills/*/SKILL.md` показывает все девять.
- **Скилл срабатывает:** дать задачу без явного `/имя` и увидеть, что модель сама подтянула нужный скилл. Если недосрабатывает — усилить `description` (см. §4).
- **Хук блокирует вызов, а не прячет:** сгенерировать текст с запрещённым словом и убедиться, что хук *останавливает* выход, а не просто помечает. Та же проверка, что для allow-list Replit: deny должен запрещать, а не скрывать.
- **references/ грузятся по ссылке:** проверить, что агент открывает `glossary.md`/`facts-sheet.md`, когда доходит до них, а не игнорирует.

## 8. Сопровождение

Скиллы обновляются обычным git-циклом: правка → commit → push → тimecommand'ы подтягивают. Это и есть аргумент за проектное размещение: канон в git обновляется одним движением для всего контура. Персональные скиллы на машинах пришлось бы обновлять поштучно — путь к дрейфу.

Разделение источников списков намеренное: `blocked-terms.md`, `glossary.md`, `facts-sheet.md` — три разных файла в трёх скиллах, а не один общий. Меняется что-то одно — правится один файл, остальные не трогаются. Слить в один удобнее в конфиге, труднее в сопровождении.

## Замечание

Девять скиллов поднимают качество машинного прохода и убирают отдельные человеческие шаги (визуальную приёмку — `render-check`; вычитку запрещённых слов — хук). Это оправдано. Но ни один не ускоряет последовательный человеческий критический путь напрямую — реальный лимит проекта остаётся прежним. Скиллы масштабируют качество и воспроизводимость, не скорость основателя. Держать это в рамке при добавлении десятого: `design-decision-journal` оправдан не превентивно, а с первого отвергнутого решения, которое дороже записать, чем перегенерировать. Пустой журнал — поверхность обслуживания, не память.
