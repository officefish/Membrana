# Канонический минимум для контура communications

**Тип:** спецификация набора
**Назначение:** минимальный набор канонических документов, которые контур communications должен читать, чтобы производить внешние документы без дрейфа истины
**Дата:** [заполняется]

---

## Принцип отбора

Не «всё, что есть», а **минимальный набор, при котором контур может описать любой внешний документ, не выдумав ни одного факта.** Каждый документ в списке отвечает на вопрос: без него контур начнёт галлюцинировать или нарушать тон? Если нет — документ не входит в минимум, он справочный.

Набор делится на два слоя: **носитель истины** (факты и стадия) и **носитель формы** (тон и термины). Контур без первого врёт по существу; без второго — по стилю. Оба слоя обязательны.

## Слой 1. Носитель истины (факты, стадия, дорожная карта)

Что описывает состояние проекта. Источник ответа на вопрос «что у нас правда есть».

| Документ | Зачем в контуре | Без него |
|---|---|---|
| `WHITE_PAPER.md` (внутренний) | Канонический источник по видению, архитектуре, физике, дорожной карте, метрикам. «Север» для любого утверждения. | Контур придумывает стадию, метрики, этапы. |
| `ARCHITECTURE.md` | Границы пакетов и слоёв. Нужен, чтобы описывать компоненты по факту, а не по догадке. | Описания компонентов расходятся с реализацией. |
| `INTEGRATIONS_STRATEGY.md` | Эшелонный подход, каталог моделей (DSP / открытые нейросети / агентные). Это технический дифференциатор — сердце внешних документов. | Семейство детекторов описывается неточно, теряется главный сигнал зрелости. |
| `SINGLE_NODE_DETECTION_FIRST_PROMPT.md` | Формулировка stage-gate (precision ≥ 85%, recall ≥ 90% на одном узле) как рабочей дисциплины. | Stage-gate подаётся как лозунг, а не как блокирующий инвариант. |

## Слой 3. Производные артефакты канона (создаются под контур)

Три документа, которых сейчас нет как отдельных файлов, но которые контуру нужны в явном виде. Они извлекаются из слоёв 1–2 и фиксируются как отдельный канон — чтобы партнёр брал факт из листа, а не вычитывал его из white paper каждый раз.

| Документ | Что содержит | Почему отдельно |
|---|---|---|
| `FACTS_SHEET.md` (лист фактов) | Только проверенные утверждения: стадия, метрики-шлюзы, размер команды (4–7), демонстрируемый артефакт, дорожная карта по этапам. Каждый факт — с пометкой «подтверждён / гипотеза / [TBD]». | Главный барьер против дрейфа фактов. Партнёр берёт цифру отсюда, не выдумывает. Первый по приоритету артефакт контура. |
| `GLOSSARY.md` (заблокированный глоссарий) | Фиксированные термины: «нижнее небо», «сенсорный узел» (не «нода», не «датчик», не «устройство»), «семейство детекторов», «stage-gate», «наблюдательная система». | Терминологическая согласованность между слайдами, лендингом, white paper. |
| `BRAND_TOKENS.md` (бренд-токены) | Палитра (графит / тёмно-синий / серый + один приглушённый акцент), шрифты (Inter / Manrope / IBM Plex Sans, кириллица), визуальные do/don't (без сток-дронов и солдат; схема и абстракция), формат футера и нумерации. | Визуальная часть канона. Нужна для презентаций, лендингов, инфографики. |

## Что в минимум НЕ входит (явные non-goals)

Чтобы набор остался минимальным, зафиксируем, что сознательно **не** переносится в контур:

- **Операционные документы дня** (`STRATEGIC_PLAN_DAY.md`, `DAILY_STANDUP.md`, `MAIN_DAY_ISSUE.md`, `DAILY_CODE_REVIEW.md`) — это оперативный шум для внешних коммуникаций. Стадия проекта берётся из листа фактов, не из вчерашнего стендапа.
- **Реестр задач, ghost-closure, session-архив** — внутренняя кухня, к внешним документам отношения не имеет.
- **Инфраструктурные каноны** (`SERVICES.md`, `BACKGROUND_SERVERS.md`, ритм разработки) — за исключением `ARCHITECTURE.md`, нужного для точного описания компонентов. Остальное контур не читает: он описывает продукт для инвестора, а не сопровождает разработку.

Правило: если документ помогает **строить** систему, но не помогает **описать её снаружи** — он не в минимуме.

## Замечание о синхронизации (связь с вопросом топологии)

Этот список — вход для решения по топологии из брифа. Наблюдение, которое консилиум учитывает:

Слой 1 (носитель истины) **дрейфует** — код и архитектура меняются. Если контур в отдельном репозитории, именно эти четыре документа требуют живой синхронизации, иначе описания компонентов устаревают. Слой 2 и производные артефакты слоя 3 меняются редко — их можно переносить снимком без большой цены.

Отсюда практический вывод для варианта B: синхронизировать надо в первую очередь слой 1, и именно на нём проверяется, остаётся ли синхронизация односторонней и дешёвой. Если да — отдельный репо жизнеспособен. Если слой 1 требует ручного переноса — теряется живое чтение, ради которого контур и строился.

## Минимальный старт

Если строить контур сегодня, порядок ввода документов:

1. `FACTS_SHEET.md` — первым. Без него любой инструмент производит дрейф фактов.
2. `CLAUDE_PROJECT_SYSTEM_PROMPT.md` + `GLOSSARY.md` — форма и термины.
3. `MEMBRANA_ONE_PAGER.md` — камертон.
4. Слой 1 целиком — для точных описаний компонентов.
5. `BRAND_TOKENS.md` — когда контур переходит от текста к визуалу.

Остальное добавляется, когда конкретный артефакт этого потребует, а не «на будущее».
