## Зачем

Коворк #487 собрал три FREE-UC декомпозицией combined («вычитанием») — графы
корректны, но subtraction-подход сработал рентгеном и вскрыл **две течи
наглядности** палитры device-board. Не корректность — честность: имя узла врёт,
владелец времени неверный. FREE отгружается на этих графах; этот спринт платит
долг наглядности fast-follow, чтобы FREE-витрина (её скачивают и по ней учатся)
не учила пользователя по врущему графу.

**Принцип консилиума:** узел вводится, когда КЕЙС требует его для честности, а не
превентивно. Обе течи — из реального кейса, заземлены в коде.

## Задача PC-1 (первая) — отчёт одиночного детектора · вес ADR

**Проблема (улика №2).** `EnsembleAnalysisRef` принимается только обобщённым
входом `DetectionAnalysisRef` (`socket-type.ts` L153–157, union-исключение basn).
Его берут `make-detection-fusion` и `make-combined-report`. Отчётник
`make-report-from-analysis` прибит к `FftTrendAnalysisRef` и нейро не принимает.
→ Нейро-одиночка (`usercase-free-neuro-detection`) вынужден носить узел
`MakeCombinedReport` — «combined»-отчёт при ОДНОЙ модальности. Холст врёт.

**Гейт: ADR** (не полный консилиум — выбор реализации по готовому канону, но с
UI-ролью в комнате: выбор именно про наглядность). Два пути, ADR выбирает:
- **A. Расширить приём `make-report-from-analysis`** до обобщённого
  `DetectionAnalysisRef`. Дёшево, обратно совместимо (union шире прежнего —
  сохранённые графы не ломаются). Минус: имя «report-from-analysis» перестаёт
  точно отражать «отчёт по любому одному детектору».
- **B. Новый узел `make-detection-report`** (одиночный детектор → отчёт),
  симметрично `make-report-from-analysis` трендового. Чище имя, но +1 вид +
  миграция сериализованных сценариев (для bundled FREE дёшево).

**DoD PC-1:**
- ADR зафиксирован (`membrana-adr`-скилл) со ссылкой на консилиум и RETROSPECTIVE #487.
- Реализация выбранного пути; контракт сокетов согласован.
- `usercase-free-neuro-detection` переведён на честный отчёт: узел и заголовок
  отчёта на карточке **не содержат «combined»** при одной модальности (условие
  приёмки Верстальщика).
- Тесты: нейро-граф использует честный отчётник; combined (2 модальности) не задет.
- scoped CI `@membrana/device-board` зелёный; `catalog:verify-client` OK.

## Задача PC-2 (вторая) — периодический гейт без рекордера · вес консилиума

**Проблема (улика №1).** Периодизации окна без рекордера в палитре нет.
`spectrum-live` вынужден использовать `is-recording-window-full` (узел рекордера)
как таймер — спектральный сценарий зависит от рекордера, которого ему не нужно.
`SCENARIO_NODE_KINDS` без timer/interval-узла. Смежно: `collectorConfig.windowSec`
рантаймом НЕ используется (окно задаёт частота flush) — декоративное поле, врущее
параметром (находка spectrum F2).

**Гейт: полный консилиум** (`yarn consilium`) — новый владелец времени затрагивает
рантайм тика и семантику лупа, это core-контракт. Спорные точки для консилиума:
- Новый узел `is-window-elapsed`/`on-interval` (чистый гейт по host-часам, честный
  `windowMs`, который рантайм реально читает) **vs** обобщение
  `is-recording-window-full` до режима «без RecorderRef». UI-роль: против
  невидимого режима, за честное имя.
- Отношение к существующему `onTick` (`loopTick`) — это системная точка входа
  лупа, не гейт «прошло N мс» в main. Как соотносятся.
- Заодно: чинить ли декоративный `collectorConfig.windowSec` (рантайм читает vs
  удалить поле).

**DoD PC-2:**
- Консилиум-протокол (docs/seanses) с письменным консенсусом по контракту.
- Реализация владельца времени; `spectrum-live` переведён с рекордера-как-часов на
  честный периодический гейт; UI показывает единицу времени, которую рантайм читает.
- Класс «декоративных полей» (`windowSec`) разрешён по вердикту консилиума.
- scoped CI зелёный; живой Run spectrum-UC = проверка сборки (борд пересоздать из пикера).
