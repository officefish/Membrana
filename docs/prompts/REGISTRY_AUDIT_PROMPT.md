## Зачем

Реестр копит «мёртвые души»: карточка `active`, а работа давно сделана и иссью закрыта.
Список активных задач перестаёт быть картой дня — им нельзя пользоваться для планирования.
Аудит отвечает на один вопрос: **что из active закрыто на самом деле.**

## Главный принцип

**Механический признак «иссью закрыта» — это гипотеза, а не вердикт.**

Ловушка, ради которой этот конспект и написан: у Membrana есть **зонтичные иссью** —
одна иссью на несколько карточек (дорожная карта, эпик). Её закрытие НЕ означает, что
закрыта каждая карточка под ней. Прогон `yarn tasks:archive-closed --execute` на таком
наборе тихо похоронит незавершённую работу.

Следствие канона «расхождение код↔Issue = находка»: расхождение — это **вход в разбор**,
а не повод дожать одну сторону под другую.

## Слои проверки

### L0 — инвентарь тулинга (до аудита)

Проверить, что заявленные скрипты существуют. В прогоне 18.07 `package.json` ссылался на
`yarn tasks:audit` → `scripts/tasks-audit.mjs`, **файла нет**. Битая запись в манифесте —
находка сама по себе.

Рабочие инструменты на 18.07:
- `yarn tasks:archive-closed` — L1, dry-run по умолчанию (`--execute` для применения);
- `yarn issues:audit --manifest <path>` — аудит иссью по манифесту;
- `yarn task:archive <id>` — архивация одной карточки.

### L1 — механический слой (дёшево, автоматизируемо)

`yarn tasks:archive-closed` — active-карточки, чей `githubIssue` в списке закрытых.
Результат: **список кандидатов**, не список на архивацию.

### L2 — разметка кандидатов (обязательный фильтр)

Сгруппировать кандидатов **по номеру иссью**:

- **1 иссью → 1 карточка** — доверенный кандидат, идёт в L3 выборочно.
- **1 иссью → N карточек** — **зонтик, красный флаг.** Каждая карточка требует
  индивидуального вердикта. Никогда не архивировать пакетом.

Затем прочитать `stateReason` иссью — он различает два разных исхода:
- `COMPLETED` — сделано;
- `NOT_PLANNED` — **отменено**, не сделано. Архивировать можно, но `archiveNotes`
  обязаны сказать «отменено», иначе история соврёт.

### L3 — доказательство по существу (дорого, выборочно)

Для зонтичных и любых сомнительных карточек искать доказательство завершённости, а не
косвенный признак. Порядок надёжности:

1. **Критерий приёмки из карточки** — если в `notes` записан численный гейт, сверить с
   фактом. Живой пример 18.07: `vdr-hard-gate` требует `trends F1 >= 85%`, по факту
   пилота — **63.4**. Иссью-зонтик закрыт, карточка живая → **ложное срабатывание.**
2. **Мёрженый PR / код** — код важнее доков.
3. **Артефакт closure-review** в `docs/archive/`.
4. **`notes` карточки** — слабейший источник: у зонтичных карточек они пустые.

## Что выдать на выходе

Три корзины, каждая с обоснованием:

| Корзина | Что это | Действие |
|---|---|---|
| **Архивировать** | одиночная иссью, `COMPLETED`, есть доказательство | `yarn task:archive <id>` |
| **Отменено** | `stateReason = NOT_PLANNED` | архив + `archiveNotes` «отменено» |
| **Разобрать руками** | зонтик, или критерий приёмки не выполнен | вердикт владельца, НЕ трогать |

## Систематические находки (чинить отдельно, не в аудите)

- **`githubIssueClosedAt` никогда не заполняется** — в прогоне 18.07: 0 из 18. Поле в
  схеме есть, синхронизации нет → состояние закрытия нельзя узнать без обращения к `gh`,
  аудит невозможен офлайн и не воспроизводим по коммиту.
- **Зонтичные иссью не помечены в схеме** — нет признака «эта иссью — эпик/дорожная
  карта». Ловушка L2 ловится только глазами; напрашивается поле `umbrella: true` или
  отдельная иссью на карточку.

## Правила агента

- **Никогда не `--execute` по результату L1** без прохождения L2+L3.
- Ложное срабатывание дешевле пропуска: сомнительное — в «разобрать руками».
- Аудит **только предлагает**; массовая архивация — слово владельца.
- Отчёт с числами и причинами, а не «почищено N карточек».
