# Бриф-интервью: device-board хакатон 1

> **Формат:** команда (5 ролей) → владелец продукта.  
> **Минимум:** 20 вопросов с фиксацией ответов.  
> **Вход:** [`DEVICE_BOARD_HACKATHON_BRIEF.md`](./DEVICE_BOARD_HACKATHON_BRIEF.md) v0.3.  
> **Выход:** протокол `docs/seanses/hackathon-brief-interview-<YYYY-MM-DD>.md` (таблица «Вопрос | Ответ | Решение»).

**Как проводить:** идти блоками; на блокирующие вопросы (§Блокеры) — ответ до `hackathon:open`.

## Продукт и пользователь (6 вопросов)

| ID | Вопрос | Ответ |
|----|--------|-------|
| **P1** | Кто primary user: полевой оператор, инженер настройки, или оба с разными правами? | Преимущественно специалист по безопасности (в т.ч. оператор ПВО), доступ зависит от тарифа |
| **P2** | Сценарий по умолчанию после pairing: пустой канвас, готовый пресет «Drone watch», или копия с сервера? | Пока показываем модули и плагины; позже возможна развилка клиента на «лабораторник» и «устройство» |
| **P3** | Нужна ли **валидация** сценария до Run (как compile errors в Blueprints) или допускаем run с предупреждениями? | Желательна строгая предзапусковая валидация |
| **P4** | Один активный сценарий на устройство или несколько профилей с переключением? | Фиксированные системные ветки: `init`, `onStop`, `base loop`, `alarm loop` + пользовательские настраиваемые триггеры |
| **P5** | Должен ли пользователь **видеть signal graph и scenario** одновременно на одном экране? | Скорее да |
| **P6** | Минимальный уровень документации в UI: тултипы на блоках, wizard первого запуска, или внешний manual? | Hybrid: тултипы + wizard на 3 шага + ссылка на manual |

## Visual scripting и блоки (6 вопросов)

| ID | Вопрос | Ответ |
|----|--------|-------|
| **V1** | Типы рёбер в scenario: только **control flow** (exec), или также **data** между блоками (значения, буферы)? | Data-передача при поддержке узлами типов; capability входов/выходов дорабатывается узлами отдельно |
| **V2** | Блок «записать 5 секунд» — фиксированная длительность или параметр на ноде / в сайдбаре? | Параметр; максимум 30 сек |
| **V3** | Переход main → alarm: по **фронту** детекции, по устойчивому N из M окнам, или настраиваемо в блоке? | По фронту детекции |
| **V4** | **Функции/subgraph**: входы/выходы как pins на границе блока — да/нет; максимальная глубина вложенности в v1? | Pins на границе; глубина вложенности ≤ 1 (временно, возможно расширение) |
| **V5** | Именование и версии при export: semver сценария, hash содержимого, автор — что обязательно в JSON? | Обязательны `version` и `hash` |
| **V6** | Импорт JSON с **более новой** `version` — отказ, миграция, или импорт «как есть» с warning? | Отказ |
