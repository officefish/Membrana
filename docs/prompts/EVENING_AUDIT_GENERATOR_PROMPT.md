## Контекст

Вечерний ритуал не показывает, **что происходило за день**. Док 1
(`DAILY_CODE_REVIEW.md` — «как написано») есть, документа «что случилось» нет.
Владельческая формулировка 18.07: наглядно видеть движение в трёх местах, где живёт
работа — **репозиторий**, **реестр задач**, **граф правды**, — и отдельно видеть,
**сколько строк ушло на какую область**: основной продукт · кабинет · тулинг ·
бизнес-процессы · витрина для менеджмента.

### Развилка формы — решена владельцем, оформить ADR

Вердикт **F1/M4** заседания `evening-auditor` (18.07) ратифицировал для Дока 2 другую
форму: блоки «сверка / перепроверка / вне-границы» + тернарный вердикт `ok/drift/stale`.
Это документ **«сошлось ли сделанное с планом»**.

Владелец просит **другой документ** — хронику «что происходило», без оценок и сверки с
планом. Это не уточнение формы M4, а её замена для `DAILY_AUDIT.md`. Решение
владельческое и старше вердикта комнаты, консилиум не созывается — но **фаза 1 обязана
оформить ADR**, иначе завтрашний аудитор справедливо спросит, почему форма не та.

Сохраняется из M4 (проверено и полезно):

- машинные данные первичны, markdown — детерминированная проекция;
- живой `docs/DAILY_AUDIT.md` + датированный снимок `docs/archive/daily-day/<date>/`;
- `exit=0` ⟺ отчёт собран;
- «нет данных» показывается честно, а не пропускается.

**Не** переносится: тернарный вердикт дня, блоки сверки/перепроверки, слот `consumed`
двухфазной записи — они принадлежат документу-аудитору, а не документу-хронике.

### Что уже есть (проба, не результат)

На ветке `feat/evening-audit-generator` лежит прототип, подтвердивший осуществимость на
живых данных 18.07: **68% строк — бизнес-процессы, 29% — тулинг, продукт и кабинет —
ноль**. Проба переиспользуется как черновик, но проходит фазы заново: у неё нет ни

### Что построить (продуктовое описание)

`yarn audit:evening` печатает отчёт «что происходило» за день:

1. **Репозиторий** — коммиты по типам с внятными темами, влитые PR и главное:
   **строки по областям** (продукт · кабинет · тулинг · бизнес-процессы · витрина)
   с долей в процентах и наглядной полосой.
2. **Реестр задач** — что заведено, что уехало в архив, что сменило статус.
3. **Граф правды** — что закристаллизовано, что отозвано, что сменило статус.

---

### Архитектура / контракт

| Слой | Путь | Ответственность |
|------|------|-----------------|
| Ядро | `scripts/lib/audit-evening.mjs` | данные → данные, `render(данные) → md`; без git/fs/сети |
| Обвязка | `scripts/audit-evening.mjs` | единственная точка ввода-вывода: git, чтение, запись |
| Артефакт | `docs/DAILY_AUDIT.md` + `docs/archive/daily-day/<date>/audit.md` | живой файл + датированный вещдок |

**База дня** — последний коммит ДО `00:00` даты. Отчёт есть функция от двух снимков
репозитория, а не от момента запуска; отсюда воспроизводимость.

**Запрещено:**

- считать строки суммой по коммитам — файл, тронутый пять раз, посчитается пятикратно,
  и «сколько ушло на кабинет» превратится в «сколько раз коммитили кабинет».
  Только один `git diff --numstat` база→HEAD;
- схлопывать `+` и `−` в одну сумму: +500/−500 это переписывание, а не простой;
- звать LLM: отчёт обязан быть детерминированным и бесплатным;
- ходить в сеть (`gh` не нужен — всё из git и двух JSON).

### Фазы

Каждая фаза = отдельный PR с ревью и зелёным CI. Следующая начинается после мёржа.

#### Фаза 1 — канон и границы (docs, без кода)

- ADR: `DAILY_AUDIT.md` = **хроника дня** (три источника), а не аудитор плана по M4;
  явно назвать сохранённое и заменённое.
- Разрез областей и **правило отнесения пути** — канон, а не случайность регэкспа.
  Спорные случаи назвать поимённо (напр. `docs/handoff`, `docs/archive`, корневые файлы).
- GitHub Issue со ссылкой на ADR.
- **DoD:** ADR принят; разрез записан; issue заведён.

#### Фаза 2 — ядро и тесты (наружу ничего не выводит)

- `registryMovement`, `truthMovement`, `repoMovement`, `linesByArea`, `renderReport`.
- **DoD:** `yarn test:scripts` зелёный; ядро не импортирует `node:fs` и
  `node:child_process`.

#### Фаза 3 — обвязка, прогон, включение в ритуал

- `scripts/audit-evening.mjs`, `yarn audit:evening` (+ `--date`, `--stdout`).
- Живой прогон за 18.07 и хотя бы один прошлый день — отчёты приложить к PR.
- Включение в `ritual:evening` — **после слова владельца**, не молча.
- **DoD:** отчёт собирается за произвольный день; повторный прогон даёт тот же файл.

### Тесты

| Область | Минимум |
|---------|---------|
| Области | отнесение путей, включая спорные из фазы 1 |
| Подсчёт строк | файл, правленный многократно, не задваивается |
| Подсчёт строк | `+` и `−` не схлопываются |
| Граф правды | отзыв ловится сменой статуса, а не исчезновением записи |
| Реестр | массовая правка одних полей не выдаётся за движение работы |
| Проекция | `render` детерминирован: один вход → побайтово один выход |

### Definition of Done

- [ ] ADR принят, разрез областей — канон (фаза 1)
- [ ] Ядро чистое, тесты на фикстурах зелёные (фаза 2)
- [ ] `yarn audit:evening` собирает отчёт за любой день, повторный прогон воспроизводим
- [ ] Отчёт отвечает на три вопроса: репозиторий · реестр · граф правды
- [ ] Строки видны по пяти областям с долями
- [ ] `yarn turbo run lint typecheck test` — зелёный
- [ ] Три PR смёржены, `yarn task:archive evening-audit-generator`

### Порядок работы ролей

1. **Структурщик** — разрез областей, границы ядра и обвязки (ведущий).
2. **Математик** — детерминизм, отсутствие задвоения, честность `+/−`.
3. **Teamlead** — LGTM по фазам, удержание скоупа.
4. **Музыкант** — не участвует (audio/DSP не затронуты).
5. **Верстальщик** — читаемость таблицы областей (доли, полоса, порядок).
