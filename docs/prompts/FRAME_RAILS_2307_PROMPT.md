## Контракт (снят с кода, не с канона)

Легальны три полосы очереди: `preflight` (гейт до) · **`frames` (автоцепочка)** · `post` (ручной хвост). ### Специальный кадр `countersign` — окончательное закрытие дыры актора

**Зачем.** Цепь звеньев с ключами из билетов Фаррелла делает невозможной цепь, **сочинённую задним числом**: серверные метки времени и порядок переходов агент подделать не может. Но она **не** доказывает участие: у нас один API-ключ на всех, поэтому автор любого перехода — одно и то же лицо. Время доказывает подлинность последовательности, а не подлинность труда. Остаётся театральное соблюдение — прогнать танец с билетами, не сделав работы.

**Что закрывает.** `countersign` требует акта, который агент **физически не может произвести**: перехода/одобрения билета **актором, чьими учётными данными агент не владеет** (отдельный аккаунт трекера или живое действие капитана).

**Предикат кадра (тотальный):**

```
countersigned(sprint) =
  ∃ t ∈ ticket.transitions :
      t.by ∉ agentActors
    ∧ t.timestamp > closureAccept.timestamp
```

**Свойства.**
- Кадр стоит **между `closure-accept` и `archive`**: цепь не может дойти до архивации без контрподписи.
- Отказ **честный и громкий** — «ждёт контрподписи», а не тихий пропуск: пустое состояние объявляется, как и всюду в проекте.
- Это единственное место контура, где невозможность держится **не на времени, а на недоступности ключа**. Всё остальное агент способен исполнить сам — здесь он обязан остановиться.

**Предусловие реализации:** второй актор должен существовать. Сегодня его нет — один ключ на всех; заведение отдельной учётной записи для приёмки является частью этого кадра, а не отдельной задачей.

## Инструктаж агента: работа с Linear через прокси

Проверено живьём 22–23.07. Всё ниже — грабли, на которые уже наступили; повторять не нужно.

### Связь

- **Напрямую из РФ Linear не отвечает (403).** Егресс идёт **через прокси**: локальный конец туннеля на media-VPS, адрес — в `HTTPS_PROXY`.
- Node сам прокси **не подхватывает**: нужен явный диспетчер. Паттерн — `ProxyAgent` из `undici`, передаётся в `fetch` как `dispatcher`. Без него запрос уйдёт напрямую и получит 403.
- Ключ — `LINEAR_API_KEY`, команда — `LINEAR_TEAM_KEY` / `LINEAR_TEAM_ID` (в окружении; **не переспрашивать владельца**, это конфиг).
- Проверка связи — дешёвый read-запрос (`viewer`, `teams`). HTTP 200 = канал жив; 403 = мимо прокси.

### Что куда кладётся

- **Спринт — это проект Linear**, а не самодельный parent-issue. Cycles (родные спринты) в команде **выключены**; включение — настройка владельца. Прецедент: спринт уже клали проектом ранее.
- **Контейнер** (если нужен) заводится руками и живёт только в Linear, без парного GitHub-issue.
- **Под-задачи руками НЕ создавать** (норма #984): заводится **GitHub-issue** → зеркало само делает Linear-карточку → карточка перевешивается под контейнер и привязывается к проекту → регистрация в реестре со связями (`--issue`, `--linear`). Ручные билеты дают дубли — уже проходили.

### Ловушки API

- У задачи поле называется **`project`**, а не `projectId` (на чтение). На запись — `issueUpdate(input: { projectId })`.
- **`description` проекта ограничен 255 символами.** Длинный текст кладётся в `content`, иначе запись падает валидацией.
- Свободный план имеет **лимит активных задач**: при исчерпании любая запись падает с ошибкой лимита. Перед записью — проверить остаток.
- Расходные старые карточки **архивируются, а не удаляются**: архивация снимает их с активного счёта и **обратима**.
- Зеркало не знает о контейнере: **перевешивание под контейнер — отдельный шаг**, само не произойдёт.

### Гигиена

- Любая запись в Linear — **внешнее необратимое действие**: массовые операции (архивация, перенос) только по явному слову владельца.
- Ключ в логи не печатать; в отчётах — идентификаторы задач, не секреты.
