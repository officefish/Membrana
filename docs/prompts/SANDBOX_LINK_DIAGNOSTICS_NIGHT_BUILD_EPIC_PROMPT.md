## Night Build — промпт целиком

Ты выполняешь замороженный ночной эпик. Три фазы, scope не расширять, чекпоинт
в `NIGHT_BUILD_LOG.md` после каждой фазы. Prod не трогаем; всё проверяется
тестами/сборкой локально. Утром — `night:close` + handoff.

### NB0 — Пробник устойчивости связи (scripts/node-link-probe.mjs)

Зачем: владельцу нужен способ проверить «есть ли устойчивая связь устройства и
сервера» именно из песочницы (прокси-окружение), без гаданий по мигающему UI.

- [ ] `scripts/node-link-probe.mjs`: подключение как role=node (token+deviceId из
  аргументов; url по умолчанию из `VITE_CABINET_API_URL`), удержание N минут,
  presence.heartbeat, pong на health-ping (PCB6), счёт разрывов и кодов закрытия,
  реконнект, итоговая сводка (uptime%, разрывы, коды) + вердикт
  (УСТОЙЧИВО / СЕТЬ-ПРОКСИ 1006 / AUTH 4401 / НЕТ СОЕДИНЕНИЯ).
- [ ] Прокси-паритет с песочницей: HTTPS_PROXY/HTTP_PROXY → https-proxy-agent.
- [ ] Явное предупреждение в help/выводе: пробник вытесняет реальный клиент из
  реестра присутствия на время прогона (закрыть вкладку клиента).
- [ ] Юнит-тест на чистые функции (parseProbeArgs, summarizeProbe, resolveDefaultWsUrl)
  + строка в `test:scripts`.
- [ ] `package.json`: скрипт `probe:node-link`.

### NB1 — Видимый deviceId сопряжения (старая просьба владельца)

Зачем: владелец дважды просил «при сопряжении показывать, с каким устройством
сопряжён узел, подсвечивая его id». Сейчас Device ID показывается только
обрезанным `shortId` в модалке `MembraneLinkedPanel`.

- [ ] `MembraneLinkedPanel`: Device ID — полностью, моноширинно, **подсвечен**
  (badge/выделение), кнопка «копировать» (clipboard) с фидбеком.
- [ ] `NodeConnectionFooterIndicator`: рядом со статусом связи — короткий deviceId
  (`shortId`), полный — в title; чтобы id был виден постоянно, не только в модалке.
- [ ] Юнит-тест на новую логику, если она выносится в функцию; иначе — прогон
  существующих тестов клиента.
- [ ] Границы: только презентационный слой, никакой бизнес-логики в JSX.

### NB2 — Разбор флапа «связь установлена ↔ переподключение» (вкладка «Узлы», #269)

Зачем: узловой сокет стабилен (подтверждено консолью владельца), мигает шапка
вкладки «Узлы» — кабинетная realtime-линия или поллинг link-state (известный P2:
rate-limit на частый поллинг).

- [ ] Найти источник сообщений шапки в коде #269 (`board-right-sidebar` /
  «Узлы»-вкладка): какая линия рождает «переподключение».
- [ ] Зафиксировать причину цикла письменно (в handoff): реконнект кабинетной WS?
  поллинг с таймаутом? реакция на transient nodeOffline из-за вытеснения реестра?
- [ ] Если фикс тривиален и в ночном scope (UI/клиент, без core) — исправить с тестом.
  Если нет — оформить находку как задачу с репро-шагами, НЕ чинить силой ночью.

## Definition of Done (ночь)

- NB0: пробник работает (smoke: `--help` и разбор аргументов), тесты зелёные.
- NB1: deviceId виден и копируется; клиентские тесты зелёные.
- NB2: причина флапа названа по коду (не гипотезой) в handoff; фикс — только если тривиален.
- Все фазы: `yarn test:scripts` + тесты клиента + `check:boundaries` зелёные;
  чекпоинты в `NIGHT_BUILD_LOG.md`; утром `night:close` → handoff.

## Non-goals (ночь)

- Прод-деплой, прод-миграции (pairingStatus), изменение heartbeat-контракта.
- Решение pcb-d2-multinode (take:1) — отдельный дневной спринт.
- Расширение scope сверх NB0–NB2.
