## Контекст

Владелец хочет давать партнёрам доступ к бордам панели panel.mmbrn.tech без ручной
возни: партнёр регистрируется **промокодом** и получает доступ ко всем разделам;
владелец в owner-разделе **«Пользователи»** галочками управляет доступом каждого
пользователя к каждому разделу и чеканит промокоды под конкретные случаи.

Сегодня: panel-auth — HMAC-токены без хранилища (office stateless, ADR 0004 Р2);
invite-коды дают только роль ally; данные бордов публичны, клиентский гейт = UX
(консилиум detector-compare-board Р4 — доктрина НЕ меняется этим эпиком).

**Консилиум-гейт пройден ДО кода:**
[`panel-promo-access-2026-07-14.md`](../seanses/panel-promo-access-2026-07-14.md)
— все 7 развилок решены (Perplexity-ресёрч ×2; сгенерирован координатором по
подписке: API-ключ исчерпан; формат канона соблюдён). **Спор НЕ переоткрывать.**

### Архитектурные инварианты (из консилиума)

- **Entitlements поверх ролей**: партнёр = роль `ally` + `grants: string[]`
  (id разделов; `'*'` = все, текущие и будущие). Единый предикат:
  `canAccess(role, minRole) ∨ id ∈ grants` (wildcard раскрывается предикатом).
  Лестница ролей не размывается: owner-разделы гейтятся ролью, не грантом.
- **Store — истина, cookie — кэш отображения.** JSON-store office:
  `/var/lib/membrana-office/panel-users.json` (docker volume, атомарная запись
  tmp→rename, in-memory кэш). Session-cookie несёт снапшот
  `{role:'ally', sub: userId, grants, pv}`; `/me` сверяет `pv` c `permVersion`
  пользователя: отстал → **тихое переиздание** cookie со свежими грантами
  (НЕ разлогин); `revoked`/нет пользователя → public. Каждое изменение галочек
  инкрементит `permVersion`.
- **Промокоды серверные и отзываемые**: `{code: base32×16 CSPRNG, label, grants,
  expiresAt, maxUses, usedCount, revoked, createdAt}`; чеканятся ТОЛЬКО ручкой
  owner-UI (секрет и генерация на office, код показывается один раз).
  Старые HMAC ally-invite продолжают работать без миграции.
- **Office автономен**: локальные zod-DTO, без импортов панели/`@membrana/core`
  (кроме легального R4 RAG); панель не импортирует office.
- **ADR 0005** пишется в PU1 ДО кода модуля; ADR 0004 не правится.

### Фазы (порядок жёсткий; фаза = PR = closure review)

1. **PU1 — office-модуль `panel-users`** (`packages/background-office/src/modules/panel-users/`):
   - ADR 0005 «panel-users store» (инвариант store/cookie — в тексте).
   - Store-сервис: чтение при старте (нет файла → пустой реестр + громкий warning,
     видимая деградация: /me без store не роняет роль, но грантов не отдаёт),
     атомарная запись, in-memory кэш; аудит-массив (кап 500) в том же файле,
     регистрации тоже пишутся в аудит.
   - Ручки: `POST /v1/panel/register {code, name}` (public + rate-limit 5/час/IP
     через существующий `createSlidingWindowLimiter`; ошибки НЕ различают причину
     наружу) → создаёт пользователя, ставит session-cookie с грантами;
     `GET /v1/panel/admin/users`, `PATCH /v1/panel/admin/users/:id/grants`,
     `POST /v1/panel/admin/users/:id/revoke`, `GET|POST /v1/panel/admin/promo-codes`,
     `POST /v1/panel/admin/promo-codes/:id/revoke` — всё за существующим
     owner-guard; **не-owner получает 404** (существование не подтверждаем);
     `Cache-Control: no-store` на всём новом.
   - `/me` дополняется сверкой эпохи (см. инварианты).
   - Volume в `docker-compose.yml` И `deploy/background-office.prod.compose.yml`.
   - Тесты: чистое ядро store/эпохи/кодов + **инвентарный контракт-тест ручек**
     (паттерн `panel-auth.inventory.test.ts`) — против него пишется PU2.
   - **Живой смоук на VDS**: редеплой → файл пережил; регистрация тестовым кодом;
     галочка снята → следующий /me отдал урезанные гранты. Бэкап-строка
     (`scp .../panel-users.json backup/`) — в runbook.
2. **PU2 — панель: регистрация + гранты в гейте**:
   - Welcome: форма «Регистрация по промокоду» (код + «как вас называть»,
     ошибки словами, состояния с первого коммита).
   - `PanelIdentity` панели расширяется `grants`; гейт карточек — единый предикат
     (включая `'*'`); operator/owner-поведение не меняется.
   - Тесты предиката и парсинга /me; ноль новых зависимостей.
3. **PU3 — owner-раздел «Пользователи»** (`panel-users` в `SECTION_BOARDS`, minRole `owner`):
   - Матрица «пользователь × раздел» из `PANEL_SECTIONS`; **три состояния ячейки**
     (грант ✓ / нет ✗ / «покрыто `*`» — серая с разворотом `*` в явный список одним
     кликом); оптимистичные галочки с откатом, `aria-live="polite"`.
   - Чеканка кодов: пресет «полный (`*`)» или набор галочками → код в `<dialog>`
     с «скопировать», показывается ОДИН раз (в списке — префикс + label + остаток
     использований + revoke).
   - Аудит-хвост читаемым текстом. A11y: нативные чекбоксы с aria-label, клавиатура,
     `tabular-nums`. `yarn verify:tailwind-coverage` зелёный.
   - Деплой панели + смоук: партнёрский сценарий конец-в-конец.
