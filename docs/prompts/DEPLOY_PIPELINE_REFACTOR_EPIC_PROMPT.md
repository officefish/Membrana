## Контекст продукта

При выкате MP7b прод-сборка cabinet **упала**, потому что часть зависимых исходников
(`NODE_REALTIME_ENABLED` в `env.schema.ts` и обвязка node-realtime) осталась **незакоммиченной**
в рабочем дереве. Локальные `dev/typecheck/build` собирают из **рабочего дерева**, а прод —
из **git** (`git reset --hard FETCH_HEAD` на VPS). Это системный дефект: **источник истины при
проверке ≠ источник истины при деплое** → класс ошибки «works on my machine».

Цель эпика — сделать деплой **детерминированным** (сборка = функция от иммутабельного коммита/образа),
**гейтированным** (на прод едет только то, что зелено в CI) и **откатываемым** (rollback по тегу),
а также закрыть смежные пробелы: гигиена git (CRLF), расширенный smoke, регламент веток/миграций,
доставка полевого клиента `apps/client`.

**Что уже есть:**

- Деплой-скрипты по SSH: `scripts/_ssh-cabinet-deploy.mjs`, `scripts/_ssh-device-board-deploy.mjs`
  (`git fetch` + `git reset --hard` → `docker build` → `down`→`up` → smoke). Ветка из `.env`
  (`CABINET_GIT_BRANCH=main`).
- Сборка идёт **до** `down` → провал сборки **не роняет** прод (ценное свойство, сохранить).
- Авто-миграция: entrypoint cabinet выполняет `prisma migrate deploy`.
- Smoke в скрипте: health + SPA-код ответа (узкий).
- Compose/Dockerfile: `packages/background-cabinet/Dockerfile`, `deploy/cabinet-stack.sh`,
  `deploy/*.compose.yml`, Caddy.
- CI: частичный workflow (ветки `techies68` и пр.) — нет обязательного гейта на `main`.

**Принятые решения консилиума (зафиксировано):**

1. **Поэтапно:** DR1 — CI-гейт + сборка из чистого checkout (закрывает инцидент сейчас);
   DR2 — образ в CI → registry → на проде только `pull` (устраняет класс ошибки полностью).
2. **`.gitattributes` `eol=lf`** — убрать CRLF-шум, маскировавший недокоммиченные файлы.
3. **Zero-downtime — отложено с условием:** сначала регламент **expand/contract** совместимых
   миграций; rolling — после появления SLA/внешних потребителей API. (DR7 — gated/deferred.)
4. **Доставка `apps/client`** — отдельный таргет + индикатор версии + тест совместимости контракта.
5. **Правила веток:** `main` всегда деплоируемое; выкат только из `main`; PR по одному эпику;
   длинные мульти-эпиковые ветки запрещены; архитектурные контракты — через `vesnin`.
6. **Расширенный smoke:** login, список узлов, `prisma migrate status`, проверка рантайм-канала.
7. **Rollback + теги релизов:** `cabinet-vX.Y.Z`; rollback = деплой предыдущего тега; раннбук.

## Мнение виртуальной команды

```text
[Teamlead — Vesnin]:
Корень — расхождение источников истины. Делаем поэтапно: DR0 (гигиена+gate, дёшево и сразу),
DR1 (обязательный CI-гейт на main, деплой блокируется на красном), DR2 (образ в registry — на
проде только pull, класс ошибки исчезает физически). DR3 rollback+раннбук, DR4 smoke, DR5
регламент веток/миграций, DR6 доставка client. DR7 zero-downtime — отложено с условием. Сохранить
свойство «build до down». Приоритет — измеримая метрика: ноль прод-сборок, не воспроизводимых в CI.

[Структурщик — Ozhegov]:
Один воспроизводимый build-context. DR2 даёт иммутабельный артефакт-образ, его «недокоммитить»
нельзя. Деплой-скрипты разделить: «build & publish» (CI) vs «pull & up» (VPS). Gate чистого
рабочего дерева — в скрипте перед push. Машиночитаемый JSON-итог деплоя. Запрещено: собирать на
VPS из изменчивого дерева как основной путь; смешивать build-context разных сервисов; деплоить из
длинной мульти-эпиковой ветки.

[Математик — Dynin]:
Инвариант корректности: сборка — чистая функция от иммутабельного входа (коммит → артефакт).
Метрика DR1/DR2: доля прод-сборок, падающих по причине, не воспроизводимой в CI, = 0. Для rolling
(DR7) инвариант expand/contract: схема совместима И со старым, И с новым кодом — это предпосылка,
вводится в DR5 раньше самого rolling. setMode/деплой идемпотентны: повторный деплой того же тега
не меняет состояние.

[Музыкант]:
Smoke сейчас «глухой» — только health/200. DR4: функциональная проверка после up — login, список
узлов, `prisma migrate status`, и проверка живости рантайм-канала (WS поднялся, команда доходит).
Зелёный деплой без функционального smoke = ложно-зелёный. Headless без устройства — не блокер.

[Верстальщик — Rodchenko]:
Фича-флаги делятся на build-time (вшиваются в образ) и runtime — это надо явно зафиксировать в
DR2, иначе «один образ — разное поведение на стендах» не выйдет. DR6: в полевом клиенте — индикатор
версии и «доступно обновление»; смотреть DESIGN.md, состояния loading/empty, a11y. Версии
сервер↔клиент видны пользователю.
```

### Что построить (по фазам)

1. **DR0 — гигиена и gate.** `.gitattributes` с нормализацией EOL для исходников; в SSH-деплой-скриптах
   добавить проверку «нет незакоммиченных изменений в собираемых пакетах» (предупреждение + явный
   флаг подтверждения для обхода).
2. **DR1 — CI-гейт.** GitHub Actions: на push в `main` и деплой-ветки — `turbo run lint typecheck test build`
   из **чистого checkout** (без рабочего дерева разработчика). Деплой-скрипт перед стартом читает
   статус CI деплоируемого коммита (`gh run list`/`gh api`) и **останавливается на красном**.
3. **DR2 — образ в registry.** CI собирает образы cabinet-api и cabinet-web, тегирует
   (`cabinet-vX.Y.Z` + commit SHA), пушит в registry. Деплой на VPS меняется с «build из git» на
   **`pull` образа по тегу** + `up`. Явно разделить env: build-time (вшивается в образ, напр. `VITE_*`)
   и runtime (через `.env`/compose). Сохранить свойство «провал перед переключением не роняет прод».
4. **DR3 — rollback + сводка.** Откат = деплой предыдущего тега одной командой. Деплой выдаёт
   машиночитаемую JSON-сводку (тег, SHA, образы, результат smoke, длительность) в артефакт/лог.
   Раннбук в `MEMBRANE_PLATFORM_DEPLOY.md` дополнить процедурой отката.
5. **DR4 — smoke-suite.** Единый скрипт пост-деплой проверок: `GET /health`, login (тест-юзер),
   `GET /v1/...` список узлов, `prisma migrate status`, проверка рантайм-канала (WS подключение +
   доставка тест-команды). Вызывается и из деплоя, и вручную; код выхода ≠ 0 при провале.
6. **DR5 — регламент.** В `CONTRIBUTING.md`: `main` всегда деплоируемое; выкат только из `main`;
   один эпик — одна ветка/серия PR; длинные мульти-эпиковые ветки запрещены как источник прод-выката;
   контракты `@membrana/core` — через `vesnin`. Отдельный раздел — регламент **expand/contract**
   совместимых миграций (что можно/нельзя в одной миграции, как делить несовместимые изменения).
7. **DR6 — доставка десктоп-продуктов (переопределено 2026-06-18).** `apps/client` — это **dev-песочница**,
   а не поставляемый продукт. Веб-клиента в продуктовой линейке нет. Конечные продукты — **Electron-приложения**:
   `membrana-studio` (Electron-оболочка вокруг `apps/client` как renderer, полная работа с контентом) и
   будущее лёгкое `device`-приложение (периферия: как можно скорее доставить данные на сервер; renderer —
   `device-board`). Доставка = **Electron-пакеты** (`electron-builder`), а не статический бандл за Caddy.
   Сборка десктопа **декаплирована** от деплоя сервера и **условная**: запускается только когда изменения
   коснулись приложения или его зависимостей (turbo affected / path-filter) — нет смысла пересобирать
   тяжёлый Electron на каждый серверный деплой. В десктоп-клиенте — индикатор версии и «доступно обновление»
   (сравнение версии/контракта с сервером); автотест совместимости контракта `runtime` сервер↔клиент.
   `device` как отдельное приложение — паттерн на будущее (follow-up), в этой фазе эталон — `membrana-studio`.
8. **DR7 — zero-downtime (отложено).** Rolling update без `down→up`. Реализуется только после DR5
   (регламент expand/contract) и при появлении SLA.
