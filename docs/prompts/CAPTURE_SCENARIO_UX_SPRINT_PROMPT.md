## Требования владельца (дословно, 2026-07-07)

1. В карточке узла в кабинете не видно, с каким устройством он связан. Макет карточки:
   «Узел 1» → его id → **«связан с устройством {id устройства}»** → статус online/offline.
2. При переходе в раздел «Журнал» (кнопка на карточке узла) произошло **отвязывание**
   (release захвата), но на устройстве (apps/client) это не было отражено.
3. Нужен **список сценариев на сервере**, чтобы было понятно, какой сценарий запускаем.
   Один сценарий всегда выбран. Так как сценарии могут быть пользовательскими, список,
   возможно, должен предоставляться клиентом при захвате устройства.
4. На клиенте при захвате должен открываться борд с **актуальным сценарием сервера**
   (выбранным на сервере). Пока устройство захвачено — оператор **не может выйти из борда**
   (и при работе, и при остановке). При отпускании кнопка выхода разблокируется.
5. Если сервер недоступен и пользователь в диалоге выбирает «Остаться в связанном режиме» —
   в **шапке клиента** должно появиться предупреждение, что связи с сервером нет.

## Фазы

### CX1 — Карточка узла: «связан с устройством {deviceId}» · S

**Пакет:** `apps/cabinet` (NodesPage / NodeCard). Данные уже есть: `node.device.mediaDeviceId`.

- Структура карточки сверху вниз: label узла → id узла (как сейчас) →
  строка «связан с устройством `{mediaDeviceId}`» (полный id, `font-mono`, копирование по
  образцу NB1 `MembraneLinkedPanel`) → бейдж online/offline (как сейчас).
- Узел без сопряжения: строка не показывается (остаётся бейдж «не сопряжён»).
- Зеркальность с NB1: id, который клиент показывает у себя («Сопряжённое устройство»),
  совпадает с id в карточке узла — оператор сверяет глазами.

### CX2 — Переход в «Журнал» не должен ронять захват; release всегда отражается на клиенте · M

**Пакеты:** `apps/cabinet` (+ при необходимости `packages/background-cabinet`).

- **Диагностика (зафиксировать в PR):** `useCabinetNodeRuntime` живёт в `NodesPage`;
  переход на JournalPage размонтирует страницу → кабинетная WS-сессия рвётся → сервер
  делает release по сессии кабинета. Подтвердить и назвать точный механизм по коду.
- Фикс: capture-сессия кабинета переживает навигацию между разделами кабинета
  (поднять runtime/WS на уровень `CabinetShell` или эквивалент). Захват отпускается
  только явной кнопкой «Отпустить», TTL или закрытием кабинета.
- **Клиентская половина требования:** если release всё же произошёл (любой reason),
  устройство обязано отразить его в UI (бейдж захвата исчезает, борд разблокируется).
  Путь `board.release` → `boardLeaseBridge` уже есть — покрыть тестом сценарий
  «release по разрыву кабинетной сессии» и починить, если broadcast в этом пути не шлётся.
- Тест: unit на «runtime переживает смену раздела»; ручной smoke-скрипт в PR-описании.

### CX3 — Список сценариев: клиент объявляет, сервер хранит выбранный · M

**Пакеты:** `@membrana/core` (контракт), `packages/background-cabinet` (хранение/ретрансляция),
`apps/client` (объявление списка), `apps/cabinet` (dropdown). Поглощает `db-scenario-selector`.

- Contract-first: тип `DeviceScenarioListPayload` в core — клиент при захвате (и при
  изменении набора сценариев под захватом) объявляет список `{id, title}` + `selectedScenarioId`.
- Инвариант: **один сценарий всегда выбран** (по умолчанию — сохранённый сценарий устройства).
- `selectScenario` уже в whitelist тарифа v2 (`TARIFF_CABINET_RUNTIME_COMMANDS`) и
  wire-заготовки готовы (CT1) — соединить, не изобретать новый канал.
- Кабинет: dropdown списка сценариев на карточке захваченного узла; «Пуск» запускает
  выбранный; смена выбора шлёт `selectScenario`.
- Сервер не хранит тела сценариев — только список объявленный устройством и выбранный id
  (сценарии пользовательские, живут на устройстве; канон CT3-решение владельца 2026-07-02).
- Тесты: валидация payload в core; gateway-ретрансляция; UI-логика выбора (не в JSX).

### CX4 — Борд-лок на клиенте под захватом · M (после CX3)

**Пакет:** `apps/client` (boardLeaseBridge, agenda-стор, device-board module).

- PCB5 уже форс-открывает борд при захвате — расширить: борд открывается с
  **актуальным выбранным сценарием сервера** (из CX3-контракта).
- Пока `capture !== null`: выход из device-board заблокирован — переключение модулей
  в agenda-сторе запрещено, кнопка выхода задизейблена с подсказкой
  «Устройство захвачено кабинетом»; работает и при запущенном, и при остановленном сценарии.
- На release (любой reason: operator / ttl-expired / server-restart) — разблокировка.
- Инвариант канона §3.3 сохраняется: **emergency stop доступен всегда**; блокировка
  навигации не блокирует остановку звука.
- Тесты: pure-логика «можно ли покинуть борд» (unit); смена сценария под захватом.

### CX5 — Предупреждение в шапке клиента «нет связи с сервером» · S

**Пакет:** `apps/client` (AppHeader, nodeConnectionStore, ConnectionFallbackDialog).

- Выбор «Остаться в связанном режиме» при недоступном сервере → в `AppHeader`
  постоянный warning-индикатор («нет связи с сервером», tone warning по DESIGN.md).
- Снимается автоматически при восстановлении связи (реконнект WS / успешный ping);
  повторный показ при следующем разрыве — без повторного диалога.
- Не дублировать футер-индикатор NB1 — шапка отвечает на «почему всё молчит», футер
  остаётся детальным статусом. a11y: `role="status"`, видимость в обеих темах.
- Тесты: стор-логика переходов (разрыв → stay-linked → баннер; реконнект → баннер убран).

## Out of scope

- Редактирование/создание сценариев из кабинета (тариф v3), пауза/resume — вне тарифа v2.
- Multi-node UI кабинета (`pcb-d2-multinode` — отдельный backlog).
- Изменение прод-контракта heartbeat/TTL (канон §3) — только чтение.
- Синхронизация тел сценариев на сервер (сценарии живут на устройстве).
- Хвост lastSeenAt на registerNode (PCB4) — отдельная задача, не тащить сюда.
