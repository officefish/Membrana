# Промпт: W2 — доп. реестры улова и ловушек; шаблон антипаттерна

> **M** · `bw-w2-registries` · [#948](https://github.com/officefish/Membrana/issues/948) · lead **ozhegov** · parent `bestiary-workshop`

## Контекст

T1/T8/T12/T16/T17: нужны **два** дополнительных носителя (не замена классов):

1. **Улов** — зверёк / гранула (пойманный пример), отдельно от specimens (T2) и от
   `BESTIARY_LIST` (T16).
2. **Ловушка / дока** — реестр или индекс документации ловушек-исходников.

T13/T14: антипаттерн — **абстрактный шаблон**; ловушка уточняет род зверей на примерах.
Цепочка смысла: шаблон → ловушка → зверёк (улов).

## Промпт целиком

1. Выбрать пути в доме (предложение; править только с LGTM):
   - улов: напр. `docs/audit/bestiary/registry/CATCH_LIST.md` (или `catch/`);
   - ловушки: напр. `docs/audit/bestiary/traps/` + индекс в `registry/TRAPS_LIST.md`;
   - шаблоны антипаттернов: напр. `docs/audit/bestiary/antipatterns/<id>.md`
     (абстракт; **не** pins кита).
2. Зафиксировать **формат строки** (T17) — таблица полей для:
   - гранулы улова (id, класс/ссылка на шаблон, evidence path/PR, дата, статус);
   - карточки ловушки (id, какие классы/шаблоны целит, paths prompts+scripts, kit pin?).
3. Написать ≥1 пример улова и ≥1 карточку ловушки (можно stub на существующий
   класс линзы + ссылка на specimen — **без** нового детектора).
4. Носитель шаблона антипаттерна: контракт «как паттерн» (проблема / форма /
   анти-примеры ссылкой); ≥1 stub (можно для уже известного класса).
5. README дома: таблица «что писать / в git?»; явное «доп. реестр ≠ BESTIARY_LIST»;
   AGENT_PROMPT — сценарии Inventory-Catch / Trap-Doc (HARD GATE: не путать specimen).

## Acceptance criteria
