# Sprint prompt: Туториалы для партнёра по внешним коммуникациям (PT0–PT3)

| Поле | Значение |
|------|----------|
| **Sprint** | `partner-tutorials` |
| **Тип** | Doc/tutorial-разработка (не имплементация скиллов/хука/MCP) |
| **Опора** | Спринты `comms-contour-environment` (CC1–CC9) + `comms-sandbox-docs-adaptation` (CD1–CD6); живой канон `docs/comms/canon/` |
| **Решение** | Консилиум `docs/seanses/partner-tutorials-design-2026-07-06.md` (5 ролей, консенсус 20 реплик) |
| **Size** | M (4 задачи) |

> **Рамка (консилиум):** партнёр — нетехническая роль. Туториалы **показывают** арсенал на живом
> рабочем сценарии (сгенерировать лендинг/дек/инфографику от канона до файла в `out/`), а **не
> перечисляют** его — иначе дубль GUIDE/RUNBOOK. Держат инвариант «сток не исток» и leaf-zero границы.

## Поправка к протоколу консилиума (зафиксировать)

В протоколе тема (2) названа «сверка со Storybook (по CC9)» — **неверно**. В стеке контура Storybook
нет; за сверку рендера отвечает **Playwright MCP** (скилл `render-check`). CC9 — «первый выход v0.1 +
срез партнёра», не установка Storybook. В спринте тема (2) = сборка v0.1 → сверка через
`render-check`/Playwright.

## Реальность, к которой привязаны туториалы

- Арсенал партнёра: 4 MCP-сервера (filesystem=живой канон, playwright=сверка рендера, figma=дизайн-
  контекст, replit=генерация лендингов без деплоя); 9 скиллов (brand-kit, frontend-design,
  lexicon-guard, glossary-lock, facts-guard, gamma-outline, landing-structure, diagram-vocabulary,
  render-check); детерминированный `tone-guard` (`checkTone` блокирует хайп/военную лексику/эмодзи);
  seam-генератор `yarn generate:v0.1`; живой канон `docs/comms/canon/`.
- Границы: leaf-zero (`check:boundaries`), запись только в `out/`, secret-scan (CC4).

## Формат (общий для всех туториалов)

Единый шаблон Верстальщика: шапка (цель + время выполнения), блок «что понадобится», нумерованные
шаги (один акцент на шаг), блок «результат», чек-лист самопроверки. Тёмная тема, палитра/типографика
из `BRAND_TOKENS.md`, `tabular-nums` для чисел. a11y: уровни заголовков, alt к скриншотам, не только
цветом. Термины — **только** из `GLOSSARY.md`; нет термина → заявка на пополнение, не локальное
определение. Тон конструктивистский, честный; каждое утверждение о метриках — «подтверждено /
гипотеза / TBD»; без PR-преувеличений и без сырых внутренних цифр.

Размещение: исходники `apps/comms-studio/docs/tutorials/`, собранные артефакты и скриншоты
`apps/comms-studio/out/tutorials/`, ссылки относительные внутри контура (без абсолютных путей
монорепо и `@membrana/*`).

## Задачи

### PT0 — Шаблон-конструкция туториала
- [ ] Единый шаблон (шапка/что понадобится/шаги/результат/чек-лист) как первый артефакт, до наполнения.
- [ ] Frontmatter-контракт: версия + дата + ссылка на версию `FACTS_SHEET.md`.
- [ ] Тёмная тема + `BRAND_TOKENS.md` + a11y-каркас.
- **Роль:** Верстальщик (rodchenko). **Size:** S. **Зависит:** —

### PT1 — Как достать актуальные факты из `FACTS_SHEET.md` и не соврать
- [ ] Фундамент: без умения читать лист фактов партнёр плодит дезинформацию (репутационный риск).
- [ ] Показать: где факт, как читается пометка «подтверждён / гипотеза / TBD», что делать при отсутствии (`[TBD]`, не выдумка).
- [ ] Чек-лист самопроверки в конце.
- **Роль:** Математик (dynin) ведёт, Teamlead (vesnin) — факты. **Size:** S. **Зависит:** PT0

### PT2 — Сквозной сценарий: первый выход v0.1 от канона до файла в `out/`
- [ ] End-to-end: собрать 3–5 компонентов, сверить через `render-check`/Playwright (НЕ Storybook), забрать файлы из `out/`.
- [ ] По ходу раскрыть арсенал в деле (канон → скиллы → генерация → сверка), не перечислением.
- [ ] Показать срабатывание `tone-guard` на нарушении (запись отклонена) — «сток не исток» на живом примере.
- **Роль:** Верстальщик (rodchenko) ведёт, Структурщик (ozhegov) — границы. **Size:** M. **Зависит:** PT1

### PT3 — Как честно рассказывать о технологии и стадии
- [ ] Пары «до/после»: честная формулировка vs преувеличенная (аранжировка контраста).
- [ ] Пометки риска на каждое утверждение о метриках; никаких сырых внутренних цифр (P/FPR — гипотезы стадии, не маркетинг).
- [ ] dual-use тон (GLOSSARY §4), ссылки на глоссарий для терминов.
- **Роль:** Музыкант (kuryokhin) ведёт, Teamlead (vesnin) — LGTM тона. **Size:** M. **Зависит:** PT1, PT2

## Порядок производства

Строго **PT0 → PT1 → PT2 → PT3**, не параллельно. PT3 выпускается только после LGTM на PT1 и PT2
(самый рискованный по тону).

## Non-goals

- **Не** имплементировать 9 скиллов, хук, MCP-конфиг (это setup партнёра по RUNBOOK).
- **Не** менять контур `apps/comms-studio` (реализация закрыта CC1–CC9).
- **Не** дублировать RUNBOOK/GUIDE — туториал показывает сценарий, а не пересказывает справочник.
- **Не** заводить отдельный «обзорный» туториал-карту арсенала (раскрывается через PT2); PT4 — только по отдельному решению владельца.

## Definition of Done

- Три туториала (+ шаблон PT0) в `apps/comms-studio/docs/tutorials/` по единому шаблону; собранные версии + скриншоты в `out/tutorials/`.
- `check:boundaries` (leaf-zero) зелёный; secret-scan (CC4) зелёный; в тексте нет `@membrana/*`, абсолютных путей монорепо, обратных импортов.
- Все термины ⊆ `GLOSSARY.md`; каждое утверждение о метриках помечено.
- Каждый туториал имеет frontmatter (версия, дата, версия `FACTS_SHEET.md`).
- Прогон «холодным» читателем: ≥80% шагов без внешнего вопроса, итоговый артефакт = эталон.
- LGTM Teamlead на тон и факты (ревью на рендере); PR только в `apps/comms-studio/**` (CODEOWNERS CC3).
