# Промпт: RT-8: daily-ревью видит код дня, а не остаток рабочего дерева

> **Стаб подзадачи эпика** `ritual-trust-contour` ([#539](https://github.com/officefish/Membrana/issues/539)).
> Детали доопределяет `rt-0-gate`. **Заполнить до кода.** Размер: **M**.
> Эпик-промпт: [`RITUAL_TRUST_CONTOUR_PROMPT.md`](./RITUAL_TRUST_CONTOUR_PROMPT.md).
> Расследование: [`code-review-honesty-report-2026-07-16.md`](../seanses/code-review-honesty-report-2026-07-16.md).

## Что построить

`daily`-ревью должно оценивать **код дня** (дифф слитых за сутки коммитов), а не остаток рабочего дерева. Плюс гвард свежести и тест на слепоту.

**Не трогать:** ревью по запросу (`--pr` / `--branch` / `--uncommitted` / `--staged`) — оно честное, тянет полный дифф.

## Прецедент (живой — на нём и демонстрировать)

15.07: `collectRepositoryContext` не обходит дерево исходников (самодокумент, строки 206–207); `git diff --name-status` на чистом после коммитов дереве пуст. Ревью перечислило крупные слияния дня (PC-1/PC-2/cowork FREE) и оценило их как **T0 docs-only «Runtime не затронут»** — потому что смотрело на остаток рабочего дерева (docs-снапшоты). Вердикт ушёл в MAIN_DAY_ISSUE 16.07 как «✅ Вход в Фазу 0».

## Источник

Расследование 16.07 (находки 1–6). Смежно с `rt-7` (вход из метаданных, не из состояния) и `rt-1` (замкнуть контур достоверности).

## Решение консилиума (`code-review-honesty-refactor-2026-07-16`, 21 реплика)

**Вариант A с гардами честности.** Deep research (Q2) подтвердил направление: вечернему ревью подают дифф main утром-vs-вечером / squash-коммиты, не сырое дерево.

- Новая чистая лемма `collectDayWorkDiff(period) → { commits, diff, stats, precision }` в `scripts/lib/`; пакетный граф не трогаем.
- `daily`-ветка генератора зовёт `collectDayWorkDiff` и ревьюит **дифф коммитов дня**, а не снимок рабочего дерева.
- `collectRepositoryContext` **сохранить** — остаётся леммой «снимок рабочего дерева» для гигиены (untracked, чистота перед деплоем). По смыслу не менять.
- Границы периода двухуровнево: primary — `gh pr view --json mergeCommit` → родитель squash-SHA (`precision: exact`); fallback — `git log --since/--until --merges` без сети (`precision: approximate`).
- **Честная деградация:** шапка `DAILY_CODE_REVIEW.md` — поля `режим` (`работа дня` / `рабочее дерево — дифф недоступен`), `precision`, `период (base..head, N коммитов)`. Конец молчаливому «Runtime не затронут».
- Гвард свежести: таймаут `gh` 5–10 с → fallback; вечерний ритуал не блокируется, `|| true`-семантика хвоста сохранена. Согласовать с `rt-5`.
- Граница: rt-8 = M, только генератор. Гейт `main-day-probe-gate` (#533) не трогаем — отдаём ему честный маркер как контракт на стыке.

### Golden-тест (обязателен)

Регресс на 15.07: день крупных слияний (PC-1/PC-2/cowork FREE) **не** должен давать вердикт `T0 docs-only`. Плюс: пустой день; день с одним PR; день с чужими пограничными коммитами. Все детерминированы, без сети (фикстуры git-лога).

## Вопросы для research (Q1–Q3)

1. **Landscape:** какие практики сложились в 2025–2026 для автоматического/LLM daily code review в командах разработки — что именно ревьюят на уровне «сводки дня» (полный дифф слитых за сутки коммитов, рабочее дерево или только per-PR), на каком шаге цикла его зовут, и как гарантируют, что сводка построена на коде, а не на метаданных вроде статуса CI и тем коммитов?
2. **Fit (Membrana):** для маленькой команды с AI-агентами, где код в течение дня уезжает в main squash-мёржами и к вечеру рабочее дерево чистое, как на практике дают вечернему автоматическому ревью реальный код для анализа и как отличают честное «code review» от «сводки статусов, оформленной как ревью»?
3. **Risk:** чем рискует команда, подавая вердикт автоматического daily-ревью (Tier, «всё зелёное») как доверенный вход в планирование следующего дня — ложная уверенность, оценка по метаданным вместо кода, устаревание сводки; какие приёмы верификации и гарды свежести применяют на практике?
