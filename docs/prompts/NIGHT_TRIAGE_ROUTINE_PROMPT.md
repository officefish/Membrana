### Что построить (продуктовое описание)

Три шага брифа, шаг ≈ один артефакт:

1. **Шаг 1 — верификация канала.** Убедиться, что Claude Code Routines (research
   preview) доступен на аккаунте, репозиторий `officefish/Membrana` подключается,
   клон видит `docs/tasks/registry.json`. Разовый ручной ран с тривиальным промптом
   («перечисли активные задачи из registry.json»). Замерить расход квоты одного рана.
   **Стоп-условие:** preview недоступен или один ран съедает заметную долю пиковой
   полосы → пилот замораживается, фиксация в INSIGHT, остаёмся на локальном `night:open`.
2. **Шаг 2 — промпт рутины + секрет-гейт.** Довести промпт (Приложение A брифа +
   поправки консилиума) до самодостаточности: ран не имеет памяти и человека в контуре.
   Промпт версионируется в репо (`docs/day-sprint/night-triage-routine-2026-07-10/ROUTINE_PROMPT.md`),
   правки — только через PR. **Блокирующий gate:** секрет-сканер `scripts/night-triage-secret-scan.mjs`
   (владелец Dynin) — паттерны распространённых форматов ключей + запрет непустых
   значений `token`/`secret`/`key` в `registry.json` и путях чтения рутины;
   non-zero exit останавливает пилот.
3. **Шаг 3 — боевой ран и приёмка.** Ручной запуск полного промпта → ревью отчёта →
   включение расписания (еженощно 02:00–04:00) только после критериев приёмки.

### Решения консилиума (обязательны, не пересматривать)

- **stale-порог:** N=14 — флаг первого рана, НЕ канон; отчёт дополнительно выводит
  dwell-time (дней без движения) каждой активной задачи; канонический N — после
  анализа распределения.
- **PR-стратегия:** daily draft PR (один срез = одна точка ревью); rolling PR —
  кандидат фазы 2.
- **Отчёты:** на пилоте копятся в `docs/reports/night-triage/` без архивации;
  после приёмки — `archive:daily-day` с окном хранения активных.
- **`issues:audit`:** эталон приёмки, НЕ постоянный CI-гейт.
- **Детерминизм:** два прогона на идентичном срезе дают совпадающие таблицы;
  сортировка строк по id.
- **Формат отчёта:** сводка первой строкой; таблицы id · категория · ссылка ·
  действие (close/relink/re-scope) · уверенность (high/low); для stale — колонка
  «дней без движения»; визуальное отделение high/low.
- **Метрика точности:** TP/FP по каждой категории отдельно, не один агрегированный %.

- Рутина НЕ закрывает issues, НЕ правит `registry.json`, НЕ пишет в Linear/Slack.
- Push рутины только в `claude/`-ветки; diff PR = ровно один новый файл.
- Изменения младше 24 ч не классифицировать (переходные состояния).
- Никакой CI-обвязки вокруг research preview.

### Definition of Done

- [ ] Шаг 1: успешный ручной ран на подключённом `officefish/Membrana`; расход квоты записан.
- [ ] Шаг 2: секрет-сканер реализован, non-zero exit блокирует; промпт рутины версионирован.
- [ ] Шаг 3: отчёт по пути `docs/reports/night-triage/NIGHT_TRIAGE_<YYYY-MM-DD>.md`; draft PR; diff = один новый файл.
- [ ] Детерминизм: два прогона на идентичном срезе совпадают (сортировка по id).
- [ ] Полнота: находки рутины ⊇ находки `yarn issues:audit` на одном срезе.
- [ ] Точность: ложные ghost/orphan ≤ 20%; TP/FP по категориям зафиксированы.
- [ ] Отчёт содержит dwell-time каждой активной задачи.
- [ ] INSIGHT со сведёнными замерами закоммичен.
- [ ] Стоп-условие отработано честно (если сработало — заморозка + INSIGHT, это тоже валидное закрытие).
- [ ] LGTM Teamlead.

## Заметки для человека-постановщика

1. GitHub Issue (`wish`) + ссылка на этот файл.
2. Запись в `docs/tasks/registry.json` (`status: active`) — сделано при регистрации.
3. После merge: отчёт в Issue → `yarn task:archive night-triage-routine-pilot --notes "…"`.
4. Шаг 1 требует действий владельца аккаунта в claude.ai/code (research preview) —
   агент готовит всё остальное и передаёт чеклист.

### Проверка после PR

```bash
yarn night-triage:secret-scan        # exit 0 на чистом registry
yarn issues:audit                    # эталонный срез для сверки полноты
```
