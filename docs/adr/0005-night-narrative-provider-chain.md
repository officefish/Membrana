# ADR 0005 — нарратив ночных агентов: цепочка провайдеров Claude → DeepSeek(direct) → graceful-пропуск

> **Статус:** ACCEPTED · 2026-07-13 (решение владельца по итогам разбора инференс-провайдеров; спринт `night-narrative-deepseek-fallback`, Issue #424)

## Контекст

LLM-нарратив ночных агентов (night-triage «Обзор», night-hunt weekly-отчёты) — единственная
недетерминированная часть контуров: детерминированное ядро (таблицы триажа, PR-механика)
живёт независимо и нарративом только украшается (graceful: ошибка/нет ключа → отчёт без раздела).

История канала:
1. **OpenRouter** (исходный NT4/night-hunt): аккаунт владельца получает **403 «security
   policy»** (настройка аккаунта, не кода), плюс `OpenRouterService` на голом `fetch`
   не видит прокси. Нарратив триажа пропускался с деплоя NT5.
2. **ClaudeService** (свап #390 для night-hunt, позже и триаж): работает через
   NL/media-прокси — жив, но зависит от прокси-инфраструктуры и гео-обстановки
   (уроки: блок ТСПУ был IP-specific, LE rate-limit, OpenRouter-403).

Одноканальный LLM — единая точка отказа для наблюдаемости. DeepSeek chat-completions
(OpenAI-совместимый) доступен с office (Москва) **напрямую, без прокси**, дёшев — идеальный
страхующий канал именно потому, что его путь отказа не пересекается с путём Claude.

## Решение

**Фиксированная в коде цепочка** (не конфиг): `ClaudeService` → `DeepSeekService` (direct)
→ graceful-пропуск.

- **Claude первым** — качество + уже живой канал; DeepSeek подхватывает только при
  ошибке/неконфигурированности Claude.
- **DeepSeek — прямой URL** (`https://api.deepseek.com/chat/completions`), НИКАКОЙ
  прокси-логики в модуле: ценность канала — независимость от прокси; если DeepSeek
  зафильтруют — это отдельное решение, не тихая деградация.
- **OpenRouter выведен из цепочки нарратива** (403). Возврат — отдельным решением после
  починки аккаунта владельцем; `OpenRouterService` не удаляется.
- **Метка провайдера обязательна**: раздел нарратива подписывается
  `сгенерировано: claude|deepseek` — оператор видит, какой канал работал; смена метки —
  ранний сигнал деградации первичного канала.
- Детерминированное ядро НЕ зависит от цепочки: оба канала упали → отчёт без раздела,
  как сейчас.

## Границы (Структурщик)

- Модуль `packages/background-office/src/modules/deepseek/` — зеркало `openrouter`
  (HTTP, конфиг, ошибки); НЕ знает про триаж/hunt.
- Цепочка живёт в потребителях (`night-triage.service`, `night-hunt.service`) — они не
  знают деталей HTTP.
- **Без** общего «LLM-роутер»-пакета/абстракции провайдера: два потребителя × три звена —
  YAGNI; пересмотреть, если потребителей/провайдеров станет больше.
- Конфиг: `DEEPSEEK_API_KEY`, `DEEPSEEK_MODEL` (default `deepseek-chat`) — опциональны
  в `env.schema`.

## Следствия

- Нарратив возвращается в отчёты без ожидания починки OpenRouter-аккаунта.
- Плата: второй LLM-аккаунт/ключ в проде office (провизионировать в `/etc/membrana/office.env`).
- Тест-матрица цепочки (Математик): claude ok → deepseek не вызывается; claude fail +
  deepseek ok → нарратив с меткой deepseek; оба fail → отчёт без раздела.
