# ADR-0006 — Прогон бенчмарка исполняет калиброванный пресет; владелец истины — JSON калибратора

> **Статус:** ACCEPTED · 2026-07-18 (LGTM владельца)
> Решения Р1–Р3 действуют. Замена — только через новый ADR со статусом SUPERSEDED-by-ADR-MMMM.

## Контекст

Заседание `dads-integration` установило DAG вопросов интеграции внешнего датасета DADS.
Корнем графа (слой нулевой, in-degree 0) оказался вопрос: **какую конфигурацию детекторов
исполняет прогон бенчмарка** — потому что базовая линия, приор тест-сплита и цена развилки
ресэмплинга производны от него.

Вопрос дважды выносился на консилиум (`M1` прогон 1 и прогон 2) и дважды был отклонён
аудитом: оба вердикта спроектированы поверх непроверенных посылок — прогон 1 ввёл контракт
поверх несуществующих сущностей, прогон 2 инвертировал фактическое направление потока
данных и описал механизм чтения, которого в коде нет.

**Причина повторного срыва — не исполнение, а форма.** У вопроса нет развилки: направление
потока и место дефекта однозначно читаются в коде. Консилиум предназначен для спора; там,
где спора нет, он изобретает сущности. Поэтому узел корня выведен из очереди заседаний с
носителем ADR — по прецеденту, созданному в том же заседании на M0, где два кандидата были
легитимно переданы другим носителям (реестр задач; обязательство отчёта).

Заседание остаётся в силе: ратифицированный DAG не пересматривается, следующим идёт узел
состава DADS и тихого fallback.

Продуктовая граница (поправка владельца, 18.07): реализация идёт **плагином модуля
микрофон**; борд (`modules/device-board`, сценарии UserCase) в scope не входит.

**Фактическое направление потока:** корпус → калибратор → JSON → *ручная транскрипция* →
TS → плагины. Ручная транскрипция и есть дефект: результат калибровки попадает в рантайм
только через человека.

## Решение

### Р1 — Прогон бенчмарка исполняет калиброванный пресет, а не дефолты пакетов

Цифра прогона обязана означать поведение боевой поверхности. Боевая поверхность — плагины
модуля микрофон, и они исполняют пресет (`analyzeStreamDetectors.ts:28,79`). Прогон на
дефолтах меряет конфигурацию, которой нет ни на одной поверхности.

**Границы:** меняется только конфигурация, подаваемая в `analyzeSample` в
`scripts/benchmark-detectors.mjs`. Пороги детекторов, шаблоны, отбор сэмплов и метрики не
трогаются. Отладочный прогон на дефолтах остаётся возможен явным параметром, но не имеет
права называться цифрой боевой поверхности.

### Р2 — Владелец истины — JSON калибратора; TS-копия производится из него

Направление генерации: **JSON → TS**, не наоборот. Обоснование грунтовано кодом: JSON есть
выход измерения (`calibrate-detectors.mjs:277-288`), TS-файл сам объявляет себя производным
(`calibration-preset.ts:4-5`), а состав полей идентичен (`aggregation`,
`sampleConfidenceThreshold`) — вычисляемых полей нет, потери при генерации нет.

Обратное направление (TS как владелец) отвергается как **вредное**: оно узаконивает ручную
транскрипцию и лишает рекалибровку пути в код. При автогенерации JSON из TS возникают два
писателя одного файла — калибратор и генератор, — и первый же прогон рекалибровки будет
откачен генератором, а тест на расхождение объявит легитимный результат калибровки дрейфом.

**Границы:** генерация — один скрипт в `scripts/`, пишущий один TS-файл. Новых пакетов,
контрактов, фасадов и переименований носителя не вводится. Имя `CALIBRATED_SAMPLE_OPTIONS`
сохраняется.

### Р3 — Совпадение с боевой поверхностью подтверждается тестом и паспортом прогона

Два дешёвых средства вместо контрактной машинерии:

1. **Тест равенства** — сгенерированная TS-копия совпадает с JSON. Расхождение = красный
   тест, а не тихий дрейф. Живёт рядом с носителем.
2. **Паспорт прогона** — отчёт бенчмарка печатает, какую конфигурацию исполнил (пресет или
   дефолты). Цифра без паспорта неинтерпретируема; сегодняшняя двусмысленность канона
   (автоген против заметки о stage-gate) выросла ровно из её отсутствия.

**Границы:** хэширования конфигурации, CI-guard с падением прогона, бейджей и требований к
вёрстке отчёта **не вводится** — это уровень нового контракта, он не обоснован, пока не
сделан ни один прогон DADS.

## Definition of Done (для будущей реализации)

- [ ] `scripts/benchmark-detectors.mjs` подаёт в `analyzeSample` калиброванный пресет; путь
      получения конфигурации — один, без дублирования значений в скрипте.
- [ ] Отладочный прогон на дефолтах доступен явным параметром и помечен в отчёте.
- [ ] TS-копия `calibration-preset.ts` генерируется из JSON скриптом; файл несёт пометку
      «сгенерировано, не редактировать руками» с указанием источника.
- [ ] Тест: сгенерированная копия равна JSON; тест краснеет при искусственном сдвиге
      значения в JSON.
- [ ] Отчёт прогона печатает исполненную конфигурацию (паспорт).
- [ ] Тесты зелёные на затронутых пакетах; границы пакетов не нарушены; борд не затронут.

## Out of scope / открытые задачи

- **Развилка ресэмплинга** (16 kHz против 48 kHz, три варианта с разной ценой) — узел слоя
  L3 заседания, назначен на M5. Ни в конфигурацию, ни в паспорт прогона не вносится.
- **Базовая линия** — какое число канона истинно (автоген 67.5/90.0 против заметки
  85.5/88.3); узел слоя L1. Настоящий ADR даёт для него предпосылку (паспорт конфигурации),
  но вопроса не решает.
- **Состав DADS и тихий fallback отбора** — узлы слоя L0, следующий этап заседания.
- **Перенос переобучения:** пресет калиброван grid-search на train-части нашего корпуса —
  насколько он правомерен на внешнем корпусе, решается после первого прогона DADS.
- **Коллизия нумерации ADR:** в реестре два файла с номером 0005 — отдельная задача.
