# ADR-0009 — Делегированное исполнение ночного спринта фоновому субагенту

> **Статус:** ACCEPTED · 2026-07-15
> Директива владельца 2026-07-15 («канонизировать»). Референс-реализация той же
> ночи: ночной спринт `graphify-public-graph` исполнен делегированным фоновым
> субагентом в изолированном worktree.

## Контекст

Формат Night Build (`NIGHT_SPRINT_REGULATION.md`) описывает жизненный цикл
`ritual:evening → night:open → agent NB0…NBn → checkpoint → close → morning merge`,
но **не уточняет, КТО и КАК** исполняет «agent NB0…NBn». По умолчанию это выполнял
координатор в основной сессии. Ночной спринт по определению **автономный и
безнадзорный**, поэтому его исполнение естественно **делегировать фоновому
субагенту**, освободив координатора и совпав с духом формата.

## Наблюдаемое состояние (подтверждено, @2026-07-15)

| Факт | Где |
|------|-----|
| Lifecycle «agent NB0…NBn» не уточняет исполнителя | `docs/NIGHT_SPRINT_REGULATION.md` §Жизненный цикл |
| Есть инструмент фонового субагента с worktree-изоляцией | Agent tool (`run_in_background`, `isolation: worktree`) |
| Референс: `graphify-public-graph` открыт `night:open`, исполнение делегировано фоновому субагенту в worktree `Membrana-night-graphify` | сессия 2026-07-15 |
| Прецедент делегирования подзадач субагентам (research/impl/adversarial-verify) | коворк #487, PC-2 |

## Решение

### Р1 — Делегирование по умолчанию, координатор ≠ нянька
После `night:open` координатор **делегирует NB0…NBn фоновому субагенту**
(`run_in_background`, worktree-изоляция), который исполняет фазы автономно,
чекпойнтит и пишет HANDOFF. Координатор не досматривает ночь — получает
уведомление о завершении. **Границы:** механика `night:open`/`checkpoint`/`close`
не меняется; это норма исполнения, не новый скрипт.

### Р2 — Промпт эпика = контракт; жёсткие инварианты в нём
Качество ночи = качество спеки. Субагент получает: список фаз NB с DoD, **жёсткие
инварианты** (что запрещено), вердикт консилиума/ADR эпика. Расплывчатый эпик →
расплывчатая ночь. Инварианты приватности/безопасности — в промпте явно, со СТОП-правилом.

### Р3 — Human-in-loop шаги остаются владельцу
Всё, что требует браузера/глаза/суждения (визуальная оценка, живой смоук с
оборудованием), субагент **НЕ фабрикует**, а откладывает на утро/владельца.
Референс: оценку `graph.html` вынесли владельцу — субагент отдал артефакт, не «вау».

### Р4 — Утренняя верификация HANDOFF, не слепой мёрж
Координатор утром **верифицирует** результат ночи. Если ночь произвела код —
**адверсариальная верификация** (как PC-2), а не доверие на слово. Честный негатив
(«не встало / утечка») — валидный исход, документируется, не переоткрывается.

### Р5 — Масштаб: один субагент = один ограниченный спринт
Один делегат покрывает ограниченный ночной эпик в пределах своего контекста. Эпик
шире одного контекста → **несколько делегатов или Workflow**, а не растягивать
одного до деградации.

### Р6 — Изоляция обязательна
Ночной субагент работает в **отдельном worktree** (ветка `night/<id>-<date>`),
чтобы не коллизить с параллельными сессиями ([[feedback_parallel_session_worktrees]]).

### Р7 — Scoped auto-yes, а не глобальный bypass
Чтобы владелец реально отошёл от компьютера, ночной агент запускается с
**авто-подтверждением на командной поверхности ночи** (`Bash(git *)`, `Bash(yarn *)`,
Edit/Write в репо, `night:*`/`turbo`/`docs:lint`) — через `permissions.allow`.
**НЕ** глобальный `--dangerously-skip-permissions`: инварианты Р2/Р3 держит
**механический deny-лист**, а не добросовестность агента. В `permissions.deny`:
прод-деплой (явные префиксы — ведущие wildcard в Claude Code ненадёжны),
`git push --force*`, `git reset --hard`, SSH, `task:close-github`, правки
`packages/core/**`. Так «отойти от компьютера» = auto-yes на рутине при жёстком
запрете опасного.

> **Реализация (2026-07-16):** скилл [`membrana-always-yes`](../../.cursor/skills/membrana-always-yes/SKILL.md)
> + `yarn always-yes:on|off|status` (`scripts/always-yes.mjs`). Профиль вливается в
> локальный `.claude/settings.local.json`; `deny` > `allow` во всех режимах Claude
> Code; делегированный ночной субагент наследует профиль координатора.
