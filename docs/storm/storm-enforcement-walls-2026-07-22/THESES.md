# Шторм: стены вместо слов — как принуждать регламент, когда у агентов руки на файловой системе

> Открыт 2026-07-22 по слову владельца («Давай обсудим это через шторм»).
> Вход: вопрос владельца «как нам это регламентировать? У агентов есть доступ к
> файловой системе, они просто берут и пользуются командами — и всё»; вещдок дня —
> сессия пушила код/каноны из-под процедурной ветки morning-ritual без ревью;
> заготовки-зубы #924 (ревью-вердикт перед мерджем) и #925 (процедурная ветка —
> не транспорт); лестница принуждения из 5 слоёв (конспект в обоих Issues).
> Секретарь: Ангелина (инлайн — фоновой пока нет; веду только конспект, кодекс #922).

## Подтверждено владельцем — тезисы

**T1. Это не архитектура принуждения — это архитектура РОУТИНГА PR-ов и
коммитов.** Цель — избавить ровно от двух вещей: (1) **безответственных
коммитов** (изменение без лица) и (2) **драки за main** (гонка параллельных
сессий; вещдоки 21–22.07: package.json ×3 конфликта, реестры ×2, «шесть PR
обгоняют друг друга»). Не стены против нарушителей, а маршрутизация потоков
изменений (владелец, шторм 22.07).

**T2. Сессия = одна ветка, рождённая на старте: id сессии + дата + модель.**
Никаких других веток сессия не заводит; ветка — её единственный дом на весь
жизненный цикл (владелец, шторм 22.07).

**T3. Сессии закрыт инструмент push/PR — полностью.** Она умеет только
подтягивать изменения с сервера (pull). На все свои изменения она подаёт
**заявку на сервер office** (владелец, шторм 22.07).

**T4. Office — роутер изменений: ревьюит код заявки, оформляет его под
конкретной веткой и ставит в очередь.** Ревью, оформление и очередь мерджа —
серверные обязанности office, не сессии. Драка за main исчезает по построению
(очередь одна), безответственный коммит невозможен (у каждой заявки есть
сессия-податель) (владелец, шторм 22.07).

**T5. След в Linear: при закрытии Issue в комментарии пишется, под какой
сессией на самом деле пришёл этот код и когда** — атрибуция исполнителя
доезжает до трекера, а не теряется в одинаковых Co-Authored-By
(владелец, шторм 22.07).

**T6. Ревью заявки — всегда две ступени: сначала исполнитель, потом Тимлид.**
Исполнитель — лицо по характеру изменений (каскад назначения уже есть:
resolveReviewLead T3); Тимлид — вторая ступень поверх (владелец, шторм 22.07).

**T7. Исполнитель не определён → исполнителя назначает Тимлид по характеру
изменений.** Незакрытого «ничьего» ревью не существует (владелец, шторм 22.07).

**T8. Неоднородная заявка не ревьюится — Тимлид отправляет её назад с
требованием разбить на Issues.** Однородность (один скоуп/одно лицо на заявку) —
входной критерий очереди; смешанный дифф — возврат, не торг (владелец,
шторм 22.07).

**T9. Контракт заявки: код едет патчем в API office — `{дифф ветки сессии
против базы, id сессии, Issue/карточка, декларация скоупа}`; office прикладывает
патч на ЧИСТОМ клоне — рабочему дереву сессии не верит** (владелец, шторм 22.07:
«да»).

**T10. Транзиентные таймауты office — АВАРИЯ, требующая немедленной починки,
не фоновое неудобство.** Server-first требование (стратегия проекта) не
выполняется: сервис, на который канон уже опирается (ласточка: «office
недоступен — повторить»), а будущий роутинг обопрётся тотально (T3/T4),
нестабилен. Приоритет — аварийный, до строительства роутера (владелец,
шторм 22.07).

**T11. Роутинг и ревью — разные процессы (владелец: «я их смешал, но согласен —
это разные вещи»).** **Роутингом и первой блокировкой занимается Ожегов**:
пакет-заявка, не удовлетворяющий по СТРУКТУРЕ (однородность, скоуп, оформление),
блокируется им до всякого ревью. **Второй отказ — по уже существующим правилам
ревью** (исполнитель по характеру изменений → Тимлид, T6). Следствие: office —
исполняющая механика без собственных решений; каждое решение в конвейере
именное (Ожегов — структура, исполнитель+Тимлид — содержание) — «безответственного
роутера» не существует (владелец, шторм 22.07).

**T12 (развилка, слово владельца: «а»). Конспект едет входом в ЗАСЕДАНИЕ
«Роутинг изменений»** — конвергенция в спеку office-роутера. **Авария office
(T10) — отдельным аварийным Issue немедленно**, не дожидаясь заседания
(владелец, шторм 22.07).

## Осталось нерешённым (в заседание)

1. Две оси атрибуции: ветка/Linear = сессия-исполнитель, карточка/фрейм =
   персона-ответственность — разведение не подтверждено явно.
2. Деградация при падении office: заявки копятся локально и доезжают — или стоп.
3. Переходный мост до роутера: судьба зубов #924/#925 (временные стены или
   поглощаются эпиком роутера).

## Журнал вдохов

- Вдох 1 (d=31): T1 — рефрейм: архитектура роутинга PR/коммитов; две цели — без безответственных коммитов и без драки за main.
- Вдох 2 (d=11): T2–T5 — сессия=одна ветка (id+дата+модель), push/PR закрыты (только pull + заявка), office ревьюит/оформляет/ставит в очередь, атрибуция сессии в Linear.
- Вдох 3 (d=5): T6–T8 — две ступени ревью (исполнитель → Тимлид), ничьего ревью нет (Тимлид назначает по характеру), неоднородная заявка → возврат с разбиением на Issues.
- Вдох 4 (d=3): T9 — контракт заявки: патч в API office, приложение на чистом клоне, дереву сессии не верим.
- Вдох 5 (d=2): T10 — таймауты office = авария (server-first нарушен), немедленная починка до роутера.
- Вдох 6 (d=3): T11 — роутинг ≠ ревью: структурная блокировка — Ожегов, содержательный отказ — исполнитель→Тимлид; office — исполняющая механика без своих решений.
- Вдох 7 (d=1): T12 — развилка «заседание Роутинг изменений» + аварийное Issue по office немедленно; шторм закрыт по развилке (S-S2).

> **Поправка развилки (владелец, 22.07, после закрытия):** задача слишком радикальная и сырая для внедрения — конспект едет в ИНСАЙТ (не в заседание немедленно). Заседание «Роутинг изменений» — когда инсайт созреет.
