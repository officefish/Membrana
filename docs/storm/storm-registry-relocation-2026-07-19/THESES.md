## Объявлено владельцем — тезисы, не кандидаты

**T1. Linear — источник истины напрямую.** Не посредник, не зеркало. Реестр задач
переезжает **полностью**. Выбор сделан осознанно: Linear позволяет каждому виртуальному
программисту команды действовать там как **отдельная личность** (поправка владельца
19.07 — председатель ошибочно утверждал обратное, не проверив).

**T2. Реестр двухконтурный:** горячий быстрый + холодный медленный архивный.

**T3. Старые задачи должны быть правильно заархивированы** — возможно через MongoDB.

**T4. GitHub из стека не выключается.** В части задач он становится
**блокчейноподобной бекап-системой для графа правды.**

**T5. Форму холодного хранилища выбирает КОМАНДА, не владелец.** Аргументы были и в
пользу Postgres; владелец допускает, что мог их не учесть. MongoDB из T3 — кандидат,
а не назначение. Решение — предмет консилиума, с явным пересмотром лестницы из
`insight-task-archive-storage`.

**T6. Реестр вырезается из git НЕ полностью — карточки самих задач остаются в git**
(владелец, 19.07). Уточнение к T4: git не «бекап» реестра, а носитель другого слоя.

**Разрез проходит по оси «движение против содержания»:**

| Слой | Носитель | Что там живёт |
|---|---|---|
| **Движение** | Linear | состояние, переходы, назначение, приёмка, отзыв, переоткрытие, эскалация |
| **Содержание** | git | карточка задачи как документ: постановка, промпт, решения, обоснование |
| **Холодный архив** | выбирает команда (T5) | закрытое, вышедшее из горячего оборота |

Это не «источник и бекап», а два разных предмета. Linear отвечает на вопрос
**где задача сейчас и кто с ней что сделал**; git — **что это за задача и почему
решено так**. Хеш-цепочка git защищает содержание, а не состояние.

**Следствие для N1 (детерминизм):** гейты расслаиваются. Читающие содержание работают
офлайн и детерминированно, как сегодня. Сеть нужна только тем, кто судит о движении.
Вердикт M3 при таком разрезе, вероятно, выживает — но проверить это обязана комната,
а не председатель.

**Следствие для вердикта M2 (носители следа):** артефакт закрытия расщепляется.
Решение о приёмке — содержание (git), событие приёмки с актором и временем — движение
(Linear). Кто из двух есть носитель следа, вердикт M2 не различал: тогда носитель был
один файл. Это **не опровержение M2, а вопрос к его применению** на новом стеке.

**T7. Задач без Issue не бывает.** Задача без Issue **незаконна**, браться за неё
нельзя. Issue — удостоверение личности задачи, а не поле в карточке.

**T8. Незаконно сделанное разруливается только через PR.** Инвариант не уничтожает
работу: PR — путь легализации. Гейт с зубами, но без выбрасывания сделанного.

**Замер 19.07 — норма противоречит сегодняшнему состоянию на две трети:**

| Показатель | Значение |
|---|---|
| Активных карточек | 219 |
| Без `githubIssue` | **151 = 69%** |
| Первые в списке | `meeting-team-execution-contour`, `team-accountability-metrics`, `generated-docs-quality-criteria`, `angelina-orchestrator-prompt` |

Четыре первых примера — **карточки, заведённые председателем сегодня, в этой сессии**.
То есть норма нарушалась не в далёком прошлом, а час назад, тем, кто её сейчас
записывает. Это не оправдание — это замер точности «69%»: цифра живая.

**Следствие:** включать T7 жёстко завтра нельзя — это ровно та несущая стена, которую
вердикт M2 запретил ломать на ходу. Форма включения у нас уже есть и проверена:
мягкое предупреждение → миграция → жёсткий отказ.

**T9. Удостоверение личности задачи — GitHub Issue** (владелец, 19.07). Именно её
отсутствие делает работу незаконной по T7.

**T10. Linear-issue и GitHub-issue НЕ тождественны.** Отношение — **один ко многим**:
одна Linear-issue может включать несколько GitHub-issue. Симметрия (один к одному) —
частный случай, а не правило.

**T11. Автогенерация Issue в Linear не зашита намертво** — контролируется отдельно.
Иногда удобна, но обязательной связкой не является. В зависимости от задачи Linear-issue
либо зеркало GitHub-issue, либо разруливается иначе.

**Разрез уточняется:**

| Единица | Носитель | Роль |
|---|---|---|
| GitHub-issue | git/GitHub | **единица законности** — паспорт работы, публична, переживает смену трекера |
| Linear-issue | Linear | **единица движения** — контейнер над одной или несколькими единицами законности |

**T12. Linear-issue не обязательно задача** — в ней может быть чеклист задач.
**GitHub-issue — всегда конкретная задача с конкретным исполнителем.**

**Это снимает вопрос об уровне.** Ответственность и метрики привязываются к
**GitHub-issue**, потому что только там исполнитель присутствует по определению.
У контейнера движения единственного исполнителя нет по природе — требовать его значит
сочинять. Вердикт M2 («ровно один ответственный на задачу») переезжает на новый стек
без правки смысла: `leadPersona` карточки становится исполнителем GitHub-issue.

**Но открывается другое — где наблюдается движение единицы ответственности.**
Если единица ответственности — GitHub-issue, а состояние и переходы живут в Linear, то
у **каждой** GitHub-issue обязано быть наблюдаемое состояние в Linear. Иначе движение
известно только для контейнера, а метрики по T12 считаются для его элементов — и мы
получим величины, у которых нет источника (ровно дефект V5, найденный сегодня).

Отсюда фактура, которую надо выяснить **проверкой, а не догадкой**: являются ли
элементы чеклиста Linear полноценными объектами с собственным состоянием и исполнителем,
или состояние есть только у контейнера. От этого зависит, нужна ли связка «одна
исполняемая задача — один объект в Linear» для всего, что берётся в работу.

**Прежний вопрос об уровне (снят T12), сохранён как след рассуждения:**
Вердикт M2 требует ровно одного ответственного «на задачу»; вердикт M3 считает время до
закрытия, незавершённую работу и пропускную способность «по карточке». Но при отношении
один-ко-многим «задача» перестаёт быть одним объектом: у контейнера движения и у единицы
законности разные жизненные циклы. Считать на обоих уровнях — двойной счёт; выбрать один
молча — тихая ложь в отчёте. Ни M2, ни M3 этого различения не знали: тогда карточка была
одна. **Это не опровержение вердиктов, а вопрос к их применению.**

**T13. Linear-issue без GitHub-issue незаконна** (владелец, 19.07). Обязана быть
подкреплена **хотя бы одной**. Обходного пути через новый стек нет: инвариант T7
действует в обе стороны — работа без паспорта незаконна, и контейнер движения без
паспорта тоже.

Владелец выбрал **строгий** вариант. Председатель предлагал мягче — планировать
свободно, гейт только на переходе в исполнение. Отвергнуто: плановый контейнер без
единой единицы законности не допускается вовсе.

**Следствие — связка обязательна по существованию, но не по механизму.** T11 говорит,
что автогенерация не зашита намертво; T13 требует, чтобы паспорт был. Значит при
заведении Linear-issue кто-то обязан завести и GitHub-issue — руками или автоматом, но
обязан. Противоречия нет: норма о существовании и норма о способе — разные нормы.
Автогенерация становится **удобством, снимающим ручную работу**, а не источником
законности.

**T14. Структура Linear-issue: одна центральная задача + возможные дополнительные.**

- центральная — **ровно одна и обязательна**;
- дополнительные — не всегда обязательны;
- центральная **законно закрывается при незакрытой дополнительной**.

**T15. Дискриминатор — НАПРАВЛЕНИЕ зависимости, не её наличие.** Если центральная явно
зависит от задачи, дополнительной та быть не может: это **цепочка двух обязательных
задач с гейтом**.

**T16. Изолированная задача обязательна по определению** — «зациклена на себе»
(владелец, 19.07). Отсутствие связей не делает задачу необязательной: у неё
самостоятельное основание, и ничто внешнее не может её отложить.

> **Исправление председателя.** В T15 было записано «необязательность определяется
> отсутствием зависимости» — **шире, чем дал владелец**. T16 это опровергает. Второе
> подряд чрезмерное обобщение чужого правила за две реплики (первое — «у контейнера нет
> исполнителя по природе», снято T14). Класс ошибки один: правило сформулировано за
> владельца в более общем виде, чем прозвучало.

**Правило целиком.** Пусть `C` — центральная задача контейнера, `X` — прочая:

| Отношение | Статус `X` | Почему |
|---|---|---|
| `C` зависит от `X` | **обязательна**, звено цепи с гейтом | блокирующая по определению (T15) |
| связей нет вовсе | **обязательна** | самостоятельное основание, зациклена на себе (T16) |
| `X` зависит от `C`, обратной зависимости нет | **может быть дополнительной** | основание производно: без неё `C` полна |

**Дополнительная — узкий класс:** только односторонне производная от центральной работа.
Всё остальное обязательно. Это согласуется с линией владельца по всему шторму: строгий
вариант там, где председатель предлагал мягкий.

**Следствие для формы контейнера.** Раз изолированное обязательно, контейнер со
свалкой несвязанных задач становится полностью блокирующим и не закроется, пока не
сделано всё. Значит **Linear-issue обязана быть связной**: центральная плюс то, что от
неё производно или её блокирует. Несвязанное — в свой контейнер, со своей центральной.

**Что T14 чинит в сказанном ранее.** Председатель утверждал: «у контейнера движения
единственного исполнителя нет по природе, требовать его значит сочинять». **Неверно** —
контейнер всегда несёт центральную задачу, а у неё исполнитель есть по T12. Значит
контейнер наследует и исполнителя, и момент закрытия от центральной. Уровень счёта
определён окончательно: ответственность и закрытие контейнера — **по центральной
задаче**; дополнительные считаются отдельно и закрытию не препятствуют.

**Риск обхода, который надо назвать.** Правило T15 проверяемо машиной **только если
зависимость объявлена**. Пометить блокирующую задачу дополнительной и не объявить
зависимость — и гейт молчит: формально одна центральная, формально дополнительная
необязательна. Это тот же класс, что **спрятанная посылка** в протоколе: чтобы понять,
что вход пропущен, надо понимать вывод. Машиной не ловится; работа человека и аудитора.

**Фактура к выяснению (не догадкой):** выражает ли Linear блокирующие отношения между
объектами так, чтобы цепочка из T15 была наблюдаема в API, — и на каком объекте живёт
состояние элемента чеклиста.

**Проверяемость:** инвариант ловится машиной с обеих сторон — GitHub-issue без задачи
и Linear-issue с нулём подкреплений одинаково видимы. Это редкий случай, когда норма
и её гейт совпадают по форме.

## Посылка, которая могла убить прежнее решение

Лестница инсайта отвергла Postgres как **избыточный**, и обоснование было конкретным:
он решает проблему конкурентной записи, «которой у последовательного вечернего
архивирования нет». Названный триггер перехода — **второй параллельный пишущий
процесс**.

**Переезд этот триггер, похоже, взводит.** Если Linear — источник истины, а события
приезжают вебхуками, то `background-office` становится писателем, работающим
**асинхронно и по событию**, а не вечерним пакетом. Рядом остаётся вечерняя
архивация. Это два писателя с непредсказуемым чередованием — ровно то условие, при
отсутствии которого Postgres был признан лишним.

Проверить это должна команда: посылка «нет конкурентной записи» была **фактом о нашей
архитектуре**, а не нормой. Архитектура меняется — факт может умереть вместе с ней.
Если умер, лестница пересматривается не потому, что владелец передумал, а потому что
исчезло основание.

## Напряжения — назвать до, а не после

**N1. Детерминизм гейтов против сети.** Вердикт M3, вынесенный за два часа до этого
разговора, требует: чистая функция, тот же вход → тот же выход бит-в-бит, **без сети**.
Если Linear — источник истины напрямую, надо явно сказать, **что читают гейты**:
материализованный снимок (и тогда снимок — часть контракта, а не кэш) или сеть (и тогда
вердикт M3 подлежит пересмотру, а не молчаливому обходу).

**N2. Инсайт про архив уже вынес решение — MongoDB его пересматривает.**
`insight-task-archive-storage` (28.06) зафиксировал **лестницу хранилищ** с правилом
«останавливаться на первой достаточной ступени»: `registry.json` (горячий) →
`archive.jsonl` append-only (холодный источник истины) → генерируемый индекс → SQLite →
Postgres **только** при выполнении названных триггеров (`database is locked` чаще раза в
неделю · второй параллельный писатель · объём в гигабайтах).

MongoDB в лестнице отсутствует. Это не запрет — это требование **явно пересмотреть**
решённое, а не переехать мимо него. Двухконтурность (T2) инсайт как раз подтверждает:
он и есть про горячее и холодное.

**T17. Назначение цепочки — НЕ защита от подлога, а защита от исчезновения**
(владелец, 19.07). Цепочка нужна, **чтобы задачи не пропадали в истории, а превращались
в законный технический долг**.

**N3 этим снимается, а не разрешается спором.** Инсайт выносил за скоуп
*tamper-evidence* с обоснованием «нет недоверенных сторон». Возражение мимо: **угроза
не злоумышленник, а забвение**. Модель угроз другая — значит и механизм другой и
дешевле: нужен **append-only без удаления плюс правило преемника** (незакрытое
порождает законного наследника-долг), а не криптографическая неподделываемость.

Отсюда GitHub годится по существу, а не по метафоре: Issue не исчезает, закрывается со
ссылкой, история коммитов только дописывается. «Блокчейноподобность» здесь — про
**непрерывность цепи преемства**, а не про защиту от подделки.

**Прецедент с замером уже есть.** `insight-ghost-task-closure-invariant`: закрытие одной
Issue делало «завершёнными» карточки-сёстры, чьи промпты не доделаны — общий
`githubIssue` **не является доказательством закрытия**. Инспекция 29.06: из 38
кандидатов безопасно архивировались 19, остальные 19 требовали решения или работы.
Это и есть «задачи пропали в истории», в цифрах.

**Новый инвариант растворяет этот класс в корне.** Ghost-closure возможен только при
**разделяемой** Issue на несколько задач. По T7+T12 каждая задача несёт свою
GitHub-issue с конкретным исполнителем — разделять нечего. Дефект исчезает не потому,
что мы построили против него защиту, а потому что исчезло условие его существования.

**T18. Граф правды к этому стеку не относится** — проектируется отдельно, есть
предложение делать его на Redis. В этом шторме вердиктов по нему не выносить.

> Единственное, что стоит проверить в том, отдельном проектировании: у Redis
> долговечность — настраиваемое свойство, а не данность. Для носителя владельческих
> фактов это первый вопрос, а не деталь развёртывания. Здесь — только пометка, не спор.

**N3 (снят T17). Тот же инсайт прямо вынес хеш-цепочку за скоуп** — «tamper-evidence хеш-цепочка
(нет недоверенных сторон в контуре)». T4 её возвращает. Значит изменилась посылка:
либо недоверенная сторона появилась, либо цепочка нужна не против подлога, а для чего-то
другого. **Назвать, для чего** — иначе построим защиту от несуществующей угрозы.

**N4. Что именно «блокчейноподобно» в GitHub.** Git уже хеш-цепочка по устройству —
это почти бесплатно. Но свойство держится **только на append-only**: мы же документы
переписываем и историю мёржим. Нужно решить, что в графе правды становится
неперезаписываемым, а что остаётся обычным документом.

**N5. Граф правды — источник или бекап?** T4 говорит «бекап-система для графа правды».
Сейчас граф живёт в `docs/truth/registry.json`, в репозитории. Вопрос: источник графа
остаётся в git (и тогда «бекап» — неточное слово, git и есть носитель), или граф тоже
уезжает, а git становится его неизменяемым журналом?

**N6. Агент без сети.** Гейты и ритуалы сегодня читают файл в рабочем дереве. После
переезда каждому агенту для проверки нужен либо снимок, либо доступ к Linear. Это
касается ночных прогонов и параллельных сессий.

**N7. Миграция.** 213 карточек в активном реестре плюс архив. Правильная архивация
старого (T3) — отдельная работа со своим критерием «правильно».
