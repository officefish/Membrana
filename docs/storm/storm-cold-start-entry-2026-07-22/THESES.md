## Подтверждено владельцем — тезисы

**T1. Тема шторма поднята уровнем выше: не «как починить утро», а как вообще
работать с процедурами.** Контейнер процедур появился, но сама сущность процедуры
не проработана до конца (владелец, шторм 22.07).

**T2. Определения слоя (владелец, шторм 22.07):**
- **Процедура** — цепочка конкретных вызываемых команд, выполняющая определённый
  сценарий; **сценарий** — высокоуровневое понятие.
- **Фрейм** — элемент сценария: определённый набор конкретных команд, который
  (1) **имеет своё ответственное лицо** и (2) **имеет версию** по принципу
  паттерна [`PINNED_SUBGRAPH_VERSIONING`](../../patterns/PINNED_SUBGRAPH_VERSIONING.md)
  (SHA-пины связного подграфа).

**T3. Применение к прецедентам: прецедент упаковывается в конкретный фрейм, и
рефакторинг заказывается ответственному лицу фрейма.** Сегодняшний утренний
прецедент (автостарт холодной сессии) при таком устройстве стал бы задачей
«отрефакторить фрейм входа» его ответственному лицу (владелец, шторм 22.07).

**T4. Провода к процедуре — часть процедуры; фрейм — этап подводки процедуры к
агенту, явно недостающий элемент.** Указатели-двери (CLAUDE.md, зеркала скиллов,
маршруты холодной сессии) — не «окружение», а этап сценария, у которого должен
быть взявший его человек/лицо. Именно этот этап третий день без хозяина
(владелец, шторм 22.07).

**T5. Смысл фрейма — адресный ремонт по прецеденту вместо инлайн-фикса.**
Имея фрейм, владелец по прецеденту находит ответственное лицо и просит починить
ЕГО фрейм. Без этого понятия агент, выдающий себя за Ангелину, при первой
возможности стремится пофиксить сам в ближайшей инлайн-сессии — ремонт без
адреса, очередной рецидив (владелец, шторм 22.07; живой пример — предложение
куратора «закрою прямо сейчас» в этой же сессии).

**T6. Сценарий полностью состоит из цепочки фреймов; процедура — буквально
очередь из фреймов.** Не «фрейм по мере боли», а полная декомпозиция
(владелец, шторм 22.07).

**T7. Родовой узор: у контейнера есть основной список/стек и поддерживающий его
сабграф; сабграфы версионируются по второму паттерну
([`PINNED_SUBGRAPH_VERSIONING`](../../patterns/PINNED_SUBGRAPH_VERSIONING.md)).
Именование по доменам — явно: в скриптах сабграф = КИТ, в процедурах сабграф =
ФРЕЙМ.** Это не одно и то же, но оба удовлетворяют рекомендации второго паттерна
(владелец, шторм 22.07). Снимает занозу питомца: фрейм — не новое слово для той
же кучи, а доменная инкарнация родового понятия «версионируемый сабграф
контейнера».

**T8. Гранулярность пина фрейма — отрезок, не документ.** Целые тексты в сабграф
не входят: документ шире процедуры (сотни инструкций, процедура — лишь фрагмент).
Но **конкретная строчка/абзац — под версией и под ответственным лицом — часть
фрейма**. Фрейм — буквально набор **разрозненных отрезков** (команды + отрезки
фундаментальных документов), складывающийся во фрагмент сценария; редакция
нескольких документов сводится к одному понятию «подводка проводов к процедуре»
и инкапсулируется во фрейм (владелец, шторм 22.07).

**T9. Уточнение к паттернам (владелец, шторм 22.07, по занозе питомца):
вчерашнее «контейнеры — вне области» остаётся в силе для САМОГО контейнера
(реестр/список — снимковая дисциплина, не пин). Оговорка снята так:
проектирование сабграфа ПОД контейнером — поддерживающего его — это НЕ оверкод;
в проекте уже есть живой пример, где сабграф поддерживает контейнер, и это
уместно.** Формулировку паттерна PINNED_SUBGRAPH_VERSIONING уточнить, чтобы
следующий агент не отказался пинить киты/фреймы со ссылкой на «вне области».

**T10. Лицо фрейма — частность, выводимая из структуры процедуры, не предмет
отдельного торга.** Для фрейма-подводки утра — **Ожегов** (владелец, шторм 22.07:
«это всегда будет понятно из структуры процедуры; я полагаю Ожегов»). Прецедент
22.07 (двери на старый сценарий) получает адресата: рефакторинг фрейма подводки —
заказ Ожегову, не инлайн-фикс первой попавшейся сессии.

**T11 (развилка, слово владельца: «а»). Конспект едет в СПРИНТ «фреймы»:**
`frames[]` в манифесте процедуры (id, лицо, пин), механика пина отрезков (K1),
упаковка прецедента 22.07 в первый фрейм «подводка утра» → ремонт дверей руками
Ожегова (владелец, шторм 22.07).

## Кандидаты (не подтверждены)

**K1 (Математик, не подтверждён).** Механика пина отрезка: SHA файла меняется от
любой правки соседних строк → пин отрезка = (файл, якорь/диапазон, хэш текста
отрезка); дрейф отрезка ≠ дрейф файла. Вопрос реализации для спринта, не шторма.

## Журнал вдохов

- Вдох 1 (d=24): тема поднята уровнем выше; T1–T3 — сущность процедуры не доработана; определения процедура/сценарий/фрейм (лицо + версия по подграфу); прецедент → рефакторинг фрейма ответственным лицом.
- Вдох 2 (d=4): T4–T5 — провода = часть процедуры; фрейм = этап подводки к агенту (недостающий элемент); смысл — адресный ремонт по прецеденту вместо инлайн-фикса.
- Вдох 3 (d=5): T6–T7 — процедура = очередь фреймов (полная декомпозиция); родовой узор «контейнер: список + версионируемый сабграф», кит (скрипты) и фрейм (процедуры) — доменные имена.
- Вдох 4 (d=2): T8 — пин отрезка, не документа (фрейм = набор разрозненных отрезков); T9 — оговорка снята: контейнер не пинится, сабграф под контейнером — не оверкод; K1 механика якоря отрезка.
- Вдох 5 (d=1): T10 — лицо фрейма выводится из структуры процедуры; подводка утра — Ожегов; прецедент 22.07 получил адресата.
- Закрытие (НЕ вдох, d=0 — механика честна): T11 — развилка «спринт фреймы» словом владельца (S-S2) при 5/7 вдохах; потолок не потребовался.
