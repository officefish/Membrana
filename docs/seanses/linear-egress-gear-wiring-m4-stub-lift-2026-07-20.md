[Teamlead]: Рамку фиксирую сразу, чтобы E6 не съел соседние комнаты. Наш вопрос ровно один: предикат снятия stub + правило переключения + пост-режим. Closure (К4b), секреты, топология egress, поля шапки (M3) — не наши, ограда явная. По «форма следует функции»: stub существует, потому что живого pull ещё нет — значит снятие обязано стоять на факте появления первого честного не-stub артефакта, а не на дате. Кандидат 3 (дата/«туннель поднят») отклоняю сразу — это нарушение нормы M0 «не дата, а факт».

[Математик]: Формализую, иначе «критерий» останется вкусовым. Пусть `S` — снимок движения, `pullOk(S)` — предикат из M3 (полная шапка ∧ producedBy=media-NL ∧ egressRegion=NL ∧ …). Событие снятия — это переход состояния флага `movementMode: deferred-egress → live-snapshot`. Критерий кандидата 1: `∃ S: pullOk(S)=true ∧ source(S)=media-NL ∧ ¬fixture(S) ∧ ¬manualHeader(S)`. Кандидат 2 добавляет `N≥3` успешных в разных сутках. Вопрос строго вероятностный: `n=1` не отличает удачу от воспроизводимости.

[Музыкант]: По касательной — аудио тут не задет напрямую, но ритм я узнаю, это как первый успешный рендер мастера на боевом тракте. Мой живой флаг из `ritual-inputs-echo`: три отражения одного снимка — это `n=1`, а не `n=3`. Но тут ловушка обратная: если мы требуем N≥3 «в разных сутках» до снятия stub, мы блокируем сам cutover ради статистики, которой у нас пока нет вообще (живой pull ещё не проведён). Один честный экспорт, который реально прошёл тракт, снимает stub с состояния «выдуманный» — а качество замеряем уже после, отдельной осью.

[Структурщик]: Начну с термина, иначе спор поплывёт. «Снятие stub» — это омоним двух лемм: (1) **снятие фикции** — флаг перестаёт быть заглушкой, потому что появился первый настоящий артефакт; (2) **гарантия стабильности** — режим доказал воспроизводимость. Это разные предикаты с разной стоимостью. Кандидат 1 закрывает лемму (1) за один честный `pullOk`. Кандидат 2 — про лемму (2), это уже SLA режима, а не снятие заглушки. Смешивать их в одном флаге — «синоним-слой», два смысла на одном имени `movementMode`.

[Верстальщик]: С точки зрения потребителя артефакта — а потребитель это агент, читающий `task:start`, и человек в PR — моя боль из `main-day-accuracy`: любой флаг врёт, если не показывает, на какой ревизии и по какому снимку он переключён. Если `movementMode: live-snapshot` появится без ссылки на конкретный `S` с `pullOk`, это **мёртвая дверь**: зелёное «live» там, где под ним ничего нет. Требую: сам факт снятия обязан нести provenance — id снимка, его хэш шапки, кто/чем переключил.

[Teamlead]: Принимаю разделение Структурщика — это ключ. Мы решаем лемму (1): снятие фикции. N≥3 (кандидат 2) — это SLA живого режима, отдельный вопрос, в E6 не тащим (иначе колонизируем будущую комнату про стабильность). Значит база — кандидат 1: **один** честный `pullOk` снимает stub. Но Верстальщик прав про provenance, и Музыкант прав, что автоматический silent-flip опасен. Форма: переключатель **явный**, не автомат.

[Математик]: Соглашусь, что N≥3 дисквалифицирован как условие снятия заглушки — он отвечает на другой вопрос (устойчивость), а не на «фикция кончилась». Но зафиксирую численную защиту: `pullOk(S)=true` обязан быть **воспроизводимым по входу**, иначе снятие стоит на шуме. То есть снятие требует не «где-то прошло», а сохранённого артефакта `S`, к которому флаг ссылается по id. `audit(S) = pullOk(S)` должно давать тот же ответ при перепроверке — детерминированно, из сохранённого файла, без сети.

[Музыкант]: Тогда я снимаю возражение — согласен на один `pullOk` как критерий снятия фикции, при условии что артефакт `S` сохранён и переигрываем. Это как первый честный take: он не обязан быть финальным мастером, но он обязан существовать на диске, чтобы к нему вернуться. Silent-flip запрещаю категорически — по опыту `code-review-honesty`: если флаг переключается сам по факту прогона, никто не заметит, что переключился на клиппинге.

[Структурщик]: Тогда финализирую контракт переключателя. Флаг `movementMode` живёт в каноне/конфиге как единственный источник истины (одна лемма — один носитель, урок `dads-integration-m1-preset-delivery`: два носителя молча расходятся). Переключение — явная запись владельцем или скриптом **после** проверки `pullOk(S)`, с обязательной ссылкой `snapshotRef: <id S>`. Автоматический флип из кода прогона — запрещён нормой. Это разрывает связь «прогон → мутация состояния режима».

[Верстальщик]: Принимаю, и добавлю несущую про наблюдаемость пост-режима. После снятия экран `task:start` обязан показывать `live-snapshot` **со ссылкой на актуальный снимок**; отсутствие ссылки = ошибка, не «пусто». По моей норме честности: индикатор режима без provenance не имеет права гореть зелёным. Незаконным становится: `task:start` печатает `deferred-egress` stub, когда режим объявлен `live` — это регресс к фикции.

[Teamlead]: Собираю пост-режим — что незаконно после снятия. Первое: `task:start` без ссылки на актуальный снимок, если движение заявлено `live` — незаконно (Верстальщик). Второе: возврат к печати stub `deferred-egress` при `movementMode: live-snapshot` без явного отката флага — незаконно. Что остаётся stub-допустимым до полного cutover: единицы, созданные ДО снятия, не переписываются задним числом — их stub легитимен исторически.

[Математик]: Уточню границу «stub-допустимого», иначе она размоется. Множество единиц `U` делится по времени относительно момента снятия `t₀`: `U_before = {u : created(u) < t₀}` — их stub легитимен, не трогаем. `U_after = {u : created(u) ≥ t₀}` — обязаны нести `snapshotRef`. Это чистый предикат по временной метке, проверяемый без интерпретации. Никакого «Date.now()» в самом аудите — сравнение сохранённых меток создания против сохранённого `t₀` переключения.

[Музыкант]: Поддержу разделение по `t₀` — это как версия плагина в проекте: старые треки сведены на старой версии, их не пересводишь ретроактивно, но новые обязаны быть на актуальной. Единственное добавлю из своего слуха: `t₀` должен быть записан там же, где флаг, одной записью — иначе будет как расхождение WAV/MP3, когда флаг переключили, а метку времени забыли, и граница `before/after` поплыла.

[Структурщик]: Согласен — `movementMode`, `snapshotRef` и `switchedAt` (=`t₀`) — это одна атомарная запись в каноне, одна словарная статья, не три разбросанных поля. Иначе получаем три носителя одной сущности. И зафиксирую строку в К4b без вердикта: **closure не снимает stub сам** — это соседняя комната, гейт закрытия иссью и снятие заглушки движения ортогональны. Наш флаг не переключается фактом закрытия задачи.

[Верстальщик]: Принимаю атомарную запись. С точки зрения потребителя это правильно: одна запись = одна честная шапка на экране (режим + снимок + когда переключён). Для К4b фиксирую видимую строку: индикатор closure и индикатор movement — два разных бейджа, не связаны; зелёный closure не красит movement в live. Иначе мёртвая дверь на стыке комнат.

[Teamlead]: Хорошо, сходимся. Итог: снятие stub = **один** честный `pullOk(S)` (кандидат 1), сохранённый и переигрываемый артефакт `S`; переключатель **явный** (владелец/скрипт), silent-flip запрещён; атомарная запись `{movementMode, snapshotRef, switchedAt}` в каноне; пост-режим запрещает stub при заявленном live и требует `snapshotRef` для единиц после `t₀`; closure stub не снимает. N≥3 — отдельный вопрос устойчивости, не наш. Все роли — подтверждаете?

[Музыкант]: Принимаю. Один честный take снимает фикцию, silent-flip запрещён, граница `before/after` записана атомарно.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Предикат снятия stub | `∃ S: pullOk(S)=true ∧ source(S)=media-NL ∧ ¬fixture(S) ∧ ¬manualHeader(S)` — **один** честный не-stub артефакт (кандидат 1). N≥3 отклонён как вопрос устойчивости, не снятия фикции |
| Условие на артефакт `S` | Сохранён и переигрываем: `audit(S)=pullOk(S)` детерминированно из файла, без сети |
| Кто/как переключает | **Явно**: владелец или скрипт **после** проверки `pullOk(S)`. Автоматический silent-flip из кода прогона — запрещён нормой |
| Носитель флага | Атомарная запись в каноне/конфиге: `{ movementMode: live-snapshot, snapshotRef: <id S>, switchedAt: t₀ }` — одна словарная статья, один носитель |
| Незаконно после снятия | (1) `task:start` печатает stub `deferred-egress` при объявленном `live`; (2) единицы `created ≥ t₀` без `snapshotRef`; (3) индикатор `live` без provenance |
| Stub-допустимо до/после | Единицы `created < t₀` сохраняют легитимный исторический stub — ретроактивно не переписываются |
| Строка в К4b | **Closure не снимает stub сам**: гейт закрытия иссью и `movementMode` ортогональны; зелёный closure не красит movement в live |

## Список посылок

- Норма владельца / M0: до провода — честные stub; снятие — по факту, не по дате. — **норма**
- M1b: `task:start` печатает stub `movement: deferred-egress` до боевого pull. — **факт**
- M3: `pullOk(S)` = полная шапка ∧ producedBy=media-NL ∧ egressRegion=NL ∧ … только от файла; на выход в эту комнату — «первый артефакт с pullOk не-stub путём». — **факт**
- Живой боевой pull с media-NL ещё не проведён. — **факт**
- Урок `dads-integration-m1-preset-delivery`: два носителя одной сущности молча расходятся со временем. — **норма**
- Норма честности (`code-review-honesty`, `main-day-accuracy`): индикатор без provenance не имеет права гореть зелёным. — **норма**

**Definition of Done (E6):**
- В каноне/конфиге определён флаг `movementMode` с допустимыми значениями `deferred-egress | live-snapshot` и атомарной записью `{ movementMode, snapshotRef, switchedAt }`.
- Предикат снятия реализован как проверяемая функция: снятие законно только при наличии сохранённого `S` с `pullOk(S)=true` (переигрываемо из файла).
- Переключение флага возможно только явным действием (владелец/скрипт) с обязательным `snapshotRef`; автоматический флип из кода прогона отсутствует (проверяется отсутствием мутации `movementMode` в путях прогона).
- После `switchedAt = t₀`: `task:start` для единиц `created ≥ t₀` требует `snapshotRef`; печать stub `deferred-egress` при `movementMode: live-snapshot` даёт ошибку.
- Единицы `created < t₀` не переписываются — исторический stub легитимен.
- Без деплоя живого pull в прод: критерий и переключатель работают на артефакте, живой pull — вне DoD этой комнаты.
