# Консилиум: обобщать ADR-контур или держать отдельный реестр правды

**Повестка:** Единственный вопрос заседания Q1 (фундамент эпика #576): свести контур принятых архитектурных решений (ADR) с остальными реестрами правды репозитория в единый обобщённый механизм — или держать ADR отдельным, самостоятельным реестром со своими правилами? От вердикта зависят форма записи, охлаждение, инструменты и подписчики (последующие заседания эпика).

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Математик → Структурщик → Верстальщик → Teamlead → Музыкант (циклически)

[Математик]: Начну с фактов, чтобы не спорить о словах. У нас есть N разных «реестров правды»: `docs/tasks/registry.json` (состояние задач), `docs/adr/*` (архитектурные решения), `importance.json` (флаги памяти), фронтматтеры протоколов `docs/seanses/*`. Вопрос сводится к одному предикату: **у этих множеств одинаковый жизненный цикл записи или разный?** Если жизненные циклы совпадают — обобщение экономит код. Если различаются — обобщение вводит ложную общность, которую придётся ветвить `if`-ами. Я утверждаю: у ADR цикл иной, и это проверяемо.

[Структурщик]: Начну с термина, иначе спор поплывёт. «Реестр правды» — это омоним. У нас минимум две разные леммы: (1) **состояние процесса** — `registry.json`, оно меняется много раз за жизнь одной задачи (scaffold → implemented → archived); (2) **принятое решение** — ADR, оно по определению **иммутабельно после принятия** (меняется только статусом `superseded`, но текст не переписывается). Сводить иммутабельный append-only лог с изменяемым состоянием в один механизм — это как в словаре слить статью «глагол» и статью «существительное» под одной леммой. Разные части речи.

[Верстальщик]: С точки зрения потребителя вердикта — а потребитель это агент утром и человек в PR — обобщённый реестр создаёт **мёртвую дверь на экране**. Если ADR и задачи в одном списке, читатель видит смешанный поток и не понимает, что здесь неотменяемое решение, а что — текущий статус, который завтра поменяется. Иерархии нет, глаз не цепляется. Мне для честности нужно, чтобы ADR визуально читался как «камень» (принято, не трогать), а задача — как «поток» (в работе). Это два разных типа карточек, не один.

[Teamlead]: Рамку фиксирую сразу, чтобы этот консилиум не повторил ошибку 17.07 — тогда рамка первого вопроса колонизировала все десять. Наш вопрос **узкий**: свести ADR с прочими реестрами в единый механизм — да или нет. Не «как устроить шестой контур», не «переписать registry.json». По конструктивизму: форма оправдана функцией. Если функция ADR (зафиксировать необратимое архитектурное решение и его контекст) отличается от функции реестра задач (отслеживать изменяемое состояние работы) — то и форма обязана быть разной. Слушаю аргументы против разделения, если они есть.

[Музыкант]: Поддержу аналогией из моего контура. ADR — это как мастер-запись трека: приняли, свели, зафиксировали — дальше только новая версия поверх, оригинал не перезаписываешь. Реестр задач — это микшерный пульт в процессе сведения: фейдеры двигаются постоянно. Слить мастер-архив с живым пультом в один интерфейс — рецепт того, что кто-то случайно подвинет фейдер на готовом мастере. Риск-флаг: обобщение повышает вероятность **порчи иммутабельного** через общий инструмент записи.

[Математик]: Формализую то, что сказал Музыкант, численно. Пусть `mutations(record)` — число легальных изменений записи за её жизнь. Для задачи `mutations ≈ 3–5` (переходы статуса). Для ADR `mutations ∈ {0, 1}` — принята, максимум однажды помечена `superseded`. Дисперсия жизненного цикла различается на порядок. Обобщённый механизм пришлось бы параметризовать политикой мутабельности — а это и есть скрытый `if (isADR) deny-edit`. Общность, которая внутри немедленно ветвится по типу, — это ложная общность. Держу позицию: раздельно.

[Структурщик]: Соглашусь с Математиком и уточню архитектурно. Обобщение имеет смысл, когда несколько потребителей переиспользуют **одну инфраструктуру** — ровно критерий выделения foundation-сервиса из `SERVICES.md`. Но здесь общее только поверхностное: «это записи в `docs/`». Индексация для RAG, git-история, PR-ревью — уже общие механизмы платформы, они работают для **любого** markdown без всякого «единого реестра». То есть выгода обобщения = ноль (инфра уже общая), а цена = связывание двух разных жизненных циклов. Отрицательный баланс.

[Верстальщик]: Добавлю про честность отображения, раз инфра и так общая. Если завтра кто-то захочет «единую ленту решений» — это **вид (view)**, а не хранилище. Лента может собирать ADR + вердикты консилиумов + ключевые задачи в одну читаемую хронику, гейтя типы бейджами (`badge-warning` «решение», `badge-ghost` «статус»). Но это презентационный слой поверх раздельных источников, а не слияние источников. Смешивать хранение и представление — та же ошибка, что бизнес-логика в JSX.

[Teamlead]: Вот это ключевое разграничение, фиксирую его как несущее. **Хранение — раздельное; представление — можно агрегировать.** Это снимает соблазн «обобщить». Никто не теряет сводную картину: если понадобится общий обзор, это read-only view, а не общий writer. По `ARCHITECTURE.md` — один writer на один тип истины, это и есть слабая связанность на уровне данных. Склоняюсь к вердикту «нет, ADR держим отдельным реестром». Есть возражения по существу до того, как зафиксирую?

[Математик]: Одно уточнение к вердикту, чтобы он был проверяемым, а не декларативным. «Отдельный реестр» должен иметь **измеримый признак иммутабельности**: у ADR есть `status ∈ {proposed, accepted, superseded}` и `superseded-by`, и предикат `isEditable(adr) = (status === proposed)`. Текст принятого ADR неизменен — это гард, а не пожелание. Тогда «отдельность» — не слово, а свойство, которое можно проверить линтером в CI.

[Структурщик]: Поддержу гард и добавлю границу пакета для чистоты. ADR-контур — это `docs/adr/` с собственным индексом (`docs/adr/README.md` или `index`), собственным шаблоном (`NNNN-title.md` с фронтматтером `{id, status, date, supersedes?, superseded-by?}`) и собственным линтером формата. Он **не импортирует** схему `registry.json` и не импортируется ею. Два раздельных словаря, каждый со своей статьёй. Пересечение только на уровне ссылок: ADR может ссылаться на задачу по id, задача — на ADR, но это гиперссылка, не общая таблица.

[Верстальщик]: Со своей стороны закреплю потребительский контракт, чтобы вердикт не завис в воздухе. Раз ADR отдельный — у него отдельная карточка в любом будущем UI: заголовок решения, статус-бейдж (`accepted` = `badge-info`, `superseded` = `badge-ghost` с зачёркиванием), дата, ссылки на контекст. Читатель за 200 мс отличает «камень» от «потока». Мёртвых дверей нет: `superseded` не кликается как активное, а ведёт на заменивший ADR.

[Teamlead]: Принимаю. Резюмирую форму решения для протокола и для зависимых заседаний эпика. Вердикт: **держать ADR отдельным реестром**, не обобщать с прочими контурами правды. Обоснование — разный жизненный цикл (иммутабельный append-only vs изменяемое состояние), нулевая выгода обобщения (RAG/git/PR уже общие для всякого markdown), отрицательная цена (связывание двух циклов). Агрегация допустима только как read-only view, не как общий writer.

[Музыкант]: Принимаю. С аудио-стороны один практический хвост для последующих заседаний: раз ADR иммутабельны, «охлаждение» (следующий вопрос эпика) для них тривиально — принятый ADR не остывает, он либо действует, либо `superseded`. А вот у реестра задач охлаждение живое. Это подтверждает, что циклы разные, и хорошо, что не слили: иначе пришлось бы одно правило охлаждения натягивать на две несовместимые сущности.

[Математик]: Согласен, и это ровно моя посылка с начала: разные `mutations` → разные политики охлаждения → раздельные механизмы. Вердикт численно устойчив. Добавлю DoD-критерий: линтер `docs:verify-adr` проверяет иммутабельность принятых (текст не менялся в diff, если `status=accepted`) и валидность фронтматтера. Это чистая функция над git-diff, живёт в `scripts/*.mjs`, моя вторичная зона.

[Структурщик]: Принимаю вердикт и границы. Для протокола последующих заседаний фиксирую посылку: **ADR = отдельная лемма** со своим writer, своим шаблоном, своим линтером; пересечение с другими реестрами — только ссылками по id, без общих таблиц и без общей схемы. Заседания по форме записи, охлаждению, инструментам и подписчикам исходят из этого как из данности.

[Верстальщик]: Принимаю. Мой хвост для эпика: любой сводный «обзор решений» проектируется как view поверх раздельных источников, с честными бейджами типов — хранилища не смешиваем.

[Teamlead]: Единогласно. Вердикт зафиксирован, зависимые вопросы эпика #576 наследуют его как посылку. Закрываю заседание.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Обобщать ADR-контур с прочими реестрами правды в единый механизм? | **Нет.** ADR держим отдельным, самостоятельным реестром |
| Причина раздельности | Разный жизненный цикл: ADR иммутабелен (append-only, `mutations ∈ {0,1}`), реестр задач изменяем (`mutations ≈ 3–5`) |
| Выгода обобщения | Нулевая: RAG-индексация, git-история, PR-ревью уже общие для любого markdown — «единый реестр» ничего не добавляет |
| Цена обобщения | Отрицательная: связывает два несовместимых цикла, вынуждает скрытый `if(isADR)` в политике мутабельности |
| Что допустимо вместо слияния | **Read-only view** (сводная лента решений), агрегирующий раздельные источники — представление, не хранилище |
| Признак «отдельности» (проверяемый) | `status ∈ {proposed, accepted, superseded}`, `isEditable(adr) = status==='proposed'`, гард иммутабельности в CI |
| Пересечение с другими реестрами | Только ссылки по id (гиперссылка), без общих таблиц и общей схемы |

**Definition of Done (для инструментального следствия вердикта):**
- `docs/adr/` имеет собственный индекс, шаблон `NNNN-title.md` с фронтматтером `{id, status, date, supersedes?, superseded-by?}`, не импортирует схему `registry.json`.
- Линтер `docs:verify-adr` (чистая функция над git-diff, `scripts/*.mjs`): проверяет валидность фронтматтера и иммутабельность (текст `status=accepted` ADR не изменён в diff).
- Любой будущий «обзор решений» реализуется как read-only view поверх раздельных источников с бейджами типов (`accepted` = `badge-info`, `superseded` = `badge-ghost`); хранилища не сливаются.
- Вердикт внесён в эпик #576 как посылка для заседаний по форме записи / охлаждению / инструментам / подписчикам.
