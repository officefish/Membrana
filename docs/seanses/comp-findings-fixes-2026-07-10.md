[Музыкант]: Начну с третьей точки, она бьёт по живому запуску сильнее всего. На старте сценария batch пуст ровно один-два тика, пока collect не наполнился. Если анализатор кидает — сценарий умирает у владельца на глазах. Это не «ошибка топологии», это нормальный transient холодного старта. Я за вариант (а): молчащий анализ — ref invalid, exec продолжается. Звук/детекция просто ещё не набрали окно, это честное состояние «пока нечего анализировать».

[Математик]: Согласен по существу, но с уточнением контракта. Пустой batch — это не ошибка входных данных, это `n=0`. Чистая функция анализа на `n=0` должна возвращать «неопределённо», а не бросать. Формально: `MakeEnsembleAnalysis(∅) → DetectionResult{ valid:false, presentCount:0 }`, без записи в store. Throw оставляем только для реально невалидного входа (NaN, неверный kind), а не для пустоты. Это вариант (а), но я настаиваю: разделить «пусто» и «сломано» на уровне предиката, не сваливать в один catch.

[Структурщик]: Тогда зафиксируем термин. Сейчас «invalid-ref» перегружен: он означает и «значение ещё не готово», и «типовое несоответствие». Это как раз то, против чего я — один термин, один смысл. Предлагаю: пустой batch → `ref{ valid:false, kind: <target> }` — то есть **ref знает свой целевой kind даже когда пуст**. Это одновременно закрывает точку (3) *и* точку (1). Если invalid-ref всегда несёт корректный kind, `assertTypeCompatible` не спотыкается на дефолтном виде.

[Верстальщик]: С точки зрения UI мне важно, чтобы «молчащий анализ» не выглядел как зависание. Если exec продолжается, но карточка детектора пустая — нужно состояние «ожидание окна», а не пустой прогресс, который читается как баг. У меня в `DESIGN.md` есть `skeleton` + `aria-busy="true"` для async 1.B — переиспользую его для «окно набирается». Владелец должен видеть: система жива, просто копит данные. Молчание в рантайме ≠ молчание в UI.

[Teamlead]: Принимаю рамку. По (3) — вариант (а) молчащий анализ, но с дисциплиной Математика: `valid:false` только для пустоты, throw сохраняем для настоящего мусора. И решение Структурщика по kind в invalid-ref я хочу сделать **сквозным** — оно закрывает две точки одной конструкцией, это конструктивно, а не косметически. Давайте добьём (1) в этой логике.

[Музыкант]: По (1) я практик: команды сейчас обходят нетипизированными рёбрами — это костыль, который будет протекать в каждый новый сценарий. Не хочу документировать костыль (вариант «в»). Хочу, чтобы типизированное ребро от collect работало из коробки до первого flush.

[Математик]: Здесь есть два предложенных механизма: (а) collect-store всегда отдаёт invalid-ref с корректным kind целевого типа, (б) assertTypeCompatible пропускает любой invalid-ref без типовой проверки. Я против (б) в чистом виде: «пропускать проверку для valid:false» звучит безопасно, но это дыра — теряем контроль kind на всём протяжении, пока значение невалидно. Правильнее (а): kind известен всегда, проверка типов идёт всегда, но сравнивает kind, а не значение. Тип — это свойство ребра, а не значения на нём.

[Структурщик]: Полностью поддерживаю (а) и это ровно моя реплика выше. Kind — атрибут контракта ребра, он определён топологией графа в момент связывания, а не наличием данных. collect batch-выход имеет объявленный тип элемента → invalid-ref обязан нести этот kind. `assertTypeCompatible` тогда проверяет kind↔kind всегда, а `valid` отвечает только на вопрос «есть ли данные». Два ортогональных свойства — не смешивать. Вариант (б) их смешивает и я его блокирую как нарушение слабой связанности семантики.

[Математик]: Уточню численно, чтобы не было регрессии: assertTypeCompatible на паре (kind=X, valid=false) и (kind=X, valid=true) должен вести себя одинаково по типу — оба пройдут; различие только в том, читает ли потребитель значение. Downstream-узел, получив valid:false, идёт по ветке «нет данных». Это и есть связка (1)↔(3): один инвариант «ref всегда типизирован».

[Верстальщик]: Для меня (1) невидим в UI напрямую, но косвенно — если type-mismatch больше не валит граф на старте, то и «красный экран ошибки сценария» не мелькает у владельца. Это плюс. Прошу только, чтобы переход valid:false→true не дёргал индикаторы резко: то же сглаживание 400–500 ms по `LIVE_DETECTION_UI.md`, что и для confidence.

[Teamlead]: (1) — вариант (а), закреплённый как общий инвариант «invalid-ref всегда несёт целевой kind». (б) отклонён Структурщиком и Математиком с обоснованием — принимаю блок. Теперь самая идеологическая — (2), вход в alarm-loop. Это прямой конфликт с духом ND3.

**Definition of Done:**
- invalid-ref от collect-store и пустых анализаторов несёт `kind` целевого типа; `assertTypeCompatible` проходит на паре одинаковых kind независимо от `valid`.
- Пустой batch → анализатор не бросает, `valid:false`, exec продолжается; throw только на невалидном kind/NaN — покрыто unit-тестами.
- `make-detection-fusion` — единственный писатель `lastDetection`; alarm-front стартует по combinedScore; при отсутствии fusion-узла работает legacy fallback (тест на оба пути).
- Все **703 базовых теста device-board зелёные**; добавлены целевые тесты на (1)(2)(3); сценарии пикера не изменены.
- Обратная совместимость bundled MVP подтверждена smoke-прогоном трёх сценариев alpha/beta/gamma до передачи владельцу.
- UI показывает «окно набирается» (skeleton/aria-busy), без ложного вида ошибки; индикатор и переключение тревоги синхронны.
- CONCEPT.md alpha/gamma лежат в архиве спринта, docs-lint зелёный.
