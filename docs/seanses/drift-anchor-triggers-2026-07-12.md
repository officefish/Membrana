# Консилиум: развилка деплоя drift-anchor (DA3) — где ловить каждый якорь

**Повестка:** Разрешить развилку в спринте #396 (DA3): где ловить каждый из двух дрейф-якорей — в CI-гейте или в серверном расписании. Всплыла тонкость frozen-image: замороженный ночной образ не пересобирает детекторы, поэтому не способен зафиксировать дрейф самого детекторного кода. Нужно определить фазы, владельцев и записать их в спринт.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Структурщик → Верстальщик → Teamlead → Математик → Музыкант (циклически)

---

[Структурщик]: Зафиксирую термины, иначе спор будет о словах. У нас два разных якоря: **data-anchor** — дрейф входных данных/фичей при фиксированном коде детектора, и **code-anchor** — дрейф самого детекторного кода при фиксированном эталонном наборе. Это две леммы, не одна. Frozen-image ловит только первый: образ заморожен, детекторы внутри него не пересобираются.

[Верстальщик]: С точки зрения кабинета мне всё равно, где считается якорь, но результат обоих должен прийти в одну панель с честной меткой источника. Если я не вижу в бейдже, откуда дрейф — из CI или из ночного расписания — оператор запутается. Прошу заранее контракт на два разных `anchorSource`.

[Teamlead]: Согласен с разделением Ozhegov'а — это ядро развилки, а не деталь. Приоритет по `ARCHITECTURE.md`: детерминизм и откатываемость (эпик DR0–DR7). Frozen-image существует ровно ради воспроизводимости data-плоскости. Значит его нельзя нагружать задачей, которая требует пересборки кода — это противоречит его роли.

[Математик]: Численно это очевидно. Data-anchor: `detector = const`, метрика `Δ = f(input_t) − f(input_ref)`. Code-anchor: `input = const (эталонный корпус)`, метрика `Δ = f_new(corpus) − f_ref(corpus)`. Смешивать нельзя — иначе два источника дисперсии складываются и якорь перестаёт быть якорем: непонятно, что дрейфануло.

[Музыкант]: Поддержу с практической стороны. Когда я гоняю детектор на живом микрофоне, я вижу оба дрейфа сразу и это ад для отладки. Если код-дрейф ловится отдельно на замороженном корпусе — я могу доверять live-метрике как чистому data-сигналу. Это сильно упрощает мне ensemble-калибровку.

[Структурщик]: Тогда предлагаю топологию. **Code-anchor → CI-гейт**: PR, меняющий `packages/services/detectors/*`, обязан прогнать эталонный корпус (`free-v1`, 120 сэмплов) и сравнить с baseline; регресс блокирует merge. Это фаза сборки, ей место в CI, где код *свежий по определению*.

[Верстальщик]: Тогда code-anchor — это не UI-панель, а статус PR/CI. Мне его рисовать не нужно, максимум badge «detector baseline: OK/regress» в кабинете как read-only. Не раздувайте мне на это отдельный экран.

[Teamlead]: Верно. Code-anchor — гейт, а не мониторинг. **Data-anchor → серверное расписание** на frozen-image: ночью гоняем замороженный образ по свежему потоку/новым сэмплам, ловим дрейф входных данных. Образ остаётся детерминированным, код внутри — та версия, что была на момент заморозки. Это честно и воспроизводимо.

[Математик]: Одно уточнение к серверному якорю: frozen-image даёт **stale code-anchor**. То есть data-дрейф он меряет версией детектора N-дневной давности. Если между заморозками код менялся, ночная метрика отстаёт. Нужно логировать `detectorVersion` образа рядом с data-anchor, чтобы Rodchenko показал возраст baseline.

[Музыкант]: Вот это важно. Иначе оператор увидит «дрейфа нет» ночью, а на живом узле днём уже другой детектор и другое поведение. Возраст образа — это не косметика, это защита от ложного спокойствия.

[Структурщик]: Значит контракт якоря в `@membrana/core` должен нести: `anchorKind: 'data' | 'code'`, `anchorSource: 'ci' | 'schedule'`, `detectorVersion`, `imageFrozenAt`, `delta`, `verdict`. Один тип, две реализации producer'ов. Никаких прямых импортов между CI-скриптом и серверным джобом — оба пишут в общий журнал через контракт.

[Teamlead]: Принимаю контракт. Но всплыла та самая тонкость из повестки: frozen-image **в принципе** не может поймать code-anchor. Значит одной ночной джобой мы дыру не закрываем. Нужна **третья фаза**: периодическая пересборка детекторов из текущего `main` и прогон по эталонному корпусу — как «страховка» на случай, если code-anchor в CI кто-то обошёл (hotfix, ручной деплой).

[Математик]: Да, это `scheduled-code-anchor`: `f_main(corpus)` из свежесобранного образа против зафиксированного baseline. Он дублирует CI-гейт по смыслу, но ловит расхождение прод-деплоя с тем, что реально в `main`. Чистая функция: вход — корпус + текущий код, выход — delta. Побочных эффектов нет, всё в мою зону.

[Музыкант]: Только не гоняйте пересборку каждую ночь — детекторную сборку + прогон 120 сэмплов это заметный CPU. Раз в сутки достаточно, а лучше по событию «деплой прошёл вне CI». Я против того, чтобы ночной цикл распух и мешал data-джобе по ресурсам.

[Структурщик]: Согласен — три producer'а, но два джоба на сервере (data-anchor ночью, scheduled-code-anchor по деплою/раз в сутки) + один в CI. Границы: CI-скрипт живёт в деплой-пайплайне (`dpr-*`), серверные джобы — в `background-media`, контракт — в `core`. Никто ни у кого не импортирует внутренности.

[Верстальщик]: Тогда мне нужны в панели три строки с разными `anchorSource` и возрастом. Сделаю карточку «Дрейф-якоря»: code (CI) — badge OK/regress; code (schedule) — с меткой `собран из main @ commit`; data (schedule) — с `detectorVersion` + `imageFrozenAt`. `tabular-nums`, честные числа, без метафор. `aria-live` на смену вердикта.

[Teamlead]: Хорошо. Расхождение между code-anchor(CI) и scheduled-code-anchor — это алерт «прод разошёлся с main», отдельный вердикт. Rodchenko, это твоя danger-строка. Не прячь её за цвет — иконка + текст «Прод ≠ main».

[Математик]: И порог. Для code-anchor delta порог жёсткий (регресс F1 > ε блокирует), для data-anchor — мягкий (дрейф → warning, не блок деплоя, потому что данные меняются легитимно). Разные семантики порога у разных якорей. Калибровку ε вынесу в отдельную запись, привяжу к `free-v1`.

[Музыкант]: Принимаю. Data-дрейф как warning — правильно: реальный звук дрона меняется от модели к модели, блокировать деплой из-за этого нельзя. А code-регресс — это наша ошибка, его блокировать надо жёстко.

[Структурщик]: Итог по границам зафиксирован, циклов зависимостей нет. Принимаю: контракт в `core`, три producer'а, две новые фазы в спринт (scheduled-code-anchor + расхождение-алерт).

[Teamlead]: Сводлю. Развилка разрешена **не как «или/или», а как три якоря с разными местами ловли**. Добавляем в #396 две фазы: DA4 (scheduled-code-anchor) и DA5 (расхождение CI↔schedule + UI). LGTM формы решения. Принимаю.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Где ловить **code-anchor** (дрейф кода детектора) | **CI-гейт**: PR в `detectors/*` прогоняет эталонный корпус `free-v1`, регресс F1 > ε — **жёсткий блок merge**. Свежий код по определению. |
| Где ловить **data-anchor** (дрейф входных данных) | **Серверное расписание** на frozen-image, ночью. Детерминированно; дрейф → **warning**, не блокирует деплой. Логировать `detectorVersion` + `imageFrozenAt`. |
| Проблема frozen-image (не видит code-drift) | Признана фундаментальной: замороженный образ не пересобирает детекторы. Закрывается **третьим якорем**, а не нагрузкой на ночную data-джобу. |
| Третий якорь: **scheduled-code-anchor** | Периодическая пересборка детекторов из `main` + прогон корпуса (раз в сутки / по событию деплоя вне CI). Ловит расхождение прод-деплоя с `main`. |
| Расхождение CI ↔ schedule | Отдельный вердикт-алерт «Прод ≠ main» (danger, иконка + текст, не только цвет). |
| Контракт | Единый тип якоря в `@membrana/core`: `anchorKind`, `anchorSource`, `detectorVersion`, `imageFrozenAt`, `delta`, `verdict`. Три producer'а через журнал, без прямых импортов. |
| Пороги | code-anchor — жёсткий (блок), data-anchor — мягкий (warning); калибровка ε — отдельная запись Математика на `free-v1`. |
| Новые фазы спринта #396 | **DA4** (scheduled-code-anchor) + **DA5** (расхождение-алерт + UI-панель якорей). |
