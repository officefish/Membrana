[Верстальщик]: Начну с потребителя вещдока — а потребитель это агент, читающий счётчик вдоха в PR, и человек в аудите. Моя норма из `main-day-accuracy`: экран не должен врать и не должен показывать «мёртвую дверь». Значит счётчик `{опорный_хеш, счётчик, K_эфф}` обязан читаться так, чтобы было видно, **сколько вдохов сделано из семи** и **почему следующий ещё не случился** (дрейф ниже `K_эфф`). Один индикатор — один смысл; не смешивать «пора» и «потолок» в один серый бейдж.

[Музыкант]: По касательной, аудио тут не задет, но ритм я узнаю — это как метроном, который щёлкает не по времени, а по смене материала. Вдох = «дорожка изменилась по существу», а не «кто-то подвигал фейдер на 0.1 дБ». Мой живой флаг из `ritual-inputs-echo`: три отражения одного снимка — это `n=1`, а не `n=3`. Косметическая правка тезиса не должна щёлкать метрономом, иначе накрутим вдохи-пустышки — чистый паразит. Гранула события должна ловить **новый сигнал**, а не эхо старого.

[Математик]: Формализую, иначе «по существу» останется вкусовым. Пусть конспект на момент события — упорядоченная последовательность нормализованных тезисов `T = [t₁,…,tₘ]`. Хешируемое состояние — **весь конспект целиком**, спроецированный в SimHash-вектор фиксированной ширины `b = 64` бита: `h(T) ∈ {0,1}⁶⁴`. Дрейф между двумя вдохами — расстояние Хэмминга `d(h_prev, h_cur) = popcount(h_prev XOR h_cur)`. Вдох происходит, когда `d ≥ K_эфф`. Всё детерминировано: SimHash — чистая функция от токенов, `popcount` — чистая функция. Ни `Date.now()`, ни `Math.random`.

[Структурщик]: Начну с термина, иначе весь словарь поплывёт. У нас три разные леммы под угрозой слипания: **событие** (единица, после которой пересчитываем хеш), **вдох** (удачный дрейф хеша за порог) и **пауза** (граница окна счёта). Событие ≠ вдох: событий много, вдохов ≤ семь. Определяю событие как **смену состояния тезиса** (добавлен тезис / тезис изменён по существу / тезис снят) — не «реплику» как строку. Реплика может не менять ни одного тезиса; тогда события нет, хеш не считаем.

[Teamlead]: Рамку фиксирую сразу, чтобы M3 не сполз в M2 (роли) и M4 (место). Мы решаем **ровно** механизм вдоха: что хешируется, что событие/пауза, критерий `K`, форма `g(a)`, счётчик и завершимость. Пол «пора» и потолок семь вдохов — решены владельцем, не пересматриваем. По конструкции «форма следует функции»: единственная функция вдоха — дать шторму детерминированный само-темп, поэтому каждая деталь механизма оправдана либо детерминизмом, либо завершимостью. Продолжаем по осям.

[Верстальщик]: Согласен с определением события Структурщика, но добавлю требование читаемости. Раз событие = смена состояния тезиса, то в вещдоке должно быть видно `счётчик_событий_с_прошлого_вдоха` рядом с `d` и `K_эфф`. Иначе читатель видит «вдохов 2 из 7» и не понимает, застрял шторм или просто копит дрейф. Три числа — три колонки, не одна. Это ровно моя норма против мёртвой двери.

[Музыкант]: Поддержу и уточню про «по существу» на языке сигнала. Нормализация тезиса перед SimHash — это как гейт-порог на входе в тракт: срезаем шум (пробелы, регистр, порядок слов внутри тезиса), пропускаем смысл. Если два тезиса дают одинаковый нормализованный токен-мультисет — это `n=1`, дубль не считается новым событием. Флаг: без нормализации SimHash задребезжит на пунктуации, и `g(a)` придётся крутить, чтобы это компенсировать — лечим болезнь на входе, а не порогом.

[Математик]: Принимаю нормализацию как предусловие чистоты. Формализую критерий «по существу» строго: событие валидно ⟺ `h(T_после) ≠ h(T_до)` **при нормализованном входе**. Косметика по построению даёт `h_после = h_до`, значит `d = 0`, вклада в дрейф нет — паразит закрыт математикой, а не эвристикой. Теперь `g(a)`. Пусть `a` — число артефактов (принятых вещдоков) на момент проверки. Требую: `g` монотонно не убывает по `a`, `g(0) = 0`, и `g` ограничена сверху: `g(a) ≤ K0 − K_min`, где `K_min ≥ 1`. Форма: `g(a) = ⌊(K0 − K_min) · a/(a + c)⌋`, `c > 0` — константа полунасыщения. Тогда `K_эфф = K0 − g(a) ∈ [K_min, K0]`.

[Структурщик]: Начну с термина «пауза», иначе окно счёта поплывёт. **Пауза** — не молчание по времени (времени у нас нет), а **событие типа «ход завершён без новой смены тезиса»** — маркер конца окна. Определяю окно счёта как последовательность событий между двумя паузами; хеш считаем после каждого события, а решение о вдохе принимаем на границе паузы по накопленному `h_cur` против `h_prev`. Это разводит две статьи: «когда пересчитываем» (каждое событие) и «когда решаем» (пауза). Один термин — один смысл.

[Teamlead]: Здесь ставлю проверку границы. Согласен, что пауза — маркер-событие, а не таймер: это держит нас в детерминизме и вне server-first этажа (гейт 3, не наш). Но требую, чтобы `g(a)` читала `a` из **уже принятых** вещдоков на момент паузы, а не из «ожидаемых» — иначе смягчение станет гаданием. Форма Дынина с полунасыщением конструктивно оправдана: она не даёт `K_эфф` ни упасть в 0 (мгновенный потолок), ни застрять в `K0` навечно. LGTM формы, но жду доказательства завершимости.

[Верстальщик]: По вещдоку паузы — потребитель должен отличать «пауза без вдоха» (дрейф не дотянул) от «пауза с вдохом» (дотянул, счётчик +1). Это две разные записи в журнале, не одна серая. Моя норма честности: если на паузе `d < K_эфф`, показываем «накоплено d из K_эфф», а не пусто. Тогда агент видит, что шторм жив и движется к следующему вдоху, а не завис.

[Музыкант]: Флаг про завершимость от практика тракта. Если `g` смягчает порог числом артефактов, а артефакты рождаются во время шторма, есть риск петли: больше говорим → больше артефактов → ниже порог → больше вдохов. Но это как раз то, что нужно для «не зависнуть» — обратная связь ускоряет финал, а не тормозит. Опасность в другом: чтобы «успеть высказаться» не проиграло. Пусть `K_min ≥ 1` гарантирует, что даже при максимальном смягчении нужен **хотя бы один бит** реального дрейфа — пустышка не пройдёт.

[Математик]: Доказываю завершимость. Утверждение: семь вдохов достижимы за конечное число событий при любом валидном `g`. По построению `K_эфф ≥ K_min ≥ 1` всегда. Значит для вдоха достаточно дрейфа `d ≥ 1` бита. Каждое событие «по существу» меняет нормализованный конспект, а значит с ненулевым структурным изменением меняет хотя бы часть SimHash-вектора; накопление таких изменений монотонно увеличивает `d(h_prev, h_cur)` до тех пор, пока `d ≥ K_эфф` не станет истинным (на конечном числе существенных событий, т.к. `d` ограничено `b=64` и растёт неубыванием по существенным правкам). После вдоха `h_prev ← h_cur`, счётчик += 1. Повтор ≤ 7 раз ⇒ конечное число событий. Верхняя оценка грубая: `≤ 7 · b = 448` бит-изменений в худшем случае. Завершимость доказана.

[Структурщик]: Уточню инвариант счётчика, иначе состояние поплывёт. Рантайм-счётчик — это кортеж `{опорный_хеш: bits64, счётчик: 0..7, K_эфф: K_min..K0}`, и он **derived** от лога событий, а не мутируется вслепую: `state = fold(events)`. Мы это ровно так решали в `insight-archive-lifecycle-c4`: source of truth — append-only лог событий, счётчик — чистая проекция. На вдохе reducer заменяет `опорный_хеш` и инкрементит `счётчик`; между вдохами `опорный_хеш` неизменен. Никакого скрытого состояния.

[Математик]: Принимаю проекцию. Тогда полный контракт вдоха — чистая функция: `step(state, event) → state'`. При `event` существенном: `h_cur = simhash(normalize(конспект))`; `d = popcount(state.опорный_хеш XOR h_cur)`; `K_эфф = K0 − g(a)`; если `d ≥ K_эфф` ∧ `state.счётчик < 7` ⇒ `state' = {опорный: h_cur, счётчик: счётчик+1, K_эфф}`; иначе `state' = {…state, K_эфф}`. Тот же вход — тот же выход, без сети, без времени, без ГСЧ. Это ровно то, что требует бриф.

[Teamlead]: Проверяю на колонизацию: мы не назначили ни одной роли (M2) и не тронули место (M4) — чисто. Числа за Дыниным, как решил владелец 20.07, и они предъявлены: `b=64`, `K_min≥1`, `g(a)=⌊(K0−K_min)·a/(a+c)⌋`, оценка `≤7·b`. Конструктивно всё оправдано функцией. Один открытый параметр — численные `K0` и `c` — это калибровка, не механизм; фиксирую их как настраиваемые константы вещдока, дефолт задаёт Дынин при реализации.

[Верстальщик]: Последнее по подаче. Дефолты `K0`/`c` в вещдоке должны быть подписаны и видимы — не магические числа в коде. Читатель журнала обязан видеть, при каком `K0` и `c` считался этот вдох, иначе аудит невоспроизводим глазами. Это a11y самого артефакта: honest data, tabular-nums, никаких скрытых кнопок.

[Музыкант]: Принимаю весь механизм. Как ритм он честный: щёлкает по существу, ускоряется артефактами, но всегда требует минимум один бит настоящего дрейфа. «Успеть высказаться» держится полом `K_min`, «не зависнуть» — обратной связью `g(a)` и потолком семь. Пустышки закрыты нормализацией. LGTM по моей зоне.

[Структурщик]: Принимаю. Словарь чист: событие / вдох / пауза разведены, счётчик — derived-проекция append-only лога, инварианты явные. Нарушения слабой связанности нет — механизм это чистая функция `step`, без сети и без чужих зависимостей.

[Математик]: Принимаю. Детерминизм и завершимость доказаны численно, а не на слух.

[Teamlead]: Принимаю. LGTM. Форма следует функции, границы фаз соблюдены, числа за Дыниным предъявлены. Фиксирую вердикт.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Хешируемое состояние | **Весь конспект целиком** (упорядоченная последовательность нормализованных тезисов) → SimHash-вектор ширины `b = 64` бита. Не дельта, не срез. |
| Событие | **Смена состояния тезиса** (добавлен / изменён по существу / снят). Реплика без смены тезиса — не событие. Хеш пересчитываем после каждого события. |
| Пауза | Событие-маркер «ход завершён без новой смены тезиса» — граница окна счёта. Решение о вдохе принимается на паузе. Времени/таймеров нет. |
| «Удачный хеш» (порог `K`) | Вдох ⟺ `d(опорный, текущий) = popcount(XOR) ≥ K_эфф` в битах Хэмминга. «По существу» гарантировано нормализацией входа: косметика даёт `d=0`. |
| `g(a)` (смягчение) | `K_эфф = K0 − g(a)`, `g(a) = ⌊(K0 − K_min)·a/(a+c)⌋`, `c>0`. Монотонна неубывающе, `g(0)=0`, `K_эфф ∈ [K_min, K0]`, `K_min ≥ 1`. `a` — число **уже принятых** артефактов. |
| Счётчик рантайма | `{опорный_хеш: bits64, счётчик: 0..7, K_эфф}` как **derived-проекция** append-only лога событий: `step(state, event) → state'`. Без сети, без `Date.now()`, без `Math.random`. |
| Завершимость | Доказана: `K_эфф ≥ K_min ≥ 1` ⇒ достаточно ≥1 бита дрейфа; существенные события монотонно наращивают `d` до `b=64`; ≤ 7 вдохов ⇒ ≤ `7·b = 448` бит-изменений в худшем случае. Конечно. |

**Definition of Done (для Z3):**
- Реализована чистая функция `step(state, event) → state'` и `simhash(normalize(конспект)) → bits64`; обе без сети, без времени, без ГСЧ; покрыты юнит-тестами на детерминизм (тот же вход → тот же выход).
- Тест: косметическая правка тезиса даёт `d = 0` (паразит-пустышка не проходит).
- Тест завершимости: последовательность существенных событий достигает `счётчик = 7` за конечное число шагов при `K_min = 1` и при максимальном смягчении.
- `g(a)` реализована с проверяемыми границами `K_эфф ∈ [K_min, K0]`; дефолты `K0`, `c`, `K_min` вынесены в подписанные константы вещдока, видимые в журнале (не magic numbers).
- Вещдок вдоха несёт `{опорный_хеш, счётчик, K_эфф, d, a, K0, c}`, читаемые как отдельные поля (норма честности артефакта).
