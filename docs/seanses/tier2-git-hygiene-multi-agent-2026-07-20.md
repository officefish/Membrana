# Консилиум: git-гигиена для четырёх параллельных ИИ-агентов на одной машине

**Повестка:** Найти наиболее правильную git-гигиену для мульти-агентной разработки Membrana. Данность: 4 агента (2× Claude Code, 1 Codex, 1 Grok) на ОДНОЙ машине, ОДИН проект, ОБЩИЙ git-объектный стор и диск. Предложение владельца (5 канонических worktree: main / tooling / product / codex / cursor) обсуждается как один из вариантов, не как данность. Backlink: `insight-tier2-worktree-topology`.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Математик → Верстальщик → Структурщик → Музыкант → Teamlead (циклически)

[Математик]: Формализую, иначе «правильная гигиена» останется вкусовой. У нас 4 писателя `W = {claude1, claude2, codex, grok}` и общий ресурс — один `.git`-объектный стор. Ключевой предикат безопасности: `∀ w: write(w)` не должен пересекаться с `write(w')` в одной изменяемой области. Гит уже даёт нам две развязки: объекты (immutable, content-addressed, конфликтов нет физически) и **refs + index + рабочее дерево** (mutable, вот здесь гонки). Значит проблема не в объектном сторе — он безопасен для конкурентного чтения/записи по конструкции. Проблема ровно в двух вещах: один shared `HEAD`/index на worktree и одновременный `gc`.

[Верстальщик]: Со стороны потребителя — а потребитель здесь агент, который утром читает состояние, и человек в PR — мне нужна **видимость владения**. Если четыре агента пишут в общий диск, я хочу однозначно на холсте видеть: эта ветка чья, этот worktree чей, кто держит lock. Мёртвая дверь тут — агент думает, что работает изолированно, а на деле делит index с соседом. Схема владельца с именованными worktree это как раз даёт: имя = владелец. Но само имя ещё не гарантия изоляции.

[Структурщик]: Начну с термина, иначе DAG поплывёт. «Worktree» здесь — это две слипшиеся леммы. Первая: **git worktree** — отдельное рабочее дерево с собственным `HEAD` и index, но общим объектным стором (`git worktree add`). Вторая: «долгоживущая ветка-роль» (`tooling`, `product`, `codex`, `cursor`) — это уже про топологию refs, не про диск. Предложение владельца их сплавляет, и это опасно. Правильно: worktree решает изоляцию **процесса** (index/HEAD не общий), а базовая ветка решает изоляцию **истории**. Это ортогональные оси, их нельзя мерить одним словом.

[Музыкант]: По касательной — аудио тут не задет, но ритм многодорожечной записи я узнаю мгновенно. Четыре агента на одном проекте — это четыре музыканта, пишущихся в одну студию. Золотое правило студии: **у каждого своя дорожка (worktree), но один общий тайм-код (main)**. Если два исполнителя пишут в одну дорожку одновременно — получаешь фазовую кашу, наложение, которое потом не расслоить. Мой флаг из опыта: `git worktree` — это именно раздельные дорожки на общем мультитреке. Схема владельца ритмически здоровая. Смущает только одно: `gc` — это как техник, который между дублями чистит ленту, пока на неё пишут.

[Teamlead]: Рамку фиксирую сразу, чтобы консилиум не расползся в «как нам вообще жить с четырьмя агентами». Наш вопрос узкий: **наиболее правильная git-гигиена**, а не оргпроцесс. По конструктивистскому «форма следует функции»: функция — конкурентная запись без порчи чужой работы и без гонок. Математик уже разделил безопасное (объекты) и опасное (refs/index/gc). Ozhegov разделил две оси (диск vs история). Это фундамент. Предложение владельца я принимаю как **сильный кандидат**, но требую от каждой роли назвать конкретную защиту от гонки, иначе это красивая схема без несущей конструкции.

[Математик]: Считаю вероятность порчи. `git gc` (в т.ч. автоматический `gc.auto` при почти каждом `commit`/`fetch`) переупаковывает и **удаляет** объекты, которые считает недостижимыми. Если агент B создал коммит, но ещё не привязал его к ref, а агент A запустил `gc` — B теряет объект. Вероятность ненулевая и растёт с частотой коммитов четырёх писателей. Вывод численный и жёсткий: **`gc.auto=0` глобально** для этого репозитория, `gc` только вручную из `main`-worktree, когда остальные не пишут. Это не опция — это инвариант безопасности.

[Верстальщик]: Поддержу с холста: инвариант, который не виден, — не инвариант. Если `gc.auto=0` живёт только в чьей-то голове, следующий агент через неделю запустит `git gc` руками и всё сломает. Норма честности: это должно стоять в `.git/config` репозитория (видимо, версионируемо через доку) и дублироваться строкой в `CONTRIBUTING.md` / worktree-README. Потребитель — агент — обязан прочитать «gc запрещён, кроме main» до первого коммита.

[Структурщик]: Согласен, и уточню границу словаря. Пять worktree владельца — это пять **точек входа процесса**, и у каждой обязан быть однозначный контракт «что здесь можно». Моя тревога — синонимы-слои: `tooling` и `product` оба «разработка на Claude». Это два фасада на одно (Claude пишет код). Различие должно быть названо явно: `tooling` = скрипты/CI/ритуалы (`scripts/*.mjs`, `docs/`), `product` = `apps/*` + `packages/*`. Если различие только «какой Claude сейчас свободен» — это не граница, это случайность, и её надо убрать. Либо предметное разделение, либо слить в один worktree разработки.

[Музыкант]: Тут вклинюсь смелой идеей, помечаю как риск. А зачем вообще привязывать worktree к **агенту** (`codex`, `cursor`)? Дорожку в студии называют по инструменту, не по музыканту — «бас», а не «Вася». Привязка к агенту хрупкая: Grok заменят на другую модель — и `cursor`-worktree врёт именем. Предлагаю называть worktree по **зоне работы**, а не по агенту: `main`, `tooling`, `product`, `support-a`, `support-b`. Риск моего варианта: теряется человеческая привязка «кто где сидит». Но по долговечности зона честнее модели.

[Teamlead]: Ozhegov и Музыкант бьют в одну точку с двух сторон — и это ценно. Фиксирую как развилку: **worktree по зоне (долговечно) vs по агенту (удобно сейчас)**. Приоритет `ARCHITECTURE.md` и «форма следует функции»: имя должно переживать смену исполнителя. Склоняюсь к зонам. Но данность владельца — 2 Claude + Codex + Grok, у них разные окружения и разные аутентификации в git/GitHub. Это аргумент за агент-привязку у **саппорт**-worktree. Компромисс архитектора: **общие зоны по функции (main/tooling/product), а саппорт-контуры — по агенту**, потому что у Codex и Grok разная физическая интеграция. Гибрид оправдан конструкцией, а не вкусом.

[Математик]: Проверю гибрид на непротиворечивость. Инвариант «один worktree — максимум один пишущий процесс в момент времени» сохраняется в обеих схемах. Гибрид добавляет только требование: маппинг `агент → worktree` должен быть **функцией** (однозначный), не отношением. Два Claude на `product` и `tooling` — это ок, если они в разных worktree. Два Claude в одном worktree одновременно — нарушение, гонка по index. Значит правило: `|writers(worktree)| ≤ 1` в любой момент. Это проверяемо: lock-файл или соглашение «один агент — один worktree за сессию».

[Верстальщик]: Тогда мне на холст нужен **индикатор занятости worktree**. Как в студии лампа «REC» над дверью. Дешёвый вариант: файл `.worktree-owner` в корне каждого worktree с именем агента и timestamp сессии; агент при старте читает, при живом чужом owner — не заходит. Это тот же паттерн, что я защищал в вечернем аудиторе: состояние обязано быть видимым, не подразумеваемым. Без лампы «REC» два Claude рано или поздно войдут в `product` вдвоём.

[Структурщик]: Принимаю лампу, но уточню лемму «lock». Есть два разных замка: **advisory** (файл-соглашение, агент может проигнорировать) и **mandatory** (git сам не даст — например, `git worktree` уже запрещает checkout одной ветки в двух worktree). Второй мы получаем бесплатно: если каждый worktree сидит на СВОЕЙ базовой ветке, git физически не позволит второму worktree взять ту же ветку. Значит базовая гигиена: **разные базовые ветки = встроенный mandatory-lock от git**. Advisory-файл — только поверх, для человекочитаемости. Это сильно надёжнее голого соглашения.

[Музыкант]: Вот это и есть чистый звук — git сам держит фазу, а не мы уговариваем агентов. Но добавлю флаг из реальной студийной боли: **спринт уходит в собственную ветку с базовой**. Владелец это и написал. Ритмически критично: долгоживущий worktree никогда не коммитит в свою базовую напрямую, он всегда ответвляется. Иначе базовая ветка `product` превращается в помойку недособранных дублей, и squash-PR (наш канон из TASKS_MANAGEMENT §7а) теряет чистоту. Базовая ветка worktree = метроном, а не дорожка записи.

[Teamlead]: Это ключевая нормализация, принимаю жёстко. Базовая ветка worktree — **держатель позиции, не место работы**. Работа всегда в task-ветке, squash-PR в `main`, как канон 2026-06. Тогда картина сходится: `main`-worktree синхронизируется с origin/main и служит источником; каждый саппорт/зонный worktree держит свою базовую ветку (mandatory-lock от git), но пишет в отдельные task-ветки. Конфликт двух агентов на одну task-ветку исключён физически. Остаётся один класс риска — общий диск вне git: `node_modules`, `dist`, `.turbo`, локальные `.env`.

[Математик]: Да, и это второй непокрытый инвариант. Общий диск ≠ общий git. `node_modules` и `.turbo`-кэш — это разделяемое мутабельное состояние ВНЕ версионирования. Если два worktree делят один `node_modules` (симлинк или монорепо-хойстинг) и один ставит зависимость, второй получает несогласованное дерево посреди сборки. Численно: turbo-кэш адресуется хешем входов — здесь коллизий нет, кэш безопасен для чтения. Опасна только **запись установки** (`yarn install`) и запись `dist`. Требование: у каждого worktree **свой** `node_modules` (worktree — отдельная FS-директория, это и так так по умолчанию), install изолирован. Общий — только turbo remote/local cache, он content-addressed.

[Структурщик]: Точная поправка. `git worktree add` создаёт отдельную директорию — значит `node_modules` уже физически раздельны, если не городить симлинки. Правило нормы: **не шарить `node_modules` между worktree руками**. Turbo-кэш (`.turbo`) — можно и полезно шарить как read-mostly, он по хешу. Итого граница словаря чистая: общий — `.git/objects` (безопасен) и turbo-cache (безопасен); раздельные — рабочее дерево, index, HEAD, node_modules, dist, .env.

[Верстальщик]: Сведу для потребителя, потому что четыре агента не прочитают десять инвариантов россыпью. Нужна одна **карточка worktree** в каждом корне: имя, зона, базовая ветка, владелец-агент, строка «gc запрещён», строка «install только свой». Это ридми-конструктив, а не декор: агент открывает worktree — сразу видит контракт. Без этой карточки гигиена держится на памяти, а память у сессии агента короткая.

[Музыкант]: Поддержу и закрою свою партию. Схема владельца в сумме звучит чисто, если снять две фальшивые ноты: (1) `tooling`/`product` разделить по предмету, не по «какой Claude свободен»; (2) `codex`/`cursor` — принять привязку к агенту как временную, с пометкой «имя по инструменту-интеграции, не по модели». Всё остальное — раздельные дорожки, общий тайм-код, метроном не пишем — это ровно правильная студия для четырёх рук.

[Teamlead]: Свожу вердикт. Схему владельца **принимаю с нормализацией**, а не как есть — потому что сырая схема сплавляла диск и историю (правка Ozhegov) и не называла защиту от гонок (правка Dynin). Наиболее правильная гигиена = worktree-per-зона/агент + разные базовые ветки как mandatory-lock + работа только в task-ветках + `gc.auto=0` + раздельный install + видимая карточка worktree. Это конструкция, где каждая деталь оправдана функцией, декора нет. Владельцу — развилка по именам саппорт-worktree (зона vs агент) с моей рекомендацией гибрида. Все роли — подтвердите.

[Математик]: Принимаю. Инварианты замкнуты: объекты безопасны, gc заперт, index раздельный, install раздельный. Численно порчи не вижу.

[Структурщик]: Принимаю. Термины разведены, синонимы-слои `tooling`/`product` требуют предметного различия в ADR — это моё условие.

[Верстальщик]: Принимаю. Условие: карточка worktree обязательна и видима, иначе гигиена невидима = мёртвая дверь.

[Музыкант]: Принимаю. Метроном не пишем, дорожки раздельны — звучит.

[Teamlead]: Принимаю и фиксирую. **LGTM** на нормализованную схему; backlink `insight-tier2-worktree-topology`.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Принять схему владельца (5 worktree)? | **Да, с нормализацией** — не в сыром виде |
| Ось изоляции | Разделить: worktree = изоляция процесса (index/HEAD/node_modules); базовая ветка = изоляция истории |
| Защита от одновременной записи в одну ветку | **Разные базовые ветки на worktree** → git даёт mandatory-lock бесплатно (нельзя checkout одной ветки в двух worktree) |
| Работа в базовой ветке worktree | Запрещена; базовая ветка = держатель позиции. Любой спринт → отдельная task-ветка → squash-PR в `main` (канон §7а) |
| `git gc` | **`gc.auto=0`** глобально для репо; ручной `gc` только из `main`-worktree, когда никто не пишет |
| Именование `tooling` vs `product` | Развести **по предмету**: `tooling` = scripts/docs/CI, `product` = apps/packages. Иначе слить в один worktree разработки |
| Именование `codex` / `cursor` | Привязка к агенту принята как временная (разные интеграции/auth); имя — по инструменту-интеграции, не по модели; долгосрочно тяготеть к зонам |
| Общий диск вне git | `.git/objects` и turbo-cache (content-addressed) — шарить безопасно; `node_modules` / `dist` / `.env` / index — **раздельные per worktree**, install изолирован |
| Видимость владения | Карточка worktree в каждом корне (имя, зона, базовая ветка, владелец, «gc запрещён», «install свой») + advisory `.worktree-owner` поверх mandatory-lock |
| Инвариант конкуренции | `|writers(worktree)| ≤ 1` в любой момент времени |

**Definition of Done (если ведёт к задаче в коде):**
- `git config gc.auto 0` применён на репозитории; факт задокументирован в `CONTRIBUTING.md` и в карточке каждого worktree.
- Пять (или нормализованных) worktree созданы через `git worktree add`, каждый на СВОЕЙ базовой ветке; проверено, что git запрещает checkout одной ветки во втором worktree.
- В каждом корне worktree лежит `WORKTREE.md` (карточка контракта) + опциональный `.worktree-owner`.
- `node_modules` физически раздельны (нет ручных симлинков между worktree); turbo-cache настроен как shared content-addressed.
- ADR фиксирует предметное различие `tooling` ↔ `product` (условие Ozhegov).
- Ни один агент не коммитит в базовую ветку worktree напрямую; линт/хук или норма отклоняет прямой коммит в базовую ветку — только task-ветки со squash-PR.
- Backlink на `insight-tier2-worktree-topology` проставлен в артефакте закрытия.
