# A3 — второй актор на сервере (вердикт ведущего, ручная чеканка)

| Поле | Значение |
|------|----------|
| Заседание | `request-container-api` |
| Комната | A3 (третья по ратифицированному порядку) |
| Повестка | [`A3_AGENDA.md`](../meeting/request-container-api/A3_AGENDA.md) |
| Канал | **рука** (цепочка провайдеров исчерпана) |
| Статус | вердикт есть, **ратификации владельца ждёт** |

## Вопрос

**A3 — Кто является вторым актором на сервере и чем достигается его отличимость от первого: (а) что подписывает `countersign` и где он живёт в контуре заявки; (б) чем второй актор отличим, раз кабинет даёт полномочия, но не личность, и один ключ обслуживает всех; (в) несёт ли A3 механизм выдачи и изменения кабинета, на который сослалась K1?**

## Итоговое решение

| Часть | Решение |
|-------|---------|
| **(а) Что подписывает и где живёт** | `countersign` подписывает **хеш содержимого закрываемой единицы работы** (подпись есть пин над содержимым: правка после приёмки роняет подпись дрейфом). Живёт на **серверном переходе «выполнено → принято»**, между исполнением заявки и архивацией; фиксируется как переход билета с автором и серверным временем. |
| **(б) Чем отличим второй актор** | Вторым актором выступает **серверный исполнитель (Фаррелл)**. Отличимость даёт **не кабинет, а асимметрия ключей**: клиент по Т2 ключей не имеет вовсе и способен только подать заявку; акт приёмки физически производится там, куда клиент не дотягивается. Отсюда `executed.by` — заявитель, `accepted.by` — серверный актор, и они различимы по построению, а не по договорённости. |
| **(б-ограничение)** | **Асимметрии ключей НЕДОСТАТОЧНО.** Если акт приёмки запускается заявкой клиента, агент авторствует собственную приёмку руками Фаррелла — театр вместо контроля. Поэтому приёмка обязана быть **критериальной, а не заявочной**: Фаррелл совершает её, когда сходится предикат над фактами, которых заявитель не контролирует (результат боевого прогона, статус проверок, серверное время и длительности, факты квоты). Заявка «прими меня» — не вход приёмки. |
| **(в) Выдача кабинета** | **НЕТ.** A3 механизма выдачи и изменения кабинета не несёт. По вердикту K1 кабинет **выдаёт сервер при авторизации**; вопрос «по чьему распоряжению он меняется» — **открытый**, и он не предмет этой комнаты. Запись K1 «владелец меняет через A3» опиралась на обещание, которого здесь нет, и подлежит уточнению. |

## Уточнение к ратифицированному вердикту K1

Формулировка K1 «кабинет меняет владелец через комнату A3» **исправляется** на: **«выдаёт и меняет сервер; по чьему распоряжению — открытый вопрос, комнатой не закрыт»**. Это не переоткрытие K1: содержательная часть (что перечисляется, чем проверяется граница, что при отказе) остаётся в силе, снимается лишь ссылка на несуществующий механизм.

## Definition of Done

- [ ] Переход «выполнено → принято» существует на сервере как отдельное состояние, а не как побочный эффект закрытия задачи.
- [ ] Подпись приёмки хранит хеш содержимого закрываемой единицы; правка содержимого после приёмки делает подпись невалидной.
- [ ] Приёмка **не вызывается заявкой клиента**: вход приёмки — предикат над фактами вне контроля заявителя; попытка запросить приёмку отклоняется как неподдерживаемое действие.
- [ ] `executed.by` и `accepted.by` различимы в записи перехода; совпадение — блокирующая находка.
- [ ] Отказ приёмки логируется в дом заявок наравне с отказами границы (норма K1).
- [ ] Запись K1 о выдаче кабинета уточнена; открытый вопрос «по чьему распоряжению» вынесен явно, а не растворён.

## Список посылок

1. **факт** — Замер по трекеру: 144 активные карточки, 111 в состоянии «сделано» при 9 с отметкой начала работы; 102 перехода прошли мимо работы. Причина — один ключ на всех, автор перехода неотличим.
2. **факт** — Серверные метки времени и порядок переходов подделать нельзя; подлинность труда они при этом не доказывают.
3. **норма** — Т2 шторма: в процедурах агент остаётся без ключей (ратифицировано владельцем).
4. **норма** — Т3 шторма: вызов внешнего API есть заявка; агент формулирует, сервер исполняет.
5. **норма** — Т4 шторма: ответственность переезжает на сервер, `countersign` остаётся нужен.
6. **норма** — Т5 шторма: кабинет есть зона полномочий, не личность.
7. **норма** — Т6 шторма: серверную механику держит Фаррелл.
8. **норма** — Вердикт K1 (ратифицирован): кабинет выдаёт сервер при авторизации; граница проверяется синхронным предикатом на сервере; отказ специфичен и логируется.
9. **норма** — Вердикт M2 (ратифицирован): Фаррелл вызывает глаголы мастерской и логирует результат; неподдерживаемый глагол даёт ошибку с записью.
10. **норма** — Подпись есть пин над содержимым: дрейф содержимого после подписи роняет подпись (формализация Математика, два круга).
11. **факт** — Механизм «спросить у сервера, какая пара установлена» существует: overlay каналов забирается с office по адресу и токену. Локальная правка реестра резолвером не читается — проверено прогоном.
