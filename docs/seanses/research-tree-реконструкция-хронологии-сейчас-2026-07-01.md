Иначе получится 500 точек в графе и невозможно ориентироваться. Гранулярность: **по PR с меткой `rt-*` или явному входу в новый stage (S0, S1, S2 и т.д.)**. Риск неточных дат решается аннотациями в комментариях узлов: если даты нет — пишем `date_uncertainty: "между PR #219 и #220"`. ## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| **Приоритет: реконструкция vs UI-таймлайн?** | Реконструкция (git-log + ручная валидация) в первую очередь; UI-таймлайн — после заполнения transitions.json данными. |
| **Гранулярность переходов** | По PR с меткой `rt-*`, по явным эпохам (S0–S5), по добавлению детекторов. **Не более 1–2 переходов в неделю**. Не каждый коммит. |
| **Как работать с неточными датами?** | Аннотация в структуре: `source.note`, `confidence < 1.0`. Скрипт: confidence = 1.0 для тегов, 0.9 для grep-совпадений, 0.7 для умозаключений. |
| **Контракт Transition** | Interface в `@membrana/core/types/research-tree.ts`: id, timestamp, label, type (`epoch` / `feature` / `fix` / `refactor`), source (commit SHA или inferred), confidence (0.8–1.0), description. |
| **Состав спринта** | RT-RECON-0 (контракт, Структурщик), RT-RECON-1/2 (скрипты, Математик), RT-RECON-3 (данные, Teamlead), RT-RECON-4 (UI, Верстальщик), RT-RECON-5 (интеграция, Структурщик). |

| **UI-таймлайн (когда данные есть)** | Горизонтальная полоса, фильтры по type (иконки, toggle), popover при hover, адаптив (мобильная свертка). Компонент: `apps/client/src/modules/research-tree/TimelineComponent.tsx`. |
