## Вопрос M3 (ровно один)

**S1 — Как реализовать стендап как персональный акт Тимлида: план разработки по Day Issue,
handoff-кеш и активация памяти исполнителей, стык с движком задач?**

Решаем только S. Не залезать в доклад (R) и сны (D). Ратифицирует владелец.

**Требование к вердикту (обязательно):** вердикт обязан нести секцию **«Список посылок»** —
явные входы (закрытые вердикты M0/M1/M2; спека стендапа; ограничение Linear). Решение без
названного списка посылок не принимается.

## Грани вопроса S (простым текстом, не под-вопросы)

- **Автор — Тимлид (Vesnin) лично.** Не безличный скрипт-свалка «ведёт X, ещё 85». По M1
  автор ∈ {5 персон, человек}, субагент автором быть не может → стендап реально пишет
  персона Тимлида. Что значит «персонально» технически.
- **Вход — Day Issue (каркас K).** Тимлид планирует разработку по существу по 5-блочному
  плану (магистраль + подкрепление). Как читает каркас, что берёт.
- **Handoff-кеш каждому исполнителю.** Смыкается с контуром исполнения («кеш `current_task`
  у каждого исполнителя, обновляется с началом задачи»). Что в кеше, как формируется.
- **Активация участков памяти.** Как Тимлид «активирует определённые участки памяти»
  исполнителя под конкретные задания (срез журнала персоны? предзагрузка контекста?),
  чтобы исполнитель был готов.
- **Стык с движком задач.** На этапе стендапа Тимлид либо **заводит новые Linear-сценарии
  исполнения**, либо **привязывает исполнителей к задачам** — что когда. Ограничение:
  Linear гео-блокирован для RU (403; egress = туннель media-VPS NL; поле делегата
  `delegate`).
- **Стендап = отчёт о готовности команды** через призму команды: соотношение плана и
  инструментов разработки. Форма отчёта.
- **Стык с Ангелиной (M1).** Стендап — документ каскада → обязан эмитить проверяемую
  структуру (обязательные маркеры), провенанс `{author=Тимлид, guard, digest, readAt}`,
  чтобы страж судил свежесть/структуру. Автор пишет, Ангелина проверяет — граница ролей.

## Посылки-подсказки (для вердикта, не сам вердикт)

- M0 (ратифицирован): S после K, зависит только от K.
- M1 (ратифицирован): автор ∈ {5 персон, человек}, субагент нельзя; провенанс
  `{author, guard, digest, readAt}` обязателен; `stale` → громкий стоп.
- M2 (ратифицирован): план — 5 детерминированных слотов; стендап читает магистраль+
  подкрепление из этого каркаса.
- Спека владельца: стендап = персональный акт Тимлида, handoff-кеш + активация памяти;
  на этапе стендапа — стык с движком задач (Linear-сценарии / привязка исполнителей).
- Перевод команды из «заявлена, но не исполняет» в готовый субъект действия.
