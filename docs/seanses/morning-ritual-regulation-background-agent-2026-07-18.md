[Teamlead]: Рамку фиксирую сразу, чтобы не сползти в контрабанду вопросов. Заседание одновопросное (канон владельца): мы регламентируем утренний ритуал и определяем **границу делегирования** фоновому агенту. Не решаем «сколько веток», «нужен ли sprintKind» — это отдельные mf-задачи. Мой профиль «форма следует функции» даёт несущий тезис: у утра две несовместимые функции — **механика** (детерминированные шаги, воспроизводимые без человека) и **диалог** (владельческие решения). Их надо развести физически: одно в фон, другое — на передний план. Гейт нельзя делегировать по определению.

[Структурщик]: Начну с термина, иначе DAG поплывёт. У нас слиплись две леммы под словом «шаг утра»: **механический шаг** (`f(state) → artifact`, детерминированный, идемпотентный) и **гейт** (`ask(owner) → decision`, требует живого субъекта). Это разные словарные статьи, и от классификации КАЖДОГО из восьми шагов 18.07 зависит его адрес: фон или диалог. Предлагаю несущую норму: шаг делегируем фону ⟺ он чистый и его выход проверяем без владельца. Всё, что меняет мир по владельческому «да», — не делегируется никогда.

[Верстальщик]: Со стороны читаемости артефакта — у фонового агента есть фундаментальный риск **мёртвой двери**. Если механика идёт в фоне, а владелец в это время ведёт диалог, то где он видит, что фон вообще работает и не проглотил гейт? Мне нужна витрина статуса фона: одна строка-акцент «фон: шаг 3/8, ожидает вашего решения по X». По моей норме — молчаливый фон недопустим: каждый гейт, до которого дошёл фон, поднимается как видимый alert с ручным закрытием, как `CaptureAlertToasts`. Не автоскрытие — переход контроля всегда озвучен.

[Математик]: Формализую, иначе «безопасно делегировать» останется вкусовым. Каждый из восьми шагов — это функция с сигнатурой. Делимость на фон определяется предикатом `pure(step) ∧ verifiable(output)`: шаг чист (нет запроса к владельцу внутри) И его выход проверяется детерминированной функцией (probe, тест, diff). Если внутри шага есть `ask(owner)` — предикат ложен, шаг остаётся в диалоге. Это тот же критерий, что мы вывели для code-review 16.07: вход детерминирован ⟹ можно автоматизировать. Гейт недетерминирован по субъекту ⟹ нельзя.

[Музыкант]: По касательной к аудио, но ритм я узнаю — это два клока, как источник и синк в аудио-графе (я это флагал в strategy-day-q3). Фон — быстрый клок механики, диалог — клок владельца. Если они бьют в разном темпе без синхронизации, на стыке «щелчок»: фон убежал вперёд и проскочил гейт, пока владелец думал. Нужен barrier: фон доходит до гейта и **останавливается**, ждёт владельца, не крутит дальше. Магическая кнопка «всё сделал за ночь» без честной остановки на гейтах — это трек, записанный в пустой зал.

[Teamlead]: Принимаю формализацию Математика как несущую и добавляю классификацию восьми шагов 18.07 по профилю. Механика (фон-кандидаты): подтяжка состояния из git, probe посылок MAIN_DAY_ISSUE, code-review диффа за период, сбор реестра активных задач, генерация черновика стратегии-горизонта. Диалог (гейты, только передний план): выбор фокуса дня, LGTM формы решения, любое `assign` меняющее мир. Барьер Музыканта — обязательное условие: фон готовит, владелец решает.

[Структурщик]: Согласен с классификацией, но зафиксирую хостинг истины, иначе разъедется. Где живёт список «какой шаг — фон, какой — гейт»? [Верстальщик]: Поддержу манифест с витрины. [Математик]: Уточню инвариант барьера численно, иначе «останавливается» размыто. Барьер — это предикат прогресса: `canAdvance(step) = step.kind == mechanic ∨ (step.kind == gate ∧ hasDecision(step))`. Фон монотонно двигается по `steps[]`, пока `canAdvance` истинно; на первом гейте без решения — `d/dt = 0`, застывает. Это ацикличный проход, как DAG-инвариант из truth-graph 17.07: нет обхода гейта в принципе, потому что `canAdvance` его не пускает. Проверяется чистой функцией без запуска фона.

Фон исполняет механику монотонно, застывает на первом непройденном гейте (барьер Математика), поднимает видимый alert (витрина Верстальщика), каждый шаг оставляет провенанс с origin-hash (Музыкант). Второй инвариант: идемпотентность механики — повторный прогон шага на той же ревизии даёт тот же артефакт (`f(state) == f(state)`). Третий: каждый лог-эход несёт origin-hash. ## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Регламентировать утренний ритуал? | **Да** — как исполняемый манифест `steps[]`, а не только prose-регламент |
| Что делегируем фону? | **Только механику**: шаг чист (`pure`) И его выход проверяем без владельца (`verifiable`) — подтяжка git-состояния, probe посылок, code-review диффа, сбор реестра, черновик горизонта |
| Что НЕ делегируем никогда? | **Гейты**: выбор фокуса дня, LGTM формы, любой `assign`, меняющий мир. `ask(owner)` внутри шага ⟹ шаг не делегируется |
| Как фон не проглотит гейт? | **Барьер**: `canAdvance(step) = mechanic ∨ (gate ∧ hasDecision)`. Фон монотонно идёт по `steps[]`, застывает на первом гейте без решения (`d/dt=0`) |
| Как владелец видит фон? | **Витрина** (лента из 8 тиков: done/active-gate/disabled, `aria-live`) + **мостик в диалоге** (Музыкант вплетает барьер в живой разговор) |
| Провенанс | Каждый механический шаг оставляет лог с **origin-hash** ревизии, на которой отработал (против эхо-камеры) |
| Граница пакета | Фон — office-контур (рядом со `scripts/*.mjs`), НЕ сервис в `packages/services/*`. Связь с витриной через артефакт, не прямой импорт |

**Definition of Done:**

1. `docs/prompts/MORNING_RITUAL_REGULATION.md` описывает норму, ссылается на манифест как источник истины (без дубля списка шагов в prose).
2. Манифест `steps[] = {id, kind: mechanic|gate, verify, blocksOn}` — исполняемые данные в office-контуре; каждый из 8 шагов 18.07 классифицирован.
3. Барьер `canAdvance` реализован; **тест-инвариант**: полный перебор состояний решений — фон НИКОГДА не проходит гейт без `hasDecision`.
4. Идемпотентность механики покрыта тестом: `f(state) == f(state)` на одной ревизии.
5. Каждый шаг фона пишет лог с origin-hash; тест на наличие провенанса.
6. Витрина статуса фона (Rodchenko): 8 тиков с состояниями done/active-gate/disabled, empty-state «Утро не запущено», ни одного молчаливого/кликабельного-заглушкой гейта, `aria-live="polite"` на смену стабильного статуса.
7. Мостик диалог↔фон: при упоре фона в гейт барьер озвучивается в живом диалоге с владельцем (не только в UI).
8. Границы связанности: фон не импортирует плагины/UI напрямую; probe переиспользует `main-day-probe`; LGTM Teamlead перед включением фона в реальный ритуал.
