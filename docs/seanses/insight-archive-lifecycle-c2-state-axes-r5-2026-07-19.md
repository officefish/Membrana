# Консилиум: C2R4 — замыкание L/O enums и supersede-модели

**Повестка:** Выбрать ровно два остатка C2 — (1) дословные positive/negative enums для осей L (delivery/SLICE) и O (outcome/SLICE) с семантикой `None` вне enum; (2) одну supersede-модель без «или». Канонические D/L/O/V и вся замороженная основа не пересматриваются. C3–C7 не решаются.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант (Kuryokhin), Верстальщик (Rodchenko).

**Порядок реплик:** Математик → Структурщик → Верстальщик → Teamlead → Музыкант

[Математик]: Формализую, иначе enum поплывёт. Обе оси по замороженной основе — `Option<assessment>`, где `None` строго вне enum и означает «нет assertion», а не «отрицательный assertion». Значит каждая ось — это тип-сумма мощности 3: `None ⊕ positive ⊕ negative`. Задача сегодня — назвать ровно два непустых значения на ось, чтобы пара была exhaustive и mutually exclusive. Никакого третьего непустого значения — иначе разрушим биекцию с product из основы.

[Структурщик]: Начну с термина, иначе спор поплывёт. L и O — две разные леммы, их positive/negative нельзя называть синонимами. L — про **факт доставки среза** (SLICE был сдан или нет), O — про **результат среза** (то, что сдали, дало эффект или нет). Если для обеих взять `delivered/realized` вперемешку — получим синонимы-слои, запрет из моего профиля. Предлагаю дословно: `L = Option<{delivered, not-delivered}>`, `O = Option<{realized, not-realized}>`. Термины не пересекаются, каждый однозначен в своём домене.

[Верстальщик]: С точки зрения потребителя проекции — а потребитель это агент и человек в архиве — эти четыре слова должны читаться без легенды. `delivered`/`not-delivered` и `realized`/`not-realized` дают ровно то, что нужно моей норме «функция видна сразу»: по значению понятно, о чём ось, без сноски. Важно, что `None` на витрине — это **пустая ячейка / «—»**, а не «отрицание». Три визуальных состояния на ось: значение-плюс, значение-минус, прочерк. Это ложится в честную таблицу без мёртвых дверей.

[Teamlead]: Рамку фиксирую сразу, чтобы repair не сполз в C3. Мы не решаем, *как* устанавливается `delivered` (это критерий, C3) и не вводим terminal/monotonic семантику (это тоже за периметром). Мы решаем ровно два имени-пары и одну модель supersede. По конструкции «форма следует функции» пара Структурщика оправдана: два домена — два разных словаря, декора нет. Принимаю `L = Option<{delivered, not-delivered}>`, `O = Option<{realized, not-realized}>` как несущий каркас. Возражения по именам — сейчас.

[Музыкант]: По касательной, аудио тут не задет, но пару positive/negative я узнаю — это как «сигнал прошёл в шину / не прошёл» против «сигнал дал слышимый эффект / не дал». Delivery — это роутинг дошёл до синка; outcome — на выходе реально звучит. Разные вещи, Структурщик прав. И флаг из опыта эхо-камеры: не сливайте `not-delivered` с `None`. Молчание (`None`) — это «не записывали», а `not-delivered` — «записывали, вышла тишина». Это `n=1` против осмысленного нуля, разный вес.

[Математик]: Подтверждаю численно различие, которое назвал Музыкант. `None` — отсутствие свидетельства (undefined в домене), `not-delivered`/`not-realized` — свидетельство отрицательного исхода (defined, значение false-подобное). Это три различимых состояния, а не два. Инвариант: `|L| = |O| = 3` как размеченных состояния, из них 2 внутри enum, 1 (`None`) — вне. Product из основы `D_opt × L_opt × O_opt × V_opt` остаётся корректным: каждый множитель — `Option` конечного enum, число комбинаций конечно и exhaustive.

[Структурщик]: Тогда закрою вопрос симметрии, иначе кто-то потом добавит третий член. Обе пары — строго **бинарные positive/negative**, никаких `partial`, `pending`, `in-progress`. Если понадобится «частично» — это не значение оси, это отсутствие assertion (`None`) до C3-критерия. Моя норма: один термин — один смысл. `delivered` = сдан полностью по критерию (критерий — C3), иначе `None` пока критерий не применён, либо `not-delivered` как явный отрицательный assertion. Пара замкнута, синонимов нет.

[Верстальщик]: Принимаю бинарность — на витрине три состояния читаются чище двух-с-половиной. Одно уточнение по отображению, не по решению: `None` не должно выглядеть одинаково с `not-delivered`. По DESIGN — прочерк `text-base-content/40` для `None`, а `not-delivered` — явный бейдж (например `badge-ghost` с текстом). Это не расширяет enum, это честность мёртвой двери: пустое и отрицательное различимы глазом. Само решение C2 не трогаю.

[Teamlead]: Верно, и это не наш вердикт — это заметка для C-visibility/представления, фиксируем как non-binding. Первое решение считаю принятым: L и O — бинарные `Option`-enum по Структурщику. Переходим ко второму остатку — supersede. Напоминаю рамку: **выбрать ровно одну модель, без «или»**, без terminal/monotonic, без generic assertion-axis. Кандидаты из брифа: (A) event-only link `supersede(oldDecisionAssertion, successorRevision)` без нового D-value; (B) добавить `superseded` в D enum. Мнения.

[Математик]: Считаю, а не голосую по вкусу. Вариант B добавляет пятое значение в D enum (`proposed|accepted|rejected|deferred` → +`superseded`). Это меняет **замороженную основу**: D-enum зафиксирован в C1, трогать его = переоткрывать решённое, запрещено сегодня. Вариант A — чистое ребро в графе, не меняет ни один axis-enum: `supersede: (oldAssertion, successorRevision) → link`. Инвариант ацикличности сохраняется, если successor всегда новее (revision monotone по времени, но это свойство данных, не семантика оси). A дешевле и не ломает заморозку.

[Структурщик]: Согласен с Дыниным по термину. `superseded` внутри D — это смешение двух лемм: D — это **assessment решения** (какое оно), а supersede — это **отношение между двумя ревизиями** (связь). Отношение не должно жить как значение атрибута — это моя базовая норма, ребро графа не кодируется полем узла. Вариант B создал бы синоним-слой: одно и то же «решение заменено» выражалось бы и через ревизию, и через enum-значение. Event-only link (A) — это ссылка на статью, а не переписанная статья. Беру A.

[Верстальщик]: С витрины A тоже честнее. Если `superseded` — значение D, старое решение на экране «перекрашивается» и теряет свой исходный assessment (`accepted` превращается в `superseded`) — это перезапись истории, мёртвая дверь: читатель не видит, чем решение *было*. При event-link старая ассерция остаётся `accepted` и рядом рисуется стрелка «заменено ревизией N». Иммутабельность архива из основы соблюдена визуально: ничего не затёрто. Голосую за A.

[Teamlead]: Конструкция сходится. B нарушает две несущих балки основы сразу — иммутабельность (перезапись assessment) и заморозку D-enum. A — ребро без нового значения, ровно то, что основа разрешает: «reopen new revision / no transcription». Принимаю **A: event-only link `supersede(oldDecisionAssertion, successorRevision)` без нового D-value**, без terminal/monotonic семантики и без generic assertion-axis. Музыкант, последнее слово — риск-флаг есть?

[Музыкант]: Риск-флаг один, из опыта декоррелированности: supersede-link не должен превратиться в generic «связь чего угодно с чем угодно». Бриф это и запрещает — только `(oldDecisionAssertion, successorRevision)`, домен ребра узкий, привязан к D. Пока ребро именно такое типизированное, а не «assertion→assertion вообще» — я спокоен, эхо-камеры из одного решения-многих-ссылок не будет. Принимаю A. Аудио-возражений нет, зона не задета по существу.

[Математик]: Тогда фиксирую замкнутость формально. После двух решений: домены L и O замкнуты (по 2 enum-значения + `None` вне), supersede — единственное ребро без axis-мутации. Product `D_opt × L_opt × O_opt × V_opt` остаётся exhaustive: ни одна ось не расширена, D-enum неизменен. Никаких C3-критериев, никакой terminal/monotonic семантики не введено. Основа инвариантна — проверяемо: diff по axis-enum'ам = 0, добавлен только один link-тип вне осей.

[Структурщик]: Подтверждаю замкнутость по словарю. Четыре леммы (D/L/O/V) не тронуты в определениях; L и O получили дословные значения без синонимов; supersede — ссылка на статью, не новая статья. Граф без циклов, exhaustive, публичный смысл каждого термина однозначен. Возражений по связанности нет — ни одно решение не создало прямую зависимость там, где нужна ссылка.

[Teamlead]: Тогда выношу единый вердикт. Оба содержательных решения приняты всеми ролями, заморозка не нарушена, C3–C7 не затронуты. Формирую блок вердикта — далее председатель делает mechanical extraction в `C2_VERDICT.md` без новых содержательных решений.

[Музыкант]: Принимаю итог. Оба звучат чисто, без щелчка на стыке заморозки.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| **L (delivery/SLICE) enum** | `L = Option<{delivered, not-delivered}>`. `delivered`/`not-delivered` — бинарная positive/negative пара; `None` вне enum = отсутствие assertion (не отрицание). |
| **O (outcome/SLICE) enum** | `O = Option<{realized, not-realized}>`. `realized`/`not-realized` — бинарная positive/negative пара; `None` вне enum = отсутствие assertion. |
| **Различие `None` vs negative** | `None` = нет свидетельства (вне enum); `not-delivered`/`not-realized` = явный отрицательный assertion (внутри enum). Три различимых состояния на ось; enum-значений — 2. |
| **Supersede-модель (единственная)** | **A — event-only link** `supersede(oldDecisionAssertion, successorRevision)` без нового D-value. Без terminal/monotonic семантики, без generic assertion-axis. |
| **Почему не B** | Добавление `superseded` в D enum нарушает заморозку D-enum (C1) и иммутабельность архива (перезапись исходного assessment). Отклонено. |
| **Замкнутость** | После двух решений домены L/O замкнуты, D-enum неизменён, product `D_opt × L_opt × O_opt × V_opt` остаётся exhaustive; вся замороженная основа инвариантна. C3–C7 не решались. |
| **Non-binding заметка (V/представление)** | Визуально `None` (прочерк) и `not-*` (явный бейдж) должны различаться — вне периметра C2, для C-visibility. |

**Definition of Done (для mechanical extraction в `C2_VERDICT.md`):**
- Вердикт содержит **только** два решения выше + подтверждение замкнутости и неизменности основы.
- Дословные enum: `L = Option<{delivered, not-delivered}>`, `O = Option<{realized, not-realized}>`; `None` явно определён как вне enum = отсутствие assertion.
- Supersede зафиксирован как единственная модель A (event-only link), без «или», без terminal/monotonic, без generic assertion-axis.
- Premises, карта C3–C7 и исправленный footer перенесены без новых содержательных решений.
- Diff по axis-enum'ам относительно замороженной основы = 0 (только добавлен link-тип вне осей).
