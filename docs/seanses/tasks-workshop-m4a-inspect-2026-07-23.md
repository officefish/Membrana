# Консилиум: inspectElement для карточки задачи

**Повестка:** Определить контракт функции `inspectElement` для карточки в реестре задач: минимальный состав паспорта, решение по сетевому доступу (живое состояние vs кэш), база рекурсии при спуске в дочерние карточки эпика.

[Верстальщик]: Начну с потребителя — того, кто смотрит `inspectElement` в PR и в CLI. Паспорт карточки должен быть **видим без раскопок**: ID, слаг, статус (live/archived/rejected), ответственный явно, эпик-родитель если есть. Связи снаружи — GitHub-issue, Linear-issue, промпт — это метаданные; показывать ссылку на них, но живое состояние не нужно. Сеть затягивается, если мы каждый `inspect` лезём в Linear; кэш из реестра достаточен.

[Teamlead]: Рамку фиксирую. Функция — read-only, детерминированная, вход = карточка из `registry.json`, выход = структурированный паспорт. Сеть запрещена: вердикт консилиума от 19.07 требует чистоты гейтов, и `inspectElement` — тот же гейт, он служит аудиту. Живое состояние Linear появляется только если владелец явно просит `--refresh`, но то уже другая команда. Минимальный паспорт: ID, слаг, статус, ответственный, эпик-родитель (если есть), временная метка последнего обновления в реестре. Ссылки (github, linear, prompt) как URN без разрешения.

[Математик]: Формализую, чтобы рекурсия не поехала. Паспорт карточки `C` — кортеж `(id, slug, status, owner, parentEpicId?, updatedAt, links[])`. Дочерние карточки возвращаются ТОЛЬКО если `parentEpicId` существует и сама карточка имеет статус live. База рекурсии: `depth ≤ 2` (карточка → дочерние ← внуки), иначе бесконечный граф эпиков создаст DDoS CLI. Предикат `hasChildren(C) := ∃ C' : C'.parentEpicId = C.id ∧ C'.status = live`. При depth=2 возвращаем `{self, children[], grandchildren[]}` на каждую дочернюю.

[Структурщик]: Уточню носители и связи. Входы функции: путь к `registry.json` + ID карточки; выход — JSON с заданным контрактом. Ссылка на GitHub-issue резолвится из поля `issues[]` в карточке, на Linear — из `linearId`, на промпт — из `taskPromptFile`. Эти поля уже есть в реестре; живой fetch запрещён. Если карточка-эпик имеет дочерних, их ID лежат в отдельном индексе `epicChildren.json` (или вычисляются единым скэном `registry.json` за O(n), но с кэшем). Структура не требует новых файлов, всё есть.

[Музыкант]: По касательной — аудио тут не задет, но ритм я узнаю, это проверка целостности на разных уровнях громкости. Паспорт — тихое знание (кэш); в CLI выводится честно. Предупреждение: если карточка-эпик `parentEpicId` указывает на несуществующий эпик (сиротство), это ошибка данных, а не молчаливое отрезание. Требую честного маркера в паспорте `orphaned: true`, чтобы аудитор видел де

фект, а не тишину.

[Верстальщик]: Согласен про orphaned-маркер. Визуально в CLI: паспорт выглядит как структурированная таблица, дочерние карточки идут отступом (tree view DaisyUI или простой текстовый indent). Ссылки показываются как URN с иконкой типа (github/linear/prompt), без живого status. Если карточка в статусе rejected или archived — badge `badge-ghost` / `badge-error` рядом с ID, чтобы человек сразу видел мёртвую дверь.

[Teamlead]: Сходится. Финальная форма: `inspectElement(registryPath, cardId, {depth=2, orphanedWarning=true})` → JSON паспорта + дочерние структуры по глубине без live-запросов, никогда не ходит в сеть. Если родитель мёртв (rejected/archived) и у него есть живые дочерние — это нарушение, маркируем как `inconsistent: true` и выводим в stderr. Стандартный вывод — основная карточка + дочерние; предупреждения в stderr; код выхода 0 при success, 1 при несогласованности данных.

[Структурщик]: Уточню реализацию. Функция — это скрипт `scripts/inspect-task-element.mjs`; вход из CLI: `--registry=docs/tasks/registry.json --id=<cardId> [--depth=2]`. Не требует новой зависимости.

[Музыкант]: Поддержу на конструкцию. Честная ошибка важнее молчания: если что-то в реестре сломано, инструмент должен вопить, а не выдавать зелёный паспорт мёртвого.

[Математик]: Уточню контракт дочерних для эпика. Если карточка A — эпик, и у неё есть дочерние {B, C}, возвращаем `{self: A, children: [B, C], grandchildren: {B: [D, E], C: [F]}}` при depth=2. Это позволяет визуализировать дерево одним проходом. При depth=1 — только `{self, children}`.

## Итоговое решение консилиума

| Компонент | Решение |
|-----------|---------|
| **Контракт `inspectElement`** | `inspectElement(registryPath: string, cardId: string, options?: {depth?: 1\|2, orphanedWarning?: bool}) → CardInspection` |
| **Минимальный паспорт** | `{id, slug, status, owner, parentEpicId?, updatedAt, links: {type, urn, label}[], orphaned?, inconsistent?}` |
| **Сетевой доступ** | **Запрещён** — read-only, кэш только из реестра. Живые данные через отдельную команду с `--refresh` (не в scope этого вопроса). |
| **Иерархия дочерних** | При depth=1: только children[]. При depth=2: children[] + grandchildren{} структуру. База рекурсии: живые карточки, статус rejected/archived не спускаются. |
| **Маркеры ошибок** | `[ORPHANED]` если parentEpicId не резолвится. `[INCONSISTENT]` если родитель мёртв, а дочерние живы. Оба видны в основном выводе. |
| **Реализация** | Скрипт `scripts/inspect-task-element.mjs`. Чистая функция внутри, CLI оболочка. Unit + интеграционные тесты на фиксуре. |
| **Вывод (CLI)** | Структурированная таблица (tree view), stdout основной паспорт, stderr предупреждения. Exit code 0 = success, 1 = data inconsistency. |
