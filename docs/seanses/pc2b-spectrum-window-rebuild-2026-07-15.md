## Повестка

PC-2b: спектр-граф перестаёт тащить рекордер как часы. Свап механически ясен
(scout). Открытый design-вопрос — **величина windowMs**, и он связан с субсэмплером
трендов (research вскрыл связь). Продуктовый режим: решаем деталь честно.

### Разведка (scout, ground truth)

Обвязка часов в `usercase-free-spectrum-live`: `collectFrames → windowFull`;
`recorderClock → windowFull` (RecorderRef); `windowFull.true → sequence`;
`windowFull.false → infinity`. Свап: снять windowFull+recorderClock → is-window-elapsed;
перевесить true→sequence, false→infinity, collectFrames→is-window-elapsed; recorder-ребро
не восстанавливать.

### Research (Dynin, WebSearch — с оговорками)

- Окно РЕШЕНИЯ (не FFT-кадр) в акустической детекции дрона кластерится на **1–2 с**;
  расширение 1→2 с стабилизирует дальние цели, на близких почти без изменений.
- Наши 5000 мс — консервативно-длинно (макс. устойчивость тренда, +2–3 с latency).
- **Самосбрасывающееся непересекающееся окно (hop=window) уместно для НАБЛЮДЕНИЯ**
  (не алярма): проще, независимые вердикты; цена — обновление раз в windowMs.
- Каденс вердикта ограничен не вычислением (latency 150–200 мс), а окном накопления.
- Рекомендация research: **windowMs 3000** (при тике 500 мс = 6 кадров ≥ measurementsCount 5).

### КРИТИЧЕСКАЯ СВЯЗЬ (деталь, ради которой продуктовый режим)

Субсэмплер трендов требует **кадров в batch ≥ measurementsCount (=5)** (иначе
`analyzeTrendsFromFftFrames` → null, insufficient-subsample, L11). Кадров = windowMs / tickMs.
Тик ~500 мс (L32, измерен, НЕ гарантирован). Значит:
- windowMs 2000 → 4 кадра < 5 → **ЛОМАЕТ субсэмплер**. Запрещено.
- windowMs 3000 → 6 кадров (запас 1; при тике 600 мс = 5, при 700 мс = 4 → падение).
- windowMs 5000 → 10 кадров (запас против джиттера тика до 1000 мс).

### Спорные точки

- **СТ-1.** windowMs: 3000 (research-оптимум, отзывчивость) vs 5000 (текущее, джиттер-запас) vs среднее.
- **СТ-2.** Самосбрасывающееся непересекающееся окно для наблюдения — подтвердить.
- **СТ-3.** Как защитить связь windowMs↔measurementsCount от тихой поломки в будущем.
- **СТ-4.** Свап (scout) — подтвердить механику и границы.

## Итоговое решение (консенсус)

- **windowMs = 4000** (не 5000, не 3000). Обоснование: research-коридор окна решения
  1–2 с + запас устойчивости для DSP-шаблонного тренда; 5000 избыточны по latency,
  3000 не держит джиттер-запас субсэмплера. 4000/500=8 кадров номинал (запас 3 над
  measurementsCount=5), субсэмплер жив до тика ~800 мс. Живее 5000 для наблюдательного
  борда, стабильнее 3000. Финальная величина — калибровка на живом ране (СБОРКА, не
  детекция; двигать к 5000 при падении субсэмплера, не ниже).
- **Гард-связи ОБЯЗАТЕЛЕН (сердце консилиума):** тест кодифицирует
  `windowElapsedMs / EXPECTED_TICK_MS >= measurementsCount` (EXPECTED_TICK_MS≈500, L32,
  одна константа). Любое будущее изменение windowMs/measurementsCount, ломающее связь,
  падает в CI, а не тихо в поле (L11 insufficient-subsample). Защита детали, вскрытой research.
- **Свап (scout, подтверждён):** removed += {windowFull(is-recording-window-full),
  recorderClock(get-recorder-168)}; is-window-elapsed(windowElapsedMs=4000):
  collectFrames→iwe, iwe.exec-true-out→sequence, iwe.exec-false-out→infinity;
  recorder-ребро НЕ восстанавливать. Непересекающееся самосбрасывающееся окно (СТ-2,
  research-подтверждён; overlap вне scope). Тексты узла/comment-группы — честные
  («окно наблюдения по host-часам, без записи», убрать «IsRecordingWindowFull/таймер»).
  Границы: только usercase-free-spectrum-live + тест; is-recording-window-full и
  collectorConfig.windowSec (F2) НЕ трогать. Живой Run = сборка, дрон не гейт.
