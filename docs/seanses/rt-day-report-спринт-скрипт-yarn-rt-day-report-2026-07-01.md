# Консилиум: RT DAY-REPORT спринт — архитектура, хранение, оценка токенов

**Повестка:** Определить архитектуру скрипта `yarn rt:day-report <date>`: место хранения выходных файлов, способ генерации нарратива DAY_GIT_FLOW, коэффициент токенов для оценки объёма работы по git diff.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант (Kuryokhin), Верстальщик (Rodchenko).

**Порядок реплик:** Структурщик → Музыкант → Верстальщик → Математик → Teamlead (циклически)

---

[Структурщик]: Хранение выходных файлов — это архитектурный вопрос слоёв репозитория. `docs/archive/rt-day-reports/<date>/` логичнее, чем внутри `apps/demos/Research-Tree/`, потому что отчёты — это **операционные артефакты**, а не часть демо-приложения. Демо может переехать, пересоздаться, а отчёты дня должны жить в единой истории спринта. Слой `docs/archive/` уже хранит архивированные задачи — туда же складываем.

[Музыкант]: Согласен на `docs/archive/`, но добавлю деталь: каждый отчёт — это моментальный снимок дня, и он должен быть self-contained. Если потом будет UI, чтобы смотреть отчёты (например, дашборд истории спринтов), то структура вроде `docs/archive/rt-day-reports/YYYY-MM-DD/` позволит легко парсить — просто переберём папочки по дате. Хорошая идея для future-proofing.

[Верстальщик]: С точки зрения UI, если когда-то будет визуализация этих отчётов в клиенте, им нужна машиночитаемая структура. Рекомендую сохранять JSON (структурированные данные) рядом с markdown-нарративом: `DAY_GIT_FLOW.md` + `DAY_COMPONENT_REPORT.json` (или .csv для таблицы). Так легче позже натянуть на граф или таблицу в React без парсинга markdown.

[Математик]: JSON + markdown — грамотно. Но добавлю метрику: для оценки работы через git diff нужна **стабильность коэффициента**. Четыре токена на строку — это предположение без базиса. Предлагаю собрать эмпирику: возьми 5–10 дней из прошлого (если есть git), посчитай lines добавленных/удалённых, оцени вручную сложность задач и сравни с 4× и 6× коэффициентом. Без этого любой выбор — гадание.

[Teamlead]: Математик прав — нужна эмпирика. Но тут важен и продуктовый контекст: **зачем** нам эта метрика? Если это чисто для аналитики (понять, сколько работы в день), то коэффициент 4 достаточен. Если это для **ресурс-плана** или **прогноза спринта**, нужна точность. Для MVP я предлагаю: 1) Хранить в `docs/archive/rt-day-reports/YYYY-MM-DD/`; 2) DAY_GIT_FLOW генерировать **шаблонным нарративом без LLM** (claude-haiku стоит денег и добавляет невоспроизводимость); 3) Коэффициент **4 по умолчанию, но логировать raw lines** — потом легче пересчитать.

[Структурщик]: Шаблонный нарратив — это риск быть неинформативным. Но Teamlead прав про воспроизводимость. Предлагаю компромисс: шаблон с **подстановками** — переходы, узлы, дельта линий, время дня. Например: «Основной фокус на [узел1] и [узел2]: +[lines] строк за [time]. Побочные: [узел3]…» Это не LLM, но информативнее, чем пустой лист.

[Музыкант]: Шаблон подстановок — хороший компромисс. И технически это проще: скрипт парсит transitions[], группирует по узлам, вычисляет метрики, вешает в шаблон. Без API-вызовов, без задержек. Вопрос про формат выхода: DAY_GIT_FLOW.md — это основной файл для чтения человеком? Или это машиночитаемое резюме?

[Верстальщик]: Оба, я думаю. DAY_GIT_FLOW.md — это **читаемый нарратив** (для review в team evening feedback), DAY_COMPONENT_REPORT.json — это **данные для графиков**. В team evening можно проецировать сводку из JSON в таблицу на слайд / в консоль. Простая структура: `{ date, nodes: [{ name, linesAdded, linesRemoved, tokens, timeRange, commitCount }], totalTokens, narrative }`.

[Математик]: JSON-схема понятна. Но вернёмся к коэффициенту 4. Если мы кодируем в lines × 4, нужно договориться: это **строки кода** или **строки в diff** (включая контекст)? git diff выдаёт контекст (@@) — его считаем или только +/–? Предлагаю: считаем только `+` и `–` строки (игнорируем контекст), и за базу берём 4 токена на добавленную/удалённую строку.

[Teamlead]: Договоримся: **raw added + deleted lines, коэффициент 4**. Переход на 6 будет, только если team evening feedback покажет занижение. Логируем raw count, потом пересчитываем легко. Шаблонный нарратив — утверждаю, структурированный JSON рядом — утверждаю. Третий пункт: **где хранить скрипт?** Это `scripts/rt.mjs` или отдельный файл?

[Структурщик]: `scripts/rt.mjs` логично — рядом с другими утилитами спринта. Или, если скрипт вырастет, `scripts/rt-day-report.mjs` + `scripts/rt/helpers.mjs` для парсинга git. Но на MVP достаточно одного файла в `scripts/`.

[Музыкант]: И добавить команду в `package.json` scripts: `"rt:day-report": "node scripts/rt-day-report.mjs"`. Плюс хелпер: `yarn rt:day-report --help` выдаёт формат даты и примеры.

[Верстальщик]: Согласен. Плюс валидация на входе: если дата неправильная (не YYYY-MM-DD или дата в будущем), скрипт выдаёт ошибку и показывает help. Пользователю понятно, что произошло.

[Математик]: Валидация даты — хорошо. Добавлю: логирование на stdout (прогресс парсинга git) и финальное резюме (всего узлов, всего строк, всего токенов). Для отладки полезно.

[Teamlead]: Утверждаю. Итоговое решение формирую в следующем блоке.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| **Место хранения выходных файлов** | `docs/archive/rt-day-reports/YYYY-MM-DD/` (в `docs/archive/` вместе с другими операционными артефактами спринта) |
| **Структура файлов в папке** | `DAY_GIT_FLOW.md` (читаемый шаблонный нарратив с подстановками) + `DAY_COMPONENT_REPORT.json` (структурированные метрики узлов) |
| **Генерация DAY_GIT_FLOW** | **Шаблонный нарратив без LLM** — переходы→узлы, дельта lines, время дня, коммиты вшиваются в шаблон. Формат: «Основной фокус на [узлы] и [узлы]: +[lines] строк за [time]. Побочные работы: [узлы]…» |
| **Коэффициент токенов** | **4 на строку** (добавленная или удалённая в git diff, контекст исключается). Логируем raw count; переход на 6 — только по итогам team evening feedback |
| **Расположение скрипта** | `scripts/rt-day-report.mjs` или `scripts/rt.mjs` (если объединяется с другими rt-утилитами) |
| **Команда в package.json** | `"rt:day-report": "node scripts/rt-day-report.mjs <YYYY-MM-DD>"` с флагом `--help` |
| **Валидация входа** | Проверка формата даты (YYYY-MM-DD), отклонение будущих дат; ошибка → help |
| **Логирование на stdout** | Прогресс парсинга git + финальное резюме (узлы, lines, tokens) |

**Definition of Done:**

- [ ] Скрипт `scripts/rt-day-report.mjs` создан и протестирован на 2–3 датах из git-истории.
- [ ] JSON-схема `DAY_COMPONENT_REPORT` задокументирована (комментарий в скрипте или `docs/RT_DAY_REPORT_FORMAT.md`).
- [ ] Шаблон DAY_GIT_FLOW вёрстан и протестирован (минимум 3 дня).
- [ ] Команда `yarn rt:day-report` работает; `yarn rt:day-report --help` выдаёт подробку.
- [ ] Первый отчёт за день прогонен и сохранён в `docs/archive/rt-day-reports/YYYY-MM-DD/`; код залит в `vesnin` (Teamlead LGTM).
- [ ] team evening feedback даёт обратную связь по коэффициенту; при необходимости переводим на 6 и логируем решение в ADR.
