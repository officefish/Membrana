# Консилиум: уточнённая формулировка PINNED_SUBGRAPH_VERSIONING — сабграф под контейнером, кит и фрейм

**Повестка:** F4 — как снять оговорку «контейнеры — вне области» в части сабграфа ПОД контейнером и как зафиксировать кит и фрейм как доменные имена родового узора PINNED_SUBGRAPH_VERSIONING.

[Teamlead]: Открываю сток графа M0 и сразу оглашаю рамку — мы не расширяем область на сам контейнер (T9 держит оговорку для списка/реестра), не проектируем новые домены узора, не пересматриваем M1–M3. Наш вопрос ровно один: точная редакция оговорки, разделяющая «сам контейнер» и «сабграф под ним», плюс таблица доменных имён кит/фрейм. По «форма следует функции»: паттерн формализует прожитое, не наоборот — прецедент M1–M3 теперь есть, значит снятие оговорки законно, а не теоретично. Список посылок оглашу отдельно, по регламенту он обязателен.

[Математик]: Формализую, иначе «сабграф под контейнером» останется вкусовым. Есть отношение `supports(g, c)`: подграф `g` поддерживает контейнер `c`, но не есть сам список `c`. Предикат области паттерна должен быть точным: `versionable(x) = supports(x, c) ∧ ¬isContainerList(x)`. Оговорка T9 — это ровно `isContainerList(c) → ¬versionable(c)` (снимковая дисциплина), при этом `supports(g,c) ∧ ¬isContainerList(g) → versionable(g)`. Две строки, взаимоисключающие по `isContainerList`. Никакого противоречия: контейнер-список вне области, сабграф под ним — в области.

[Структурщик]: Начну с термина, иначе снятие оговорки поплывёт. «Контейнер» в текущем тексте — омоним двух лемм: (1) **сам список/реестр** — снимковая дисциплина, каждая ревизия — целый снимок, пина отрезков нет; (2) **сабграф ПОД контейнером** — поддерживающая структура (провода, отрезки), которую паттерн версионирует. Оговорка «контейнеры вне области» писалась про лемму (1) и по неаккуратности накрыла (2). Снятие = не «отменить оговорку», а **расщепить омоним**: оговорка остаётся дословно для леммы (1), для леммы (2) вводится явное разрешение. Это ровно формализация Математика: `isContainerList` — граница между двумя статьями словаря.

[Верстальщик]: С точки зрения потребителя паттерна — а потребитель здесь следующий агент, читающий текст как топливо, и человек в PR — оговорка обязана быть **видна первой строкой и не двусмысленна**. Сейчас жёсткий текст стоит дважды: в шапке и в «Границах применимости». Моя норма честности: если снятие спрятано в абзаце, агент прочтёт шапку «контейнеры НЕ версионируются» и откажется пинить фрейм — ровно провал, который F4 закрывает. Требую: шапку переписать на разведённую формулировку, а таблицу доменных имён дать **видимым блоком** — две строки, кит и фрейм, с колонками «домен / что пинится / SHA». Не только текст — таблица, чтобы различие читалось глазом.

[Музыкант]: По касательной, аудио тут не задет, но родовой узор я слышу как один эффект с двумя пресетами. `PINNED_SUBGRAPH_VERSIONING` — это ревербератор; кит и фрейм — два пресета того же алгоритма, отличаются только гранулярностью тапа: кит пинит целый файл (длинный хвост), фрейм — отрезок (короткий тап). Мой флаг мёртвой двери: если в тексте останется старая шапка рядом с новой таблицей — это как WAV и забытый MP3, две копии правила разъедутся молча. Требую: одна точка истины на оговорку, зеркало в GROUP_CONTAINERIZATION — ссылкой, не копией.

[Teamlead]: Принимаю расщепление омонима Структурщика как несущую конструкцию редакции. Оглашаю **Список посылок** вслух, до споров по деталям — регламент требует, и шесть комнат до нас падали, не назвав основания. Вердикты M0 (порядок, сток = F4), M1 (контракт frames), M2 (пин отрезка), M3 (утренний фрейм) — легальные посылки-предшественники, все норма. T7, T9 шторма 22.07 — норма владельца. Текущий двойной текст оговорки и зеркало в GROUP_CONTAINERIZATION — факт по коду. Прожитые киты `angelina-morning`, `dream-master` и фрейм из M1–M3 на ядре `auditPins` — факт.

[Математик]: По зеркалу в GROUP_CONTAINERIZATION формализую выбор «править синхронно vs одна точка истины». Пусть `copies(rule) = n`. Инвариант рассинхрона: при `n ≥ 2` и независимой правке `P(divergence) → 1` со временем — это доказанный эхо-механизм («Ссылка вперёд», прецедент MEETING_REGULATION). Единственное устойчивое решение — `n = 1`: одна каноническая формулировка в PINNED_SUBGRAPH_VERSIONING, GROUP_CONTAINERIZATION несёт **ссылку**, не копию. «Править синхронно» — это `n=2` с ручной дисциплиной, а ручная дисциплина против энтропии проигрывает. Выбираю ссылку.

[Структурщик]: Поддержу `n=1` и уточню носитель ссылки. GROUP_CONTAINERIZATION в «Границах применимости» вместо «К контейнерам он не применяется» несёт: «Область версионирования сабграфов — см. единую формулировку в PINNED_SUBGRAPH_VERSIONING §<оговорка>». Это ссылка на статью, а не копия абзаца — ровно моя норма «импорт = ссылка на словарную статью, не на случайный абзац». Таблицу доменных имён держу тоже в **одной** точке (PINNED_SUBGRAPH_VERSIONING); признак заведения нового домена формулирую словарно: новый домен заводит своё имя, когда его гранулярность пина отлична от уже названных (файл — кит, отрезок — фрейм; третий домен — только при третьей гранулярности).

[Верстальщик]: Приму таблицу доменных имён как главный diffable-артефакт правки. Колонки строго: `домен` (кит / фрейм), `слой` (скрипты / процедуры), `что пинится` (файл целиком / отрезок), `механизм` (`auditPins` + resolver). Признак Структурщика «новая гранулярность → новое имя» ставлю подписью под таблицей, курсивом — чтобы следующий агент видел правило расширения, а не гадал. По шапке: старую строку «контейнеры НЕ версионируются» заменяю на две явные — «список/реестр контейнера — снимковая дисциплина, вне паттерна» и «сабграф под контейнером — законный адресат паттерна». Первое, что читает агент — обе стороны, без ловушки.

[Музыкант]: Флаг по чеклисту соответствия — пункт «пины — git SHA; копий файлов нет» при вводе segmentHash. Отрезок нельзя пинить одним SHA файла: правка соседних строк дрейфует SHA файла, но не сам отрезок (K1-шторм, доказанный факт). Значит чеклист получает уточнение: для кита пин = git SHA файла, для фрейма пин = (git SHA файла + segmentHash отрезка + якорь). Копий файлов по-прежнему нет — segmentHash это дайджест содержимого отрезка, а не копия. Иначе тап поедет, и фрейм зазвучит по старой партитуре.

[Teamlead]: Принимаю уточнение Музыканта в чеклист — но фиксирую границу: сам механизм пина отрезка уже решён в M2, мы здесь только **отражаем** его в чеклисте паттерна, не переоткрываем M2. Это отражение, не проектирование. Согласны все?

[Математик]: Согласен — `verify(frame) = validGitSHA(file) ∧ segmentHash(anchor) = stored ∧ resolveSegment ≠ ∅`. Это предикат проверки, живёт в M2, в паттерне лишь строка чеклиста «пин отрезка = SHA файла + segmentHash; копий нет». Отражение, не новый алгоритм. Принимаю.

[Структурщик]: Согласен, границу M2 держу. Добавлю термин в таблицу, чтобы не было синонимов-слоёв: «пин» у кита и «пин» у фрейма — одна лемма (версионируемая ссылка на прожитое состояние сабграфа), два значения гранулярности. Не заводим два слова на одно понятие — заводим два **значения** одного слова, различённые колонкой «что пинится». Это чистый Ожегов: одно слово — один смысл в контексте, контекст задаёт домен.

[Верстальщик]: Согласен. Свожу подачу в один блок PINNED_SUBGRAPH_VERSIONING: (1) разведённая шапка, (2) таблица доменных имён с подписью-правилом, (3) строка чеклиста про segmentHash. GROUP_CONTAINERIZATION — только ссылка. Три видимых артефакта в одном PR, дифф читается по секциям — подарок ревьюеру.
