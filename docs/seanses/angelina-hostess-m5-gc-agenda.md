# Заседание «Ангелина — хозяйка утра» — M5 (GC: затирание следов альтернативных сценариев)

**Объединяющее задание:** утро не воспроизводится старым; гейты — механика; ведёт хозяйка.
Порядок (ратифицирован, с поправкой владельца): C → B → G → H → GC. Четыре закрыты и
ратифицированы. Сейчас — финальный GC, вершина DAG: затирать можно только когда новый мир
полностью встал (все вердикты закрыты) и абсорбция завершена (H). Поправка-основание:
BRIEF_AMENDMENT (слово владельца: «нужен garbage collector — я не вижу, как мы затираем
следы альтернативных сценариев»).

## Вопрос M5 (ровно один)

**Z1 — Как систематически затирать следы альтернативных сценариев, чтобы ни холодная
сессия, ни параллельный агент не могли подобрать мёртвый путь?**

⚠️ **Обязательное эхо в вердикте:** строки «Реестр следов: <решение>», «Механика
затирания: <решение>», «Гейт против возрождения: <решение>», «Процессные следы: <решение>».
Вердикт обязан нести секцию-заголовок **«Список посылок»** (иначе не состоялось, S-M2).

Решаем только GC как класс. Конкретная вычистка утреннего конкурента уже решена в H
(манифест, атомарность, |entry|=1) — GC обобщает её в системный процесс. Ратифицирует
владелец.

## Грани вопроса Z (простым текстом)

- **Реестр следов:** откуда берётся полный список альтернатив? Известные экспонаты:
  барьерный раннер утра (код+манифест+промпт+скрипт+фазы в issue+карточка) · старый
  генератор плана и его читатели · retired-sources в assertions (вещдоки — маркированы,
  но читаемы) · старые описания утра в канонах (добито в C) · следы старого ночного
  канала. Реестр — рукописный или добываемый (grep-инвентаризация по предмету — тезис
  питомца из шторма)?
- **Механика затирания:** статусная механика из C (status/supersededBy, тело не читается)
  покрывает документы; что с кодом (deprecated-скрипты: удалять или статус-заглушка с
  ошибкой?), с данными (retired-sources), с зеркалами. «Затереть» ≠ «удалить из git-истории»
  — история неприкосновенна; затирается ЧИТАЕМОСТЬ мёртвого пути живой сессией.
- **Гейт против возрождения:** чем ловится возврат затёртого (шаг вернулся в цепочку,
  скрипт снова позван, канон снова описывает старое)? Кандидат: расширение docs:verify-canon
  из C + grep-гейт «мёртвые пути не вызываются» в CI. Периодичность: GC — разовый спринт
  или регулярный проход (например, в вечернем ритуале)?
- **Процессные следы (живой экспонат сегодняшнего дня):** альтернативные сценарии — не
  только файлы: нескоординированные процессы (три разрушения node_modules основного дерева
  за сутки install-войной параллельных агентов; агрессивная чистка, зацепившая 2019 файлов
  main tree). GC-класс включает процессную гигиену или она — отдельный контур (worktree-
  гигиена уже имеет свой эпик)? Провести границу явно.

## Посылки-подсказки (для вердикта, не сам вердикт)

- Слово владельца (BRIEF_AMENDMENT, ратифицирован): нужен GC; «стратегия охотников за
  привидениями» (тезис шторма: манифест следов до сноса, снос со ссылкой).
- Закрытые вердикты этого заседания: C (статусная механика, verify-canon), H (манифест +
  атомарность + |entry|=1 для утреннего конкурента — образец для класса).
- Прецеденты подбора мёртвого пути: холодная сессия 21.07 (старый сценарий с боковой
  ветки); зомби-акцент стратегии на отозванных кристаллах (research-канал).
- Живой инцидент: чистка деревьев зацепила основное дерево (2019 файлов, восстановлены);
  node_modules-война; предохранитель «dry-run не резолвится в main tree» предложен в issue
  гигиены.
- Тезис питомца (шторм, не подтверждён — кандидат): инвентаризация пересечений перед
  проектированием.
