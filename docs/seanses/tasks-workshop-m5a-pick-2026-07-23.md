# Консилиум: контракт подбора кандидатов для one shot

**Повестка:** Решить, что возвращает функция подбора кандидатов для one shot, по чему судит про серверность без диффа, как стыкуется со штампом тимлида.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант (Kuryokhin), Верстальщик (Rodchenko).

**Порядок реплик:** Структурщик → Математик → Teamlead → Музыкант → Верстальщик

---

[Структурщик]: Начну с термина, иначе DAG поплывёт. «Функция подбора» — омоним двух лемм: (1) **фильтрация по критериям** (вход: список карточек → выход: подмножество, ранжированное или с вердиктом) и (2) **валидация допуска** (проверка, может ли задача вообще быть one shot). Первая — процедурная, вторая — охранная. У нас уже есть охрана в вердикте тимлида; ловушка в том, что обе иногда путают. Предлагаю разделить: функция = (1) только ранжирование + объяснение, **без вердикта допуска**. Вердикт остаётся у тимлида.

[Математик]: Формализую разделение, чтобы не было подмены. Пусть `candidates: Card[]` — входные карточки. Функция возвращает `ranking: (card, score, reasoning) []`, где `score ∈ [0, 1]` (1 = идеальный one shot), а `reasoning` — именованный набор условий: `{ sizeCheck, serverImpactClue, scopeClarity, … }`. Каждый элемент `reasoning` — это **булев предикат с обоснованием**, не вердикт. Тимлид видит: карточка B имеет score 0.9 с обоснованием «size малый ✓, нет упоминания `background-*` ✓, scope ясен ✓», и решает сам: берёшь или нет. Функция не говорит «одобряю / отклоняю»; она говорит «вот факты».

[Teamlead]: Принимаю. Это ровно то, что нужно для штампа: функция вооружает тимлида фактами, тимлид остаётся гейтом. Теперь про источник суждения о серверности **при отсутствии диффа** — когда карточка только зарегистрирована и diff ещё не скачен. Можно ли судить по `title` / `description` / `labels`? Или это подставлять ногу? Я вижу две стратегии: (А) консервативная — серверность = неизвестна, score падает до 0.3 по одному этому, (Б) эвристическая — ищем ключевые слова (`background-`, `server`, `deploy`, `database`) в описании и понимаем вероятность. Голосую за (А) как safe default: if no diff scanned, serverImpactClue = `unknown` → `reasoning.serverImpactClue = false` с пояснением «диф не просмотрен».

[Музыкант]: По касательной, но флаг про честность экрана: что видит тимлид в moment запроса штампа? Если функция возвращает ранжирование, но некоторые поля `unknown`, это обязано быть видимо как `⚠ не все данные готовы`. Молчаливое `unknown` за радостным 0.8 score = мёртвая дверь. Требую явный индикатор готовности каждого источника суждения: `diff: scanned | pending | error`, `labels: found | absent`, `size: measured | estimated`. Тимлид тогда знает, на чём он стоит.

[Верстальщик]: Верно, и в список посылок внесу: функция обязана возвращать не только `score`, но и `metadata: { dataReadiness }`. По видимости в PR: когда тимлид кликает «штамп one shot» в PR-интефейсе (или через CLI), он видит ранжированный список с явными дырами — это норма из `main-day-accuracy`. Иконка ⚠ жёлтая, если `pending`; красная, если `error`. Без этого тимлид подпишется слепо. Заострю: функция возвращает **четыре обязательных источника** (сигнатура в `types.ts`): размер + серверность + scope + историческая репутация карточки (есть ли у неё прошлые one shot успехи).

[Структурщик]: Уточню четыре источника через мою линзу лемм. (1) **размер** — `size` из карточки реестра, источник = человек-регистратор; может лгать, но это точка останова. (2) **серверность** — `serverImpactClue`, источник = диф (если готов) или описание (эвристика); пойдёт `unknown` при отсутствии обоих. (3) **scope clarity** — есть ли в описании явные граждане (файлы, модули, роли)? Парсим текст; логирование — если нашли ровно одного держателя в `leadPersona`, это +0.3 к score. (4) **историческая репутация** — переиспользование лемм из прошлых карточек; есть ли в истории задач реестра закрытые one shot карточки с теми же ключевыми словами? Если да — этот кандидат уже проверен, +0.2 к score. Все четыре поля идут в `metadata.sources[]`.

[Математик]: Формализую жёсткие отсечки и мягкие признаки. **Жёсткие отсечки** (если хоть один нарушен — карточка выпадает из списка): `size ≤ M` (М = параметр, предлагаю M=5 часов), `serverImpactClue ≠ 'critical'` (т.е. `unknown | low | medium` допустимы, `critical` исключает). **Мягкие признаки** (поднимают или опускают score внутри диапазона [0.3, 1.0]): `scopeClarity ∈ [0, 1]` (норм. число гражданских элементов в description), `historicalReputation ∈ {0, 0.2, 0.4}` (есть ли успешные предки). Формула итогового `score = w₁·sizeNorm + w₂·(1 - serverImpactRisk) + w₃·scopeClarity + w₄·historicalReputation`, где w-коэффициенты = [0.3, 0.3, 0.2, 0.2]. Это детерминировано и тестируемо.

[Teamlead]: Принимаю формулу Математика. Проверю граничные случаи: карточка размером 6 часов (чуть выше отсечки) падает сразу — верно. Карточка размером 3 часа, но с `serverImpactClue = 'critical'` падает сразу — верно. Карточка размером 2 часа, unknown серверность, но отличная историческая репутация — пройдёт фильтр с score ~0.65, попадёт в ранжирование — верно. Теперь про стык со штампом тимлида. Функция выдаёт ранжированный список с score и reasoning. Тимлид видит топ-N (N = 3 или 5?) и может кликнуть на любого (или сказать «не беру ни одного»). Клик = вызов отдельного промпта, где тимлид ставит штамп явно (доп. вопрос к нему: почему именно этот кандидат, несмотря на score). Штамп = непреодолимо; функция не может его отменить, только проинформировать.

[Музыкант]: По касательной, но флаг про failover: что если функция упадёт (диф не скачается, парсинг description отказал)? Требую graceful degradation — функция **всегда** возвращает результат (может быть с ошибками в metadata), но не выбрасывает исключение. Тимлид видит ⚠ и может взять карточку вручную (свой выбор, не фильтра). Это честно и не блокирует workflow.

[Структурщик]: Согласен, добавлю в контракт: `Result = { candidates: Candidate[], errors?: Error[] }`. Ошибка в parsing не валидирует весь прогон, только помечает одного кандидата как `dataReadiness: 'error'`. Функция live, в CI не фейлится — это норма из `checklists-soft-fail`.

[Верстальщик]: По видимости ранжирования в UI. Предлагаю простой список с картами: для каждого кандидата — title, size badge, score как полоса (0–100%), четыре кнопки-индикатора (зелёный ✓ / жёлтый ⚠ / красный ✗) под каждый source из metadata. Клик на карту открывает modal с полным reasoning (ссылка на диф, цитаты из description). Кнопка «Штамп» только в modal; она ведёт на вторую форму (вопросы к тимлиду). Иконка ⚠ на source — tooltip с кратким текстом («диф не готов», «историческая репутация не найдена»). Никаких чисел в score напрямую; только визуальная полоса + обоснование через reasoning.

[Математик]: Финальное согласование на числа. Жёсткие отсечки: `size ≤ 5h` + `serverImpactClue ∈ {unknown, low, medium}`. Мягкие: `sizeNorm = 1 - (size / 5)`, если size ≤ 5h (норм на [0, 1]), иначе отсечка срабатывает раньше. `scopeClarity` вычисляется как ratio найденных лемм (файлы, модули) к ожиданию — от 0 до 1. Веса w = [0.3, 0.3, 0.2, 0.2] фиксирую как констанду в `constants.ts`. Threshold попадания в ранжирование: score ≥ 0.4 (ниже — даже не показывать в списке, только в `excluded[]` с пояснением). Тестов: юнит на граничные score (size=5, score=1.0; size=6, отсечка; serverImpactClue='critical', отсечка).

[Teamlead]: Принимаю. Фиксирую итоговый контракт. Функция `rankOneShotCandidates(cards: Card[], options?: RankOptions) → Result`, где `Result = { candidates: RankedCandidate[], excluded: ExcludedCandidate[], errors?: Error[] }`. Каждый `RankedCandidate` несёт `{ card, score, metadata: { sources: Source[] }, reasoning: { sizeCheck, serverCheck, scopeCheck, reputationCheck } }`. Функция чистая (без побочных эффектов), детерминирована при одних входах. Она **не ставит штамп**, не хранит состояние, не знает о тимлиде. Тимлид вызывает её как `yarn one-shot:rank`, получает JSON, смотрит в UI, кликает на кандидата, вызывает второй промпт `yarn one-shot:stamp <cardId>` — там уже вопросы к тимлиду и его явное слово. Разделение ответственности чистое.

[Структурщик]: Носитель функции — `scripts/one-shot-rank.mjs`, экспорт также в `packages/core/scripts/utils.ts` для переиспользования в других ритуалах (if появятся). Входные данные — реестр карточек из `docs/tasks/registry.json` (парсим его, не скачиваем дифф в batch-режиме — только по запросу). Выход — JSON stdout (для пайплайна) + опциональный флаг `--ui` для показа в браузере. Тесты: fixtures с карточками разного размера, с разными описаниями, с ошибками парсинга. Граница ясна: ровно функция подбора, не сама регистрация, не штамп, не мердж.

[Музыкант]: Флаг про отмену. Если тимлид вызвал `rank`, потом передумал и закрыл UI, не теряется ничего — функция stateless. Можно вызвать снова, получишь тот же результат. Это важно, потому что владелец может позвать ранжирование в утро несколько раз до решения. Молчаливое состояние здесь мертво.

[Верстальщик]: По UI — финальная норма. Интерфейс ранжирования не входит в `apps/client/` (это backend-утилита). Он живёт как `scripts/one-shot-rank-ui.html` — standalone страница, которую тимлид может открыть в браузере после `yarn one-shot:rank --ui`. HTML генерируется из JSON результата функции. Мобильный адаптив не требуется (утилита для тимлида за десктопом). Кнопка «Штамп» в modal кликает на `yarn one-shot:stamp`, shell перехватывает и показывает вторую форму.

## Список посылок

1. **Факт:** Формат one shot (инсайт 7.2) определил, что вердикт S-ности ставит тимлид по запросу, самоаттестация исключена.
2. **Норма:** Функция подбора не должна подменять тимлида; она может только информировать с фактами и ранжированием.
3. **Факт:** Карточка при регистрации содержит поле `size` (оценка человека), но диф готов не всегда в момент запроса штампа.
4. **Норма:** Серверность судится по четырём независимым источникам: диф (если готов) + описание (эвристика) + историческая репутация + явное указание в description.
5. **Факт:** Метаданные карточки (title, description, labels) парсятся автоматически; парсинг может ошибиться.
6. **Норма:** Ошибки в парсинге не блокируют весь прогон; функция gracefully деградирует и помечает источник как `error` в metadata.
7. **Норма:** Жёсткие отсечки применяются первыми; только карточки, прошедшие отсечки, попадают в ранжирование и получают score.
8. **Норма:** Score вычисляется как взвешенная сумма четырёх нормализованных компонент; формула детерминирована и тестируема.
9. **Норма:** Функция чистая (stateless); её можно вызвать несколько раз и получить идентичный результат.
10. **Норма:** UI ранжирования (if любой) не входит в `apps/client/`; это backend-утилита для тимлида (`scripts/one-shot-rank-ui.html`).

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| **Что возвращает функция подбора** | Ранжированный список `RankedCandidate[]` с `score ∈ [0, 1]` и именованным `reasoning` (четыре источника: size, serverImpactClue, scopeClarity, historicalReputation). **Не вердикт**, только информация. |
| **Вердикт допуска** | Остаётся у тимлида. Функция выдаёт facts; тимлид кликает на кандидата → вызывает второй промпт `yarn one-shot:stamp <cardId>` с явными вопросами. |
| **Источник суждения о серверности** | Четыре слоя (приоритет): (1) diff-анализ, если готов; (2) ключевые слова в description; (3) историческая репутация (успешные one shot с теми же леммами); (4) явная роль в `leadPersona`. При отсутствии всех — `unknown`. |
| **Жёсткие отсечки** | `size ≤ 5h` И `serverImpactClue ∈ {unknown, low, medium}` (не `critical`). Выше — исключение без попадания в ранжирование. |
| **Мягкие признаки** | `sizeNorm`, `scopeClarity`, `historicalReputation` — поднимают score внутри [0.4, 1.0]. Threshold попадания в список: score ≥ 0.4. |
| **Формула score** | `score = 0.3·sizeNorm + 0.3·(1 - serverImpactRisk) + 0.2·scopeClarity + 0.2·historicalReputation`. Детерминирована, все компоненты нормализованы на [0, 1]. |
| **Метаданные выхода** | Каждый источник несёт `dataReadiness ∈ {ready, pending, error}`. UI показывает индикаторы; тимлид видит дыры в данных. |
| **Graceful degradation** | Ошибка в парсинге одного поля помечает соответствующий source как `error`, не валидирует весь результат. Функция всегда возвращает `Result`, никогда не выбрасывает исключение в бизнес-коде. |
| **Состояние** | Функция stateless и чистая; может быть вызвана многократно с одними входами и выдаст идентичный результат. |
| **Носитель** | `scripts/one-shot-rank.mjs` + экспорт утилит в `packages/core/scripts/utils.ts`. UI (if любой) — `scripts/one-shot-rank-ui.html` (backend-утилита, не в apps/client). |

**Definition of Done:**

- [ ] Функция `rankOneShotCandidates(cards, options) → Result` с сигнатурой в `types.ts` (не меняется); все четыре источника обязательны.
- [ ] Жёсткие отсечки (`size ≤ 5h`, `serverImpactClue ≠ 'critical'`) применены до score; карточки, прошедшие отсечки, попадают в `candidates[]`, остальные — в `excluded[]`.
- [ ] Score = детерминированная формула (веса w = [0.3, 0.3, 0.2, 0.2] в constants); юнит-тесты на граничные случаи (size=5, score=1.0; size=6, отсечка; serverImpactClue='critical', отсечка).
- [ ] Metadata несёт `dataReadiness` для каждого источника (`ready | pending | error`); ошибка в парсинге одного поля не блокирует весь результат.
- [ ] Тесты: fixtures карточек разного размера, разные description, парсинг ошибок, историческая репутация (успешные и отсутствующие предки).
- [ ] Вывод — JSON stdout (для пайплайна) + опциональный `--ui` флаг; UI (если сделан) — standalone HTML в scripts, не в apps/client.
- [ ] Функция чистая (no side effects); вызов многократно с одними входами выдаёт идентичный результат.
- [ ] Граница ясна: ровно подбор и ранжирование; регистрация, штамп, мердж — вне scope.
