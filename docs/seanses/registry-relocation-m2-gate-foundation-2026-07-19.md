# Консилиум: вход проверок и метрик после переезда на Linear (F1)

[Математик]: Формализую, иначе «вход» останется вкусовым. Гейт — чистая функция `gate(x) → verdict`, где по наследству эпика `x` детерминирован и результат бит-в-бит воспроизводим без сети. Живой запрос к Linear делает `x` функцией времени: `gate(fetch()) ≠ gate(fetch())` через минуту. Значит **живой запрос дисквалифицирован как вход по конструкции** — он ломает уже ратифицированную аксиому чистоты. Вход обязан быть снимком.

[Структурщик]: Начну с термина, иначе спор поплывёт. «Снимок» и «кэш» — две разные леммы, слипшиеся под одним словом. Кэш — это **оптимизация**: его можно молча не обновить, и корректность не страдает, только скорость. Снимок — это **контракт**: он адресуемый, версионированный артефакт с провенансом, и его молчаливое устаревание — нарушение, а не потеря производительности. F1 требует именно контракта: у снимка есть имя, момент съёмки и ревизия источника, зафиксированные в самом артефакте.

[Музыкант]: По касательной — аудио тут не задет, но ритм я узнаю, это ровно воспроизводимость сведения, на которой я обжигался в `code-review-honesty-refactor`. Мастер обязан звучать одинаково при рендере сегодня и через год: фиксируешь не только исходники, но и снимок состояния сессии на момент экспорта. Живой Linear — это как рендерить из живого DAW, где кто-то в параллели двигает фейдеры. Снимок = замороженная сессия. Плюс флаг из фактуры: `blockedBy` вебхуками не приходит — значит движение в принципе не событийно, его надо **опрашивать и материализовать**, иначе поймаешь тишину вместо клиппинга.

[Teamlead]: Рамку фиксирую сразу, чтобы комната не расползлась в форму хранилища и порядок миграции — это следующие фазы, запрет повестки. Наш вопрос ровно один: вход гейта. По конструкции «форма следует функции»: функция гейта — выносить воспроизводимый вердикт, значит вход обязан быть **снимком, а не сетью**, и это не компромисс, а прямое следствие ратифицированной чистоты. Напряжение «сетевой источник против чистой функции» разрешается не правкой аксиомы, а вставкой слоя материализации между сетью и гейтом. Принимаю линию Математика и Структурщика как корневую.

[Верстальщик]: С точки зрения потребителя вердикта — а потребитель это агент утром и человек в PR — снимок без провенанса рождает **мёртвую дверь**: зелёная галочка на данных недельной давности выглядит как свежая правда. Моя норма из `main-day-accuracy`: любой вердикт врёт, если не показывает, на какой ревизии построен. Значит шапка отчёта обязана нести момент съёмки снимка **и** ревизию источника, к которой он привязан. Без видимого провенанса свежесть — вечный ложный варнинг, это уже проходили.

[Математик]: Уточняю пригодность численно, потому что «снимок соответствует источнику» без предиката — вкусовщина. Пригодность = пара `(capturedAt, sourceRevision)`, где `sourceRevision` — это версия/updatedAt-курсор от Linear на момент съёмки. Предикат расхождения: `stale(snapshot) := source.updatedAt > snapshot.capturedAt`. Это проверяемо ровно одним лёгким запросом курсора, не полным опросом — то есть проверка свежести дешева, а сам гейт при этом остаётся офлайн на теле снимка.

[Структурщик]: Тогда фиксирую производителя снимка как отдельную лемму, иначе адрес поплывёт. Снимок производит **не гейт**. Уже построено: `@membrana/background-office` умеет читать задачу и принимать вебхук с проверкой подписи. Значит производитель — batch-джоб на стороне office: он опрашивает Linear, материализует снимок и кладёт его в известное место. Гейт — чистый потребитель файла, он **не ходит в сеть вообще**. Это держит слабую связанность: сетевая граница локализована в одном узле, а не размазана по каждой проверке.

[Музыкант]: Поддержу с живой стороны — и добавлю про парность, я на этом настаивал в корпусной спеке. Снимок один — это `n=1`, мы не увидим движения. Для метрик по наследству эпика нужен **вечерний батч из двух снимков** — начало окна и конец окна — плюс артефакты закрытия. Значит производитель кладёт снимок с меткой времени как трек в архив, а метрика сводит два трека. Один снимок = стоп-кадр; движение слышно только на паре.

[Teamlead]: Свожу и решаю про расслоение гейтов — это несущий шов. Проверки **содержания** (оно в git по решению владельца: содержание — в git, движение — в Linear) сети не требуют вовсе и работают на рабочем дереве. Проверки **движения** (статусы, блокировки, исполнители) работают на снимке Linear. Граница проходит ровно по шву git/Linear, как и объявлено владельцем. Ни одна проверка не ходит в сеть в момент вердикта: содержание читает git, движение читает снимок. Сеть живёт только в производителе снимка. Принимаю.

[Верстальщик]: С точки зрения потребителя добавлю про поведение при расхождении — оно обязано быть **видимым и различимым по коду возврата**, иначе экран соврёт. По брифу: отказ обязан отличаться от замечания. Моя раскладка: `stale` снимок → это **замечание** (warning, ненулевой мягкий код), потому что гейт всё ещё может дать честный вердикт на теле снимка, просто помечает «данные на ревизии X, источник ушёл вперёд». А вот **отсутствие** снимка или битый провенанс → **отказ** (жёсткий exit-код), потому что вход не определён. Два разных цвета, два разных кода.

[Математик]: Соглашусь с раскладкой кодов и уточню границу. `stale` — не отказ, потому что чистота от свежести не зависит: `gate(snapshot)` детерминирован на любом снимке, включая устаревший. Отказ — только когда `x` не определён: снимка нет, провенанс отсутствует или `capturedAt`/`sourceRevision` пусты. Формально: `verdict.exit ∈ {OK=0, STALE=warn, NO_INPUT=fail}`. Свежесть — это аннотация вердикта, а не его валидность. Это спасает от вечного false-fail, которым Родченко справедливо пугает.

[Структурщик]: Закреплю судьбу вердикта о чистой функции, чтобы не было молчаливого обхода — бриф это запрещает прямо. Вердикт эпика **выживает с уточнением входа, без пересмотра сути**. Уточнение ровно одно: слово «два снимка» в эпике теперь означает «два **материализованных снимка Linear**, произведённых office-батчем», а не абстрактные снимки git. Функция осталась чистой, детерминизм цел, сеть по-прежнему вне гейта. Меняется не аксиома, а **адрес источника снимка**. Это уточнение, а не отзыв.

[Музыкант]: Поддержу — и брошу один живой флаг из фактуры, чтобы не поймать тишину. Вебхуки могут терять события, есть лимиты частоты — значит снимок нельзя строить «по накопленным вебхукам», это дырявый мастер. Снимок строится **полным опросом состояния** на момент съёмки (pull, не накопление push). Вебхук — только триггер «пора снимать», но тело снимка — всегда свежий pull. Иначе снимок молча разъедется с источником, а мы этого не услышим.

[Teamlead]: Принимаю уточнение Музыканта — оно снимает риск потери событий и держит провенанс честным: снимок = pull-состояние на `capturedAt`, вебхук не является источником тела. Фиксирую, что форму хранилища снимка (где именно лежит, попадает ли в git) мы **называем требованием, но не выбираем** — это следующая фаза, запрет повестки. Наше обязательство: снимок адресуем, несёт `(capturedAt, sourceRevision)`, производится office-батчем pull-опросом. Где он материально живёт — не наш вердикт.

[Верстальщик]: Принимаю, с одной оговоркой потребителя. Раз форму хранилища мы не выбираем, то в **отчёте** провенанс обязан быть самодостаточным независимо от места хранения: шапка несёт `capturedAt`, `sourceRevision`, окно метрики и обе ревизии пары снимков — чтобы человек утром прочитал честность вердикта, не зная, где физически лежит файл. Видимость провенанса — несущая конструкция, а не украшение.

[Математик]: Согласен, и закрою последнюю численную дыру — доказательство пригодности. Пригодность снимка доказывается **не** повторным полным опросом (дорого, снова вводит сеть в проверку), а сравнением курсора: `fresh := source.cursor == snapshot.sourceRevision`. Один дешёвый запрос курсора, вне тела гейта, в момент производства или в отдельной проверке свежести. Тело вердикта остаётся офлайн. Провенанс — это `(capturedAt, sourceRevision)`, свежесть — булев предикат над ним, стоимость O(1).

[Структурщик]: Свожу термины в словарь, чтобы ни одна лемма не осталась омонимом: **снимок** — адресуемый контракт с провенансом; **кэш** — здесь не используется, слово запрещаю в этой подсистеме, чтобы не спутали контракт с оптимизацией; **производитель** — office-батч, pull; **потребитель** — гейт, офлайн; **свежесть** — аннотация, не валидность. Граф без циклов: `office → снимок → гейт`, гейт не знает про Linear. Слабая связанность соблюдена: сетевая зависимость в одном узле.

[Teamlead]: Консенсус собран, выношу вердикт. Вход гейта — снимок, не сеть. Снимок — контракт с провенансом `(capturedAt, sourceRevision)`, производится office-батчем полным pull-опросом, вебхук — лишь триггер. Гейты расслоены по шву git/Linear: содержание — git, движение — снимок; сеть только в производителе. Расхождение: `STALE` — замечание (мягкий код), `NO_INPUT` — отказ (жёсткий код). Вердикт эпика о чистой функции выживает с уточнением адреса входа, суть цела. Форму хранилища и порядок миграции не трогаем — чужие комнаты. **LGTM.**

[Верстальщик]: Принимаю. Провенанс в шапке отчёта — несущая, коды возврата различимы, мёртвой двери нет.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Вход гейта: снимок или сеть | **Снимок.** Живой запрос дисквалифицирован — ломает ратифицированную чистую функцию (`gate(fetch()) ≠ gate(fetch())`). |
| Чем снимок отличается от кэша | Снимок — **контракт**: адресуемый, версионированный, с провенансом; молчаливое устаревание = нарушение. Кэш — оптимизация, молча не обновить можно. Слово «кэш» в этой подсистеме запрещено. |
| Производитель снимка | **office-батч** (`@membrana/background-office`) **полным pull-опросом** состояния на момент съёмки. Вебхук — только триггер «пора снимать», не источник тела (вебхуки теряются, есть rate-limit). |
| Доказательство пригодности | Провенанс `(capturedAt, sourceRevision)`; предикат свежести `fresh := source.cursor == snapshot.sourceRevision` — один дешёвый запрос курсора **вне тела гейта**, O(1). Тело вердикта офлайн. |
| Поведение при расхождении | `OK=0`; `STALE=warn` (мягкий код — замечание, вердикт валиден на теле снимка); `NO_INPUT=fail` (жёсткий код — вход не определён: нет снимка/провенанса). Отказ ≠ замечание по коду возврата. |
| Расслоение гейтов | Содержание → git (сеть не нужна); движение (статусы/блокировки/исполнители) → снимок Linear. Граница по шву git/Linear. **Ни одна проверка не ходит в сеть в момент вердикта.** |
| Судьба вердикта о чистой функции | **Выживает с уточнением входа**, без пересмотра сути. Уточнение: «два снимка» = два материализованных pull-снимка Linear от office-батча. Детерминизм и офлайн-вычисление целы. Не отзыв. |

**Definition of Done (F1):**
- Контракт снимка объявлен: обязательные поля `capturedAt`, `sourceRevision`; артефакт адресуем и несёт провенанс.
- Функция гейта принимает на вход снимок (файл/структуру), **не** делает сетевых вызовов в теле вердикта; свойство «тот же снимок → тот же вердикт бит-в-бит» покрыто тестом.
- Предикат свежести `fresh(snapshot, sourceCursor)` — отдельная функция, вызывается вне тела гейта; стоимость — один запрос курсора.
- Коды возврата различимы и задокументированы: `OK` / `STALE` (warn) / `NO_INPUT` (fail); отказ отличается от замечания кодом.
- Расслоение проверок зафиксировано: список гейтов размечен `git-only` / `snapshot-based`; ни один не помечен `network-at-verdict`.
- Шапка отчёта несёт `capturedAt`, `sourceRevision`, окно метрики и ревизии обоих снимков пары — провенанс самодостаточен независимо от места хранения.
- Уточнение вердикта эпика зафиксировано письменно (ADR/запись): «снимок» = материализованный pull-снимок Linear; форма холодного хранилища и порядок миграции **не решаются** здесь.
