[Математик]: Формализую детерминированность сразу, иначе слово останется вкусовым. Требование одно: `audit(x) = audit(x)` для одного и того же входа `x = (logs, dayPlan, truthCrystals[])` — чистая функция, `∀ прогонов результат бит-в-бит равен`. Это значит: **никаких** `Date.now()`, `Math.random()`, обхода файловой системы вне переданного `x`, порядка ключей объекта как источника недетерминизма. Практический тест: `sha256(audit(x)) == sha256(audit(x))` на двух прогонах в разных процессах.

[Структурщик]: Начну с термина, иначе спор поплывёт. «Воспроизводимость» — омоним двух лемм. Первая: **детерминированность функции** (Дынин её закрыл — один вход → один выход). Вторая: **воспроизводимость входа** — то есть возможность через N дней собрать тот же самый `x`. Это разные обязательства к разным владельцам. Функцию гарантирует автор `audit`, а вот сохранность `x` гарантирует тот, кто пишет `logs` и снимает срез `dayPlan`. Без второго первое бесполезно: чистая функция от испарившегося входа невоспроизводима.

[Teamlead]: Рамку фиксирую сразу, чтобы не расползлось. DA1 — **ровно один** вопрос: какие поля обязаны существовать во входе и кто владелец каждого предиката. Не «форма отчёта» (вопрос F), не «читатель» (вопрос T) — их сюда не тащим по границе повестки. Приоритет `ARCHITECTURE.md`: `audit` — чистое ядро, живёт как функция без побочных эффектов. Разделение Структурщика на две леммы — несущее, беру в основу: **детерминизм функции** и **пиннинг входа** — два разных контракта.

[Музыкант]: По касательной, аудио тут не задет, но ритм я узнаю — это ровно как воспроизводимость сведения. Мастер должен звучать одинаково при рендере сегодня и через год: значит фиксируешь не только исходники, но и **версии плагинов и семплрейт**. Мой живой флаг из `main-day-issue-accuracy` 16.07: мы уже обжигались на том, что артефакт не показывал ревизию кода, на которой построен. Здесь то же — во входе обязан быть **хэш ревизии** самого кода `audit`, иначе «тот же вход» даст другой выход после рефактора функции.

[Верстальщик]: С точки зрения потребителя отчёта — а он у меня болел в `code-review-honesty-refactor` и `main-day-accuracy` — воспроизводимость должна быть **видимой на холсте**, не только логичной. Если два прогона дали разный результат, читатель обязан увидеть, что именно разъехалось: какое поле входа отсутствовало или изменилось. Значит `derivedReport` должен нести провенанс входа — хэши всех трёх аргументов — как честную подпись, а не как скрытую метаданную.

[Математик]: Согласен с Музыкантом и уточню численно. Вход `x` неполон без **версии кода**: `audit@commit`. Тогда контракт детерминизма звучит строго: `audit_v(x) = audit_v(x)`, где `v` пиннится хэшем ревизии. Формализую обязательные поля входа как множество с владельцами:
- `logs`: `{entries[], sourceCommit, capturedAt}` — владелец: ритуал, породивший логи.
- `dayPlan`: `{planId, revision, snapshotHash}` — владелец: `registry-reaper` (срез дня, ревизия реестра вынесена, по границе повестки).
- `truthCrystals[]`: `{id, claim, originHash, ratifiedAt}[]` — владелец: граф правды.
Каждое поле имеет **origin-hash**; отсутствие хэша = поле не пиннится = невоспроизводимо.

[Структурщик]: Origin-hash как обязательное поле принимаю — это ровно норма из `truth-graph-unify-or-split` 17.07: derived-токен обязан ссылаться на origin. Но зафиксирую жёстко про **владельца предиката**, иначе размажем ответственность. Предикат — это проверяемое утверждение отчёта, например `plan.item[3].claimed_done == true, но truthCrystal T-12 говорит not-done`. Владелец такого предиката — **не `audit`**. `audit` только *вычисляет* столкновение; владелец истины — тот, кто выпустил кристалл. `audit` не имеет права быть источником правды, только её со-поставителем. Иначе получим второй реестр правды под именем «отчёт» — а это анти-паттерн, мы его хоронили в Q1 truth-graph.

[Teamlead]: Принимаю разграничение Структурщика как несущее и формулирую вердикт по владельцам. `audit` — **derived**, никогда **owner**. Три класса владельцев входа:
1. Владелец `logs` — ритуал-источник (капчурит поток дня).
2. Владелец `dayPlan` — `registry-reaper` (срез, по границе повестки).
3. Владелец `truthCrystals[]` — граф правды.
`audit` владеет **только** предикатом соответствия — «сошлось / разъехалось», и этот предикат derived, с обязательной ссылкой на origin обоих операндов. Автодействие по нему запрещено инвариантом H1 — здесь не переигрываем.

[Музыкант]: Поддержу с живой стороны и подниму флаг про декоррелированность — он у меня из `ritual-inputs-echo`. Если два `truthCrystal` выведены из одного origin-hash, они не два независимых свидетельства, а `n=1`. Аудитор, который посчитает их как два подтверждения, соврёт весом — как ансамбль детекторов от эха. Значит в поля кристалла обязан входить `originHash`, и `audit` при подсчёте согласия должен **дедуплицировать по origin**, а не по `id`. Это не форма отчёта — это корректность самого предиката.

[Верстальщик]: Тогда с витрины провенанс — это не одна подпись, а **три честных блока**: хэш logs, хэш dayPlan, список кристаллов с их origin. По моей норме «функция видна сразу»: если кристалл дедуплицирован по origin, читатель должен видеть, что три строки схлопнулись в одно свидетельство, а не притворяться тремя галочками — та же боль, что эхо-камера в `strategy-day-q5`. Мёртвая дверь недопустима: отсутствующее поле входа показывается явно как «нет данных», а не как молчаливый ноль.

[Математик]: Закреплю численный критерий детерминированности как исполнимый тест, чтобы «на практике» не осталось словом. Три проверки:
1. **Идемпотентность**: `sha256(audit(x)) == sha256(audit(x))` в двух процессах — ловит скрытое состояние.
2. **Полнота пиннинга**: каждое поле `x` имеет непустой origin-hash; иначе fail до вычисления.
3. **Дедуп-инвариант**: `|distinct(originHash(crystals))| == вес согласия`, не `|crystals|` — ловит эхо.
Если все три зелёные — `audit` воспроизводим по определению. Это чистая функция, покрывается юнит-тестом без DOM и аудиостека.

[Структурщик]: Три теста Дынина принимаю как контракт. Добавлю границу пакета, чтобы норма не расползлась по коду. `audit` живёт как **чистое ядро сервиса** — вход типизирован, origin-hash'и — часть `types.ts`, никаких обращений к git/fs изнутри функции: пиннинг делает **вызывающий**, передавая уже собранный `x`. Это ровно разделение слоёв из `SERVICES.md`: `math/` не читает файловую систему. Сбор `x` (чтение логов, снятие среза, git-хэш) — обязанность обвязки, не ядра. Так тест №1 Дынина становится тривиально выполнимым: ядру неоткуда взять недетерминизм.

[Teamlead]: Свожу к вердикту. Все три обязательства покрыты: детерминизм функции (три теста), пиннинг входа (origin-hash на каждое поле), владельцы (три источника + `audit` только derived). Границу повестки удержали: форму отчёта (F) и читателя (T) не решали, ревизию реестра оставили `registry-reaper`. Это соответствует `ARCHITECTURE.md` (чистое ядро) и не создаёт второго реестра правды. Выношу на принятие.

[Музыкант]: Принимаю. Дедуп по origin — несущий, без него аудитор врёт весом, как ансамбль от эха.

[Верстальщик]: Принимаю — при условии, что провенанс входа виден в отчёте, а не спрятан; отсутствующее поле = явная мёртвая дверь, не ноль.

[Математик]: Принимаю. Три теста детерминированности — исполнимый контракт, ядро чистое.

[Структурщик]: Принимаю. Пиннинг — обязанность обвязки, ядро не трогает fs/git; владелец предиката соответствия — `audit` (derived), владелец истины — граф правды.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Что значит детерминированность на практике | `audit(x) = audit(x)` бит-в-бит; проверяется тремя тестами: идемпотентность (sha256 в двух процессах), полнота пиннинга, дедуп-инвариант по origin |
| Какие поля обязаны существовать во входе | `logs {entries[], sourceCommit, capturedAt}`, `dayPlan {planId, revision, snapshotHash}`, `truthCrystals[] {id, claim, originHash, ratifiedAt}`, плюс версия кода `audit@commit` |
| Обязательное поле каждого элемента | непустой `origin-hash`; отсутствие = fail до вычисления (невоспроизводимо) |
| Владелец `logs` | ритуал-источник (капчур потока дня) |
| Владелец `dayPlan` | `registry-reaper` (срез дня; ревизия реестра — по границе повестки, здесь не решаем) |
| Владелец `truthCrystals[]` | граф правды |
| Владелец предиката соответствия | `audit` — **derived**, никогда owner; со-поставитель, не источник правды |
| Кто пиннит вход | обвязка (вызывающий), не ядро; `audit` не читает git/fs — разделение слоёв `SERVICES.md` |
| Дедупликация свидетельств | по `originHash`, не по `id`; вес согласия = `|distinct(originHash)|` (защита от эха) |

**Definition of Done (если решение ведёт к задаче):**
- `audit(logs, dayPlan, truthCrystals[]) → derivedReport` реализован как чистая функция в `math/`-слое сервиса; без обращений к `Date.now`, `Math.random`, git, fs.
- Типы входа в `types.ts` включают обязательный `origin-hash` для каждого поля; версия кода пиннится как `audit@commit`.
- Юнит-тесты покрывают три инварианта: идемпотентность (sha256 в двух процессах), полнота пиннинга (fail при пустом origin-hash), дедуп по origin (вес согласия = число различных origin-hash).
- Сбор входа `x` (чтение логов, срез `dayPlan`, git-хэш) вынесен в обвязку, не в ядро.
- `derivedReport` несёт провенанс входа (хэши трёх аргументов); отсутствующее поле отображается явно, дедуплицированные кристаллы не показываются как независимые свидетельства (согласование формы — вопрос F, здесь только требование к данным).
- Инвариант H1 (запрет автодействия по механическому предикату) не нарушается.
