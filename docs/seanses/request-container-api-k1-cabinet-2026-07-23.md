# Консилиум: Контракт кабинета как зоны полномочий

**Повестка:** Определение структуры кабинета агента на сервере, механизма выдачи полномочий, проверки границ и обработки нарушений.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант (Kuryokhin), Верстальщик (Rodchenko).

**Порядок реплик:** Структурщик → Верстальщик → Teamlead → Математик → Музыкант

---

[Структурщик]: Начну с термина, иначе кабинет расползётся на три разных предмета. **Кабинет** — это омоним двух лемм: (1) **зона полномочий** (агент может просить X, но не Y) и (2) **хранилище состояния** (где живут параметры, кэш, история). Мы решаем только лемму (1), её граница на **сервере через предикат**. Хранилище — мастерская M2, не наш вопрос.

[Верстальщик]: Со стороны потребителя — а потребитель это агент и человек в логах — кабинет должен быть **видимым в ответе сервера**. Первая реплика ответа: HTTP-заголовок или поле `authorization` в теле с явным перечислением прав. Никаких молчаливых ограничений — если агент может просить, сервер должен объявить это до выполнения. Иначе это мёртвая дверь: агент срабатывает успешно, но ничего не получает.

[Teamlead]: Держу рамку: мы решаем ровно **контракт проверки**, не сами глаголы и не механику выдачи. Глаголы — дело мастерской M2 (каноны, лемматизация). Выдачу полномочий — дело комнаты A3 (кто имеет право раздавать). Я определяю форму проверки и точку входа на сервер. По принципу конструктивизма: **форма следует функции** — функция кабинета **отклонить недопустимый запрос до исполнения**, значит проверка — **синхронный фильтр перед основной логикой**.

[Математик]: Формализую, иначе граница останется вкусовой. Кабинет `C = (P, L, T)`, где: **P** — множество разрешённых процедур (замкнутый перечень); **L** — множество лимитов (запросов/день, токенов/запрос, каналов); **T** — текущее состояние счёта (израсходовано). Предикат проверки: `canProceed(request) := request.procedure ∈ P ∧ isWithinLimits(request, L, T)`, чистая функция. Возврат: `{passed: bool, reason?: string, remainingQuota?: {...}}`. Ни один запрос не исполняется без `canProceed(…) = true`.

[Музыкант]: Флаг на надёжность цепочки. При отказе сервера на гранце кабинета — это не ошибка, это **отказано в праве**, и клиент должен различить два сценария: (1) «вы запросили недопустимую процедуру» и (2) «вы израсходовали лимит». Первое — требует корректировки конфига агента или повтора с другой процедурой. Второе — требует ожидания или эскалации. Молчаливое отклонение обоих ведёт к гаданию по логам.

[Структурщик]: Развожу два сценария по носителям данных. **(А) Закрытый кабинет**: полномочия выдаёт сервер один раз при создании агента (bootstrap), потом передаёт клиенту в `AuthResponse`, клиент кэширует, может устаревать. Проверка гибридная — клиент может фильтровать до отправки (быстро), сервер **должен** перепроверить (безопасно). **(Б) Динамический кабинет**: полномочия живут только на сервере, клиент запрашивает перед каждым вызовом или кэширует с TTL. Я рекомендую **(А) для начала** — проще, выше производительность, защита через двойную проверку.

[Verстальщик]: По подаче на клиента. Если полномочия в ответе `AuthResponse`, нужна иерархия: основные права — в корне, лимиты — отдельный объект, счёт — отдельный (обновляется в фоне). Структура: `{ procedures: string[], limits: { requestsPerDay: 100, tokensPerRequest: 2000 }, quota: { used: 45, remaining: 55 } }`. Человек может заглянуть в логи или DevTools и увидеть, почему запрос отклонили. Бейдж в кабинете UI (мастерская M2 или кабинет A3) — потом.

[Teamlead]: Утверждаю разводку Структурщика как норму. **Закрытый кабинет с двойной проверкой** — архитектурно верно: граница ясна (сервер — источник истины), отказ на сервере перекрывает любой обход клиентом. Точка входа: middleware/guard в `background-office` проверяет `canProceed(…)` для каждого `POST /api/llm-call` и подобного. При отказе — `403 Forbidden` с телом `{ error: "quota_exceeded", details: "…" }`.

[Математик]: Уточню счётчик. Счёт должен быть **атомарным за сессию дня**: инкрементируется при успешном исполнении, откатывается при ошибке (повтор не штрафует). `T = { requestsToday: N, tokensConsumed: M, channelsFailed: K }`. Проверка `isWithinLimits(request, L, T)` вычисляется как: `T.requestsToday + 1 ≤ L.requestsPerDay ∧ T.tokensConsumed + request.estimatedTokens ≤ L.tokensPerRequest * L.requestsPerDay`. Если граница затронута — сервер отклоняет до попытки вызова API.

[Музыкант]: Флаг на обработку отказа. Когда клиент получает `403 quota_exceeded`, он должен: (1) остановить текущий цикл (не пытаться повторить), (2) оставить запрос в очереди (если это асинхронный контур вроде `swallow`), (3) вернуть управление владельцу с явной ошибкой (не молчаливый отказ). Это как тайм-аут записи: если микрофон выключился, рекордер не молчит, а ловит ошибку и выводит на экран.

[Структурщик]: Заостряю разницу между **процедурой** и **действием**. Процедура — это ветка цепочки (`morning-ritual`, `swallow-send`). Действие — конкретный вызов внутри процедуры (например, отправка сообщения в Telegram или вызов LLM). Кабинет перечисляет **процедуры**, которые агент может запускать. Действия внутри процедуры — они либо разрешены (входят в список), либо нет. **Не смешиваем зёрна**: одна таблица процедур, одна таблица лимитов.

[Верстальщик]: По видимости в логах. Когда процедура стартует, сервер логирует: `{ timestamp, procedure, agent, status: "started", cabinet: { procedures: […], quota_before: {…} } }`. При отклонении: `{ timestamp, procedure, agent, status: "rejected", reason: "not_in_cabinet", cabinet: {…} }`. Дифф между двумя логами показывает, почему вторая попытка не запустилась. Человек видит цепочку, агент может отладить.

[Teamlead]: Утверждаю разводку: **(1) Процедуры** — замкнутый перечень в кабинете; **(2) Лимиты** — квоты на день (запросы, токены, каналы); **(3) Счёт** — текущее израсходованное за день; **(4) Проверка** — синхронный предикат на сервере перед исполнением; **(5) Отказ** — `403 Forbidden` с явной причиной в теле.

[Математик]: Формализую границу выхода за кабинет. Случай (А): агент запрашивает недопустимую процедуру (`procedure ∉ P`). Ответ: `403 { error: "procedure_not_allowed", procedure: "…", available: […] }`. Случай (Б): лимит исчерпан (`T.requestsToday >= L.requestsPerDay`). Ответ: `403 { error: "quota_exceeded", reset_at: "2026-07-22T00:00:00Z", remaining: 0 }`. Эскалации нет — это граница, не оуп.

[Музыкант]: По маршруту отказанного запроса. Если процедура отклонена на гранце кабинета, она не должна оставить фантомный лог на сервере (вроде запроса к LLM). Это ответственность гейта: проверить ПЕРЕД тем, как дёргать внешние системы. Логика: `guard(request) → checkCabinet(request) → checkChannelAvailable(request) → executeMainLogic(request)`. Если гейт упал — `executeMainLogic` не вызывается.

[Структурщик]: Уточню носитель кабинета. Кабинет хранится на сервере в таблице `agent_cabinets (agent_id, procedures: json, limits: json, created_at, updated_at)` или во встроенной структуре, если БД в памяти. Клиент получает **снимок** при авторизации, кэширует, может устаревать — но сервер при каждом вызове перепроверяет по источнику истины. Изменение кабинета (например, владелец уменьшил лимит) отражается на новый запрос, не на старый кэш.

[Верстальщик]: По синхронизации кэша на клиенте. Если кабинет выдан в `AuthResponse`, клиент кэширует его на время сессии; при 403 с причиной `quota_exceeded` или `procedure_not_allowed` — кэш **не обновляется** (он всё равно устарел). Клиент может заправить UI индикатором: «осталось X запросов из Y», но это расчётный, не истина. Истина на сервере.

[Teamlead]: Итоговое решение консилиума. **Контракт кабинета** — это запись `{ procedures: [str], limits: { requestsPerDay, tokensPerRequest }, quota: { requestsToday, tokensConsumed } }`, выдаваемая при авторизации, проверяемая на сервере перед каждым запросом, возвращаемая при отказе как `403 { error, details, remaining }`. Это инструмент **автономии агента при контроле сервера**: агент видит свои права, может фильтровать локально, но не может обойти сервер.

[Математик]: Уточню функцию выхода за границу. При нарушении кабинета возвращается **не общая ошибка**, а **специфическая бизнес-ошибка**: коды `PROCEDURE_NOT_ALLOWED`, `QUOTA_EXCEEDED`, `CHANNEL_UNAVAILABLE`. Код позволяет клиенту различить три случая и действовать разумно. Числовой эквивалент: `procedure_not_allowed = -1, quota_exceeded = -2, channel = -3`.

[Музыкант]: Финальный флаг на надёжность. Кабинет как граница работает только если сервер **не молчит** при отказе. Требую, чтобы логирование отказов было **обязательным и видимым**: каждый отказ на гранце кабинета попадает в лог `llm-calls` с причиной и квотой. Это не ошибка приложения (ошибка — когда кабинет не сработал и запрос прошёл), это штатное событие, и оно обязано быть слышно.

[Структурщик]: Принимаю. Кабинет — замкнутая система с чистыми границами: сервер их охраняет, клиент их соблюдает, оба видят отказы. Нарушения обрабатываются специфичными кодами, не общей ошибкой.

[Верстальщик]: Принимаю. Интерфейс кабинета визуально ясен: права видны в ответе, лимиты кэшируются и обновляются при отказе, логи показывают причину отклонения.

[Teamlead]: Принимаю. **Кабинет** — это зона полномочий агента, определённая тройкой (процедуры, лимиты, счёт), выдаваемая сервером, проверяемая на сервере, отклоняемая с причиной.

[Математик]: Принимаю. Граница кабинета — это вычислимый предикат, возвращающий bool + причину, исполняемый перед основной логикой.

[Музыкант]: Принимаю. Отказ на гранце кабинета — это не молчаливый отказ, а явное событие с кодом и логированием.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Что перечисляется в кабинете? | **Процедуры** (замкнутый перечень: `["morning-ritual", "swallow-send", …]`), **лимиты** (квоты на день: `requestsPerDay`, `tokensPerRequest`), **счёт** (текущее израсходованное: `requestsToday`, `tokensConsumed`). |
| Кто выдаёт и меняет кабинет? | **Сервер** (`background-office`) выдаёт при авторизации агента. **Владелец** (через комнату A3) меняет полномочия. Клиент получает **снимок** в `AuthResponse` и кэширует. |
| Чем проверяется граница на сервере? | **Синхронный предикат** `canProceed(request, cabinet)` перед исполнением: `request.procedure ∈ cabinet.procedures ∧ isWithinLimits(request, cabinet.limits, cabinet.quota)`. Если ложь — `403 Forbidden` с кодом причины. |
| Что при выходе за границу? | **Специфичный отказ** (не generic error): `403 { error: "procedure_not_allowed" \| "quota_exceeded" \| "channel_unavailable", reason: "…", quota: {remaining, reset_at} }`. **Логирование обязательно**: каждый отказ на гранце попадает в лог `llm-calls`. |
| Структура кабинета в ответе? | `{ procedures: [str], limits: { requestsPerDay: int, tokensPerRequest: int }, quota: { requestsToday: int, tokensConsumed: int, reset_at: ISO8601 } }` |
| Повтор при отказе? | **Не повторяем** без изменения конфига. Клиент рассчитывает время сброса квоты (`reset_at`) и может переопланировать процедуру или остановить текущий цикл. |

## Список посылок

1. **факт** — Контейнер запросов существует в `docs/audit/llm-calls` с полями для отслеживания вызовов (см. `ARCHITECTURE.md` §1.2).

2. **факт** — Слой каналов реализует резолвинг пары модель+провайдер на сервере с fallback на запасной канал, проверено в прогоне M0 (консилиум `request-container-api-2026-07-21`).

3. **норма** — Agент остаётся без ключей в процедурах; секреты хранятся на сервере (штормовое решение T2, `ARCHITECTURE.md` §2).

4. **норма** — Вызов внешнего API происходит через заявку (агент формулирует, сервер исполняет); контракт заявки должен быть проверяемым (требование комнаты K1).

5. **факт** — `background-office` принимает HTTP-запросы к `/api/llm-call` и подобным маршрутам без текущего гейта на полномочия агента (код лежит в корне).

6. **норма** — Ответственность за исполнение лежит на сервере; полномочия раздаются там, у клиента их копия для локальной фильтрации (штормовое решение T4).

7. **факт** — `AuthResponse` возвращает токен и метаданные агента, но не содержит явного перечня процедур, которые агент может вызвать.

8. **норма** — Граница должна быть проверяемой на сервере перед исполнением запроса, а не только на клиенте (архитектурное правило `ARCHITECTURE.md` безопасность).

9. **норма** — Отказ системы на границе полномочий должен различаться от ошибки исполнения (два разных сценария требуют разных действий клиента).

10. **норма** — Логирование событий на границе кабинета обязательно для аудита и отладки (требование комнаты K1 к видимости).
