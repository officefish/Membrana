[Teamlead]: Принимаю разделение Ozhegov — это и есть конструкция дозора без украшательств. Напоминаю ограду: мы решаем ГДЕ живёт проверка, КАКОЙ порог, КАКОЙ каденс. Версии vite/fastify — это M2, не сюда. Даю форму решения: Dependabot включаем как корень, `yarn npm audit` встраиваем в ритуал, порог маппим на действия реестра, каденс фиксируем. Дальше — детали от вас.

[Верстальщик]: Со своей стороны замечу: у дозора должен быть **видимый артефакт**, иначе он невидим и его перестают читать. Не UI-компонент в клиенте (это не наша зона тут), а строка в выводе ритуала: «audit: 2 critical / 34 high» с явной иерархией и без визуального шума. Критичное — сверху, контрастно; low — приглушённо. Честные числа, `tabular` выравнивание в логе, никаких «всё ок» поверх реальных critical.

[Музыкант]: Про место в ритуале. Утро или вечер? Я за **утро** — `yarn ritual:day`. Если critical всплыл ночью через Dependabot, утром мы его видим первым делом и планируем день вокруг него, а не узнаём в 18:00, когда уже нет сил чинить. Вечер — для сверки, что за день ничего нового не прилетело.

[Математик]: Возражу частично. Dependabot и так работает 24/7 и шлёт алерт в момент публикации advisory — это и есть непрерывный дозор. `yarn npm audit` в ритуале — это **контрольный замер** локального lock-файла, который может расходиться с алертами GitHub (транзитивные, не покрытые Dependabot). Логика: Dependabot — событийный триггер (задача сразу при critical/high), `audit` — детерминированный срез. Ставить audit в оба ритуала избыточно; достаточно **утро — обязательно**, вечер — только быстрый diff «появилось новое?».

[Структурщик]: Тогда зафиксирую владельцев вывода, иначе «читает вывод» повиснет. Dependabot alerts → **Teamlead** (он превращает alert в задачу реестра, у него право заводить эпик). `yarn npm audit` в `ritual:day` → тот, кто ведёт ритуал дня (Teamlead), но техническую интерпретацию severity+разрез рантайм/dev делегируем **Математику** как ops-компетенции (Linux/security в его личном деле). [Teamlead]: Принимаю. Я — владелец решения «alert → задача»; Dynin — владелец интерпретации severity и разреза рантайм-прод/dev-тулинг. Теперь порог. Кандидат из брифа M0: critical/high → задача в реестр **сразу**; moderate → **планово**; low — **накопительно** к ближайшему апгрейду. Возражения по маппингу?

[Математик]: Порог принимаю с одним уточнением-разрезом. **Рантайм-прод** (fastify-стек, electron, tar, multer, ws, undici) и **dev-тулинг** (vite, vitest) реагируют по-разному:
- рантайм-прод: critical/high → задача сразу; moderate → планово; low → накопительно;
- dev-тулинг: critical → сразу; high → планово (эксплуатация только на машине разработчика, не в проде); moderate/low → накопительно.
Это не смягчение ради удобства — это разная численная модель угрозы: у dev-тулинга нет сетевой поверхности прода.

[Музыкант]: Поддерживаю разрез. Но настаиваю: **critical — всегда сразу, в обоих разрезах**, без обсуждения. Критикал не спит.

Три корзины, три чётких определения — без синонимов-слоёв.

[Верстальщик]: По видимости вывода добавлю DoD-требование: в `ritual:day` шаг audit печатает **сводку по severity + разрез рантайм/dev одной строкой** и подсвечивает наличие critical/high цветом (danger-токен из DESIGN — красный для critical, warning для high). [Музыкант]: Про каденс пересмотра порога. Предлагаю **раз в квартал** плюс внеочередной пересмотр после любого инцидента класса patchhog. Порог — живой инструмент, а не высеченный камень; если за квартал разрез рантайм/dev перестал ловить реальную угрозу, чиним модель.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Где живёт дозор (пассивный) | **Dependabot alerts** для экосистемы **npm**, режим **только мониторинг** (без auto security-PR — это M2). Владелец вывода: **Teamlead** (alert → задача реестра). |
| Где живёт дозор (активный) | **`yarn npm audit`** — обязательный шаг в **`yarn ritual:day` (утро)**; лёгкий diff «появилось новое?» в `yarn ritual:evening`. Интерпретацию severity+разрез ведёт **Математик (Dynin)**. |
| Порог — рантайм-прод (fastify/electron/tar/multer/ws/undici) | critical/high → **задача в реестр сразу**; moderate → **планово** (вход в №1b/M2); low → **накопительно** к апгрейду. |
| Порог — dev-тулинг (vite/vitest) | critical → **сразу**; high → **планово**; moderate/low → **накопительно**. Основание: у dev-тулинга нет сетевой поверхности прода. |
| Три корзины действий | «сразу» = запись в `docs/tasks/registry.json` с label по зоне; «планово» = маркированный вход в апгрейд-спринт №1b (M2); «накопительно» = список, вскрываемый при плановом апгрейде. |
| Видимость вывода | Строка в `ritual:day`: сводка severity + разрез рантайм/dev; critical/high подсвечены (danger/warning) **и** текстовый префикс `[CRIT]`/`[HIGH]` (DESIGN: не только цвет). |
| Каденс пересмотра порога | **Ежеквартально** + **внеочередной пересмотр после инцидента** класса patchhog. Владелец: Teamlead; фиксация в протоколе следующей волны. Метрики пересмотра: доля advisory→задача и медиана «alert → задача». |
