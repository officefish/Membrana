# Консилиум: подготовительный спринт клиента office (panel.mmbrn.tech)

**Повестка:** Разрешить 5 пронумерованных развилок: (1) форма приложения `apps/panel` vs секция в кабинете — заодно вердикт drift-panel-placement #396; (2) auth-модель при stateless office с уровнями public/ally/operator/owner; (3) судьба публичного `/v1/drift-anchor/digest` и Q3-чеклист; (4) деплой Caddy/DNS/LE; (5) скоуп и форма эпика OP1..OPn.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант (Kuryokhin), Верстальщик (Rodchenko).

**Порядок реплик:** Верстальщик → Математик → Структурщик → Teamlead → Музыкант (циклически)

[Верстальщик]: По форме (развилка 1) я за `apps/panel` как отдельный SPA. Аудитория панели — оператор/владелец и нетехнические союзники; это другой визуальный контракт, чем кабинет пользователя продукта. Смешивать welcome-окно для союзников с рабочими экранами клиента — значит ломать иерархию и приоритет одного акцента на зону. Отдельное приложение даёт мне чистый layout без легаси-навигации кабинета.

[Математик]: Согласен с формой, но хочу зафиксировать вход развилки 2 численно. «Уровень доступа» — это не строка, а полный порядок: `public < ally < operator < owner`, монотонная функция `canAccess(role, section) = role ≥ section.minRole`. Default-deny из Q2 — это `canAccess = false` для всего, что явно не помечено `@Public()`. Это детерминированный предикат, тестируемый без БД и без UI. Требую, чтобы AuthGuard был именно таким предикатом, а не набором ad-hoc проверок.

[Структурщик]: Форма (a) — новый `apps/panel` — ложится в граф чисто: `apps/*` уже 7 приложений, turbo дёшев, изоляция по comms-прецеденту (CC1-CC9) — норма, а не украшательство. Секция в кабинете (b) создала бы синоним-слой: две аудитории под одним публичным API кабинета, связанность растёт. Термин фиксирую: `panel` — операторская витрина office, `cabinet` — кабинет пользователя продукта; это разные леммы, не сливать.

[Teamlead]: Принимаю форму (a) как решение владельца, research-дефолт (b) не переубеждает — разные аудитории плюс изоляция закрывают вопрос. Этим **вердиктом разрешаю развилку drift-panel-placement #396**: дрейф-борд и trends-таблица живут в `apps/panel`, не в кабинете. Панель — отдельное приложение под panel.mmbrn.tech, потребители-борды делаются после каркаса. Форма следует функции: у оператора своя конструкция экрана.

[Музыкант]: По форме без возражений — отдельная витрина честнее. Но помечу риск деплоя заранее (развилка 4): урок OM4-C с сожжённым LE rate-limit — это не «случайность», а следствие ретраев на неконсистентном DNS. Если панель поедет тем же путём — получим второй ожог. Предлагаю в DoD явный гейт: не трогать LE, пока NS не консистентен по всем резолверам.

[Верстальщик]: Возвращаюсь к auth с точки зрения UX (развилка 2). Для союзников-нетехнарей GitHub OAuth — барьер: у нетехнической аудитории telegram-спринта GitHub может не быть. Значит для уровня `ally` нужен invite-код/токен — низкий порог входа. Для `operator`/`owner` — сильная аутентификация. Welcome-окно (public) не требует логина вообще: видно всем, один акцент, кнопка «войти» вторична.

[Математик]: Тогда фиксирую гибрид как две независимые функции выдачи роли, а не один механизм. `operator/owner` → GitHub OAuth + allowlist по `user_id` (детерминированный файл, без БД). `ally` → invite-код из статического подписанного списка (HMAC от кода → роль, проверка без БД, `O(1)`). Оба сходятся в один предикат `role`. Хранение — httpOnly cookie (secure, sameSite), НЕ localStorage; это требование Q2, не вкусовщина.

[Структурщик]: Здесь важно не завести mini-БД украдкой. Invite-коды как persisted-таблица = связывание панели с состоянием, а office stateless по ADR 0004 Р2. Поэтому: allowlist операторов — файл в репо/секрете; ally-коды — подписанные (HMAC-secret в env), проверяются чистой функцией без хранилища. Отказываю варианту (а) «переиспользовать background-cabinet auth» — он связывает панель с кабинетом и его Postgres, нарушает изоляцию, которую мы только что выбрали.

[Teamlead]: Принимаю auth-модель: office остаётся stateless. `operator/owner` = GitHub OAuth + allowlist-файл; `ally` = подписанный invite-код (HMAC, без БД); `public` = welcome-окно без логина. Cookie httpOnly + CSRF, default-deny AuthGuard + явный `@Public()` на витрину — как в Q2. Отклоняю привязку к cabinet-auth (Ozhegov прав по изоляции) и статический общий токен (не различает роли). Роль выдаёт office по предикату Dynin.

[Музыкант]: По CSRF для cookie — раз уж мы на httpOnly, нужен double-submit token или sameSite=strict на мутациях. Но замечу: панель на этом этапе почти read-only (дайджесты, таблицы). Мутаций мало. Не будем городить тяжёлый CSRF-контур там, где нет записи — sameSite=lax + no-store хватит для read-only, а точечный CSRF добавим, когда появится первая мутирующая ручка. Иначе это украшательство.

[Верстальщик]: Согласен, для read-only каркаса CSRF не раздуваю. По welcome-окну: тёмная тема DESIGN.md, один заголовок-акцент `--color-accent`, короткий текст «что это», кнопка входа. Для союзников — человеческий язык (ALLY_PRIMER.md), без жаргона. Состояния login/error/loading обязательны с самого начала — это несущая конструкция, не опция.

[Математик]: Развилка 3 — судьба публичного digest. Численно: `/v1/drift-anchor/digest` уже отдаёт агрегаты (3 строки якорей, возраст), не сырьё. По Q3 главный вектор — прямые API за панелью и утечка операционных данных. Агрегированный дайджест — это уже data-минимизация. Значит его можно оставить публичным без нарушения Q3, при условии `Cache-Control: no-store` и rate-limit. Обещание ADR 0004 Р3 сохраняется.

[Структурщик]: Поддерживаю оставить digest публичным, но сформулирую общий принцип для будущих панельных API, чтобы не решать каждый раз заново. Норма: любой новый office-эндпоинт по умолчанию за `operator`-уровнем (default-deny); публичным становится только явным `@Public()` + чек «отдаёт агрегат, не сырьё». Digest уже прошёл этот чек (решение владельца 2026-07-13). Один термин — одно правило.

[Teamlead]: Принимаю по развилке 3: `/v1/drift-anchor/digest` **остаётся публичным** (ADR 0004 Р3 не нарушаем), новые панельные API — default-deny + явный `@Public()`. В DoD из Q3-чеклиста берём: `Cache-Control: no-store` на JSON, rate-limit, data-минимизация (агрегаты), блок `/debug`/`/metrics`, аудит доступа. Панель — investigation trigger, не источник сырых данных. Это принцип, а не разовое исключение.

[Музыкант]: По деплою (развилка 4) конкретика: Caddy site-block `panel.mmbrn.tech` — static SPA (turbo build → скопированный `dist` в volume) + `reverse_proxy` к office API для `/v1/*`. Никакого отдельного бэкенда у панели нет — она бьёт в существующий office. Это упрощает: один процесс, один сертификат. Гейт LE: сначала `dig` по всем NS, потом только выпуск сертификата.

[Математик]: Как ops добавлю: LE-гейт делаю проверяемым скриптом (`.mjs`) — резолвим `panel.mmbrn.tech` через ≥2 независимых резолвера, сравниваем A-записи, только при консистентности запускаем issuance. Это защита от повторения OM4-C, детерминированная: одинаковый DNS-стейт → одинаковое решение go/no-go. Без ручного «на глаз».

[Структурщик]: Топология: `apps/panel` собирается turbo, артефакт `dist` кладётся в volume, который монтирует Caddy; API-запросы `/v1/*` проксируются на office-контейнер (тот же, что office.mmbrn.tech). Панель НЕ импортирует internals кабинета — только свой код + `@membrana/core` при необходимости общих типов. Публичный API office для панели — через существующие HTTP-контракты, не через прямой доступ к чужим модулям.

[Teamlead]: Принимаю деплой: Caddy site-block panel.mmbrn.tech (static SPA + reverse_proxy `/v1/*` → office), сборка turbo → volume, LE только после DNS-гейта (скрипт Dynin). Развилка 4 закрыта. Переходим к скоупу эпика (развилка 5).

[Верстальщик]: Скоуп каркаса с моей стороны: welcome-окно (public), shell с навигацией разделов, состояния auth (login/error/loading), заглушки разделов с бейджами уровня доступа. Out of scope — содержимое drift-борда и trends-таблицы, они потребители. Каркас = конструкция без наполнения, но с честными состояниями.

[Музыкант]: Согласен с out of scope. Явно вывожу за скоуп: сами борды drift/trends, LLM-фичи, любые мутирующие ручки, тяжёлый CSRF. Спринт — каркас, не контент. Иначе расползётся.

[Структурщик]: Форма эпика: OP1 scaffold `apps/panel` (SPA стек кабинета, turbo) → OP2 auth-контур office (GitHub OAuth allowlist + HMAC invite + AuthGuard default-deny + `@Public`) → OP3 welcome + shell разделов с уровнями (Верстальщик) → OP4 деплой Caddy/DNS-гейт/LE (Dynin ops + Музыкант) → OP5 Q3-hardening публичных API (no-store, rate-limit, аудит). Владельцы по зонам, границы пакетов не нарушены.

[Teamlead]: Принимаю форму эпика OP1..OP5 как предложил Ozhegov. **LGTM** на весь план при условии: границы (панель не тянет cabinet internals), stateless office сохранён, LE-гейт обязателен. Все пять развилок разрешены — да с планом. Собираю итог.

## Итоговое решение консилиума

| Вопрос (развилка) | Решение |
|--------|---------|
| 1. Форма приложения | **Да, `apps/panel`** — отдельный SPA (стек кабинета) на panel.mmbrn.tech. Отклонена секция в кабинете (разные аудитории + изоляция). **Вердикт drift-panel-placement #396: борды живут в panel, не в кабинете.** |
| 2. Auth-модель (stateless office) | `operator/owner` = GitHub OAuth + allowlist-файл; `ally` = подписанный HMAC invite-код (без БД); `public` = welcome без логина. httpOnly cookie (secure, sameSite), default-deny AuthGuard + `@Public()`. Отклонены: cabinet-auth (связность), общий статический токен (не различает роли), persisted invite-БД. |
| 3. Публичный digest + Q3 | `/v1/drift-anchor/digest` **остаётся публичным** (ADR 0004 Р3). Новые панельные API — default-deny + явный `@Public()` + чек «агрегат, не сырьё». Q3 в DoD. |
| 4. Деплой | Caddy site-block panel.mmbrn.tech (static SPA + reverse_proxy `/v1/*` → office). Сборка turbo → volume. LE только после DNS-консистентности (скрипт-гейт, урок OM4-C). |
| 5. Скоуп и форма эпика | Эпик **OP1..OP5** (scaffold → auth → welcome/shell → деплой → Q3-hardening). Out of scope: сами борды drift/trends, LLM-фичи, мутирующие ручки, тяжёлый CSRF. |

**Definition of Done:**
- `apps/panel` scaffold собирается через turbo; не импортирует internals `apps/cabinet`; зависит максимум от `@membrana/core`.
- AuthGuard = детерминированный предикат `canAccess(role, section)` с полным порядком `public<ally<operator<owner`; default-deny; юнит-тесты без БД/DOM.
- GitHub OAuth + allowlist-файл (operator/owner); HMAC invite-код (ally) — проверка чистой функцией без хранилища; office остаётся stateless (ADR 0004 Р2).
- Роль хранится в httpOnly cookie (secure, sameSite); localStorage не используется.
- Welcome-окно public по DESIGN.md (тёмная тема, один акцент, язык ALLY_PRIMER.md); состояния login/error/loading; a11y (`focus-visible`, `aria-label`).
- `/v1/drift-anchor/digest` публичный, `Cache-Control: no-store`, rate-limit; новые API default-deny + `@Public()`.
- Q3-hardening: no-store на JSON, rate-limit, data-минимизация, блок `/debug`/`/metrics`, аудит доступа.
- Деплой: Caddy site-block + reverse_proxy `/v1/*`; LE-issuance только после прохождения DNS-гейта (скрипт проверяет ≥2 резолвера).
