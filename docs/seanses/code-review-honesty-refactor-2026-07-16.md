# Консилиум: рефакторинг daily code-review — сделать ревью честным

**Повестка:** Даёт ли расследование (генератор `daily` code-review структурно слеп к коду дня — видит только имена файлов рабочего дерева, а не дифф коммитов) достаточное основание для рефакторинга rt-8? Какова целевая форма: что именно ревьюит `daily` — работу дня (коммиты за период) или остаток рабочего дерева? Deep research по практикам 2025–2026 — вход, не решение.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Структурщик → Верстальщик → Музыкант → Математик → Teamlead → (циклически)

[Структурщик]: Начну с термина, иначе спор поплывёт. «Code-review» у нас — омоним двух разных лемм: **ревью-по-запросу** (`--pr`/`--branch`/`--uncommitted`) — честное, получает полный дифф; и **daily-ревью** — читает `collectRepositoryContext`, который по самодокументу «не обходит дерево исходников». Это два разных инструмента под одним именем — и именно смешение имён породило слепоту. Расследование фиксирует: 15.07 день крупных слияний получил вердикт T0 docs-only. Основание для рефакторинга — не «баг», а расхождение имени и функции. Достаточно.

[Верстальщик]: С точки зрения потребителя вердикта — а потребитель это MAIN_DAY_ISSUE и человек утром — сейчас экран врёт. «✅ Runtime не затронут» в день крупных слияний — это мёртвая дверь: индикатор показывает зелёное там, где была работа. По моей норме (деградация — это состояние экрана, а не тишина) вердикт обязан честно называть источник: «оценено рабочее дерево» vs «оценена работа дня». Пока метка не различает эти два режима — доверие ко всему утреннему ритуалу подорвано.

[Музыкант]: Поддержу аналогией из моего контура: это как мерить качество трека по тому, что осталось в буфере после экспорта, а не по самому экспортированному потоку. Рабочее дерево после коммитов — пустой буфер. Ревьюить надо **поток дня** — коммиты за период. Риск-флаг: если возьмём «диапазон коммитов», надо честно определить границы периода, иначе поймаем чужие коммиты в диапазон — та же боль, что мы решали в ti-1 (squash-SHA → родитель, а не «HEAD перед pull»).

[Математик]: Формализую, что значит «работа дня», иначе останется вкусовым. Вход ревью — детерминированное множество коммитов `C = {c : merged_at ∈ [day_start, day_end)}`, дифф = `git diff <base>..<head>` где `base` — родитель первого коммита периода, `head` — последний. Это чистая функция от (период, git-лог) → дифф. Ключ из ti-1 переносится дословно: границу брать через `gh pr view --json mergeCommit` → squash-SHA → его родитель (`git show -s --format=%P`), НЕ «HEAD перед pull». Тогда диапазон воспроизводим и не захватывает чужое.

[Teamlead]: Рамку принимаю. Расследование даёт достаточное основание: имя `code-review` обещает ревью кода, функция `daily` его не выполняет — это нарушение соответствия формы и функции, мой профиль. Целевая форма: **daily ревьюит работу дня (дифф коммитов за период), а не остаток рабочего дерева**. Research — вход. Прошу пройти три развилки предметно: (1) источник границ периода, (2) что делаем со «слепым» `collectRepositoryContext`, (3) деградация, когда дифф не собрать (нет gh, локальный прогон).

[Структурщик]: Развилка 2 — несущая. `collectRepositoryContext` — это лемма «снимок рабочего дерева»; она не «сломана», она про другое. Не переписывать её в «сборщик диффа дня» — это смешает два словаря в одной функции. Ввожу вторую чистую лемму: `collectDayWorkDiff(period) → { commits, diff, stats }`. `daily` вызывает её; `collectRepositoryContext` остаётся для проверок гигиены рабочего дерева (untracked, чистота перед деплоем). Два имени — два смысла, без синонимов-слоёв.

[Верстальщик]: Тогда по развилке 3 — деградация как состояние. Если дифф собрать нельзя (нет `gh`, офлайн, нет сети до GitHub) — ревью не должно молча падать в старый режим и печатать «Runtime не затронут». Это ровно та ложь, что расследуем. Нужен явный badge-режим в шапке артефакта: `[режим: работа дня]` / `[режим: рабочее дерево — ДИФФ НЕДОСТУПЕН]`. Второй режим честно предупреждает, что вердикт неполон. Не только текст — машиночитаемый маркер, чтобы MAIN_DAY_ISSUE-гейт мог его прочесть.

[Музыкант]: По развилке 1 добавлю практический слой. `gh pr view` — сеть; в вечернем ритуале это хвост после archive:daily-day, и он может отвалиться (урок LE rate-limit: ретраи на неконсистентной сети жгут). Предлагаю двухуровневый источник границ: primary — `gh` (точный squash-SHA), fallback — локальный `git log --since=<day_start> --until=<day_end> --merges` без сети. Fallback помечается как «приблизительный» тем самым badge Родченко. `|| true`-семантика хвоста сохраняется: рефреш не роняет ритуал.

[Математик]: С fallback Музыканта соглашусь численно, но зафиксирую иерархию точности. `gh mergeCommit` даёт **точный** squash-SHA → детерминированный `base` = родитель. `git log --since/--until` по датам — **приблизительный**: зависит от таймзоны коммитера и может включить/пропустить пограничные. Значит fallback обязан нести флаг `precision: "approximate"` в структуре, а точный путь — `precision: "exact"`. Badge Родченко читает именно это поле. Без сети мы не обещаем точности — мы её честно понижаем. Это и есть научная честность вердикта.

[Teamlead]: Фиксирую развилки. (1) источник границ — двухуровневый: gh-exact → git-log-approximate, поле `precision`; принято. (2) новая чистая лемма `collectDayWorkDiff`, `collectRepositoryContext` не трогаем по смыслу — принято, это профиль Структурщика. (3) деградация — явный machine-readable режим в шапке + `precision`, никакого молчаливого «Runtime не затронут» — принято. Теперь граница: rt-8 — это M, только генератор ревью. Не расширяемся в MAIN_DAY_ISSUE-гейт (это `main-day-probe-gate`, #533) — только отдаём ему честный маркер.

[Структурщик]: Согласен с границей M и с тем, что маркер — контракт на стыке, а не логика гейта. Уточню размещение: `collectDayWorkDiff` — чистая функция в `scripts/lib/` (организационный слой, пакетный граф `ARCHITECTURE.md` не трогается, как в rt-серии). Тесты — на границу периода (пустой день, день с одним PR, день с чужими пограничными коммитами), потому что именно там живёт баг. Без этих трёх тестов лемма не считается принятой.

[Математик]: Тесты Структурщика беру и добавлю четвёртый — **регрессия 15.07**: скормить генератору тот самый день крупных слияний и проверить, что вердикт НЕ `T0 docs-only`, а отражает runtime-изменения. Это золотой тест против рецидива: фиксирует ровно ту ошибку, ради которой всё затеяно. Детерминированно, воспроизводимо, без сети (фикстура git-лога дня).

[Музыкант]: Поддержу золотой тест — это как эталонный трек для регрессии артефактов. И одно UX-ритма замечание: `collectDayWorkDiff` в вечернем хвосте не должен блокировать. Если `gh` висит — таймаут (5–10 с) → fallback, а не зависание ритуала. Мягкая деградация с честным badge лучше, чем точный вердикт ценой висящего вечера.

[Верстальщик]: И по читаемости самого артефакта: шапка `DAILY_CODE_REVIEW.md` получает три явных поля — `режим`, `precision`, `период (base..head, N коммитов)`. Числа — `tabular-nums`-стиль в смысле выравнивания markdown-таблицы. Оператор утром за одну секунду видит: что оценивали, насколько точно, за какой диапазон. Не только цвет/слово — полный контекст вердикта.

[Teamlead]: LGTM по форме. Собираю решение: расследование — достаточное основание; rt-8 = M, только генератор; `daily` ревьюит работу дня через новую чистую лемму; деградация честная и машиночитаемая; золотой тест 15.07 обязателен. Research остался входом — мы взяли из него идею диапазона коммитов, но границу закрепили нашим ti-1-каноном, а не чужой практикой. Все роли — подтвердите.

[Структурщик]: Принимаю. Границы словаря соблюдены: два имени — два смысла.

[Математик]: Принимаю. Диапазон детерминирован, precision честный, золотой тест фиксирует регрессию.

[Музыкант]: Принимаю. Деградация мягкая, ритм вечера не рвётся.

[Верстальщик]: Принимаю. Вердикт перестаёт врать, оператор видит режим и точность.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Даёт ли расследование основание для рефакторинга rt-8? | **Да.** Имя `code-review` обещает ревью кода, функция `daily` его не выполняет (слепа к диффу дня) — нарушение соответствия формы и функции. Кейс 15.07 (T0 docs-only в день крупных слияний) — документированный рецидивоопасный симптом. |
| Что ревьюит `daily`: работу дня или остаток рабочего дерева? | **Работу дня** — дифф коммитов за период `[day_start, day_end)`, а не снимок рабочего дерева. |
| Источник границ периода | Двухуровневый: primary — `gh pr view --json mergeCommit` → squash-SHA → родитель (`precision: exact`); fallback — `git log --since/--until --merges` без сети (`precision: approximate`). Канон границы — из ti-1 (родитель squash-коммита, НЕ «HEAD перед pull»). |
| Судьба `collectRepositoryContext` | **Не трогаем по смыслу.** Остаётся леммой «снимок рабочего дерева» (гигиена: untracked, чистота перед деплоем). Вводится новая чистая лемма `collectDayWorkDiff(period) → { commits, diff, stats, precision }`. |
| Деградация при недоступном диффе | **Честная и машиночитаемая.** Шапка артефакта: `режим` (`работа дня` / `рабочее дерево — дифф недоступен`) + `precision` + `период (base..head, N коммитов)`. Никакого молчаливого «Runtime не затронут». Таймаут gh 5–10 с → fallback, `|| true`-семантика хвоста сохраняется. |
| Граница задачи | rt-8 = **M**, только генератор ревью. MAIN_DAY_ISSUE-гейт (`main-day-probe-gate` #533) не трогаем — отдаём ему только честный маркер как контракт на стыке. |

**Definition of Done:**
- Новая чистая лемма `collectDayWorkDiff(period)` в `scripts/lib/` → `{ commits, diff, stats, precision }`; пакетный граф `ARCHITECTURE.md` не затронут.
- `daily`-ветка генератора вызывает `collectDayWorkDiff` и ревьюит дифф коммитов дня, а не снимок рабочего дерева.
- `collectRepositoryContext` сохранён для проверок гигиены рабочего дерева (по смыслу не изменён).
- Двухуровневый источник границ: `gh`-exact → `git-log`-approximate; поле `precision` в структуре.
- Шапка `DAILY_CODE_REVIEW.md`: поля `режим`, `precision`, `период (base..head, N коммитов)` — человеко- и машиночитаемые.
- Таймаут на `gh` (5–10 с) с fallback; вечерний ритуал не блокируется, `|| true`-семантика хвоста сохранена.
- Юнит-тесты границы периода: пустой день; день с одним PR; день с чужими пограничными коммитами; **золотой регрессионный тест 15.07** (день крупных слияний НЕ даёт вердикт `T0 docs-only`).
- Все тесты детерминированы и работают без сети (фикстуры git-лога).
