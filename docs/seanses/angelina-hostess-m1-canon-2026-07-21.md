[Структурщик]: Начинаю с рамки владельца, она же — предмет: developer-rhythm слишком
абстрактный, из него утро вычёркивается, рождается отдельный скилл. Термины развожу так:
скилл «утренний ритуал» — **проводник одной леммы «утро»**; developer-rhythm остаётся
проводником леммы «ритм дня» (вечер, read-order M/L). Абстракция и была дырой: скилл
«обо всём» дал холодной сессии старую таблицу шагов — прецедент 21.07, корень №4.

[Математик]: Формализую разбиение, иначе «вычеркнуть» останется вкусовым. Пусть S —
множество шагов старого скилла. Требую три предиката: **покрытие** `S_morning ∪ S_day = S`
(ни один шаг не потерян), **непересечение** `S_morning ∩ S_day = ∅` (нет двойного
источника истины), **отсутствие сироты** (каждый шаг имеет ровно один скилл-носитель).
Проверяемо тестом над списком шагов — без LLM.

[Teamlead]: Рамку держу: мы НЕ проектируем здесь сборку канона (B), гейты (G) и фоновую
хозяйку (H) — только каноны/скиллы. Архитектурно поддерживаю разделение: один скилл —
одна лемма — одна зона ответственности. Имя и триггеры — Структурщику.

[Структурщик]: Решение по новому скиллу: `membrana-morning-ritual` (канонический
`.cursor/skills/membrana-morning-ritual/SKILL.md`, тонкие зеркала `.claude`/`.agents`).
**Состав** — новый сценарий целиком: пред-ритуальный порядок (ветка утра = main,
эскалация при грязном дереве; чтение вчерашнего фидбека и открытых развилок ДО запуска),
цепочка с Ангелиной-стражем, двухгейтовое утро (magistral owner-choice из топ-3,
swallow-send с «ок»), запрет принимать магистраль от скрипта. **Триггеры description:**
«утро», «утренний ритуал», «ritual:day», «standup», «main-day-issue» — чтобы холодная
сессия по слову «утро» попадала в НОВЫЙ скилл, не в общий.

[Музыкант]: Флаг на маршрут сигнала: если новый скилл недоступен (файл потерян, зеркало
битое) — старый НЕ должен молча подхватывать утро. Это была бы мёртвая дверь: сломался
новый мир → тихо заиграл старый. Требую **failover-терминал**: недоступен
`morning-ritual` → СТОП с явной ошибкой, developer-rhythm утро не замещает.

[Верстальщик]: Со стороны читателя-агента: скилл должен кричать о своём статусе ДО тела.
Предлагаю фронт-маттер `status ∈ {live, deprecated, superseded}` во всех skills; при
`superseded` — обязательный `supersededBy: <path>` на существующий `live`. Deprecated/
superseded — иконка+текст «не источник истины», не только пометка. `skills/README.md` —
генерируемый ряд карточек с колонкой `status` (`yarn skills:sync-readme`), руками не
правится — как memory-журналы.

[Математик]: Беру статусную механику в предикат: `fresh(skill) = (supersededBy = ∅) ∧
(status = live)`. Холодная сессия при `superseded` обязана перейти по ссылке, тело не
читается. Цепочка `supersededBy` — DAG с обязательным live-терминалом; цикл — красный CI.
Всё проверяемо детерминированно.

[Структурщик]: Вычеркивание из developer-rhythm — не удаление в пустоту: утренние
разделы заменяются **ссылкой** «утро → membrana-morning-ritual» (не копией текста —
копия возродит двойной источник). Description старого скилла чистится от утренних
триггеров. Канон `DEVELOPER_RHYTHM.md` — той же операцией: утренний раздел → ссылка.

[Teamlead]: Проверяю против прецедента 21.07. Холодная сессия тогда: (1) открыла
developer-rhythm по слову «утро» → теперь description не отвечает на «утро», а тело
шлёт по ссылке; (2) пошла по старой таблице шагов → таблицы в старом скилле больше нет;
(3) память не сработала гейтом → статусная механика работает без памяти, на чтении.
Три из трёх закрыты. Четвёртый корень (ветка) — внутри нового скилла (пред-ритуальный
порядок). Принимаю.

[Музыкант]: Дополню терминал провенансом: переход по `supersededBy` и failover-стоп
пишутся в журнал (кто, откуда, куда) — иначе аудит не увидит, что сессии реально ходят
новым путём. Тихая маршрутизация = тот же молчун в профиль.

[Верстальщик]: По подаче итога: `docs:verify-canon` должен стать красным CI при
нарушении статусной механики — отсутствие `status`, битая ссылка `supersededBy`, цикл,
или утренний шаг в любом `live`-скилле кроме `morning-ritual`. Гейт, не соглашение.

[Математик]: Свожу тестируемое: (1) три предиката разбиения над множеством шагов;
(2) `fresh()` — чистая функция, юнит без сети/mtime; (3) ацикличность DAG `supersededBy`
с live-терминалом; (4) запрет утренних шагов вне `morning-ritual` — грепом по составу
шагов в CI. Всё детерминированно, LLM нигде не судья.

[Структурщик]: Порядок правок, чтобы между коммитами не жил полусценарий: (1) создать
`membrana-morning-ritual` (live); (2) в тот же PR — вычеркнуть утро из developer-rhythm
+ ссылка; (3) канон `DEVELOPER_RHYTHM.md` + зеркала; (4) гейт verify-canon. Один PR —
одна операция разделения, атомарно.

[Teamlead]: Границы: B (сборка канона генератором), G (гейты в код), H (фоновая
хозяйка) — не тронуты; статусная механика — обвязка канонов, не рантайм ритуала. Скоуп
чист. LGTM на форму; сборка вердикта — ниже, посылки — отдельной секцией, как требует
формат.

[Музыкант]: Принимаю. Один вдох тревоги оставляю в протоколе: статусная механика
защищает от чтения старого, но не от копипасты старого в новое — на это есть предикат
непересечения Математика. Пара работает только вместе.

[Верстальщик]: Принимаю. Прошу в DoD явную строку про генерируемый README со статусами —
чтобы колонка не появилась «когда-нибудь».

[Математик]: Финальная проверка инвариантов: разбиение тотально (три предиката), статусный
граф ацикличен, failover терминален, всё покрывается тестами без сети. Возражений нет.

[Структурщик]: Термины закреплены: «скилл-проводник» ≠ «канон-норма»; проводник ссылается,
норма описывает намерение; машинное (README, зеркала) генерится. Возражений нет.

[Верстальщик]: Последняя сверка подачи: строка статуса в шапке скилла — ДО описания и
триггеров, чтобы холодная сессия увидела её первой строкой чтения. Принято всеми.

[Teamlead]: Единый вердикт принят всеми. Фиксирую таблицу, посылки и DoD.
