У нас три разных объекта под угрозой слипания: **GitHub-issue** (удостоверение задачи), **Linear-issue как контейнер** (parent, движение) и **Linear sub-issue** (полноценный объект со своим статусом и assignee — это факт из фактуры). «Задача» — омоним минимум двух лемм: *единица работы* (то, что производится и принимается) и *единица движения* (то, что ползёт по колонкам). Пока мы это не развели, «закрыто» будет означать три разных события.

[Математик]: Формализую, иначе останется вкусовым. Пусть есть отношение ответственности `owns: Task → Persona`, требование заседания `team-execution-contour` — `|owns⁻¹(t)| = 1` для каждой задачи, где ответственная = принявшая выход. Вопрос E1 сводится к тому, на каком **объекте графа** мы объявляем `Task`. Кандидатов три: контейнер, центральная под-задача, дополнительная под-задача. Аксиома «ровно один ответственный» проверяема только если носитель `owns` — единственный тип объекта, иначе `owns` становится многозначным и инвариант `=1` неформулируем.

[Музыкант]: По касательной к аудио, но ритм графа я узнаю — это как проект в DAW: один **мастер-трек** (контейнер) и дорожки внутри (под-задачи). Мой живой флаг из `dads-integration-m1-preset-delivery`: два носителя одной сущности молча расходятся, если мастером не назначен один. Если ответственность повесить и на контейнер, и на под-задачу — это два мастера на один микс, гарантированный дрейф. Ответственный должен сидеть на **одном** уровне, а не размазываться по иерархии.

[Верстальщик]: С точки зрения потребителя — а потребитель это агент утром и человек в PR — у меня сразу боль **мёртвой двери** из `main-day-accuracy`. Если «закрыто» означает три разных события, экран покажет зелёную галочку на контейнере, пока центральная под-задача ещё живая. Читатель обязан видеть, **какое** «закрыто» сработало: контейнер, центральная или подчинённая. Без этого различения любой вечерний вердикт врёт.

[Teamlead]: Рамку фиксирую сразу, чтобы аксиома не колонизировала соседние фазы. Мы определяем **словарь счёта**, не читателя гейтов и не форму хранилища — это запрещено повесткой. По конструктивизму «форма следует функции»: функция ответственности — чтобы у каждого выхода был один человек, принявший его. Значит носитель `owns` = объект, у которого есть **принимаемый выход**. Контейнер выхода не производит, он движется; выход производит задача. Предлагаю рабочую гипотезу: единица ответственности = **GitHub-issue**, потому что владелец решил — Issue есть удостоверение конкретной задачи с конкретным исполнителем.

[Структурщик]: Поддержу, но уточню границу, иначе получим синоним-слой. GitHub-issue — удостоверение личности; Linear sub-issue — контейнер движения этой же личности. Один-к-одному? Фактура говорит: под-задача Linear имеет свой assignee и статус, а GitHub-issue незаконна вне Linear. Значит связь **центральная под-задача ↔ GitHub-issue** — это лемма-мост, одна сущность в двух носителях. Ответственность живёт на **GitHub-issue** как на удостоверении, а движется через привязанную Linear-под-задачу. Контейнер (parent) ответственности **не несёт** — он агрегатор.

[Математик]: Здесь надо проверить численно, а не на глаз. Если `owns` определён на GitHub-issue, а величины считаются на контейнере — получим двойной счёт. Разложу пять операционных величин по объектам. **Время до закрытия**, **доля задач без владельца**, **пропускная способность**, **доля эскалаций** — определены на единице ответственности, то есть на GitHub-issue (законная задача = есть Issue + есть один владелец). **Незавершённая работа (WIP)** — единственная, что естественно считается на **движущемся** объекте, то есть на Linear-под-задаче в активной колонке. Смешивать нельзя: `throughput` по Issue, WIP по под-задачам — но обе привязаны один-к-одному, так что дрейфа нет.

[Музыкант]: Тут соглашусь с оговоркой из своего контура. WIP по движущемуся объекту — это как счётчик активных дорожек в записи, он про **сейчас играет**. Но если центральная законно закрывается при незакрытой дополнительной (владелец так решил), то дополнительная продолжает висеть в WIP как «фоновый ревёрб, который забыли выключить». Надо явно сказать: дополнительная под-задача **не** добавляет отдельную единицу счёта к пяти величинам — иначе одна работа посчитана дважды, как три отражения одного снимка это `n=1`, а не `n=3`.

[Верстальщик]: С точки зрения потребителя закрою эту дыру визуально-семантически. Дополнительная под-задача — не самостоятельная строка в счётчике, а **атрибут** центральной. Иначе экран покажет две задачи там, где владелец видит одну. По моей норме честности из `code-review-honesty`: индикатор не имеет права дробить одну единицу работы на две галочки. Событие для величин = **закрытие центральной** (она же — закрытие её GitHub-issue). Закрытие контейнера — производное, наступает когда центральная закрыта; закрытие подчинённой — не событие для гейта вовсе.

[Teamlead]: Принимаю разложение Математика и уточнение про WIP. Свожу двухслойность Linear, это четвёртая обязательная вещь. `assignee` (человек, отвечает за результат) + `delegatedAgent` (агент, отвечает за выполнение) — это **не то же самое**, что «принявшая выход + производитель». Наследство `team-execution-contour`: ответственная = **приняла выход**, а не физически произвела. Значит `assignee` = наша единица ответственности (принявшая выход), а `delegatedAgent` — исполнитель. Но фактура прямо говорит: агент не может быть единственным ответственным. Это совпадает с нашим каноном: `leadPersona` в карточке = принявшая выход, живёт на `assignee`.

[Структурщик]: Здесь возражу против прямого склеивания, иначе термин поплывёт. «Принявшая выход» и `assignee` совпадают по **функции** (один человек, отвечает за результат) — это чистое соответствие. А вот `delegatedAgent` ≠ «физический производитель» из старого канона автоматически: в старом заседании производитель был любой ролью-персоной, а `delegatedAgent` в Linear — конкретный слот агента. Соответствие есть, но это **адаптер-мост**, а не тождество. Запишем: `assignee` ↔ ответственная (тождество по функции); `delegatedAgent` ↔ исполнитель (соответствие через адаптер, не склеивать имена).

[Математик]: Проверю запрет двойного счёта формально — это пятая обязательная вещь. Пусть `U` — множество единиц счёта. Двойной счёт исключается тогда и только тогда, когда каждая работа отображается **ровно в одну** единицу. Правило: `count-unit(work) = центральная под-задача ⟺ её GitHub-issue`, биекция. Контейнер и дополнительные под-задачи **исключены** из области счёта по построению: контейнер = агрегатор (не производит выход), дополнительная = производная от центральной (не самостоятельный выход). Инвариант проверяем: `Σ по контейнерам ≠ счёт`, `Σ по центральным = счёт`. Один объект — одна величина.

[Музыкант]: Хочу зафиксировать один риск из практики, помечу как флаг. Если дополнительная под-задача **блокирующая** (по фактуре — `blockedBy` видна в GraphQL, но вебхуками не приходит), то она влияет на «время до закрытия» центральной, хоть и не считается отдельной единицей. Это как латентность в цепочке: не отдельный трек, но добавляет задержку в мастер. Значит блокирующая обязательна и её незакрытость **держит** центральную — а изолированная/производная дополнительная не держит. Владелец это уже решил направлением зависимости; мы только подтверждаем, что счёт это уважает.

[Верстальщик]: Поддержу и добавлю несущую про наблюдаемость. Раз блокирующие связи вебхуками не приходят, а видны только в GraphQL — любой счётчик «время до закрытия» обязан **явно показать**, какие блокеры учтены, иначе цифра непроверяема на экране. Моя норма из `main-day-accuracy`: величина без видимого основания — мёртвая дверь. Но это уже про форму отчёта и про то, что читают гейты — граница нашей комнаты, дальше не лезу, только помечаю для следующей фазы.

[Teamlead]: Правильно остановился — это фаза чтения гейтов, не наша. Свожу к вердикту. Пять вещей закрыты: (1) единица ответственности = **центральная задача = GitHub-issue**, на ней `assignee` = принявшая выход; (2) единица счёта четырёх величин = та же GitHub-issue, WIP = движущаяся Linear-под-задача (биекция с Issue, дрейфа нет); (3) «закрыто» для величин и гейта = **закрытие центральной** (= закрытие её GitHub-issue); закрытие контейнера производно, закрытие подчинённой — не событие; (4) `assignee` ↔ ответственная (тождество), `delegatedAgent` ↔ исполнитель (адаптер, не склеивать); (5) двойной счёт исключён биекцией центральная↔Issue, контейнер и дополнительные вне области счёта.

[Структурщик]: Принимаю. Словарная статья вышла однозначной: контейнер — движение, центральная — единица счёта и ответственности, дополнительная — атрибут центральной. Ни одного синонима-слоя.

[Математик]: Принимаю. Инвариант `|owns⁻¹(t)| = 1` формулируем и проверяем, биекция центральная↔Issue исключает двойной счёт численно. Возражений нет.

[Музыкант]: Принимаю. Один мастер на микс, дополнительные — не отдельные треки в счётчике. Флаг про блокирующие связи передан следующей фазе.

[Верстальщик]: Принимаю. Мёртвая дверь закрыта: одна работа — одна галочка, «закрыто» на экране означает закрытие центральной, не контейнера.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| **Единица ответственности** | **Центральная задача = GitHub-issue.** На ней ровно один ответственный — `assignee` = персона, принявшая выход (`leadPersona`). Контейнер (parent Linear-issue) ответственности не несёт. |
| **Единица счёта величин** | Четыре величины (время до закрытия, доля задач без владельца, пропускная способность, доля эскалаций) — на **GitHub-issue = центральной задаче**. **WIP** — на движущейся Linear-под-задаче (биекция с Issue). |
| **Что значит «закрыто»** | Событие для величин и гейта = **закрытие центральной задачи** (= закрытие её GitHub-issue). Закрытие контейнера — производное (наступает при закрытой центральной). Закрытие подчинённой под-задачи — **не событие**. |
| **Двухслойность Linear** | `assignee` ↔ ответственная (принявшая выход) — **тождество по функции**. `delegatedAgent` ↔ исполнитель — **соответствие через адаптер**, имена не склеивать. Агент не может быть единственным ответственным. |
| **Запрет двойного счёта** | Биекция **центральная под-задача ↔ GitHub-issue**; каждая работа → ровно одна единица. Контейнер (агрегатор) и дополнительные под-задачи (атрибут/производная центральной) **исключены** из области счёта. |

**Definition of Done (для E1):**
- Словарь E1 зафиксирован: контейнер = движение; центральная = единица счёта и ответственности; дополнительная = атрибут центральной, вне счёта.
- Инвариант `ровно один ответственный на единицу` формулируется на GitHub-issue и проверяем.
- Разложение пяти операционных величин по объектам зафиксировано (4 — на Issue, WIP — на под-задаче, биекция).
- Соответствие `assignee`↔ответственная (тождество) и `delegatedAgent`↔исполнитель (адаптер) записано без склейки имён.

## Список посылок

- **GitHub-issue — удостоверение личности задачи, всегда конкретная задача с конкретным исполнителем; задача без Issue незаконна** — *норма* (решение владельца).
- **Linear-issue — контейнер движения; покрывает несколько GitHub-issue; несёт ровно одну центральную задачу и может нести дополнительные; без GitHub-issue незаконна** — *норма* (решение владельца).
- **Центральная законно закрывается при незакрытой дополнительной** — *норма* (решение владельца).
- **Дискриминатор дополнительной — направление зависимости: блокирующая обязательна, изолированная обязательна, дополнительная может быть только односторонне производной от центральной** — *норма* (решение владельца).
- **Движение живёт в Linear, содержание — в git** — *норма* (решение владельца).
- **Подзадача Linear — полноценный объект: собственный статус и собственный `assignee`, оба через API; чеклист в описании — не сущность** — *факт* (проверяемо по живому Linear API).
- **Ответственность в Linear двухслойная: `assignee` (человек, за результат) + `delegatedAgent` (агент, за выполнение); агент не может быть единственным ответственным** — *факт* (проверяемо по живому Linear API).
- **Блокирующие связи `blockedBy` / `blocking` видны в GraphQL, вебхуками не приходят** — *факт* (проверяемо по живому Linear API).
- **Наследство `team-execution-contour`: ответственность — ровно одна персона на задачу, та, что приняла выход, а не физический производитель** — *норма* (ратифицированный вердикт первого заседания).
- **Пять операционных величин: время до закрытия, доля задач без владельца, незавершённая работа, пропускная способность, доля эскалаций** — *норма* (ратифицированный вердикт первого заседания).
- **Носители следа — карточка реестра (`leadPersona`) и артефакт закрытия** — *норма* (ратифицированный вердикт первого заседания).
- **Вердикты первого заседания выносились при одной карточке, без отношений «один ко многим»** — *факт* (проверяемо по истории заседания).
