Не сломать 703 базовых теста, сохранить обратную совместимость bundled MVP и не трогать сценарии в пикере (только платформу).

Дух ND3: тревога стартует по `combinedScore ≥ threshold`, где combined приходит из `DetectionFusion` main-тика (`presentCount>0 && combinedScore>=порога`). С точки зрения корректности это разные величины: `isDrone` от одного детектора — оценка с высокой дисперсией; `combinedScore` от fusion — сглаженная агрегированная статистика по 2–4 анализам. Смешивать источники front — ошибка контракта.

Если мы жёстко переключим front на fusion, мы потеряем чувствительность на первых кадрах. Значит вопрос не «isDrone или combined», а «какой порог и какое сглаживание combined» — чтобы не платить латентностью за корректность.

Сейчас, если loop переключается по одному сигналу, а карточки детекторов и ensemble-строка рисуют combinedScore — оператор видит рассинхрон: экран «в тревоге», а ensemble-прогресс ещё 40 %. Это прямое нарушение DESIGN.md §«Live-детекция»: сырой `isDrone` не вешаем на индикатор напрямую, только сглаженный стабильный статус.

Проблема в том, что `branch-on-detection` уже считает combined-порог, но на переключение лупов НЕ влияет — то есть у нас **два независимых механизма решения о тревоге**. Это дубль-фасад. Норма — один источник истины.

Дух ND3 и `WHITE_PAPER.md §8` (Single-Node Detection First, но fusion на одном узле) требуют: **alarm по combinedScore, не по одиночному детектору**. ## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Источник входа в alarm-loop | **Да, вариант (б) с уточнением**: `make-detection-fusion` при наличии в графе становится последним писателем `lastDetection`, пишет сырой `combinedScore` + `presentCount`. Соло-`isDrone` как источник front — убрать. |
| Кто бинаризует combined | Рантайм переключения лупов через чистую функцию `front(lastDetection, threshold)` = `presentCount>0 && combinedScore>=threshold`. Fusion остаётся чистым продюсером метрики (без знания о топологии). |
| Порог | Единый: из узла `branch-on-detection` в main-графе; дефолт **0.5** при отсутствии узла. Один порог и для ветвления, и для лупа. |
| Дребезг / мигание сирены | Отдельный слой `loop-transition-policy`: асимметричный гистерезис (вход `threshold`, удержание `threshold − 0.15`) + debounce N=3 кадра. Единственный, кто дёргает main↔alarm. |

**Definition of Done:**
- `make-detection-fusion` пишет `combinedScore ∈ [0,1]` + `presentCount` в `lastDetection`; является последним писателем **только** при наличии узла в графе.
- Чистая функция `front(lastDetection, threshold)` + `loop-transition-policy` (гистерезис Δ=0.15, debounce N=3) — покрыты юнит-тестами без графа/DOM.
- Порог читается из `branch-on-detection` main-графа, дефолт 0.5; единый источник для ветвления и лупа.
- Единственная точка переключения main↔alarm — `loop-transition-policy`; старый front по соло-`isDrone` удалён.
- 703 базовых теста device-board зелёные; сценарии в пикере не изменены (правки только в платформе); bundled MVP без fusion сохраняет поведение.
- Предусловие «молчащий пустой batch» (точка 3, вариант а) смёржено до/вместе с рефакторингом лупов.
- Три сценария соревнования проверены Музыкантом на слух перед живым тестом владельцем; сирена не мигает на границе порога.
- Точка 1 (type-mismatch) заведена отдельным тикетом со ссылкой на этот протокол.
