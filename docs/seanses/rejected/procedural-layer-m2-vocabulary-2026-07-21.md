# Консилиум: словарь категорий процедурного слоя (M2)

**Повестка:** E2 — словарь категорий процедурного слоя: закрытый, открытый или полуоткрытый с процедурой расширения; где он записан (носитель внутри дома слоя, решённого в M1) и чем проверяется, что процедура оперирует только объявленными категориями. Категории-кандидаты владельца: виртуальная команда, задача, ответственность, стратегическая цель.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Структурщик → Музыкант → Математик → Teamlead → Верстальщик (циклически)

[Структурщик]: Начну с термина, иначе E2 поплывёт. «Словарь категорий» — это две разные леммы, слипшиеся под одним словом. Первая: **множество допустимых терминов** (что процедура вообще имеет право называть). Вторая: **носитель этого множества** (файл, схема, enum). Повестка спрашивает про обе плюс третью — предикат проверки. По моей норме «граница пакета = граница словаря»: если процедурный слой оперирует категориями, то категории обязаны быть закрытым перечнем с одним владельцем-файлом, иначе каждый агент дорисует свою «виртуальную команду» и мы получим синонимы-слои, как «code-review» 16.07.

[Музыкант]: По касательной, аудио тут не задет, но ритм я узнаю — это как фиксированный набор дорожек в проекте перед сведением. Если дорожек ровно четыре и они названы, микс предсказуем; если каждый плагин добавляет свою шину «на лету», мастер поплывёт. Мой живой флаг из `procedural-layer-m1`: открытый словарь без процедуры расширения — это шумовая дорожка, которая молча просачивается в мастер. Голосую за **закрытый на старте**, с явной калиткой для добавления, а не за самозаселение.

[Математик]: Формализую, иначе «закрытый/открытый» останется вкусовым. Пусть словарь — множество `V = {виртуальная команда, задача, ответственность, стратегическая цель}`, `|V| = 4`. Процедура — функция `proc: term → action`. Требуемый инвариант: `∀ term, использованный процедурой → term ∈ V`. Это проверяемый предикат: линтер обходит все процедурные документы, извлекает употреблённые категории, считает разность `used \ V`. Если разность непуста — отказ с кодом возврата. «Открытый словарь» ломает предикат: `V` перестаёт быть константой, разность всегда пуста тривиально, проверка вырождается в тавтологию и ничего не ловит. Значит открытый = отсутствие проверки. Отвергаю.

[Teamlead]: Рамку фиксирую сразу, чтобы E2 не съел M3. Наш вопрос ровно три части: статус словаря, носитель, механизм проверки. Не «где граница слоёв» (это M3) и не «что делает каждая категория» (это словарные статьи, следующий слой). По конструкции «форма следует функции»: функция словаря — быть замком, а не декором. Замок с открытой дужкой — не замок. Поддерживаю закрытый перечень из четырёх категорий владельца. Но требую **процедуру расширения через тот же гейт, что и любой канон**: PR + человек-ратификация, не автозаселение. Это не «полуоткрытый словарь» — это закрытый словарь с легальной калиткой, где калитка = решение владельца.

[Верстальщик]: С точки зрения потребителя — а потребитель здесь агент, читающий процедурный документ, и человек в PR — словарь обязан быть **видимым в одном месте**, не размазанным по десяти файлам. Моя норма из `main-day-accuracy`: если категория употреблена, а её нет в перечне, читатель обязан увидеть отказ, а не молчаливое «сойдёт». Мёртвая дверь здесь — документ, который называет пятую категорию, а линтер зелёный, потому что словарь открыт. Поддерживаю закрытый перечень + видимую шапку «категории слоя: 4» в самом носителе.

[Структурщик]: Тогда назову носитель точно, иначе «в одном месте» останется лозунгом. Дом слоя решён в M1 — процедурный слой живёт в своём каталоге. Носитель словаря: **машиночитаемый JSON** `categories.json` внутри дома (`id`, `title`, `definition-ref`) как единственный источник истины, плюс сгенерированная из него markdown-таблица для человека — не две рукописные копии, иначе повторим `dads-integration-m1-preset-delivery`, где TS и JSON молча разъезжались. Один владелец-файл, генерация вниз, а не рукописная синхронизация вбок.

[Музыкант]: Поддержу единый мастер-файл — это ровно урок про два носителя одного пресета, я на нём обжигался. Но добавлю флаг про калитку Тимлида: расширение через PR — правильно, только процедура расширения обязана сама быть **записана рядом со словарём**, а не жить в устном предании. Иначе через месяц никто не вспомнит, как легально добавить пятую дорожку, и начнётся самосев в обход. Строчка в шапке `categories.json`: «расширение — только PR + ратификация владельца».

[Математик]: Уточню предикат проверки, чтобы он не был декоративным — иначе повторим находку аудита 19.07, где `meeting-audit.mjs` молча искал не тот файл и рапортовал «нарушений 0». Линтер обязан: (1) читать `V` из `categories.json`, не из хардкода; (2) извлекать категории из процедурных документов по явной разметке (не по вольному тексту — распознавание в прозе недетерминировано); (3) при пустом множестве найденных документов — **отказ, не успех**, иначе «проверил ноль файлов» читается как чистота. Это чистая функция `lint(docs, V) → {violations[]}`, детерминированная, без сети.

[Teamlead]: Принимаю носитель Структурщика и предикат Математика. Ужесточу одно: разметка категорий в документах должна быть машинной — иначе пункт (2) Математика повиснет на регэкспе по прозе, а это тот самый недетерминизм, что уронил комнату M0 четыре раза. Пусть процедурные документы объявляют использованные категории явным front-matter или тегом `[category:<id>]`, линтер сверяет теги с `V`. Прозу не парсим. Форма следует функции: если категория несущая — она объявлена конструктивно, а не угадывается.

[Верстальщик]: Согласен с машинной разметкой, но со стороны читателя добавлю: front-matter невидим в отрендеренном документе. Нужна **видимая на холсте** плашка «категории этого документа: задача, ответственность» вверху, генерируемая из того же front-matter — не рукописная. Иначе получим честный линтер и слепого человека: машина знает, читатель нет. Одна разметка — два потребителя, генерация в обе стороны.

[Структурщик]: Свожу в контракт, чтобы не осталось синонимов. Три артефакта, один источник: `categories.json` (истина, `|V|=4`) → генерит markdown-таблицу словаря (для человека) и питает линтер (`V` для предиката). Документы объявляют категории во front-matter (`categories: [task, responsibility]`), линтер проверяет `frontmatter.categories ⊆ V`, рендер поднимает плашку из того же front-matter. Ноль рукописных копий множества. Расширение `V` — только PR в `categories.json` + ратификация владельца, процедура записана в шапке файла.

[Математик]: Закрываю численную сторону. Инвариант один: `used(doc) ⊆ V` для всех процедурных документов, где `used` берётся из front-matter (детерминированно), `V` — из `categories.json`. Предикат отказа: `∃ doc: used(doc) \ V ≠ ∅` ∨ `docs_scanned = 0`. Второй дизъюнкт защищает от находки 19.07. Функция чистая, вход `(docs, categories.json)`, выход `violations[]`, без `Date.now()`, без сети. Это тестируемо на фикстуре из двух документов — один законный, один с чужой категорией.

[Teamlead]: LGTM по форме. Свожу вердикт: словарь **закрытый**, `|V|=4` (виртуальная команда, задача, ответственность, стратегическая цель), носитель — `categories.json` в доме слоя, проверка — детерминированный линтер по front-matter с защитой от нулевого сканирования, расширение — PR + ратификация владельца. Границы M2 удержаны: где проходит граница слоёв — вопрос M3, что означает каждая категория — словарные статьи следующего слоя. Не открываю их здесь.

[Верстальщик]: Принимаю. Единственное дополнение в DoD со стороны потребителя: markdown-таблица словаря и плашка в документах должны нести **источник и дату генерации** («сгенерировано из categories.json»), чтобы читатель не принял устаревшую копию за истину — это моя норма против мёртвой двери.

[Музыкант]: Принимаю. Мастер один, дорожек четыре, калитка записана рядом с пультом. По моему контуру возражений нет — аудио не задет, ритм словаря чистый.

[Структурщик]: Принимаю. Один источник, ноль синонимов-слоёв, граница словаря = граница дома слоя. Термин закрыт.

[Математик]: Принимаю. Предикат детерминирован, защита от нулевого сканирования на месте, `V` — честная константа из файла.

[Teamlead]: Принимаю и фиксирую как вердикт M2. Ратификация владельцем — следующий гейт.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Статус словаря | **Закрытый**, `|V| = 4`: виртуальная команда, задача, ответственность, стратегическая цель |
| Носитель | `categories.json` в доме процедурного слоя (решён в M1) — единственный источник истины (`id`, `title`, `definition-ref`) |
| Производные артефакты | Markdown-таблица словаря (для человека) + плашка категорий в документах — **генерируются** из `categories.json`, не рукописные |
| Разметка в документах | Машинный front-matter `categories: [...]`; проза не парсится |
| Механизм проверки | Детерминированный линтер: `used(doc) ⊆ V` из front-matter; **отказ при `docs_scanned = 0`** (защита от находки аудита 19.07) |
| Процедура расширения | PR в `categories.json` + ратификация владельца; процедура записана в шапке файла (не устное предание) |
| Границы (не решаем здесь) | Граница слоёв — M3; значение каждой категории — словарные статьи следующего слоя |

**Definition of Done:**
- `categories.json` создан в доме процедурного слоя, `|V| = 4`, поля `id` / `title` / `definition-ref`, шапка с процедурой расширения.
- Генератор markdown-таблицы словаря из `categories.json`; таблица несёт источник и дату генерации.
- Линтер `lint(docs, categories.json) → violations[]` — чистая функция, без сети/`Date.now()`; предикат `used(doc) ⊆ V`; **отказ при нулевом сканировании**; покрыт фикстурой (законный документ + документ с чужой категорией).
- Процедурные документы объявляют категории во front-matter; рендер поднимает видимую плашку из того же front-matter (с указанием источника генерации).
- Ноль рукописных копий множества `V`.
