# Консилиум: E3 — конституция границы слоёв (процедуры → киты → скрипты)

**Повестка:** E3 — чем обеспечивается направленность «процедуры → киты → скрипты», чем даётся зуб на нарушение, где живёт парсер ответственности как жилец (адрес, не механизм). Ответ обязан назвать: правило направленности и машинную проверку; статус кит-манифеста в ревью; адрес парсера ответственности с маршрутом расследования прецедента (вердикт → SHA правил → история правок).

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Teamlead → Математик → Структурщик → Верстальщик → Музыкант

[Teamlead]: Рамку фиксирую сразу, чтобы E3 не расползся в реализацию. Мы на узле L3 ратифицированного DAG: M1 дал дом слоёв, M2 — словарь, наш вопрос узкий — **конституция границы**. Три обязательных выхода из повестки: (1) правило направленности + его машинная проверка; (2) статус кит-манифеста в ревью; (3) адрес парсера ответственности с маршрутом расследования. По профилю «форма следует функции»: граница между слоями оправдана только если у неё есть зуб — правило без машинной проверки это декор, а декора без смысла у нас не существует. Не решаем, *как* писать парсер — решаем, *где он живёт* и *что он гарантирует*.

[Математик]: Формализую направленность, иначе «процедуры → киты → скрипты» останется стрелкой на слайде. Пусть слои — вершины `L = {proc, kit, script}` с рангом `rank(proc)=0, rank(kit)=1, rank(script)=2`. Правило направленности — предикат на рёбрах импорта: `∀ import(a→b): rank(a) ≤ rank(b)` запрещено, разрешено только `rank(a) ≥ rank(b)` для вызова вниз, а объявление зависимости — строго `rank(caller) < rank(callee)` нет — уточню: скрипт может звать кита, кит — процедуру-контракт, но **не наоборот**. Проверяемо: строим граф импортов, ищем ребро `rank(from) > rank(to)` в направлении объявления — если процедура импортирует скрипт, `rank 0 → rank 2`, это цикл вверх, exit 1. Это чистая функция `checkLayerDirection(importGraph) → violations[]`, без состояния.

[Структурщик]: Начну с термина, иначе граница поплывёт. «Кит» — это не синоним «сервиса» и не «утилита»: по M2-словарю кит — **несущая абстракция слоя посередине**, лемма «опора, на которой стоит скрипт и которую диктует процедура». Направленность Математика правильна, но машинная проверка обязана жить не в глазах ревьюера, а в `scripts/check-layer-direction.mjs` с чтением реального графа импортов (`tsconfig` references + `import`-парс). Зуб — exit-код в CI, не «LGTM на глаз». Кит-манифест — это `docs/.../kit-manifest.json`: словарная статья каждого кита (имя, слой, что диктует, что опирается). В ревью манифест — **обязательный вход**: PR, добавляющий кита без записи в манифест, отклоняется машиной, а не вкусом.

[Верстальщик]: С точки зрения потребителя вердикта — а потребитель это агент утром и человек в PR — направленность и манифест должны быть **видимы**, иначе получим мёртвую дверь: зелёный CI при живом нарушении. Моя норма из `main-day-accuracy`: любой вердикт врёт, если не показывает, на какой ревизии построен. Значит отчёт `check-layer-direction` обязан нести шапку: SHA графа, список рёбер-нарушителей с адресами `from → to`, а не абстрактное «нарушений: N». Про парсер ответственности как жильца: адрес — это то, что человек откроет и прочитает; если адрес есть, но отчёт нечитаем, дверь всё равно мёртвая.

[Музыкант]: По касательной, аудио тут не задет, но ритм маршрута я слышу как сигнальную цепь: source → эффект → выход, где нельзя, чтобы выход переподключался ко входу. Направленность `proc → kit → script` — ровно это: обратная связь вверх = самовозбуждение, свист в мониторах. Мой живой флаг из инцидента атрибуции 19.07: правило получает зубы на выходе (гейт), но молчит на входе — комната не знает требования. Так что машинная проверка должна не только ловить нарушение постфактум, но и быть **вызываемой локально** до PR, иначе автор узнаёт о свисте только на сведении.

[Teamlead]: Принимаю три уточнения и свожу к парсеру ответственности — третьему выходу. Повестка требует **адрес, не механизм**: где живёт жилец и какой у него маршрут расследования прецедента. Предлагаю форму: парсер ответственности — это не новый сервис, а **read-only маршрут** поверх уже существующих артефактов. Маршрут из повестки: вердикт → SHA правил → история правок. Структурщик, где физический адрес этого жильца, чтобы он не стал шестым «менеджером» без предмета?

[Математик]: До адреса — формализую сам маршрут, иначе «расследование» останется вкусовым. Три звена, каждое — чистая функция чтения: `verdict → rulesSHA` (в шапке вердикта записан хеш файла правил на момент решения), `rulesSHA → ruleText` (`git show <sha>:<path>`), `ruleText → history` (`git log --follow <path>`). Ключевое свойство — **детерминизм**: один вердикт всегда указывает на один SHA, один SHA всегда даёт один текст. Никаких `Date.now()`, никакого «текущего состояния файла» — только зафиксированный на момент вердикта хеш. Иначе через месяц вердикт будет ссылаться на переписанное правило, и расследование соврёт. Это ровно урок 19.07: воспроизводимость входа, не только детерминизм функции.

[Структурщик]: Адрес называю через границу словаря. Парсер ответственности не заводит новый пакет — он **жилец `docs/meeting/procedural-layer/` как регламент** + тонкий читатель `scripts/trace-accountability.mjs`. Регламент — словарная статья: «вердикт обязан нести поле `rulesSHA`; расследование = три `git`-команды по фиксированному SHA». Скрипт — не механизм ответственности, а **навигатор по следу**: на вход имя вердикта, на выход — цепь `verdict → SHA → git show → git log`. Он read-only, ничего не решает. Это критично по норме: ответственность живёт в правиле и в истории git, парсер только показывает маршрут. Если завести «сервис ответственности» — родим синоним-слой, который сам станет объектом расследования.

[Верстальщик]: Принимаю адрес, прошу про presentational-контракт следа. Чтобы `rulesSHA` не стал очередным незаполненным полем — как `githubIssueClosedAt`, ноль из восемнадцати, — он обязан быть **обязательным в шаблоне вердикта** и проверяться при записи. Мёртвая дверь здесь: вердикт без SHA выглядит валидным, а расследовать по нему нельзя. Значит гейт записи вердикта (по образцу `premisesSection` из M0-контура) отклоняет вердикт без `rulesSHA`. И отчёт `trace-accountability` рисует цепь как три блока с якорями — вердикт, правило-на-SHA, история — чтобы человек прошёл маршрут глазами, а не собирал команды руками.

[Музыкант]: Поддержу аналогией из моего контура: `rulesSHA` в вердикте — это как записать версии плагинов и семплрейт в проект, а не только исходники. Через год откроешь — свёл бит-в-бит, потому что зафиксировал цепь. Без SHA — мастер, который на новой машине звучит иначе. Один флаг риска: три `git`-команды в маршруте — это внешняя зависимость от целостности истории. Если ветку переписали (rebase, force-push), `--follow` соврёт. Значит норма из инцидента 19.07 — **история не переписывается** — не просто гигиена, а несущая опора парсера ответственности. Без неё маршрут рвётся.

[Teamlead]: Фиксирую — норма «история не переписывается» становится посылкой конституции слоя, а не отдельной гигиеной. Свожу три выхода. Есть ли возражения по существу перед вердиктом?

[Математик]: Возражений нет, одна оговорка к машинной проверке направленности: она обязана падать на **любом** ребре вверх, включая транзитивные через кита. `proc → kit → proc'` где `proc'` тянет `script` — цикл маскируется. Проверка строит полный транзитивный граф, не только прямые рёбра. Это условие на реализацию, граф E3 не меняет — помечаю как норму, не факт.

[Структурщик]: Согласен с транзитивностью. Добавлю в DoD термин-замок: кит-манифест и правило направленности ссылаются на **один** словарь слоёв из M2, без второго определения «кита». Два определения = синоним-слой, повод для ADR, а не для тихого расхождения.

[Верстальщик]: Принимаю. С витрины одно: отчёты обеих проверок (`check-layer-direction`, `trace-accountability`) несут SHA графа/правил в шапке — чтобы вердикт был отзываем адресно, а не декоративен.

[Музыкант]: Флагов больше не поднимаю. Ремарка на будущее: локальный вызов `check-layer-direction` до PR — обязателен, иначе зубы только на выходе, а вход глухой. Это вне нашей комнаты — задача носителя.

[Teamlead]: Свожу вердикт консилиума. Направленность — предикат ранга слоёв с машинной проверкой в скрипте и exit-кодом в CI (транзитивный граф). Кит-манифест — обязательный вход ревью, отклонение машиной. Парсер ответственности — read-only жилец: регламент в `procedural-layer/` + навигатор `trace-accountability.mjs`, маршрут `вердикт → rulesSHA → git show → git log`, опора — норма неизменяемости истории. LGTM.

**Definition of Done:**
- `scripts/check-layer-direction.mjs`: чистое ядро `checkLayerDirection(importGraph) → violations[]`, транзитивное замыкание графа, unit-тесты (прямое ребро вверх, транзитивное ребро вверх, чистый граф → 0), exit 1 при нарушении, отчёт с SHA графа + адреса рёбер, локальный запуск без CI.
- `kit-manifest.json` + гейт ревью: PR с новым китом без записи → отклонение (exit 1); формат словарной статьи по M2.
