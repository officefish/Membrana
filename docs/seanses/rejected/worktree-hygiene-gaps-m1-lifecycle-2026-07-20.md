# Консилиум: lifecycle рабочего дерева (worktree) — W1

**Повестка:** W1 — каким должен быть lifecycle рабочего дерева: кто/когда заводит, кто/по какому проверяемому предикату сносит, нужен ли гейт «нет хвостов» и куда он встроен, как формально отличать каноническое долгоживущее дерево от временного спринт-дерева.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Teamlead → Математик → Музыкант → Верстальщик → Структурщик (циклически)

[Teamlead]: Рамку фиксирую сразу, чтобы M1 не расползся в синхрон базы — это K1, соседняя комната. Наш вопрос ровно один: жизненный цикл дерева как процедура. По «форма следует функции»: worktree заводится под конкретную работу, живёт ровно пока эта работа не влита, сносится по проверяемому предикату. Нам нужны три вещи: событие «завести», событие «снести» с предикатом, и место гейта «нет хвостов». Каноническое дерево (`main`-worktree, персона-ветки `ozhegov`/`dynin`/`boyarskiy`) — исключение из процедуры сноса, оно долгоживущее по определению.

[Математик]: Формализую, иначе «lifecycle» останется вкусовым. Дерево — это состояние `w = (path, branch, hasUncommitted, hasUnpushed, prState)`. Предикат сноса должен быть **монотонным и проверяемым без сети там, где можно**: `safeToRemove(w) ⟺ ¬hasUncommitted ∧ ¬hasUnpushed ∧ prState ∈ {merged, closed}`. Первые две компоненты — локальные (`git status --porcelain`, `git log @{push}..`), третья требует `gh` — один сетевой вызов. Без всех трёх снос — потеря данных с ненулевой вероятностью, а «совпало, что пусто» — не доказательство.

[Музыкант]: По касательной к аудио, но ритм я узнаю — это как временные рендер-файлы при сведении. Пока микшируешь — bounce-файлы плодятся, свёл мастер — подчищаешь, иначе диск забит копиями одного трека в разных стадиях. Мой живой флаг из `repo-clean` 15.07: там снесли 93 мёртвых ветки и 4 worktree, и главная боль была именно **отличить временное от канона**. Персона-ветки чуть не попали под нож. Значит признак «каноническое» должен быть явным флагом, а не выводиться эвристикой из имени.

[Верстальщик]: С точки зрения потребителя — а потребитель здесь агент, который завтра наступит на чужой worktree, и человек в PR. Моя норма честности: состояние дерева должно быть **видимо**, а не выясняться постфактум `du -sh`. Нужен реестр или хотя бы `git worktree list` с аннотацией «temp/canon» и «safe/dirty». Мёртвая дверь здесь — worktree, который выглядит завершённым (PR merged), но держит незапушенный коммит: зелёно снаружи, потеря внутри. Предикат Математика ровно это закрывает — три компоненты, не одна.

[Структурщик]: Начну с термина, иначе DAG поплывёт. «Worktree» здесь — омоним двух лемм: **каноническое дерево** (долгоживущий рабочий контекст персоны или `main`, снос запрещён) и **спринт-дерево** (эфемерный контекст одной task-ветки, снос обязателен после влития). Это разные сущности с разным lifecycle, и их нельзя мерить одним предикатом. Каноническое различается не именем, а **явной записью** — как лемма в словаре помечена «постоянная». Предлагаю носитель: список канонических путей/веток в конфиге (`repo:clean` уже имеет гард персона-веток — это де-факто такой список).

[Teamlead]: Принимаю разделение Структурщика — это несущая конструкция, а не косметика. Тогда процедура ветвится по типу: каноническое дерево заводится **вручную** (решение владельца/архитектора), в lifecycle сноса не входит вообще. Спринт-дерево заводится **автоматически** при старте task-ветки (`pr:ship` уже работает в worktree — эталон `pr-ship-worktree-merge`), сносится автоматически после merge. Гейт «нет хвостов» встраиваю в **две точки**: перед merge (не влить грязное) и в `repo:clean` (подмести пропущенное).

[Математик]: Уточню про монотонность предиката во времени, иначе гейт соврёт. `hasUnpushed` не монотонен: коммит может появиться между проверкой и сносом. Значит предикат обязан вычисляться **в момент сноса атомарно**, а не кэшироваться из отчёта. Формально: `remove(w)` начинается с повторного `safeToRemove(w)` под тем же процессом, и если false — отказ с кодом «найдены хвосты» (не сбой скрипта, находка — это находка, `exit-code-semantics-audit` 19.07). Кэшированный вердикт из вечернего аудита годится только как **сигнал к проверке**, не как разрешение.

[Музыкант]: Поддержу атомарность живой аналогией: нельзя решить «трек готов к экспорту» вчера и жать render сегодня — за ночь могли докрутить автоматизацию. Но добавлю флаг риска из своего опыта: у спринт-дерева бывает **брошенное состояние** — task-ветка, по которой PR так и не открыли, работа заглохла. Такое дерево не merged и не closed, предикат Математика скажет «не сносить» вечно. Нужен явный путь для этого: либо человек закрывает PR/ветку, либо дерево висит с меткой «заброшено, требует решения» — но молча не гнить.

[Верстальщик]: Ровно, и это моя мёртвая дверь на экране. Брошенный worktree без метки — зелёная тишина, за которой мусор. Предлагаю в аннотации `git worktree list` три класса, видимых сразу: `canon` (не трогать), `ready-to-remove` (merged/closed + чисто), `needs-attention` (грязно ИЛИ pr отсутствует/open дольше N дней). Последний класс — не автоснос, а сигнал человеку. По DESIGN честность: не только цвет — текст статуса и причина («незапушенный коммит», «PR не открыт»).

[Структурщик]: Согласен с тремя классами, но зафиксирую границу владельца предиката, иначе размажется. `safeToRemove` — **чистая функция** Математика, живёт в утилите (`scripts/*.mjs` или lib), один вход `w`, один выход `{class, reasons[]}`. `repo:clean` и merge-гейт — **потребители** этой функции, они не переопределяют логику, только вызывают и действуют по классу. Это развязывает: изменишь правило безопасности в одном месте — оба потребителя согласованы. Прямого дублирования проверки в двух скриптах быть не должно — это «спагетти» через копипаст.

[Teamlead]: Принимаю. Собираю процедуру целиком. Каноническое дерево: заводится вручную, помечено в конфиге канонических путей, lifecycle сноса не касается. Спринт-дерево: заводится при старте task-ветки, сносится автоматически при классе `ready-to-remove`, задерживается при `needs-attention` с видимой причиной. Предикат — одна чистая функция `classifyWorktree(w) → {class, reasons[]}` (владелец Математик). Гейт «нет хвостов» — атомарная проверка перед merge + периодический проход `repo:clean`.

[Математик]: Закрою численную часть. `classifyWorktree` детерминирована при фиксированном входе: `w` собирается из `git status --porcelain` (uncommitted), `git rev-list @{push}..HEAD` (unpushed), `gh pr view --json state` (prState). Порог для `needs-attention` по возрасту открытого PR — параметр, предлагаю 7 дней как дефолт (недельный ритм), но это норма, не факт — владелец может сдвинуть. Ошибка сети при `gh` → класс `unknown`, не `ready-to-remove`: недоказанное безопасным не считаем.

[Музыкант]: Мне ложится. Единственное — попрошу, чтобы канонический список включал не только персона-ветки, но и `main`-worktree и любые долгоживущие спринт-деревья формата, если владелец такое заведёт явно. Из `cowork-format-regulation` 14.07 помню: бывают спринты в три изолированных блока с поздней стыковкой — их деревья живут дольше одной ветки. Пусть флаг «canon» ставится решением, а не автоматом по сроку жизни.

[Верстальщик]: Поддержу и добавлю потребительскую деталь: вывод `repo:clean --report` (он уже есть, `canon-tooling-debt` 15.07) должен печатать три класса таблицей с колонками path / branch / class / reasons. Агент утром читает это как топливо следующего шага, не как стену. `unknown` из-за сети — отдельная строка с пометкой «проверь вручную», не прячется в `needs-attention`.

[Структурщик]: Термин закрою финально, чтобы реестр не поплыл. Каноническое дерево = запись в списке (path ИЛИ branch), спринт-дерево = всё остальное с worktree. Один термин — один носитель: список канонов живёт в **одном** месте (расширить существующий гард персона-веток `repo:clean`, не плодить второй конфиг). `classifyWorktree` сначала проверяет членство в каноне → класс `canon` без дальнейших вычислений; иначе считает три компоненты. Это и определение, и порядок вычисления.

[Teamlead]: Консенсус вижу. Фиксирую как решение — да, с пошаговым планом. Владельцы: `classifyWorktree` — Математик (dynin); интеграция в `repo:clean` и merge-гейт + список канонов — Структурщик (ozhegov) как обвязка; вывод `--report` таблицей — Верстальщик (rodchenko) по DESIGN. Музыкант — ревьюер класса `needs-attention` на предмет брошенных деревьев. LGTM даю на форму; существенный код — после scaffolding по этому протоколу.

[Математик]: Принимаю. Контракт зафиксирован: чистая функция, детерминизм, `unknown` при недоказуемости.

[Верстальщик]: Принимаю. Три класса видимы, причины текстом, `unknown` не прячется.

[Структурщик]: Принимаю. Два термина, один носитель канона, функция — единственный источник правила.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Кто/когда заводит дерево | **Каноническое** — вручную (решение владельца/архитектора), помечается в списке канонов. **Спринт-дерево** — автоматически при старте task-ветки (эталон `pr-ship-worktree-merge`). |
| Кто/по какому предикату сносит | Чистая функция `classifyWorktree(w) → {class, reasons[]}` (владелец Математик). Снос только при классе `ready-to-remove`; проверка **атомарна в момент сноса**, не кэшируется. |
| Предикат «безопасно сносить» | `ready-to-remove ⟺ ¬hasUncommitted ∧ ¬hasUnpushed ∧ prState ∈ {merged, closed}`. Сеть-ошибка `gh` → класс `unknown` (не сносить). |
| Классы дерева | `canon` (не трогать), `ready-to-remove` (чисто + merged/closed), `needs-attention` (грязно ИЛИ PR нет/open > 7 дн), `unknown` (сеть недоступна). |
| Нужен ли гейт «нет хвостов» и где | Да, в **двух точках**: атомарная проверка перед merge + периодический проход `repo:clean`. Оба — потребители одной функции, без дублирования логики. |
| Каноническое vs временное | По **явной записи** в списке канонов (path ИЛИ branch), не по имени/сроку. Один носитель — расширить существующий гард персона-веток `repo:clean`. |
| Брошенные деревья | Класс `needs-attention` с видимой причиной; не автоснос — сигнал человеку. Не гниют молча. |

**Definition of Done:**
- `classifyWorktree(w)` — чистая функция в `scripts/*.mjs`/lib: вход `w`, выход `{class, reasons[]}`; детерминизм при фиксированном входе; юнит-тесты на все четыре класса + `unknown`.
- Список канонических деревьев — единый носитель (расширение гарда персона-веток `repo:clean`), включает `main`-worktree, персона-ветки и явно помеченные долгоживущие деревья.
- Merge-гейт: атомарный повтор `classifyWorktree` перед влитием; отказ при не-`ready-to-remove` с осмысленным кодом (находка ≠ сбой, по `exit-code-semantics-audit`).
- `repo:clean --report`: таблица path / branch / class / reasons; `unknown` — отдельной строкой «проверь вручную».
- Порог возраста open-PR для `needs-attention` — параметр (дефолт 7 дней), задокументирован как норма.
- Отсутствие дублирования предиката: `repo:clean` и merge-гейт вызывают одну функцию.
