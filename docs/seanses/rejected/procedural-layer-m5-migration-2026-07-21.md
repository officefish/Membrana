# Консилиум: M5 — миграция существующих процедур в слой (сток цепи)

**Повестка:** E5 — форма включения существующих процедур в процедурный слой: сразу / по касанию / только новые; критерий «процедура мигрирована»; первые кандидаты и держатели.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Математик → Музыкант → Структурщик → Верстальщик → Teamlead (циклически)

---

[Математик]: Формализую, иначе «мигрирована» останется вкусовым. Пусть множество существующих процедур `P`, и для каждой определён предикат `migrated(p) → bool`. Три стратегии — это три разных отношения на множестве времени: «сразу» = `∀p: migrated(p)` к моменту T; «по касанию» = `migrated(p)` истинно только когда `p` редактируется после T; «только новые» = `migrated(p)=false ∀p∈P, =true ∀новых`. Первый вопрос — не какую выбрать, а **что вообще делает предикат истинным**. Без определения `migrated` мы будем спорить о темпе миграции фикции.

[Музыкант]: По касательной, аудио тут не задет, но ритм я узнаю — это как перевод старого каталога треков в новый формат тегов. Заставить всё разом — застрянешь на неделю и ничего нового не выпустишь. Мой живой флаг из миграций: массовый ремастеринг ломает то, что играло. Я за «по касанию» плюс явный список первых доноров — те процедуры, что и так на столе сейчас (ритуал, заседание), пусть первыми получат контейнер, раз мы их всё равно трогаем.

[Структурщик]: Начну с термина, иначе E5 поплывёт. «Мигрирована» — это омоним трёх лемм: (1) **контейнер существует** — `docs/procedures/<id>/` заведён; (2) **словарь заполнен** — держатель, вход/выход, границы (термины M2); (3) **грамматика соблюдена** — процедура описана по правилам M4. Миграция без всех трёх — это пустая папка, синоним-слой. Предлагаю `migrated(p) = container(p) ∧ vocabulary(p) ∧ grammar(p)`. Тогда «по касанию» безопасно: касаемся — дозаполняем до полного предиката, а не заводим огрызок.

[Верстальщик]: С точки зрения потребителя слоя — а потребитель это агент, который ищет процедуру по её контейнеру, и человек в PR — «только новые» рождает **мёртвую дверь**: половина процедур в слое, половина вне, и по внешнему виду не отличить, где искать. Моя норма честности: если слой существует, его граница должна быть **видима**. Значит нужен реестр `docs/procedures/registry.json` со статусом каждой: `migrated | legacy | pending`. Тогда «по касанию» не врёт — читатель видит, что ещё не переехало.

[Teamlead]: Рамку фиксирую сразу, чтобы M5 не превратился в план миграции всего сразу — это сток цепи, а не новый эпик. Наш вопрос ровно три части брифа: **форма включения, критерий, первые кандидаты с держателями**. По «форма следует функции»: функция слоя — чтобы процедура имела адрес и словарную статью, а не чтобы мы за вечер переписали шесть регламентов. Принимаю предикат Ожегова из трёх конъюнктов и стратегию «по касанию» Музыканта как базовую. Но требую от Верстальщика реестр со статусом — без него «по касанию» неразличимо от «забыли».

[Математик]: Поддержу предикат Ожегова, но уточню численно. `migrated = container ∧ vocabulary ∧ grammar` — это конъюнкция, значит **самое слабое звено определяет истинность**. Опасность: `container` дешёвый (mkdir), `grammar` дорогой (осмысленное описание). Если не развести стоимость, получим массовый `container=true` при `grammar=false` — папки-призраки, ровно антипаттерн M0-заседания. Предлагаю: реестр хранит три булевых поля раздельно, а `migrated` — производное. Тогда статус `legacy` честно значит «контейнера ещё нет», а не «папка есть, смысла нет».

[Музыкант]: Согласен разнести поля — это как стемы в проекте: не сводить в один мастер, пока каждая дорожка не готова. И добавлю про темп из своего опыта: «по касанию» без форсирования умрёт — процедуру, которую год не трогают, никто не мигрирует. Нужен один принудительный первый залп: те 2-3 процедуры, что мы и так держим в руках на этой неделе. Ритуал — очевидный донор, его каждый день гоняют. Заседание — второй, мы прямо в нём сидим.

[Структурщик]: Принимаю раздельные поля Дынина. Уточню границу реестра, иначе он станет вторым синонимом-слоем к `docs/tasks/registry.json`. Процедурный реестр описывает **процедуры** (ритуал, шторм, заседание, коворк, ночь) — повторяющиеся способы работы; task-реестр описывает **задачи** — разовые единицы. Не сливать. Ключ реестра процедур = `id` процедуры, значение = `{holder, container, vocabulary, grammar, homePath}`. `homePath` — ссылка на `docs/procedures/<id>/`, единственная точка входа, как `index.ts` для пакета.

[Верстальщик]: Реестр принимаю, holder — обязательное поле, иначе процедура без держателя это мёртвая дверь: никто не отвечает. По первым кандидатам: соглашусь с Музыкантом на ритуал и заседание, но добавлю видимость статуса в самом контейнере — `docs/procedures/<id>/README.md` с шапкой «статус: migrated/legacy, держатель: X, ревизия: hash». Чтобы читатель, открыв папку, а не реестр, тоже видел правду. Дублирование? Нет — реестр машиночитаем, README для человека, один источник (реестр), README генерируется.

[Teamlead]: Сведу к решению, спор сходится. Форма — **«по касанию» + один принудительный первый залп**. Критерий — `migrated = container ∧ vocabulary ∧ grammar`, три поля раздельно в реестре, `migrated` производное. Реестр `docs/procedures/registry.json`, отдельный от task-реестра (граница Ожегова), README генерируется из него (видимость Родченко). Первые кандидаты: **ритуал** и **заседание** — те, что в руках. Держателей назову: ритуал — Ангелина (ведущая контура исполнения), заседание — Teamlead (я, как фасилитатор консилиума/заседания).

[Математик]: Держателей проверю на непротиворечие. Ангелина — ведущая, не советующая; она держит рантайм ритуала — это согласуется с её ролью в VIRTUAL_TEAM_PROMPT. Заседание держит Teamlead — тоже согласуется, он фасилитатор консилиума. Но зафиксирую посылку явно: **держатель ≠ автор кода процедуры**, держатель = владелец адреса и словарной статьи, тот, к кому идут при расхождении. Иначе `holder` станет синонимом `leadPersona`, а это разные леммы.

[Музыкант]: Поддержу разведение держателя и автора — это как продюсер против сессионного музыканта: продюсер отвечает за трек в целом, играют другие. По залпу закрою свою тревогу: два донора на этот вечер — потолок. Третьим не гружу, иначе grammar поедет на скорость, а мы только что договорились grammar не занижать. Ритуал и заседание — и хватит на M5.

[Структурщик]: Согласен, два донора — граница объёма M5. Уточню критерий «мигрирована» для приёмки, чтобы не приняли огрызок: процедура считается мигрированной, когда `homePath` существует, README сгенерён из реестра, и все три поля `true` с provenance (кто заполнил vocabulary, на какой ревизии). Без provenance поля `grammar=true` — это самозаявление, которое M0-заседание уже ловило как ложь. Provenance = минимальная защита от папки-призрака.

[Верстальщик]: Provenance принимаю, добавлю в README видимую строку «vocabulary заполнен: <persona>@<hash>». По остальным процедурам (шторм, коворк, ночь, родня) — статус `legacy` в реестре с самого создания, чтобы дверь не была мёртвой: читатель сразу видит «эта ещё не переехала, идёт по старому адресу». Не `pending` — `pending` обещает скорый переезд, которого мы не планируем. `legacy` честнее: живёт, но вне слоя.

[Teamlead]: Принимаю `legacy` вместо `pending` — Родченко прав, `pending` даёт ложное обещание. Финализирую границы M5: мигрируем **два** (ритуал, заседание), остальные заводим в реестр статусом `legacy`, дальше — «по касанию». Provenance обязателен на каждое `true`-поле. Это сток цепи: M5 ничего вниз не отдаёт, потребляет M1–M4. LGTM на форму решения. Definition of Done ниже.

[Математик]: Последнее численное замечание для DoD. Критерий приёмки должен быть машинопроверяем: скрипт-валидатор реестра, который для каждой записи проверяет `container ⟺ существует homePath`, `migrated ⟺ (container ∧ vocabulary ∧ grammar)`, и что каждое `true`-поле несёт непустой provenance `<persona>@<hash>`. Иначе «мигрирована» опять станет вкусовым словом на ревью.

[Структурщик]: Согласен на валидатор — он же гарант границы между двумя реестрами: падает, если ключ процедуры пересёкся с task-id. Моя итоговая позиция: принимаю форму, предикат, границу реестров, два донора. Слой получает однозначный вход (`homePath`), однозначный словарь (три поля + holder), однозначную грамматику. Синонимов-слоёв нет.

## Итоговое решение консилиума

| Форма включения | **«По касанию» + один принудительный первый залп** (2 донора сейчас, остальные — по мере редактирования) |
| Критерий «мигрирована» | `migrated = container ∧ vocabulary ∧ grammar`, три поля раздельно, `migrated` — производное; каждое `true` несёт provenance `<persona>@<hash>` |

| Держатель | Владелец адреса и словарной статьи, точка обращения при расхождении; **≠ автор кода** процедуры (≠ `leadPersona`) |
| Объём M5 | Ровно 2 донора; сток цепи — вниз ничего не отдаёт, потребляет M1–M4 |

**Definition of Done:**
- Заведён `docs/procedures/registry.json`: все существующие процедуры (ритуал, шторм, заседание, коворк, ночь, родня) как записи; поля `{holder, container, vocabulary, grammar, homePath}`; `migrated` — производное.
- Для **ритуала** и **заседания**: создан `docs/procedures/<id>/`, все три поля `true` с provenance `<persona>@<hash>`, README сгенерён из реестра с видимой шапкой (статус, держатель, ревизия, «vocabulary заполнен: <persona>@<hash>»).
- Остальные процедуры: статус `legacy`, поля `container/vocabulary/grammar = false`, homePath пустой.
- Скрипт-валидатор реестра: проверяет `container ⟺ homePath существует`, `migrated ⟺ (container ∧ vocabulary ∧ grammar)`, непустой provenance на каждое `true`-поле, отсутствие пересечения ключей с `docs/tasks/registry.json`; включён в `yarn test:scripts`.
- Держатели зафиксированы: ритуал → Ангелина, заседание → Teamlead (Vesnin).
