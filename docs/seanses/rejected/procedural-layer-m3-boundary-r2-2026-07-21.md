# Консилиум: E3 — конституция границы слоёв (процедуры → киты → скрипты)

**Повестка:** E3 — конституция границы слоёв: чем обеспечивается направленность «процедуры → киты → скрипты», чем даётся зуб на нарушение, и где живёт парсер ответственности как жилец (адрес, не механизм). Ответ обязан назвать: правило направленности и его машинную проверку; статус кит-манифеста в ревью; адрес парсера ответственности с маршрутом расследования прецедента (вердикт → SHA правил → история правок).

[Музыкант]: Аудио тут не задет, голосую по касательной — но маршрут сигнала я узнаю: `процедуры → киты → скрипты` это ровно source → эффекты → выход. У меня один живой флаг из `dads-integration-m1-preset-delivery`: если направленность держится только на договорённости, две копии молча разъезжаются, как WAV и забытый MP3. Значит зуб обязан кусать не человека постфактум, а импорт на входе — иначе «выход» начнёт диктовать «источнику». И ещё: направленность должна быть слышна в артефакте, а не жить в чьей-то голове.

[Структурщик]: Начну с термина, иначе E3 поплывёт. «Направленность» — это лемма о графе импортов: ребро разрешено только `procedure → kit → script`, обратное и в обход — запрещено. Это то же правило, что `ARCHITECTURE.md` §1 держит для пакетов (`core` не знает потребителей). Машинная проверка — статический линтер на импортах: `depcruise`-подобное правило `no-upward-import` по трём слоям с явным списком адресов. Кит-манифест — это `index.ts` слоя китов: единственная словарная статья, через которую скрипт видит процедуру. Обход манифеста = `нарушена слабая связанность`, возврат с точным путём.

[Teamlead]: Рамку фиксирую сразу, чтобы E3 не расползся в переписывание тулинга. Мы отвечаем ровно на три вопроса брифа: правило направленности + его машинная проверка; статус манифеста в ревью; адрес парсера ответственности с маршрутом расследования. По конструктивизму «форма следует функции»: слой китов существует, потому что скрипт не должен знать внутренностей процедуры — манифест это несущая стена, не декор. Опираюсь на M1 (дом) и M2 (словарь): адреса и термины уже ратифицированы, мы кладём на них границу и зуб. Ничего нового не изобретаем.

[Математик]: Формализую, иначе «направленность» останется вкусовой. Пусть слои — упорядоченное множество `L = {procedure=2, kit=1, script=0}`, импорт `a→b` разрешён ⟺ `layer(a) > layer(b)` и `b` экспортирован через манифест слоя. Проверка — чистая функция `checkImports(graph) → violations[]`, детерминированная: один граф → один список нарушений, без `Date.now()`, без сети. Зуб — exit-код в CI: `violations.length > 0 ⇒ exit 1`. Это то же, что мы закрыли в `dads-integration-m1-preset-delivery`: пока предикат `equal` не машинный, `P(целостность)` монотонно падает со временем.

[Верстальщик]: С точки зрения потребителя — а потребитель здесь агент, читающий вердикт, и человек в PR — направленность обязана быть **видима**, не только логична. Моя норма из `main-day-accuracy`: если парсер ответственности выдаёт вердикт «правило X нарушено», в отчёте обязан стоять SHA версии правил и ссылка на историю правок — иначе это мёртвая дверь: зелёная галочка без адреса. Прошу заложить в формат отчёта парсера три поля: `verdict`, `rulesSha`, `blameTrail`.

[Музыкант]: Поддержу Верстальщика живой стороной. Парсер ответственности без `blameTrail` — это индикатор «export done», который горит, когда в буфере тишина. Обжигался на `code-review-honesty-refactor`: вердикт мерил не то дерево. Маршрут расследования должен быть проигрываемым: взял вердикт → достал SHA правил на момент вердикта → прокрутил `git blame` до автора правки. Если хоть одно звено не воспроизводится — вердикт не отзываем.

[Структурщик]: Уточню адрес, иначе парсер станет синонимом-слоем. Парсер ответственности — не новый механизм, а **жилец** уже существующего дома M1. По M1 (ратифицирован) контур процедур живёт в `scripts/`-контейнере с манифестом; парсер — это read-only функция в слое китов, `packages/` или `scripts/lib/` по адресу M1, но не в скриптах-исполнителях. Один термин — один смысл: «парсер ответственности» = функция `attributeViolation(verdict) → {rulesSha, blameTrail}`, а не отдельный сервис. Дублировать `git blame` своей реализацией — запрещено, вызывает существующий.

[Teamlead]: Согласен со Структурщиком по адресу, но зафиксирую статус манифеста в ревью — это второй вопрос брифа, его нельзя проглотить. Кит-манифест в ревью — **обязательный чек-лист LGTM**: без обновлённого манифеста слоя китов PR, добавляющий межслойную связь, я блокирую. Это не совет, а BLOCK-условие. Аналог из `evening-auditor`: индикатор не имеет права гореть зелёным без разбора. Манифест — несущая стена: меняешь связь слоёв — обновляешь статью, иначе стена трещит молча.

[Математик]: Добавлю про воспроизводимость маршрута — иначе `blameTrail` будет несравним между прогонами. Маршрут `вердикт → SHA правил → история правок` детерминирован ⟺ SHA правил зафиксирован **в момент вердикта**, а не вычисляется постфактум. Иначе `attribute(v)` сегодня и через месяц дадут разные `blameTrail` — правила успели измениться. Требование: вердикт несёт `rulesSha` как поле, парсер читает его, не `HEAD`. Это ровно урок `evening-auditor-m2p-reproducibility`: фиксируешь версии, не только исходники.

[Верстальщик]: Поддержу и добавлю про наблюдаемость нарушения на холсте. Линтер направленности из реплики Математика должен падать **с адресом**: не «есть нарушение», а «`script/foo.mjs:12` импортирует `procedure/bar` в обход манифеста». Моя больная тема — `code-review-honesty`: вердикт без точного пути нечитаем и неотзываем. Формат violation: `{from, to, line, rule, expectedManifest}`. Тогда человек в PR видит стену, а не тревогу без адреса.

[Музыкант]: Ещё один флаг, чтобы зуб не оказался декоративным. Значит линтер направленности обязан иметь **тест на собственную слепоту**: заведомо нарушающий импорт в фикстуре должен ловиться, иначе зелёный CI лжёт. Проверка проверки — не паранойя, а мастеринг перед релизом.

[Структурщик]: Принимаю флаг Музыканта — это стандарт для любого нашего гейта. ## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Правило направленности | Граф импортов: ребро разрешено ⟺ `layer(a) > layer(b)` при `L={procedure:2, kit:1, script:0}` **и** цель экспортирована через манифест слоя. Правила: `no-upward-import` + `manifest-only-crossing` |
| Машинная проверка | Чистая функция `checkImports(graph) → violations[]`, детерминированная; зуб = exit-код CI (`violations.length > 0 ⇒ exit 1`); violation несёт `{from, to, line, rule, expectedManifest}` |
| Тест на слепоту гейта | Обязательна фикстура с заведомо нарушающим импортом — ловится линтером; иначе зелёный CI недоказуем (урок `team-execution-contour`) |
| Статус кит-манифеста в ревью | `index.ts` слоя китов — **BLOCK-условие LGTM**: PR с новой межслойной связью без обновлённого манифеста блокируется Teamlead |
| Адрес парсера ответственности | Read-only `attributeViolation(verdict) → {rulesSha, blameTrail}` — жилец слоя китов по адресу M1 (`packages/`/`scripts/lib/`), не сервис; `git blame` не дублируется, вызывается существующий |
| Маршрут расследования | `вердикт → SHA правил (зафиксирован в момент вердикта) → git blame → автор правки`; воспроизводим бит-в-бит; неотзываем при разрыве любого звена |
| Голос | **Да** — единогласно (Teamlead LGTM) |

**Definition of Done:**
- Линтер направленности реализован как чистая функция `checkImports(graph) → violations[]`; интегрирован в CI с exit-кодом; violation несёт `{from, to, line, rule, expectedManifest}`.
- Тест-фикстура с нарушающим импортом ловится линтером (тест на слепоту), покрыта в `yarn test:scripts`.
- Кит-манифест (`index.ts` слоя китов) добавлен в LGTM-чек-лист как BLOCK-условие; правило зафиксировано в `CONTRIBUTING.md` / регламенте M1.
- Парсер `attributeViolation(verdict)` живёт по адресу M1, read-only, возвращает `{verdict, rulesSha, blameTrail}`; `rulesSha` берётся из вердикта, не из `HEAD`.
- Маршрут расследования (`вердикт → rulesSha → git blame`) документирован и проигрываем на одном реальном прецеденте; отсутствие любого звена ⇒ вердикт не отзываем.
- Без новых пакетов; адреса и термины — на ратифицированных M1/M2; дисциплина импортов согласована с `ARCHITECTURE.md` §1.
