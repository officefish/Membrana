# Консилиум: конституция границы слоёв «процедуры → киты → скрипты»

**Повестка:** E3 — чем обеспечивается направленность «процедуры → киты → скрипты», чем даётся зуб на нарушение, и где живёт парсер ответственности как жилец (адрес, не механизм). Ответ обязан назвать: (1) правило направленности и его машинную проверку; (2) статус кит-манифеста в ревью; (3) адрес парсера ответственности с маршрутом расследования прецедента (вердикт → SHA правил → история правок).

[Teamlead]: Рамку фиксирую сразу, чтобы M3 не расползся в тулинг-стройку. M1 дал дом («процедуры → киты → скрипты» — три этажа), M2 дал словарь. Наш вопрос ровно один: **конституция границы**, три обязательных пункта из повестки. По принципу «форма следует функции»: направленность — это не пожелание, а несущая стена. Если её нельзя проверить машиной, её нет. Начинаем с правила направленности, затем зуб, затем адрес парсера. Никаких новых сущностей сверх трёх пунктов.

[Верстальщик]: С точки зрения потребителя вердикта — а это агент утром и человек в PR — направленность обязана быть **видимой на холсте**, а не только истинной в коде. Моя норма из `code-review-honesty`: если правило есть, но нарушение не показано читателю с адресом, это мёртвая дверь. Прошу заранее: вывод машинной проверки — не «нарушений: 0» с exit 0 (мы на этом обожглись 19.07, когда маска `-topic.md` молча не нашла повестку), а явный список рёбер с направлением и вердиктом по каждому.

[Математик]: Формализую направленность, иначе «→» останется вкусовым. Три этажа — это множество узлов `V = {процедуры, киты, скрипты}` и множество импортов `E`. Правило направленности — предикат: `∀ (a→b) ∈ E: level(a) ≥ level(b)`, где `level(процедуры)=2, level(киты)=1, скрипты=0`. Разрешён импорт вниз и вбок, запрещён вверх. Это в точности проверка ацикличности + монотонности по уровню — линейная по числу рёбер, детерминированная, без сети. Машинная проверка = функция `checkDirection(importGraph) → {violations[]}`, чистая.

[Музыкант]: По касательной, аудио тут не задет, но маршрут сигнала я узнаю — это ровно source → эффекты → выход, ток течёт в одну сторону. Мой флаг из `code-review-honesty-refactor` 16.07: правило без голоса на входе получает зубы только на выходе и молчит на входе — как гейт 19.07, что судил запись, но не сказал комнате формат. Значит правило направленности должно **звучать дважды**: детектор рёбер на CI (выход) и строка в кит-манифесте, которую видит ревьюер (вход).

[Структурщик]: Начну с термина, иначе DAG поплывёт. «Направленность» здесь — это лемма о **графе разрешённых импортов между слоями**, и она уже частично живёт в `ARCHITECTURE.md`: foundation → core, analyzer → foundation, детекторы без импортов между собой. Наш M3 не изобретает механизм — он поднимает существующую норму слабой связанности на слой процедуры/киты/скрипты. Проверка обязана лежать там же, где живут остальные import-guard'ы: пакет границы (`scripts-boundary-container` в реестре). Один термин — один смысл: «кит» = единица словаря M2, манифест китов = реестр лемм слоя.

[Teamlead]: Принимаю формализацию Математика как правило направленности и уточнение Структурщика про адрес. Фиксирую пункт 1: **правило** — `level(a) ≥ level(b)` для любого импорта, проверка живёт в пакете границы (`scripts-boundary-container`), выход — список рёбер-нарушителей, не голое «0». Теперь пункт 2 — **зуб**. Зуб = CI-гейт, который роняет сборку на нарушении с ненулевым exit и печатает нарушившее ребро. Без exit≠0 правило декоративно — 19.07 нас этому научило прямо.

[Верстальщик]: Поддержу зуб, но добавлю про кит-манифест — это пункт 2 повестки, «статус манифеста в ревью». Манифест должен быть **обязательным входом ревью**, а не приложением. По моей норме наблюдаемости: если в PR трогают слой (процедуру/кита/скрипт), diff манифеста обязан быть в поле зрения ревьюера, и его отсутствие — сам по себе повод для BLOCK. Иначе манифест устаревает молча, как `githubIssueClosedAt` — ноль из восемнадцати заполнений.

[Математик]: Уточню зуб численно. Гейт = функция `gate(graph) → exitCode`, где `exitCode = violations.length > 0 ? 1 : 0`. Но добавлю условие, которого 19.07 не хватило: **гейт обязан провалиться, если сам не выполнился**. То есть если множество проверенных рёбер пусто при непустом графе (маска не нашла файлы) — это не «0 нарушений», это `exitCode = 2` (проверка не состоялась). Различие «чисто» и «не проверено» — обязательная часть контракта, иначе `-topic.md` повторится.

[Музыкант]: Вот это ключевое, поддержу живой стороной. У меня был `ritual-inputs-echo`: три отражения одного снимка выдал за `n=3`. Пустой прогон, выданный за чистоту — та же болезнь. Гейт с тремя исходами (`0=чисто, 1=нарушение, 2=не проверено`) — это как индикатор, который отдельно показывает «тишина» и «сигнал в норме»: их нельзя путать. По кит-манифесту согласен с Верстальщиком: строка в манифесте — это партитура, ревьюер сверяет исполнение с ней.

[Структурщик]: Соглашусь с трёхзначным exit — это честный контракт функции. Теперь пункт 3, **адрес парсера ответственности**. Здесь важно не построить механизм (это не наш M3), а назвать **жильца**: где физически живёт разбор «кто ответственен за нарушение». По границе пакетов — это `scripts-boundary-container`, подмодуль ответственности; не в `apps/*`, не в детекторах. Маршрут расследования из повестки: `вердикт → SHA правил → история правок` — это read-only трасса по git, не рантайм-сервис. Парсер = чистая функция `resolve(verdict) → {rulesSha, editHistory[]}`.

[Teamlead]: Хорошо, но зафиксирую границу M3 жёстко: мы называем **адрес и контракт** парсера, но не пишем его — это отдельная задача. Принимаю: жилец — `scripts-boundary-container`, парсер ответственности как read-only чистая функция `resolve(verdict) → {rulesSha, editHistory[]}`, маршрут `вердикт → SHA правил (из коммита, где правило действовало) → git log правил`. Это отвечает на «где живёт жилец, не механизм». Три пункта повестки закрыты вчерне. Возражения по существу?

[Математик]: Одно уточнение к трассе, иначе SHA будет вкусовым. «SHA правил» обязан быть **SHA на момент вынесения вердикта**, а не HEAD — иначе воспроизводимость рушится: правило могли поправить после. Значит вердикт обязан нести в себе `rulesSha` как поле (детерминированный вход), а парсер лишь разворачивает его в историю. Это ровно то, что мы решили в `evening-auditor-m2p-reproducibility`: фиксируешь не только исходники, но и версию правил. Иначе `resolve` недетерминирован.

[Верстальщик]: Поддержу и добавлю наблюдаемость трассы. Читатель расследования — человек в PR утром: он обязан увидеть **на холсте** три звена цепи явно: какой вердикт, какой SHA правил был активен, какие правки правил после этого. Без явного `rulesSha` в шапке вердикта расследование — мёртвая дверь: красивая кнопка «показать историю», ведущая в HEAD, который уже не тот. Прошу: `rulesSha` — обязательное поле артефакта вердикта, видимое, не выводимое.

[Музыкант]: Согласен, это как версия плагина в сессии мастера — без неё рендер через год звучит иначе. Один флаг из опыта: трасса должна работать **офлайн по git**, без обращения к внешнему API (GitHub может лечь). `git log` по файлу правил — локальный, детерминированный. Если парсер полезет в сеть за историей — это уже другой зверь, и латентность/отказ появятся там, где их не ждут. Read-only по локальному git — вот честный контур.

[Структурщик]: Закрою термин, чтобы манифест и парсер не слиплись. **Кит-манифест** (пункт 2) и **парсер ответственности** (пункт 3) — две разные леммы в одном пакете: манифест — это *декларация* лемм слоя (что есть кит, к какому этажу принадлежит), парсер — *расследование* по вердикту. Манифест — вход ревью и вход детектора направленности; парсер — read-only функция постфактум. Они не зависят друг от друга напрямую, только через общий словарь M2. Прямого импорта манифест↔парсер быть не должно — иначе нарушим ту самую слабую связанность, которую конституируем.

[Teamlead]: Принимаю разведение манифеста и парсера как две леммы одного пакета без взаимного импорта — это внутренне согласовано с тем, что мы конституируем. Свожу три пункта. Кто-то видит нарушение границы M3 (заезд в реализацию вместо адреса/контракта)?

## Итоговое решение консилиума

| Вопрос повестки | Решение |
|-----------------|---------|
| **1. Правило направленности + машинная проверка** | Правило: `∀ (a→b) ∈ E: level(a) ≥ level(b)`, где `level(процедуры)=2, киты=1, скрипты=0` — импорт вниз/вбок разрешён, вверх запрещён (расширение нормы слабой связанности `ARCHITECTURE.md` на слой процедуры/киты/скрипты). Проверка: чистая функция `checkDirection(importGraph) → {violations[]}`, живёт в пакете `scripts-boundary-container`. Вывод — явный список рёбер-нарушителей с направлением, **не** голое «0». |
| **2. Статус кит-манифеста в ревью** | Манифест — **обязательный вход ревью**, не приложение: при изменении любого слоя diff манифеста обязан быть в поле зрения ревьюера; его отсутствие/устаревание — повод для **BLOCK**. Манифест = декларация лемм слоя (что есть кит, к какому этажу принадлежит), общий словарь с M2. Одновременно — вход детектора направленности. |
| **3. Адрес парсера ответственности + маршрут** | Жилец: пакет `scripts-boundary-container`, подмодуль ответственности (не `apps/*`, не детекторы). Парсер = read-only чистая функция `resolve(verdict) → {rulesSha, editHistory[]}`. Маршрут: **вердикт → `rulesSha` (обязательное поле артефакта, зафиксированное на момент вынесения, не HEAD) → `git log` по файлу правил** (локально, офлайн, без внешнего API). Манифест и парсер — две леммы одного пакета **без взаимного импорта**. |
| **Зуб на нарушение** | CI-гейт с **трёхзначным exit**: `0 = чисто`, `1 = нарушение` (печать ребра-нарушителя, сборка падает), `2 = проверка не состоялась` (пустое множество проверенных рёбер при непустом графе — защита от повтора инцидента `-topic.md` 19.07). |
| **Граница M3** | Назван **адрес и контракт**, реализация — отдельная задача. Ни строки кода в этом заседании. |

**Definition of Done (для задачи-наследника, вне M3):**
- В `scripts-boundary-container` реализована `checkDirection(importGraph) → {violations[]}` — чистая, покрыта unit-тестами (кейсы: импорт вниз — ok, вбок — ok, вверх — violation, пустой граф при наличии файлов — exit 2).
- CI-гейт с трёхзначным exit подключён; тест на исход `2` (маска не нашла файлы) обязателен.
- Кит-манифест заведён как файл-декларация лемм слоя; ревью-чеклист требует его diff при изменении слоя.
- `resolve(verdict) → {rulesSha, editHistory[]}` реализована как read-only по локальному git; артефакт вердикта несёт обязательное поле `rulesSha`, видимое в шапке.
- Нет прямого импорта манифест↔парсер; нет импортов «вверх» по уровням.
