# Консилиум: синхрон базовых веток долгоживущих деревьев с `main` (W2, K1)

**Повестка:** W2 — как базовые ветки долгоживущих worktree (`main`, `tooling`, `product`, `codex`, `cursor`) синхронизируются с `main`: частота, событие-триггер, исполнитель (агент/скрипт/хук), проверяемый предикат «протухла», режим вмешательства (сигнал vs авто-действие). Область синхрона задана K2 (M1): только легитимные деревья (`canon` / `sprint-open`), не хвосты под снос.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант (Kuryokhin), Верстальщик (Rodchenko).

**Порядок реплик:** Структурщик → Математик → Teamlead → Музыкант → Верстальщик (циклически)

[Структурщик]: Начну с термина, иначе W2 поплывёт. «Синхрон» — омоним двух лемм: (1) **fast-forward базы** (`git fetch` + `git merge --ff-only origin/main` в дерево, где база = ветка `tooling`/`product`/etc), и (2) **rebase рабочих коммитов** дерева поверх свежего `main`. Это радикально разные операции: первая безопасна и не трогает историю, вторая переписывает и может конфликтовать. K1 обязан говорить только про первую — синхрон *базы*, а не рабочего слоя. Иначе мы залезаем в lifecycle, который закрыл K2.

[Математик]: Формализую предикат «протухла», иначе слово останется вкусовым. База дерева `B` протухла относительно `main`, если `git rev-list --count B..origin/main > 0` — то есть в `origin/main` есть коммиты, недостижимые из `B`. Это чистый предикат: вход — две ссылки, выход — целое `drift = |B..origin/main|`. Порог тривиален: `drift > 0` уже протухание, но осмысленный сигнал — `drift ≥ threshold`. Предлагаю два уровня: `drift > 0` → info, `drift ≥ 20` коммитов **или** возраст базы `> 72h` → warning. Оба измеримы `rev-list` и `git log -1 --format=%ct`.

[Teamlead]: Рамку фиксирую сразу, чтобы K1 не съел K2. Наш вопрос ровно один: **как базы синхронизируются с `main`** — четыре среза (частота, событие, руки, режим). Мы не проектируем rebase рабочих веток и не трогаем снос хвостов — это закрыто M1. По «форма следует функции»: функция синхрона базы — чтобы дерево стартовало новую работу от свежего фундамента, а не от вчерашнего. Значит событие-триггер обязано быть привязано к **началу работы**, а не к часам на стене. Поддерживаю разделение Ozhegov: K1 = ff-only базы, rebase рабочего слоя — вне зоны.

[Музыкант]: По касательной, аудио тут не задет, но ритм я узнаю — это как синхронизация метронома перед записью дубля. Если база отстала, дубль пишется под старый клок, а на сведении разъезжается. Мой флаг из `dads-integration-m1-preset-delivery`: два носителя одного пресета молча расходятся, если синхрон ручной — `P(equal)` падает со временем. Тут ровно то же: ручной ff-only базы никто не сделает вовремя. Значит руки должны быть **не человеческие** — скрипт или хук, привязанный к событию.

[Верстальщик]: С точки зрения потребителя — а потребитель здесь агент, стартующий работу в дереве, и человек в PR — синхрон обязан быть **видимым**. Моя норма из `main-day-accuracy`: если база протухла, агент должен увидеть это до первого коммита, а не узнать на PR-конфликте. Значит предикат Дынина рисуется в баннер старта: `drift=N коммитов, возраст Nh`. Мёртвая дверь — молчаливо стартовать от старой базы и получить конфликт при мердже; зелёное там, где нужен разбор.

[Структурщик]: Соглашусь с Teamlead про триггер, но уточню носитель. У нас два кандидата на «руки»: (1) **git hook** (`post-checkout`/`post-merge`) — живёт в дереве, но хрупок и не переносится между машинами без установки; (2) **скрипт ритуала** (`scripts/*.mjs`, вызывается из `ritual:day` или `worktree:sync`) — переносим, тестируем, единый носитель. По норме «один термин — один смысл»: хук размажет логику по хостам, скрипт держит её в одном месте. Голосую за скрипт как единственный носитель, хук — максимум тонкая обёртка, вызывающая тот же скрипт.

[Математик]: Поддержу скрипт численно. Хук `post-checkout` срабатывает недетерминированно — зависит от того, как агент вошёл в дерево (checkout vs уже там сидел). Скрипт, вызванный явно в начале работы, даёт `P(запуск) = 1` при условии, что старт проходит через ритуал. И важное: **ff-only не может создать конфликт** по построению — если `B..origin/main` линеен относительно `B`, merge либо проходит бит-в-бит, либо отклоняется с непустым `origin/main..B` (у базы есть свои коммиты — тогда это уже не чистая база, а рабочее дерево, и мы её НЕ трогаем). Предикат чистоты базы: `git rev-list --count origin/main..B == 0`.

[Teamlead]: Принимаю чистоту базы как предусловие. Это важная граница: синхроним ff-only **только** ту базу, у которой нет собственных коммитов впереди `main` (`origin/main..B == 0`). Если у базы есть свои коммиты — это либо не база, либо кто-то работал не в том слое, и синхрон обязан **отказаться и сигналить**, а не форсить. Так конструкция сама защищает от потери работы: авто-действие разрешено только там, где оно математически безопасно (ff-only чистой базы), во всех остальных случаях — сигнал человеку.

[Музыкант]: Тогда режим вмешательства раздваивается честно, и мне это нравится — как в цепочке эффектов: авто-нормализация только когда клиппинга нет, иначе руку на фейдер. Чистая база, `drift>0` → **авто ff-only**, тихо, лог в отчёт старта. База с собственными коммитами или конфликт → **стоп + сигнал**, никакого авто. Это ровно моя норма «риск осознанный»: авто там, где артефакт невозможен, ручка там, где он вероятен.

[Верстальщик]: Тогда баннер старта имеет три состояния, и это ложится в DESIGN.md без нового токена. `drift=0` → `badge-ghost` «база свежая», тихо. Авто-ff прошёл → `badge-info` «база подтянута: +N коммитов». База грязная/конфликт → `badge-warning` «синхрон остановлен: у базы N своих коммитов, нужен разбор», `aria-live="polite"`, ручное закрытие как у alert вытеснения. Не только цвет — иконка плюс текст. Мёртвой двери нет: каждое состояние озвучено.

[Структурщик]: Зафиксирую границу пакета, чтобы не расползлось. Логика синхрона — чистая функция в `scripts/`-контейнере (эпик `scripts-boundary-container`): вход `{tree, base, originMain}`, выход `{drift, baseClean, action: 'ff'|'skip'|'halt', evidence}`. Никаких сетевых вызовов внутри чистого ядра — `git fetch` делает тонкая обёртка до вызова, ядро получает уже готовые ref-состояния. Это отделяет «посчитать, что делать» от «сделать», и делает предикат тестируемым без реального репозитория.

[Математик]: Поддержу разделение fetch/decide/act — это ровно `audit(x)=audit(x)` детерминированность. Чистое ядро `decideSync(refs) → action` бит-в-бит воспроизводимо; побочка (`fetch`, `merge --ff-only`) — в тонкой оболочке. По частоте: не нужен таймер. Событие — **начало работы в дереве** (старт ритуала/старт задачи). `P(протухания)` растёт монотонно только между стартами, а синхрон перед стартом обнуляет drift ровно тогда, когда он вреден. Таймерный синхрон фоном — лишняя сущность, режем по «декора без смысла не существует».

[Teamlead]: Собираю четыре среза в вердикт. **Частота:** по событию, не по расписанию. **Событие:** старт работы в дереве (вход в `ritual:day` / старт задачи через тулинг). **Руки:** скрипт `scripts/` (чистое ядро + тонкая оболочка fetch/act), хук — опциональная обёртка, не источник логики. **Режим:** авто-ff-only для чистой базы с `drift>0`; стоп+сигнал для грязной базы или конфликта. Предикаты Дынина — в основу. Это не расширяет K2: рабочий слой не трогаем.

[Музыкант]: Одна поправка по ритму, из живого опыта. Пять деревьев в списке (`main`, `tooling`, `product`, `codex`, `cursor`), но синхроним **только то дерево, куда агент заходит работать** — не пачкой все пять при каждом старте. Иначе это как ремастерить весь альбом ради одного трека: лишний I/O и сетевые вызовы. Синхрон адресный: одно дерево = одно событие старта в нём.

[Верстальщик]: Поддержу адресность и добавлю про `main` как особый случай. Дерево `main` — само по себе `origin/main`, у него drift к самому себе всегда 0. Значит в баннере для `main`-дерева синхрон-строка либо скрыта, либо `badge-ghost` «база = main». Не показывать пустой прогресс там, где операции нет — это моя норма против мёртвой двери: индикатор без события не рисуем.

[Структурщик]: Согласен, `main`-дерево — вырожденный случай, `B == origin/main`, ядро вернёт `action:'skip'` по `drift=0`. Последнее уточнение по термину для реестра: назову функцию `resolveBaseSyncAction` (не `syncManager` — без предмета), артефакт-эвиденс — `BaseSyncReport {tree, base, drift, baseClean, action, ts}`. Ложится в тот же контейнер, что и остальной ритуал-тулинг, без прямых импортов между скриптами.

[Математик]: И зафиксирую пороги как значения по умолчанию, не как догму: `drift>0` → кандидат на ff; `baseClean=false` → halt независимо от drift; warning-порог `drift≥20 ∨ age>72h` — только для окраски сигнала, на решение `ff/halt` не влияет. Пороги окраски — параметр конфига, предикат безопасности (`baseClean`) — жёсткий инвариант, не настраивается. Это разделяет косметику от несущей конструкции.

[Teamlead]: LGTM на всё собранное. Один инвариант поверх: синхрон **никогда** не переписывает историю и не форсит — только ff-only или halt. Это конституционально для K1, как и в M1 было для сноса. Даю форму задачи Структурщику (ядро + оболочка + реестр-функция), Дынину — предикаты и тесты чистого ядра, Верстальщику — баннер старта по DESIGN.md, Музыканту — проверку на реальных деревьях. Порядок: ядро → оболочка → баннер → прогон. Вопрос W2 закрыт.

## Итоговое решение консилиума

| Срез вопроса W2 | Решение |
|-----------------|---------|
| **Частота** | По событию, не по расписанию; таймерный фоновый синхрон отклонён («декор без смысла») |
| **Событие-триггер** | Старт работы в дереве (вход `ritual:day` / старт задачи через тулинг); синхрон **адресный** — только то дерево, куда заходит агент, не все пять |
| **Руки (исполнитель)** | Скрипт в `scripts/`-контейнере: чистое ядро `resolveBaseSyncAction(refs)` + тонкая оболочка (`fetch` до, `merge --ff-only`/halt после); git-хук — опциональная обёртка над тем же скриптом, не источник логики |
| **Предикат «протухла»** | `drift = git rev-list --count B..origin/main`; `drift>0` → протухание |
| **Предикат безопасности (жёсткий)** | База чистая ⟺ `git rev-list --count origin/main..B == 0`; `baseClean=false` → **halt**, синхрон запрещён |
| **Режим вмешательства** | Чистая база + `drift>0` → **авто ff-only**, тихо, лог в отчёт старта. Грязная база / конфликт → **стоп + сигнал**, ручной разбор |
| **Пороги окраски (мягкие, конфиг)** | `drift>0` → info; `drift≥20 ∨ age>72h` → warning; на решение `ff/halt` не влияют |
| **Вырожденный случай** | `main`-дерево: `B==origin/main`, `drift=0`, `action:'skip'`; синхрон-строку не рисуем |
| **Граница с K2** | Синхрон трогает **только базу** легитимного дерева (ff-only); рабочий слой и снос хвостов — вне зоны, закрыты M1 |
| **Артефакт-эвиденс** | `BaseSyncReport {tree, base, drift, baseClean, action, ts}` |
| **Инвариант** | Синхрон никогда не переписывает историю и не форсит — только ff-only или halt |

**Definition of Done:**
- Чистая функция `resolveBaseSyncAction(refs) → {drift, baseClean, action: 'ff'|'skip'|'halt', evidence}` в `scripts/`-контейнере; юнит-тесты без реального git (вход — фиктивные ref-состояния), бит-в-бит детерминизм.
- Тонкая оболочка: `git fetch` до вызова ядра, `git merge --ff-only` при `action:'ff'`, отказ без побочек при `action:'halt'`.
- Интеграция в старт работы: синхрон адресный (только целевое дерево), вызывается перед первым коммитом задачи.
- Баннер старта по `DESIGN.md`: три состояния (`badge-ghost` свежая / `badge-info` подтянута +N / `badge-warning` остановлен, ручное закрытие, `aria-live="polite"`), иконка + текст, не только цвет; для `main`-дерева строка скрыта.
- Инвариант ff-only/halt покрыт тестом: грязная база (`origin/main..B > 0`) → `halt`, никогда не форс.
- Реестр: `leadPersona: ozhegov`, support `dynin`/`rodchenko`/`kuryokhin`; артефакт `BaseSyncReport` задокументирован.
- Границы K2 не нарушены: рабочий слой и снос хвостов не затронуты.
