**Повестка:** E1 — где физически живёт процедурный слой: адрес контейнеров процедур, контракт минимального контейнера, соотношение с существующими носителями, первый жилец и достаточный для него минимум.

[Музыкант]: Начну честно с зоны: аудио тут задет по касательной, но ритм графа я слышу. E1 — это source в маршруте: пока не назван адрес контейнера, все следующие узлы (словарь M2, граница M3) звучат в пустоту. Мой флаг из `dads-integration-m1-preset-delivery`: два носителя одной сущности молча расходятся. Значит с самого начала — **один канонический адрес процедуры, один носитель**, а не «где-то в docs и где-то в packages».

[Математик]: Формализую, иначе «где живёт» останется вкусовым. Процедурный слой — это множество контейнеров `P = {p}`, где каждый `p` — чистая спецификация: вход (категории), тело (шаги над категориями), выход. Ключевой предикат фундамента: `addressable(p)` — у контейнера есть единственный стабильный путь и машиночитаемая схема. Без схемы контейнер нельзя ни валидировать, ни воспроизвести. Требую: минимальный контейнер = **файл по фиксированной схеме** (id, категории-вход, шаги, категории-выход), детерминированно парсимый.

[Верстальщик]: С точки зрения потребителя — а потребитель здесь агент, читающий процедуру как топливо, и человек в PR — адрес обязан быть **видимым и однотипным**. Моя норма из `main-day-accuracy`: если контейнер живёт в двух местах, читатель не отличит канон от копии — мёртвая дверь. Прошу: один каталог, одно расширение, предсказуемое имя файла = id. UI-работы прямой нет, но когда процедуры понадобится показать витриной, схема с явными полями сэкономит целый слой адаптеров.

[Teamlead]: Рамку фиксирую сразу, чтобы E1 не расползся в проектирование словаря (это M2) и границы (это M3). Наш вопрос ровно три пункта: **адрес, контракт минимального контейнера, первый жилец**. По «форма следует функции»: процедурный слой — это спецификации, а не исполняемый код и не React. Значит его дом — **не** `packages/services/*` (там сервисы с логикой), **не** `apps/*` (там приложения). Это документо-подобные артефакты с машиночитаемой схемой. Предлагаю `docs/procedures/` как корень слоя.

[Структурщик]: Начну с термина, иначе DAG поплывёт. «Процедурный слой» — омоним двух лемм: (1) **спецификация процедуры** (декларативный контейнер, что делаем над категориями) и (2) **исполнитель процедуры** (код, который её проигрывает). E1 решает только лемму (1) — носитель спецификации. Исполнитель — отдельный вопрос, в `packages/` позже. Смешаем — получим `packages/services/procedures` с логикой внутри и потеряем чистоту. Поддерживаю Teamlead: спецификация живёт в `docs/`, но с **строгой JSON/YAML-схемой**, а не в свободном Markdown.

[Музыкант]: Соглашусь со Структурщиком про две леммы — это ровно как партитура против исполнителя: ноты лежат отдельно от музыканта, который их играет. Но добавлю живой флаг: если контейнер — чистый JSON без единой человекочитаемой строки, ревьюер-человек ослепнет. Предлагаю гибрид: машиночитаемое ядро (schema-valid) + короткое `title`/`intent` для глаза. Как в `insight-own-field-corpus` мы закладывали и данные, и подпись для человека.

[Математик]: Принимаю гибрид, но с оговоркой о детерминизме. `title`/`intent` — поля-декор, они **не влияют** на `run(p)`; валидатор обязан игнорировать их при проверке эквивалентности процедур. Иначе получим то, на чём обжигались в `evening-auditor-m2p`: два «разных» контейнера, различающихся только комментарием. Схема: обязательные поля `{id, inputs[], steps[], outputs[]}` — несущие; `{title, intent}` — опциональные, вне хеша эквивалентности.

[Верстальщик]: Поддержу разделение несущее/декор — это моя же норма «форма = сообщение, декор без роли убираем». Но настою: `title` пусть будет **обязательным для человека**, даже если не входит в хеш. Пустой заголовок в списке процедур — это стена из id, нечитаемая витрина. Компромисс: `title` обязателен синтаксически, но не участвует в семантической эквивалентности. Так и глаз сыт, и хеш чист.

[Teamlead]: Принимаю. Свожу форму: `docs/procedures/` — корень; один файл = один контейнер; имя файла = `<id>.procedure.yaml` (YAML читаемее JSON для человека-ревьюера, при этом машиночитаем). Схема в `docs/procedures/schema/procedure.schema.json`. Обязательные несущие поля — по Дынину; `title` обязателен синтаксически, вне хеша — по Родченко. Это не нарушает `ARCHITECTURE.md`: никакого нового пакета, никаких зависимостей в графе.

[Структурщик]: Согласен с адресом и именем — `<id>.procedure.yaml` даёт «имя файла = id» без синонимов-слоёв. Зафиксирую границу словаря: `docs/procedures/` экспортирует контейнеры только как файлы по схеме; **никакой** `.mjs`-исполнитель туда не кладём — он появится позже как отдельная лемма (валидатор/раннер) в `scripts/` или сервисе. Валидатор схемы — да, он read-only и живёт в `scripts/procedures/validate.mjs`, зона Математика как ops.

[Математик]: Беру валидатор. `validate(dir) → {valid[], invalid[], errors[]}` — чистая функция над множеством файлов: парс YAML → проверка по JSON Schema → проверка уникальности `id` и совпадения `id == filename`. Детерминированно, без сети, exit-код по инвариантам. Это тот же профиль, что `evening-auditor`: один вход → один выход, бит-в-бит. Готов дать схему и валидатор в одном PR.

[Музыкант]: По первому жильцу — вот где мой контур реально задет. Предлагаю в жильцы не абстракцию, а живую процедуру из нашего дня: **вечерний ритуал** или **консилиум**, у которых уже есть шаги над категориями (роли, артефакты, вердикт). Но самый честный минимальный жилец — что-то маленькое и полностью наблюдаемое. Как первый трек на новом пульте берут простой синус, а не оркестр.

[Верстальщик]: Поддержу «простой синус» как первого жильца. С точки зрения потребителя: первый контейнер должен быть **читаем целиком за один экран**, чтобы проверить схему глазами. Если первым сунуть консилиум с пятью ролями и ветвлением — не отличим ошибку схемы от сложности процедуры. Первый жилец = минимальная процедура с 1-2 шагами, где `inputs`/`steps`/`outputs` видны без прокрутки.

[Teamlead]: Решаю по первому жильцу: не консилиум (он L, ветвление, риск колонизации схемы сложностью), а **минимальная линейная процедура** — кандидат `yarn ask` (одна роль → один вопрос → один ответ-артефакт): вход = {персона, вопрос}, шаги = {выбор персоны, формирование промпта, запись ответа}, выход = {файл в docs/discussions}. Она реальна, линейна, наблюдаема за один экран. Достаточно для неё: пройти валидатор + ревью человека в PR.

[Структурщик]: Принимаю `yarn ask` как первого жильца — это существующий процесс, не выдуманный, термин однозначен. Уточню контракт достаточности: жилец «заселён», когда (1) файл `ask.procedure.yaml` проходит `validate.mjs` зелёным, (2) `id == filename`, (3) все категории в `inputs`/`outputs` — пока просто строки-имена (словарь категорий M2 их ратифицирует позже, здесь мы их только называем, не определяем). Границу с M2 не переходим.

[Математик]: Согласен с границей M2. Зафиксирую численно достаточность: категории на этапе E1 — **непрозрачные строки** (opaque labels), валидатор проверяет только их непустоту и синтаксис, но `не` проверяет принадлежность словарю — словаря ещё нет. Это корректный частичный порядок: E1 (носитель) → M2 (словарь) → M3 (граница). Валидатор E1 не должен падать на неизвестной категории, иначе мы протащим M2 в M1.

[Музыкант]: Мне нравится, что граница чистая. Последний флаг из опыта: заложите в схему поле `version` контейнера с самого начала. Мастер без версии — источник тихих расхождений, я на этом горел в `preset-delivery`. Одно поле сейчас дешевле миграции потом.

[Верстальщик]: Поддержу `version` — и как потребитель добавлю: пусть в схеме будет ровно один обязательный `schemaVersion`, чтобы витрина и валидатор знали, по каким правилам читать. Больше полей «на будущее» не тащим — это уже украшательство, которого я сам избегаю.

[Teamlead]: Принимаю `schemaVersion` как обязательное, прочие «на вырост» — нет (YAGNI, конструкция без декора). Свожу к вердикту. Все несогласия сняты: адрес — `docs/procedures/`, контракт — YAML по схеме, первый жилец — `ask.procedure.yaml`. LGTM форме. Передаю в DoD.

[Структурщик]: Принимаю целиком. Границы пакетов не тронуты, синонимов-слоёв нет, исполнитель отложен явной леммой. Схема — единственная точка правды формата. Готов ревьюить PR на слабую связанность: валидатор не должен импортировать ничего из `apps/` или `packages/services/`.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Где физически живёт процедурный слой | Корень слоя — **`docs/procedures/`**; спецификации, а не исполняемый код. Не `packages/services/*`, не `apps/*` |
| Контракт минимального контейнера | Один файл = один контейнер, имя `<id>.procedure.yaml` (YAML). Несущие поля: `{schemaVersion, id, inputs[], steps[], outputs[]}`; декор вне хеша эквивалентности: `{intent}`; `title` обязателен синтаксически, но не влияет на семантику |
| Схема формата | `docs/procedures/schema/procedure.schema.json` (JSON Schema) — единственная точка правды формата |
| Категории на этапе E1 | Непрозрачные строки (opaque labels): валидатор проверяет непустоту/синтаксис, **не** проверяет принадлежность словарю (словарь — M2) |
| Соотношение с носителями | Новый каталог в `docs/`; **никакого нового пакета**, граф зависимостей `ARCHITECTURE.md` не тронут; исполнитель/раннер — отдельная лемма позже (`packages/` или `scripts/`) |
| Валидатор | `scripts/procedures/validate.mjs` — чистая read-only функция (парс → JSON Schema → уникальность id → `id == filename`), детерминированная, без сети, exit-код по инвариантам. Зона Математика (ops) |
| Первый жилец | **`ask.procedure.yaml`** (`yarn ask`): линейная, наблюдаемая за один экран. Не консилиум (L, ветвление) |
| Достаточность заселения | (1) проходит `validate.mjs` зелёным; (2) `id == filename`; (3) категории названы как строки без ратификации словарём; (4) ревью человека в PR |

**Definition of Done:**

- [ ] Создан `docs/procedures/ask.procedure.yaml` — первый жилец, линейный, читаем за один экран, `id: ask`.

- [ ] `yarn` -скрипт `procedures:validate` в корневом `package.json`.
- [ ] `validate.mjs` зелёный на `ask.procedure.yaml`; категории обрабатываются как opaque strings (не падает на неизвестной категории).
