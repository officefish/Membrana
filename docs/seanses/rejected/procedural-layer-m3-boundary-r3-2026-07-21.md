# Консилиум: конституция границы слоёв «процедуры → киты → скрипты» (E3)

**Повестка:** E3 — назвать правило направленности слоёв и его машинную проверку; статус кит-манифеста в ревью; адрес парсера ответственности как жильца (не механизма) с маршрутом расследования прецедента (вердикт → SHA правил → история правок). Ратифицировано: M3 зависит от M1 (дом) и M2 (словарь).

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Верстальщик → Музыкант → Математик → Структурщик → Teamlead (циклически).

[Верстальщик]: С точки зрения потребителя вердикта — а потребитель это агент утром и человек в PR — граница слоёв нечитаема, если её не видно на холсте. Моя больная тема из `code-review-honesty`: направленность «процедуры → киты → скрипты» обязана быть **видимым артефактом**, а не устной договорённостью. Прошу зафиксировать: любой вердикт о нарушении границы показывает конкретное ребро (кто на кого сослался в запрещённую сторону) и SHA правил, по которым судили. Иначе на экране зелёная галочка там, где дверь мёртвая.

[Музыкант]: По касательной, аудио тут не задет напрямую, но маршрут сигнала я узнаю — это ровно цепочка source → эффекты → выход, только для процедуры. «Процедуры → киты → скрипты» — это однонаправленный тракт: скрипт не имеет права дёргать процедуру назад, как выходной каскад не переписывает вход. Мой флаг из `team-execution-contour` 19.07: когда правка тулинга посреди заседания унеслась чужим коммитом — это ровно нарушение направленности, только в git. Значит зуб нужен именно на обратное ребро.

[Математик]: Формализую, иначе «направленность» останется вкусовой. Пусть слои — упорядоченное множество `L = {процедура=0, кит=1, скрипт=2}`, а зависимость — ребро `a → b`. Правило направленности: `∀ ребро (a → b): layer(a) ≤ layer(b)` — импорт разрешён только на свой уровень или ниже по потоку. Нарушение — предикат `∃ (a → b): layer(a) > layer(b)`, вычислимый на графе импортов без запуска кода. Это чистая функция `checkDirection(graph) → violations[]`, детерминированная, покрывается юнит-тестом на синтетическом графе.

[Структурщик]: Начну с термина, иначе DAG поплывёт. «Слой» здесь — это **лемма о границе пакета**, а не о папке. Процедура = регламент (`docs/prompts/*`, канон), кит = манифест публичного контракта скрипт-контейнера, скрипт = `scripts/*.mjs` внутри контейнера. Машинная проверка направленности — это `dependency-cruiser`-правило (или наш `import-boundary`-линтер) на графе: ребро из `scripts/` в сторону, объявляющую процедуру исполняемой зависимостью, — `error`. Экспорт только через объявленный кит-манифест; обход манифеста = `нарушена слабая связанность` с точным путём.

[Teamlead]: Рамку фиксирую сразу, чтобы M3 не расползся в реализацию линтера. Мы на узле L3 ратифицированного DAG: дом закрыт M1, словарь — M2, наш вопрос ровно один — **конституция границы**: правило + зуб + адрес жильца. По «форма следует функции»: направленность оправдана тем, что процедура — это норма (медленно меняется, ревьюится людьми), скрипт — исполнитель (меняется часто). Норма не смеет зависеть от исполнителя — иначе смена скрипта тихо переписывает конституцию. Принимаю формулу Математика как правило. Дальше — про кит и парсер.

[Верстальщик]: Тогда про статус кит-манифеста в ревью. Кит — это **словарная витрина слоя**: что слой экспортирует наружу и на что имеет право ссылаться. Для читателя PR манифест обязан быть diffable-артефактом: изменение публичного контракта видно в дифе как отдельный файл, а не растворено в коде. Моя норма честности: если кит поменялся, ревьюер видит это первым экраном, а не выуживает из тридцати файлов.

[Музыкант]: Поддержу, но добавлю живой флаг про статус в ревью. Манифест не должен быть декоративным — как индикатор «export done», который горит при клиппинге. Значит статус кита в ревью — **блокирующий гейт**, а не рекомендация: если код нарушил направленность или обошёл манифест, CI роняет PR до человека. Иначе повторим `team-execution-contour`, где гейт получил зубы на выходе, но не голос на входе — и три прогона врезались в стену.

[Математик]: Уточню про зуб численно, чтобы «блокирующий» не остался словом. Зуб — это exit-код проверки в CI: `checkDirection(graph)` возвращает `violations`, и `|violations| > 0 ⟹ exit 1`. Важно: проверка обязана падать **громко** — молчаливый `повесток: 0` из аудита 19.07 показал, как «нарушений: 0» с exit 0 читается как чистота, которой не было. Поэтому контракт: если граф не построился (парсер упал, файл не найден) — это **не** PASS, это отдельный exit-код ошибки инструмента, отличимый от честного нуля нарушений.

[Структурщик]: Согласен, и здесь ключевая лемма M3 — **парсер ответственности как жилец, а не механизм**. «Жилец» = у него есть постоянный адрес в репозитории и словарная статья (что это, вход, выход), а не разовый скрипт. Предлагаю адрес: `scripts/*` внутри скрипт-контейнера (пакет `@membrana/ritual-tools` или его аналог из `scripts-boundary-container`), с публичным контрактом через кит-манифест контейнера. Парсер читает вердикт → извлекает SHA правил, по которым судили → отдаёт историю правок этих правил. Это три чистых шага, каждый — своя функция.

[Математик]: Разложу маршрут расследования формально, потому что «вердикт → SHA правил → история» — это композиция трёх тотальных функций. `f₁: verdict → rulesetSHA` (вердикт обязан нести SHA правил, по которым вынесен — иначе неотзываем). `f₂: rulesetSHA → rulesFile@commit` (по SHA достаём точную версию правил направленности). `f₃: rulesFile → editHistory[]` (git-blame/log по файлу правил). Композиция `f₃ ∘ f₂ ∘ f₁` детерминирована при условии, что вердикт самодостаточен: несёт SHA. Без SHA в вердикте маршрут рвётся на первом шаге — это hard-требование к формату вердикта.

[Teamlead]: Принимаю маршрут, но зафиксирую границу ответственности жильца — чтобы парсер не подменил линтер. Линтер (`checkDirection`) выносит **вердикт о нарушении сейчас**; парсер ответственности реконструирует **как правило дошло до этого состояния** — это разные функции с разными входами. Первый смотрит на граф кода, второй — на историю правил. Не сливать: иначе получим один God-скрипт, который и судит, и расследует, и тихо роняет обе роли, как мы уже видели в аудите 19.07.

[Верстальщик]: Тогда про наблюдаемость маршрута для человека. Расследование прецедента — это не только три функции, это **экран отчёта**: вердикт (что нарушено, какое ребро), SHA правил (ссылка на точную версию), история (кто и когда правил правило). Моя норма из `main-day-accuracy`: отчёт без ревизии врёт. Значит артефакт расследования несёт шапку: head-ревизия кода, SHA правил, timestamp. Без этой шапки читатель не отличит свежий вердикт от вчерашнего.

[Музыкант]: Соглашусь и добавлю про парность из моего корпусного опыта. Вердикт и правила, по которым он вынесен, — это как оригинал и производная: их нельзя разлучать. Если вердикт живёт без ссылки на SHA правил — это трек без указания версии мастера, через год не воспроизведёшь. Поэтому в формат вердикта жёстко: `rulesetSHA` — обязательное поле, не опциональное. Тогда парсер-жилец всегда найдёт вход для `f₁`.

[Структурщик]: Закреплю термин, чтобы кит и линтер не слиплись. **Кит-манифест** — это декларация контракта слоя (что экспортирую / на что ссылаюсь), статический артефакт. **Линтер направленности** — процесс, сверяющий фактический граф с этим манифестом. **Парсер ответственности** — жилец, читающий историю правил. Три разные леммы, три адреса: манифест — в корне контейнера, линтер — правило в `dependency-cruiser`-конфиге, парсер — именованный модуль в `scripts/*`. Ни один не импортирует другой напрямую в запрещённую сторону — сами соблюдаем нашу же конституцию.

[Математик]: Последнее численное замечание — про полноту зуба. Одного правила `layer(a) ≤ layer(b)` мало, если граф импортов неполон: динамический `import()` или чтение файла по строковому пути линтер не видит. Значит зуб честен только на **статически разрешимом** графе; всё, что резолвится в рантайме, — вне области определения `checkDirection`, и это надо явно объявить допущением, а не молча считать чистым. Иначе повторим фантом «нарушений: 0» на непокрытом ребре.

[Teamlead]: Принимаю допущение Математика как явное ограничение конституции: зуб покрывает статический граф, динамические ссылки — вне гарантии и требуют отдельной нормы (не в этом M3). Свожу вердикт. Правило направленности — формула Математика на статическом графе. Зуб — блокирующий exit-код в CI с отличимой ошибкой инструмента. Кит-манифест — блокирующий diffable-артефакт в ревью. Парсер ответственности — жилец по адресу в скрипт-контейнере, маршрут `f₃ ∘ f₂ ∘ f₁` при обязательном `rulesetSHA` в вердикте. Все против сваливания линтера и парсера в один God-скрипт.

[Структурщик]: Принимаю. С оговоркой в DoD: адрес парсера окончательно приземляется на пакет из `scripts-boundary-container` — если тот контейнер к моменту реализации ещё scaffold, парсер живёт временно в `scripts/*` с явной пометкой «переезжает в контейнер», чтобы не плодить второй дом.

[Музыкант]: Принимаю. Мой единственный флаг на будущее: блокирующий гейт без голоса на входе бесполезен — правило направленности должно быть **опубликовано в каноне** (M1-дом), чтобы автор кода знал границу до CI, а не узнавал о ней от красного экрана.

[Верстальщик]: Принимаю. Держу за собой наблюдаемость: артефакт расследования — с честной шапкой (head, rulesetSHA, timestamp), ребро нарушения — явным текстом, не только цветом.

[Математик]: Принимаю. Правило, зуб и маршрут — три детерминированные функции; допущение о статическом графе зафиксировано явно.

[Teamlead]: Принимаю и закрываю M3. Конституция границы слоёв определена: направленность, зуб, кит, парсер-жилец. LGTM на вердикт, реализация — отдельными задачами, не в этом протоколе.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Правило направленности | Слои упорядочены `L = {процедура=0, кит=1, скрипт=2}`; импорт разрешён только `layer(a) ≤ layer(b)`. Норма (процедура) не зависит от исполнителя (скрипт) |
| Машинная проверка | Чистая функция `checkDirection(graph) → violations[]` на **статическом** графе импортов (`dependency-cruiser` / `import-boundary`-линтер); детерминирована, покрыта юнит-тестом |
| Зуб на нарушение | Блокирующий CI: `|violations| > 0 ⟹ exit 1`. Ошибка инструмента (граф не построился) — **отдельный** exit-код, отличимый от честного нуля нарушений |
| Динамические ссылки | Вне области определения зуба — явное допущение конституции; отдельная норма вне M3 |
| Статус кит-манифеста в ревью | **Блокирующий** diffable-артефакт: декларация контракта слоя (экспорт / допустимые ссылки), видимый первым экраном PR; обход манифеста = `нарушена слабая связанность` |
| Публикация правила | Направленность опубликована в каноне (M1-дом) — автор знает границу до CI, не от красного экрана |
| Адрес парсера ответственности | Жилец (постоянный адрес + словарная статья) в скрипт-контейнере (`scripts-boundary-container` / `@membrana/ritual-tools`); временно `scripts/*` с пометкой «переезжает», если контейнер ещё scaffold |
| Маршрут расследования | Композиция `f₃ ∘ f₂ ∘ f₁`: `verdict → rulesetSHA → rulesFile@commit → editHistory[]`. Обязательное поле `rulesetSHA` в формате вердикта — иначе маршрут неотзываем |
| Наблюдаемость расследования | Артефакт с шапкой: head-ревизия, `rulesetSHA`, timestamp; ребро нарушения — явным текстом, не только цветом |
| Разделение ролей | Линтер (вердикт о нарушении сейчас) и парсер (реконструкция истории правила) — **разные функции**, не сливать в God-скрипт |

**Definition of Done (для последующих задач реализации):**
- `checkDirection(graph)` — чистая функция в скрипт-контейнере, юнит-тест на синтетическом графе с нарушением и без; отдельный exit-код для «граф не построился».
- Кит-манифест каждого слоя — отдельный diffable-файл; CI-гейт роняет PR при нарушении направленности или обходе манифеста.
- Правило направленности опубликовано в каноне (M1-дом) со ссылкой на машинную проверку.
- Формат вердикта включает обязательное поле `rulesetSHA`; без него запись вердикта отклоняется.
- Парсер ответственности — именованный модуль со словарной статьёй (вход: `verdict`; выход: `editHistory[]`); адрес приземлён на контейнер или помечен «переезжает».
- Артефакт расследования несёт шапку (head, `rulesetSHA`, timestamp) и явный текст нарушенного ребра.
- Линтер и парсер — раздельные модули, не импортируют друг друга в запрещённую сторону (соблюдают собственную конституцию).
