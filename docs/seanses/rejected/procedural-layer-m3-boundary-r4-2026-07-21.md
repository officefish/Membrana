# Консилиум: конституция границы слоёв «процедуры → киты → скрипты» (E3)

**Повестка:** E3 — чем обеспечивается направленность зависимостей «процедуры → киты → скрипты», чем даётся машинный зуб на нарушение, и где живёт парсер ответственности как жилец (адрес, не механизм). Ответ обязан назвать: правило направленности + его проверку; статус кит-манифеста в ревью; адрес парсера с маршрутом расследования прецедента (вердикт → SHA правил → история правок).

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Математик → Структурщик → Teamlead → Верстальщик → Музыкант (циклически).

---

[Математик]: Формализую направленность, иначе «слои» останутся вкусовыми. Пусть есть три множества узлов: P (процедуры/регламенты), K (киты — стабильные разделяемые модули), S (скрипты-исполнители). Направленность — это ацикличный частичный порядок на рёбрах импорта: `import: S→K`, `S→P` (скрипт читает регламент как данные), `K→P` разрешены; обратные `P→S`, `K→S` — запрещены. Проверяемый предикат нарушения: существует ребро импорта `p∈P → s∈S` или `k∈K → s∈S`. Это чистая функция над графом импортов: вход — список рёбер, выход — множество запрещённых. Пока предикат не кодирован, «направленность» — декларация.

[Структурщик]: Начну с термина, иначе E3 поплывёт. «Кит» — это не синоним «сервиса» и не «пакет вообще». Кит — **стабильный разделяемый модуль-лемма**, на который скрипты ссылаются как на словарную статью: один смысл, публичный API через `index.ts`, без обратной ссылки на потребителя. Направленность у нас уже есть словарём в `ARCHITECTURE.md` (`core` не зависит ни от кого; foundation → core; analyzer → foundation). E3 просит распространить тот же принцип на пару «регламент ↔ скрипт»: регламент — это статья, скрипт — её потребитель. Правило одно: **скрипт импортирует кит и читает регламент; регламент и кит не знают об именах скриптов**.

[Teamlead]: Рамку фиксирую сразу, чтобы E3 не расползся в «перепишем весь тулинг». Мы отвечаем ровно на три вопроса брифа: (1) правило направленности + машинная проверка; (2) статус кит-манифеста в ревью; (3) адрес парсера ответственности с маршрутом расследования. По конструктивизму «форма следует функции»: слой оправдан, только если у него есть зуб. Декларация без проверки — архитектурное украшательство, которое мы запрещали 19.07, когда гейт получил зубы на выходе и не получил голоса на входе. Значит правило Математика должно стать exit-кодом, а не абзацем в `.md`.

[Верстальщик]: С точки зрения потребителя нарушения — а потребитель это агент в CI и человек в PR — правило без **видимого** отчёта о нарушении рождает мёртвую дверь. Моя норма из `code-review-honesty`: индикатор не имеет права гореть зелёным, если проверка молча не выполнилась. Помню находку 19.07: `meeting-audit.mjs` искал повестки по маске, файла не нашёл, вывел «нарушений: 0» с exit 0 — чистота, которой не было. Значит проверка направленности обязана в отчёте называть **что именно** прошло: сколько рёбер проверено, какие запрещённые найдены, с путём импорта. Ноль рёбер — это красный, а не зелёный.

[Музыкант]: По касательной к аудио, но маршрут сигнала я узнаю — это как цепочка эффектов, где обратная связь без контроля рождает вой. Регламент → кит → скрипт — это однонаправленный тракт: source → processing → output. Если скрипт начнёт диктовать регламенту (обратное ребро `P→S`), получишь фидбэк-петлю: тулинг подстраивает правила под себя, ровно как я боялся про аудитора, мерящего своё же дерево. Мой флаг: кит должен быть как разделяемый DSP-модуль — один, стабильный, версионированный; три скрипта тянут один кит, а не три копии формулы.

[Математик]: Уточню машинную проверку до кода. Граф импортов строится статически: обходим `scripts/*.mjs` и `packages/**/index.ts`, извлекаем `import`-специфdocument. Классификация узла — по адресу: `docs/**` и `*_AGENDA.md`/регламенты → P (читаются как данные, не импортируются как код); `packages/services/**`, `packages/core` → K; `scripts/**` → S. Предикат: `∀ edge(a→b): ¬(a∈P ∧ b∈S) ∧ ¬(a∈K ∧ b∈S)`. Регламент физически не может импортировать `.mjs` — он `.md`; но обратное ребро прячется иначе: **скрипт хардкодит логику, которая должна жить в регламенте-данных**. Это уже не граф импортов, это дубль-детектор: одна константа в двух носителях (как `PREMISES_SECTION_TITLE` 19.07). Второй предикат: единственность источника правила.

[Структурщик]: Вот здесь ключевое различие, иначе манифест выродится. Есть **два** класса нарушения, и у них разные проверки. Первый — топологический: обратный импорт `K→S`, ловится графом (Математик). Второй — семантический: **правило-константа продублировано** в ките и в скрипте вместо единого источника. Первый ловит `madge`/статический обход, второй — только реестр китов с полем «единственный носитель правила». Поэтому кит-манифест — это не список пакетов, а **словарь**: `{ кит, публичный API, какие правила он единолично держит }`. `PREMISES_SECTION_TITLE` — эталон кита: одна константа, два потребителя (гейт + промпт), ноль дублей после починки 19.07.

[Teamlead]: Принимаю разведение на два класса. Тогда статус кит-манифеста в ревью — второй вопрос брифа — определяю так: **манифест — это ратифицированный артефакт, изменение которого требует LGTM, как изменение `ARCHITECTURE.md`**. Не декоративный список, а конституционный. Правило: PR, добавляющий скрипт с новым правилом-константой, обязан либо сослаться на существующий кит, либо завести новый кит в манифесте. Без записи в манифесте — BLOCK. Это ровно «декора без смысла не существует»: каждый кит оправдан тем, какое правило он единолично держит.

[Верстальщик]: Тогда манифест должен быть **читаемым артефактом**, не JSON ради JSON. Потребитель — человек в PR-ревью и агент. Прошу форму: таблица `кит | адрес index.ts | правило-лемма | потребители`. Ровно как таблицы в `SERVICES.md` и `DESIGN.md`, которые я читаю не разворачивая. И якоря: каждый кит — кликабельный адрес, чтобы вердикт нарушения мог сослаться на `#кит-premises-title`, а не на пересказ. Мёртвых дверей нет: нашёл дубль — прыгнул на строку манифеста.

[Музыкант]: Поддержу единственность носителя живым флагом. `dads-integration-m1-preset-delivery` 18.07: два носителя пресета (TS + JSON) молча расходятся, `P(equal)<1` и падает с каждым ручным коммитом. То же с правилом-константой: пока правило живёт и в ките, и захардкожено в скрипте — разъедется. Кит как master-трек: правишь мастер, скрипты рендерятся из него. Значит проверка единственности — не «красиво бы», а зуб: grep константы по `scripts/**`, найден литерал вместо импорта кита → красный.

[Математик]: Свожу третий вопрос — парсер ответственности как жилец. «Адрес, не механизм» — значит называем **где живёт** функция расследования, а её реализация отдельным промптом. Формализую маршрут как чистую функцию: `investigate(verdict) → { rulesSHA, editHistory[] }`. Вход — вердикт (протокол консилиума/гейта со ссылкой на кит). Шаг 1: из вердикта извлечь `#кит` → адрес `index.ts`. Шаг 2: `git rev-parse HEAD:<путь-кита>` → SHA правил на момент вердикта. Шаг 3: `git log --follow <путь-кита>` → история правок кита. Детерминировано, без сети, воспроизводимо: один вердикт → один маршрут.

[Структурщик]: Адрес парсера называю по границе словаря. Парсер ответственности — это **аналитический скрипт-потребитель**, он S, не K: `scripts/accountability/investigate.mjs`. Он импортирует кит-манифест (K) и читает вердикты (P как данные) — направленность соблюдена, `S→K`, `S→P`. Он **не** живёт в ките: расследование — не разделяемая лемма, а разовая операция. Манифест китов — `docs/meeting/procedural-layer/WHALES_MANIFEST.md` (регламент-данные, P), сам граф-чекер — `scripts/lint/layer-direction.mjs` (S). Три жильца, три адреса, ни один не знает имени другого сверх контракта.

[Teamlead]: Свожу к вердикту, проверяю границы. Три жильца названы: чекер направленности (S, `scripts/lint/layer-direction.mjs`), манифест китов (P, `WHALES_MANIFEST.md`, ратифицируемый как `ARCHITECTURE.md`), парсер ответственности (S, `scripts/accountability/investigate.mjs`). Граф-предикат + предикат единственности носителя — два зуба. Маршрут расследования: вердикт → `#кит` → SHA (`git rev-parse`) → история (`git log --follow`). Это отвечает на все три вопроса брифа без расширения в «перепишем тулинг». LGTM на форму.

[Верстальщик]: Принимаю, но зафиксирую два условия видимости в DoD. Первое: чекер печатает **сколько рёбер проверено** — ноль рёбер это красный (урок `meeting-audit` 19.07). Второе: манифест — таблица с якорями, не сырой JSON. Тогда потребитель — человек и агент — увидит и нарушение, и адрес правила. Без этого зуб есть, а голоса у него нет — как гейт 19.07 до починки.

[Музыкант]: Принимаю. Одна ремарка на будущее, не переоткрытие: предикат единственности носителя (grep литерала по `scripts/**`) стоит записать так, чтобы он ловил и семантический дубль, не только точное совпадение строки — но это уже задача реализации чекера, не наша комната. Тракт направленный, петля обратной связи закрыта, кит один. Флагов больше не поднимаю.

[Структурщик]: Принимаю. Термины разведены (кит ≠ сервис ≠ скрипт), два класса нарушения размечены (топологический + семантический), три адреса названы без циклов. Единственная норма процесса в DoD: новый кит заводится в манифесте **до** первого скрипта-потребителя — определение до реализации, иначе скрипт унаследует невидимый носитель правила.

[Математик]: Принимаю. Оба предиката детерминированы, без сети, воспроизводимы бит-в-бит на фиксированном дереве. Полнота маршрута расследования проверена тестом «удали шаг — сломается»: без SHA нельзя привязать вердикт к версии правил; без `--follow` теряется история переименований кита. Лишних шагов нет.

[Teamlead]: Консенсус собран, границы M3 удержаны — мы не проектировали реализацию чекера и парсера, назвали конституцию и адреса. Финальный LGTM. Вердикт уходит на ратификацию владельцем.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Правило направленности | Ацикличный порядок импорта: разрешены `S→K`, `S→P`, `K→P`; запрещены обратные `P→S`, `K→S`. Регламент (P) — данные, не импортируемый код; кит (K) — стабильная лемма; скрипт (S) — потребитель |
| Машинная проверка (зуб) | **Два предиката.** (1) Топологический: `∀ edge(a→b): ¬(a∈K ∧ b∈S)` — статический обход графа импортов. (2) Семантический: единственность носителя правила — grep литерала-константы по `scripts/**`, найден вместо импорта кита → красный. Чекер: `scripts/lint/layer-direction.mjs` |
| Классификация узлов | P = `docs/**`, регламенты/`*_AGENDA.md`; K = `packages/services/**`, `packages/core`; S = `scripts/**` |
| Статус кит-манифеста в ревью | Ратифицируемый артефакт уровня `ARCHITECTURE.md`: изменение требует LGTM. Новый скрипт с правилом-константой обязан сослаться на кит или завести новый кит; иначе BLOCK. Адрес: `docs/meeting/procedural-layer/WHALES_MANIFEST.md` |
| Форма манифеста | Таблица с якорями: `кит | адрес index.ts | правило-лемма | потребители`. Не сырой JSON. Каждый кит — кликабельный адрес для ссылки вердикта |
| Адрес парсера ответственности | `scripts/accountability/investigate.mjs` (S, не K — расследование не разделяемая лемма). Импортирует манифест (K), читает вердикты (P) |
| Маршрут расследования прецедента | `investigate(verdict) → {rulesSHA, editHistory[]}`: вердикт → извлечь `#кит` → адрес `index.ts` → `git rev-parse HEAD:<путь>` (SHA правил) → `git log --follow <путь>` (история правок). Детерминировано, без сети |

**Definition of Done (при реализации чекера и манифеста):**
- `scripts/lint/layer-direction.mjs`: строит граф импортов, применяет топологический предикат; **печатает число проверенных рёбер** (ноль рёбер → exit ≠ 0, красный — урок `meeting-audit` 19.07); при нарушении выводит запрещённое ребро с путём импорта.
- Предикат единственности носителя: grep правил-констант по `scripts/**`; литерал вместо импорта кита → exit ≠ 0 с адресом дубля.
- `WHALES_MANIFEST.md`: таблица с якорями (`#кит-<имя>`), колонки `кит | адрес index.ts | правило-лемма | потребители`; новый кит заводится **до** первого скрипта-потребителя.
- Изменение манифеста требует LGTM Teamlead (гейт как для `ARCHITECTURE.md`).
- `scripts/accountability/investigate.mjs`: чистая функция `investigate(verdict)` → `{rulesSHA, editHistory[]}`; без сети, детерминирована; юнит-тест на маршрут «вердикт → SHA → история».
- Тесты: `yarn test:scripts` зелёный; отдельный тест на предикаты чекера (позитив/негатив рёбер).
