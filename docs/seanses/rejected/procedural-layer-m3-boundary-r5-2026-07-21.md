# Консилиум: конституция границы слоёв — направленность «процедуры → киты → скрипты» и жилец-парсер ответственности

**Повестка:** E3 — чем обеспечивается направленность зависимостей «процедуры → киты → скрипты», чем даётся машинный зуб на нарушение, каков статус кит-манифеста в ревью и где живёт парсер ответственности как жилец (адрес, а не механизм) с маршрутом расследования прецедента: вердикт → SHA правил → история правок.

[Teamlead]: Рамку фиксирую сразу, чтобы M3 не расползся в переписывание всей архитектуры. Мы на узле L3 ратифицированного DAG: M1 закрыл «дом» (где живёт процедурный слой), M2 — «словарь» (термины процедуры/кита/скрипта). Наш единственный вопрос — **конституция границы**: правило направленности, его машинная проверка, статус кит-манифеста в ревью и адрес парсера ответственности. По конструктивистскому принципу «форма следует функции»: граница оправдана, только если её нарушение ловится машиной, а не вкусом ревьюера. Три обязательных выхода из брифа — правило+проверка, статус манифеста, адрес парсера с маршрутом расследования.

[Структурщик]: Начну с термина, иначе DAG поплывёт. «Направленность» здесь — не метафора, а **отношение частичного порядка на трёх слоях**: `процедуры → киты → скрипты`, где стрелка значит «может импортировать / вызывать». Это ровно тот же паттерн, что foundation→analyzer в `SERVICES.md`: слой N видит слой N+1, обратное ребро запрещено. «Кит» в нашем словаре (M2) — это опорный модуль-фасад между процедурой и скриптом, единственная легальная дверь вниз. Значит нарушение — это либо обратный импорт (скрипт тянет процедуру), либо обход кита (процедура зовёт скрипт напрямую).

[Математик]: Формализую, иначе «направленность» останется вкусовой. Пусть слои — размеченные вершины графа импортов `G=(V,E)`, функция `layer: V → {0:процедуры, 1:киты, 2:скрипты}`. Легальное ребро `(u→v)` удовлетворяет `layer(v) = layer(u) + 1` (сосед вниз) либо `layer(v) = layer(u)` (внутри слоя). Два запрещённых класса: `layer(v) < layer(u)` — обратное ребро; `layer(v) > layer(u)+1` — прыжок через кита. Проверка — чистая функция `checkLayering(G, layer) → violations[]`, детерминированная, exit-код по `|violations| > 0`. Это тот же аппарат, что `checkLayering` для detectors — переиспользуем, не изобретаем.

[Музыкант]: По касательной, аудио тут не задет, но маршрут сигнала я узнаю — это ровно цепочка source → эффект → выход, где нельзя пустить выход обратно во вход без петли обратной связи, а петля даёт вой. Мой живой флаг из `dads-integration-m1-preset-delivery`: если между слоями два носителя одного правила (манифест в доке и проверка в скрипте), они молча разъедутся. Значит правило направленности и его проверка должны читаться из **одного источника**, а не из двух копий. Иначе манифест-кит станет мёртвой дверью.

[Верстальщик]: С точки зрения потребителя вердикта — а потребитель это агент утром и человек в PR — граница обязана быть **видимой в самом отчёте проверки**. Моя норма из `code-review-honesty`: индикатор не имеет права гореть зелёным, если проверка не запускалась на этой ревизии. Значит вывод `checkLayering` — не просто exit-код, а отчёт с честной шапкой: ревизия HEAD, SHA файла правил, список нарушений с путями импорта. Иначе «✅ границы соблюдены» — мёртвая дверь.

[Teamlead]: Принимаю формализацию Математика как несущую конструкцию. Но зафиксирую приоритет по `ARCHITECTURE.md`: машинная проверка — это **CI-гейт**, не совет. Без прохождения `checkLayering` PR не сливается — это и есть «зуб». Теперь второй выход брифа: статус кит-манифеста в ревью. Мой вопрос к Структурщику — манифест это декларация (что кит экспортирует) или это и есть источник разметки `layer`?

[Структурщик]: Это ключевое различие, и я держу его как две леммы. **Кит-манифест = словарная статья кита**: одно предложение «что это», плюс явный список того, что кит открывает вниз (какие скрипты легально доступны через него). Разметка `layer` — производная от адреса в файловой структуре (M1 закрыл дом), а не от манифеста. Манифест — это **контракт двери**, проверяемый: если процедура импортирует скрипт, которого нет в манифесте кита, — нарушение, даже если слои формально соблюдены. Статус в ревью: манифест — обязательный артефакт кита, его отсутствие = BLOCK, как отсутствие `index.ts` у пакета.

[Математик]: Согласен, и это даёт вторую проверяемую величину помимо слоёв. Первая — `checkLayering` (топология). Вторая — `checkManifestCoverage(G, manifests) → violations[]`: для каждого ребра `процедура→скрипт` проверяем, что скрипт объявлен в манифесте кита-посредника. Обе чистые, обе детерминированные, обе в один отчёт. Важно: манифест не должен дублировать разметку слоёв — иначе `P(рассогласование) → 1` со временем, ровно флаг Музыканта. Слой из адреса, покрытие из манифеста — два ортогональных предиката, один источник каждый.

[Музыкант]: Тогда закрою свой флаг конкретикой. Единый источник разметки — **файловая структура (дом M1)**, а не отдельный YAML со слоями. Манифест кита живёт рядом с китом (как `README` сервиса), и он — единственное место, где перечислены легальные скрипты вниз. Тогда нет двух копий: топология из дерева каталогов, покрытие из манифеста-соседа. Это как трек и его мастер в одной папке с версией — не разъедутся, потому что физически вместе.

[Верстальщик]: Поддержу, и добавлю про третий выход брифа — парсер ответственности как **жилец**. Потребитель здесь — человек, расследующий прецедент: «почему этот вердикт?». Ему нужен не механизм, а **адрес**, куда прийти. По моей норме наблюдаемости: маршрут расследования обязан быть напечатан. Вердикт должен нести ссылку: `вердикт → SHA файла правил на момент проверки → git-история этого файла`. Три клика, а не археология.

[Teamlead]: Вот здесь притормозю — «парсер ответственности» рискует стать новым механизмом, а бриф прямо просит **жильца, адрес, не механизм**. Структурщик, где он живёт логически? Не «как он парсит», а «по какому адресу к нему приходят».

[Структурщик]: Начну с термина, чтобы не построить лишнюю дверь. «Парсер ответственности» — это не сервис и не новый скрипт, это **лемма о том, кто отвечает за вердикт**. Адрес = сам отчёт проверки плюс его provenance. Конкретно: отчёт `checkLayering`/`checkManifestCoverage` пишет в шапку `rulesSha` — SHA файла с правилами направленности (тот, что кодирует `layer` и легальные рёбра) на момент прогона. Парсер ответственности — это **функция чтения этой шапки**, живущая там же, где отчёт, а не отдельный демон. Маршрут: открыл вердикт → прочитал `rulesSha` → `git log <rules-file>` даёт историю правок и автора каждой. Никакого нового пакета.

[Математик]: Формализую маршрут как composition, чтобы он был воспроизводим. `responsibility(verdict) = (verdict.rulesSha, gitHistory(rulesFile @ rulesSha))`. Чистая часть — извлечение `rulesSha` из вердикта (детерминировано). Нечистая, но воспроизводимая — `git log`, который при фиксированном SHA даёт бит-в-бит одну историю. Ключ: `rulesSha` должен вычисляться **в момент проверки** и вписываться в вердикт, иначе через месяц никто не докажет, по какой версии правил вынесен вердикт. Это ровно требование воспроизводимости входа из `evening-auditor-m2p`.

[Музыкант]: Тут соглашусь без оговорок — фиксировать версию правил в момент вердикта это как записать версию плагина в метадату мастера. Через год открываешь — знаешь, каким инструментом сведено. Без этого расследование прецедента превращается в «а какие тогда были правила?» — и ответа нет.

[Верстальщик]: Именно. И на холсте это три поля в шапке отчёта: `Ревизия HEAD`, `SHA правил`, `Нарушений: N`. Каждое нарушение — строка `путь-импорта → тип (обратное ребро / прыжок / вне манифеста)`. SHA правил — кликабельная ссылка в git-историю. Пустое состояние («0 нарушений») тоже честное: показываем, что проверка **прогонялась**, а не молчим. Без цвета-единственного-признака: иконка + текст «Границы соблюдены».

[Teamlead]: Собираю конструкцию. Три несущих: (1) направленность — частичный порядок слоёв из адреса в доме M1, машинная проверка `checkLayering` + `checkManifestCoverage`, CI-гейт как зуб; (2) кит-манифест — обязательный контракт двери вниз, отсутствие = BLOCK; (3) парсер ответственности — не механизм, а адрес: `rulesSha` в шапке вердикта → git-история файла правил. Один источник разметки — файловое дерево. Возражения по существу есть?

[Структурщик]: Одно уточнение термина, чтобы не осталось омонима. Файл правил, чей `rulesSha` мы фиксируем, — это **не** манифесты китов (их много) и **не** дерево каталогов (оно и есть разметка). Иначе «SHA правил» станет неоднозначным. С этим уточнением — принимаю.

[Математик]: Принимаю. Численный итог: два предиката (`layering`, `manifestCoverage`), оба чистые, оба детерминированные, оба гейтят CI по `|violations|>0`. Провенанс — один `rulesSha`, вычисляемый в момент прогона. Маршрут расследования воспроизводим при фиксированном SHA. Возражений нет.

[Музыкант]: Принимаю. Единый источник, версия правил в вердикте, никакой второй копии-манифеста со слоями. Мой флаг мёртвой двери закрыт конструкцией.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Правило направленности | Частичный порядок трёх слоёв `процедуры(0) → киты(1) → скрипты(2)`; легально ребро `layer(v)=layer(u)` или `layer(v)=layer(u)+1`; запрещены обратное ребро (`layer(v)<layer(u)`) и прыжок через кита (`layer(v)>layer(u)+1`) |
| Источник разметки `layer` | **Один** — файловое дерево (дом M1); плюс единый файл правил, кодирующий отношение легальности (его `rulesSha` — провенанс вердикта). Никаких второй копии |
| Машинная проверка (зуб) | Две чистые детерминированные функции: `checkLayering(G, layer)` и `checkManifestCoverage(G, manifests)`; CI-гейт по `|violations|>0` — без прохождения PR не сливается |
| Статус кит-манифеста в ревью | Обязательный артефакт кита = контракт двери вниз (список легальных скриптов); отсутствие = **BLOCK** (как отсутствие `index.ts`); импорт скрипта вне манифеста — нарушение |
| Парсер ответственности (жилец, адрес) | **Не механизм**, а адрес: шапка вердикта несёт `rulesSha` (вычислен в момент прогона) → маршрут `вердикт → rulesSha → git log <rules-file>` (история правок + автор). Нового пакета не заводим |
| Отчёт проверки (наблюдаемость) | Честная шапка: `Ревизия HEAD`, `SHA правил`, `Нарушений: N`; каждое нарушение — путь импорта + тип; честное пустое состояние; иконка+текст, не только цвет |

**Definition of Done:**
- Единый файл правил кодирует функцию `layer(path)` и список легальных межслойных паттернов; его SHA фиксируется в каждом вердикте.
- `checkLayering` и `checkManifestCoverage` реализованы как чистые функции (unit-тесты без DOM/сети, детерминированность бит-в-бит), переиспользуют аппарат `checkLayering` детекторов.
- CI-гейт валит PR при `|violations|>0` для любого из двух предикатов.
- Каждый кит имеет манифест-контракт двери вниз рядом с собой (дом M1); отсутствие манифеста = BLOCK в ревью.
- Отчёт печатает шапку `HEAD / rulesSha / N` и кликабельный маршрут в `git log` файла правил; пустое состояние отображается явно (не молчит).
- Маршрут расследования прецедента воспроизводим: при фиксированном `rulesSha` `git log` даёт бит-в-бит одну историю.
