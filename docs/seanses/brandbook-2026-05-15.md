Консилиум: отдельный пакет брендбука в Membrana
Повестка: нужен ли @membrana/brandbook (или аналог) как самостоятельный workspace-пакет, или достаточно docs/DESIGN.md + DaisyUI в клиенте?

Участники: Vesnin (Teamlead), Ozhegov (Структурщик), Dynin (Математик), Музыкант, Rodchenko (Верстальщик).

Порядок реплик (случайный): Музыкант → Dynin → Vesnin → Rodchenko → Ozhegov → …

[Музыкант]: С моей стороны брендбук — это не логотип на сплэше, а то, как пользователь чувствует интерфейс при работе со звуком: красный «запись», фиолетовый акцент, спокойный фон без мерцания. Сейчас это размазано между DESIGN.md, DaisyUI-темами и цветами canvas в @membrana/audio-data-viz. Отдельный пакет имеет смысл, если он станет единым источником «семантики опасности / записи / нейтрали» для всех визуализаций.

[Dynin]: Уточню границу: брендбук как npm-пакет для меня — набор инвариантов (числа, контрасты, шкала отступов), а не React-компоненты. Математикам нужны только стабильные константы для графиков: палитра линий спектра, пороги яркости, чтобы heatmap не конфликтовал с WCAG. Если пакет тащит JSX — я против; если экспортирует типизированные токены и функции colorForLevel(dB) — за.

[Vesnin]: Стоп на терминах. «Брендбук» в маркетинговом смысле (логотипы, tone of voice, мерч) в репозитории нет и в roadmap v0.1 не фигурирует. Вопрос консилиума переформулирую: пакет дизайн-токенов vs документ DESIGN.md. По ARCHITECTURE.md новый пакет не должен нарушать граф: только @membrana/core или leaf-lib без React — иначе agenda и services утонут в UI-зависимостях.

[Rodchenko]: Я за единый контракт, но Rodchenko уже живёт в трёх местах: DESIGN.md, tailwind.config.js клиента, getCanvasThemeColors() в audio-data-viz. Верстальщику больно: в #41 и #43 снова ловили text-gray-400 вместо base-content. Отдельный пакет с токенами и маппингом на DaisyUI снял бы дубли — но только если content Tailwind и темы подтянутся автоматически, иначе снова рассинхрон.

[Ozhegov]: Структурно предлагаю различать слои. L0 — markdown-спека (человекочитаемо). L1 — @membrana/design-tokens (JSON/TS, zero deps). L2 — опционально @membrana/ui-primitives позже. Пакет с названием brandbook звучит шире, чем нам нужно сейчас, и провоцирует тащить шрифты, иконки, Storybook — это другой бюджет. Имя вводит в заблуждение.

[Музыкант]: Согласен с Ozhegov по имени. Но без L1 я снова увижу в плагине FFT свой оттенок «опасности», не совпадающий с --color-danger из DESIGN.md. Для осциллограммы и индикатора клиппинга нужен один импорт: import { semantic } from '@membrana/design-tokens'.

[Dynin]: Добавлю метрику принятия: пакет оправдан, когда ≥2 потребителя импортируют одни и те же константы. Сейчас потребители: apps/client, audio-data-viz, косвенно agenda (темы). Три — порог есть. Но дублирование пока терпимое: в DESIGN.md ~15 токенов, не сотни.

[Rodchenko]: Практика: если завтра появится device-board с другим UI-стеком, общий пакет нужен вчера. Пока один клиент — риск «пакета ради пакета» и лишний шаг в Turbo ^build. Я бы сначала вынес токены в L1 без React, а логотипы и гайд по тону оставил в docs/brand/ когда появится маркетинг.

[Vesnin]: Фиксирую ограничения из репо: MODULE_AND_PLUGIN_UI.md требует согласовать content Tailwind; agenda не должен зависеть от тяжёлого UI-пакета. Значит L1 — leaf, зависит ни от кого. agenda и audio-data-viz могут зависеть от L1. Запрещено: @membrana/core → design-tokens (core остаётся чистым доменом).

[Ozhegov]: Тогда граф: design-tokens ← audio-data-viz, design-tokens ← apps/client (через alias), опционально design-tokens ← agenda только для типов темы, не наоборот. CI: тест-снэпшот, что hex в canvasThemeColors совпадает с токенами — иначе консилиум бессмысленен.

[Музыкант]: Тест на совпадение цветов — да. И отдельно: анимации 150–200 ms из DESIGN.md в токены не кладём, это CSS; в пакете только то, что импортируется в TS для canvas и WebGL.

[Dynin]: Поддерживаю узкий scope L1: palette, spacing scale, typography numbers, semantic aliases (record, error, accent). Без компонентов, без Storybook в v1.

[Rodchenko]: Для верстки L1 недостаточен без генерации CSS variables в клиентском index.css или скрипта sync — иначе DaisyUI и --color-accent снова разъедутся. Либо tokens → postcss plugin, либо один tokens.css, импортируемый клиентом. Это работа на M-задачу, не на час.

[Vesnin]: Оценка: сейчас активны телеметрия, FFT-плагины, DSP-детектор. Втаскивать L1+sync в середине — шум в diff. Предлагаю решение в два шага, не «всё или ничего».

[Ozhegov]: Шаг 0 (сейчас): усилить DESIGN.md ссылкой «единственный источник имён токенов»; запретить новые raw Tailwind gray в ревью. Шаг 1 (когда закроем текущий волновой плагин или появится второй app): завести @membrana/design-tokens + тест паритета с canvas.

[Музыкант]: Шаг 0 меня устраивает, если Teamlead добавит в чек-лист LGTM пункт «цвета из DESIGN / DaisyUI semantic».

[Dynin]: Шаг 1 — когда будет ADR и список потребителей в registry.json. Иначе пакет умрёт неиспользуемым.

[Rodchenko]: Полный «брендбук-пакет» с логотипами и PDF — нет до появления публичного сайта или второго продукта. Узкий design-tokens — да, но отложенно. Сейчас синхронизируем вручную audio-data-viz с DESIGN.md одним PR, если расхождение найдём.

[Vesnin]: Сводлю голосование. Против @membrana/brandbook как маркетингового монолита — единогласно. За отложенный @membrana/design-tokens (L1) при выполнении порога — консенсус. За немедленный полный пакет — никто.

[Ozhegov]: Формулировка итога: не внедряем brandbook-пакет; ведём дизайн-контракт в DESIGN.md; задачу L1 регистрируем в backlog после стабилизации текущей волны UI-плагинов.

Итоговое решение консилиума
Вопрос Решение
Отдельный пакет «брендбук» (логотипы, гайдлайны, маркетинг, React-примитивы)
Не внедрять на текущем этапе
Отдельный пакет @membrana/design-tokens (типизированные токены, zero deps, потребители: client, audio-data-viz, позже agenda)
Запланировать (M-задача в backlog), не начинать до завершения текущей волны крупных UI-плагинов
До появления L1
Усилить соблюдение docs/DESIGN.md в ревью; при необходимости — разовый PR на паритет canvas ↔ токены
Именование
Не использовать имя brandbook для npm-пакета — только design-tokens или аналог
Definition of Done для будущей L1-задачи: пакет без React; граф зависимостей по ARCHITECTURE.md; тест паритета цветов canvas/DaisyUI; обновлён DESIGN.md + чек-лист в MODULE_AND_PLUGIN_UI.md.

Контекст репозитория: визуальный контракт уже в DESIGN.md; темы — в @membrana/agenda; canvas-цвета — в @membrana/audio-data-viz; отдельного brand-пакета нет. Реплик в диалоге: 24; каждый из пяти участников высказался не менее одного раза.
