# Консилиум: контракт `frames[]` в MANIFEST.json процедуры

**Повестка:** F1 — каков контракт поля `frames[]` в MANIFEST.json процедуры: схема элемента `{id, holder, pin}`, способ расширения закрытой пятиполевой схемы Р1 (`id, leadPersona, kitVersion, engines, precedents`), зуб в `validateProcedure` и судьба шести живущих процедур?

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Музыкант → Teamlead → Структурщик → Математик → Верстальщик (циклически)

[Музыкант]: По касательной — аудио тут не задет, но маршрут сигнала я узнаю: `frames[]` — это как автоматизация треков в проекте, отдельная шина поверх дорожек. Мой первый флаг из `angelina-hostess-m2-assembly`: тотальность и честная дыра. Если у процедуры нет `frames[]` — это не должно молча ронять `validateProcedure` или подхватывать пустоту. Требую: отсутствие поля = легальное состояние «нет кадров», а не дефект.

[Teamlead]: Рамку фиксирую сразу, чтобы F1 не сполз в проектирование самой сборки кадров (это следующая фаза M-цепи) и не переоткрыл Р1. Мы решаем ровно три вещи: (1) схема элемента `frames[]`, (2) легальный способ расширить закрытую пятиполёвку Р1, не превращая манифест в свалку, (3) зуб валидатора и судьба шести живущих процедур. По конституции «форма следует функции»: `frames[]` оправдан, только если несёт функцию, которой пять полей не покрывают. Иначе — не вводим.

[Структурщик]: Начну с термина, иначе DAG поплывёт. «Расширить закрытую схему» — омоним двух лемм: (1) **добавить шестое поле в ту же статью** (ломает вердикт Р1, где ровно пять и лишний ключ = дефект) и (2) **ввести именованное необязательное поле-массив с собственной суб-схемой**, объявленное в том же источнике правил. Р1 закрыл набор *обязательных* полей. `frames[]` — не шестой синоним пяти, а новая лемма «кадры процедуры». Это законно только если оно `optional` и объявлено явно, а не просачивается.

[Математик]: Формализую, иначе «расширение» останется вкусовым. Схема Р1 — предикат `valid(m) = keys(m) = {id, leadPersona, kitVersion, engines, precedents}` (строгое равенство, лишний ключ ⇒ false). Ввести `frames[]` = ослабить предикат до `keys(m) ⊆ Base ∪ {frames} ∧ Base ⊆ keys(m)`, где `frames` — единственный разрешённый необязательный ключ. Элемент кадра — `{id: string, holder: persona, pin?: SHA}`; `pin` опционально ⇒ `frame` без пина легален (latest). Инвариант: `∀ f ∈ frames: f.id` уникален внутри процедуры, `holder ∈ Persona`, `pin` при наличии — валидный SHA графа.

[Верстальщик]: Со стороны потребителя манифеста — а потребитель это агент, читающий MANIFEST как вход промпта, и человек в PR — `frames[]` обязан быть **виден как отдельная секция**, а не растворён среди пяти скалярных полей. Моя норма честности из `main-day-accuracy`: если поле опционально и отсутствует — читатель должен видеть явное «кадров нет», а не гадать. В human-проекции реестра: таблица кадров с колонками id / holder / pin(latest|SHA), пустой массив = строка-заглушка «— кадров нет —».

[Музыкант]: Держу failover-терминал из `angelina-hostess-m1-canon`: если `pin` указывает на битый/несуществующий SHA — это НЕ «тихо взять latest». Это громкий стоп с явной ошибкой, как мёртвая дверь: pin обещал воспроизводимость, а её нет. Разведу с Математиком: `pin` отсутствует ⇒ legal latest; `pin` присутствует, но не резолвится ⇒ **block**. Два разных исхода, не один.

[Teamlead]: Поддерживаю развод Музыканта — это ровно «форма следует функции»: pin существует, чтобы фиксировать, значит битый pin обязан кусаться, иначе поле декоративно. По расширению схемы: принимаю формулировку Структурщика и Математика — не шестое обязательное поле, а единственный разрешённый `optional`-ключ с суб-схемой, объявленный в том же файле правил Р1. Это не переоткрытие Р1, а его **аддитивное** уточнение: обязательное ядро неизменно, вводится именованное расширение. Апеллирую к прецеденту `pl-r3-boundary`: правила живут в одной точке истины.

[Структурщик]: Тогда фиксирую границу словаря. `frames` объявляется в **том же** едином файле правил, что и пятиполёвка (посылка из Р3/Р4 — одна точка истины), с пометкой `optional: true` и суб-схемой элемента. `manifestSchemaProblems` перестаёт быть строгим равенством и становится: `unknownKey ⇔ key ∉ (Base ∪ {frames})`. Любой другой ключ — по-прежнему дефект «свалка». Это сохраняет норму Р1 «манифест — контракт, а не свалка» и добавляет ровно одну легальную статью.

[Математик]: Уточню зуб `validateProcedure`, чтобы он был тотальной чистой функцией. Три независимых предиката: `P1 = Base ⊆ keys ∧ keys ⊆ Base∪{frames}` (структура); `P2 = ∀f∈frames: unique(f.id) ∧ f.holder∈Persona` (внутренняя целостность); `P3 = ∀f: f.pin=∅ ∨ resolves(f.pin)` (пины). Разделяю намеренно: `P1/P2` проверяются без сети (чистая структура), `P3` требует доступа к графу SHA — это отдельный слой проверки. Отсутствие `frames` ⇒ `P2,P3` вакуумно истинны ⇒ процедура без кадров валидна, как требовал Музыкант.

[Верстальщик]: Принимаю разделение P3 — и на холст его выношу честно. В human-проекции: pin со статусом «не резолвится» рисуется `badge-error` «битый пин», latest — `badge-ghost` «latest», валидный SHA — `badge-info` с короткой формой хэша. Не только цвет: иконка + текст, по DESIGN.md a11y. Дыра P3 не должна быть тихой — если сетевой слой проверки не отработал, статус пина = «не проверен» (`badge-warning`), а не ложно-зелёный.

[Музыкант]: Про судьбу шести живущих процедур — слышу это как миграцию старых сессий на новый пульт. Мой флаг из `procedural-layer-m5-migration`: «сразу все» — замаскированный спринт. Раз `frames` опционально и его отсутствие легально (P2/P3 вакуумны) — шесть процедур мигрировать НЕ нужно вовсе. Они валидны как есть, `frames` добавляется по мере надобности той процедуре, которой кадры реально нужны. Zero-cost миграция — вот подарок опциональности.

[Teamlead]: Согласен — и это снимает главный риск переоткрытия Р1. Шесть процедур остаются валидными без изменений, потому что `frames` не обязателен. Никакой миграционной задачи, никаких шести пустых массивов «для галочки». Кто первый жилец `frames[]` — решает следующая фаза (сборка кадров), не мы. Наша зона: контракт и зуб. Фиксирую: судьба шести = «не трогаем, валидны по построению».

[Структурщик]: Одно уточнение по именам, иначе разъедется. `frame.id` — не тот же термин, что `procedure.id` и не `engine.id`: это лемма «идентификатор кадра внутри процедуры», уникальный **в пределах** `frames[]`, не глобально. Требую в суб-схеме явный комментарий-определение, чтобы потребитель не спутал области видимости. И `holder` — это `persona` (нижний регистр, латиница, канон пятерых + `angelina`), не свободная строка; иначе получим `manager`-без-предмета.

[Математик]: Поддержу и заострю: `holder ∈ Persona` — замкнутое множество из канона, проверяется предикатом принадлежности, не regex по строке. Это делает `P2` полностью детерминированным без сети. И зафиксирую численно судьбу шести: при отсутствии `frames` число проверок пинов = 0, число нарушений = 0 по построению — миграция пуста не по решению, а по теореме. «Совпало» тут не при чём: опциональность математически влечёт zero-cost.

[Верстальщик]: Закрою свою зону. В human-проекции реестра процедур `frames[]` — отдельный сворачиваемый блок под пятью скалярными полями, не в одну строку с ними (иерархия через контраст, не через цвет). Пустой блок — статическая строка «— кадров нет —», не скрытый. Diff в PR читается по кадрам атомарно: добавление одного кадра = одна строка таблицы. Это подарок ревью, как и per-slot сборка в `angelina-hostess-m2`.

[Teamlead]: Свожу. Все три вопроса F1 закрыты без переоткрытия Р1: (1) элемент кадра `{id, holder, pin?}`; (2) расширение — единственный `optional`-ключ `frames`, объявленный в той же точке истины правил, ядро пяти полей неизменно; (3) зуб = три предиката P1/P2/P3 (P3 — сетевой слой), судьба шести = не трогаем. LGTM формы. Владелец ратифицирует, дальше — фаза сборки кадров с первым жильцом.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Схема элемента `frames[]` | `{ id: string, holder: persona, pin?: SHA }`. `id` уникален внутри `frames[]` (не глобально); `holder` ∈ канон персон (`vesnin/ozhegov/dynin/kuryokhin/rodchenko/angelina`); `pin` опционален |
| Способ расширения пятиполёвки Р1 | **Не** шестое обязательное поле. Единственный разрешённый `optional`-ключ `frames`, объявленный в том же едином файле правил. Ядро (`id, leadPersona, kitVersion, engines, precedents`) неизменно. Аддитивное уточнение Р1, не переоткрытие |
| `manifestSchemaProblems` | Со строгого равенства → `unknownKey ⇔ key ∉ (Base ∪ {frames})`. Любой иной ключ — по-прежнему дефект «свалка» |
| Зуб `validateProcedure` | Три предиката: **P1** структура (`Base ⊆ keys ⊆ Base∪{frames}`); **P2** целостность (`unique(id) ∧ holder∈Persona`) — без сети; **P3** пины (`pin=∅ ∨ resolves(pin)`) — сетевой слой. Отсутствие `frames` ⇒ P2/P3 вакуумно истинны |
| Битый `pin` | `pin=∅` ⇒ legal latest; `pin` не резолвится ⇒ **block** (громкий стоп, не тихий latest). Сетевой слой недоступен ⇒ статус «не проверен» (`badge-warning`), не ложно-зелёный |
| Судьба шести живущих процедур | **Не трогаем.** `frames` опционален ⇒ шесть валидны по построению (zero-cost, миграция пуста по теореме). Первый жилец `frames[]` — предмет следующей фазы |
| Human-проекция | Отдельный сворачиваемый блок под пятью полями; пустой = статическая строка «— кадров нет —»; pin-статус иконкой+текстом (`badge-error`/`badge-ghost`/`badge-info`/`badge-warning`), не только цветом |

**Definition of Done:**
- В едином файле правил объявлен `optional`-ключ `frames` с суб-схемой `{id, holder, pin?}` и комментарием-определением области видимости `id`.
- `manifestSchemaProblems` обновлён: `Base ∪ {frames}` — легальные ключи; прочие → дефект.
- `validateProcedure` реализует P1/P2/P3 как чистые предикаты; P3 вынесен в отдельный сетевой слой проверки SHA; отсутствие `frames` не даёт ложных нарушений.
- Битый `pin` → block с явной ошибкой; недоступный сетевой слой → статус «не проверен».
- Unit-тесты: процедура без `frames` валидна; дубль `frame.id` → P2 fail; `holder` вне канона → P2 fail; битый `pin` → P3 block; пустой `pin` → latest.
- Human-проекция реестра рендерит блок `frames[]` (пустой = явная заглушка), pin-статус — иконка+текст по DESIGN.md a11y.
- Шесть живущих процедур не изменяются; регресс-тест: они остаются валидными после ввода зуба.
