## Решения для обновления брифа

1. Зафиксировать архитектуру UI как режимы/вкладки `Signal` и `Scenario`.
2. Уточнить scope хакатона 1: web client + редактирование в кабинете (минимальный parity).
3. Зафиксировать на уровне триггеров `onDisconnect` = stop.
4. Отметить timer-based restart как потенциальный v1.1/v2 эпик.
5. В разделе alarm loop оставить задачу формализации порога на базе плагина качества микрофона (A4).

## Открытые вопросы после блокеров

- Формула/порог для «достаточно тихо» (A4): шкала, пороги, длительность удержания.
- Границы server-side редактирования v1: какие операции точно обязаны совпадать с client.
- Политика хранения и синхронизации JSON сценария между клиентом и сервером.

## Дополнительно зафиксировано (P1–P5)

| ID | Решение |
|----|---------|
| P1 | Primary user: специалист по безопасности (возможен профиль оператора ПВО), права зависят от тарифа |
| P2 | После pairing на клиенте пока остаётся текущая модель «модули/плагины»; в будущем возможна развилка клиентов на «лабораторник» и «устройство» |
| P3 | Предзапусковая валидация сценария желательна и должна быть включена в MVP board mode |
| P4 | На устройстве фиксируются 4 системные ветки: `init`, `onStop`, `base loop`, `alarm loop` + дополнительные пользовательские триггеры |
| P5 | Предпочтительно одновременно видеть `Signal` и `Scenario` (например split-view или linked tabs) |

| P6 | Hybrid onboarding: тултипы + wizard на 3 шага + ссылка на manual |
