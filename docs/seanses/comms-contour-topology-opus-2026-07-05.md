# Консилиум: топология контура внешних коммуникаций

**Повестка:** Выбрать топологию для контура communications (нетехнический партнёр, инвариант «сток, не исток»): A — изолированный leaf-воркспейс в монорепо; B — отдельный репозиторий с импортом канона; или гибрид. Решение строго через критерий §5: живое чтение канона при минимальной поверхности синхронизации и выполненном инварианте.

[Структурщик]: Начну с определения, иначе спор расплывётся. «Контур communications» — это leaf-пакет графа: ноль исходящих рёбер к нему, много входящих рёбер от него (он читает канон). Инвариант «сток, не исток» на языке графа зависимостей — это `indegree(comms) = 0` для продуктовых пакетов. У нас уже есть `yarn check:boundaries`, который умеет ловить запрещённые рёбра. [Математик]: Сформулирую критерий §5 как функцию стоимости, чтобы решать численно, а не «на глаз». Пусть `C = S + M`, где `S` — поверхность синхронизации канона (число рукотворных шагов на цикл обновления), `M` — поверхность обслуживания инфраструктуры. Живое чтение — это `S = 0` по определению: контур читает актуальный источник, шага синхронизации нет. Вариант A даёт `S_A = 0`. Вариант B даёт `S_B ≥ 0`, причём `S_B = 0` **только** если зеркало автоматическое, одностороннее и дешёвое; иначе `S_B ≥ 1` рукотворный шаг за цикл. По §5 любой `S_B ≥ 1` активирует оговорку «тогда A предпочтительнее». Это ключевой инвариант расчёта.

[Teamlead]: Согласен с рамкой Математика, но напомню развилку факта, которую задал вопрос: **можно ли партнёру дать read-доступ к продуктовому репозиторию?** Если да — A выигрывает тривиально, потому что живое чтение бесплатно, а инвариант держится границами. Если нет — тогда даже A не даёт партнёру прямого чтения продуктового кода, и надо смотреть, кто именно читает. Уточняю по §3: в обеих топологиях партнёр касается контура **тонким срезом** — монитор каталога + Drive, не git. Читает актуальный код не партнёр, а **агент** внутри контура. Значит вопрос read-доступа — это вопрос доступа агента, не человека.

[Верстальщик]: Как потребитель канона (тон, глоссарий, лист фактов, DESIGN-токены) скажу с позиции выхода: мне важно, чтобы описание компонента совпадало с текущей реализацией без ручной сверки. Живое чтение снимает мой человеческий шаг верификации. Это прямо §3.

[Музыкант]: Добавлю смелую, но честную ноту. Соблазн — сделать красивый отдельный репо с зеркалом, submodule, CI-мостом. Это джаз ради джаза. Если уж рисковать — рисковать в сторону простоты: leaf внутри монорепо, агент читает живьё.

[Структурщик]: Уточню границу термина, чтобы не смешать. Партнёр в git **не сидит ни в A, ни в B** — это зафиксировано брифом §3. Значит аргумент B «партнёр не касается продуктового кода» частично ложен как разделитель: партнёр и в A его не касается. Различие между A и B сводится к одному: **имеет ли агент контура живое чтение продуктового исходника.** В A — да, по построению (один воркспейс, один граф). В B — только если зеркало.

[Математик]: Тогда посчитаю B честно. Пусть зеркало — CI-джоба, которая на каждый push в `main` продуктового репо копирует минимальный канон (`CANON_MINIMUM_FOR_COMMS_CONTOUR.md` и перечисленные файлы) в comms-репо односторонним коммитом. Это `S_B = 0` рукотворных шагов, но `M_B > 0`: надо написать и обслуживать джобу, следить за её падениями, обновлять список файлов при изменении канона. При этом остаётся **лаг**: между push и завершением зеркала канон в B устарел. Лаг → окно дрейфа. В A лага нет: `latency = 0`. Численно: `C_A = 0 + M_A(boundaries, уже есть)`, `C_B = 0 + M_B(зеркало) + risk(лаг)`. `C_A < C_B`.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Топология | **Вариант A** — изолированный leaf-воркспейс `apps/comms-studio` внутри монорепо |
| Обоснование через §5 | Живое чтение канона `S_A = 0` (нулевой лаг, нулевой рукотворный шаг). B даёт `S_B = 0` только при авто-зеркале, но с ненулевым `M_B` (обслуживание + лаг + риск ручной правки копии) → «дешёвая» не выполнена → оговорка §5 «B может перевесить» **не активируется** |
| Гибрид | Отклонён: несёт `M` монорепо и `M` зеркала одновременно — худшее из двух миров |
| Развилка факта (read-доступ) | **Агенту** контура даётся read-доступ к продуктовому исходнику (чтение — весь смысл §3). **Партнёру** git-доступ запрещён в обеих топологиях; партнёр = тонкий срез (монитор каталога + Drive) |
| Инвариант «сток, не исток» | Держится **запретом исходящих рёбер к контуру** (`check:boundaries`), не запретом чтения. Изоляция по записи, не по чтению |
| Механизм синхронизации | **Отсутствует по построению** — один источник, один файл, живое чтение. Односторонность тривиальна: нет копии → нечего расщеплять |
| Точка дрейфа | Единственная — кэш агента; блокируется чтением рабочей копии на каждый прогон (`git pull` в ритме). Копии канона не создаётся → §2 single source of truth выполнен |

**Механизмы изоляции (для A):**
- **Граф:** `apps/comms-studio` — leaf; `dependencies` без `@membrana/*`; чтение канона через fs-read рабочей копии, **не** через `import` (иначе появляется ребро — блок `check:boundaries`).
- **Границы:** `yarn check:boundaries` проверяет `indegree(comms от продуктовых) = 0` и `outdegree(comms к продуктовым) = 0` автоматом.
- **CODEOWNERS:** `apps/comms-studio/**` → owner comms-агент; продуктовые пути → инженеры.
- **CI:** отдельная полоса через `paths`-фильтры workflow; продуктовый `lint typecheck test build` не триггерится правкой контура и наоборот (Turbo scope + path filters).
- **Секреты:** изолированный `apps/comms-studio/.env`, вне продуктового скоупа; вычистка секретов — блокирующее условие перед архивом (канон проекта).
- **Партнёр:** тонкий срез — монитор каталога `apps/comms-studio/out/**` + Drive для ручной правки; прямого git нет.

**Non-goals (что контур НЕ делает):**
- Не деплоит продакшен-лендинг `membrana.space` (решение о хостинге отложено stage-gate, §6).
- Не является вторым источником истины — только потребитель канона (§2).
- Не сажает партнёра в продуктовый git и не даёт ему запись за пределами своего каталога.
- Не импортирует `@membrana/*` пакеты (чтение ≠ импорт).

**Definition of Done:**
- `apps/comms-studio` создан как leaf-воркспейс; `package.json` без `@membrana/*` в зависимостях.
- `yarn check:boundaries` проходит и падает на искусственно добавленном ребре к/от контура (позитивный + негативный тест инварианта).
- Workflow с `paths`-фильтрами: правка `apps/comms-studio/**` не запускает продуктовый пайплайн; правка `packages/**` / `apps/client/**` не запускает comms-полосу.
- CODEOWNERS покрывает `apps/comms-studio/**`.
- Изолированный `.env`; чек вычистки секретов в comms-CI.
- Агент читает канон и живые исходники из рабочей копии (fs-read); документирован ритм `git pull` перед прогоном (закрытие точки дрейфа кэша).
- Канон коммуникаций (тон, dual-use, запрещённые слова, лист фактов, глоссарий) применяется к выходу контура — проверка на этапе рендера.
- INSIGHT-артефакт в каноническом формате зафиксирован (устное решение ≠ канон, §7).
