## Повестка

PC-2 спринта `palette-clarity-nodes`. Консилиум `palette-nodes-free-clarity`
пометил улику №1 **весом консилиума** (не ADR): нет периодического гейта окна без
рекордера — спектральный наблюдательный сценарий вынужден тащить рекордер как
таймер. Затрагивает рантайм тика и владельца времени (core-контракт), потому
полный гейт.

### Улики (заземлены в коде @2026-07-15)

| Факт | Где |
|------|-----|
| `is-recording-window-full` требует на входе `RecorderRef` + `windowSec` (Integer, nullable), ветки true/false | `packages/device-board/src/graph/is-recording-window-full-node.ts:19-45` |
| Спектр-граф заводит `get-recorder` **только как часы**: «IsRecordingWindowFull работает здесь как таймер. Трек не создаётся» | `packages/device-board/src/graph/usercase-free-spectrum-live.ts:281` |
| Спектр использует `collect-fft-frames` (FIFO-путь) | `usercase-free-spectrum-live.ts:72` |
| Тик существует на хосте как **pacing лупа** (`waitUntilNextLoopTick`), не как гейт «прошло N мс» в main | `packages/device-board/src/runtime/host.ts:245` |
| `collectorConfig.windowSec` РЕАЛЬНО читается в collect-samples-пути (`shouldFlushCollect(tickState, config, nowMs)`) | `packages/device-board/src/runtime/collect-node-executor.ts:118` |

### Спорные точки

- **СТ-1.** Новый узел `is-window-elapsed` (гейт по host-часам, без RecorderRef) vs обобщение `is-recording-window-full` до режима «без рекордера».
- **СТ-2.** Отношение к `onTick`/`waitUntilNextLoopTick` (pacing лупа) — не дублируем ли; где граница «pacing» vs «гейт окна».
- **СТ-3.** Находка блока F2 («`collectorConfig.windowSec` рантаймом не используется») — противоречит collect-node-executor:118. Разрешить: какой collect-путь у спектра, декоративно ли поле, чинить ли.
- **СТ-4.** Владелец времени: признать рекордер общими часами ИЛИ ввести явного владельца времени наблюдательного лупа.
- **СТ-5.** Скоуп: это FREE fast-follow (улучшает отгруженный спектр-граф) или шире.

## Итоговое решение (консенсус)

- **Вводим новый узел `is-window-elapsed`** — чистый periodic-гейт по host-часам,
  БЕЗ RecorderRef: вход `windowMs` (Integer, читается рантаймом), выходы exec
  true/false (как `is-valid`/`is-recording-window-full`), per-node состояние «время
  последнего срабатывания» на хосте (периодическое самосбрасывающееся окно). НЕ
  обобщение `is-recording-window-full` (наглядность > компактности палитры;
  невидимых режимов-по-входу не заводим — Rodchenko). Гейт узла палитры пройден
  ЭТИМ консилиумом.
- **Граница с тиком чёткая (СТ-2):** `onTick`/`waitUntilNextLoopTick` — это PACING
  (частота итераций лупа); `is-window-elapsed` — ГЕЙТ (накопилось ли окно),
  ПОТРЕБИТЕЛЬ тика, не дубль. Записать в контракт узла, чтобы не слили. Host-хук
  «прошло ли windowMs» ставится поверх уже существующих host-часов лупа — новой
  инфраструктуры времени НЕ вводим.
- **Скоуп — FREE fast-follow (СТ-5):** узел + пересборка `usercase-free-spectrum-live`
  (снять get-recorder+is-recording-window-full как часы → is-window-elapsed). Живой
  Run = проверка СБОРКИ (пикер, борд держит снапшот), не детекции; живой дрон не
  гейт. **Out of scope:** alarm-loop (свой onTick by design); долг
  `collectorConfig.windowSec` в collect-fft-frames (F2 — path-specific декоративное
  поле, отдельная карточка); обратная совместимость держится (старый узел остаётся).
