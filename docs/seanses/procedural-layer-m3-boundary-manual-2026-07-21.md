Посылки оглашены. Прошу реплики строго по трём пунктам повестки.

[Математик]: По пункту 1 — формализую направленность. Слоям приписан ранг: `rank(процедура)=0, rank(кит)=1, rank(скрипт)=2`. Инвариант границы: любая ссылка/импорт допустимы только при `rank(from) < rank(to)` либо внутри слоя; ребро с `rank(from) > rank(to)` — нарушение (скрипт, знающий о ките или процедуре; кит, знающий о процедуре). Проверка — чистая функция `checkLayerDirection(importGraph) → violations[]` на **статическом** графе импортов; детерминирована, без сети. Динамические ссылки (require по строке) вне гарантии — фиксирую это ограничение явно, оно наследует посылку 10: не молчать о слепой зоне.

[Структурщик]: Уточню носители, чтобы функция Математика имела вход. Граф импортов строится по файловому дереву: принадлежность слою выводится из адреса (посылки 2 и 4 — `docs/procedures/` это ранг 0 через манифест, `scripts/` — ранг 2; киты, когда появятся, — ранг 1 через свои манифесты). Отношение легальности кодирует **единый файл правил** — определение `layer(path)` плюс разрешённые межслойные паттерны. Один файл, один SHA, один blame. Это не кит-манифесты (их много) и не дерево (оно — данные): смешение сделало бы `rulesSha` неоднозначным.

[Музыкант]: Поддержу и заострю посылку 10: зуб обязан кусать на входе, не постфактум. Если направленность живёт только в договорённости — две копии разъедутся молча, как WAV и забытый MP3. Значит `checkLayerDirection` встаёт в CI блокирующим гейтом: `|violations| > 0 → exit 1`, а «граф не построился» — **отдельный** код выхода, отличимый и от нуля, и от нарушений. И правило обязано быть опубликовано в каноне дома (M1), чтобы автор узнавал границу до красного экрана, а не от него.

[Верстальщик]: По пункту 2 — кит-манифест в ревью. Для ведущего ревьюера манифест — **обязательный вход и главный diffable-артефакт**: PR, добавляющий или меняющий кита без записи в манифест (или со ссылками, которые не резолвятся), отклоняется машиной до вкусовых споров. Отчёт проверки — с честной шапкой: ревизия HEAD, `rulesSha`, число нарушений; каждое нарушение — строкой `путь-импорта → тип (обратное ребро / вне манифеста)`. Пустое состояние честное: «0 нарушений» показывает, что проверка прогонялась.

[Математик]: По пункту 3 — маршрут расследования, воспроизводимость. Маршрут детерминирован тогда и только тогда, когда SHA правил зафиксирован **в момент вердикта**: вердикт несёт поле `rulesSha`, расследование читает его, не `HEAD`. Цепочка: вердикт → `rulesSha` → git-история файла правил → прецеденты. При фиксированном SHA `blameTrail` совпадает между прогонами; посылка 9 гарантирует, что история не уедет.

[Структурщик]: Тогда адрес парсера ответственности складывается по слоям без нового God-места: **определение** (канон + манифест) — контейнер процедуры `docs/procedures/attribution/` по контракту M1; **правила** — тот самый единый файл правил, версионируемый конфигом отдельно от кода (Р5); **код**, когда появится, — плоский `scripts/` по ссылке из манифеста, и это подтверждает K9 адресно: реализация станет первым китом в режиме pinned (посылки 6, 7); **прецеденты** — `precedents[]`/`analysis` того же контейнера. Расследование прецедента всегда стартует из одного места — контейнера `attribution` — и идёт по маршруту Математика.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Правило направленности | Ранги `процедура=0 < кит=1 < скрипт=2`; ссылка законна только при `rank(from) < rank(to)` или внутри слоя; принадлежность слою — из адреса, легальность кодирует **единый файл правил** (один файл — один SHA) |
| Машинная проверка | Чистая `checkLayerDirection(importGraph) → violations[]` на статическом графе импортов; динамические ссылки — названная слепая зона |
| Зуб | Блокирующий CI: `|violations|>0 → exit 1`; ошибка построения графа — отдельный exit-код; правило опубликовано в каноне дома до гейта |
| Кит-манифест в ревью | Обязательный вход и главный diffable-артефакт: кит без манифеста или с нерезолвящимися ссылками — машинный BLOCK; отчёт с шапкой HEAD + `rulesSha` + N, честное пустое состояние |
| Адрес парсера ответственности | Определение — `docs/procedures/attribution/` (контракт M1); правила — единый версионируемый файл (Р5); код по ссылке из плоского `scripts/`, реализация — первый кит в режиме pinned (K9 подтверждён адресно); прецеденты — тот же контейнер |
| Маршрут расследования | вердикт (`rulesSha`, зафиксирован в момент вердикта) → git-история файла правил → прецеденты контейнера; воспроизводим при фиксированном SHA |

Вердикт вынесен по E3 полностью; вопросы M4 (грамматика веток) и M5 (миграция) не колонизированы.
