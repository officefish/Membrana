[Верстальщик]: Начну с потребителя вердикта — а он у меня уже болел в `main-day-issue-accuracy` и `truth-graph-q1`. По моей норме честности мёртвых дверей нам нужны минимум три разных статуса на холсте, а не один терминальный. Значит уже до всякой формализации: закрываемых сущностей больше одной.

Наш вопрос узкий: **назвать закрываемую сущность или набор сущностей и их границы**, не рисуя переходы состояний и не проектируя evidence. Мой глаз архитектора видит: «форма следует функции» требует, чтобы у каждой закрываемой сущности была своя функция в системе. [Музыкант]: По касательной, аудио тут не задет, но ритм я слышу и метафора рабочая. Инсайт с многолетним roadmap — это как альбом: он не «закрыт», когда сведён один трек. REVIEW-мандат — это конкретный трек, который ты пообещал свести к дедлайну. А outcome — это когда трек реально зазвучал в зале, у слушателя. Три разных события, три разных «готово». Флаг из опыта `sunrise-flash`: если объявить альбом законченным после первого трека — остальные фазы записаны в пустой зал.

[Структурщик]: Начну с термина, иначе DAG поплывёт. У нас под словом «закрыть инсайт» слиплись **три разные леммы о носителе**: (1) **INSIGHT** — контейнер проблемы + гипотезы + многолетнего roadmap; (2) **REVIEW-мандат** — принятый на ревью следующий шаг с ограничениями, который транскрибируется в task; (3) **OUTCOME** — доказанный продуктовый результат среза. Это ровно тот омоним, что мы разводили в `code-review-honesty` и `truth-graph-q1`: неотменяемое решение ≠ живой статус процесса. Один closable здесь — нарушение нормы «один термин — один смысл».

[Математик]: Формализую, иначе останется вкусовым. У нас три сущности с **разными областями определения и разной динамикой во времени**. INSIGHT живёт на оси идеи: `d/dt ≈ 0` на горизонте месяцев — почти инвариант. REVIEW-мандат живёт на оси суток/спринта: дискретный, `Δ` при каждом ревью. OUTCOME живёт на оси свидетельства: булев предикат `proven(slice) ∈ {true, false}`, проверяемый фактом. Три разных `d/dt` — три разных объекта. Склейка в один — это выдать `n=1` за `n=3`, ровно как эхо-камера в `strategy-day-q5`.

[Верстальщик]: Поддержу и уточню границы для витрины. Тогда идентичность различима так: INSIGHT — крупный заголовок H1 (идея), REVIEW-мандат — карточка среза под ним, OUTCOME — бейдж «доказано/не доказано» на карточке. Три уровня иерархии — читатель за секунду видит, что закрыто, а что открыто. Ключ: закрытие карточки среза **не гасит** заголовок идеи — визуально это отдельные слои фотомонтажа.

[Teamlead]: Согласен с разведением на три, но зафиксирую несущую конструкцию, иначе C1 расплывётся. **Закрываемых сущностей — две терминальные + одна нетерминальная по умолчанию.** REVIEW-мандат (транскрибированный в task) закрывается штатно — это его функция: принят срез, поставлен, закрыт. OUTCOME закрывается как отдельное свидетельство. А INSIGHT **терминально закрывается только явным решением, что идея исчерпана либо отвергнута** — по дефолту он остаётся открытым, пока есть непокрытые фазы roadmap. Это прямо решает кейс Persona: фазы 1/1.5 поставлены, фазы 2–3 живы.

[Музыкант]: Подхвачу кейсы из фактуры, они у меня ложатся на слух. Hermes — это альбом, где принятый срез был полным: все обещанные треки сведены, мандат исчерпал roadmap. Значит его INSIGHT можно закрывать терминально — редкий чистый случай. Persona — альбом с записанными фазами 1/1.5 и открытыми 2–3: закрываешь мандат и, если хочешь, outcome фазы, но альбом играет дальше. Разная судьба INSIGHT при одинаковой механике мандата — вот почему их нельзя слить.

[Структурщик]: Приму разведение Teamlead и уточню идентичность каждой леммы, иначе partial delivery не смоделируется. **INSIGHT**: идентичность = стабильный `insight-id`, границы = проблема+гипотеза+roadmap[]; терминально закрыт ⇔ roadmap исчерпан ИЛИ идея явно отвергнута. **REVIEW-мандат**: идентичность = `(insight-id, review-date, accepted-slice)`, границы = принятый шаг + ограничения; закрыт ⇔ task-транскрипт поставлен. **OUTCOME**: идентичность = `(mandate-id, phase)`, границы = доказанный результат конкретного среза. Тогда **partial delivery** = закрытый мандат + open roadmap; **новая фаза** = новый мандат под тем же `insight-id`; **достигнутый outcome** = отдельный closable над мандатом.

[Математик]: Подтверждаю формально и уточню моделирование трёх кейсов из фактуры численно. Пусть у INSIGHT есть `roadmap = [phase₁…phaseₙ]`, множество закрытых мандатов `M` и множество доказанных outcome `O`. Тогда: **partial delivery** ⇔ `|M| ≥ 1 ∧ covered(M) ⊊ roadmap` (Persona: covered = {1, 1.5}, roadmap ⊋ covered). **Новая фаза** ⇔ добавление мандата `m` с `phase(m) ∉ covered(M)`. **Достигнутый outcome** ⇔ `proven(slice) = true`, элемент O. **Терминальное закрытие INSIGHT** ⇔ `covered(M) = roadmap ∨ rejected`. Hermes: `covered = roadmap` → INSIGHT закрываем. Это чистые предикаты, без state machine — только идентичность и границы, как требует C1.

[Верстальщик]: Тогда закрою тему ложного терминала с витрины — это была исходная боль. Никакой галочки «идея закрыта» без предиката `covered = roadmap`. Это ровно та честность источника, что я требовала в `strategy-day-q4`: экран не показывает терминал там, где его нет.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Какой объект lifecycle закрываемый? | **Три раздельные сущности**, не одна: `INSIGHT`, `REVIEW-мандат`, `OUTCOME` |
| Идентичность INSIGHT | `insight-id`; границы = проблема + гипотеза + `roadmap[]`. **Нетерминален по умолчанию** |
| Идентичность REVIEW-мандата | `(insight-id, review-date, accepted-slice)`; границы = принятый шаг + ограничения; закрыт ⇔ task-транскрипт поставлен |
| Идентичность OUTCOME | `(mandate-id, phase)`; границы = доказанный продуктовый результат среза; закрываем независимо |
| Терминальное закрытие INSIGHT | Только при `covered(mandates) = roadmap` **или** явном `rejected` |
| Partial delivery | `|M| ≥ 1 ∧ covered(M) ⊊ roadmap` (кейс Persona: covered = {1, 1.5}) |
| Новая фаза | Новый мандат `m` с `phase(m) ∉ covered(M)` под тем же `insight-id` |
| Достигнутый outcome | Отдельный closable `proven(slice) = true` над мандатом, не влечёт терминала идеи |
| Кейс Hermes | `covered(M) = roadmap` → INSIGHT **можно** закрывать терминально |
| Ложный терминал | Исключён: галочка «идея закрыта» только по предикату покрытия/отклонения |

**Использованные посылки:**
- Ратифицированный M0; C1 — корень DAG (из брифа).
- INSIGHT содержит проблему, гипотезу и иногда многолетний roadmap.
- REVIEW принимает срез; task транскрибирует мандат.
- Delivery среза не гарантирует outcome и не исчерпывает будущие фазы.
- Hermes = полный принятый срез; Persona = фазы 1/1.5 при живых 2–3.
- MADR отделяет decision status от confirmation реализации.
- Прецеденты команды: омоним-разведение (`code-review-honesty`, `truth-graph-q1`), честность мёртвых дверей (`main-day-issue-accuracy`), эхо/декорреляция (`strategy-day-q5`).

- [x] Определена идентичность и границы каждой сущности.
- [x] Показано моделирование partial delivery, новой фазы, достигнутого outcome.
- [x] Перечислены использованные посылки.
