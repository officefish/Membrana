**Вопрос (по итогам сессии 2026-07-14):**

Спринт `agent-tooling-friction-2` — 7 инструментов против трения агентской работы:
(ti-1) `task:review:ship` — one-shot закрытия задачи после squash-мёржа;
(ti-2) `consilium --secretary-file` + `insight review --review-file` — оффлайн-протоколы с валидацией канона;
(ti-3) `task:register` — регистрация карточки без ручного JSON и merge-конфликтов;
(ti-4) `neighbors` — сводка активности соседних сессий перед стартом работы;
(ti-5) `persona:memory --all` + хвост в ritual:evening;
(ti-6) mojibake/BOM-гард на docs;
(ti-7) унифицированная ошибка «LLM-кредит исчерпан» с подсказкой фолбэка.
Развилки: состав/порядок; дизайн ti-1 (деривация SHA/базы, чеки при ушедшем main); стратегия ti-3 против конфликтов; ti-4 — читать существующие сигналы или вводить coordination-файл; границы (что НЕ трогаем).

[Структурщик]: Начну с границы всего спринта: все семь инструментов — организационный слой `scripts/` + хвосты ритуалов; пакетный граф не трогается вовсе, как в #433. Два жёстких запрета сразу. Первый: ti-1 — **обёртка над существующим** `lib/task-closure-review.mjs`, а не его переписывание: инварианты runner'а (immutable per-SHA артефакт, fail-closed evidence) выстраданы и сегодня четырежды доказали пользу — ship автоматизирует ХОРЕОГРАФИЮ вокруг них, не ослабляет гварды. Второй: ti-3 НЕ меняет формат registry.json — миграция хранилища задач это отдельный adopted-инсайт task-archive-storage (7.6, гейт Voyage), туда не лезем.

[Математик]: Дизайн ti-1 по research Q2 — деривация детерминированная: `gh pr view <N> --json mergeCommit` → точный squash-SHA; база = его родитель (`git show -s --format=%P <sha>`), НЕ «HEAD перед pull» — это устраняет сегодняшнюю ошибку захвата чужих коммитов в диапазон (persona-memory: база пришлось пересчитывать руками). Главный подводный камень (research его подтверждает): между merge и локальным pull в main успевают въехать чужие коммиты — HEAD ≠ mergeCommit, а runner требует равенства для `--check`. Решение видели сегодня руками: ship делает **detached checkout mergeCommit** для прогона чеков и возвращается на main; для финальных коммитов bookkeeping — обычный main с rebase-retry. Это должно быть встроено, а не оставлено человеку.

[Teamlead]: Подтверждаю состав: все семь берём, форма — один M-зонтик с независимыми ti, мёрж по готовности (форма #433 отработала: 4 PR за день без блокировок друг друга). Порядок «дешевле→дороже»: ti-6 (гард) → ti-5 (memory-хвост) → ti-7 (ошибка кредита) → ti-4 (neighbors) → ti-3 (register) → ti-2 (secretary-file) → ti-1 (ship). ti-1 самый ценный, но и самый рискованный — идёт последним, когда рука набита на мелких. Память: ровно так #433 шёл ti-1→ti-3→ti-2→ti-4 и закрылся за сутки.

[Верстальщик]: По ti-4 neighbors — отвечаю на развилку «читать сигналы или вводить coordination-файл». Вводить второй координационный файл — два источника правды, дрейф гарантирован. Вердикт: neighbors ТОЛЬКО ЧИТАЕТ существующие сигналы: (а) коммиты origin/main за N часов, (б) открытые PR, (в) active-карточки за сегодня, (г) свежие remote-ветки, (д) worktree-список. [Музыкант]: ti-5 в моей зоне ритма: рефреш журналов — хвост `ritual:evening` ПОСЛЕ archive:daily-day и ДО code-review (журналы должны впитать протоколы дня до вечерней оценки), с `|| true`-семантикой как telegram-хвост: сломанный рефреш не роняет ритуал. [Структурщик]: ti-3 по research Q1. Наш случай — единый JSON-массив со вставкой в начало у ВСЕХ сессий: конфликт в одной точке гарантирован при параллели (сегодня дважды). File-per-entry — это та самая миграция, которую мы запретили. Из оставшегося research честен: append/prepend сам по себе конфликт не убирает — убирает **регенерация вместо ручного merge**. Вердикт: `task:register` (а) валидирует поля по схеме карточки, (б) вставляет запись детерминированно (в начало — конвенция «свежие сверху» сохраняется), (в) при отказе push делает `pull --rebase` и **перегенерирует вставку заново** (ephemeral regeneration из research) — конфликт руками не решается никогда. Плюс тот же путь для insight-карточек позже — но это follow-up, не ti.

[Математик]: ti-2 secretary-file — валидатор канона протокола, чистые функции: (1) метаданные-таблица присутствует; (2) счёт реплик `[Роль]:` ≥ min (20 консилиум / 5 ролей инсайт-ревью); (3) каждая роль ≥1 реплики; (4) секция итога/консенсуса есть; (5) для инсайта — таблица голосования парсится и средний балл сходится с заявленным (сегодня я бы сам себя проверил).
