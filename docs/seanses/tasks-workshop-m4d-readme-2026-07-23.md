# Заседание M4D: судьба проверки README против реестра

[Структурщик]: Вопрос кажется мелким, но он про правило, а не про файл. `docs/tasks/README.md`
производен от `registry.json`. Вердикт M4B уже сказал: групповая валидность — это то, что нельзя
проверить, глядя на одну карточку. Расхождение производного файла с источником ровно такое:
одну карточку смотреть бесполезно, нужен весь реестр. Значит проверка **по природе** групповая,
и держать её отдельным инструментом — значит завести второй носитель одного правила.

[Teamlead]: Тогда решаем судьбу #1014 честно, а не «закроем и забудем». Иссью просила две
вещи: режим `--check` и строку в pre-commit. Первая — поглощается: это одна из групповых находок.
Вторая — не поглощается, это **место зуба**, вопрос доставки, а не логики. Поэтому #1014 не
закрывается ссылкой на вердикт: она переформулируется в «поставить зуб», а сама проверка уезжает
в валидность.

[Математик]: Про дублирование добавлю предикат, чтобы это не осталось словами. Правило: у
проверки один носитель — предикат групповой валидности; `task:sync-readme --check` допустим, но
обязан быть **тонкой обёрткой** над ним, а не своей реализацией сравнения. Проверяется это
просто: если удалить предикат, `--check` обязан перестать компилироваться, а не продолжить
работать по-своему.

[Верстальщик]: Про место зуба. Pre-commit ловит раньше — и это важнее, чем кажется: живой
дефект 23.07 случился при `cherry-pick`, когда файл разошёлся молча, и заметили его случайно
грепом. Pre-push поздно: к этому моменту коммит уже собран и его придётся править. Но зуб обязан
падать **только** если в индексе есть `registry.json` или `README.md` — иначе он будет тормозить
каждый коммит ради файла, которого никто не трогал.

[Музыкант]: И не чинить молча. Соблазн большой: раз файл производный, пусть хук перегенерит
сам. Но тогда исчезает сигнал — а сигнал здесь и есть ценность. Пусть говорит, что разошлось, и
роняет; чинит человек одной командой.

[Teamlead]: Свожу и подписываю. Логика — в валидности, зуб — в pre-commit, `--check` —
тонкая обёртка, автофикса нет, #1014 переформулируется. LGTM.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| **Поглощение** | Проверка «README против реестра» **уходит в групповую валидность** (`validateRegistry`): по природе она групповая — одну карточку смотреть бесполезно, нужен весь реестр |
| **Судьба #1014** | **Не закрывается ссылкой.** Иссью просила две вещи: логику (`--check`) и место (pre-commit). Логика поглощается; иссью переформулируется в «поставить зуб» и остаётся открытой до этого |
| **`task:sync-readme --check`** | Допустим, но обязан быть **тонкой обёрткой** над предикатом валидности, а не своей реализацией сравнения |
| **Правило против дублирования** | У проверки один носитель. Проверяемо: удаление предиката обязано ломать `--check`, а не оставлять его работать по-своему |
| **Место зуба** | **pre-commit**, и только при наличии в индексе `docs/tasks/registry.json` или `docs/tasks/README.md`. Pre-push поздно: коммит уже собран |
| **Строгость** | Падать, а не предупреждать. Расхождение производного файла — не риск, а свершившийся дефект |
| **Автофикс** | **Запрещён.** Хук не перегенерирует README сам: исчезает сигнал, ради которого зуб и ставится. Хук называет расхождение, чинит человек одной командой |

**Definition of Done:**

- [ ] Проверка расхождения README ↔ registry реализована как групповая находка `validateRegistry`.
- [ ] `task:sync-readme --check` — обёртка над предикатом; своей логики сравнения не содержит.
- [ ] Зуб в `.githooks/pre-commit` срабатывает только при затронутых `registry.json` / `README.md`.
- [ ] Хук падает, не чинит; сообщение называет, что именно разошлось.
- [ ] #1014 переформулирована в «поставить зуб» со ссылкой на этот вердикт.
