# Консилиум: taxonomy тест-гейта по классу правки (M3 / night-build-format)

**Повестка:** T1 — как регламент различает классы правки (код / конфиг / инфраструктура / интеграция), что считать достаточной проверкой для каждого класса, и как честно оформить квант, который юнитом не проверяется в принципе — без фиктивных тестов и без молчаливого обхода гейта.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант (Kuryokhin), Верстальщик (Rodchenko).

**Порядок реплик:** Teamlead → Верстальщик → Структурщик → Музыкант → Математик (циклически)

---

[Teamlead]: Ограда жёсткая: только taxonomy тест-гейта, guardrails и вес-каденс закрыты. Начинаю с формы. У нас безусловный тезис «каждая правка с тестом» дал три протечки за ночь 17.07 — значит тезис не ложен, а недоспецифицирован: гейт обязателен для всех, но его *форма* зависит от класса. Задача — операциональная таблица «класс → достаточная проверка» плюс честная форма для «юнитом не проверяется». Никаких новых сущностей сверх этого.

[Верстальщик]: Поддержу с точки зрения читаемости регламента. Три протечки — это три разных состояния кванта, и сейчас они все выглядят как «зелёный тест», хотя это неправда. Как в UI: нельзя один индикатор вешать на три разных статуса. Нужен явный маркер класса у кванта и явный маркер «проверка нестандартная», чтобы обход был *виден*, а не прятался за галочкой. Честность = состояние не спрятано за метафорой.

[Структурщик]: Определим термины до реализации, иначе расплывётся. Предлагаю четыре леммы, ровно из research:
— **код** — исполняемая логика (`.ts/.mjs` с ветвлениями);
— **конфиг** — декларативные данные (`.json/.yaml/.markdownlint`);
— **инфра** — то, что меняет процесс сборки/деплой/CI;
— **интеграция** — квант, чей смысл — взаимодействие с внешней системой (LLM, сеть, FS-гонки).
Один квант — один первичный класс. Смесь классифицируется по «несущему» изменению, а не по числу файлов.

[Музыкант]: Согласен с четвёркой, но у меня живой кейс, который ломает границу. `NB5` — libuv-гонка `process.exit` после fetch. Это не «код» в смысле логики и не чистая интеграция: баг проявляется только в реальном таймлайне event-loop с живым LLM-ответом. Юнит тут не фиктивен — он *невозможен* без живого вызова. Это отдельное подсостояние: «интеграция, не воспроизводимая в sandbox дёшево». Его нельзя мести под ковёр пометкой, его надо назвать.

[Математик]: Вот здесь я поставлю численный акцент. «Проверяется» — это не булево, это про *правдоподобие ложного зелёного*. Юнит на 12 строк `.markdownlint.json` не даёт информации: он проходит по построению, значит его информационная ценность ≈ 0. Фиктивный тест хуже отсутствия — он повышает ложную уверенность. Поэтому критерий достаточности проверки: **проверка должна быть способна упасть на реальном дефекте этого класса**. Если по построению упасть не может — это не проверка, это декорация.

[Teamlead]: Принимаю формулировку Dynin как несущую: достаточная проверка = та, что способна упасть на реальном дефекте класса. Это отсекает и фиктивный юнит на конфиг, и «зелёный» на печать строки. Тогда таблица строится не «какой инструмент модно», а «что реально ловит дефект данного класса». Структурщик, зафиксируй по классам инструмент.

[Верстальщик]: До инструментов — один нюанс формы для `NB6` (клиентский вывод ласточки, печать строки). Это презентационный вывод: его «дефект» — не логика, а то, что человек прочитал не то. Честная проверка тут — snapshot строки или golden-output, а не «юнит на печать». Snapshot *может* упасть при изменении формата — значит по критерию Dynin он валиден, в отличие от фиктивного юнита. Не всё, что «глазами», беспроверочно.

[Структурщик]: Тогда таблица «класс → достаточный гейт», операционально:
— **код** → юнит-тест, покрывающий ветвление (обязателен, падуч по построению);
— **конфиг** → валидатор схемы/линтер (`docs:lint`, JSON-schema) — юнит запрещён как фиктивный;
— **инфра** → dry-run / `--plan` / симуляция пайплайна, лог результата в квант;
— **интеграция** → sandbox или интеграционный прогон против стенда;
— **интеграция-невоспроизводимая** (подкласс) → декларация + живой прогон с фиксацией. Это пятая строка, о ней ниже.

[Музыкант]: По `NB5` в пятую строку: живой прогон — это не «обход», если он *записан как обязательный шаг*. Предлагаю: квант помечается `verify: live-run`, и close-гейт требует приложить лог живого прогона (timestamp, что именно наблюдалось: гонка не воспроизвелась N раз). Без лога — квант не закрывается. Тогда «верифицируется живым прогоном» перестаёт быть отговоркой и становится артефактом.

[Математик]: Поддержу, но добавлю строгость к пятому классу, иначе он станет мусорной корзиной. Живой прогон обязан фиксировать: (1) число повторов, (2) наблюдаемый инвариант («exit code 0 после fetch, без зависания»), (3) почему sandbox невозможен — одним предложением. Три поля. Если автор не может заполнить «почему sandbox невозможен» — значит sandbox возможен, и квант падает обратно в класс «интеграция». Это защита от злоупотребления пятой строкой.

[Teamlead]: Хорошо. Тогда пятый класс — не лазейка, а *самый дорогой* путь: он требует трёх обоснований и лога, тогда как юнит требует просто зелёного. Стоимость обхода выше стоимости честного теста — это правильный стимул. Rodchenko прав про snapshot, Kuryokhin про лог, Dynin про «способность упасть». Сводим в одну таблицу из пяти строк.

[Верстальщик]: И маркер класса должен быть в самом кванте, в видимом месте — как badge статуса. Предлагаю поле в записи кванта: `edit-class: code|config|infra|integration|integration-live` и `gate: <что предъявлено>`. Тогда при ревью ночи видно глазами, где стандартный гейт, а где заявлен нестандартный. Молчаливого обхода не будет, потому что пустое поле = красный флаг.

[Структурщик]: Согласен, но имя поля — по словарю. `edit-class` однозначно, `gate` слишком общее. Предлагаю `gate-kind` (unit / lint / plan / sandbox / live-run) и `gate-evidence` (ссылка/лог/строка результата). Два поля с одним смыслом каждое, без синонимов. Это ложится в существующую запись кванта в `NIGHT_SPRINT_REGULATION.md` без новой структуры.

[Музыкант]: Один живой риск: не превратить это в бюрократию на каждый чих. Для кванта класса «код» это авто — `gate-kind: unit`, `gate-evidence: имя теста`. Ручное обоснование включается ТОЛЬКО для `config` (почему не юнит), `infra` и `integration-live`. Иначе замедлим ночь и получим формализм ради формализма — а это ровно то, против чего Vesnin.

[Математик]: Численно поддержу это правило пропорциональности. Стоимость проверки должна быть монотонна по риску ложного зелёного. Код с ветвлением — риск высок, юнит дёшев → обязателен. Конфиг — риск низок, юнит бесполезен → линтер. Live — риск высок, sandbox невозможен → дорогая декларация оправдана. Нигде проверка не дороже, чем нужно для отлова дефекта класса. Это и есть «целое (гейт) управляет частями (квантами)», а не наоборот.

[Teamlead]: Фиксирую консенсус. Меняем ровно один раздел регламента — тест-гейт — вводим таксономию пяти классов, два поля в записи кванта, критерий достаточности «способна упасть», и запрет фиктивного юнита как явную норму. Guardrails, вес-каденс, night:open/close, scripts-boundary — не трогаем. Верстальщик и Структурщик, подтвердите формат полей; Математик — критерий; Музыкант — лог live-run.

[Верстальщик]: Принимаю. Поля `edit-class` + `gate-kind` + `gate-evidence`, обоснование обязательно для config/infra/integration-live. Пустое поле = блок.

[Структурщик]: Принимаю. Термины и таблица без синонимов, правится только раздел «тест-гейт» `NIGHT_SPRINT_REGULATION.md`.

[Музыкант]: Принимаю. `integration-live` требует лог живого прогона с числом повторов и наблюдаемым инвариантом.

[Математик]: Принимаю. Критерий достаточности — проверка способна упасть на реальном дефекте класса; фиктивный юнит запрещён; пятый класс требует обоснования невозможности sandbox.

[Teamlead]: Единогласно. Оформляю вердикт и DoD.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Различать ли классы правки при тест-гейте | **Да.** Пять классов: `code`, `config`, `infra`, `integration`, `integration-live` |
| Критерий «достаточной проверки» | Проверка обязана быть **способна упасть** на реальном дефекте класса; фиктивный (непадучий по построению) тест **запрещён** — он хуже отсутствия |
| code → гейт | Юнит-тест на ветвление (обязателен) |
| config → гейт | Линтер / валидатор схемы (`docs:lint`, JSON-schema); юнит запрещён как фиктивный |
| infra → гейт | dry-run / `--plan` / симуляция пайплайна, лог в квант |
| integration → гейт | Sandbox / интеграционный прогон против стенда |
| integration-live (напр. `NB5` libuv-гонка) | Декларация + **лог живого прогона**: число повторов, наблюдаемый инвариант, одно-строчное «почему sandbox невозможен». Без всех трёх → квант падает обратно в `integration` |
| Как квант объявляет класс | Поля в записи кванта: `edit-class`, `gate-kind` (`unit`/`lint`/`plan`/`sandbox`/`live-run`), `gate-evidence` |
| Ручное обоснование | Обязательно для `config`, `infra`, `integration-live`; для `code` авто-заполнение |
| Молчаливый обход | Невозможен: пустое `gate-*` = красный флаг на ревью, блок закрытия |
| Стимул честности | Стоимость обхода (декларация + лог) **выше** стоимости честного теста; проверка монотонна по риску ложного зелёного |
| Разбор трёх ночных протечек | `NB1` markdownlint → `config` + `lint`; `NB6` вывод ласточки → `code`/snapshot golden-output; `NB5` libuv → `integration-live` + лог |
| Границы правки регламента | Меняется **только** раздел «тест-гейт» `NIGHT_SPRINT_REGULATION.md` |

**Definition of Done:**
- В `NIGHT_SPRINT_REGULATION.md` (раздел тест-гейта) добавлена таблица `класс правки → достаточный гейт` из пяти строк, операционально, без синонимов.
- Введены поля записи кванта: `edit-class`, `gate-kind`, `gate-evidence`; описано авто-заполнение для `code` и обязательное обоснование для `config`/`infra`/`integration-live`.
- Зафиксирован критерий достаточности: «проверка способна упасть на реальном дефекте класса»; явный запрет фиктивного юнита.
- Класс `integration-live` требует лог живого прогона (повторы + инвариант + «почему sandbox невозможен»); отсутствие поля блокирует закрытие кванта.
- Три ночных кванта (`NB1`, `NB6`, `NB5`) переразмечены как эталонные примеры в регламенте.
- Не изменены: guardrails, вес/каденс, `night:open/close`, scripts-boundary.
- Прогон валидации регламента `--no-rag --no-memory`.

**Полный список посылок:**
1. Тезис «каждая правка с тестом» верен, но недоспецифицирован — гейт обязателен для всех классов, форма различна (Q1 §4).
2. Research-соответствие: код→юнит, конфиг→линтер/схема, инфра→plan, интеграции→sandbox.
3. Фиктивный тест повышает ложную уверенность → хуже отсутствия (критерий «способна упасть», Dynin).
4. `NB5` не воспроизводится без живого LLM-вызова → нужен пятый подкласс, а не обход пометкой.
5. Стоимость проверки монотонна по риску ложного зелёного (пропорциональность, без бюрократии для `code`).
6. Класс объявляется явным полем → молчаливый обход становится видимым красным флагом.
7. Ограда соблюдена: правится только раздел тест-гейта; guardrails/каденс/night:open-close/scripts-boundary не тронуты.
