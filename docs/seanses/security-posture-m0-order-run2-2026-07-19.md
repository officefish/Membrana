Это чистое отношение «контракт → реализация»: целое (правило дозора) задаёт форму частям (задачам триажа). Ребро `№2 → №1` математически честное.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Корень DAG (фундамент) | **№2 Постоянный дозор** — задаёт порог severity и каденс, без которого триаж зарастает снова |
| Единственное ребро зависимости | `№2 (порог severity + каденс) → №1b (апгрейд-спринт мажоров)` |
| Изолированный лист-действие | **№1a гигиена-патчи** (tar, multer, ws, undici, form-data, fast-uri) — без консилиума |
| Параллельные корни | **№3, №4, №5** — ортогональны (люди/ссылки ↔ OAuth-скоупы ↔ содержимое коммитов), рёбер между ними нет |
| Разрыв мнимого ребра №3→№4 | Инцидент patchhog — **общая причина**, не зависимость; ревизия доступов обязательна независимо от протокола |
| Кандидаты без консилиума (исполнять) | №1a гигиена; critical рантайма (@fastify/middie + 6 high middleware); ревизия выданных OAuth/Apps (фаза №4); gitleaks в pre-commit/CI (фаза №5) |
| Требуют заседания | №2 (дозор), №1b (апгрейд-спринт vite 7 + vitest 3 + fastify-мажор), регламент-фазы №3/№4/№5 |
| Порядок созыва | **M1** — дозор (№2) · **M2** — апгрейд-спринт (№1b) · **M3** — регламенты реагирования (№3 + правило-фаза №4 + дисциплина-фаза №5, один M-lite) |
| Расщепление узлов | №1 = {№1a гигиена-лист, №1b апгрейд-спринт-M}; №4 = {ревизия-лист, правило-M-lite}; №5 = {gitleaks-лист, .env-дисциплина-M-lite} |
