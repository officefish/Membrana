# Повестка консилиума office-panel-contour (2026-07-14)

## Решение владельца (вход, не оспаривается)

Выходим в **подготовительный спринт клиента для office** по адресу
**panel.mmbrn.tech**. Мотивация — две задачи указывают на потребность office в
собственной визуализации: (1) UI-панель «Дрейф-якоря» (эпик #396, развилка
drift-panel-placement «кабинет vs отдельный операторский UI» поднята владельцем
2026-07-13 — этот консилиум её разрешает); (2) таблица trends DRONE_TIGHT vs
yamnet. Оба борда делаются ПОЗЖЕ, опираясь на подготовленный каркас.
Требование владельца: **минимальное приветственное окно доступно всем**, у
конкретных разделов — **свой уровень доступа** ⇒ нужна система авторизации.

## Факты репозитория (проверено кодом)

- `background-office` — stateless-шлюз, **без Postgres** (ADR 0004 Р2, docker-compose
  «stateless, без PostgreSQL»); публичный HTTPS office.mmbrn.tech за Caddy на MSK VDS.
- Уже существует **публичный** `GET /v1/drift-anchor/digest` (решение владельца
  2026-07-13: данные несекретны) — источник данных для drift-борда готов и жив.
- Прецедент клиента: `apps/cabinet` — React+Vite+Tailwind+DaisyUI, token-auth
  (login/password → bearer token → fetchMe) против background-cabinet (Postgres).
- Аудитории: кабинет = пользователи продукта; панель = оператор/владелец + «союзники»
  (нетехническая аудитория telegram-спринта #428, docs/comms/ALLY_PRIMER.md).
- Требования к drift-борду уже проголосованы (консилиум drift-anchor-triggers):
  3 строки якорей, возраст записей, danger «Прод ≠ main» иконка+текст, tabular-nums,
  aria-live, DESIGN.md; устаревание видимо.
- Уроки деплоя: LE rate-limit сожжён ретраями на устаревшем DNS (OM4-C) — ждать
  консистентности NS; comms-контур = прецедент изоляции по записи (эпик CC1-CC9).

## Research-дайджест (Perplexity, 3 запроса — полный текст: office-panel-contour-research-2026-07-14.md)

- **Q1 (форма):** мейнстрим-рекомендация для малых команд — role-gated секция в
  существующем кабинете (переиспользование auth/CI, нет CORS); отдельный SPA
  оправдан, когда другая аудитория/стек или продуктовая изоляция; SSR-страницы —
  только для read-only без интерактива. «Сожаление» отдельного SPA — двойные
  package.json/CI (в нашем turbo-монорепо это дёшево: apps/* уже 7 приложений).
- **Q2 (auth без новой БД):** лучшие варианты — (а) переиспользовать существующий
  JWT/bearer-сервис платформы (кабинетный auth); (б) GitHub OAuth + allowlist-файл
  (роль по user_id, без БД). Magic links и ручные токены тянут mini-БД; статический
  общий токен НЕ различает роли — отклонить. Хранение: **httpOnly cookie**
  (secure, sameSite), НЕ localStorage; CSRF-защита для cookie; **default-deny**:
  глобальный AuthGuard + явный @Public() на витрину.
- **Q3 (риски публичных статус-панелей):** утечка операционных данных полезна
  атакующим/конкурентам; прямые API за панелью — главный вектор обхода;
  Cache-Control: no-store на JSON, rate-limit, data-минимизация (агрегаты, не сырьё),
  блок /debug|/metrics, аудит доступа. Панель — investigation trigger, не источник
  сырых данных.

## Спорные точки (пронумерованы)

1. **Форма приложения**: (a) новый `apps/panel` (SPA, стек кабинета) на
   panel.mmbrn.tech — как задал владелец; (b) research-дефолт «секция в кабинете» —
   есть ли причины переубеждать владельца, или разные аудитории (союзники ≠ клиенты)
   + изоляция (comms-прецедент) закрывают вопрос в пользу (a)? Вердикт разрешает
   развилку drift-panel-placement (#396).
2. **Auth-модель при stateless office**: уровни public / ally / operator / owner.
   Где живут учётки: (а) переиспользовать background-cabinet auth (Postgres есть,
   но связывает панель с кабинетом); (б) GitHub OAuth + allowlist на office (без БД,
   но союзники-нетехнари без GitHub?); (в) invite-токены/коды доступа уровня
   (простота для союзников vs учёт). httpOnly cookie vs bearer; кто выдаёт роль.
3. **Существующий публичный digest**: остаётся публичным (обещание ADR 0004 Р3) или
   уходит за public-уровень панели? Общий принцип для будущих панельных API office
   (default-deny + @Public, Cache-Control, rate-limit — что из Q3-чеклиста в DoD).
4. **Деплой**: Caddy site-block panel.mmbrn.tech (static SPA + reverse_proxy к office
   API), DNS/LE (урок OM4-C), где собирается SPA (turbo build → volume/copy).
5. **Скоуп подготовительного спринта и форма**: welcome-окно (public) + auth +
   каркас разделов с уровнями + деплой = эпик OP1..OPn? Что явно out of scope
   (сами борды drift/trends, контент разделов, LLM-фичи). Размер/владельцы фаз.
