# Консилиум: пилот routine «Ночной триаж реестра задач»

**Повестка:** go/no-go по замене самодельной ночной обвязки облачной Claude Code Routine на одной read-mostly задаче — триаж ghost/orphan/stale в `docs/tasks/registry.json` с отчётом в `docs/reports/night-triage/` + draft PR. Спорные точки: порог N для stale, судьба отчётов, стратегия PR (daily vs rolling), нужна ли перекрёстная проверка через `yarn issues:audit`.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант, Верстальщик (Rodchenko).

**Порядок реплик:** Структурщик → Музыкант → Teamlead → Математик → Верстальщик (циклически)

---

[Структурщик]: По сути пилот не создаёт новую инфраструктуру, а замещает звено — это правильный шаг с точки зрения связанности. Ключевой инвариант «sink not source» соблюдён: отчёт — производная от состояния репо и трекера. Меня беспокоит одно: рутина читает `registry.json` и `README.md`, но классификация ghost требует сопоставления с GitHub Issues. Это внешняя граница, и её контракт должен быть зафиксирован в промпте, а не подразумеваться.

[Музыкант]: Мне нравится сам жанр — «ночной аудитор без человека в контуре». Это как оставить магнитофон писать эмбиент, пока все спят, а утром слушать что вышло. Но у любого автономного прогона без ушей есть риск артефактов: ран без памяти каждый раз изобретает интерпретацию заново. Дисциплина должна жить целиком в тексте промпта, иначе получим дрейф звучания от ночи к ночи.

[Teamlead]: Согласен с рамкой. Это read-mostly задача, push только в `claude/`-ветки, ноль модифицирующих действий вне отчёта — риск дешёвый, откат дешёвый, критический путь не затронут. Это соответствует нашему принципу: конструкция оправдана функцией, лишней обвязки нет. Предварительно склоняюсь к **go с условиями**, но хочу услышать Математика по порогу N и метрикам приёмки — это ядро вопроса «шум vs сигнал».

[Математик]: Порог N=14 дней выбран без обоснования — это гипотеза, а не константа. Правильнее не фиксировать N волюнтаристски, а сделать его **параметром отчёта** на пилоте: помечать stale при разных порогах (7 / 14 / 30) и смотреть распределение. За внешним хаосом реестра стоит структура — я хочу увидеть эмпирику dwell-time задач, прежде чем канонизировать 14. На первом ране N — это измеряемая величина, а не решение.

[Верстальщик]: С точки зрения формата отчёта бриф уже почти всё задал: три таблицы, строка вида id · категория · ссылка · действие · уверенность. Это конструктивистски честно — числа и метки, без метафор. Добавлю только: сводка «сколько ghost/orphan/stale» должна быть первой строкой и читаться за секунду, а колонка «уверенность» — визуально отделять high от low, чтобы утренний ревьюер сразу видел, что требует человека.

[Структурщик]: По границе секретов — это блокирующее условие из брифа, и я его усиливаю. Прежде чем рутина клонирует репо, нужно формально проверить, что `registry.json` и пути чтения не содержат токенов, ключей, приватных URL. Это не «на глаз»: нужен явный grep-чеклист в шаге 2. Нормализационная граница либо есть, либо пилот не стартует.

[Математик]: Проверку секретов возьму как ops-задачу — это моя вторичная зона (Linux, security, bash/mjs). Сделаю простой сканер `scripts/*.mjs`: паттерны на распространённые форматы ключей + запрет на любые значения вида `token`, `secret`, `key` с непустым содержимым в `registry.json`. Блокирующий, exit-code non-zero останавливает шаг 2. Это дешевле, чем доверять глазам.

[Teamlead]: Принимаю границу секретов как gate шага 2. По открытому вопросу №4 — перекрёстная проверка через `yarn issues:audit`: критерий приёмки уже гласит «находки рутины ⊇ находки ручного аудита». Значит `issues:audit` — это **эталон приёмки на пилоте**, а не постоянный CI-гейт. Не будем строить CI-обвязку вокруг research preview, который сам может измениться. Верификация — да, институционализация — нет.

[Музыкант]: Поддержу: не прибивать гвоздями то, что в research preview. Курехинский принцип — риск осознанный и помеченный. Предлагаю на первом ране прогнать промпт дважды на одном срезе с интервалом, чтобы услышать, насколько детерминирован результат. Если два прогона дают разные ghost-списки на неизменном репо — это артефакт, и промпт надо стабилизировать до расписания.

[Структурщик]: Дельно. Детерминизм на неизменном входе — это свойство контракта. Добавлю в критерии приёмки: «два прогона на идентичном срезе дают совпадающие таблицы находок (расхождение только в порядке строк допустимо)». Порядок строк, кстати, надо зафиксировать — сортировка по id, иначе diff PR будет шумным и Верстальщику нечего будет ревьюить глазами.

[Верстальщик]: Стабильная сортировка по id — да, обязательно. Иначе rolling PR раздувается ложными diff'ами. Кстати, это выводит на вопрос №3: daily draft PR против одного rolling PR в неделю. С точки зрения читаемости — daily draft PR лучше: один PR = один срез = один момент ревью. Rolling PR превращается в ленту, в которой теряется «что нового сегодня».

[Teamlead]: Согласен по PR-стратегии: **daily draft PR** на пилоте. Один срез — одна точка ревью, чистая приёмка «diff = ровно один новый файл». Rolling PR оставим как кандидата на фазу 2, если daily окажется шумным. По вопросу №2 (судьба отчётов) — слово Структурщику, это вопрос топологии хранения.

[Математик]: До PR-стратегии добавлю по метрике точности. Критерий «доля ложных ghost/orphan ≤ 20%» на первом ране требует знаменателя — нужно ручное определение истины по каждому срабатыванию. Я предлагаю фиксировать confusion-строку: TP/FP по каждой категории отдельно, потому что ложный ghost и ложный stale имеют разную цену. Один агрегированный процент скрывает структуру ошибки.

[Структурщик]: По хранению отчётов (вопрос №2): `docs/reports/night-triage/` — это накопитель производных артефактов, не источник истины. Логично **архивировать по регламенту** `archive:daily-day`, а не держать бесконечную кучу. Активными в дереве держим последние N отчётов, старше — в архив. Иначе директория станет свалкой, а свалка нарушает «граница пакета = граница словаря».

[Музыкант]: Согласен, но с оговоркой: пока идёт пилот и мы измеряем dwell-time и детерминизм, отчёты лучше **не архивировать сразу** — они нужны рядом для сравнения ночь-к-ночи. Архивацию включаем после приёмки пилота. На фазе измерений история под рукой ценнее чистоты директории.

[Teamlead]: Разумный компромисс: на время пилота отчёты копятся в `docs/reports/night-triage/` без архивации; после приёмки — подключаем `archive:daily-day` с окном хранения активных. Это фиксирую как двухфазное решение. По stale — Математик, финализируй: что уходит в промпт на первом ране?

[Математик]: На первом ране промпт помечает stale при N=14 как основной порог, но **дополнительно** выводит dwell-time каждой активной задачи в отчёт (числом дней с последнего движения). Это даёт эмпирику для калибровки без ветвления логики классификации. Решение по каноническому N принимаем после первого среза — не раньше, чем увидим распределение. Гипотезу не канонизируем без данных.

[Верстальщик]: Тогда в таблицу stale добавляю колонку «дней без движения» (`tabular-nums`, справа). Это и есть честные данные вместо метафоры «давно». Ревьюер видит не только флаг, но и величину — и сам чувствует, где реальный порог. A11y отчёта — markdown, так что забота минимальна, но заголовки таблиц и сводка должны быть машиночитаемы для будущей доставки в фазе 2.

[Структурщик]: Со стороны структуры возражений на go нет. Границы чистые: чтение — registry + README + gh CLI, запись — один файл в `claude/`-ветку, внешних зависимостей от наших пакетов рутина не создаёт. Это не трогает граф `@membrana/*`. Принимаю при условии секрет-гейта и детерминизм-критерия.

[Teamlead]: Тогда фиксирую **GO на пилот** с перечнем условий. LGTM формы решения. Собираем в итоговую таблицу и Definition of Done.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Go/No-go по пилоту | **GO** — read-mostly, риск и откат дешёвые, критический путь не затронут, инвариант «sink not source» соблюдён |
| Порог N для stale | **Не канонизируем**. На первом ране N=14 как основной флаг + вывод dwell-time (дней без движения) каждой задачи. Канонический N — после анализа распределения |
| Судьба отчётов | Двухфазно: во время пилота копятся в `docs/reports/night-triage/` без архивации (нужны для сравнения ночь-к-ночи); после приёмки — `archive:daily-day` с окном хранения активных |
| Стратегия PR | **Daily draft PR** (один срез = одна точка ревью, diff = ровно один новый файл). Rolling PR — кандидат на фазу 2 |
| Перекрёстная проверка `issues:audit` | Эталон приёмки на пилоте, **не** постоянный CI-гейт (не институционализируем обвязку вокруг research preview) |
| Граница секретов | Блокирующий gate шага 2: `scripts/*.mjs`-сканер (владелец — Математик), non-zero exit останавливает пилот |
| Детерминизм | Новый критерий приёмки: два прогона на идентичном срезе дают совпадающие таблицы (порядок строк — сортировка по id) |
| Формат отчёта | Сводка первой строкой; таблицы с колонками id · категория · ссылка · действие · уверенность + для stale колонка «дней без движения» (`tabular-nums`); визуальное отделение high/low; сортировка по id |
| Метрика точности | Фиксировать TP/FP по каждой категории отдельно (не один агрегированный %); цена ложного ghost ≠ цена ложного stale |

**Definition of Done (пилот):**
- Шаг 1: успешный ручной ран research preview на подключённом `officefish/Membrana`; замер расхода квоты одного рана записан.
- Шаг 2: секрет-сканер (`scripts/*.mjs`) реализован и проходит по `registry.json` и путям чтения; non-zero exit блокирует; промпт версионирован в репо.
- Шаг 3: боевой ран → отчёт по правильному пути `docs/reports/night-triage/NIGHT_TRIAGE_<YYYY-MM-DD>.md`; daily draft PR открыт; diff PR = ровно один новый файл.
- Детерминизм: два прогона на идентичном срезе совпадают (расхождение только в порядке строк недопустимо — сортировка по id).
- Полнота: находки рутины ⊇ находки `yarn issues:audit` на одном срезе.
- Точность: доля ложных ghost/orphan ≤ 20% на первом ране; TP/FP по категориям зафиксированы отдельно.
- Отчёт содержит dwell-time каждой активной задачи для калибровки N.
- Итоги (квотная стоимость рана, распределение dwell-time, детерминизм, точность) сведены в `INSIGHT_2026-07-10_NIGHT_TRIAGE_ROUTINE.md`.
- Стоп-условие соблюдено: при недоступности preview или заметном квотном расходе одного рана — заморозка, фиксация в INSIGHT, остаёмся на локальном `night:open`.
