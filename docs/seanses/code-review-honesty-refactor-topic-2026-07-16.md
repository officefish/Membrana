# Повестка: рефакторинг daily code-review — расследование + внешняя практика

> Консилиум перед кодом (rt-8, эпик #539). Два входа: внутреннее расследование
> честности генератора и deep research по практикам 2025–2026. Research — **вход**,
> а не решение: проверяй по нашему коду.

## Вход 1 — расследование (наш код)

Полный разбор: [`code-review-honesty-report-2026-07-16.md`](./code-review-honesty-report-2026-07-16.md). Шесть находок:

1. **daily-ревью не видит кода** (структурно). `daily` → `collectRepositoryContext`, который «не обходит дерево исходников» (самодокумент). Диффа коммитов дня нет; `git diff --name-status` на чистом после коммитов дереве пуст.
2. **Оценивает рабочее дерево, а не работу дня.** 15.07: день крупных слияний (PC-1/PC-2/cowork FREE) → вердикт **T0 docs-only «Runtime не затронут»**, потому что вечером в дереве остались только docs-снапшоты. Ревью пересказывает день, а грейдит остаток.
3. **Слеп и к задаче дня** — `appendTaskContext` для daily не вызывается.
4. **Потреблён вниз по цепочке** — MAIN_DAY_ISSUE 16.07: «DAILY_CODE_REVIEW (15.07): T0, зелёный ✅ Вход в Фазу 0».
5. **Свежесть без гварда** — `|| true` в вечернем ритуале глотает сбой (15.07 фолбэк без кредита отдавал 127), гварда даты вниз по цепочке нет. Латентно.
6. **Слепота не покрыта тестом** (как PG3 #533: гейт врал при 12 зелёных тестах).

Что честно: ревью по запросу (`--pr`/`--branch`/`--uncommitted`) тянет полный дифф — не трогать. Числа тестов реальны. Замороженного хардкода тут нет.

## Вход 2 — внешняя практика (deep research 2025–2026)

Полная выжимка: [`docs/tasks/research/rt-8-code-review-sees-code.md`](../tasks/research/rt-8-code-review-sees-code.md). Ключевое (проверять по нашему коду):

- **Q2 прямо про наш случай:** вечернему ревью подают «**полный дифф (разница между состоянием main утром и вечером) или сжатый набор squash-коммитов**, а не сырое рабочее дерево, чтобы ИИ анализировал итоговую логику». Это дословно направление находки 1–2.
- **Честное ревью vs сводка** — три критерия: глубина (архитектура/логика, не имена файлов), конструктивность (пример кода на каждое замечание), классификация (`must fix` / `should fix` / `nit`). «Если агент пишет „нужно добавить тесты“ без указания граничного случая — это сводка».
- **Лимит объёма** 200–400 строк на ревью; большой дифф разбивать.
- **Q3 риски совпали с находками:** оценка по метаданным, устаревание сводки, ложная уверенность. Гарды: risk-based routing (high-stakes → человек), human-in-the-loop, теневой режим 1–2 недели, блокер только на серьёзный риск, не на стиль.
- **AI как «вход в разговор», не «вердикт»** — найденное превращается в комментарии к PR, а не в авто-мёрж.

Оговорка: в выжимке есть шум генерации («basado», «menschного» — иноязычные вставки), но он не CJK и смысла не искажает.

## Спор для консилиума

Главная развилка — что делать с daily:

**Вариант A — научить видеть код.** daily получает дифф слитых за сутки коммитов (`todaysCommits()` уже есть) вместо рабочего дерева. Ревью становится честным по содержанию. Риск: дифф дня может быть огромным (лимит 200–400 строк из research → как резать?).

**Вариант B — переименовать в «сводку дня».** Признать, что вечерний прогон — это дайджест статусов, не code review; убрать Tier-вердикт, чтобы downstream не читал T0 как «код проверен». Ревью кода остаётся только per-PR (`--pr`, уже честный). Риск: теряем вечерний обзор кода как класс.

**Вариант C — гибрид.** daily = честная сводка дня БЕЗ вердикта качества + отдельный per-PR ревью на критичных изменениях (risk-based routing из research). Downstream получает и обзор, и явную метку «это сводка, не ревью».

## Вопросы

1. A, B или C — и почему? Что первично: вернуть daily код или перестать выдавать его за ревью?
2. Лимит 200–400 строк: дифф целого дня Membrana часто больше. Резать по PR, по пакету, семплировать, или ревьюить только high-stakes-файлы (auth/core/runtime)?
3. Гвард свежести (находка 5): падать при протухшем ревью, как решили для `main-day-probe`, или мягкий `[warn]`? Согласовать с решением по `|| true` из `rt-5`.
4. Вердикт daily кормит планирование (находка 4). Если daily остаётся сводкой — какой сигнал честно давать MAIN_DAY_ISSUE вместо «T0 зелёный»?
5. Классификация `must fix` / `should fix` / `nit` (research) — вводить ли в наш регламент вместо/вместе с Tier T0–T2?
6. Не переусложняем ли? Размер rt-8 — M. Что минимальный честный срез, реализуемый после FREE, а что — отдельная задача?
