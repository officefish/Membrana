# Консилиум: Управление сцепкой карточки (Linear ↔ GitHub)

**Повестка:** Какие инварианты сцепки обязательны и какого уровня каждое нарушение? Порядок проверки, восстановления, фиксации нарушений при запуске утра и в течение дня.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант (Kuryokhin), Верстальщик (Rodchenko).

**Порядок реплик:** Математик → Верстальщик → Музыкант → Структурщик → Teamlead

[Математик]: Начну с формализации, иначе «инвариант» растворится в предчувствиях. Пусть **карточка** — это кортеж `(taskId, linearId, githubIssueNumber, status, dates)`. Есть три **соглашения о синхронизме**: (1) если `linearId ≠ null`, то должен существовать Linear-тикет с этим ID; (2) если `githubIssueNumber ≠ null`, то должен существовать GitHub Issue; (3) если Issue закрыта, то `githubIssueClosedAt` обязана быть заполнена. Без этих трёх — либо мёртвая ссылка, либо дыра в провенансе.

[Верстальщик]: Со стороны читателя карточки в реестре — видимость нарушений обязательна. Предлагаю трёхуровневый цвет: **🟢 green** (все сцепки живы и синхронны), **🟡 yellow** (одна или несколько ссылок неживы, но задача ещё не закрыта — допустимо временно), **🔴 red** (нарушен инвариант синхронизма: fechived Linear остался в Active, или Issue закрыта без даты). Реестр `docs/audit/tasks/registry.json` должен рисовать эту иерархию без раскопок.

[Музыкант]: Флаг на цепь жизненного цикла карточки. Когда задача переходит в Done, должна срабатывать автоцепочка: (1) пометить Issue как `closed-by-workflow`; (2) заполнить `githubIssueClosedAt ← Date.now()`; (3) отправить Linear уведомление о закрытии (если это разрешено). Если хотя бы один шаг молчаливо провалится — получим рассинхрон. Требую явный failover: "не смогли закрыть Issue → СТОП с ошибкой в логе утра".

[Структурщик]: Начну с носителя и условия проверки. Три слоя данных: **Слой 1** — `docs/audit/tasks/registry.json` (единая точка истины на диске); **Слой 2** — Linear API (живой источник для `linearId`); **Слой 3** — GitHub API (живой источник для `githubIssueNumber`). Инвариант: Слой 1 должен быть **консервативен** — если API недоступны, we read Слой 1, не выдумываем. Проверка целостности — чистая функция `checkCardIntegrity(card, cache) → violations[]`, без побочных эффектов, запускается в утреннем ритуале как dry-run, потом в вечернем ритуале — с правом писать `status: invalid` в карточку.

[Teamlead]: Рамку держу — ставлю вопрос по порядку. **Вопрос 1:** какие три инварианта и их уровни критичности? **Вопрос 2:** кто (какой скрипт / когда запускается) отвечает за проверку и восстановление? **Вопрос 3:** что делать, если нарушение обнаружено слишком поздно и карточка уже в репозитории помечена ошибочно? Пока рамка не ясна, ответы разойдутся. Математик формализовал три соглашения — принимаю. Верстальщик описал **видимость** — хорошо. Музыкант назвал **failover** — правильно. Структурщик дал **слои и функцию** — держу. Уточняю: нам нужен не реактивный мониторинг, а **профилактика в ритуалах**.

[Математик]: Уточню уровни. **Критический инвариант** (блокирует save карточки): если задача в Active и linearId ≠ null, то Linear-тикет обязан существовать и быть либо Open, либо In Progress. Если Linear-тикет удалён или заархивирован — пока задача Active, это нарушение уровня 🔴 **HARD_BLOCK** (save отклоняется, пока не чиним). **Мягкий инвариант** (warning, не блокирует): если задача Backlog и у неё есть githubIssueNumber, но Issue уже закрыта три недели — это жёлтое предупреждение, не критично, но стоит почистить. **Инвариант провенанса** (обязательная метаданная): если Issue закрыта, обязана быть заполнена `githubIssueClosedAt` с датой закрытия из GitHub. Это не блокирует save, но при экспорте провенанса будет 🔴 **DATALOSS**.

[Верстальщик]: Со стороны пользователя ревью — картина должна быть прозрачна в один взгляд. Предлагаю в `registry.json` добавить поле `sycnStatus ∈ {green, yellow, red}` с причиной (текст). В UI реестра рядом с ID карточки — иконка цвета + хувер-подсказка причины (ссылка на гайд восстановления, если red). Пример: 🔴 + хувер = «Linear тикет DRU-369 удалён; восстановление: yarn tasks:linear-recover 'containerization-coldstart-sync'».

[Музыкант]: Уточню failover-сценарий при закрытии. Когда задача переходит в Done: (1) `closeGithubIssue(issueNumber)` → попытка закрыть Issue; (2) `updateLinearStatus(linearId, 'Done')` → попытка отметить в Linear; (3) `updateRegistry(cardId, { status: 'Done', closedAt: Date.now() })` → запись на диск. Если (1) или (2) падают — не переходим к (3). Логируем в `docs/audit/tasks/sync-errors.log` с меткой времени и попытаемся ещё раз в вечернем ритуале. Если вечер тоже упал — выводим красный флаг: "Не удалось закрыть карточку; проверьте доступ к Linear/GitHub; вручную в Linear: …".

[Структурщик]: Развожу слои контроля. **Слой 0** — **init-time проверка** (при запуске `yarn ritual:day`): dry-run `checkCardIntegrity` на всех active-карточках; выводим сводку: сколько green/yellow/red. Если красные — выводим список и условия восстановления, но не блокируем ритуал (может быть плановая переиндексация). **Слой 1** — **commit-time проверка** (при попытке `git push`): custom hook проверяет `registry.json` diff — если красная карточка заливается в main, hook требует явного флага `--force-invalid` (фиксация нарушения в commit-message обязательна). **Слой 2** — **evening-time автовосстановление**: скрипт `tasks:sync-repair` переиндексирует все карточки через Linear/GitHub API, освежает сцепки, пишет лог изменений в `docs/audit/tasks/sync-repair.log`.

[Teamlead]: Вердикт по вопросам. **(1) Три инварианта:** Linear-existence (active card → Live Linear ticket), GitHub-existence (если Issue указана, → существует и доступна), Closure-metadata (Issue closed → `githubIssueClosedAt` заполнена). Уровни: Hard-block (инвариант 1), Dataloss (инвариант 3), Warning (инвариант 2 + мягкие ситуации). **(2) Проверка и восстановление:** Слой 0 (утро, dry-run) → Слой 1 (коммит, гейт) → Слой 2 (вечер, авто). **(3) Позднее обнаружение:** если карточка уже в main помечена красной, пишем ADR в `docs/decisions/` с описанием причины и человеческого решения; следующий вечер автоматически переиндексирует.

[Математик]: Формализую function signature для Структурщика. `checkCardIntegrity(card: TaskCard, apiCache?: {linear?, github?}) → {status: 'green'|'yellow'|'red', violations: Array<{invariant: string, level: 'hard'|'soft'|'dataloss', message: string}>}`. Если `apiCache` не передан — работаем только с локальным Слоем 1 (реестр); если передан — проверяем через API. Это позволяет запустить dry-run быстро утром (без API), а ночью запустить с полной проверкой (с API).

[Верстальщик]: Уточню фронт-маттер registry.json. Каждая карточка получает поле `_sync`: `{ status: 'green'|'yellow'|'red', lastChecked: ISO8601, reason?: string, suggestions?: string[] }`. При ревью PR в GitHub видно, какая карточка сломалась и почему. В CLI утра выводим colour-таблицу: `| ID | syncStatus | Reason |` с красным/жёлтым/зелёным фоном.

[Музыкант]: Одна оговорка про failover. При закрытии карточки стоит проверить, не находится ли Linear-тикет в режиме «архив»: если заархивирован — это нормально (человек его закрыл), но не вызываем API updateLinearStatus (это вызовет ошибку). Логируем: "Linear ticket already archived; skipping update". GitHub Issue закрывается всё равно, провенанс заполняется.

[Структурщик]: Добавлю ветвь восстановления. Если обнаружена мёртвая ссылка (Linear-тикет удалён, но в карточке linearId ≠ null): (1) выводим диагноз; (2) предлагаем три действия: `--clear` (обнулить linearId, если задача orphan), `--link-new` (создать новый тикет в Linear и пересцепить), `--manual` (указать новый ID вручную). Выполнение — через интерактивный prompt в утреннем ритуале или через флаг `yarn tasks:repair --clear 'card-id'`.

[Teamlead]: Итоговый вердикт. Принимаю архитектуру из пяти слоёв: **(0) Инварианты** (три кубика, уровни критичности). **(1) Видимость** (цвет + причина в реестре и CLI). **(2) Dry-run утром** (без блокировки). **(3) Гейт коммита** (при push в main). **(4) Авто-репарь вечером** (с логом изменений). Определение Done: (a) `checkCardIntegrity` реализована и покрыта тестами; (b) три инварианта задокументированы в `docs/audit/tasks/SYNC_INVARIANTS.md`; (c) скрипты `tasks:sync-check` и `tasks:sync-repair` работают dry-run / wet-run; (d) обновлена схема `registry.json` с полем `_sync`; (e) ревью diff-ом в PR.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| **Три обязательных инварианта сцепки** | **1.** Если карточка Active и `linearId ≠ null` → Linear-тикет обязан существовать и быть Open/In Progress (уровень: 🔴 HARD_BLOCK). **2.** Если карточка указывает `githubIssueNumber` → Issue обязана существовать и быть доступной (уровень: 🟡 WARNING, если мёртвая ссылка). **3.** Если Issue закрыта → `githubIssueClosedAt` обязана быть заполнена датой (уровень: 🔴 DATALOSS). |
| **Уровни критичности нарушений** | 🔴 **HARD_BLOCK** (инварианты 1, 3 при Active карточке) — блокирует save в registry; 🟡 **WARNING** (инвариант 2, мёртвые ссылки на Backlog карточках) — не блокирует, но выводит alert; 🔴 **DATALOSS** (инвариант 3 без даты) — не блокирует save, но флагируется при экспорте провенанса. |
| **Порядок проверки и восстановления (пять слоёв)** | **(Слой 0)** Сутра: `yarn tasks:sync-check --dry-run` → таблица green/yellow/red, рекомендации, без блокировки ритуала. **(Слой 1)** При `git push`: commit-hook проверяет `registry.json` diff, красная карточка требует `--force-invalid` с обоснованием в commit-message. **(Слой 2)** Вечером: `yarn tasks:sync-repair` → полная переиндексация через API, автозаполнение `_sync`, лог в `docs/audit/tasks/sync-repair.log`. **(Слой 3)** При закрытии карточки: failover-цепь (GitHub close → Linear update → registry update); логирование ошибок, повтор в вечернем ритуале. **(Слой 4)** Позднее обнаружение: ADR в `docs/decisions/` с причиной; автовосстановление не применяется без явного одобрения. |
| **Видимость нарушений в реестре и CLI** | Каждая карточка получает поле `_sync: { status, lastChecked, reason?, suggestions? }`. CLI утра выводит таблицу с цветовой кодировкой. UI реестра (если есть фронт) показывает иконку + хувер-подсказку с ссылкой на гайд восстановления. |
| **Восстановление мёртвых ссылок** | Интерактивный prompt с тремя вариантами: (1) `--clear` (обнулить ID, если orphan); (2) `--link-new` (создать новый тикет в Linear); (3) `--manual <newId>` (указать ID вручную). Выполняется через флаг в утреннем ритуале или через отдельный скрипт `yarn tasks:repair --<действие> 'card-id'`. |
| **API-кэш для быстрой проверки** | Функция `checkCardIntegrity(card, apiCache?)` работает в двух режимах: (a) без API — только локальный реестр, fast path для утра; (b) с API — полная валидация через Linear/GitHub, для вечернего ритуала. Результат кэшируется в `_sync.lastChecked` с TTL 4 часа. |
| **Обработка архивированных Linear-тикетов** | Если Linear-тикет заархивирован → логируем как предупреждение, но не вызываем API updateLinearStatus (это ошибка). GitHub Issue закрывается обычно, провенанс заполняется. |

**Definition of Done:**
- [ ] `checkCardIntegrity()` реализована в `packages/audit/` и покрыта юнит-тестами (минимум 3 сценария: green, yellow, red).
- [ ] Три инварианта задокументированы в новом файле `docs/audit/tasks/SYNC_INVARIANTS.md` с примерами каждого нарушения.
- [ ] Схема `docs/audit/tasks/registry.json` обновлена с полем `_sync` (по умолчанию `{ status: 'unknown', lastChecked: null }`).
- [ ] Скрипты `yarn tasks:sync-check --dry-run` и `yarn tasks:sync-repair` работают и выводят лог.
- [ ] Git hook в `.git/hooks/pre-push` проверяет карточки и требует `--force-invalid` при красной карточке в diff.
- [ ] Вечерний ритуал (`yarn ritual:evening`) вызывает `tasks:sync-repair` автоматически.
- [ ] PR содержит diff-обзор изменений по слоям; `yarn tasks:sync-check` запущен локально без ошибок.

## Список посылок

1. На 23.07 карточка `containerization-coldstart-sync` имела `linearId: null` при попытке save — **факт** (DRU-369, замер).
2. На 18.07 у 0 из 39 active-карточек заполнено поле `githubIssueClosedAt` — **факт** (инвентаризация).
3. Linear API доступен только через прокси с ключом в `.env` worktree — **факт** (DRU-364, архитектура доступа).
4. GitHub API доступен без прокси, Issue содержат поле `closed_at` с датой закрытия — **факт** (GitHub REST v3).
5. Карточка может быть в одном из состояний: Active, Backlog, Done, Archived — **норма** (жизненный цикл из регистра).
6. При переходе карточки в Done должна срабатывать цепь закрытия Issue и обновления Linear — **норма** (требование М4С из повестки).
7. Проверка целостности сцепки должна быть чистой функцией без побочных эффектов — **норма** (архитектурное решение).
8. Утренний ритуал должен предотвращать ошибки, вечерний — исправлять их — **норма** (двуслойная стратегия из MAIN_DAY_ISSUE).
9. Мёртвая ссылка (удалённый Linear-тикет, но `linearId ≠ null` в registry) — состояние, требующее явного выбора действия — **норма** (следует из инвариантов).
10. Реестр `docs/audit/tasks/registry.json` — единственная точка истины для состояния карточек на диске — **норма** (слой 1 архитектуры проверки).
