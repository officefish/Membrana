# Консилиум: F3 — полиморфизм и рекурсия инструментов мастерской в двумерном доме

**Повестка:** Как устроен полиморфизм и рекурсия инструмента `inspectElement` в процедурном (двумерном) доме: диспетчер по роду процедуры, якорный документ как база рекурсии и конечность спуска, поведение стек-подобных операций по этажам.

**Порядок реплик:** Верстальщик → Математик → Teamlead → Структурщик → Музыкант (циклически)

[Верстальщик]: С точки зрения потребителя — а потребитель здесь агент, вызывающий `inspectElement` над процедурой, и человек в PR — начну с честности экрана. Диспетчер по роду должен быть **виден в артефакте обзора**: заголовок «Обзор спринта» vs «Обзор ритуала» — не косметика, а честная метка, какая реализация отработала. Если глагол один, а результат разный, читатель обязан видеть, чей это результат, иначе полиморфизм — «мёртвая дверь», как я говорил в `procedural-layer-m2a-rod`.

[Математик]: Формализую, иначе «полиморфизм» останется вкусовым. Пусть `inspectElement: Element → Detail | recurse`. Элемент типизируется предикатом `isContainer(e) → bool`. База рекурсии — предикат `hasAnchor(e) → bool`: элемент, у которого есть якорный документ и `¬isContainer(e)`, — лист, терминал. Конечность спуска доказывается монотонно убывающей мерой: `depth(e)` строго уменьшается на каждом шаге, дерево контейнеров конечно по построению (T7), значит спуск завершается за конечное число шагов. Это не «на глаз» — это фундированность отношения вложенности.

[Teamlead]: Рамку фиксирую сразу, чтобы M3 не сполз в достройку процедурного дома (T9 отложено) и не переоткрыл `frames[]` (#900). Мы решаем **ровно форму** диспетчера и конечность спуска. По «форма следует функции»: функция полиморфизма — открытость для расширения (новый род процедуры не трогает мастерскую). Значит выбор между «свитч внутри мастерской» и «род приносит свою реализацию» решается не вкусом, а этим критерием. M2 уже склонил ко второму («что/как») — я склонен закрепить, но пусть Структурщик проверит связанность.

[Структурщик]: Начну с термина, иначе DAG поплывёт. «Диспетчер» — омоним двух лемм: (1) **свитч по роду** — мастерская знает конечный набор родов и ветвит внутри себя; (2) **общий интерфейс** — мастерская зовёт `inspectElement` по контракту, реализацию поставляет род. Лемма (1) нарушает открытость: новый род = правка мастерской, прямая зависимость мастерская → все роды. Лемма (2) — слабая связанность: мастерская знает только интерфейс, роды регистрируют реализации. Закрепляю M2: **род приносит свою реализацию**, мастерская объявляет `что`, кит/процедура поставляет `как`.

[Музыкант]: Флаг тихой дыры — как в `angelina-hostess-m2-assembly`. Что если у элемента **нет якоря**? Спуск упёрся в лист, а якорного документа нет — это дыра в полу. Требую тотальности: `inspectElement` ВСЕГДА терминирует, и отсутствие якоря — не тихое падение, а видимый маркер `⚠ anchor-missing`. Это либо зуб (CI-гейт: каждый лист обязан иметь якорь), либо честная находка в обзоре. Молчаливого проваливания сквозь пол быть не должно.

[Верстальщик]: Поддержу Музыканта на холсте. `⚠ anchor-missing` — это `badge-error` в строке элемента обзора, с текстом «якорь не найден», не только цвет (моя норма a11y из `main-day-accuracy`). Читатель обзора видит дыру построчно. А корректный лист с якорем — ссылка на якорный документ, `badge-ghost` «якорь», кликабельно. Обзор становится картой: где твёрдый пол, где провал.

[Математик]: Уточню предикат конечности формально, чтобы зуб Музыканта имел вход. Инвариант границы: `∀ leaf ∈ tree: hasAnchor(leaf) = true`. Нарушение — `∃ leaf: ¬hasAnchor(leaf)`. Это проверяемый предикат над деревом контейнеров, чистая функция без побочных эффектов. `checkAnchorCoverage(tree) → violations[]` — вход для CI-гейта: `|violations| > 0 → находка`, но НЕ обязательно `exit 1` — здесь я разведу с Музыкантом: отсутствие якоря может быть легальным промежуточным состоянием недостроенного дома (T9 отложено), тогда это находка обзора, не блок сборки.

[Teamlead]: Развожу спор Математика и Музыканта — это про уровень строгости зуба. Пока процедурный дом достраивается (T9 отложено), жёсткий CI-блок «нет якоря → красный билд» преждевременен: заблокируем легальную стройку. Решение: **находка обзора сейчас, зуб потом**. `inspectElement` при отсутствии якоря отдаёт `Detail` со статусом `anchor-missing` (видимый маркер), обзор его честно показывает; превращение в блокирующий гейт — отдельная задача после достройки дома. Это форма, не механика.

[Структурщик]: Принимаю развод Teamlead и уточню носитель реализации рода — вопрос повестки «где физически живёт реализация». Не у мастерской (она знает только интерфейс). Варианта два: у **кита процедуры** или у **самой процедуры** (её структура, эхо T10 «лицо выводится из структуры»). Термин развожу: кит — сторона поставки (T6), процедура — сторона спроса. Реализация `inspectElement` для рода — это **поставляемый набор**, значит носитель — **кит процедуры**. Процедура заказывает у кита обзор своего рода; мастерская зовёт по интерфейсу, не зная, чей кит отвечает.

[Музыкант]: По касательной к аудио, но маршрут сигнала слышу — это как плагинная цепочка в DAW: хост (мастерская) знает интерфейс плагина, а сам плагин (кит рода) поставляется отдельно и регистрируется. Согласен с носителем-китом. Но флаг на регистрацию: если кит рода не найден (файл потерян, зеркало битое) — мастерская НЕ должна молча вернуть пустой обзор. Failover-терминал: род без кита → СТОП с явной ошибкой `kit-missing`, как я требовал в `angelina-hostess-m1-canon`. Мёртвая дверь «сломался кит → тихо пустой обзор» запрещена.

[Верстальщик]: Зафиксирую на витрине разницу двух отказов, чтобы читатель не путал. `anchor-missing` (нет якоря у листа) — `badge-error` внутри строки обзора, обзор построился, дыра локальна. `kit-missing` (нет реализации рода) — обзор НЕ построился вовсе, это page-level `alert alert-error` «обзор рода X недоступен: кит потерян», с кнопкой диагностики. Два разных экрана для двух разных провалов — не сваливать в один маркер.

[Математик]: Формализую стек-подобные операции по этажам — вопрос повестки про `decompose`. Пусть `decompose: Container → Element[]`. На плоском доме (ветки, задачи) элемент = лист, `decompose` раскладывает список, `inspectElement(leaf)` не рекурсирует — глубина 1. На процедурном доме `decompose(процедурный-контейнер) → процедуры[]`, каждая процедура — контейнер, `decompose(процедура) → фреймы[]`. Ключевое: это **не бесконечная рекурсия одного глагола**, а полиморфизм верхнего этажа + конечный спуск. Мера `depth` убывает, дерево конечно (T7) — спуск терминирует на якоре. Один глагол на этажах, но рекурсия ограничена фундированностью.

[Teamlead]: Здесь ставлю границу, чтобы M3 не съел механику `frames[]` (#900). Мы фиксируем **что** `decompose` полиморфен по этажам и **что** спуск конечен — это форма. **Как** именно спринт раскладывается на фреймы — это заседание #900, не наше. Реплика Математика про меру `depth` — это доказательство конечности, беру как норму вердикта. Предел спуска: якорный документ, ниже листа нет (T7). Точка.

[Структурщик]: Соберу контракт слоями, чтобы связанность была видна. Мастерская (ранг выше) знает интерфейс `Tool = { inspect, decompose, audit }`. Род процедуры (через свой кит) реализует интерфейс. Мастерская → интерфейс (не → конкретный род) — направленное ребро вниз, ноль обратных. Регистрация родов — реестр родов, аналог `docs/procedures/registry.json` из `procedural-layer-m5`. Новый род = новая запись в реестре + кит, мастерская не трогается. Это открытость для расширения, замкнутость на модификацию — критерий Teamlead выполнен.

[Музыкант]: Принимаю контракт, но добавлю флаг идемпотентности, эхо `swallow-delivery-idempotency`. `inspectElement` и `decompose` — операции **чтения** (обзор не мутирует дом). Повторный вызов на неизменном доме → тот же результат. Если кто-то захочет, чтобы `decompose` создавал фреймы (запись) — это уже не наш глагол чтения, это #900. Держу оси раздельно: обзор читает, стройка пишет. Смешение — та самая заноза, что губила заседания.

[Верстальщик]: Последнее по подаче — рекурсивный обзор должен быть **навигируемым**, а не стеной. Дерево контейнеров рисую как раскрываемые уровни: процедурный дом → процедуры (свёрнуты) → фреймы (по клику). `line-clamp` на глубокие уровни, `truncate` на узкой ширине — моя норма адаптива. Якорный лист — терминальная ссылка, дальше не раскрывается (визуальный сигнал конца рекурсии). Читатель видит глубину, не тонет в ней.

[Математик]: Закрою доказательную часть одним предложением, чтобы вердикт стоял на теореме, а не на вкусе. Конечность гарантирована фундированностью отношения «контейнер ⊃ элемент»: строго убывающая мера `depth` на конечном дереве (T7) плюс терминатор `hasAnchor ∧ ¬isContainer`. Иных исходов, кроме `Detail` или спуска-на-шаг-глубже, у `inspectElement` нет — функция тотальна. Принимаю.

[Teamlead]: Свожу и выдаю форму. Диспетчер — общий интерфейс, род приносит реализацию через свой кит (закрепление M2). Носитель — кит процедуры. База рекурсии — якорный документ, конечность доказана. `anchor-missing` — находка обзора сейчас (зуб потом, после T9). `kit-missing` — жёсткий failover-стоп. `decompose`/`inspectElement` — чтение, идемпотентны, механика фреймов — #900. Границы соблюдены, `ARCHITECTURE.md` не нарушен. **LGTM формы.** Принимаю.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Устройство диспетчера (свитч vs общий интерфейс) | **Общий интерфейс**: мастерская объявляет `что` (`Tool = { inspect, decompose, audit }`), род процедуры приносит реализацию `как`. Свитч по роду отклонён — нарушает открытость и слабую связанность. Закрепление вердикта M2. |
| Где физически живёт реализация рода | В **ките процедуры** (сторона поставки, T6). Процедура заказывает у кита обзор своего рода; мастерская зовёт по интерфейсу, не зная чей кит. Регистрация родов — реестр родов (аналог `docs/procedures/registry.json`). |
| База рекурсии и конечность спуска | **Якорный документ** (T7). Терминатор: `hasAnchor(e) ∧ ¬isContainer(e)`. Конечность доказана фундированностью — строго убывающая мера `depth` на конечном дереве контейнеров. Функция `inspectElement` тотальна. |
| Отсутствие якоря у листа | **Находка обзора сейчас** (видимый маркер `anchor-missing`, `badge-error` + текст), НЕ блокирующий CI-гейт — легальное промежуточное состояние недостроенного дома (T9). Превращение в зуб — отдельная задача после достройки. |
| Отсутствие кита рода | **Жёсткий failover-стоп**: `kit-missing` → обзор не строится, page-level `alert alert-error`. Мёртвая дверь «сломался кит → тихо пустой обзор» запрещена. |
| Стек-подобные операции по этажам | `decompose: Container → Element[]` полиморфен по этажам (плоский дом → лист; процедурный → процедуры; спринт → фреймы). **Не** бесконечная рекурсия — полиморфизм верхнего этажа + конечный спуск до якоря. |
| Природа операций обзора | `inspectElement`/`decompose` — операции **чтения**, идемпотентны, не мутируют дом. Создание фреймов (запись) — #900, вне этой комнаты. |
| Витрина двух отказов | `anchor-missing` — локальная строка обзора (`badge-error`); `kit-missing` — page-level alert. Не сваливать в один маркер; a11y (иконка+текст, не только цвет). |

**Definition of Done (при переходе к задаче в коде):**
- Контракт `Tool` (интерфейс мастерской) и реестр родов с реализациями `inspectElement`/`decompose`/`audit` в отдельном носителе (кит процедуры) — без прямой зависимости мастерская → конкретный род.
- Чистая функция `checkAnchorCoverage(tree) → violations[]` (без побочных эффектов), unit-тесты на конечность спуска и тотальность `inspectElement`.
- `inspectElement` возвращает `Detail` со статусом `{ ok | anchor-missing }`; `kit-missing` — явная ошибка (failover-стоп), не пустой результат.
- UI обзора: раскрываемое дерево уровней, `anchor-missing` = `badge-error`+текст, `kit-missing` = page-level `alert-error`; a11y (иконка+текст, `aria-live` на смену статуса), соответствие `DESIGN.md`.
- Нет прямых импортов между родами процедур; регистрация — только через реестр родов; границы пакетов по `ARCHITECTURE.md`/`SERVICES.md` соблюдены.
- Механика `frames[]`/`auditPins` (#900) НЕ затронута; T9-достройка дома НЕ выполняется в рамках этой задачи.
