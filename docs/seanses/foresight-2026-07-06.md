## Ключевой вывод: free выпускается ДО ~17.07

Нейро-детектор по канону (`INTEGRATIONS_STRATEGY §4.2.1`, Способ A) — **zero-shot, в браузере, без
обучения** (YAMNet TF.js / CLAP onnx). Следствия:

- **Весь критический путь free НЕ заблокирован оборудованием** — строится сейчас.
- Полевые записи (~17.07, VDR hard-gate) — это **валидация качества детекции** и возможный fine-tune
  (эшелон 2), **параллельный/последующий трек, не блокер выпуска**.
- Free уходит с детекцией в статусе «бета»; VDR-сертификация приходит следующим релизом.

Итог: ~2 недели чистой продуктовой работы без ожидания железа.

## Критический путь и горизонты

```
S1 Нейро-плагин (UC2) ─► S2 Combined UC (fusion + alarm-loop) ─► S3 Упаковка UserCases ─► S5 Лендинг
                                                                     ▲
                        S4 Студия → download (параллельно) ─────────┘
```

- **S1 — Нейро-плагин детекции (keystone).** Zero-shot YAMNet(TF.js) по образцу существующих
  спектральных/trends-плагинов; scaffold `packages/services/detectors/yamnet`. Разблокирует S2.
- **S2 — Комбинированный UC.** Fusion DRONE_TIGHT + нейро-скор + alarm-loop (громкость → относительная
  дистанция). Зависит от S1.
- **S3 — Упаковка user cases** в device-board UserCase (сейчас развёрнут ~1 из 4).
- **S4 — Студия к скачиванию** (Electron-сборка + флоу download). Почти независима — студия есть.
- **S5 — Лендинг** с CTA «скачать студию» + витриной сценариев. Последний.

Критический путь: **S1 → S2 → S3 → S5**; **S4** параллельно.

## Принятые решения (go/no-go)

1. **Модель UC2:** первый плагин — **YAMNet** (TF.js, score по Aircraft/Helicopter/Propeller — быстрее
   выпустить). **CLAP** — второй итерацией.
2. **Alarm-loop:** грубый индикатор «ближе/дальше» по громкости, честно помеченный как ориентировочный
   (не калиброванные метры).
3. **Выпуск vs сертификация:** free уходит до 17-го с пометкой «детекция в бете»; VDR-валидация —
   следующим релизом, выпуск ради неё не держим.
4. **Упаковка:** пакуем все 4 UserCase; combined (UC4) — витринный.

## Следствие для «фокуса дня»

Утренний план должен вести по критическому пути (S1→S5), а не по детекционной полировке (A/B/C уже
сделаны) и не по comms. Механическая причина дрейфа фокуса — `scripts/_strategic-plan.mjs` +
`scripts/lib/detection-planning-priorities.mjs` (зашита детекция как магистраль). Правка этих входов —
отдельная небольшая задача.
