# Консилиум: Классовый состав тест-сплита DADS для боевого смысла цифры прогона (M4)

**Повестка:** M4 — какой классовый состав обязан иметь тест-сплит DADS, чтобы цифра прогона означала поведение детектора в бою? Вердикт называет: пропорцию сплита и обоснование, главную метрику, соотношение с гейтом 85/90.

**Участники:** Teamlead (Vesnin), Структурщик (Ozhegov), Математик (Dynin), Музыкант (Kuryokhin), Верстальщик (Rodchenko).

**Порядок реплик:** Teamlead → Структурщик → Верстальщик → Математик → Музыкант (циклически)

---

[Teamlead]: Рамку фиксирую сразу, чтобы M4 не расползся. Мы на узле L2 ратифицированного DAG: корень закрыт ADR-0006 (прогон исполняет калиброванный пресет), состав DADS — факт. Наш единственный вопрос: **каким должен быть тест-сплит, чтобы цифра означала бой**, а не удобную статистику на перекошенном датасете. Приоритет `WHITE_PAPER.md` §8: Single-Node Detection First — мерим качество детекции на узле, а не мультиузел. Диагноз по профилю «форма следует функции»: перекос 163591 drone / 16729 no-drone (≈9.8:1) — это не датасет для боевой оценки, это генератор ложного оптимизма по accuracy. Открываю обсуждение.

[Структурщик]: Начну с термина, иначе спор поплывёт. «Тест-сплит» — это две разные леммы, которые здесь слиплись: **train-распределение** (что скармливаем при обучении/калибровке пресета) и **eval-распределение** (на чём меряем цифру прогона). ADR-0006 закрыл train: пресет калиброван. Мы решаем **только eval**. И это ключ: eval-состав не обязан повторять природную частоту дрона — он обязан отражать **операционную ценность ошибок**. Смешивать эти две леммы — та же ошибка, что «code-review» как омоним 16.07.

[Верстальщик]: С точки зрения потребителя цифры — а потребитель это оператор и человек в отчёте прогона — перекошенный сплит рождает **мёртвую дверь на экране**. Accuracy 94% на 9.8:1 выглядит как зелёная галочка, а за ней детектор, который просто чаще говорит «дрон». По моей норме честности: цифра, которую нельзя показать рядом с её базовой ставкой, — это враньё в UI. Мне для отчёта нужны **две раздельные оси**: P_d (обнаружение) и P_fa (ложная тревога), а не одно слитое число.

[Математик]: Формализую, иначе останется вкусовым. Accuracy на 9.8:1 имеет тривиальный baseline: классификатор «всегда drone» даёт 90.7% точности, ничего не детектируя. Это `n=1`, выданный за качество. Значит accuracy **дисквалифицирована** как главная метрика по определению. Правильная ось: детектор — это система с рабочей точкой, и её поведение в бою описывается парой `(P_d, P_fa)` на фиксированном пороге плюс интегральная AUC. По составу: eval-сплит должен быть **сбалансирован 1:1** (drone:no-drone), потому что только на балансе P_fa измерима с приемлемой дисперсией — при 16729 no-drone у нас достаточно негативов для оценки P_fa с узким CI, а позитивы подсэмплируем до равного числа.

[Музыкант]: Поддержу с живой стороны — я на перекосе обжигался в `ritual-inputs-echo`: три отражения одного снимка это `n=1`, а не `n=3`. Тут та же болезнь по-другому: 163591 drone могут быть коррелированы — один дрон, много окон одной записи. Если мы возьмём природную частоту, мы не только перекосим класс, мы ещё и раздуем **псевдо-повторами**. Балансный сплит 1:1 без утечки записей между train/eval — вот честный слух. И флаг: no-drone класс должен быть **разнообразным по фону** (ветер, город, тишина), иначе P_fa мерим в стерильной комнате.

[Teamlead]: Принимаю оба вектора. Структурщик развёл train/eval — это несущая: мы не трогаем ADR-0006, мы задаём eval. Математик дисквалифицировал accuracy численно — согласен, это не мнение, это baseline-факт. Фиксирую форму решения: **eval-сплит 1:1**, главная метрика — **пара (P_d, P_fa) на калиброванном пороге + ROC-AUC как интегральная характеристика**. Но у меня вопрос к гейту: 85/90 — это по каким осям? Пока не свяжем цифру с гейтом, вердикт неполный (rt-6 нас ловил на непокрытии повестки).

[Структурщик]: По гейту — здесь нельзя допустить омоним «85/90». Из контекста и `WHITE_PAPER.md` §8 гейт 1→2 — это порог качества детекции на узле. Я утверждаю жёстко: 85/90 обязаны читаться как **два разных предиката на двух осях**, не как одно число. Предлагаю определение: гейт пройден, если `P_d ≥ 0.90` **при** `P_fa ≤ 0.15` (то есть 85 = минимум по «специфичности» 1−P_fa, 90 = минимум по recall). Иначе «90% точности» снова схлопывается в accuracy и мы возвращаемся к вранью.

[Верстальщик]: Со стороны читаемости отчёта поддержу связку Структурщика и добавлю конструкцию экрана. Отчёт прогона по DESIGN.md — карточка с **двумя progress-барами** (`P_d` и `1−P_fa`), `tabular-nums`, `NN %`, и явная подпись базовой ставки сплита «50/50 balanced». Гейт — два бейджа: `P_d ≥ 90` (warning→success) и `P_fa ≤ 15` (warning→success). Один зелёный без другого — это `badge badge-warning`, не success. Так оператор физически не сможет прочитать «прошли», если прошла только одна ось.

[Математик]: Уточню численную сторону гейта, иначе цифры повиснут. `P_d ≥ 0.90` при `P_fa ≤ 0.15` — это точка на ROC, и я поддерживаю, но добавлю обязательный **доверительный интервал**. При balanced eval с N позитивов и N негативов CI по Уилсону для P_d ширины ±0.02 требует N ≈ 900 на класс. У нас 16729 no-drone — с запасом; drone подсэмплируем до ~16729 с деконтаминацией по записям (флаг Музыканта). Вердикт должен требовать: цифра прогона = `(P_d, P_fa)` **с 95% CI**, гейт засчитан только если **нижняя граница CI** P_d ≥ 0.90 и **верхняя граница** P_fa ≤ 0.15. Это защита от `n=1`-оптимизма.

[Музыкант]: Принимаю CI-условие, оно честное. И закрою свою зону: деконтаминация по записям — это не деталь, это несущая. Если окна одной полевой записи попали и в train, и в eval, P_d раздувается искусственно — детектор «узнаёт» запись, а не дрон. Требую в вердикт: **split по source-record, не по окну**, origin-hash записи не пересекает границу train/eval. Это ровно тот принцип декоррелированности, что мы ратифицировали для ансамблей.

[Teamlead]: Всё сходится, свожу к вердикту. Никаких «зависит»: eval-сплит 1:1 balanced, split по source-record, метрика — `(P_d, P_fa)` с 95% CI + ROC-AUC, гейт по двум осям с проверкой границ CI. Accuracy изгнана как главная метрика — остаётся только справочно рядом с baseline. LGTM формы. Структурщик, зафиксируй адрес хранения eval-манифеста; Математик — владелец метрики.

[Структурщик]: Адрес фиксирую по норме «граница пакета = граница словаря». Eval-манифест (список source-record id в сплите + seed подсэмплирования) живёт рядом с DADS-датасетом как версионируемый артефакт, чистая функция `buildEvalSplit(dataset, seed) → {records[], balance}` — в детекторном сервисе метрик (`detector-metrics-characterization`, #565), не в UI и не в apps/client. Отчёт потребляет результат через порт, не считает сам. Так цифра воспроизводима: тот же seed + тот же source-set = та же цифра.

[Верстальщик]: Принимаю. С витрины закреплю DoD-часть: отчёт не показывает единый «score», показывает две оси + AUC + подпись «balanced 50/50, split by record, seed=NN». Мёртвых дверей нет: если CI широкий (мало данных) — бейдж `badge badge-ghost` «недостаточно данных», не фальшивый зелёный.

[Математик]: Принимаю. Владельцем метрики беру: чистая функция `evaluate(detector, evalSplit) → {pd, pfa, pdCI, pfaCI, auc}`, без DOM, покрыта юнит-тестами на синтетике с известным ROC. Гейт — отдельный предикат `gatePassed(metrics) → bool` по границам CI.

[Музыкант]: Принимаю. Добавлю в DoD слуховую страховку: no-drone класс eval должен покрывать ≥3 категории фона (природа/город/тишина), иначе P_fa нерепрезентативна — помечу как условие валидности сплита, проверяемое на этапе `buildEvalSplit`.

[Teamlead]: Консенсус полный. Вердикт принят всеми пятью. Оформляю.

## Итоговое решение консилиума

| Вопрос | Решение |
|--------|---------|
| Классовый состав eval-сплита | **1:1 balanced** (drone:no-drone), подсэмплирование drone до ≈16729 с фиксированным seed |
| Что при этом train | Не трогаем — закрыто ADR-0006 (калиброванный пресет); решаем **только eval** |
| Главная метрика | **Пара (P_d, P_fa)** на калиброванном пороге + **ROC-AUC**; каждая с **95% CI** |
| Судьба accuracy | Дисквалифицирована (baseline «всегда drone» = 90.7%); остаётся только справочно рядом с base-rate |
| Защита от утечки | **Split по source-record** (origin-hash записи не пересекает train/eval), а не по окну |
| Разнообразие негативов | no-drone eval покрывает **≥3 категории фона** (природа/город/тишина) |
| Гейт 85/90 | Два предиката на двух осях: **нижняя граница CI P_d ≥ 0.90** И **верхняя граница CI P_fa ≤ 0.15**; один без другого = не пройдено |
| Где живёт | `buildEvalSplit(dataset, seed)` + `evaluate()` + `gatePassed()` в `detector-metrics-characterization` (#565); отчёт — через порт |

**Definition of Done:**
- [ ] Чистая функция `buildEvalSplit(dataset, seed) → {records[], balance}` в сервисе метрик; split by source-record, без пересечения origin-hash между train/eval.
- [ ] Валидация сплита: баланс 1:1 (±допуск), no-drone покрывает ≥3 категории фона.
- [ ] Чистая функция `evaluate(detector, evalSplit) → {pd, pfa, pdCI, pfaCI, auc}`, юнит-тесты на синтетике с известным ROC.
- [ ] Предикат `gatePassed(metrics) → bool`: нижняя граница CI P_d ≥ 0.90 И верхняя граница CI P_fa ≤ 0.15.
- [ ] Eval-манифест (record id + seed) — версионируемый артефакт рядом с DADS; воспроизводимость: seed + source-set → та же цифра.
- [ ] Отчёт (DESIGN.md): две оси (`P_d`, `1−P_fa`) + AUC, `tabular-nums`, подпись «balanced 50/50, split by record, seed=NN»; два бейджа гейта; `badge-ghost` при широком CI — без фальшивого зелёного.
- [ ] Accuracy показывается только справочно рядом с baseline «всегда drone».
