# Weekly-отчёт — `services-api-contract-drift` — неделя 2026-30

## Находки

- **Дублирующиеся deprecated-контракты в `fft-analyzer`.** В публичном API одновременно живут актуальный слой (`FftAnalyzer`, `useFftAnalyzer`, `useFftFileAnalyzer`, `useFftMicrophoneAnalyzer`) и три пары legacy-экспортов (`AudioAnalyzer`, `useAudioAnalyzer`, `useFileAnalyzer`, `useMicrophoneAnalyzer`). Drift накапливается: старый и новый контракты не сведены к единому, срок удаления не зафиксирован.
- **Отсутствие маркера версии/удаления у `@deprecated`.** JSDoc-пометки не содержат целевой версии выпила и since-метки, поэтому автоматический дифф контрактов не может отличить «плановый вывод» от «случайного дрейфа».
- **Асимметрия глубины экспортов.** `fft-analyzer` реэкспортирует внутреннюю математику полным списком (`FftCore`, `spectralFluxL2`, `SPECTRAL_FLUX_BYTE_SCALE`, `SPECTRAL_FLUX_L2_DIVISOR` и др.), тогда как `audio-engine` держит поверхность узкой. Гранулярность публичного API между двумя связанными сервисами расходится.
- **Экспорт числовых внутренних констант как части контракта.** `SPECTRAL_FLUX_BYTE_SCALE`, `SPECTRAL_FLUX_L2_DIVISOR`, `FADE_OUT_FLOOR_GAIN` вынесены в публичную поверхность; изменение значения/имени будет считаться breaking-change без явной на то причины.
- **Конфигурация дефолтов разнесена по разным именам.** В `audio-engine` — `DEFAULT_LIVE_CAPTURE_CONFIG`, в `fft-analyzer` — `DEFAULT_CONFIG`/`PRESETS`. Норматив v0.1 (`fftSize=2048`, `overlap=50%`) завязан на согласованность этих дефолтов между сервисами, но контракт не гарантирует их синхронность.

## Риски

- **Скрытый breaking-change для внешних потребителей.** Так как сервисы собираются в `dist/` для потребления вне монорепо, любое молчаливое переименование экспортированной математики/константы ломает внешних клиентов без сигнала в CI.
- **Затягивание удаления deprecated → эрозия слоёв.** Пока legacy-хуки (`useAudioAnalyzer` и пр.) остаются в `index.ts`, новый код может продолжать на них завязываться, увеличивая объём будущей миграции.
- **Рассинхрон нормативов захвата.** Расхождение между `DEFAULT_LIVE_CAPTURE_CONFIG.bufferSize` и `DEFAULT_CONFIG.fftSize` нарушит цепочку `audio-engine → fft-analyzer` (требование v0.1 = 2048) и даст тихий дрейф качества анализа перед полевыми тестами.
- **Широкая поверхность математики = хрупкий контракт.** Чем больше чистых функций реэкспортировано, тем выше вероятность, что рефактор внутренней реализации регистрируется дифф-инструментом как contract drift и порождает шум/ложные срабатывания.

## Рекомендации

- **Ввести since/removal-метки в `@deprecated`.** Формат `@deprecated since 0.x, remove in 0.y — use <X>`; тогда drift-джоба сможет классифицировать плановые выводы и алертить только на незапланированные изменения.
- **Зафиксировать окно удаления legacy-слоя `fft-analyzer`.** Назначить релиз для выпила `AudioAnalyzer` и трёх устаревших хуков, добавить lint-правило, запрещающее новые импорты deprecated-символов внутри монорепо.
- **Разделить «стабильный» и «внутренний» экспорт.** Внутренние константы (`SPECTRAL_FLUX_*`, `FADE_OUT_FLOOR_GAIN`) и низкоуровневую математику перенести в отдельный subpath-экспорт (`/internal`) или пометить как unstable, чтобы их изменения не считались breaking для основного контракта.
- **Добавить контракт-тест на синхронность дефолтов захвата.** Тест, проверяющий `DEFAULT_LIVE_CAPTURE_CONFIG.bufferSize === DEFAULT_CONFIG.fftSize === 2048` и `overlap = 50%`, чтобы норматив v0.1 держался автоматически.
- **Снять снапшот публичного API обоих сервисов.** Зафиксировать `index.ts`-экспорты `audio-engine` и `fft-analyzer` как baseline (api-extractor / snapshot-тест), чтобы дальнейший drift выявлялся построчно в PR, а не постфактум.
