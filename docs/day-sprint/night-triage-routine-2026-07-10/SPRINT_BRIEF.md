# SPRINT BRIEF · 2026-07-10 · Пилотный routine «Ночной триаж реестра задач»

Статус: seed для консилиума · Автор постановки: techies68 + Claude (Project «Membrana»)
Формат: бриф спринта; после прохождения — фиксация решений в `INSIGHT_2026-07-10_NIGHT_TRIAGE_ROUTINE.md`

---

## 1. Тезис

**Заменить самодельную ночную обвязку готовым механизмом Claude Code Routines на одной read-mostly задаче — триаже структурного долга реестра задач.** Это не новая инфраструктура (не конфликтует со stage-gate-дисциплиной), а замещение существующего звена: ghost-задачи и сироты без tracker ID уже диагностированы, ghost-closure prompt уже написан; ночной ран превращает разовую уборку в регулярный контроль.

## 2. Формализация задачи

- **Сигнал:** расхождения между `docs/tasks/registry.json`, состоянием GitHub Issues и архивом задач.
- **Шум:** легитимные переходные состояния (задача закрыта вчера вечером, batch `task:close-github` ещё не прогнан).
- **Канал:** один облачный ран Claude Code в сутки; полоса ограничена квотой Pro (раны делят лимит с интерактивными сессиями, 5 ранов/сутки максимум).
- **Приёмник:** репозиторий (ветка `claude/night-triage-YYYY-MM-DD` + draft PR). Инвариант «sink not source» соблюдён: отчёт — производная от состояния репо и трекера, не источник.

## 3. Фиксированные решения (вход спринта)

1. **Триггер:** расписание, еженощно, окно 02:00–04:00 местного времени (стаггер допустим, точность не критична).
2. **Права:** push только в `claude/`-ветки (дефолт не отключаем). Рутина **не** закрывает issues, **не** правит `registry.json`, **не** пишет в Linear на пилоте.
3. **Выход:** один markdown-отчёт `docs/reports/night-triage/NIGHT_TRIAGE_YYYY-MM-DD.md` в ветке + draft PR. PR — одновременно уведомление и точка ревью.
4. **Классификация находок** (три категории, по одной таблице на каждую):
   - ghost — активная задача, делящая GitHub Issue с архивным сиблингом;
   - orphan — активная задача без tracker ID;
   - stale — декомпозированная задача без движения (без коммитов/комментариев) дольше N дней; N — открытый вопрос, стартовое значение 14.
5. **Формат строки отчёта:** id задачи · категория · ссылка на issue · рекомендуемое действие (close / relink / re-scope) · уверенность (high / low). Рекомендация ≠ действие: исполняет человек.

## 4. План спринта (3 шага, шаг ≈ один артефакт)

**Шаг 1 — верификация канала (гейт: эмпирическая проверка перед фиксацией зависимости).**
Открыть claude.ai/code/routines, убедиться: research preview доступен на аккаунте, репозиторий `officefish/Membrana` подключается, клон видит `docs/tasks/registry.json`. Разовый ручной ран с тривиальным промптом («перечисли активные задачи из registry.json»). Артефакт: факт успешного рана + замер расхода квоты одного рана.
*Стоп-условие: если preview недоступен или квотный расход одного рана съедает заметную долю пиковой полосы — пилот замораживается, фиксируется в INSIGHT, остаёмся на локальном night:open.*

**Шаг 2 — промпт рутины (артефакт: текст промпта, приложение A как черновик).**
Довести промпт до самодостаточности: ран не имеет памяти и человека в контуре, вся дисциплина — в тексте. Проверить, что registry.json не содержит секретов (условие нормализационной границы — блокирующее).

**Шаг 3 — боевой ран и приёмка.**
Ручной запуск полного промпта → ревью отчёта утром → включение расписания только после прохождения критериев приёмки.

## 5. Критерии приёмки пилота

- Отчёт сгенерирован, лежит в правильном пути, draft PR открыт.
- **Ноль модифицирующих действий** вне отчёта (проверка diff PR: ровно один новый файл).
- Полнота: находки рутины ⊇ находки ручного аудита (`yarn issues:audit` как эталон на одном срезе).
- Точность: доля ложных ghost/orphan в отчёте ≤ 20% на первом ране (иначе итерация промпта, не расписания).
- Квотная стоимость рана измерена и записана в INSIGHT.

## 6. Не-цели

- Не строим Linear-агента (отложен до прохождения single-node stage-gate — без изменений).
- Не переносим session collector в облако: ему нужна локальная файловая система, рутины её не видят.
- Не даём рутине право закрывать/менять задачи ни на каком горизонте пилота.
- Не подключаем Slack/Linear-доставку до стабилизации отчёта (двухуровневая доставка — кандидат на фазу 2).

## 7. Риски и митигация

| Риск | Митигация |
|---|---|
| Квотная конкуренция с пиковой полосой | Один ран/ночь; замер на шаге 1; стоп-условие явное |
| Ложные срабатывания на переходных состояниях | Правило в промпте: изменения младше 24 ч не классифицировать |
| Дрейф промпта (ран без человека в контуре) | Промпт версионируется в репо; правки — только через PR |
| Секреты в клоне | Проверка registry.json и путей чтения на шаге 2; блокирующее условие |
| Research preview: API и лимиты могут измениться | Пилот не встраивается в критический путь; отказ дешёвый |

## 8. Открытые вопросы (на консилиум)

2. Судьба отчётов: накапливать в `docs/reports/night-triage/` или архивировать по регламенту `archive:daily-day`?

## 9. Приложение A — черновик промпта рутины

```
Ты — ночной аудитор реестра задач проекта Membrana. У тебя один ран,
без памяти о прошлых ранах и без человека в контуре.

Вход:
- docs/tasks/registry.json (канонический реестр)
- docs/tasks/README.md (регламент active/archive)
- открытые GitHub Issues репозитория (через gh CLI)

Задача — найти три класса расхождений:
1. ghost: активная задача, чей GitHub Issue разделён с архивной задачей.
2. orphan: активная задача без tracker ID (нет ни Issue, ни Linear ID).
3. stale: активная задача без коммитов и комментариев более 14 дней.

Правила:
- Изменения младше 24 часов не классифицируй — это переходные состояния.
- Ты НЕ закрываешь issues, НЕ редактируешь registry.json, НЕ пишешь
  во внешние сервисы. Единственное изменение — один новый файл отчёта.
- Если данных недостаточно для уверенной классификации, помечай
  уверенность low и объясняй, чего не хватает, одной строкой.

Выход:
- Файл docs/reports/night-triage/NIGHT_TRIAGE_<YYYY-MM-DD>.md.
- Структура: краткая сводка (3 строки: сколько ghost/orphan/stale),
  затем три таблицы: id · категория · ссылка · рекомендуемое действие
  (close / relink / re-scope) · уверенность (high/low).
- Закоммить файл в ветку claude/night-triage-<YYYY-MM-DD>,
  открой draft PR с заголовком "Night triage <YYYY-MM-DD>".
- Если расхождений нет — отчёт всё равно создаётся, с пустыми
  таблицами и строкой "реестр чист".
```
