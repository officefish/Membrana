# Weekly-отчёт: `services-api-contract-drift` — неделя 2026-28

## Находки

- **Дублирование дефолтных конфигов между слоями капчи.** Норматив v0.1 требует `bufferSize = 2048` одновременно в `LiveCaptureConfig.bufferSize` (audio-engine, `DEFAULT_LIVE_CAPTURE_CONFIG`) и `AudioAnalyzerConfig.fftSize` (fft-analyzer, `DEFAULT_CONFIG`). Значение продублировано в двух пакетах без единого источника истины — классический дрейф-риск: обновление одной константы не гарантирует синхронизацию второй.

- **Накопление deprecated-поверхности в публичном API `fft-analyzer`.** В `index.ts` одновременно экспортируются старый класс `AudioAnalyzer` и три устаревших хука (`useAudioAnalyzer`, `useFileAnalyzer`, `useMicrophoneAnalyzer`) рядом с их актуальными заменами (`FftAnalyzer`, `useFftAnalyzer`/`useFftFileAnalyzer`/`useFftMicrophoneAnalyzer`). Контракт раздвоен: два поколения API живут параллельно без пометки о дате удаления.

- **Отсутствие константы/дефолта в контракте у ряда сервисов.** SERVICES.md фиксирует обязательную категорию «Константы/defaults» в публичном API. `audio-engine` экспортирует только `DEFAULT_LIVE_CAPTURE_CONFIG`; наличие `PRESETS`/`applyPreset` присутствует лишь в `fft-analyzer`. Единообразие defaults-контракта между foundation и analyzer не гарантируется.

- **Смешение математики и engine-хелперов в едином barrel.** `fft-analyzer/index.ts` реэкспортирует и `src/math/*` (`FftCore`, `spectralFlux*`, `statistics`), и engine-класс `FftAnalyzer` из `src/core/`. Публичная точка входа не разграничивает pure-математику и слой с побочными эффектами — потребитель может импортировать math-примитивы, минуя предполагаемый API-контракт.

## Риски

- **Тихий рассинхрон параметров захвата (sampleRate/fftSize/overlap).** Раздельно заданные `bufferSize` и `fftSize` могут разъехаться при правке одного пакета; fallback `48000 → 44100` без синхронной пометки в telemetry ломает воспроизводимость анализа. Средний-высокий риск для полевых тестов Этапа 1.

- **Ломающие изменения при удалении deprecated-хуков.** Внешние потребители (за пределами монорепо, ради которых собирается `dist/`) не имеют сигнала о версии удаления `AudioAnalyzer`/старых хуков. Их вырезание станет неконтролируемым breaking change.

- **Утечка внутренних math-примитивов в стабильный контракт.** Широкий реэкспорт `math/*` фиксирует внутренние функции как публичное API де-факто; их рефакторинг превратится в contract drift даже без изменения «главного» класса.

## Рекомендации

- **Ввести единый источник истины для параметров захвата.** Вынести `sampleRate`/`bufferSize`/`fftSize`/`overlap` в общий контракт (в `@membrana/core` или разделяемую константу), а `DEFAULT_LIVE_CAPTURE_CONFIG` и `DEFAULT_CONFIG` вычислять из него, исключив ручное дублирование `2048`.

- **Задать deprecation-политику с версией удаления.** Каждому `@deprecated`-экспорту добавить целевую версию/спринт удаления и запись в CHANGELOG сервиса; добавить lint-правило, запрещающее новые импорты устаревших символов.

- **Разграничить публичный контракт по слоям.** Рассмотреть выделение math-примитивов в отдельный подэкспорт (`@membrana/fft-analyzer-service/math`) или пометку «internal», чтобы стабильный API ограничивался классом, хуками, типами и defaults согласно SERVICES.md.

- **Добавить контрактный тест на synced-defaults и обязательные категории API.** CI-проверка: у каждого сервиса присутствуют все четыре категории (ядро/типы/хуки/defaults) и совпадают ключевые параметры захвата между audio-engine и fft-analyzer.

- **Ввести проверку telemetry-пометки fallback sampleRate.** Гарантировать, что при переходе на 44100 Гц запись явно маркируется, — покрыть тестом на уровне engine, чтобы дрейф норматива v0.1 отлавливался автоматически.
