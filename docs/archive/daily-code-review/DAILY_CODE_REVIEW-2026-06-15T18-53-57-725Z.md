# Code Review: Состояние репозитория на 2026-06-15

## [Teamlead — Vesnin]

**Стратегический обзор:**
Виден мощный спринт по закрытию эпика **Telemetry Journal UX hardening (#81)** с полным цеклом TJ0–TJ10. Архитектурно команда придерживается слабой связанности: новые сервисы (`telemetry-journal`, `detector-report`) встраиваются через публичные контракты, а не прямыми импортами.

**Риски границ пакетов:**
1. **background-cabinet ↔ telemetry-journal**: В коммите `ea0185a` упоминается sync-бекенда — нужно убедиться, что кабинет вызывает журнал только через публичный API (`SERVICES.md`), а не через приватный state.
2. **detector-report в Docker-контексте** (коммиты `d5102e2`, `7e0b1d6`): Если `detector-report` потянул зависимость на `detector-base`, а тот не был в образе — это указывает на *пропущенный синтез* в `ARCHITECTURE.md`. Проверить, не нарушена ли переносимость на отдельный хост.
3. **client → telemetry-journal**: UI в `e9e4f65` создаёт карточки треков и отчётов. Убедиться, что компоненты (`TJ5 live journal UI`) не содержат бизнес-логику; вся трансформация должна быть в сервисе-фасаде.

**Проверить по ARCHITECTURE.md:**
- [ ] Контракты новых сервисов (TJ2 cabinet sync) задокументированы в `SERVICES.md` v-последний.
- [ ] Цепочка импортов `client → services → cabinet` не содержит циклов.

**LGTM-статус:** Ждём ревью Структурщика по слабой связанности и Математика по качеству DSP.

## [Математик — Dynin]

**Качество DSP-ядра:**

Видны два алгоритмических слоя:

1. **Detector algorithms (FFT, spectral-flux, drone):**
   - Коммит `03ebc81` (VDR5): `template-match FFT trends` — требует документации на функцию сигнатуры (вход: `Float32Array`, выход: `{freq, confidence}`).
   - Коммит `5c79cc1` (SLD3): `cepstral and spectral-flux on free-v1` — свежие функции, нужны unit-тесты на граничные случаи (тишина, импульсный шум, клиппинг).

2. **Calibration (VDR4, SLD4):**
   - Коммит `f7c433c` (VDR4): `calibrate DSP on curated ground truth` — алгоритм кальбровки не виден в коммит-сообщении. Требуется explicit функция `calibrateDetector(groundTruth: AnalysisSample[]): CalibrationParams`.
   - Коммит `9da3e25` (SLD4): Tooling есть, но надо убедиться, что нет *hardcoded magic numbers* в параметрах.

3. **Тестовые данные (VDR):**
   - Коммиты `4a07007`, `2e249d5`: Ground truth экспортируется в `benchmark manifest`.
   - ⚠️ **Проверить:** Есть ли версионирование датасета? Если параметры кальбровки плывут, это признак нестабильности.

**Требуемые артефакты:**
- [ ] В `packages/services/detectors/` должны быть `*.spec.ts` для каждого детектора (FFT, spectral-flux, drone) с edge cases: пустой буфер, DC смещение, Nyquist clipping.
- [ ] Функция кальбровки должна быть *чистой* (вход: массив сэмплов + ground truth labels, выход: params, без I/O).
- [ ] Документация на `analyzeSample` контракт в `SERVICES.md`.
