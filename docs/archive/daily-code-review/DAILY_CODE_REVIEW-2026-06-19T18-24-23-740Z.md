1. **Граница плагин ↔ сервис**: В истории коммитов видны обновления `device-board` как клиентского компонента, но нет явной информации о том, были ли **новые сервисы** созданы или использованы только существующие.    - Все потоковые вычисления должны идти через **сервис-фасад** (в `packages/services/`), не напрямую в компоненте.

3. **Mintlify docs Phase 0**: Документация должна отражать **публичный контракт** модуля, а не реализацию. Если doc-PR содержит примеры кода — они должны соответствовать `DESIGN.md` (если UI) или `SERVICES.md` (если сервис).

**Анализ UI:**

Основные коммиты указывают на значительную переработку layout'а и UX:
- Header layout
- Typed ports UI
- Variable inspector  
- Edge styles
- Runtime port inspector с Print-фичей

**Проблемы и требования:**

1. **Design compliance**: Все эти компоненты должны соответствовать `DESIGN.md` по:
   - Типографике (шрифты, размеры, вес)
   - Цветовой палитре (light/dark mode)
   - Отступам и сетке (grid-based layout)
   - A11y (aria-labels, keyboard navigation, contrast)

2. **Отсутствие видимого PR review по DESIGN.md**: В истории не видно явной ревью верстальщика. Риск: UI может содержать не-консистентные иконки, неправильные отступы, нарушения темизации.

3. **Print-фича**: Если это CSS `@media print` — нужно тестировать в browser dev-tools. Если это экспорт в PDF/PNG — требует отдельного модуля, который **не должен жить в компоненте** (нарушение SRP).

4. **Variable inspector overlay**: Сложный компонент с вложенностью. Рекомендуется разбить на атомы (card, badge, input) по `DESIGN.md`.
