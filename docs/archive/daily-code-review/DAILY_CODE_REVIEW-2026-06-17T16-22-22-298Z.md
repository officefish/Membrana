## [Структурщик — Ozhegov]

**Проверка модульных границ:**

1. **Проблема: циклические зависимости (риск высокий)**
   ```
   client → telemetry-journal → electron-journal-port
   client → resolveJournalBackend (выбирает бэкенд)
   ```
   - `resolveJournalBackend.ts` не должен импортировать конкретные реализации; нужен **фасад** (сервис-локатор или IoC контейнер).
   - Рекомендация: переместить логику в `packages/services/journal-router/` с единственной ответственностью: "выбрать правильную реализацию по ENV".

2. **Новый слой (Electron Desktop):**
   ```
   membrana-studio/
   ├── journal/ (своя file-system реализация)
   ├── media-library/ (своя file-system реализация)
   └── trends/ (чей??)
   ```
   - Дублирование бизнес-логики из `background-media` + `background-cabinet`.
   - Предложение: **не** создавать отдельные store'ы, а использовать `@membrana/services` как монолит для обоих web и desktop.
   - Если Electron должен быть офлайн-first, это требует явного контракта (CQRS? Event Sourcing?).

3. **Положительное:**
   - `telemetry-journal` остаётся чистым (не видит UI);
   - `device-board` изолирован (только DI в cabinet'е).

**Действия:**
- [ ] Создать `packages/services/journal-router/index.ts` → фасад с резолюцией по `JOURNAL_BACKEND`;
- [ ] Убрать прямые импорты из `client` в конкретные реализации;
- [ ] Обсудить с **Teamlead**: membrana-studio использует общие services или пишет своё?
