## Проблема / наблюдение

В команду вводится нетехнический партнёр по внешним коммуникациям (презентации, лендинги,
брендбук, инфографика). Ему нужен полный контекст проекта, но без доступа к продуктовому коду
на запись. Открытый вопрос брифа: **где живёт контур communications** — изолированным
воркспейсом внутри монорепо (A), отдельным репозиторием с импортом канона (B) или гибридом.

Два инварианта держат конструкцию:
1. **«Сток, а не исток»** — контур читает много, пишет мало; пишет только в свой каталог; ни один
   продуктовый пакет от него не зависит.
2. **Single source of truth** — факт рождается в одном месте; контур — потребитель истины, не
   второй её источник.

## Гипотеза → Решение

**Вариант A** (leaf-воркспейс `apps/comms-studio` в монорепо) сохраняет живое чтение канона при
нулевой поверхности синхронизации и выполненном инварианте — и потому канонически предпочтителен
по критерию §5 брифа. Первый прогон консилиума (на `claude-haiku-4-5`) ошибочно выбрал B/гибрид,
переиграв критерий §5 организационным доводом; повторный прогон на `claude-opus-4-8` пришёл к A
единогласно (24 реплики).

### Два кристаллизующих тезиса

1. **Изоляция по записи, не по чтению.** Инвариант «сток, не исток» держится запретом *исходящих*
   рёбер к контуру (`yarn check:boundaries`: `indegree(comms) = 0` от продуктовых пакетов), а не
   запретом чтения. Контур свободно читает весь актуальный контекст; писать может только в свой
   каталог. Изоляция — по направлению стрелки зависимости, не по физическому разделению репо.

2. **Чтение ≠ импорт.** Канон и живые исходники читаются через `fs-read` рабочей копии, **не**
   через `import @membrana/*`. Импорт создал бы ребро зависимости → блок границ. Поэтому
   синхронизации нет по построению: один источник, копии не создаётся → расщеплять нечего
   (§2 single source of truth выполнен тривиально).

## Обоснование через критерий §5

> «Какая топология сохраняет живое чтение канона при минимальной поверхности синхронизации и при
> выполненном инварианте?»

| Топология | Синхронизация `S` | Обслуживание `M` | Вердикт |
|-----------|-------------------|------------------|---------|
| **A (монорепо leaf)** | `S = 0` — нулевой лаг, нулевой рукотворный шаг | ≈ 0 (без пакета/паблишинга/второго репо) | ✅ выбран |
| B (отдельный репо, авто-зеркало) | `S = 0` только при авто-зеркале | `M > 0` (зеркало + лаг + риск ручной правки копии) → «дешёвая» не выполнена | оговорка §5 «B может перевесить» **не активируется** |
| B (отдельный репо, ручной перенос) | `S > 0` — ручной шаг | `M > 0` | ❌ проваливает §5; воспроизводит «Claude Project + Drive», ради обхода которого контур строился (§3) |
| Гибрид | несёт `M` монорепо **и** `M` зеркала | максимум | ❌ худшее из двух миров (§71) |

**Развилка факта (read-доступ) растворена:** read-доступ — это про **агента**, не про партнёра.
Агент читает продуктовый исходник (в этом весь смысл §3); партнёр git-доступа не имеет **ни в одной**
топологии — он касается контура тонким срезом (монитор каталога `out/**` + Drive). Значит вопрос
«доверенный ли партнёр» топологию не решает: партнёра в git нет по построению.

**Подтверждение из `CANON_MINIMUM` (§59–61):** Слой 1 (носитель истины — WHITE_PAPER, ARCHITECTURE,
INTEGRATIONS_STRATEGY, SINGLE_NODE_DETECTION_FIRST) **дрейфует**; именно на нём тестируется
односторонность и дешевизна синхронизации B. Вариант A читает Слой 1 живьём → тест не возникает.

## Механизмы изоляции (реализация для A)

- **Граф:** `apps/comms-studio` — leaf; `dependencies` без `@membrana/*`; чтение канона через
  `fs-read` рабочей копии, не через `import`.
- **Границы:** `yarn check:boundaries` проверяет `indegree(comms от продуктовых) = 0` и
  `outdegree(comms к продуктовым) = 0` автоматом (позитивный + негативный тест инварианта).
- **CODEOWNERS:** `apps/comms-studio/**` → comms-owner; продуктовые пути → инженеры.
- **CI:** отдельная полоса через `paths`-фильтры workflow (Turbo scope + path filters); продуктовый
  `lint typecheck test build` не триггерится правкой контура и наоборот.
- **Секреты:** изолированный `apps/comms-studio/.env`, вне продуктового скоупа; вычистка секретов —
  блокирующее условие перед архивом (канон проекта).
- **Партнёр:** тонкий срез — монитор каталога `apps/comms-studio/out/**` + Drive для ручной правки;
  прямого git нет.
- **Канон коммуникаций** (тон, dual-use, запрещённые слова, лист фактов, глоссарий) применяется к
  выходу контура на этапе рендера — инструмент рендерит канон, не отменяет его.

## Non-goals (что контур НЕ делает)

- Не деплоит продакшен-лендинг `membrana.space` (решение о хостинге отложено stage-gate, §85; не
  смешивать с решением об авторстве контура).
- Не является вторым источником истины — только потребитель канона (§2).
- Не сажает партнёра в продуктовый git и не даёт запись за пределами своего каталога.
- Не импортирует `@membrana/*` пакеты (чтение ≠ импорт).

## Точка дрейфа

Единственная — **кэш агента**. Блокируется чтением рабочей копии на каждый прогон (ритм `git pull`
перед запуском агента). Копии канона не создаётся → §2 выполнен.

## Scope

- **In scope:** топологическое решение (A) + механизмы изоляции + non-goals; вход для спринта
  создания окружения партнёра.
- **Out of scope:** состав канонического минимума (см. `CANON_MINIMUM_FOR_COMMS_CONTOUR.md` →
  производные артефакты FACTS_SHEET / GLOSSARY / BRAND_TOKENS); хостинг боевого лендинга;
  multi-node и прочие продуктовые магистрали.

## Вопросы для research (Q1–Q3)

1. **Landscape:** как другие монорепо изолируют нетехнический / маркетинговый контент leaf-воркспейсом
   (Turbo/Nx path-фильтры CI, CODEOWNERS, boundary-линтеры) — паттерны и грабли?
2. **Fit (Membrana):** покрывает ли текущий `yarn check:boundaries` негативный тест (искусственное
   ребро к/от контура должно ронять сборку), или нужен доп. чек?
3. **Risk:** где ещё, кроме кэша агента, возможен дрейф истины при живом чтении, и достаточно ли
   ритма `git pull` для его закрытия?
