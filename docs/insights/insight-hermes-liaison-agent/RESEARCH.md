# Research: Hermes — вестник-лиазон между сессиями

> Источник: perplexity-api

## Q1 — Landscape

**Запрос:** Landscape: как другие агентские окружения решают cross-session state/handoff (memory-файлы, context packs, session briefs) — паттерны и анти-паттерны?

**Выжимка:**

Для кросс-сессийной передачи контекста (handoff) и сохранения памяти в агентских окружениях используется разделение **сессийного состояния** (conversational context, эфемерного и scoped к одной сессии) и **состояния рабочего процесса** (workflow state, структурированного, агентно-независимого и persisting во внешнем хранилище) [5]. Основной паттерн — хранение workflow state в task store (индексированном по ID задачи), а не в истории сообщений, где сессия лишь ссылается на задачу (например, «сессия #123 о задаче #4521») [5].

### Ключевые паттерны решения

| Паттерн | Описание реализации | Преимущества |
| :--- | :--- | :--- |
| **Workflow State Separation** | Workflow state хранится в структурированном внешнем хранилище (JSON, DB), а не в истории чата. Это позволяет агенту загружать контекст в начале сессии без реконструкции из истории [5]. | Queryable, serializable, debuggable; поддержка параллельной работы агентов над одной задачей [5]. |
| **SessionContext (Durable State)** | Использование компактной технической карточки `SessionContext` (не транскрипта), отвечающей на вопрос «в какой сцене находится пользователь», как durable-состояние диалога [1]. | Компактность, отсутствие «помойки» из полного лога, base64 файлов и traceback [1]. |
| **Client Session State (опционально)** | Сохранение данных состояния сеанса на стороне клиента; клиент передает серверу все сведения при каждом запросе [3]. | server не хранит состояние (но требует передачи полного объема данных) [3]. |
| **Server Session State** | На клиенте только ID сеанса, все данные хранятся на сервере в специальной коллекции [3]. | Передается только ID (но требует значительных ресурсов сервера) [3]. |

### Анти-паттерны (что нельзя делать)

| Анти-паттерн | Проблема | Как исправить |
| :--- | :--- | :--- |
| **Conflation (слияние)** | Слияние сессийного и workflow состояний приводит к загрязнению контекстного окна (context window pollution), broken task resumption и сбитым multi-agent handoffs [5]. | Четкое разделение: структурированный workflow state в persisting store, история чата в session store [5]. |
| **SessionContext as Dump** | Превращение `SessionContext` в «помойку» с полным prompt, base64 файлов и простыней traceback [1]. | Использовать `AgentEvent`, файловое хранилище, blob storage и job-таблицы для громоздких данных [1]. |
| **Reconstruction from History** | Попытка реконструировать контекст across sessions из истории сообщений (например, для личных ассистентов, знающих предпочтения) [5]. | Загружать workflow state из внешнего хранилища в начале сессии, а не реконструировать [5]. |
| **Loose Workflow State** | Использование неструктурированного (loose) текста для workflow state, что делает его неqueryable и неserializable [5]. | Использовать JSON, DB записи, typed objects для workflow state [5]. |

### Важные принципы

1.  **Idempotency:** Каждый шаг workflow должен быть по возможности идемпотентным (повторение не меняет результат) [5].
2.  **Agent-Agnosticism:** Workflow state должен быть агентно-независимым — любой агент в системе может читать и обновлять его без реконструкции истории [5].
3.  **Distinguishing Scope:** Сессия отвечает за диалог, проект — за рабочую область [1].
4.  **Discrete State:** Для фактических состояний создаются отдельные классы с общим интерфейсом, где поведение зависит от состояния [2].

Если ваша задача требует запоминания контекста across sessions (персональный ассистент, проект, запоминающий решения), это **workflow state**, а не сессийный; его нужно хранить externally и загружать в начале каждой сессии [5].

## Q2 — Fit (Membrana)

**Запрос:** Fit (Membrana): что даёт больше — детерминированный сбор (скрипт) или LLM-обобщение; где граница, за которой бриф начинает врать?

**Выжимка:**

Детерминированный сбор (скрипт) **даёт больше** в сценариях, где входные данные чётко определены и требуется **предсказуемость, скорость и 100% надёжность** выхода, а LLM-обобщение превосходит только в «долгих» (long-tail) случаях с неструктурированными данными или открытыми запросами, где нужна **проблемная гибкость**, но с риском галлюцинаций [1][2][4].

Граница, за которой бриф начинает «врать» (генерировать ложную информацию), — это момент, когда **пропорция неопределённости в входных данных превышает способность модели следовать явным инструкциям**, что характерно для критических сфер (здоровье, финансы), где LLM могут правдоподобно, но ложно заполнять недостающие факты [3][6].

### Ключевые различия подходов

| Параметр | Детерминированный сбор (скрипт) | LLM-обобщение |
|----------|----------------------------------|---------------|
| **Надёжность** | **Всегда одинаковый выход** на одинаковый вход; любое отклонение фиксируется тестами [2] | Риск **галлюцинаций**: модель может генерировать правдоподобную, но ложную информацию [3] |
| **Гибкость** | Работает только при **жёстко заданных правилах**; ломается при малейших изменениях интерфейса [8] | Эффективна для **открытых исследований**, неструктурированных данных и контекстно-зависимых шагов [4][8] |
| **Стоимость** | Низкая стоимость масштабирования, высокая пропускная способность (throughput) [3] | Высокая стоимость токенов, энергопотребление и задержки (latency) [3] |
| **Применение** | Суммаризация, статистика, **проверяемые данные** [6] | Интерактивная идентификация, **долгосрочные use cases** [4] |

### Где граница «вранья» брифа?

1.  **Недостаток данных**: когда в брифе не хватает ключевых фактов, LLM склонна **предсказывать** (инферировать) недостающее, создавая ложную, но правдоподобную информацию [3].
2.  **Сложность контекста**: если задача требует анализа множества разрозненных источников или тонкой эмпатии, скрипт перестает работать, а LLM может «врать» из-за **недопонимания** нюансов [4][8].
3.  **Критические области**: в сферах здоровья и финансов пользователи **не готовы полагаться** на нейросети без встроенных механизмов факт-чекинга, так как риск ошибки невозможно полностью устранить [3][6].

### Оптимальная стратегия

Для производственных задач лучше использовать **гибридный подход**:
*   **Детерминированное ядро (скрипт/DAG)** для основных шагов, где важен точный расчёт и проверка [2].
*   **LLM для узкой экстракции** (извлечения данных) с последующей **валидацией через JSON-схемы** и автоматическим повтором при сбое [4].
*   **Разделение** извлечения (LLM) и трансформации (код), чтобы предотвратить смещение названий (naming drift) [4].

Такой подход обеспечивает **95% детерминированность**, сохраняет гибкость LLM для сложных шагов и минимизирует риск генерации ложной информации [4].

## Q3 — Risk

**Запрос:** Risk: как не превратить Hermes в дубль `plan:day`/`standup` и не отрастить преждевременный оркестратор?

**Выжимка:**

Чтобы не превратить Hermes Agent в упрощённый скрипт для `day`/`standup` и не создать преждевременный оркестратор, **обеспечьте агенту согласованность в цепочке действий**, требуя от модели помнить цель, сохранять ограничения и избегать повторения неудачных шагов[1].

**Ключевые аспекты для эффективной работы:**
*   **Согласованность действий:** Модель должна сохранять ограничения задачи и «знать», когда неудачный шаг был повторён, чтобы не зацикливаться[1].
*   **Архитектурные требования:** Используйте `async/await` для всех операций с базой данных и разместите тесты в `tests/` с использованием `pytest-asyncio`[2].
*   **Безопасность:** Никогда не коммитируйте файлы `.env` в систему контроля версий[2].

Если Hermes не запоминает полную цель или ограничения, он может деградировать в простую функцию для ежедневных задач, лишаясь гибкости для сложных исследований[5]. **Секрет успеха** — в ручном управлении и отказе от массового автоматизации, где каждая «сумка» (задача) создается с учётом уникальных нюансов[6], а не как стандартный оркестратор.
