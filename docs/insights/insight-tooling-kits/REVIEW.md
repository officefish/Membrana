[Структурщик]: Архитектурно чисто. Ключевое решение — «кит = именованная декларация-запрос над живыми источниками», а не новый снимок — снимает главный риск дублирования и рассинхрона (та же болезнь, что породила TF-6). Источники (`package.json` scripts, `skills/README.md`, экспорты `scripts/lib`, `.githooks/`) уже существуют, `scripts/kit.mjs` — тонкий фильтр-рендер поверх них. Три требования к контрактам: (1) декларации китов должны жить в **одном** реестре (по аналогии с `registry.json`), а не размазываться по коду `kit.mjs` — иначе новый кит = правка скрипта; (2) поле `kit` в карточке — enum, валидируемый против списка деклараций, иначе тихий дрейф значений; (3) гейт свежести (хэш источников в шапке кита) — не дублировать логику, а переиспользовать хелпер из `tooling-doctor`. Слабая связанность соблюдена: кит не импортирует скиллы, а индексирует их. `9/10`, adopted, этап «неделя».

[Математик]: Механика версии математически корректна: версия = `git log -1 <file>` (последний коммит, канон `lastTouchedAt`), changelog = `git diff` между рендерами. Это детерминированно и проверяемо — никакого mtime. Гейт свежести — хэш источников (упорядоченный `sha256` конкатенации нормализованных списков), сравнение хэшей O(1). Тут важна **нормализация**: сортировать команды перед хэшем, иначе перестановка scripts в `package.json` даст ложное «протух». Детали чистой функции: `sourceHash(sources) → string`, `kitDiff(oldRender, newRender) → {added, removed, changed}` — обе без побочных эффектов, тестируются на фикстурах.

Детерминированный маппинг `kind + keywords → kit` — это классификатор с ручными правилами; он проверяем на golden-наборе задач (взять существующие карточки реестра как разметку). Q3 честно признал, что риски маппинга Research не покрыл — значит их надо специфицировать самим: миссмаппинг детектируется постфактум по сигналу «агент вызвал команду не из выданного кита» (лог сессии → пересечение вызванных команд с составом кита; низкое пересечение = не тот кит). Это дешёвая метрика. `7/10`, adopted, этап «неделя».
