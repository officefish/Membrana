[Teamlead]: Идея бьёт в реальную боль: консилиумы и `yarn ask` рождают персон «с холода», позиции дрейфуют, споры переигрываются. Слой 1 (память-как-файлы-в-git с человек-гейтом) — правильная граница: он наблюдаем, обратим, не тянет за собой заморожённые уроки #344 (автономность без наблюдаемости). Ключевое ценностное ядро — не журнал сам по себе, а **петля калибровки прогноз↔факт**: это то, что нельзя купить внешним знанием стека (`insight-team-stack-watch`) и что отличает нас от памяти кодинг-агента (`insight-mcp-hindsight-agent-memory`). Но объём работ реалистичен только фазами: сначала read-only инъекция дистиллята в consilium/ask, калибровку — вторым шагом. Немедленный merge не предлагаю: нужен отдельный task-промпт после MVP-контура (сам инсайт-режим и consilium уже живут). Внедрять — да, этап «месяц», старт с узкого пилота на одной персоне (Dynin — у него самые проверяемые прогнозы).

[Структурщик]: Архитектурно чисто ложится как **детерминированный read-only контур ритма** — тот же паттерн, что `hermes-brief` (PR #316): extractor читает `docs/seanses/*-PROTOCOL.md` и `docs/insights/*/REVIEW.md`, пишет `docs/virtual-team/memory/<persona>.md`, загрузчик consilium/ask инъектирует top-N с токен-бюджетом. Требования к связанности: extractor не должен знать про промпты персон, загрузчик не должен знать про формат протоколов — общий контракт через журнал-файл со строгой схемой (provenance-ссылка + дата + тип записи). Research Q2 прямо подтверждает: при журнале <5K токенов, статичном, file/line-provenance — **plain Markdown в git достаточно**, вектора не нужны. Это снимает риск раздувания зависимостей. Инкрементальный дописок после консилиума — отдельный шаг пайплайна, не сцеплять с генерацией протокола. Один сервис (`packages/services/persona-memory` или расширение office) без прямых импортов в UI.

[Математик]: Экстракция должна быть **чистой и детерминированной** — regex/тег-фильтр по имени персоны, фиксированный парсер таблиц голосований, без LLM на этапе extraction (Research Q2 явно: LLM только на summarization с фиксированным шаблоном). Это даёт воспроизводимость: одинаковый вход → одинаковый журнал, тестируемо снапшотами. Калибровочный слой — моя зона: `score = align(prediction, outcome)`, чистая функция над парами «прогноз из №N ↔ факт». Предлагаю Brier-подобную метрику, не бинарную — она непрерывна и накапливается. Ротация: `score = 0.6·recency + 0.4·importance` из Research — приемлемо как старт, но importance должен быть человек-флагом, не авто-эвристикой (иначе poisoning-риск Q3). TTL/staleness — обязателен для прогнозов (у факта появляется срок), для «позиций» — бессрочно с пометкой superseded.

[Музыкант]: Моя зона тут минимальна — аудио-контур не затронут. Единственное: если у меня как персоны появится журнал, важно, чтобы он не окостенял позицию по эффектам/алгоритмам сильнее, чем реальность DSP. Прогнозы про звук проверяются слухом и клиппингом, не голосованием, — калибровка для меня должна опираться на измеримый исход (нет клиппинга / соответствие 24bit·48kHz), а не на «я так голосовал». Поддерживаю, но приоритет для недельной стратегии низкий: это инфраструктура команды, не продуктовый звук.

[Верстальщик]: UI-поверхности пока нет и не требуется — журналы это git-Markdown, читаются в редакторе/на GitHub. Не форсировать вьюер: любой ранний UI над памятью персон — преждевременная бизнес-логика в JSX. Если позже понадобится визуализация калибровки (прогноз↔факт по персоне), это отдельный презентационный компонент по DESIGN.md поверх готового чистого API, не раньше. Сейчас участие — нулевое, возражений нет.
