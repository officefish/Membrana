# Research: Публикация документации туллинга на docs.mmbrn.tech (Mintlify custom domain)

> Источник: perplexity-api

## Q1 — Landscape

**Запрос:** как Mintlify подключает custom domain — тип DNS-записи (CNAME и его цель), верификация, автоматический SSL, ограничения по тарифу (доступен ли custom domain на текущем плане), сроки propagation; конвенции для документационного поддомена (`docs.` vs корень).

**Выжимка:**

Mintlify подключает **custom domain** через **CNAME-запись** для поддомена: в примерах из документации указывается `docs` → `cname.mintlify.builders` в китайской версии и `docs` → `cname.mintlify-dns.com.` в испанской версии, то есть цель записи — направить ваш поддомен на хост Mintlify, который обслуживает документацию.[2][3] После добавления домена в dashboard Mintlify просит настроить DNS, затем дождаться **DNS propagation** и автоматического выпуска **TLS/SSL-сертификата**.[1][2][3]

По верификации: после создания DNS-записи Mintlify показывает дополнительные **TXT**-записи для проверки владения доменом; в китайской инструкции указаны `_acme-challenge.<домен>` и `_cf-custom-hostname.<домен>`, после чего нужно нажать **Retry validation**.[2] В других Mintlify-примерах процесс также описан как ожидание propagation, затем проверка статуса/валидации в панели.[4][7][9]

По SSL: Mintlify заявляет, что **TLS certificates are automatically provisioned** после того, как DNS начнёт резолвиться; внешние примеры Mintlify-страниц уточняют, что это обычно происходит автоматически после успешной DNS-валидации.[1][3][7]

По срокам propagation: официальная базовая формулировка Mintlify — просто “allow time” для распространения DNS.[1] В сторонних Mintlify-инструкциях встречаются ориентиры **5–10 минут** и **1–24 часа**; это не противоречит друг другу, а показывает, что реальный срок зависит от DNS-провайдера и кеширования.[4][7][9]

По тарифу: в предоставленных результатах **нет прямого подтверждения**, что custom domain доступен или недоступен на вашем текущем плане; нужна проверка страницы billing/pricing или вашей dashboard-конфигурации. Из результатов видно только, что в dashboard есть отдельный workflow “Custom domain setup”, но ограничение по плану в них не указано.[1][2][3]

По конвенции поддомена документации: Mintlify в примерах явно использует **`docs.`** как типичный поддомен, и в Spanish-документации даже приводит примеры `docs.example.com` и `www.example.com`.[3] В документах и сторонних гайдах по современным хостингам чаще рекомендуют именно **поддомен** для документации, а не корень, потому что так проще настраивать DNS и не затрагивать основной сайт; для apex/root это может требовать отдельной схемы DNS у некоторых провайдеров, но в найденных Mintlify-результатах отдельного правила про root-апекс не приведено.[3][13]

## Q2 — Fit (Membrana)

**Запрос:** что именно нужно в дашборде Mintlify и в DNS `mmbrn.tech`, чтобы `docs.mmbrn.tech` указывал на наш проект; не конфликтует ли с существующими записями (`office.`); какой минимальный репо-side шаг (если есть) сверх уже влитого контента.

**Выжимка:**

Чтобы `docs.mmbrn.tech` указывал на Mintlify-проект, обычно нужны **две вещи**: в дашборде Mintlify — добавить **custom domain** `docs.mmbrn.tech`, а в DNS — создать **CNAME** для `docs`, который Mintlify покажет в настройках домена; Mintlify в документации и гайдах прямо говорит идти в **Settings / Custom Domain** или **Domain Setup** и затем обновить DNS по инструкции.[4][3][1]

По конфликту с `office.`: **не конфликтует**, если вы добавляете отдельную запись именно для поддомена `docs`. `CNAME docs -> ...` живёт независимо от `office.<ваш-домен>`, потому что это разные имена хостов; конфликт был бы только если бы вы попытались создать **две записи для одного и того же имени** `docs` или если `docs` уже был занят другой записью.[3][4]

Минимальный **repo-side шаг**, если контент уже влит, обычно такой: **ничего дополнительного в репозитории не нужно**, кроме того что репозиторий уже подключён к Mintlify через GitHub App и содержит нужные файлы, после чего Mintlify автоматически деплоит изменения из репо.[1][4][5] Если GitHub App ещё не подключён, минимальный шаг — **install/authorize GitHub App** для нужного репозитория в Mintlify dashboard; это требуется для авто‑деплоя и редактора, но не для самого DNS‑маппинга домена.[1][4]

Практически это выглядит так:
- В Mintlify: открыть **Settings → Custom Domain / Domain Setup**, добавить `docs.mmbrn.tech`, получить подсказанный CNAME/проверку домена.[4][3]
- В DNS `mmbrn.tech`: добавить **CNAME** `docs` → значение, которое покажет Mintlify; не трогать `office`-записи.[3][4]
- Если репозиторий уже подключён и контент уже есть, дальше остаётся только дождаться DNS/SSL-пропагации.[3][9]

Если хотите, я могу следующим сообщением дать вам **точный набор DNS-записей в формате “что добавить в Cloudflare/Route53/GoDaddy”** и отдельно расписать, **какие записи нельзя удалять**, чтобы не сломать `office.`.

## Q3 — Risk

**Запрос:** стоимость/тариф (custom domain на бесплатном плане Mintlify обычно ограничен); кто держит доступ к дашборду Mintlify и к DNS; риск «полу-подключено» (домен указывает, а проект не привязан → 404/чужой сайт); публичность — не утечёт ли внутреннее (агент-truth в `docs/catalog/`, а не в mintlify — проверить границу «что публично»).

**Выжимка:**

Если коротко: **custom domain на Mintlify требует доступа и к дашборду Mintlify, и к DNS у регистратора/Cloudflare**, а на бесплатном плане наличие custom domain обычно зависит от текущего тарифа и может быть ограничено правилами плана, так что это нужно перепроверять по вашему аккаунту и актуальной странице pricing.[11][1][2]

Что важно по вашему сценарию:

- **Тариф и custom domain.** В источниках есть расхождение: одни описывают бесплатный план Mintlify как включающий custom domain, другие — как ограниченный free-tier с урезанным набором функций, а платные планы начинаются примерно с $250–$300/мес.[5][6][7][8][1][2]  
- **Кто держит доступ.** Для подключения домена нужны действия в **дашборде Mintlify** и изменения DNS у **провайдера домена**; в официальной инструкции Mintlify прямо сказано сначала добавить домен в dashboard, затем настроить DNS-записи у registrar/DNS-провайдера.[11]  
- **Риск “полу-подключено”.** Этот риск реален: если DNS уже указывает на Mintlify, но домен ещё не добавлен/не подтверждён в проекте, сайт может временно отдавать ошибку или некорректный TLS-сертификат во время provisioning; также без правильной привязки домен может не вести на нужный проект.[11]  
- **Публичность и утечки.** По самой схеме работы Mintlify видно, что публичным становится то, что вы публикуете в documentation project и подключаете к домену; если “agent-truth” остаётся в `docs/catalog/` и не загружается/не маппится в Mintlify, то он не должен становиться публичным через Mintlify, но границу публикации надо проверять по тому, какие файлы/разделы синхронизируются в проект.[11]

Практически безопасная модель доступа такая:

- **DNS** должен быть только у того, кто отвечает за домен.
- **Mintlify dashboard** должен быть у того, кто управляет публикацией docs.
- Если это разные люди, нужен явный процесс согласования, чтобы не было ситуации, когда DNS уже указывает на Mintlify, а проект ещё не готов.
- Перед публичным включением стоит проверить, что в Mintlify подключён только тот каталог/контент, который действительно можно показывать внешним пользователям.[11]
