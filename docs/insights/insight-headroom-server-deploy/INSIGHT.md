## Проблема / наблюдение

headroom-ai[mcp] компилируется через maturin (Rust + PyO3) — установка на Windows требует MSVC Build Tools + Windows SDK и занимает 5–10 минут каждый раз при смене окружения. Это тормозит онбординг и не масштабируется: каждый разработчик и каждая CI-машина должны самостоятельно собирать бинарь. Кроме того, headroom держит модель компрессии в процессе — ресурсоёмкость локально нежелательна.

## Гипотеза

Если запустить headroom как HTTP/SSE MCP-сервер (или совместимый с MCP proxy) на удалённой машине, разработчики подключатся к нему как к remote MCP endpoint — без компиляции, без MSVC, без загрузки модели локально. Это снизит порог входа до конфига одной строки и сделает компрессию общим ресурсом команды.

## Scope (черновик)

- In scope: упаковка headroom в Docker-контейнер; выбор транспорта (SSE / stdio-proxy); конфиг Claude Code для remote MCP; базовая auth (token).
- Out of scope: кластеризация, масштабирование под нагрузку, fine-tuning модели компрессии.

## Вопросы для research (Q1–Q3)

2. **Fit (Membrana):** Как Claude Code подключается к remote MCP по SSE — какие изменения нужны в `.claude/settings.json`, есть ли ограничения на latency или аутентификацию?
3. **Risk:** Каковы риски безопасности при открытии headroom как сетевого сервиса — утечка контекста, auth, transport security?
