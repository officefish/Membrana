[Teamlead]: Идея правильная и своевременная — прецеденты уже регистрируются, вопрос лишь в оснастке. Ключевая ценность: прецедент → фрейм → адресный ремонт. Но главный риск подсвечен исследованием прямо: **три пересекающихся каталога** (прецеденты / бестиарий-антипаттерны / уроки). Прежде чем строить дом, обязан быть один абзац таксономии: прецедент = единичный случай с root cause и рецидив-счётчиком; антипаттерн = обобщённый класс; урок = вывод для процесса. Без явной границы получим ложную повторяемость и дубли. Внедрять — да, «сейчас» частично: контракт + схема + регистрация — на этой неделе; осмотр/декомпозиция/inspectElement — следующим слоем. Детектор рецидива держим на уровне простого счётчика класса (Q3 прямо предостерегает от ML и оверинжиниринга). Дом — третий живущий жилец HOME_WORKSHOP, боевая проверка паттерна на greenfield — это ценно само по себе.

[Структурщик]: Схема ложится на GROUP_CONTAINERIZATION чисто: README-контракт + снимок-реестр с Meta + схема записи + кеш под gitignore — тот же скелет, что `docs/audit/git/` и `docs/audit/tasks/`. Настаиваю: реестр — **снимок** (regenerated), не рукописный источник; источник истины = сами файлы прецедентов, реестр агрегируется. `yarn precedent:register` пишет файл по схеме и пересобирает снимок — единственная точка записи, никакого ручного дублирования формы. Минимальная схема из Q1 (canonical cause как нормализованное поле класса, recurrence-счётчик, ссылки related) хорошо мапится. Три глагола — фасады над одним стором прецедентов, без прямых зависимостей между инструментами. Возражение к scope: миграцию 4 файлов делать **в той же задаче**, иначе схема будет жить в вакууме без валидатора на реальных данных.

[Математик]: Здесь чистого DSP нет, но есть мой вторичный контур — скрипт `precedent:register` (`scripts/*.mjs`) и валидатор схемы. Рекомендую: детектор рецидива = чистая функция `countByClass(precedents[]) → Map<class, count>` плюс `recurrenceRate`, порог из Q1 (≤15%) как справочный, не блокирующий. Никакого ML — это детерминированная агрегация по полю `class`/`canonicalCause`. Валидатор `isValid` на схему записи обязателен до записи в контейнер. Один момент: «ложная повторяемость» — это проблема нормализации ключа класса; без канонизации строки класса счётчик врёт. Значит `class` должен быть из закрытого перечня (enum), а не свободный текст.

[Музыкант]: Вне моего контура (аудио/IoT). Формально воздержусь, но замечу как «повторяемость отказов»: холодный старт рецидивил трижды — ровно тот случай, где ранний счётчик класса окупается. Голос не влияющий, приоритет ставлю нейтральный.

[Верстальщик]: UI пока нет — контейнер и мастерская живут в markdown + CLI. Каталог-UI прецедентов (аналог U9) — потенциальный будущий слой, но сейчас out of scope и правильно, что так. Форма снимка-реестра — таблица с Meta, читаемая глазами; когда дойдёт до визуализации раскладки по классам, приду по DESIGN.md. На этой неделе меня не задействовать.
