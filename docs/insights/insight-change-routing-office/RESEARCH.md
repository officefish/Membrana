# Research: Роутинг изменений: сессии без push, заявки-патчи, office-очередь, именные гейты

> Источник: perplexity-api

## Q1 — Landscape

**Запрос:** как в 2025–2026 устроены merge-queue/патч-ориентированные конвейеры (Gerrit change-request модель, GitHub merge queue, Zuul, bors) для парка автономных агентов; есть ли прецеденты «агент без права пуша, только заявка»?

**Выжимка:**

Да: в 2025–2026 уже вполне устоялся паттерн, где **агент не получает прямого права пуша в protected branch**, а работает через **заявку / patch / change-request**, после чего отдельный контур валидации и мержа решает, попадёт ли изменение в основную ветку. Это особенно хорошо ложится на **Gerrit change-request**, **GitHub merge queue**, **Zuul** и **bors**, потому что все они по сути отделяют *производство изменения* от *его включения в trunk*.[2][3][10]

В практическом виде это обычно выглядит так:

- **Агент создаёт патч или change** в изолированной ветке/patchset, а не пушит напрямую в trunk. В Gerrit изменение живёт как reviewable change request; в GitHub изменения попадают в pull request и затем в merge queue; bors и Zuul тоже работают через очереди/батчи/проверяемые изменения, а не через свободный push в основную ветку.[2][10]
- **Мерж-очередь** собирает одно или несколько изменений, прогоняет проверки в контексте актуального base branch и только потом разрешает merge. GitHub Merge Queue специально создаёт временные `merge_group` branches и требует успешных checks для слияния; при этом проверки должны срабатывать на специальной временной ветке с префиксом `gh-readonly-queue/...`.[2]
- **Состояние процесса хранится вне агента**, а не “в его памяти”: это общий принцип мультиагентных оркестровок и очередь/хранилище задач воспринимаются как источник истины о том, что сейчас в работе.[3][7]

### Что это значит именно для парка автономных агентов

Для множества агентов обычно делают так:

- **Агент-исполнитель** может только сформировать patch, PR или Gerrit change.
- **Агент-рецензент / валидатор** проверяет стиль, тесты, безопасность, конфликтность.
- **Отдельный merge-бот / orchestrator** имеет право ставить change в очередь и доводить до merge.
- **Права на push в protected branch отсутствуют** у всех обычных агентов, а у них есть только право создавать заявку или patchset.[2][7][10]

Это хорошо соответствует современному multi-agent подходу: специализация, явные стадии конвейера и жёсткое разделение обязанностей вместо “всем всё можно”.[1][5][8]

### По инструментам

| Система | Как устроен вход изменения | Как происходит merge | Насколько подходит для “агент без права пуша” |
|---|---|---|---|
| **Gerrit** | change-request / patchset | после review и проверок изменение попадает в целевую ветку | Очень подходит: модель по умолчанию именно “заявка на изменение” |
| **GitHub Merge Queue** | PR, затем постановка в очередь | GitHub создаёт временный `merge_group`, прогоняет required checks и мерджит FIFO | Подходит: агент может только открыть PR, а merge делает очередь[2] |
| **Zuul** | change на review-платформе, обычно через очереди зависимостей | тестирует speculative merge / gate pipeline | Подходит для строгого gatekeeping и пачек изменений |
| **bors** | PR, который ставится в очередь | бот мержит только после зелёного CI | Подходит, если нужен простой “bot-controlled merge” |

### Есть ли прецеденты именно “агент без права пуша, только заявка”?

Да, и это, по сути, наиболее безопасная и распространённая схема для автономных кодящих агентов. В литературе и практических кейсах по multi-agent orchestration прямо встречается идея, что агент формирует **патч / merge request / change**, а дальше другой компонент управляет попаданием в репозиторий.[10] В более общих архитектурных материалах также подчёркивается, что агенты должны работать в изолированных песочницах и через явные guardrails, а не иметь неограниченный доступ к репозиторию или инфраструктуре.[7]

### Практический вывод

Если вы строите парк автономных агентов в 2025–2026, наиболее надёжная схема — это:

- **no direct push** для агентов;
- только **patch/PR/change-request**;
- **централизованная merge-queue или gate**;
- **отдельный агент/сервис с привилегиями на merge**, но не на свободный push;
- **все проверки и состояние — вне агента**, в очереди, базе или CI/CD контуре.[2][3][7]

Если хотите, я могу дальше разложить это в виде **референс-архитектуры**:  
1) для **GitHub/GitHub Actions**, 2) для **Gerrit**, или 3) для **self-hosted Zuul/bors-подобного контура** с автономными агентами.

## Q2 — Fit (Membrana)

**Запрос:** минимальный контракт patch-based change submission для шести параллельных автономных coding-агентов в одном монорепо: обязательные поля (diff, scope-декларация, ссылка на тикет, id агента), приложение патча на чистом клоне; где достаточно кастомных git merge drivers для генерируемых файлов, а где нужна серверная merge queue.

**Выжимка:**

Для **шести параллельных автономных coding-агентов в одном монорепо** минимальный практический контракт обычно состоит из: **patch/diff**, **scope-декларации**, **ссылки на тикет** и **id агента**, плюс обязательная проверка, что патч **применяется на чистом клоне** и не зависит от незаявленного состояния репозитория. Идея согласуется с практикой проектных правил через `AGENTS.md`/`rules`, где поведение агента фиксируется на уровне репозитория и применяется только в пределах этого дерева.[1][6]

**Минимальный submission contract** можно оформить так:

- **agent_id** — уникальный идентификатор исполнителя.
- **ticket_ref** — ссылка или ключ тикета/задачи.
- **scope** — явное перечисление затронутых путей, пакетов или модулей.
- **diff** — унифицированный патч или git-дифф, достаточный для применения без ручной доработки.
- **clean_apply proof** — подтверждение, что патч применяется к **чистому клону** целевого состояния.
- **assumptions** — только если агент делал допущения; иначе поле пустое.
- **generated_files_policy** — пометка, изменялись ли файлы, которые генерируются/перезаписываются автоматически.

Если нужен совсем минимальный машинно-проверяемый JSON-контракт, он может выглядеть так:

```json
{
  "agent_id": "agent-3",
  "ticket_ref": "PROJ-1842",
  "scope": ["packages/payments/**", "apps/web/src/features/invoice/**"],
  "diff": "git-format-patch-or-unified-diff",
  "clean_apply": true,
  "assumptions": []
}
```

Для **приложения патча на чистом клоне** лучше требовать не просто `git apply`, а воспроизводимую проверку вида: взять свежий checkout нужного коммита, применить diff, запустить минимальную проверку целостности, и только после этого считать submission валидным. Это особенно важно при параллельной работе нескольких агентов, чтобы исключить скрытую зависимость от локального рабочего дерева.

**Когда достаточно custom git merge drivers для генерируемых файлов:**

- Файл является **детерминированным производным артефактом**.
- Его содержимое можно безопасно восстановить из исходников или генератора.
- Конфликты в таком файле не несут смысловой нагрузки, потому что источник истины — не сам файл, а генерация.
- Наиболее типичный случай: lock-файлы, сгенерированные манифесты, автосборочные артефакты, API-клиентские файлы, где merge driver может либо принять “ours/theirs”, либо пересобрать файл из источника после merge.

Custom merge driver достаточен, когда задача сводится к **локальному, предсказуемому разрешению конфликтов** в отдельном классе файлов и не требует глобальной координации между агентами. Это хорошо сочетается с проектными правилами и специализацией агентов, где поведение ограничивается в рамках репозитория и роли агента.[1][2][6]

**Когда нужна server-side merge queue:**

- Несколько агентов могут менять **одни и те же области** монорепо одновременно.
- Есть **межфайловые инварианты**, которые нельзя гарантировать локальным merge driver’ом.
- Важна последовательная проверка **на актуальном base** перед интеграцией.
- Нужны гарантии, что патч, корректный сам по себе, не ломает дерево после слияния с другими параллельными изменениями.
- Есть риск, что локально “чистый” патч конфликтует по семантике с другими PR, хотя синтаксически применяется.

Иначе говоря: **merge driver** решает *форматный/генерационный* конфликт внутри файла, а **server-side merge queue** решает *координационный* конфликт между несколькими независимыми изменениями в общем коде. При шести параллельных агентах в одном монорепо очередь с серверной валидацией обычно нужна там, где изменения затрагивают общие интерфейсы, сборку, зависимостные графы, CI-критичные файлы и места, где один агент не может корректно знать итоговое состояние после интеграции остальных.

Практическое правило выбора:

- **Custom merge driver достаточно**, если файл:
  - генерируется;
  - детерминированно пересобирается;
  - не является вручную редактируемым источником истины.
- **Нужна merge queue**, если:
  - файл/код — источник истины;
  - изменения пересекаются между агентами;
  - требуется проверка итогового состояния на сервере перед merge.

Если хотите, я могу сразу предложить **готовый YAML/JSON контракт** для PR submission и **политику разделения файлов**: что отдавать merge driver’ам, а что — только через merge queue.

## Q3 — Risk

**Запрос:** self-hosted сервис как единственный канал приёма патчей и очереди слияний от шести автономных агентов: режимы отказа (single point of failure, затор очереди, ревью-пробка), паттерны деградации (локальное накопление патчей, офлайн-режим) и observability-метрики раннего предупреждения до остановки разработки.

**Выжимка:**

Если **сам-hosted сервис — единственный канал** приёма патчей и очереди слияний от шести автономных агентов, то главный риск — не только обычный **single point of failure**, но и накопление системной *backpressure*: один отказ или задержка быстро превращаются в очередь, а затем в ревью-пробку и остановку всей ветки поставки изменений. Для такого контура нужны отдельные механизмы **graceful degradation**: локальное накопление патчей у агентов, офлайн-режим с последующей синхронизацией и наблюдаемость, которая заранее показывает рост латентности, очередей и доли «зависших» изменений до полной остановки разработки.  

По аналогии с self-hosted агентными платформами, где автономные агенты работают внутри собственной инфраструктуры, а действия логируются локально для трассируемости, критичны централизованный контроль канала, аудит и предсказуемое восстановление после сбоев[9]. Практики вроде резервного сценария/отката и проверки статуса сервиса также важны, потому что single-channel архитектура без страховки легко становится точкой простоя[8][10].

- **Режимы отказа**
  - **Single point of failure**: если единый сервис приёма недоступен, шесть агентов теряют общий путь доставки патчей и merge queue, даже если сами агенты продолжают работать[9][10].
  - **Затор очереди**: когда сервис жив, но не успевает принимать/маршрутизировать входящие изменения, очередь растёт быстрее, чем обрабатывается.
  - **Ревью-пробка**: если merge queue зависит от ограниченного числа ревьюеров или автоматических проверок, то throughput падает, а незавершённые изменения накапливаются; в агентных системах отдельная проверка и логирование используются именно для контроля таких опасных и блокирующих действий[12].

- **Паттерны деградации**
  - **Локальное накопление патчей**: каждый агент держит изменения в своей локальной очереди/спуле до появления связи с центральным сервисом. Это снижает риск потери работы при кратковременной недоступности канала и позволяет продолжать генерацию изменений.
  - **Offline-mode**: агент продолжает анализ, подготовку патчей и тестирование локально, но не пытается завершить merge до восстановления связи. Это согласуется с асинхронной моделью работы автономных агентов в self-hosted окружении[9][12].
  - **Буферизация с дедупликацией**: полезно хранить не только патчи, но и их хэши, метаданные ревизии и статус попыток отправки, чтобы при восстановлении связи избежать повторной доставки и конфликтов.
  - **Fail-open для работы, fail-closed для мержа**: агент может продолжать работу автономно, но слияние в основной поток блокируется до подтверждённой доставки и ревью.

- **Observability-метрики раннего предупреждения**
  - **Очередь на приём патчей**: длина очереди, возраст самого старого патча, процент патчей старше SLA.
  - **Merge queue latency**: время от создания патча до попадания в merge-ready состояние; важнее среднего — p95/p99.
  - **Throughput**: патчей в минуту на входе и на выходе; если вход устойчиво выше выхода, система приближается к перегрузке.
  - **Rework rate**: доля патчей, возвращённых на доработку после ревью; рост означает ревью-пробку или плохое качество входа.
  - **Stall time**: время, сколько патч находится без изменения состояния.
  - **Retry rate / delivery failures**: число повторных отправок, сетевых ошибок, timeouts и отказов в обработке.
  - **Agent-side backlog**: объём локально накопленных патчей у каждого из шести агентов и время их ожидания синхронизации.
  - **Health of control plane**: доступность API сервиса, ошибки 5xx, latency p95/p99, saturation CPU/memory/queue workers.
  - **Review capacity**: число активных ревью-потоков, среднее время до первого ревью, доля изменений, ожидающих человеческого или автоматического review.
  - **Change failure indicators**: рост отменённых/откаченных мерджей и конфликтов при интеграции; self-hosted агентные системы подчёркивают важность логирования и отката для контроля таких ситуаций[12].

- **Ранние сигналы до остановки разработки**
  - очередь растёт несколько интервалов подряд, а throughput не увеличивается;
  - p95 времени от патча до мержа выходит за рабочий SLA;
  - у одного или нескольких агентов локальный backlog растёт быстрее, чем очищается;
  - увеличивается доля патчей, ожидающих ревью дольше обычного;
  - растёт число ретраев и timeouts при отправке в центральный сервис;
  - сервис приёма показывает деградацию по latency/5xx, но ещё не упал полностью[8][10].

- **Практический дизайн деградации**
  - держать **локальный спул** у каждого агента;
  - ввести **лимиты на размер очереди** и политики приоритизации;
  - разрешить **автономную подготовку патчей**, но не автономный merge;
  - добавить **dead-letter**-путь для проблемных изменений;
  - иметь **резервный канал управления** или хотя бы проверенный сценарий восстановления, потому что практика self-hosted систем показывает необходимость быстрого возврата к рабочему состоянию[10];
  - собирать метрики на уровне агента и центрального сервиса, чтобы видеть не только падение сервиса, но и *предвестники* падения.
