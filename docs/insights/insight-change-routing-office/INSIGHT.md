## Проблема / наблюдение

Параллельная разработка шестью+ сессиями страдает от двух системных болезней
(вещдоки 21–22.07):

1. **Безответственные коммиты** — изменение без лица: `Co-Authored-By` одинаков
   у всех сессий, атрибуции нет; сессия 22.07 увезла в main прецеденты и кодекс
   вообще без ревью, из-под процедурной ветки — и ни одна стена не встала.
2. **Драка за main** — гонка мерджей: за два дня package.json бился трижды,
   реестры дважды; ни одного содержательного конфликта — только перекрёстки.

Промпты и регламенты не держат: у агентов руки на файловой системе, «они просто
берут и пользуются командами». Стены есть (харнесс-профили, хуки, branch
protection), но это латание — владелец увидел за ним архитектуру.

## Гипотеза

Если превратить поток изменений в **роутинг заявок** — обе болезни исчезают
по построению:

- **Сессия = одна ветка** (id сессии + дата + модель), рождается на старте;
  push/PR сессии закрыты полностью — только pull и **заявка** на office.
- **Заявка = патч в API office**: `{дифф против базы, id сессии, Issue/карточка,
  декларация скоупа}`; office прикладывает патч на чистом клоне — дереву сессии
  не верит.
- **Роутинг ≠ ревью, оба именные**: первая блокировка — **Ожегов** (структура:
  однородность, скоуп; неоднородная заявка → возврат «разбей на Issues»);
  затем ревью **исполнителем** по характеру изменений (не определён — назначает
  Тимлид), затем **Тимлид**. Office — исполняющая механика без своих решений.
- **Очередь мерджа на office** — драка за main исчезает (очередь одна).
- **Атрибуция до трекера**: при закрытии Issue в Linear пишется, какая сессия
  и когда принесла код.

Слово владельца при заведении: задача **слишком радикальная и сырая для
внедрения** — зреет инсайтом, заседание «Роутинг изменений» — когда созреет.

## Scope (черновик)

- In scope: контракт заявки-патча; office-очередь; закрытие push/PR у сессий
  (профили харнесса); структурный гейт Ожегова; двухступенчатое ревью;
  Linear-атрибуция сессий.
- Out of scope (пока зреет): миграция текущих worktree-практик; ночные/cron
  контуры; внешние контрибьюторы.
- **Пререквизит-блокер: стабильность office** — аварийное Issue #933 (таймауты,
  server-first нарушен) чинится ДО любого движения по инсайту.

## Вопросы для research (Q1–Q3)

1. **Landscape:** как в 2025–2026 устроены merge-queue/патч-ориентированные
   конвейеры (Gerrit change-request модель, GitHub merge queue, Zuul, bors) для
   парка автономных агентов; есть ли прецеденты «агент без права пуша, только
   заявка»?
2. **Fit (Membrana):** минимальный контракт patch-based change submission для шести параллельных автономных coding-агентов в одном монорепо: обязательные поля (diff, scope-декларация, ссылка на тикет, id агента), приложение патча на чистом клоне; где достаточно кастомных git merge drivers для генерируемых файлов, а где нужна серверная merge queue.
3. **Risk:** self-hosted сервис как единственный канал приёма патчей и очереди слияний от шести автономных агентов: режимы отказа (single point of failure, затор очереди, ревью-пробка), паттерны деградации (локальное накопление патчей, офлайн-режим) и observability-метрики раннего предупреждения до остановки разработки.

## Поправка владельца (22.07, после ревью — беседа с Ангелиной и питомцем)

**Двухпоточная модель — office не блокирует всю разработку.** Office-очередь
регулирует именно **спонтанные** пуши. Поток **спринтовый** идёт иначе: пуши
разработчиков регулируются **сценариями самих спринтов** — код-ревью делается
программистами ЕЩЁ ДО отправки, они же подкладывают код под нужные ветки, и
всё это — на основе **Linear-контейнера, спланированного перед запуском
спринта**. Следствия: (1) возражение «SPOF на всю разработку» ослаблено —
при падении office спринтовый поток живёт; (2) ревью-пробка не концентрируется
в office — спринтовый код приходит уже проревьюенным; (3) балл 3.4 ревью
частично основан на снятом возражении — при созревании пересмотреть.

**Монитор сетевого окружения** — real-time наблюдение отказов office —
назван владельцем первым шагом (дополнение к DoD аварии #933).

**Поправка 2 (владелец, 22.07): спринтовое ревью — специальный ФРЕЙМ.**
Ревью спринтового потока — не общий механизм, а выделенный фрейм сценария
спринта, спроектированный именно под спринтовое ревью, **с собственным API**.
Сшивка двух штормов дня: сценарий спринта = очередь фреймов (шторм фреймов),
и ревью — один из них: лицо + версия по подграфу + программный контракт,
через который программисты ревьюят и подкладывают код под нужные ветки ДО
отправки. **Поправка 3 — «билет Фаррелла» (владелец, 22.07: «подтверждаю»; имя присвоено владельцем там же):**
членство в спринтовом потоке = **Linear-контейнер, спланированный до запуска
спринта**; нет контейнера → поток спонтанный → office-очередь. Проверяется
МЕХАНИКОЙ, не честным словом: предикат бинарный, самообъявление «я спринтовый»
невозможно.
