# INSIGHT 2026-06-28 — Архивация сессий AI-агентов

> **Статус:** решения зафиксированы, два вопроса открыты до старта кода.
> **Категория:** инфраструктура / операционная память.
> **Связь с тиром:** операционный тир двухтиерной RAG-схемы (архив), не Pinecone.

---

## Контекст

Нужен процесс, собирающий, нормализующий и архивирующий сессии работы с
AI-агентами (сейчас — Cursor / Claude Code / Codex, позже другие окружения).
Сейчас встраивается в вечерний ритуал; в перспективе — ночной фон без
изменения логики.

Целевая функция уточнена в ходе обсуждения: **первичен архив без потерь —
верный воспроизводитель хода мысли разработчика и агента.** Конечная задача —
возможность восстановить историю разработки не только по репозиторию, но и
по сессиям.

---

## Зафиксированные решения

- **Два требования с разными целевыми функциями, не путать.**
  Архив — канал без потерь (сырая сессия, append-only, для аудита и
  реконструкции). Индекс — канал с потерями (информационный остаток:
  решения, диффы, тупики). На этом этапе **первичен архив**; индекс вторичен.

- **Индекс отложен.** Пока корпус адресуется напрямую (манифест +
  frontmatter + grep), вектор не нужен. Pinecone — по достижении порога,
  когда прямая адресация перестаёт справляться. Не тащим в первую версию.

- **Хранение вне git, через общий backend.** Цель — возможность
  использовать данные в продакшн-процессах. Не эфемерный локалхост.

- **Скруб секретов на границе нормализации — блокирующее условие, не «потом».**
  Выбор «persist + реюз в проде» поднимает `P(секрет доедет до реплицируемого
стора)` с нуля до ненуля по построению. Отдельно: в сессиях оседают
  scoped-ключи второго разработчика — чужой секрет, его утечку чинит ротация,
  а не `DELETE`. Фильтр на известный шумный класс: `sk-ant-…`, `ghp_…` /
  `github_pat_…`, `lin_api_…`, `ANTHROPIC_API_KEY=…`, высокоэнтропийный хвост
  после `=`. Замена на `[REDACTED:kind]` + хеш — видно «здесь был секрет»,
  самого секрета нет.

- **Скруб и дедуп не конфликтуют с «не потерять ход мысли».** Ключ — максимум
  энтропии и ноль сигнала для реконструкции рассуждения; вырезается ровно
  нулевой по целевой функции класс байтов. Дедуп по хешу хода убирает повторы
  одного turn'а (JSONL перечитывается каждый вечер), а не различающийся
  контент. Ни одно рассуждение не теряется.

- **Адаптер на окружение, общая нормализованная схема.** Источники
  гетерогенны (Claude Code — JSONL; Cursor — SQLite `state.vscdb`; Codex —
  отдельный формат). Адаптеры приводят к единой записи `Session`.

- **Инкрементальность через per-turn хеш, не «видел файл».** Сессии Claude
  Code растут append-only, один файл читается каждый вечер; дедуп на уровне
  хода, иначе архив распухнет копиями.

- **Корреляты сессия ↔ репо снимаются в момент архивации.** Поздняя
  реконструкция по таймстемпам и нечётким диффам ненадёжна. Снимаем `branch`,
  `HEAD` на границах сессии, таймстемпы коммитов в окне, предложенный дифф —
  и `linked_issue` (из `MAIN_DAY_ISSUE.md` дня или имени ветки) как нить,
  сшивающую сессии в эпик через дни.

- **Размещение скрипта.** `scripts/sessions-archive.mjs` + yarn-команда, по
  образцу `proxy-claude.mjs`. Не в графе сервисов. В вечернюю цепочку:
  `ritual:evening = archive:daily-day → archive:sessions → code-review →
save-code-review`. Позже выносится в ночной фон без изменения логики.

## Открытые вопросы (до старта кода)

1. **Домен backend.** `office` концептуально ближе всего (это и есть
   Claude/Codex/Cursor), но stateless by design — персистентный стор ломает
   инвариант. `media`/`cabinet` — чужие домены. Чистый вариант по
   single-source-of-truth — новый `background-sessions` с собственным
   Postgres. Рекомендация — отдельный сервис; решение не принято.

2. **Первый адаптер — claude-code** как самый чистый источник (JSONL по
   сессиям). Точные пути и поля снимаются с рабочей машины: формат CLI плывёт
   между версиями, за структуру Codex по памяти ручаться нельзя.

## Нормализованная схема и контракт адаптера

```ts
type AgentId = 'claude-code' | 'cursor' | 'codex';

interface Session {
  id: string; // стабильный, из источника (uuid сессии claude-code)
  agent: AgentId;
  project: string; // repo/workspace
  started_at: string; // ISO8601
  ended_at: string;
  source_ref: string; // путь/row-id — чтобы перечитать при ре-деривации
  turns: Turn[];
  files_touched: string[];
  content_hash: string; // хеш нормализованных turns → инкрементальность

  branch: string;
  head_at_start: string; // git rev-parse HEAD на старте
  head_at_end: string; // на момент архивации
  linked_issue?: string; // из MAIN_DAY_ISSUE дня или имени ветки
}

interface Turn {
  index: number;
  role: 'user' | 'assistant' | 'tool';
  ts: string;
  text: string; // уже после скруба
  tool?: { name: string; input_summary: string; output_summary: string };
  diff?: string; // unified diff, если ход породил правку
  hash: string; // per-turn хеш → дедуп между запусками
}

interface SessionAdapter {
  agent: AgentId;
  discover(): SessionRef[]; // найти кандидатов (пути/строки БД)
  read(ref: SessionRef): RawSession;
  normalize(raw: RawSession): Session; // маппинг + скруб внутри
}
```

Петля сборщика: `for adapter → discover → read → normalize(+scrub) →
dedup по turn.hash → upsert в backend`.

## Потолок канала (честное ограничение)

Даже со снятыми корелятами связь «что агент предложил» → «что легло в коммит»
остаётся **вероятностной, не побитовой.** Между предложением и коммитом стоят
человек и git (отложенный коммит, связка с другими правками, rebase) — это
канал с потерями.

- Репозиторий — основа истины о том, **что попало**.
- Сессии — основа истины о том, **что рассматривалось и почему**.

Вместе дают нарратив, но шов между ними приблизителен. «Полностью
восстановить» читать как «восстановить обе стороны без потерь и сшить их с
явной погрешностью», а не «вывести точное соответствие». Это свойство
предметной области, не дефект сборщика.
