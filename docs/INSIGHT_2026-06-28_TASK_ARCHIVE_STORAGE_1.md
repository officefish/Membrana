# INSIGHT 2026-06-28 — Хранилище архива закрытых задач

**Статус:** к регистрации
**Область:** реестр задач, регламент закрытия, `background-office`
**Связанные документы:** `docs/tasks/registry.json`, `TASK_CLOSURE_REGULATION.md`, `docs/tasks/README.md`

---

## Постановка

Реестр актуальных задач (`registry.json`) должен оставаться лёгким и
устойчивым к повреждению по мере роста истории закрытых задач. Рассматривался
перенос архива закрытых задач в полноценную БД (PostgreSQL) на `background-office`.

Формализация. Здесь две независимые оси, которые в исходной постановке склеены:

- **Лёгкость активного реестра** — функция объёма горячих данных.
- **Устойчивость к повреждению** — функция формата записи, а не объёма.

БД решает объём и конкурентный доступ. Повреждение `registry.json` — это риск
формата (кривой коммит, конфликт слияния, обрыв записи при одновременной правке),
и он снимается разнесением горячего и холодного пути, а не обязательно сервером БД.

## Решение

Разнесение горячего и холодного хранилища — верное и имеет имя: **CQRS поверх
append-only log**. Активные задачи — read-оптимизированная проекция; закрытые —
write-оптимизированный источник истины. Приём в архив и выборки из него
масштабируются независимо. Из разнесения **не следует**, что холодный путь должен
быть сервером БД: это отдельное решение, продиктованное конкурентной записью,
которой у последовательного вечернего архивирования нет.

Append-only log как форма архива даёт ровно требуемое:
последовательная дозапись в конец, никакой правки на месте, повреждение
локально на уровне одной строки. Источник истины по истории — `archive.jsonl`
(одна задача = одна строка с `id` и `timestamp`); активный `registry.json` —
проекция «что сейчас в работе».

## Лестница хранилищ

Останавливаться на первой достаточной ступени. Каждая следующая ступень — плата
без отдачи, пока не выполнено условие перехода.

- **Активный реестр** — маленький JSON, только открытые задачи; источник истины по «в работе».
- **Холодный архив** — `archive.jsonl`, append-only, строка на задачу; источник истины по истории.
- **Индекс над архивом** — генерируемый, не источник истины; плоский JSON, пока грузится в память, далее SQLite-файл. Теряется — пересобирается из JSONL.
- **Postgres на background-office** — только при конкурентной записи в архив с нескольких процессов либо объёме порядка единиц ГБ.

При появлении запроса на аналитику по истории (время закрытия, распределение по
epic, нагрузка по персонам) правильный шаг — не «положить архив в Postgres», а
разнести OLTP и OLAP: DuckDB прямо по тем же JSONL-файлам, без отдельного сервиса.

## Не-цели

- **Не tamper-evidence.** Хеш-цепочка (`prev_hash`/`curr_hash`) избыточна: недоверенной стороны в контуре нет.
- **Не сервер БД под холодные данные.** Postgres резервирует ~35–50 МБ RAM на простаивающем инстансе — постоянная плата за данные, которые почти не читаются, при том что RAM на VDS уже лимитирует выбор Ollama-модели.
- **Не потеря git-версионируемости.** Архив в репозитории остаётся человекочитаемым и diff-аемым; перенос в Postgres это свойство ломает.

## Триггеры пересмотра (stage-gate)

Переход на следующую ступень — по факту выполнения условия, а не по ощущению
«кажется, пора». Переход на Postgres оправдан при выполнении любого из:

- `database is locked` (или эквивалент) в логах архивирования чаще одного раза в неделю.
- Появляется второй процесс, пишущий в архив параллельно с вечерним ритуалом.
- Объём архива пробивает единицы гигабайт.

До выполнения любого из условий Postgres решает несуществующую проблему.

## Обоснование (внешние источники, переформулировано)

- Append-only log / event sourcing — устоявшийся паттерн: последовательная дозапись, неизменяемость устраняет конфликты записи, формат пригоден для аудита и восстановления.
- CQRS поверх event log позволяет независимо масштабировать чтение и запись, так как приём событий и оптимизированные под запросы проекции работают раздельно.
- Триггер перехода SQLite → Postgres в индустрии — давление конкурентной записи (несколько процессов в одну базу), а не объём; ниже этого порога SQLite для одного пишущего процесса точнее и быстрее.
- Порог объёма SQLite — порядка 10 ГБ; ось решения: конкурентность записи, объём данных, потребность в сетевом доступе. Архив закрытых задач не попадает ни под одну.

## Резюме

Интуиция о разнесении горячего и холодного пути верна и имеет имя. Выбор Postgres
как формы холодного хранилища — избыточен: он решает проблему конкурентной записи,
которой у последовательного вечернего архивирования нет, ценой постоянной RAM и
потери git-версионируемости. Целевая форма — `archive.jsonl` (append-only) +
генерируемый индекс, с явным stage-gate на переход к серверу БД.
