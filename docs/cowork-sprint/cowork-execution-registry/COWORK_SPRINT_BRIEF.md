## Problem (общая разработка)

Шторм об утреннем ритуале 19.07 вскрыл корень: **у процессов нет субъекта** —
утренние документы собирает детерминированная склейка, ни у одного шага нет того,
кто решает. Два заседания закрыли эту дыру вердиктами с двух сторон:

- **Эпик 1** [`team-execution-contour`](../../meeting/team-execution-contour/EPIC.md) —
  как персона начинает *делать*, а не только советовать: ведущая персона Ангелина →
  зубы регламента (след исполнителя) → измерение работы. Порядок Р1→Р2→Р3 ратифицирован.
- **Эпик 2** [`registry-relocation`](../../meeting/registry-relocation/EPIC.md) —
  переезд реестра на внешний стек (Linear): словарь единиц (аксиома) → снимок-контракт
  (фундамент) → холодный контур и миграция (независимые ветви).

**Общая разработка спринта:** восемь вердиктов двух эпиков превратить в работающий
контур — промпт и предикаты ведущей персоны, гейты следа с кодами возврата, батч
измерения, производитель снимка, холодный `archive.jsonl`, классификатор долга.

**Естественный шов резки — три слоя системы:**

1. **слой субъекта** (кто решает и делегирует) — работа 1 эпика 1;
2. **слой учёта** (что считаем, чей след, как меряем) — работы 4, 2, 3; хендофф
   явно предупредил: работа 2 (носители следа) и работа 4 (единица счёта) говорят
   об одном объекте с разных сторон — разводить их по разным блокам нельзя,
   поэтому они в одном блоке;
3. **слой данных** (откуда факты берутся и где лежат) — работы 5, 6, 7; работы 6 и 7
   независимы между собой, обе держатся на 5 — зависимость остаётся внутри блока.

Ранняя договорённость об интерфейсах навредила бы: форма следа (блок 2) и схема
снимка (блок 3) должны сначала честно вырасти из своих вердиктов, а не из
компромисса «чтобы состыковалось». Сведение — задача Interface Consilium.

### Общие корневые файлы — НЕ трогает никто

В изолированной фазе запрещены правки:

- `package.json` (скрипты `angelina:*`, `snapshot:*` и т.п. — вносятся на интеграции);
- `docs/tasks/registry.json` — **живой реестр данных не трогает даже блок миграции**:
  классификатор работает на фикстурной копии; карточку спринта ведёт координатор;
- `.husky/**`, `scripts/task-closure-review.mjs`, `scripts/task-archive.mjs`,
  `scripts/lib/task-registry.mjs` — вшивание гейтов отказа-I/II в хуки и lifecycle —

- `docs/virtual-team/PROMPT_TEAMLEAD.md` и прочие промпты пяти советующих персон —

  неприкосновенны, только чтение.

## Constraints

1. **Закрытые вердикты не пересматриваются — только применяются.** Обнаружил, что
   вердикт неисполним, — `team-<block>/BLOCKER.md`, не самодеятельность.
2. **Гейты владельца не закрыты** (ратификация уточнения `insight-task-archive-storage`,
   дата рубежа законности, разбор `parked`, судьба `STRATEGIC_PLAN_DAY.md`).
   Проектировать и стабить — можно и нужно; **живой перенос 258 архивных записей и
   любое исполнение, упирающееся в гейт, — в спринт не входит**.
3. **Три факта о Linear проверить по живому API, не по выжимке ресёрча** — каждый
   блок проверяет своё: `units-trace-measure` — двухслойность `assignee`/`delegatedAgent`
   и статус под-задачи как сущности; `snapshot-cold-migration` — «`blockedBy`/`blocking`
   вебхуками не приходят» и прецедент `linearId = DRU-93`. Канал — office
   (`v1/linear/issue/:id`). API недоступен → зафиксировать пробел в EXPECTATIONS,
   это не блокер (контур всё равно снимко-центричный).
4. **Вне спринта:** разбор токсичных карточек (отдельный процесс, решение владельца);
   граф правды (проектируется отдельно); тулинг заседаний (`meeting-audit` маска —
   только в `../Membrana-tooling`).
5. **Изоляция** (регламент §Hard rules): чужие ветки и чужие `EXPECTATIONS.md` не
   читать; об интерфейсах не договариваться до Phase 3; merge/rebase между ветками
   блоков запрещены.
6. Каждая команда — **в своём worktree** (скилл `membrana-worktree`); коммитить
   строго свои файлы поимённо, никогда `git add -A`.

## Ожидаемые швы (гипотеза координатора — уточняется контрактом Phase 3)

1. **Субъект ↔ след** (`lead-persona` ↔ `units-trace-measure`): след требует
   «персону, принявшую выход», подпись субагента не требуется (тот безымянен по
   контракту). Контракт фиксирует, как имя персоны попадает в артефакт закрытия.
2. **Снимок ↔ измерение** (`units-trace-measure` ↔ `snapshot-cold-migration`):
   «два снимка реестра» батча величин = два материализованных pull-снимка.
   Контракт сводит форму honest-шапки снимка и honest-шапки отчёта.
3. **Словарь ↔ схема данных** (`units-trace-measure` ↔ `snapshot-cold-migration`):
   единица счёта (GitHub-issue, биекция) определяет, что снимает producer и что
   хранит холод. До контракта блок данных живёт на допущениях, зафиксированных
   в своём EXPECTATIONS.
4. **Единый глоссарий кодов возврата**: отказ-I/II (зубы), три кода снимка,
   три режима миграционного гейта — на консилиуме сводятся в одну таблицу
   exit-кодов (термин = граница; «отказ отличается кодом, не тоном» — общий закон).
5. **Владелец времени**: `capturedAt` снимка vs окно отчёта vs «день» журнала
   Ангелины — чьи часы каноничны на каждом шве.

## Интеграционный smoke (набросок — уточняется контрактом Phase 3)

Сквозной прогон на интеграционной ветке, офлайн, без сети в момент вердикта:

1. Фикстурный источник → producer собирает снимок с провенансом; предикат свежести
   отрабатывает оба исхода (свеж / протух → мягкий код).
2. Гейт отказа-I читает снимок офлайн: карточка без `leadPersona` → `exit ≠ 0`;
   отказ-II на артефакте закрытия без приёмки → soft-код.
3. Батч измерения из двух фикстурных снимков + артефактов закрытия выдаёт отчёт
   с полной шапкой и подписью «wip = очередь намерений».
4. `classifyDebt` на смешанном наборе: счётчик незакрытых работ считает только
   `work`; `parked` и `archived-pre-passport` его не поднимают.
5. Cold-writer принимает запись с honest-шапкой и отвергает без неё.
6. `isValid` Ангелины валидирует возврат фикстурного аналитика и писаря; предикат
   делегирования на кейсе «долгоживущий документ» выбирает писаря.
7. Все exit-коды — по единой таблице контракта; собственные тесты трёх блоков
   зелёные в собранной ветке.
