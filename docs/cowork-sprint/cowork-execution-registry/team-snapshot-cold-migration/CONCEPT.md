## 0. Слой блока и внутренний порядок

Блок — **слой данных**: откуда факты берутся (производитель снимка), где лежат
(холод) и как существующий реестр переезжает законно (миграция).

Внутренний порядок строгий:

```
Снимок (Р2) ──┬──→ Холод (Р3)      ┐ независимы между собой,
              └──→ Миграция (Р4)   ┘ после снимка идут параллельно
```

Холод хранит снимки карточек — без контракта снимка нечего принимать.
Классификатор долга вычисляется на снимке — без контракта нечего классифицировать.
Между Р3 и Р4 зависимости нет: writer холода и `classifyDebt` не знают друг о друге,
их сводит только предикат `inCold` (читается через офлайн-reader).

Топология данных (вердикт M2, граф без циклов):

```
Linear ──(полный pull)──→ office-батч ──(материализация)──→ СНИМОК ──→ гейты (офлайн)
   │                            ↑                              │
   └──(вебхук = только триггер)─┘                              └──→ холод archive.jsonl
                                                                    (append-only, один писатель)
```

Сеть живёт ровно в одном узле — производителе. **Ни одна проверка не ходит
в сеть в момент вердикта.** Слово «кэш» в подсистеме запрещено (вердикт M2):
снимок — контракт с провенансом, его молчаливое устаревание — нарушение,
а не потеря производительности.

## 1. Р2 — Снимок-контракт

### 1.1 Схема снимка

Снимок — адресуемый, версионированный артефакт. Один файл = одна съёмка.

```jsonc
{
  "header": {                          // провенанс — обязателен целиком
    "format": "linear-snapshot@1",     // версия контракта
    "capturedAt": "2026-07-19T18:00:00.000Z",  // момент съёмки (часы производителя, UTC)
    "sourceRevision": "<курсор Linear на момент съёмки>",
    "source": "office-batch",          // производитель — всегда office-батч
    "trigger": "webhook | evening-signal | manual",  // что попросило снять; телом НЕ является
    "recordCount": 779
  },
  "records": [                         // слой ДВИЖЕНИЯ (содержание живёт в git — не дублируем)
    {
      "linearId": "DRU-93",
      "state": "…", "stateType": "…",
      "assignee": "…",                 // ответственная персона (двухслойность — допущение A3)
      "delegatedAgent": "…",           // исполнитель; имена НЕ склеивать (вердикт M1)
      "parentId": "…",                 // контейнер
      "blockedBy": ["…"], "blocking": ["…"],   // из GraphQL; вебхуками не приходят (допущение A2)
      "githubIssueRefs": ["#660"],     // паспорт(а) — биекция центральной под-задачи ↔ Issue
      "createdAt": "…", "updatedAt": "…", "completedAt": null
    }
  ]
}
```

Что именно снимаем (набор полей записи) — окончательно определяет словарь единиц
(зона соседа); до Interface Consilium блок живёт на явных допущениях A1–A6 из
`EXPECTATIONS.md`. Схема закрыта версией `format` — расширение полей не ломает
потребителей.

### 1.2 Производитель

- **Office-батч полным pull-опросом** состояния на момент съёмки. Снимок никогда
  не строится из накопленных вебхуков: вебхуки теряются, есть rate-limit (M2).
- **Вебхук — только триггер «пора снимать»**, отделён от тела производителя и
  архитектурно (отдельный вход `trigger`), и в коде (отдельный файл). Тело снимка —
  всегда свежий полный pull. Триггеры равноправны: вебхук, вечерний сигнал
  (редукция вечерней архивации — вердикт M3), ручной запуск.
- Модуль — `packages/background-office/src/linear-snapshot/**`.
  `app.module.ts` не трогаем — провод на интеграции.

### 1.3 Предикат свежести — вне тела гейта

```
fresh(snapshot, sourceCursor) := sourceCursor == snapshot.header.sourceRevision
```

Один дешёвый запрос курсора O(1), вызывается **вне тела гейта** (в момент
производства или отдельной проверкой свежести). Тело вердикта офлайн: свежесть —
аннотация вердикта, не его валидность (`gate(snapshot)` детерминирован на любом
снимке, включая протухший). Способ получения курсора инъецируется — в тестах это
фикстура, сети в тестах нет (вердикт M2, он же DoD блока).

### 1.4 Три кода возврата

| Код | Имя | Класс | Когда |
|---|---|---|---|
| `0` | `OK` | в порядке | снимок есть, провенанс полон, свежий |
| `10` | `SNAPSHOT_STALE` | **мягкий** — замечание | снимок есть и валиден, источник ушёл вперёд; вердикт честен на теле снимка, шапка отчёта обязана показать «данные на ревизии X» |
| `20` | `SNAPSHOT_NO_INPUT` | **жёсткий** — отказ | снимка нет, провенанс отсутствует или битый (`capturedAt`/`sourceRevision` пусты), формат не распознан — вход не определён |

Отказ отличается от замечания **кодом, не тоном**. Конвенция кодового
пространства блока — §4.

## 2. Р3 — Холодный контур `archive.jsonl`

### 2.1 Форма

- **Append-only** один файл `archive.jsonl`; одна строка = одна запись = снимок
  карточки на момент архивации. Читается офлайн `.mjs`-скриптом без сети.
- Ступень 2 лестницы хранилищ (28.06, уточнение M3 ждёт ратификации владельца —
  проектируем, не исполняем перенос). Postgres и MongoDB **отклонены как
  избыточные** — вердикт закрыт, не пересматриваем.
- **Единственный производитель** — office-батч по триггеру. Вечерняя архивация
  редуцируется до сигнала триггера, своей дорожки записи не ведёт. Один холодный
  источник, не два равноправных.

### 2.2 Запись и honest-шапка — условие приёмки

```jsonc
{
  "snapshot": {                        // honest-шапка — БЕЗ неё writer запись ОТВЕРГАЕТ
    "capturedAt": "2026-07-19T18:00:00.000Z",  // дата снимка
    "headRevision": "<git SHA head на момент снимка>", // ревизия head
    "source": "office-batch/trigger=evening-signal"     // источник
  },
  "card": { /* содержание карточки целиком: постановка, промпт-путь, решения */ },
  "domain": {                          // доменные поля, которых нет во внешнем источнике
    "dependsOn": ["…"],                // граф зависимостей — обязателен (413 записей, наш актив)
    "strategicWave": "…",
    "leadPersona": "…"
  }
}
```

- **Writer отвергает запись без honest-шапки** — это условие приёмки, покрывается
  тестом (вердикт M3, восстановлен финальным аудитом). Отвержение — жёсткий код.
- Хранится **содержание + доменные поля**. **Движение (open/close, переходы)
  не дублируется** — оно живёт во внешнем аудите; дублирование = синоним-слой.
- Reader — офлайн-скан; критерий перехода на индекс (ступень 3) фальсифицируем:
  `scan_time(archive.jsonl) > T_gate`. Индекс генерируется из jsonl, не заменяет
  источник. Критерий 3→Postgres: `database is locked` чаще раза в неделю **при
  доказанном пересечении писателей во времени**, ИЛИ гигабайты, ИЛИ второй
  доказанно пересекающийся писатель.
- Спека формата — `docs/tasks/archive/ARCHIVE_FORMAT.md` (Phase 2).

## 3. Р4 — Миграция и включение законности

### 3.1 Критерий «правильно заархивировано»

```
wellArchived(r) = inCold(r) ∧ hasPassport(r) ∧ archivedAt(r) ≠ null
```

Три проверяемых предиката, чистая функция на снимке-контракте, без сети.
`inCold` читается офлайн-reader'ом холода; `hasPassport` — наличие GitHub-issue;
`archivedAt` — поле карточки.

### 3.2 Классы долга — вычисляются ДО переноса

```
debtClass(r) ∈ { record, work, legality, owner-knowledge }

¬hasPassport ∧ ¬wellArchived            → legality        (нет паспорта — долг законности)
hasPassport ∧ issueClosed ∧ ¬archivedAt → record          (сделано, карточка открыта — долг записи)
hasPassport ∧ ¬issueClosed ∧ ¬workDone  → work            (не сделано — долг работы)
статус неустановим по коду и докам      → owner-knowledge (вопрос владельцу)
```

- Чистая функция, детерминирована, на снимке. `hasPassport = false` **не влечёт**
  `workDone = false` — пилот 20 старейших дал 17 сделано / 3 частично / 0 не сделано.
- **Счётчик незакрытых работ читает только `debtClass = work`.** Тест-инвариант:
  258 архивных без паспорта (класс `legality`) не поднимают счётчик ни на единицу;
  фиктивный долг синтезироваться не может по конструкции.
- Легализация рубежом видима: класс `archived-pre-passport`, поле `legalizedBy` —
  читатель отличает «паспорт был всегда» от «принят рубежом».

### 3.3 Три режима гейта законности — коды возврата

Комната задала коды **мягкого** (exit 0) и **жёсткого** (exit ≠ 0); **код
миграционного режима комнатой НЕ назван** — блок называет его здесь явно,
как собственное решение постановки, а не как вердикт комнаты:

| Режим | Поведение при нарушении | Exit-код | Кем назван |
|---|---|---|---|
| `soft` | `¬wellArchived` → замечание в отчёте, работа не встаёт | **`0`** | комната (M4) |
| `migrating` | `¬wellArchived ∧ ¬legalized` → нарушение зафиксировано, счётчик `N_illegal` в шапке, работа не встаёт | **`11` = `LEGALITY_MIGRATING`** | **блок** — кандидат на сведение в единую таблицу exit-кодов на Interface Consilium |
| `hard` | `¬wellArchived ∧ ¬legalized` → отказ | **`21` = `LEGALITY_HARD`** (≠ 0, жёсткий диапазон) | комната задала «≠ 0»; конкретное значение — блок |

- Обоснование `11`: миграционный режим обязан быть **различим по коду** и от
  чистого `soft` (0 — не видно, что миграция не закончена), и от `hard`
  (≥ 20 — остановил бы работу, что запрещено вердиктом «работа между шагами
  не встаёт»). Мягкий диапазон 1–19 — замечание, различимое машинно.
- Переключение `migrating → hard` — **по предикату `N_illegal = 0`**, не по дате.
  `N_illegal = |{r : ¬wellArchived(r) ∧ ¬legalized(r)}|`.
- Режим (`soft|migrating|hard`), счётчик `N_illegal` и дельта за прогон —
  **видимы в шапке отчёта**, переключение не случается «вдруг».

### 3.4 Состояние `parked`

Не пятый класс долга, а состояние ожидания внутри `owner-knowledge`:
не мигрирует, не блокирует, из счётчиков долга исключено (тест: `parked` не
поднимает счётчик), помечено датой запроса владельцу («ждёт владельца с DD.MM»).
Дефект `db-h1b…db-h4` — кандидаты в `parked`: даты машинно не досочиняем.

### 3.5 Порядок переноса (проектируется; исполнение — за гейтами владельца)

1. Мягкий гейт + снимок: `N_illegal` измерен, ничего не блокируется.
2. Классификация: `debtClass` каждой записи на снимке-контракте.
3. Легализация: 153 активных → выписка Issue задним числом (через PR);
   258 архивных → рубеж законности с датой; `owner-knowledge` → `parked`.
4. Перенос архивных в холод office-батчем (единственный производитель, M3).
5. `N_illegal = 0` → переключение на `hard`.

**Вне спринта** (незакрытые гейты владельца): живой перенос 258 архивных, дата
рубежа законности, разбор `parked`, ратификация уточнения
`insight-task-archive-storage`. Блок это **проектирует и стабит** (классификатор
гоняется на фикстурной копии реестра), но **не исполняет**. Живой
`docs/tasks/registry.json` блок не трогает вовсе.

## 4. Кодовое пространство блока (сводная таблица)

Конвенция: `0` — чисто; `1–19` — мягкий диапазон (замечание: вердикт валиден,
работа не встаёт; вызывающая сторона обязана обработать явно, не трактовать как
крах); `≥ 20` — жёсткий диапазон (отказ: вход не определён или инвариант нарушен
в hard-режиме).

| Код | Имя | Источник |
|---|---|---|
| 0 | `OK` | все гейты блока |
| 10 | `SNAPSHOT_STALE` | контракт снимка (M2: мягкий) |
| 11 | `LEGALITY_MIGRATING` | гейт законности, режим migrating (**назван блоком**) |
| 20 | `SNAPSHOT_NO_INPUT` | контракт снимка (M2: жёсткий) |
| 21 | `LEGALITY_HARD` | гейт законности, режим hard (M4: «≠ 0») |

Известное напряжение для Interface Consilium: M4 задал мягкому режиму законности
exit `0`, M2 задал мягкому `STALE` ненулевой различимый код — две комнаты дали
разную семантику «мягкости». Блок не сглаживает это втихую: таблица выше
исполняет оба вердикта буквально и выносится как вход в единый глоссарий
exit-кодов спринта (ожидаемый шов №4 из brief).

### Собственный DoD блока (из brief — чем закрывается)

- Схема снимка + предикат свежести + три кода — тесты на фикстурах, сеть в тестах
  запрещена (это и есть вердикт M2) → `snapshot-*.mjs` + `.spec.ts`.
- Producer pull-опросом за стабом API, триггер отдельно от тела → `linear-snapshot/*`.
- Cold-writer принимает с honest-шапкой, **отвергает без неё** (тест); офлайн-reader
  → `cold-*.mjs`.
- `classifyDebt`/`wellArchived` чистые функции: 153 активных → путь легализации
  (класс `legality`, судьба — выписка), 258 архивных → `legality`/рубеж, `parked`
  не поднимает счётчик → `debt-*.mjs`.
- `node --test` своих файлов + scoped CI пакета office — зелёные.
