# CONCEPT — Block `units-trace-measure`

> Cowork Sprint `cowork-execution-registry`, Phase 1.
> Работы: **Р1 эпика 2 (словарь единиц) → Р2 эпика 1 (зубы регламента) → Р3 эпика 1
> (измерение)** — внутренний порядок строгий, обратных рёбер нет.
> Вердикты закрыты (m1-counting-unit · m2-regulation-teeth · m3-measurement 19.07);
> здесь они **применяются**, не пересматриваются.

---

## 1. Блок изнутри: один объект с трёх сторон

Хендофф предупредил: единица счёта (Р1) и носитель следа (Р2) говорят об **одном
объекте** — единице работы — с разных сторон. Измерение (Р3) считает этот же объект.
Поэтому блок устроен как три слоя над одним словарём:

```
UNITS_DICTIONARY (аксиома: что есть единица работы и её «закрыто»)
   ↓ термины
trace-* (зубы: чей след обязан быть на единице, отказ = exit-код)
   ↓ носители
measure-* (пять величин: чистая функция от снимков + артефактов закрытия)
```

Всё, что ниже словаря, не имеет права вводить собственные значения слов «задача»,
«закрыто», «ответственный» — только импортировать их.

## 2. Словарь единиц — `docs/tasks/UNITS_DICTIONARY.md` (работа Р1)

Документ-аксиома. Состав по вердикту m1-counting-unit:

### 2.1 Словарные статьи

| Термин | Значение |
|---|---|
| **Центральная задача** | = **GitHub-issue** (удостоверение задачи). Единица ответственности и счёта. |
| **Контейнер** | Linear-issue (parent) — единица движения, агрегатор. Ответственности **не несёт**, в счёт **не входит**. |
| **Дополнительная под-задача** | Атрибут центральной, производная. Не самостоятельная единица счёта. |
| **Ответственный** | Ровно одна персона, **принявшая выход** (`leadPersona` карточки, `assignee` в Linear). Не физический производитель. |
| **«Закрыто»** | = закрытие **центральной задачи** (= её GitHub-issue). Закрытие контейнера — производное; закрытие подчинённой под-задачи — **не событие**. |

### 2.2 Биекция и запрет двойного счёта

- Правило: `count-unit(work) = центральная под-задача ⟺ её GitHub-issue` — **биекция**.
- Каждая работа отображается ровно в одну единицу. Контейнер и дополнительные —
  вне области счёта по построению. Инвариант проверяем: `Σ по центральным = счёт`,
  `Σ по контейнерам ≠ счёт`.

### 2.3 Двухслойность Linear — через адаптер

- `assignee` ↔ ответственная (принявшая выход) — **тождество по функции**.
- `delegatedAgent` ↔ исполнитель — **соответствие через адаптер, имена не склеивать**:
  `delegatedAgent` — конкретный слот агента Linear, а исполнитель нашего канона —
  обезличенный субагент без имени. Адаптер — предмет интеграции, не словаря.
- Агент не может быть единственным ответственным (факт Linear).

### 2.4 Таблица «величина → единица счёта»

| Величина | Единица счёта |
|---|---|
| `leadTime` | GitHub-issue (центральная) |
| `ownerlessRate` | GitHub-issue (центральная) |
| `throughput` | GitHub-issue (центральная) |
| `escalationRate` | GitHub-issue (центральная) |
| `wip` | движущаяся Linear-под-задача — **через биекцию** с Issue, дрейфа нет |

### 2.5 Пометка достоверности

Факты Linear (двухслойность `assignee`/`delegatedAgent`, под-задача как полноценная
сущность со своим статусом) взяты из выжимки ресёрча
`docs/tasks/research/linear-agent-identity-facts.md` — **вход, не решение**.
Проверка по живому API (office `v1/linear/issue/:id`) — Phase 2; до проверки словарь
несёт явную пометку «факт по ресёрчу, не подтверждён живым API».

## 3. Зубы регламента — `scripts/lib/trace-*.mjs` (работа Р2)

По вердикту m2-regulation-teeth. Функция следа — сделать приёмку **отзывной**,
а не украсить коммит подписью.

### 3.1 Носители следа — ровно два

1. **Карточка реестра** — поле `leadPersona` (назначение ответственной).
2. **Артефакт закрытия** — поле приёмки `{ acceptedBy, headRev }`:
   кто дал LGTM и на какой ревизии head. Ставится на closure review; декоррелирован
   от назначения фазой (назначение — акт регистрации, подтверждение — акт LGTM
   на конкретной ревизии).

**Коммит-сообщение и тело PR носителями НЕ являются** — обезличенные каналы
авторства-субагента, уязвимые к сборке чужой рукой (инцидент 19.07). Гейты их
**не читают** — ни как источник, ни как fallback.

След обезличенного исполнителя: требуется след **персоны, принявшей выход**,
не подпись субагента-автора (тот безымянен по контракту; требовать подпись =
сочинять имя).

### 3.2 Два отказа — две чистые функции

| | Отказ-I «ответственный не назначен» | Отказ-II «приёмка не подтверждена/ложна» |
|---|---|---|
| Предикат | `leadPersona == null` на карточке | поле приёмки отсутствует, `acceptedBy` пуст, или `headRev` не совпадает с ревизией закрытия |
| Вход | карточка (из снимка реестра) | артефакт закрытия |
| Фаза жизненного цикла | `pre-push` / `task:archive` | closure review (LGTM) |
| Режим | **hard сразу** — цена ноль (212/213 карточек поле имеют) | **soft → hard после миграции** активных карточек |
| Код | `exit 1` | soft: `exit 0` + строка в отчёт; hard: `exit 1` |

Контракт функций (форма из brief):

```
checkLeadPersona(card)                      → { code, verdict, reason }
checkAcceptance(closureArtifact, { mode })  → { code, verdict, reason }
  mode ∈ { 'soft', 'hard' }   // переключение — решение постановки, не Date.now()
  verdict ∈ { 'pass', 'soft', 'hard' }
  code = 0 ⟺ verdict ∈ { 'pass', 'soft' }
```

- **Отказ отличается от замечания кодом возврата, не тоном**: отказ = `exit ≠ 0`,
  замечание = `exit 0` + запись в отчёт. Граница проверяется машинно.
- Обе функции **чистые**: вход — данные (карточка/артефакт), никакой сети, `gh`,
  git и `Date.now()` внутри. CLI-обёртка (если понадобится) — тонкая, отдельно.
- Три честных состояния в выводе: зелёный «ответственность есть» · жёлтый
  «подтверждение отсутствует — soft, пропущено» · красный «leadPersona пуст —
  hard-отказ». Жёлтое не выглядит как отказ, красное не пропускается молча.
- **Вшивание в `.husky/**` и `task:archive`/closure-lifecycle — НЕ здесь**: это
  адаптеры интеграции (Phase 4). Блок отдаёт функции и их exit-контракт.

### 3.3 Локальная таблица кодов (до глоссария Phase 3)

| Код | Значение в этом блоке |
|---|---|
| `0` | в порядке, либо замечание (soft) — выход пройден |
| `1` | отказ (hard) — выход непройден |

На Interface Consilium таблица вливается в единый глоссарий exit-кодов спринта
(«отказ отличается кодом, не тоном» — общий закон); до тех пор коды локальны
и явно названы в словаре блока.

## 4. Измерение — `scripts/lib/measure-*.mjs` (работа Р3)

По вердикту m3-measurement с поправкой владельца.

### 4.1 Контракт вычисления

```
computeTeamMetrics(prevSnapshot, currSnapshot, closureArtifacts, window)
  → TeamMetricsReport
```

- **Чистая функция**: тот же вход → тот же отчёт бит-в-бит. Без сети, без git-blame,
  без `Date.now()` внутри предикатов — только timestamps из данных.
- Вход — **два снимка реестра** (материализованные, с провенансом) + множество
  артефактов закрытия с `closedAt ∈ window`. Окно `window = [срез(t−1), срез(t))`.
- Момент — вечерний батч, один раз за цикл; носитель результата — отчёт в
  репозитории. UI/ритуал только читает.

### 4.2 Пять величин и пары против Гудхарта

| Величина | Формула | Источник | Пара |
|---|---|---|---|
| `leadTime` | `closedAt − createdAt` | карточка + артефакт закрытия | `reworkRate` (риторическая, см. 4.4) |
| `ownerlessRate` | доля `leadPersona = ∅` | поле карточки | ограничитель, норма 0% — пары не требует |
| `wip(p)` | число open/in-progress карточек с `leadPersona = p` | срез реестра | `throughput` |
| `throughput(p,w)` | закрытых карточек персоны за окно | артефакты закрытия | `reworkRate` |
| `escalationRate` | доля карточек со сменой `leadPersona` между снимками или переприсвоением `acceptedBy` | два снимка | ограничитель, направление вниз — пары не требует |

Ни одна объёмная величина не выводится в отчёт в одиночку — рядом всегда её пара
или явная ссылка на риторическую пару в аудите дня.

### 4.3 Honest-шапка отчёта и обязательные строки

Шапка: окно `[t−1, t)` · ревизии **обоих** снимков · множество учтённых артефактов
закрытия. Обязательные строки тела:

- **«wip = очередь намерений, не производительность»** — при каждом выводе `wip(p)`.
  Честного источника производительности нет — заявлено явно.
- Секция **«намеренно неизмеримое»** явным текстом, три пункта:
  1) индивидуальная производительность аватара (нет носителя, субагент обезличен);
  2) вклад в строках/коммитах (не носитель, инцидент 19.07);
  3) «качество мышления» персоны (не редуцируется к следу).

### 4.4 Шестая величина — риторическая, вход назван здесь явно

«Доля переделок» (`reworkRate`) — **риторическая** (поправка владельца): живёт
**только в аудите конца дня**, хранимым событием не подкрепляется, парной метрики
не требует (целью не является — не оптимизируется). Открытый хвост эпика 1 требовал
назвать вход её чистой функции при постановке — **называю**:

```
computeReworkRhetoric(prevSnapshot, currSnapshot, closureArtifacts) → доля
```

Вход — **те же два снимка реестра дня и артефакты закрытия за окно дня**, что и у
батча; отдельного хранимого события нет. Переделка наблюдается как один из
фиксируемых переходов между снимками:

1. карточка `closed` в prev → `open`/`in-progress` в curr (переоткрытие);
2. в артефакте закрытия `acceptedBy` снят или переприсвоен (отзыв приёмки);
3. `headRev` артефакта заменён на регресс относительно принятой ревизии.

Функция экспортируется отдельно (`measure-rework-rhetoric.mjs`) и вызывается
**только** аудитом дня; в вечерний батч пяти величин не входит. В отчёте батча
`leadTime`/`throughput` несут пометку «пара — риторическая reworkRate, аудит дня».

## 6. Чего блок сознательно НЕ делает

- Не трогает `registry.json` (живые данные), `.husky/**`, `task-archive`/`closure`
  скрипты — вшивание гейтов = адаптеры интеграции.
- Не определяет форму снимка соседа — живёт на фикстурах-допущениях (EXPECTATIONS).
- Не решает форму стендапа/ритуала показа отчёта — чужая комната.
- Не считает производительность, LOC, «качество мышления» — намеренно неизмеримое.
- Не легализует и не мигрирует карточки — за гейтами владельца.
