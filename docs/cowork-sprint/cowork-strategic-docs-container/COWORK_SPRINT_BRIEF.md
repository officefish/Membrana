## Problem (общая разработка)

Стратегические документы, по которым АИ-агенты сверяют курс (README и не только),
живут россыпью и **дрейфуют**: один факт повторён в нескольких документах и расходится,
потому что чат/прецеденты структурно не держат документы синхронными. Нужен **контейнер**:
три списка данных — **гранулы** (атом-инструкция; литерал ИЛИ чистая функция), **шаблоны**
(композиция гранул с пинами версий), **релизы** (сгенерированные документы, read-only).
**Генераторы** (скрипты) рендерят валидный шаблон → в релизы, невалидный → в эксперименты
(карантин). Источник истины — git; **движки** (Notion/Coda/Affine) лишь подкрепляют. Акт
валидации — **серверный запрос на API office**.

**Естественный шов резки:** данные (A), порождение+валидация (B), поверхность-движок (C) —
три концептуально независимых слоя. Ранняя договорённость об интерфейсах навредила бы:
каждый блок сначала честно отвечает «как устроен МОЙ слой изнутри», а сведение схемы гранулы
и формата релиза — задача Interface Consilium.

### Общие корневые файлы — НЕ трогает никто (изолированная фаза)

- Реальные `README.md`, `AGENTS.md`, `.claude/CLAUDE.md`, `.cursorrules` — переключение на генерацию из гранул только **на интеграции**; в изоляции блок A работает на стаб-копиях.
- `package.json`, `docs/tasks/registry.json`, любые barrel/index-экспорты, `apps/panel` общий вход/роутер.

## Швы для Интерфейс-консилиума (что сведём в Phase 3)

- **Схема гранулы** (литерал vs чистый-вызов; поле версии/пина) — `canon-data` ↔ `generators-validation`.
- **Формат релиза** (что рендерит генератор ↔ что синкает движок) — `generators-validation` ↔ `engine-renderer`.
- **Контракт office-валидации** (запрос/ответ серверного акта) — `generators-validation` ↔ office.
