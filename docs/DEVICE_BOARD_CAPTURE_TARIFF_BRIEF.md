## Вопрос консилиуму (повестка)

Как переопределить модель server-first под тариф с **явным захватом устройства**, где до захвата у сервера **нет никакого контроля**, а после захвата доступно только **выбор + запуск + остановка** уже существующего пользовательского сценария — и что делать с уже реализованным (но не выкаченным) superset'ом v1.0 (edit-lease + pause + setMode)?

---

## Продуктовые требования (от владельца)

1. **Явный процесс захвата.** Пока устройство не захвачено — сервер не имеет контроля ни над устройством, ни над его сценариями. Захват — отдельный явный шаг, **не** побочный эффект команды `run`.
2. **После захвата** сервер может **проигрывать и останавливать** сценарии устройства — как из общего списка устройств (Узлы), так и из самого борда.
3. **Режима редактирования и режима паузы/отладки на сервере в этом тарифе НЕТ** — это следующий тариф.
4. Создание нового сценария с сервера **невозможно**; возможно только проигрывание **уже созданного** пользовательского сценария (выбор из существующих).
5. **На клиенте чётко видно**, захвачено устройство сервером или отпущено.
6. **Вытеснение (pre-emption).** Клиент может сам запустить сценарий **без захвата**. Но если сервер в этот момент захватывает управление — сценарий **останавливается**, и устройство становится **полностью ведомым** от сервера.
7. **Два режима захвата:**
   - **Мягкий:** с клиента можно **запускать и останавливать** сценарии; редактирование и пауза заблокированы.
   - **Жёсткий:** устройство **полностью ведомое** — клиент только смотрит, никакого управления. Переключение normal/alarm с сервера в тариф **не входит** (только select + play + stop).
8. **Enforcement — server-side.** Тарифные ограничения (нет pause/edit/setMode с сервера) проверяются **в gateway/ACL** (whitelist команд по тарифу); скрытие кнопок в UI кабинета — не защита. Иначе «следующий тариф» вырождается в UI-флаг, а произвольный WS-клиент шлёт pause в обход тарифа.
9. **Вытеснение — с активным уведомлением.** Когда сервер захватывает устройство и останавливает клиентский сценарий (п. 6), на клиенте показывается **активный alert** («Сервер захватил управление, сценарий остановлен»), а не только смена badge. Молчаливая остановка звука неотличима от бага.

## Контекст репозитория (факты)

Модель v1.0 (три оси, канон `DEVICE_BOARD_SERVER_FIRST.md`):

| Ось | v1.0 |
|-----|------|
| **Edit lease** | `holder: cabinet \| field \| none` — кабинет acquire/release при входе в редактор |
| **Runtime authority** | `cabinet \| field` — **инициируется командой `run`** (захват неявный) |
| **Follower mode** | `soft \| strict \| null` при `authority=cabinet` |

- Wire: канал `board` (`board.edit-lease`, `board.capture-state`) + расширенный `runtime` (`run/stop/pause/resume/setMode`, `isPaused`, `authority`, `followerMode`).
- Хранение lease: `background-cabinet` (Postgres `NodeScenarioEditLease`), REST acquire/release + WS fan-out.
- UI: `resolveServerFirstFlags()` в `packages/device-board`; follower enforcement в `apps/client`; кнопки Пуск/**Пауза**/Стоп + normal/alarm + «▶ последний трек» на `NodesPage`.
- v1 follower **soft** = поле может pause/stop/setMode, но **run локально запрещён**; **strict** = только просмотр графа.

## Дельты v1.0 → v2 (что консилиум должен переутвердить)

| # | Аспект | v1.0 (реализовано) | v2 (требование) |
|---|--------|--------------------|-----------------|
| D1 | Захват | Неявный: `run` несёт `authority=cabinet` | **Явный двухшаговый:** (1) захват устройства → (2) выбор+запуск сценария. Без захвата — 0 контроля |
| D2 | Возможности сервера | run/stop/pause/resume/setMode + edit-lease | **Только** выбор существующего сценария + **play** + **stop** |
| D3 | Редактирование с сервера | edit-lease активен | **Нет** (следующий тариф) |
| D4 | Пауза/отладка с сервера | pause/resume на NodesPage | **Нет** (следующий тариф) |
| D5 | normal/alarm с сервера | setMode есть | **Нет в тарифе** (подтверждено владельцем) |
| D6 | Follower **soft** | поле: pause/stop/setMode, run запрещён | **поле: start/stop разрешены**; edit+pause заблокированы |
| D7 | Follower **strict → «жёсткий»** | только просмотр графа | **полностью ведомый** (view-only, 0 управления) |
| D8 | Вытеснение | не специфицировано | клиентский run **без захвата** → при захвате сервером **стоп** + полное подчинение + **активный alert** (п. 9) |
| D9 | Индикация на клиенте | badge «Захват: мягкий/строгий» | явный статус **«захвачено сервером / отпущено»** |
| D10 | Enforcement тарифа | UI-флаги (`showRunControls`, flags) | **gateway/ACL whitelist команд** по тарифу (п. 8); UI — вторично |

## Открытые вопросы для ролей (спорные точки)

- **OQ1 (Структурщик/Teamlead).** Реализованный код v1 (edit-lease + pause + setMode) — **удалять** или **прятать за tariff-flag** («следующий тариф»)? Prod ещё не выкачен — окно чистой ревизии открыто.
- **OQ2 (Структурщик/Математик).** Явный захват — это **новый lease на оси runtime authority** (capture-lease: `holder=cabinet`, TTL, heartbeat, как edit-lease), или новое событие `board.capture` (acquire/release)? Переиспользуем канал `board`?
- **OQ3 (Математик).** Вытеснение (D8): при захвате устройства, где уже крутится клиентский сценарий — **жёсткий стоп немедленно** или graceful fade (звук на поле через `audio-engine`)? Инвариант перехода состояния.
- **OQ4 (Математик/Музыкант).** При мягком захвате клиент запустил сценарий — `authority` остаётся `cabinet`? Кто выигрывает при одновременном `run` с обеих сторон? Правило разрешения конфликта.
- **OQ5 (Teamlead).** Семантика **release**: возврат в полную автономию клиента? Авто-release по TTL / потере WS? Что с уже играющим (по команде сервера) сценарием при release.
- **OQ6 (Музыкант/Структурщик).** Выбор сценария с сервера: захват — на уровне устройства; список пользовательских сценариев сервер читает откуда (scenario registry / journal)? Read-path без права на создание/редактирование.
- **OQ7 (Верстальщик).** Индикация D9: единый badge состояния (Отпущено / Захват мягкий / Захват жёсткий) на `NodesPage` и на борде; DESIGN.md — info для мягкого, warning для жёсткого; a11y (aria-live) на смену состояния.
- **OQ8 (Структурщик/Математик).** Wire: нужна ли отдельная команда `capture`/`release` на канале `runtime`/`board`, отличная от `run`? Payload захвата: `{ mode: 'soft'|'hard', holder, sessionId, expiresAt }`.
- **OQ9 (Teamlead/Структурщик).** Терминология: привести RU/EN — **захват/capture**, **мягкий/жёсткий → soft/hard** (переименовать текущий `strict` → `hard`?). Обновить `followerMode` enum.
- **OQ10 (Музыкант/Математик/Teamlead).** **Аварийный стоп и offline при жёстком захвате.** Жёстко захваченное устройство теряет WS → клиент заблокирован, сервер недостижим: устройство, играющее звук, нельзя остановить ничем, кроме питания. SCADA-практика (см. Perplexity-выводы консилиума 2026-06-26): локальный **emergency stop сохраняется всегда**, даже в safety-critical follower. Проработать: (а) что именно разрешено клиенту как emergency stop при жёстком захвате (только stop? выход из ведомого режима?); (б) **auto-release захвата** по потере соединения / TTL heartbeat — семантика и тайминги; (в) индикация «захват потерян» на обеих сторонах.
