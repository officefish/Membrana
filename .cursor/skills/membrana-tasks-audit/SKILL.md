# Membrana tasks audit — разбор устаревших карточек

Движок: `scripts/tasks-audit.mjs`, `scripts/archive-task.mjs`. Реестр:
`docs/tasks/registry.json`, архив: `docs/tasks/archive/<id>.md`.
Концепт движка: [`LINEAR_TASKS_GEAR`](../../../docs/tasks/) (паспорт движка задач).

## Неподвижные правила

- **«Иссью закрыта» — гипотеза, не вердикт.** Архив только после проверки
  свидетельства в main (SHA/PR/файлы). Прецедент 18.07: зонтик #47 накрывал
  незакрытую vdr-hard-gate.
- **Никакой пакетной архивации по механическому признаку.** Каждая карточка —
  индивидуальные `--notes` со свидетельством.
- **Закрытие зонтика ≠ выполнение каждой карточки под ним.** Только повердиктно.
- **Отмена — честно.** NOT_PLANNED в notes так и пишется: «ОТМЕНЕНО, не сделано:
  причина + условие переоткрытия».
- NOT_DONE остаётся active; UNCLEAR/superseded — решение владельца, не агента.
- Работать в отдельном worktree/ветке; коммитить только `docs/tasks/` поимённо.

## Workflow

1. **Свежесть**: `git fetch origin main`. Ветка от main (или rebase на него —
   у registry.json есть merge-driver, сливает чисто).
2. **Аудит**: `yarn tasks:audit` (живой, через gh). Офлайн-fallback
   `tasks:audit:offline` опирается на кэш — только для прикидки. Выход: блоки
   АРХИВИРОВАТЬ (одиночная иссью COMPLETED) / ОТМЕНЕНО (NOT_PLANNED) /
   РАЗОБРАТЬ РУКАМИ (зонтики).
3. **Блок одиночных**: для каждой карточки найти свидетельство —
   `git log origin/main -i --grep="<id|ключи из title>"`, наличие заявленных
   файлов/скриптов в дереве, `gh issue view N` (финальный комментарий часто
   и есть вердикт). Затем `yarn task:archive <id> --notes "<PR/SHA + суть>"`.
4. **Блок отменённых**: прочитать причину закрытия в иссью,
   `task:archive --notes "ОТМЕНЕНО, не сделано: …"`.
5. **Зонтики**: на каждый — параллельный **read-only** субагент с мандатом:
   прочитать registry-запись и промпт карточки (acceptance), найти свидетельства
   в main, вернуть по каждой карточке вердикт DONE / NOT_DONE / CANCELLED /
   UNCLEAR + готовый текст `--notes`. Агенты НЕ пишут — применяет оркестратор.
6. **Применение вердиктов**: дети раньше эпика (dependsOn). DONE и
   CANCELLED-по-условию → архив; NOT_DONE → оставить active, назвать живой
   остаток; UNCLEAR → списком владельцу с вариантами (supersede/переформулировать).
7. **Проверка**: повторный `tasks:audit` — механических кандидатов 0; счётчик
   active упал ровно на число архиваций.
8. **Коммит и мердж**: только `docs/tasks/`; итоги повердиктно в теле коммита;
   мердж через `yarn pr:ship` (норма #700).

## Грабли

- «Воскресшие» карточки: архивная карточка есть, а статус в реестре снова
  active (перетёрт мерджем) → `task:archive --force`.
- Пакет `yarn task:archive` в одном shell-вызове держать ≤5 штук: каждый запуск
  yarn ~15-25с, батч больше упирается в таймаут терминала.
- `gh` может падать `EOF`/таймаутом — повторить, не считать красным.
- Ранние работы (до эпохи pr:ship) влиты прямыми коммитами без PR — grep по
  ключам из title, не только по id и номеру PR.
- Squash-merge: ветка «не в main» по ancestry, но контент в main —
  проверять `git cherry` / `git log --grep`, не `merge-base`.
- Системный дефект схемы: зонтик/эпик не помечен в реестре — аудит ловит их
  эвристикой; находки писать в отчёт, чинить отдельной задачей.
