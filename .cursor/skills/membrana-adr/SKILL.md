# Membrana ADR

Канон: [`docs/adr/README.md`](../../../docs/adr/README.md) · шаблон [`docs/adr/ADR_TEMPLATE.md`](../../../docs/adr/ADR_TEMPLATE.md).

## Когда ADR vs консилиум

- **Консилиум-гейт (НЕ ADR):** новый контракт `@membrana/core` (типы/сокеты/node kinds),
  новый пакет, новые узлы палитры device-board, L-эпик, спор value-vs-ref / границ слоёв.
- **ADR (этот скилл):** решение по готовому канону, рантайм-рефактор без нового core-контракта,
  решение «не вводить контракт» (гейт не нужен), выбор реализации, фиксация развилки.

## Процесс

1. `docs/adr/ADR_TEMPLATE.md` → `docs/adr/ADR-NNNN-<slug>.md` (следующий номер).
2. Заполнить: контекст → **наблюдаемое состояние, грунтованное кодом** (файл:строка @ дата) →
   решения Р1..Рn с явными **границами** → DoD → out of scope.
3. Статус **DRAFT** (merge ≠ решение) до LGTM владельца. Ревью `code-review --staged`, шип `pr:ship`.
4. LGTM → **ACCEPTED**; замена → **SUPERSEDED-by-ADR-MMMM**. Обновить реестр в `docs/adr/README.md`.

## Правила

- Каждое решение грунтовать конкретным кодом (не абстракции); строчные ссылки — с датой-якорем.
- Если по ходу вскрывается новый core-контракт/узел палитры → эскалировать в консилиум-гейт, не ADR.
- ADR не заменяет Issue/task-промпт/PR-LGTM — это запись решения, реализация отдельно.

## Output

Сказать: путь к ADR, статус (DRAFT/ACCEPTED), 3 буллета решений, что дальше (LGTM/реализация).
