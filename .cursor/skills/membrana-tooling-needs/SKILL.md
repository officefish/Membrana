# Membrana tooling needs — ретроспектива трения сессии

Прецедент-эталон: ретроспектива 2026-07-13 → спринт `agent-tooling-friction`
(#433): 4 инструмента (`insight:drift`, `llm:probe`, gitignore ревью-артефактов,
proxy-чистка тестов), каждый снял реальную повторявшуюся боль.

## When to use

- Владелец пишет «tooling needs» (обычно в конце плотной сессии).
- Закрыт спринт/эпик и трение ещё свежо в контексте.

## When NOT to use

- Регистрация уже одобренных инструментов → `membrana-task-lifecycle`.
- Общая ретроспектива дня с оценками → `membrana-team-evening-feedback`.

## Ключевой принцип

**Только evidence из текущей сессии.** Каждая боль обязана иметь конкретный
эпизод: «делал X вручную N раз», «потерял M минут на ложное Y», «обошёл Z
костылём». Теоретический тулинг без эпизода не предлагается. Реализация не
начинается без слова владельца.

## Шаги

1. **Собрать фрикции по фактам сессии** — пройтись по своей работе и выписать:
   - команды/последовательности, повторённые вручную ≥2 раз;
   - ложные красные (тесты/чеки, падавшие не по делу) и время на диагноз;
   - обходы и костыли (временные exclude, ручные правки одного и того же,
     самописные однострочники, которые пришлось писать заново);
   - ловушки окружения (кодировки, прокси, worktree, гонки параллельных сессий);
   - места, где пришлось нарушить/обойти регламент из-за отсутствия инструмента.
2. **Проверить, что не существует** — перед каждым предложением: греп по
   `package.json` (yarn-скрипты), `scripts/`, `.cursor/skills/README.md`,
   `.githooks/`. Дубликат существующего = не предложение, а напоминание о нём.
3. **Категоризовать** каждую боль: `скрипт` (детерминированная механика) /
   `хук` (git/harness-автоматика) / `скилл` (playbook для агента) /
   `норма` (строка в регламент/канон) / `фикс` (правка существующего).
4. **Оценить**: размер (XS/S/M), что конкретно снимает (одной фразой),
   приоритет = частота боли × стоимость обхода. Отсортировать по убыванию.
5. **Выдать список** в чат: `№ · название · категория · размер · какую боль
   снимает (с эпизодом-evidence) · эскиз реализации (1-2 фразы)`. В конце —
   рекомендуемый порядок внедрения и форма (S-пакетом одним спринтом /
   отдельными pr:ship / консилиум, если есть архитектурная развилка).
6. **Ждать слова владельца.** По «заводи»/«идём в спринт» → регистрация по
   `membrana-task-lifecycle` (консилиум-гейт оценить честно: скрипты по готовому
   канону обычно inline, пакет инструментов — как #433).

## Anti-patterns

- Предлагать инструменты «на вырост» без эпизода из сессии.
- Начинать реализацию в той же реплике, не дождавшись решения владельца.
- Пропустить шаг проверки существующего (предложить уже существующий скрипт).
- Смешивать с ретроспективой качества дня (это `team-evening-feedback`).
- Забыть уроки-нормы: иногда правильный выход — строка в регламент, а не код.

## Handoff

Одобренные пункты → `membrana-task-lifecycle` (регистрация) или прямой
`pr:ship` для XS-фиксов. Уроки-нормы → соответствующий регламент/канон + память.
