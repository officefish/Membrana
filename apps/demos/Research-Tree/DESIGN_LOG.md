# Research Tree — журнал проектирования (дайджест треда)

Запись о том, как в одном треде собиралась модель графа знаний проекта. Позволяет
войти в содержание без чтения переписки: отправная точка, слои модели в порядке
их появления, сквозные принципы, произведённые артефакты, текущее состояние.

---

## Отправная точка

Идея — дерево исследований из игры Ancestors: карта того, что проект **вскрывает**
по мере роста. Прежде всего про продакшн и разработку; математика детекции — как
одна из категорий. Формально это направленный ациклический граф зависимостей плюс
две надстройки из механики игры: **туман** (узел не виден, пока не подошёл) и
**двухфазное освоение** (синапс → закрепление). Второе легло на существующую
дисциплину stage-gate почти без натяжки. Решение на старте: **data-first**,
единственный источник истины — JSON, рендер вторичен.

## Слои модели (в порядке появления)

| # | Слой | Ключевая мысль |
|---|------|----------------|
| 1 | Узлы и состояния | Узел `{тип, ветка, requires}`; 4 состояния `fog→available→exploring→established`; `fog`/`available` выводятся из графа. Рёбра пересекают типы: знание → слой → сервис. |
| 2 | Эпохи ↔ tier продукта | Два порога — `entered` (≥1 узел закреплён) и `tier_reached` (закреплён tier_node) — снимают цикличность «эпоха от одного узла». `balanced` vs `breakthrough` = технический долг. |
| 3 | Полный стек + грань facets | `skill × process` = 4 квадранта. Ось совпала со швом Claude Code (hard/внутр) ↔ этот Project (soft/внешн). Баланс по квадрантам как диагностика портфеля. |
| 4 | Контур | Эпистемическая ось «ясность пути»: `возможность→горизонт→план→реальность`. Ортогональна `state`, транзитивна (узел не яснее слабейшей предпосылки). |
| 5 | Материальность M1–M4 | Квадрант × материальность = 16 ячеек. Материя→субстрат→структура→действие. Каденция capex→opex, цена разворота. Знание — вне шкалы. |
| 6 | Артефакты-спутники | Зернистость внутри узла: коллекция с прогресс-баром. Many-to-many. Энергия живёт на артефактах, не на абстрактном узле. |
| 7 | Ресурсы | Инфра (потолок), токены (поток, реальные тарифы), стамина (WIP-лимит). Виртуальная команда множит токен-поток, **не** стамину. |
| 8 | Денежная ось | Три формы биллинга: metered / subscription / overhead. Базовый burn — фиксированный пол до любой работы. |
| 9 | Полосы дня + каденция | Пик / поддержка / фон; регенерирующая квота подписок; 3–4 спринта/день; `sprint ≈ артефакт-шаг`. |
| 10 | Венчурный слой | Стамина — вектор по квадрантам; `capacity_gate` эпох; жизненный цикл `capital` vs `sustained` (upkeep + распад). |

## Сквозные принципы (что всплыло по ходу)

- **Единый источник истины.** Граф — в JSON; документы и рендеры ссылаются, не
  дублируют.
- **Не выдумывать числа.** Приватное (баланс, спеки, пороги, стамина) = `null`,
  заполняется из реального учёта.
- **Цена ≠ доказательство.** Токены покупают попытки; узел закрывает `gate`, не
  трата. `spent > budget` при открытом gate — отдельное честное состояние.
- **Энергию тарифицирует только план.** Для горизонта и возможности цена не
  определена — путь с пробелами не тарифицируется.
- **Биндит серийный критический путь, не ресурсы.** По калибровке — запас по
  токенам и стамине есть; потолок ставит каденция спринтов (Амдаль). Фон
  параллелит необязательное, серийное ядро не ускоряет.
- **Tool sprawl.** Подписки и агенты множат машинную пропускную способность, не
  стамину; ёмкость сверх WIP-лимита простаивает.
- **Венчурная беговая дорожка.** `sustained`-узлы копят стоячий upkeep; подъём по
  эпохам гейтится ёмкостью удержать развёрнутое (`capacity_gate ≈ Σ upkeep`).
- **Каналов восприятия мало.** Лишние оси рендера уходят в фильтры, не в
  постоянные визуальные каналы.

## Конвейер и контекст

Replit прототипирует изолированные элементы → импорт в `apps/demos/Research-Tree`
→ репо-агенты дорабатывают → Kodik мониторит код и документы. Контекст —
демонстрационная версия под формат Kodik Launchpad, не продукт целиком.
