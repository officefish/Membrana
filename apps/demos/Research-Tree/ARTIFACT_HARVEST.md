# AGENT_TASK — Research Tree: заземлить артефакты на репозиторий

## Контекст

Граф в `apps/demos/Research-Tree/membrana-knowledge-graph.json`. Артефакты —
собираемые спутники узлов (см. `KNOWLEDGE_GRAPH_SPEC.md` → «Артефакты»). Сейчас
заполнены 19 из 60 узлов; остальное — место для реальных файлов репозитория.
У тебя есть доступ к репо; у автора модели его не было.

## Цель

Привязать к узлам реальные деливераблы как артефакты — с настоящими путями,
видами и статусами. **Не выдумывать файлы и пути.**

## Задачи

1. Для каждого узла найти в репозитории конкретные тела, которые его воплощают, и
   добавить объект в массив `artifacts`:
   `{ id, title, node, also?, kind, status, path }`.
   Ориентиры (сверять с фактической структурой, не принимать на веру):
   - `layer.*`, benchmark, детекторы → `packages/services/*`, каркас детекторов;
   - `stack.nestjs`/`stack.prisma-pg` → `packages/background-*`;
   - `stack.vite-react` → `apps/client`, `apps/cabinet`;
   - `stack.mjs-scripts`, `process.*` → `scripts/*`, регламенты в `docs/`;
   - `comm.*` → `docs/` (white paper, pitch, one-pager, грант-отчёт).
2. `status` — из реальности: `collected` (есть), `in_progress` (частично),
   `missing` (требуется узлом, но нет).
3. `path` — настоящий путь. Файла нет → не выдумывать: `status:"missing"` + `note`.
4. `kind` ∈ `dataset|config|module|report|doc|script|command|hardware`.

## Не-цели

- Менять состояния узлов произвольно, менять схему, выдумывать файлы/пути.
- Заполнять приватные числа (ресурсы, пороги).
- Добавлять новые узлы.

## Валидатор (зелёный до PR)

Тот же, что в основном `AGENT_TASK.md`: ацикличность, целостность ссылок,
инвариант «`exploring/established` ⇒ предпосылки закрыты» и «`established` ⇒ нет
`missing` требуемых артефактов». Если узел `established`, а его артефакт оказался
`missing` — значит либо узел не `established`, либо артефакт не обязателен:
разрешить пометкой, не подгонкой.

## Критерии готовности

1. Существенно больше узлов имеют ≥1 артефакт с реальным `path`.
2. Статусы отражают факт репозитория.
3. Валидатор проходит без ошибок.
4. Изменения в отдельном PR с описанием, какие узлы заземлены и чем.
