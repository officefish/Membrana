# SPRINT_SEED — Research Tree: запуск спринта через консилиум

Закваска для нового спринта по линии Research-Tree. Не план — **основание** для
него: онбордит виртуальную команду в сделанное, ставит вопрос на консилиум,
задаёт каркас планирования. План спринта рождается из `yarn consilium` +
`yarn plan:week`, а не из этого файла.

---

## 1. Что уже сделано (онбординг команды)

Перед обсуждением — общий контекст. Канон лежит в `apps/demos/Research-Tree/`.

**Модель.** Граф знаний проекта: узлы (`knowledge|capability|service|process`)
на ортогональных осях — ветка, эпоха (лестница зрелости продукта, два порога +
двойной гейт), квадрант (`skill×process` × материальность M1–M4 = 16 ячеек),
состояние (`fog→available→exploring→established`, готовность пути) и контур
(`возможность→горизонт→план→реальность`, ясность пути). К узлам орбитируют
**артефакты** — собираемые спутники. Под графом — слой **ресурсов**: инфра
(потолок), `spend` (metered/subscription/overhead), стамина (вектор по
квадрантам, WIP-лимит), команда, полосы дня, каденция спринтов. Узлы бывают
`capital` (открыл и держится) и `sustained` (дренит ресурс, деградирует).

**Единый источник истины** — `membrana-knowledge-graph.json`. Конвенции —
`KNOWLEDGE_GRAPH_SPEC.md`. Data-first: рендер вторичен и платформо-зависим.

**Файлы в папке:**
- `membrana-knowledge-graph.json` — граф (60 узлов, 19 артефактов, ресурсы).
- `KNOWLEDGE_GRAPH_SPEC.md` — конвенции и правила вывода.
- `README.md` — вводная; `ONE_PAGER.md` — для внешнего наблюдателя.
- `AGENT_TASK.md` — репо-агент: verify-сверка + PR.
- `RENDER_SPEC.md` — Replit: бриф на прототип рендера.
- `ARTIFACT_HARVEST.md` — репо-агент: заземление артефактов на файлы.
- `DEMO_STACK.md` — технический стек демо, архитектура состояния.
- `STATE_ARCHITECTURE.md` — решение по state-слою.
- `DESIGN_LOG.md` — журнал проектирования модели.

**Диагнозы, которые стоит знать команде:**
- Перекос портфеля: `hard/внутр` глубок, оба внешних квадранта почти пусты.
- Фронтир — `service.single-node-detector`, держит закрытый шлюз precision/recall.
- Текущий лимитер — серийный критический путь (каденция спринтов), не стамина:
  запас по токенам и стамине есть, но в скорость критпути не конвертируется.
- Пробелы: артефакты на 19 из 60 узлов, приватные числа ресурсов `null`.

## 2. Цель спринта

Собрать оставшиеся полезные документы и промпты, чтобы демо стало связным к
защите, а workstream — самоходным. Три кандидата от автора закваски:

- **`OUTER_RING_SEED`** — ночной фоновый агент предлагает кандидатов во внешнее
  кольцо (`возможность`/`горизонт`) на утверждение основателя.
- **`CONSILIUM_REVIEW`** — прогон графа через команду на структурные дыры и
  спорные классификации.
- **`RESOURCE_LEDGER`** — подцепить реальный токен-расход в `cost`/`draws`.

Это **кандидаты, не мандат**. Что из них в спринт, в каком порядке и чего ещё не
хватает — решает консилиум.

## 3. Повестка консилиума

Вопросы на обсуждение пятью ролями:

1. **Приоритет.** Какие документы реально нужны к защите демо, какие позже?
2. **Пробелы.** Чего нет в наборе? Кандидаты на проверку: отдельный
   `validate.py`, встроенный data-snapshot для рендера, `CONTRIBUTING` для
   агентов, `DATA_DICTIONARY` по полям графа.
3. **Риски.** Плотность кодировки рендера; платформенный разрыв Kodik/Replit;
   тонкость внешних квадрантов в демо.
4. **Распределение ролей** (предложение, не догма):
   - *Teamlead* — приоритизация, план, не-цели.
   - *Структурщик* — схема графа, целостность документов.
   - *Математик* — производные (контур/state), валидатор, инварианты, экономика.
   - *Музыкант* — нарратив: one-pager, демо-история, ритм изложения.
   - *Верстальщик* — рендер и Replit-бриф, визуальная кодировка.

## 4. Каркас планирования

- **Объём:** документы и промпты, не код продукта и не сам рендер (его строит
  Replit отдельно по `RENDER_SPEC`).
- **Каденция:** 3–4 спринта/день. По желанию — литературная арка
  (завязка/развитие/кульминация/финал) на под-спринты.
- **Полоса:** дневной пик — приоритизация и каркасы; фон (ночь) — черновики
  кандидатных документов дешёвым стеком на утверждение утром.
- **Не-цели спринта:** заполнение приватных чисел; рендер; продукт целиком;
  изменение схемы графа без согласования.

## 5. Запуск

```bash
yarn consilium "Research-Tree демо: какие документы и промпты собрать в этом \
спринте, в каком приоритете, чего не хватает в наборе для связной защиты?"
yarn plan:week   # либо планирование спринта по принятому консенсусу
```

Зарегистрировать по `docs/prompts/TASK_PROMPT_WORKFLOW.md`, занести в
`docs/tasks/registry.json`.

## 6. Критерии готовности спринта

1. Консилиум дал консенсус: приоритет и состав документов зафиксированы.
2. План спринта составлен (объём, роли, под-спринты, не-цели).
3. Согласованный набор документов/промптов в черновиках, по дом-стилю.
4. Задание зарегистрировано в реестре.
